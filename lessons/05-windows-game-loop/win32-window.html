Creating a Win32 Window

Every game needs a window to draw into, and on Windows that means talking to the Win32 API. Before DirectX can render a single triangle, we need a live window and its handle. In this lesson you'll register a window class, create a real window, and put it on screen — laying the exact foundation our engine's platform layer will sit on.

What you'll learn
  • What the Win32 API is and why wWinMain replaces main for GUI apps
  • How to fill and register a WNDCLASSEX with RegisterClassEx
  • How to create a window with CreateWindowEx and control its look with styles
  • What an HWND and an HINSTANCE handle actually are
  • Why this window becomes our DirectX render surface

The Win32 API and handles

The Win32 API is the C-style interface the Windows operating system exposes for creating windows, handling input, drawing, and much more. It has been around for decades, so it looks a little old-fashioned — lots of capitalised type names and functions that take many parameters — but it is rock solid and it is exactly what DirectX expects to attach to.

The most important idea in Win32 is the handle. A handle is an opaque number that identifies a resource owned by the operating system. You never look inside it; you just pass it back to Windows when you want to act on that resource. Two handles show up constantly:

HandleMeaning
HINSTANCEIdentifies your running program (its module) in memory. Windows hands it to you at startup.
HWNDIdentifies a specific window. This is the value DirectX needs later to create its swap chain.
Handles are just IDs

Think of an HWND like a coat-check ticket. The ticket is not the coat — but hand it back and Windows returns the right one. You'll store the HWND in our engine's Window class and pass it around whenever something needs to talk to the window.

The entry point: wWinMain

A console program starts at main. A Windows GUI program starts at WinMain instead — or, for wide (Unicode) strings, wWinMain. We'll use the wide version everywhere, because modern Windows is Unicode and DirectX examples assume it. Its signature is fixed by Windows:

ParameterWhat it is
hInstanceYour program's module handle.
hPrevInstanceAlways nullptr on modern Windows (a 16-bit relic). Ignore it.
lpCmdLineThe command-line arguments as a wide string.
nCmdShowA hint for how to show the window (normal, minimised, maximised).

Step 1 — Register a window class

Before you can create a window, you must describe what kind of window it is by filling a WNDCLASSEX structure and registering it. The class ties together an icon, a cursor, a background, and — most importantly — a window procedure: the function Windows calls whenever something happens to the window. We'll build a real message loop in the next lesson; for now we supply a minimal procedure that just lets Windows do the default thing and handles the close message.

Step 2 — Create and show the window

CreateWindowEx is the workhorse. It takes the class name you registered, a title, a set of style flags that control borders and buttons, a position, a size, and your HINSTANCE. It returns the new HWND. Then ShowWindow makes it visible and UpdateWindow forces an initial paint.

Here is a complete, compilable program that puts a window on screen:

// Build (MSVC): compile as a Windows app, not a console app.
#include <windows.h>

// The window procedure: Windows calls this for every message.
// We handle destruction and defer everything else to Windows.
LRESULT CALLBACK WndProc(HWND hWnd, UINT msg, WPARAM wParam, LPARAM lParam)
{
    switch (msg)
    {
    case WM_DESTROY:
        PostQuitMessage(0); // tell our (future) message loop to exit
        return 0;
    default:
        return DefWindowProc(hWnd, msg, wParam, lParam);
    }
}

int WINAPI wWinMain(HINSTANCE hInstance, HINSTANCE hPrevInstance,
                    PWSTR lpCmdLine, int nCmdShow)
{
    const wchar_t* kClassName = L"EngineWindowClass";

    // 1) Describe and register the window class.
    WNDCLASSEX wc = {};                       // zero-initialise every field
    wc.cbSize        = sizeof(WNDCLASSEX);
    wc.style         = CS_HREDRAW | CS_VREDRAW; // repaint on resize
    wc.lpfnWndProc   = WndProc;                 // our message handler
    wc.hInstance     = hInstance;               // owning module
    wc.hCursor       = LoadCursor(nullptr, IDC_ARROW);
    wc.hbrBackground = (HBRUSH)(COLOR_WINDOW + 1);
    wc.lpszClassName = kClassName;

    if (!RegisterClassEx(&wc))
    {
        MessageBox(nullptr, L"Failed to register window class.",
                   L"Error", MB_ICONERROR);
        return -1;
    }

    // 2) Create the window. WS_OVERLAPPEDWINDOW gives a normal
    //    title bar, border, and minimise/maximise/close buttons.
    HWND hWnd = CreateWindowEx(
        0,                            // extended styles (none)
        kClassName,                   // class we just registered
        L"Learn C++ & DirectX",       // title-bar text
        WS_OVERLAPPEDWINDOW,          // window style
        CW_USEDEFAULT, CW_USEDEFAULT, // x, y position
        1280, 720,                    // width, height in pixels
        nullptr,                      // no parent window
        nullptr,                      // no menu
        hInstance,                    // owning module
        nullptr);                     // no extra creation data

    if (hWnd == nullptr)
    {
        MessageBox(nullptr, L"Failed to create window.",
                   L"Error", MB_ICONERROR);
        return -1;
    }

    // 3) Show it and force the first paint.
    ShowWindow(hWnd, nCmdShow);
    UpdateWindow(hWnd);

    // A real app pumps messages here — that's the next lesson.
    // For now, keep the window alive until the user closes it.
    MSG msg = {};
    while (GetMessage(&msg, nullptr, 0, 0) > 0)
    {
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }

    return static_cast<int>(msg.wParam);
}
Zero-initialise your structs

Writing WNDCLASSEX wc = {}; sets every field to zero first, so any field you forget has a safe default. Win32 structs are large, and a stray uninitialised member is a classic source of crashes.

Understanding the styles

The WS_OVERLAPPEDWINDOW value is a bundle of flags OR-ed together: a caption bar, a thick sizing border, a system menu, and the minimise/maximise buttons. That is the standard desktop-app look. If you later want a borderless full-screen game window, you'll swap this for WS_POPUP — but a normal resizable window is perfect while we develop the engine.

Client size vs. window size

The 1280×720 you pass is the whole window, including the border and title bar — so the drawable client area is a bit smaller. When we wire up DirectX we'll use AdjustWindowRect (or read the real client size with GetClientRect) so the back buffer matches the pixels we can actually draw into.

Building as a Windows app, not a console app

Because we use wWinMain instead of main, the linker must target the Windows subsystem. In a Visual Studio project this is a project setting; from the command line you pass it to the linker:

cl /EHsc /std:c++17 WinMain.cpp /link /SUBSYSTEM:WINDOWS user32.lib

If you accidentally build as a console app, you'll get a linker error complaining it can't find WinMain or main — that's the tell-tale sign the subsystem is wrong.

Why this matters for DirectX

That HWND is not just a pretty rectangle. When we initialise Direct3D 11, the swap chain description (DXGI_SWAP_CHAIN_DESC) has an OutputWindow field, and we hand it exactly this handle. The window is our render surface: DirectX fills a back buffer and then "presents" it to this window. Everything visual our engine ever does ends up here.

Where this goes in the engine

We'll wrap all of this in a small Window class that owns the HWND and exposes its width and height. The rest of the engine — renderer, input, the game loop — will simply ask the Window for its handle. Keeping platform code in one place is exactly how Unity and Unreal isolate the operating system from the game.

Recap
  • The Win32 API creates windows on Windows, and everything is referenced through opaque handles like HINSTANCE (your program) and HWND (a window).
  • GUI programs start at wWinMain, which receives the HINSTANCE and a show-command hint.
  • Fill a WNDCLASSEX (including a window procedure) and register it with RegisterClassEx before creating anything.
  • CreateWindowEx builds the window and returns its HWND; styles like WS_OVERLAPPEDWINDOW control its appearance; ShowWindow and UpdateWindow put it on screen.
  • Build with the /SUBSYSTEM:WINDOWS linker setting, and remember the HWND is the render surface DirectX will draw into.
