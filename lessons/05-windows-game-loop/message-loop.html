The Message Loop & Input

Last lesson you created a Win32 window and it appeared on screen — but it just sat there, deaf to the world. That is because Windows is event-driven: every click, keypress and resize is a message waiting in a queue for you to collect. In this lesson we build the loop that pumps those messages and the callback that reacts to them. This is the beating heart every game — and our engine — is built around.

What you'll learn
  • How the Windows message queue and the WndProc callback work together
  • The common messages you will handle: WM_DESTROY, WM_CLOSE, WM_KEYDOWN/WM_KEYUP, WM_SIZE and WM_MOUSEMOVE
  • Why games use the non-blocking PeekMessage loop instead of GetMessage
  • What TranslateMessage and DispatchMessage actually do

Windows is event-driven

In a console program from earlier lessons, you were in charge: your code ran top to bottom and asked for input when it wanted it. A GUI program flips that around. The operating system watches the hardware — keyboard, mouse, the window's title bar — and whenever something happens it packages the event into a message and drops it into your window's message queue. Your job is to keep emptying that queue.

A message is just a small bundle of numbers. The important pieces are a message ID (like WM_KEYDOWN) and two parameters, wParam and lParam, whose meaning depends on the message. For a key press, wParam holds which key; for a resize, lParam packs the new width and height.

The WndProc: your window's brain

When you registered your window class last lesson, you gave it a function pointer called lpfnWndProc. That function — the window procedure — is where messages are actually handled. Windows calls it for you, one message at a time. Every message you do not care about should be passed to DefWindowProc, which supplies sensible default behavior (drawing the frame, moving the window, and so on).

// The window procedure. Windows calls this once per dispatched message.
LRESULT CALLBACK WndProc(HWND hWnd, UINT msg, WPARAM wParam, LPARAM lParam)
{
    switch (msg)
    {
    case WM_CLOSE:
        // The user clicked the X or pressed Alt+F4. We choose to destroy
        // the window; DestroyWindow will then send us WM_DESTROY.
        DestroyWindow(hWnd);
        return 0;

    case WM_DESTROY:
        // The window is being torn down. Post WM_QUIT so our message
        // loop knows it is time to exit.
        PostQuitMessage(0);
        return 0;

    case WM_SIZE:
    {
        // The client area was resized. Low/high word of lParam hold the
        // new width and height in pixels. We'll resize our swap chain here later.
        UINT width  = LOWORD(lParam);
        UINT height = HIWORD(lParam);
        // TODO(engine): notify the renderer so the back buffer matches.
        return 0;
    }

    case WM_KEYDOWN:
        // wParam is the virtual-key code of the pressed key.
        if (wParam == VK_ESCAPE)
            DestroyWindow(hWnd); // Esc quits, handy during development.
        return 0;

    case WM_KEYUP:
        // Fired when a key is released. We'll feed this to our input system.
        return 0;

    case WM_MOUSEMOVE:
    {
        // Cursor moved inside the client area. Coordinates are in lParam.
        int mouseX = GET_X_LPARAM(lParam);
        int mouseY = GET_Y_LPARAM(lParam);
        return 0;
    }
    }

    // Everything we didn't handle goes to the default handler.
    return DefWindowProc(hWnd, msg, wParam, lParam);
}
Always call DefWindowProc for the rest

If you forget the final return DefWindowProc(...) and instead return 0 for every message, your window will misbehave in strange ways — no title-bar dragging, broken resizing, no close animation. Handle what you need, then defer everything else.

To read WM_MOUSEMOVE coordinates with the GET_X_LPARAM / GET_Y_LPARAM macros, include <windowsx.h> at the top of the file alongside <windows.h>.

The WM_CLOSE → WM_DESTROY → WM_QUIT chain

These three messages trip people up, so let's be precise about the order:

MessageWhen it arrivesWhat we do
WM_CLOSEUser clicks X / presses Alt+F4Call DestroyWindow (or ask "Save first?")
WM_DESTROYThe window is actually being destroyedCall PostQuitMessage(0)
WM_QUITPosted by PostQuitMessageOur loop sees it and exits
WM_QUIT never reaches WndProc

WM_QUIT is special: it is not delivered to your window procedure. It is caught directly by the message loop, which is exactly how the loop knows to stop. That is why PostQuitMessage is the clean way to end a program.

GetMessage vs. PeekMessage

Now for the loop that drains the queue. There are two ways to pull a message out, and the difference decides whether you are writing a form or a game.

GetMessage is blocking. If the queue is empty it puts your thread to sleep until a message shows up. That is perfect for a calculator or a text editor: nothing needs to happen while the user is idle, so sleeping saves power.

// Classic blocking loop — great for normal apps, WRONG for a game.
MSG msg = {};
while (GetMessage(&msg, nullptr, 0, 0) > 0) // returns 0 on WM_QUIT, -1 on error
{
    TranslateMessage(&msg); // turn key events into WM_CHAR where relevant
    DispatchMessage(&msg);  // hand the message to our WndProc
}
// msg.wParam holds the exit code passed to PostQuitMessage.

A game cannot afford to sleep. Even when the player is perfectly still we still need to advance physics, animate, and render the next frame. So games use PeekMessage, which is non-blocking: it checks the queue and returns immediately whether or not a message was waiting. That lets us process any pending input and then fall through to the game work every single frame.

// Non-blocking loop — the shape every real-time engine uses.
bool running = true;
MSG  msg = {};

while (running)
{
    // Drain ALL pending messages without blocking. PM_REMOVE pops each
    // message off the queue so we don't see it twice.
    while (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        if (msg.message == WM_QUIT)
        {
            running = false;
            break; // stop pumping; we're shutting down
        }
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }

    // ---- No messages left: do one frame of game work ----
    // Update(deltaTime);   // move objects, run systems
    // Render();            // draw the frame with DirectX
}
Drain the queue, then render once

Notice the two loops: the inner while (PeekMessage...) empties every message that piled up, then a single Update/Render pass runs. If you rendered inside the inner loop you'd draw multiple times when several messages arrive at once — wasted work and stutter.

What TranslateMessage and DispatchMessage do

Two calls appear in every loop, and it is worth knowing why:

  • TranslateMessage looks at raw key events (WM_KEYDOWN/WM_KEYUP) and, when they represent a typed character, generates a WM_CHAR message with the actual letter — respecting Shift, Caps Lock and the keyboard layout. If you only care about physical keys you can skip it, but it costs nothing and text input needs it.
  • DispatchMessage is the call that actually invokes your WndProc for that message. Without it, you would pull messages off the queue and throw them away — your window procedure would never run.
Don't stall the pump

Everything in your loop — updating, rendering, loading — runs on the same thread that pumps messages. If a single frame takes too long, Windows notices the queue isn't being drained and paints your window with "(Not Responding)". Keep per-frame work bounded and push heavy jobs onto worker threads later.

How this feeds the engine

Right now we react to input directly inside WndProc, which is fine for quitting on Escape. But scattering game logic through a giant switch statement does not scale. The professional pattern — the one Unity and Unreal use — is to let WndProc do one job: record raw state. On WM_KEYDOWN we set a flag, on WM_KEYUP we clear it, on WM_MOUSEMOVE we store the cursor position. Then our gameplay code asks a small Input class questions like "is the W key held?" during Update.

We will build exactly that Input abstraction in an upcoming module, and the WM_SIZE handler above is where our DirectX swap chain will resize its back buffer so the 3D image always matches the window. For now, the takeaway is the structure: a WndProc that classifies messages, and a PeekMessage loop that pumps them and then runs a frame. Every lesson from here builds inside that loop.

Recap
  • Windows is event-driven: the OS posts messages to your window's queue, and you must keep draining them.
  • The WndProc callback handles messages in a switch, and passes anything it ignores to DefWindowProc.
  • The close sequence is WM_CLOSE → DestroyWindow → WM_DESTROY → PostQuitMessage → WM_QUIT, which ends the loop.
  • GetMessage blocks (good for apps); games use non-blocking PeekMessage with PM_REMOVE so a frame renders even when idle.
  • TranslateMessage produces character messages; DispatchMessage actually calls your WndProc.
  • Keep WndProc lean — record input state there and let the game ask an Input class about it later.
