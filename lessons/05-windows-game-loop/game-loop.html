The Game Loop & Delta Time

In the last lesson you learned to pump Windows messages and read input. But a window that only wakes up when a message arrives is not a game — a game is alive, redrawing itself dozens of times per second even when nothing is happening. This lesson builds the beating heart of every engine: a real-time loop that reads input, updates the world, and renders a frame, over and over. We will also learn to measure delta time precisely, so your objects move at the same real-world speed whether the game runs at 30, 60, or 240 frames per second.

What you'll learn
  • Why a real-time loop uses PeekMessage instead of GetMessage.
  • The three phases of every frame: input → update(dt) → render.
  • How to measure delta time accurately with QueryPerformanceCounter.
  • Why you multiply movement by dt for frame-rate independence.
  • The difference between a variable and a fixed timestep.

From a message loop to a game loop

The message loop from the previous lesson called GetMessage, which blocks — it puts your program to sleep until Windows has something to deliver. That is perfect for a text editor, but fatal for a game. If nothing on the keyboard moves, your game freezes too. We want the opposite: keep running flat out, and only handle messages when they happen to be waiting.

The fix is PeekMessage. Unlike GetMessage, it returns immediately — with a message if one is queued, or with zero if the queue is empty. We drain every pending message in an inner loop, then fall through to update and render our world. That gives us the classic structure below.

while (running)
{
    MSG msg = {};
    // Handle EVERY message waiting in the queue, but never block.
    while (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        if (msg.message == WM_QUIT)
            running = false;

        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }

    float dt = timer.Tick();  // seconds elapsed since the last frame

    Update(dt);   // advance the simulation by dt
    Render();     // draw the current state of the world
}

Those three lines after the message pump are the whole game. Everything an engine does — physics, animation, AI, audio, rendering — is just work scheduled inside Update and Render. When we build our component-based engine, this exact loop will call Update(dt) on every component in the scene.

Always check for WM_QUIT

When the user closes the window, your WM_DESTROY handler calls PostQuitMessage, which places a WM_QUIT message in the queue. If you forget to test for it and set running = false, your loop spins forever and the process never exits cleanly.

Measuring time correctly

A frame takes some amount of real-world time to compute. On a fast machine it might be 4 milliseconds; on a slow one, 33. We call that gap the delta time, usually written dt and measured in seconds. To move a spaceship at a constant speed, we need to know exactly how much time passed since the previous frame.

You might reach for GetTickCount or std::chrono. For a game, the professional choice on Windows is the high-resolution performance counter, exposed through two API calls:

  • QueryPerformanceFrequency — how many counter ticks happen per second (a fixed value for the life of the program).
  • QueryPerformanceCounter — the current tick count, a steadily increasing 64-bit number.

Divide the ticks that elapsed this frame by the frequency, and you get seconds. Here is a small, reusable timer that wraps it up cleanly:

#include <windows.h>

class Timer
{
public:
    Timer()
    {
        // Frequency is constant, so we only need to read it once.
        QueryPerformanceFrequency(&m_frequency);
        QueryPerformanceCounter(&m_lastCount);
    }

    // Returns the seconds elapsed since the previous call to Tick().
    float Tick()
    {
        LARGE_INTEGER now;
        QueryPerformanceCounter(&now);

        // Raw ticks that passed during this frame.
        long long elapsedTicks = now.QuadPart - m_lastCount.QuadPart;
        m_lastCount = now;  // remember for next frame

        // ticks / (ticks per second) = seconds.
        return static_cast<float>(elapsedTicks) /
               static_cast<float>(m_frequency.QuadPart);
    }

private:
    LARGE_INTEGER m_frequency;  // counter ticks per second
    LARGE_INTEGER m_lastCount;  // counter value at the last Tick()
};

LARGE_INTEGER is a Windows union that holds a signed 64-bit value in its QuadPart member. We use it because the raw counter can grow past the range of a 32-bit int. Notice that both API calls take the address of a LARGE_INTEGER, which is why we pass &m_frequency and &now.

Clamp huge deltas

If the user drags the window or the debugger pauses, one frame can span several seconds. A giant dt can teleport objects through walls. In real engines we clamp it, e.g. if (dt > 0.25f) dt = 0.25f; so a hiccup never breaks the simulation.

Why we multiply movement by delta time

Here is the mistake almost every beginner makes. They move an object a fixed amount every frame:

// WRONG: speed depends on how fast the computer is.
playerX += 5.0f;  // 5 units every frame

On a 60 FPS machine that is 300 units per second. On a 144 FPS machine it is 720 units per second — the same game runs more than twice as fast. The cure is to express speed in units per second and scale it by dt:

void Update(float dt)
{
    const float speed = 5.0f;  // units PER SECOND, not per frame

    // High bit of the return value means the key is currently down.
    if (GetAsyncKeyState('D') & 0x8000)
        playerX += speed * dt;   // move right
    if (GetAsyncKeyState('A') & 0x8000)
        playerX -= speed * dt;   // move left
}

Now the math works out the same regardless of frame rate. At 60 FPS, dt is about 0.0167, and over 60 frames the player moves 5.0 * 0.0167 * 60 = 5 units. At 144 FPS each step is smaller, but there are more of them, and the total is still 5 units per second. Movement, gravity, animation playback, timers — everything that changes over time should be scaled by dt.

Variable vs fixed timestep

What we built above is a variable timestep: dt is whatever the last frame actually took, and we feed that straight into Update. It is simple, smooth, and perfect for rendering and most gameplay. The catch is that physics can behave slightly differently at different frame rates, because the simulation advances in uneven chunks.

For rock-solid, reproducible physics, engines use a fixed timestep. You pick a constant step — say 1/60th of a second — accumulate real time, and run the physics update in whole steps of that size:

const float kStep = 1.0f / 60.0f;  // simulate physics at a steady 60 Hz
float accumulator = 0.0f;

// ...inside the loop, after measuring dt...
accumulator += dt;
while (accumulator >= kStep)
{
    FixedUpdate(kStep);   // always the SAME dt, fully deterministic
    accumulator -= kStep;
}

Render();  // still drawn every frame, as fast as we can

This is exactly the split Unity exposes as Update versus FixedUpdate. Don't worry about wiring up physics yet — for now, a plain variable timestep is the right starting point. Just remember the pattern exists, because our engine will grow into it.

This loop never changes

Whether your game draws sprites or a full 3D scene, the loop stays the same three phases. In the next module we start filling in Render with real Direct3D 11 draw calls — but the skeleton you wrote today is the one your engine keeps forever.

Recap
  • A game loop uses PeekMessage (non-blocking) so it keeps running instead of sleeping until input arrives.
  • Every frame runs three phases in order: process input, Update(dt), then Render.
  • Measure delta time with QueryPerformanceFrequency and QueryPerformanceCounter, dividing elapsed ticks by the frequency to get seconds.
  • Multiply anything time-based by dt so movement is frame-rate independent — speeds are expressed in units per second.
  • A variable timestep is simple and smooth; a fixed timestep with an accumulator gives deterministic physics, mirroring Unity's Update and FixedUpdate.
