Headers, Source Files & the Preprocessor

So far every program you have written lived inside a single main.cpp. That is fine for a toy, but our engine will grow to hundreds of files — a math library, a renderer, components, systems. In this lesson you will learn how C++ programs are split into headers (declarations) and source files (definitions), and how the preprocessor stitches them together. This is the skeleton every real C++ project hangs on.

What you'll learn
  • The difference between a declaration and a definition, and why the split matters
  • How #include and the preprocessor physically build a translation unit
  • Protecting headers with include guards and #pragma once
  • The One Definition Rule (ODR) and how to avoid breaking it
  • Forward declarations to break dependencies and speed up builds

Two kinds of statements: declare vs. define

Every name in C++ is either declared or defined. A declaration tells the compiler that something exists and what its shape is — its name, its return type, its parameters. A definition actually provides the body or the storage.

Think of a declaration as a promise ("there is a function called Add that takes two ints and returns an int") and a definition as delivering on it (the actual { return a + b; }). You can declare something as many times as you like, but you may only define it once.

Why separate them?

Any file that wants to use Add only needs the declaration. The definition lives in one place and is compiled once. This is what lets a big project compile files independently and in parallel.

Headers and source files

By convention we put declarations in a header file (extension .h or .hpp) and definitions in a source file (extension .cpp). Other files #include the header to learn about the declarations; the source file is compiled separately to provide the definitions.

Let's build a tiny math library — exactly the kind of thing our engine's Transform and Vector math will eventually live in. First the header, which declares what the library offers:

#pragma once   // include this header at most once per translation unit

// A minimal 2D vector. Small, plain data — perfect for a header.
struct Vec2 {
    float x = 0.0f;
    float y = 0.0f;
};

// Declarations only: no function bodies here.
float Dot(const Vec2& a, const Vec2& b);
Vec2  Add(const Vec2& a, const Vec2& b);
float Length(const Vec2& v);

Now the source file, which #includes its own header and supplies the definitions:

#include "Math.h"   // pull in the declarations we are about to define
#include <cmath>     // for std::sqrt

// Each definition matches a declaration in Math.h exactly.
float Dot(const Vec2& a, const Vec2& b) {
    return a.x * b.x + a.y * b.y;
}

Vec2 Add(const Vec2& a, const Vec2& b) {
    return Vec2{ a.x + b.x, a.y + b.y };
}

float Length(const Vec2& v) {
    return std::sqrt(Dot(v, v));   // reuse Dot for v · v
}

Finally, main.cpp uses the library. It includes the header — not the source file — and the two .cpp files are compiled separately, then linked together into one program.

#include "Math.h"
#include <iostream>

int main() {
    Vec2 a{ 3.0f, 4.0f };
    Vec2 b{ 1.0f, 2.0f };

    Vec2 sum = Add(a, b);
    std::cout << "sum = (" << sum.x << ", " << sum.y << ")\n";
    std::cout << "length of a = " << Length(a) << "\n"; // 5

    return 0;
}
Quotes vs. angle brackets

Use #include "Math.h" (quotes) for your own headers — the compiler looks next to the current file first. Use #include <iostream> (angle brackets) for the standard library and system headers.

The preprocessor: what #include really does

Before the real compiler ever sees your code, a stage called the preprocessor runs. It is a simple text-substitution engine. When it sees #include "Math.h", it literally pastes the entire contents of Math.h in place of that line. The result — one .cpp plus everything it pulled in — is called a translation unit, and that is what the compiler actually compiles.

Macros, briefly

The preprocessor also handles #define, which creates a macro — a name it replaces with text everywhere. You will see these in real code, so recognize them:

#define PI 3.14159265f          // object-like macro
#define SQUARE(x) ((x) * (x))    // function-like macro

float area = PI * SQUARE(radius); // expands before compilation
Prefer real C++ over macros

Macros ignore types and scope, so they cause confusing bugs. In modern C++ prefer constexpr float PI = 3.14159265f; and real functions. We keep #define for one job: header guards, next.

Include guards and #pragma once

Because #include is blind copy-paste, a header can easily be pasted twice into the same translation unit — for example if two headers you include both include Math.h. That would define struct Vec2 twice in one unit, which is an error.

The classic fix is an include guard: a macro that makes the second paste do nothing.

#ifndef MATH_H      // if MATH_H is not yet defined...
#define MATH_H      // ...define it, then include the body once

struct Vec2 { float x = 0.0f; float y = 0.0f; };
float Dot(const Vec2& a, const Vec2& b);

#endif // MATH_H    // second time around, the body is skipped

Almost every compiler also supports a shorter, equivalent directive that does the same thing with less boilerplate. It is what we used in the very first Math.h above:

#pragma once   // one line, no unique macro name to invent

struct Vec2 { float x = 0.0f; float y = 0.0f; };
Our convention

We'll put #pragma once at the top of every header in the engine. It is supported by Clang and GCC on both course platforms, and you never have to invent a unique guard name.

The One Definition Rule

The One Definition Rule (ODR) is the law that ties all this together: across your whole program, every function and every variable may have exactly one definition. Types like struct Vec2 may appear in many translation units, but the definition must be identical everywhere — which is exactly why we put it in a header and guard it.

This is the reason headers hold declarations: a declaration can appear in a hundred files, but the single definition of Add lives only in Math.cpp. Put a plain function definition in a header and include it from two .cpp files, and the linker will complain about a duplicate symbol.

Goes in the header (.h)Goes in the source (.cpp)
Function declarations (prototypes)Function definitions (bodies)
Type definitions: struct, enum, classNon-inline, non-template implementation
constexpr / inline constantsFile-local helpers and static data
#pragma once at the very top#include of its own header first
"Multiple definition" linker errors

If the linker says a symbol is defined multiple times, you almost certainly put a function body in a header. Move it to a .cpp, or mark it inline if it truly must live in the header.

Forward declarations

Sometimes a header only needs to know that a type exists — for instance to declare a pointer or reference to it — without needing its full definition. In that case you can use a forward declaration instead of including the whole header. This breaks dependency chains and dramatically speeds up compilation, something you will appreciate once the engine has dozens of headers.

#pragma once

// Forward declaration: "a struct named Mesh exists somewhere."
// We do NOT need to #include "Mesh.h" here because we only use Mesh*.
struct Mesh;

class Renderer {
public:
    void Draw(const Mesh* mesh);   // pointer only — declaration is enough
};

The rule of thumb: if you only use a pointer or reference to a type, a forward declaration is enough. If you store it by value, call its methods, or access its members, you need the full definition — so you must #include its header (typically in the .cpp).

Why big projects need this

Imagine our engine as one giant main.cpp with fifty thousand lines. Every tiny edit would recompile everything, merge conflicts would be constant, and nobody could work on the renderer and the physics at the same time. Splitting into headers and source files means each .cpp compiles independently, only changed files rebuild, and clean interfaces (the headers) let subsystems evolve without disturbing their neighbors. That is exactly how Unity and Unreal organize their code, and it is the structure we will use for every component we build next.

Recap
  • A declaration promises a name exists; a definition provides the body or storage — declare many times, define once.
  • Put declarations in headers (.h) and definitions in source files (.cpp); other files #include the header.
  • #include is preprocessor text-pasting that builds a translation unit; guard every header with #pragma once.
  • The One Definition Rule forbids duplicate definitions — a body in a header shared by two .cpp files causes linker errors.
  • Use a forward declaration when you only need a pointer or reference to a type, to cut dependencies and build time.
