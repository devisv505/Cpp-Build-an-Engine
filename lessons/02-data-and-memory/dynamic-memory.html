Dynamic Memory: new, delete & the Heap

In the last lesson you learned that a pointer just stores an address. Now we ask a deeper question: where does the data those pointers point to actually live, and who decides how long it stays alive? Answering that unlocks one of the most important tools in C++: allocating memory yourself, on demand, at runtime. Every mesh, texture and game object our engine loads will live in this kind of memory.

What you'll learn
  • The difference between the stack and the heap, and when each is used
  • How to allocate and free single values with new and delete
  • How to allocate arrays whose size is only known at runtime with new[] and delete[]
  • Why heap allocation is necessary for lifetime beyond a scope and runtime-sized data
  • The two classic bugs — memory leaks and dangling pointers — and how to spot them

Two places memory comes from

Every variable your program uses lives somewhere in memory. There are two regions we care about right now: the stack and the heap. Understanding which is which explains almost every "why did my data disappear?" bug beginners hit.

The stack: automatic and fast

When you write a normal local variable, it lives on the stack. The stack is a region the compiler manages for you automatically. Each time a function is called, a block of stack space (a stack frame) is reserved for its locals; when the function returns, that block is instantly reclaimed. You never call anything to make this happen.

#include <iostream>

void demo() {
    int health = 100;   // lives on the stack
    int mana   = 50;    // also on the stack
    std::cout << health + mana << '\n';
}   // <-- health and mana are destroyed here, automatically

int main() {
    demo();
    return 0;
}

The stack is wonderfully convenient, but it has two hard limits. First, a stack variable dies the moment its scope (the { } block it was declared in) ends. Second, the compiler must know the exact size at compile time. Both of those limits are exactly what the heap frees us from.

The heap: manual and flexible

The heap (also called the free store) is a large pool of memory you can request pieces of whenever you want, in whatever size you compute at runtime. Memory you take from the heap stays alive until you explicitly hand it back — it ignores scopes entirely. That power is the whole point, and also the responsibility.

A quick mental model

Think of the stack as a stack of plates that's tidied up for you automatically, and the heap as a locker room where you rent a locker for as long as you like — but you must remember to return the key, or that locker stays occupied forever.

Allocating one value: new and delete

To grab a single object from the heap, use the new keyword. It does two things: it reserves enough memory for the type, and it returns a pointer to that memory. When you're finished, you return the memory with delete.

#include <iostream>

int main() {
    // Ask the heap for one int. `p` is a pointer to it.
    int* p = new int;      // memory reserved, value uninitialised
    *p = 42;               // write through the pointer

    // You can also allocate and initialise in one step:
    int* q = new int(7);   // heap int holding 7

    std::cout << *p << ' ' << *q << '\n';   // 42 7

    delete p;   // give the memory back to the heap
    delete q;   // every `new` needs exactly one `delete`

    p = nullptr;   // good habit: null out after delete (explained below)
    q = nullptr;
    return 0;
}

The golden rule is simple to say and easy to forget: every new must be matched by exactly one delete. Forget the delete and the memory is lost until the program exits (a leak). Call delete twice on the same pointer and you corrupt the heap (undefined behaviour, often a crash).

Allocating arrays: new[] and delete[]

The single most common reason beginners reach for the heap is a size that isn't known until the program runs. Suppose the user tells us how many particles to spawn, or a model file reports how many vertices it contains — we can't hard-code that. A stack array needs a compile-time size, but new[] accepts a runtime value.

#include <iostream>

int main() {
    int count;
    std::cout << "How many scores? ";
    std::cin  >> count;                 // known only at runtime

    // Allocate an array of `count` ints on the heap.
    int* scores = new int[count];

    for (int i = 0; i < count; ++i)
        scores[i] = i * 10;            // index just like a normal array

    for (int i = 0; i < count; ++i)
        std::cout << scores[i] << ' ';
    std::cout << '\n';

    delete[] scores;   // NOTE the [] — required for array allocations
    scores = nullptr;
    return 0;
}
Match the brackets

Memory from new[] must be freed with delete[], and memory from plain new with plain delete. Mixing them (for example delete scores; without the brackets) is undefined behaviour. The compiler will not warn you.

Why bother? Lifetime and runtime size

There are two situations where the stack simply can't help, and the heap is the answer.

1. Data that must outlive the function that created it

Imagine a factory function that builds an object and hands it back to the caller. If it lived on the stack, it would be destroyed the instant the function returned, and the caller would receive a pointer to garbage. Heap memory ignores scope, so it survives.

#include <iostream>

// Returns a pointer to a heap int that survives after we return.
int* makeCounter(int start) {
    int* c = new int(start);   // heap: outlives this function
    return c;                  // caller now owns this memory
}

int main() {
    int* counter = makeCounter(10);
    *counter += 5;
    std::cout << *counter << '\n';   // 15

    delete counter;   // the caller is responsible for freeing it
    counter = nullptr;
    return 0;
}

2. Sizes decided at runtime

As we saw with new int[count], the amount of data often depends on user input, a file, or a network message. This is precisely our situation in the engine: when we load a mesh, the vertex and index counts come from the file. We'll allocate exactly the right amount of memory to hold them.

You've already been using the heap

Remember std::vector and std::string? They call new[] and delete[] internally so you don't have to. Learning the raw mechanics here is what lets you understand why those containers exist and trust what they're doing under the hood.

The two bugs everyone meets

Manual memory management is powerful, but it introduces two failure modes that have haunted C++ programmers for decades. Learn to recognise them now.

BugWhat happenedSymptom
Memory leakYou called new but never delete. The memory is never returned.Program's memory use climbs over time until it slows or crashes.
Dangling pointerYou deleted the memory but kept using the pointer that referred to it.Reads garbage, corrupts data, or crashes — often unpredictably.
 
#include <iostream>

void leak() {
    int* data = new int[1000];   // allocated...
    // ...function ends without delete[] -- 1000 ints leaked forever
}

void dangling() {
    int* p = new int(5);
    delete p;              // memory returned to the heap
    // std::cout << *p;    // BUG: `p` now dangles, *p is undefined behaviour
    p = nullptr;           // the fix: null it out so misuse is obvious
    // now `if (p)` is false, and *p would crash loudly instead of silently
}

int main() {
    leak();
    dangling();
    return 0;
}
Using memory after delete is undefined behaviour

A dangling pointer might appear to work today and crash tomorrow, because the freed memory hasn't been reused yet. These are among the hardest bugs to track down. Two habits tame them: set a pointer to nullptr right after delete, and never delete the same pointer twice. Deleting a nullptr is always safe and does nothing.

A glimpse ahead: we'll stop doing this by hand

You might already be feeling the tension: matching every new with exactly one delete, along every possible path a function can take, is genuinely hard to get right. Miss one and you leak; do it twice and you crash. Real engines don't manage memory this way by hand.

In Module 3 we'll replace almost all raw new/delete with smart pointers — std::unique_ptr and std::shared_ptr — which free the memory automatically when it's no longer needed. That's how our engine's components will be owned: a unique_ptr<Mesh> here, a shared_ptr<Texture> there, with the cleanup handled for us. But you need today's raw mechanics first, because a smart pointer is just a thin, safe wrapper around exactly the new and delete you've now learned.

The rule of thumb going forward

Prefer stack variables and std::vector for everyday code; reach for raw new/delete only to understand the machinery. From Module 3 on, prefer smart pointers whenever an object must live on the heap.

Recap
  • Local variables live on the stack: automatic, fast, but bound to their scope and fixed in size at compile time.
  • The heap gives you memory that survives scopes and can be sized at runtime — you request it with new/new[] and return it with delete/delete[].
  • Every new needs exactly one matching delete; match new[] with delete[].
  • The heap is essential when data must outlive the function that made it, or when the size is only known at runtime.
  • Forgetting to free leaks memory; using a pointer after delete leaves it dangling — set pointers to nullptr after freeing.
  • In Module 3 we'll replace raw new/delete with smart pointers that clean up automatically.
