Pointers & Memory Addresses

In the last lesson you met references — a friendly alias for an existing variable. Pointers are the next step down, closer to the metal: instead of aliasing a variable, a pointer stores the numeric address of where that variable lives in memory. This is the idea every serious engine is built on. When our DirectX engine hands a GPU buffer or a device context around, it passes a pointer, not a fat copy of the data.

What you'll learn

  • How to declare a pointer with T* and take an address with &.
  • How to read and write the pointed-to value by dereferencing with *.
  • Why nullptr exists and how to check for it safely.
  • When to reach for a pointer instead of a reference.

Memory is just a row of numbered boxes

Your program's memory is one enormous sequence of bytes. Every byte has a unique number called its address, counting up from zero. When you declare a variable, the compiler reserves one or more of those bytes and remembers where they are. An int is typically 4 bytes, so if an int lives at address 1000, it occupies bytes 1000 through 1003.

You almost never care about the exact number — it changes every run — but the concept matters: a variable is a named window onto a specific place in memory. A pointer lets you hold that place as a value you can pass around, store, and follow later.

Declaring a pointer and taking an address

A pointer type is written by adding a * to the type it points at. A pointer to int is int*; a pointer to double is double*. To get the address of an existing variable, use the address-of operator & (the same symbol you used to declare references, but here it sits on the right-hand side, applied to a value).

#include <iostream>

int main() {
    int score = 42;      // an ordinary int
    int* ptr = &score;   // ptr holds the ADDRESS of score

    // Printing a pointer shows the raw address (a hex number).
    std::cout << "score lives at: " << ptr << "\n";
    std::cout << "value of score: " << *ptr << "\n"; // dereference

    return 0;
}

Read the declaration right-to-left: "ptr is a pointer to int." The &score on the right produces score's address, and we store it in ptr.

Two jobs for one asterisk

The * means different things in different places. In a declaration (int* ptr) it is part of the type — "pointer to int." In an expression (*ptr) it is the dereference operator — "the value at this address." Same symbol, opposite directions.

Dereferencing: reading and writing through a pointer

Holding an address is only useful if you can follow it. The dereference operator * takes a pointer and gives you the actual variable it points at. And because it gives you the variable itself — not a copy — you can assign through the pointer to change the original.

#include <iostream>

int main() {
    int health = 100;
    int* hp = &health;   // hp points at health

    *hp = 75;            // write through the pointer...
    std::cout << health << "\n";   // ...health is now 75

    *hp = *hp - 10;      // read then write: 75 - 10
    std::cout << health << "\n";   // health is now 65

    return 0;
}

Notice we never touched health by name after the first line, yet its value changed. That indirection is the whole point: a function can be handed a pointer to your data and modify it in place, without ever copying it. Later, our engine's systems will take a Transform* and update an object's position exactly this way.

The null pointer

Sometimes a pointer points at nothing — maybe a resource has not been created yet, or a lookup failed. C++ has a dedicated value for "points at nothing": nullptr. Always give a pointer a value; an uninitialized pointer holds garbage and dereferencing it is undefined behavior.

#include <iostream>

int main() {
    int* device = nullptr;   // explicitly points at nothing

    // Guard before you dereference.
    if (device != nullptr) {
        std::cout << *device << "\n";
    } else {
        std::cout << "no device yet\n";
    }

    return 0;
}
Never dereference a null (or dangling) pointer

Writing *device when device is nullptr crashes your program — this is the classic "null pointer dereference." In DirectX, functions like D3D11CreateDevice hand you back a pointer and a status code; you must check the result before you use the pointer.

Pointer vs. reference

Pointers and references both give you indirect access to another variable, so when do you use which? References are simpler and safer: they must bind to something on creation and can never be reseated or null. Pointers are more powerful: they can be null, can be reassigned to point at different objects over time, and can be stored in arrays and containers.

A good rule of thumb

Reach for a reference when the target always exists and never changes. Reach for a pointer when the target might be absent (nullptr) or when it needs to change over the object's lifetime — which is exactly how an engine holds onto GPU resources such as an ID3D11Device*.

Why engines live and breathe pointers

Graphics resources — devices, textures, vertex buffers — are large and owned by the driver. You would never copy one around; instead the Direct3D API hands you a pointer to it, and every system that needs it shares that same pointer. A tiny sketch of what that looks like:

// Simplified: a system receives the shared resource by pointer.
struct Mesh {
    int vertexCount = 0;
};

// 'mesh' is passed by pointer, so nothing is copied — the caller
// still owns the Mesh and we just read through the pointer.
void draw(const Mesh* mesh) {
    if (mesh == nullptr) return;          // nothing to draw
    // *mesh gives us the real Mesh; -> reads a member through a pointer.
    for (int i = 0; i < mesh->vertexCount; ++i) {
        // ...submit vertex i to the GPU...
    }
}
A sneak peek: the arrow operator

You saw mesh->vertexCount above. When you have a pointer to a struct or class, ptr->member is shorthand for (*ptr).member — dereference, then access the member. We'll lean on it constantly once we start writing components.

Everything so far assumes the pointed-to variable already exists somewhere on the stack. But engines also need memory that outlives the function that created it — that is where the heap comes in, with new and delete. That is exactly the next lesson.

Recap
  • Memory is a long line of numbered bytes; a variable's address is where it starts.
  • A pointer is declared with T* and holds an address; &value gives you one.
  • *ptr dereferences — it reads or writes the actual variable the pointer targets.
  • nullptr means "points at nothing"; always check before you dereference.
  • Prefer references for guaranteed, fixed targets; use pointers when the target may be null or change — the model engines use to share GPU resources.
