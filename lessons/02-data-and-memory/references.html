References

So far, when you pass a variable into a function, the function gets its own private copy. That is safe, but it means the function cannot change your original value, and copying a big object like a std::string can be wasteful. References fix both problems by giving a function direct access to your variable. They are the everyday tool we will reach for constantly when our engine passes around Transforms, meshes, and other components.

What you'll learn
  • What a reference is — an alias for an existing variable — and the T& syntax.
  • How to pass by reference so a function can modify the caller's variable.
  • How const T& gives you fast, read-only access to large objects without copying.
  • Why references must be initialized and can never be re-pointed at something else.

A reference is just another name

A reference is an alias: a second name for a variable that already exists. Once you bind a reference to a variable, the two names refer to the exact same piece of memory. Change one, and you have changed both — because there is really only one value underneath.

You create a reference by putting an ampersand (&) after the type:

#include <iostream>

int main() {
    int health = 100;
    int& hp = health;   // hp is now another name for health

    hp = 75;            // changes health too — same storage

    std::cout << health << "\n";   // prints 75
    std::cout << hp << "\n";       // prints 75
    return 0;
}

Notice there is no new integer here. hp is not a copy of health; it is health, viewed through a different name. This is the whole idea, and everything else about references follows from it.

Read the & by context

The same & symbol means different things depending on where it sits. In a type like int& it declares a reference. Later, when you study pointers, you will see &x in an expression meaning "address of x". Different jobs, same character.

Passing by reference to modify a variable

Here is the problem references were made for. Suppose you write a function that should swap two values. If you pass the values by copy, the function swaps its own copies and the originals never change:

// BROKEN: a and b are copies, so the caller sees no change
void swapBroken(int a, int b) {
    int temp = a;
    a = b;
    b = temp;
}

Add an ampersand to each parameter and the function now works on the caller's actual variables:

#include <iostream>

// a and b are aliases for the caller's variables
void swap(int& a, int& b) {
    int temp = a;
    a = b;
    b = temp;
}

int main() {
    int x = 3;
    int y = 9;

    swap(x, y);   // no & needed here — just pass the variables

    std::cout << "x = " << x << ", y = " << y << "\n";  // x = 9, y = 3
    return 0;
}

Inside swap, the name a is bound to x and b is bound to y. Assigning through those names edits x and y directly. Notice the call site is unchanged — you still write swap(x, y). The reference lives in the parameter list, not the call.

Passing by const reference to avoid copies

References are not only for changing things. Their second big use is efficiency. When you pass a large object by value, the whole thing is copied — every character of a std::string, every element of a std::vector. For a big object that copy costs time and memory, and it is pure waste if the function only wants to read the value.

The fix is a const reference, written const T&. It gives the function direct, no-copy access, and the const promises the function will not modify the original:

#include <iostream>
#include <string>

// const std::string&: no copy is made, and name cannot be changed here
void greet(const std::string& name) {
    std::cout << "Hello, " << name << "!\n";
    // name = "other"; // would be a COMPILE ERROR — name is const
}

int main() {
    std::string player = "Ada";
    greet(player);          // passes a reference, not a copy
    greet("Grace Hopper");  // string literals work too
    return 0;
}

This pattern — const std::string&, const std::vector<int>&, and so on — is one of the most common sights in real C++. When our engine has a function like loadMesh(const std::string& path), the const reference lets it read the file path cheaply without duplicating the string on every call.

A simple rule of thumb

For small built-in types like int, float, or bool, pass by value — copying them is essentially free. For larger objects the function only needs to read, prefer const T&. Use a plain T& when the function is meant to modify the caller's object.

Two rules you cannot break

References come with two strict rules. Both fall straight out of "a reference is just another name for something that already exists".

1. A reference must be initialized

A name has to name something. You cannot declare a reference and leave it empty to fill in later:

int value = 42;

int& good = value;   // OK: bound to value at birth
// int& bad;         // ERROR: a reference must be initialized

2. A reference cannot rebind

Once bound, a reference is stuck to that variable for its entire life. Later assignments do not re-point the reference — they write through it into the original variable:

int a = 1;
int b = 2;

int& r = a;   // r is an alias for a
r = b;        // does NOT make r refer to b;
              // it copies b's value (2) INTO a. Now a == 2.
Common beginner trap

Writing r = b feels like "make r point to b", but a reference can never be re-seated. It quietly overwrites the value it was already bound to. When you truly need something that can point at different objects over time — that is a job for pointers, the topic of the next lesson.

Recap
  • A reference (T&) is an alias — another name for an existing variable, sharing the same storage.
  • Pass by reference (T&) when a function needs to modify the caller's variable, as in swap(int& a, int& b).
  • Pass by const T& to read large objects like std::string efficiently, with no copy and no risk of modification.
  • References must be initialized when declared and can never be rebound; assigning to a reference writes through it to the original.
  • Small built-in types are cheap to pass by value; reserve references for big objects or when you must edit the original.
