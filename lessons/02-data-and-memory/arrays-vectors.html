Arrays & std::vector

So far our variables have held a single value at a time. But a game engine is really a machine for managing lists of things: entities in a scene, vertices in a mesh, high scores on a leaderboard. In this lesson you'll meet C-style fixed arrays, understand why they're dangerous, and then learn std::vector — the growable, safe, modern container we'll reach for again and again as we build our engine.

What you'll learn
  • How to declare, fill and index a C-style fixed array — and why indices start at 0
  • Why reading or writing past the end of an array is a serious bug
  • How to use std::vector to grow a list at runtime with push_back
  • Reading elements with size(), operator[], and the range-based for loop
  • When and why to prefer std::vector in engine code

C-style fixed arrays

An array is a block of memory holding several values of the same type, laid out one after another. You declare one by writing the element type, a name, and the count in square brackets:

#include <iostream>

int main()
{
    // An array of 4 ints. The [4] is the fixed size.
    int scores[4] = { 90, 85, 70, 100 };

    // Access elements by index. Indices start at 0!
    std::cout << "First score:  " << scores[0] << "\n"; // 90
    std::cout << "Fourth score: " << scores[3] << "\n"; // 100

    // You can also assign to an element.
    scores[1] = 88;

    return 0;
}

Notice that a 4-element array is indexed 0, 1, 2, 3. The index is really an offset from the start of the block, so the first element sits at offset 0. This trips up almost everyone at first, so say it out loud: the last valid index is the size minus one.

Because the array lives on the stack (remember the stack from the previous lesson), its size must be known when you compile — you cannot decide scores[n] where n is read from the user. That fixed nature is the array's biggest limitation.

The bounds danger

C++ does not check that your index is valid. If scores has 4 elements and you write scores[4] or scores[7], the compiler happily generates code that reads or writes memory it doesn't own. This is called an out-of-bounds access, and it's one of the most common sources of crashes and security bugs in C and C++.

int scores[4] = { 90, 85, 70, 100 };

// BUG: valid indices are 0..3, but we touch index 4.
scores[4] = 55; // undefined behavior — may crash, may corrupt data
Off-by-one is undefined behavior

Reading or writing outside an array's bounds is undefined behavior: the program might crash immediately, silently corrupt a nearby variable, or appear to work until it mysteriously breaks later. There's no exception, no warning — just a landmine. Always keep indices in the range 0 to size − 1.

Enter std::vector — the dynamic array

Most of the time we don't know up front how many things we'll have. A scene might load 3 objects or 3,000. That's exactly what std::vector is for: it's a dynamic array from the C++ standard library that lives on the heap, grows as you add elements, and cleans up after itself automatically. Include <vector> to use it.

#include <iostream>
#include <vector>

int main()
{
    // An empty vector of ints — no size baked in.
    std::vector<int> scores;

    // Add elements one at a time; the vector grows to fit.
    scores.push_back(90);
    scores.push_back(85);
    scores.push_back(70);
    scores.push_back(100);

    // size() tells you how many elements are currently stored.
    std::cout << "We have " << scores.size() << " scores.\n";

    // Index just like an array, with operator[].
    std::cout << "First score: " << scores[0] << "\n";

    return 0;
}

The std::vector<int> syntax means "a vector whose elements are int." The part in angle brackets is the element type — you'll see std::vector<float>, std::vector<GameObject>, and so on. This angle-bracket style is a template; we'll explore templates properly later, but for now just read it as "vector of ___."

Reserve when you know the count

If you know roughly how many elements you'll add, call scores.reserve(1000) first. It pre-allocates memory so the vector doesn't have to keep resizing as it grows — a small habit that pays off in hot engine loops.

Looping with range-based for

To visit every element, C++ gives us a clean, readable loop that works for arrays and vectors alike: the range-based for loop. It reads almost like English — "for each score in scores":

#include <iostream>
#include <vector>

int main()
{
    std::vector<int> scores = { 90, 85, 70, 100 };

    // Read-only pass: 'score' is a copy of each element.
    int total = 0;
    for (int score : scores)
        total += score;

    std::cout << "Total: " << total << "\n"; // 345

    // Modifying pass: 'auto&' is a reference, so changes stick.
    for (auto& score : scores)
        score += 5; // bonus points for everyone

    std::cout << "First score is now " << scores[0] << "\n"; // 95

    return 0;
}

The distinction is important. for (int score : scores) hands you a copy of each element — great for reading, but writing to score wouldn't affect the vector. for (auto& score : scores) gives you a reference to the real element (and auto just lets the compiler figure out the type), so any changes you make are written back into the vector.

Use const & to read big things cheaply

When elements are large (say a struct holding several floats), copying each one just to read it is wasteful. Write for (const auto& item : items): the reference avoids the copy, and const promises you won't modify it. We'll lean on this pattern constantly when iterating over components.

A tiny engine-flavored example

Let's make this concrete with something closer to what we're building. Imagine each object in a scene has a position in 3D space. We can store those positions in a vector and update them every frame. Here's a stripped-down version using a simple struct:

#include <iostream>
#include <vector>

// A minimal 3D position. Later this becomes part of our Transform component.
struct Vec3
{
    float x, y, z;
};

int main()
{
    std::vector<Vec3> positions;

    // Spawn three objects at different spots.
    positions.push_back({ 0.0f, 0.0f, 0.0f });
    positions.push_back({ 1.0f, 0.0f, 2.0f });
    positions.push_back({ -3.0f, 0.5f, 4.0f });

    // Simulate one frame: nudge everything upward on the Y axis.
    const float deltaY = 0.1f;
    for (auto& p : positions) // reference, so the update sticks
        p.y += deltaY;

    // Report the final state.
    std::cout << "Tracking " << positions.size() << " objects:\n";
    for (const auto& p : positions) // const reference: read-only, no copy
        std::cout << "  (" << p.x << ", " << p.y << ", " << p.z << ")\n";

    return 0;
}

This is the shape of a lot of engine code: a vector of data, a loop that transforms it each frame, and a loop that reads it back. When we later build a proper Transform component, its positions, rotations and scales will live in containers exactly like this one.

Array vs. vector at a glance

FeatureC-style array int a[4]std::vector<int>
Size fixed at compile timeYesNo — grows at runtime
Add elements laterNoYes, via push_back
Knows its own lengthNoYes, via size()
Lives onStack (usually)Heap (managed for you)
Cleans up automaticallyYes (but can't grow)Yes
Bounds-checked optionNoat() throws on bad index

Both operator[] and at() let you index a vector. operator[] is fast but unchecked (just like an array); at() checks the index and throws an exception if it's out of range. Use at() when you're unsure and operator[] in tight loops where you've already guaranteed the index is valid.

Don't hold references across a push_back

When a vector grows, it may move all its elements to a bigger block of memory. Any pointer or reference you saved to an element beforehand becomes dangling — it points at freed memory. If you're going to keep adding elements, grab references only after you're done growing, or store an index instead.

Which should you use?

The recommendation is simple: reach for std::vector by default. It's safe, it manages its own memory, it knows its length, and it grows when you need it to. Plain C-style arrays still have their place — small fixed-size lookup tables, or interoperating with C APIs and some parts of the DirectX interface — but for the everyday job of "I have a list of things," the vector is the right tool. Nearly every collection in our engine, from mesh vertices to the list of active game objects, will be a std::vector.

Recap
  • A C-style array int a[N] is a fixed-size block indexed from 0 to N − 1.
  • Indexing outside those bounds is undefined behavior — no warning, just bugs and crashes.
  • std::vector is a dynamic array: grow it with push_back, ask its length with size(), and index it with operator[] or the checked at().
  • The range-based for (auto& x : v) loop iterates cleanly; use auto& to modify, const auto& to read big elements without copying.
  • Prefer std::vector for engine data — it's safe, self-managing, and the container we'll build most of the engine around.
