Render Targets: Drawing Into a Texture

Every frame so far, MiniEngine has drawn straight to the window — the swap chain's back buffer, the pixels you see. But to add bloom, tone mapping, shadows or a mirror-like reflection, you first need the scene as data you can read back. The trick is simple and it powers the entire rest of this module: render the scene into a texture you own, then feed that texture into a later draw. In this lesson we build the RenderTarget abstraction that makes off-screen rendering a one-liner.

What you'll learn
  • Why post-processing, shadow maps and reflections all begin with rendering into a texture.
  • How one texture carries two views — a render target view to write, a shader resource view to read.
  • How to create a color target (plus a matching depth target) in Direct3D 11, and the Metal equivalent.
  • The read/write hazard: why you cannot sample a texture while rendering into it, and how ping-pong fixes it.
  • How to wrap it all in a RenderTarget class with Bind(), GetSRV() and a Resize() that survives window resizing.

Two passes, one frame

Up to now our frame has had exactly one destination. The renderer binds the back buffer with OMSetRenderTargets, draws the scene, and calls Present. There is nowhere in that flow to read the finished image and do something to it — by the time the pixels exist, they are already on their way to the monitor.

Post-processing needs a detour. Instead of drawing the scene to the screen, we draw it into a texture we allocated ourselves — an off-screen render target. Now the completed scene lives in GPU memory as a sampleable image. A second pass reads that image, applies an effect (brighten, blur, tone-map), and that result goes to the back buffer. The screen never sees the raw scene directly; it sees the processed one.

One idea, many features

This "render into a texture, then read it" pattern is not just for post-FX. A shadow map renders the scene from the light's point of view into a depth target, then samples it while shading. A planar reflection renders the mirrored scene into a color target and samples it on the water. The entire post stack you build across the rest of this module — HDR, blur, bloom — is chains of exactly this. Master the render target and you have unlocked all of them.

One texture, two views

Back in Module 06 you learned that a GPU resource and the views onto it are separate things. A render target leans on that split hard, because we need the same texture two different ways:

  • While we write the scene into it, it must be an output — a ID3D11RenderTargetView (RTV), the "writable lens" you bind with OMSetRenderTargets.
  • Later, when we read it, it must be a shader input — a ID3D11ShaderResourceView (SRV), the handle a pixel shader samples through.

Both views point at one underlying ID3D11Texture2D. The catch is that the texture must be created knowing it will serve both roles, so we set two bind flags at once: D3D11_BIND_RENDER_TARGET | D3D11_BIND_SHADER_RESOURCE. A back buffer only ever needs the first flag; a render target you intend to post-process needs both.

A color target on its own is not enough for a 3D scene. Depth testing needs somewhere to store per-pixel depth, so an off-screen pass also wants its own depth target at the same resolution — its own ID3D11Texture2D with D3D11_BIND_DEPTH_STENCIL and a matching ID3D11DepthStencilView, exactly like the depth buffer you made for the swap chain. We will not sample the depth in this lesson, so it needs only the depth-stencil flag.

Choosing the color format

For now DXGI_FORMAT_R8G8B8A8_UNORM — eight bits per channel — is fine and matches the back buffer. When we reach HDR and tone mapping, we will switch the color target to DXGI_FORMAT_R16G16B16A16_FLOAT so it can hold brightness values above 1.0 without clipping. That is the whole reason to render off-screen for HDR: the back buffer can't store those values, but our own float texture can.

Creating the target in Direct3D 11

Here is the raw creation code for one off-screen target: a color texture with both views, plus a matching depth texture and its view. Because we give the textures fully-typed formats (not typeless), we can pass nullptr for the view descriptions and Direct3D infers them from the resource.

#include <d3d11.h>
#include <wrl/client.h>

using Microsoft::WRL::ComPtr;

// ---- Color texture: written as an RTV, read as an SRV ----
D3D11_TEXTURE2D_DESC cd = {};
cd.Width            = width;
cd.Height           = height;
cd.MipLevels        = 1;
cd.ArraySize        = 1;
cd.Format           = DXGI_FORMAT_R8G8B8A8_UNORM;
cd.SampleDesc.Count = 1;                       // no MSAA on the target
cd.Usage            = D3D11_USAGE_DEFAULT;      // GPU reads and writes it
cd.BindFlags        = D3D11_BIND_RENDER_TARGET  // we draw into it, and
                    | D3D11_BIND_SHADER_RESOURCE; // later sample from it

ComPtr<ID3D11Texture2D> colorTex;
device->CreateTexture2D(&cd, nullptr, &colorTex);

// The two views onto that one texture. nullptr desc = "the whole resource".
ComPtr<ID3D11RenderTargetView>   rtv;
ComPtr<ID3D11ShaderResourceView> srv;
device->CreateRenderTargetView(colorTex.Get(),   nullptr, &rtv);
device->CreateShaderResourceView(colorTex.Get(), nullptr, &srv);

// ---- Matching depth texture (same size), depth-only ----
D3D11_TEXTURE2D_DESC dd = {};
dd.Width            = width;
dd.Height           = height;
dd.MipLevels        = 1;
dd.ArraySize        = 1;
dd.Format           = DXGI_FORMAT_D24_UNORM_S8_UINT;
dd.SampleDesc.Count = 1;
dd.Usage            = D3D11_USAGE_DEFAULT;
dd.BindFlags        = D3D11_BIND_DEPTH_STENCIL;

ComPtr<ID3D11Texture2D> depthTex;
device->CreateTexture2D(&dd, nullptr, &depthTex);

ComPtr<ID3D11DepthStencilView> dsv;
device->CreateDepthStencilView(depthTex.Get(), nullptr, &dsv);

That is every object we need. Notice the color texture is the only one with two views — the depth texture has just its DSV, because we only test against depth here, we never read it in a shader. If you later want soft shadows or depth-based fog, you would give the depth texture a typeless format and add an SRV, but that is a Module 12 concern.

The read/write hazard

Here is the single most important rule of render targets, and the source of a bug every engine author hits once. A texture cannot be an output and an input at the same time. The GPU may run pixels in parallel and cache freely, so reading a texel you are simultaneously writing has no defined answer. Direct3D refuses to let you: if a resource is bound as a render target and you try to bind it as a shader resource, the runtime silently unbinds one of them — usually your SRV drops to null and you sample black.

Unbind before you sample

Before pass 2 samples the target, pass 1 must be finished and the RTV must be unbound. In practice you switch the output back to the back buffer first — OMSetRenderTargets with the back buffer's RTV — and only then call PSSetShaderResources with the target's SRV. Turn on the D3D11 debug layer: it prints a clear "resource is bound as both input and output" warning the instant you get this wrong.

What if an effect genuinely needs to read an image and write a modified version of it — a blur that reads the scene and outputs a blurrier scene? You use two targets and ping-pong: read from target A, write to target B; then for the next pass read B, write A; and so on. Because the source and destination are always different textures, there is never a hazard. The separable blur two lessons from now is built entirely on this back-and-forth.

A RenderTarget class for MiniEngine

All of that creation and binding is boilerplate we do not want scattered through the renderer. Let's wrap it in a small engine class that owns the textures and views, and exposes the three verbs a pass actually needs: Bind() to make it the output, GetSRV() to hand the finished image to a later shader, and Resize() to rebuild when the window changes size.

#pragma once
#include <d3d11.h>
#include <wrl/client.h>

class RenderTarget
{
public:
    // withDepth: allocate a matching depth-stencil target too.
    void Init(ID3D11Device* device, UINT w, UINT h,
              DXGI_FORMAT format = DXGI_FORMAT_R8G8B8A8_UNORM,
              bool withDepth = true);

    // Make this the output of the pipeline and set a matching viewport.
    void Bind(ID3D11DeviceContext* ctx);
    void Clear(ID3D11DeviceContext* ctx, const float rgba[4]);

    // Rebuild at a new size. Cheap no-op if the size is unchanged.
    void Resize(ID3D11Device* device, UINT w, UINT h);

    ID3D11ShaderResourceView* GetSRV() const { return m_srv.Get(); }
    ID3D11RenderTargetView*   GetRTV() const { return m_rtv.Get(); }
    UINT Width()  const { return m_width; }
    UINT Height() const { return m_height; }

private:
    void CreateResources(ID3D11Device* device);

    UINT        m_width  = 0;
    UINT        m_height = 0;
    DXGI_FORMAT m_format = DXGI_FORMAT_R8G8B8A8_UNORM;
    bool        m_withDepth = true;

    Microsoft::WRL::ComPtr<ID3D11Texture2D>          m_colorTex;
    Microsoft::WRL::ComPtr<ID3D11RenderTargetView>   m_rtv;
    Microsoft::WRL::ComPtr<ID3D11ShaderResourceView> m_srv;
    Microsoft::WRL::ComPtr<ID3D11Texture2D>          m_depthTex;
    Microsoft::WRL::ComPtr<ID3D11DepthStencilView>   m_dsv;
};

The implementation is the creation code from earlier, plus a Bind() that also sets the viewport — a detail that bites people, so read the next section. Resize() just releases the old ComPtrs and rebuilds; because everything lives in Microsoft::WRL::ComPtr, Reset() releases the GPU resource for us.

void RenderTarget::Init(ID3D11Device* device, UINT w, UINT h,
                       DXGI_FORMAT format, bool withDepth)
{
    m_width = w; m_height = h;
    m_format = format; m_withDepth = withDepth;
    CreateResources(device);
}

void RenderTarget::Bind(ID3D11DeviceContext* ctx)
{
    ID3D11RenderTargetView* rtvs[] = { m_rtv.Get() };
    ctx->OMSetRenderTargets(1, rtvs, m_dsv.Get()); // m_dsv may be null

    // CRUCIAL: the viewport must match THIS target's size, not the window's.
    D3D11_VIEWPORT vp = {};
    vp.Width    = static_cast<float>(m_width);
    vp.Height   = static_cast<float>(m_height);
    vp.MaxDepth = 1.0f;
    ctx->RSSetViewports(1, &vp);
}

void RenderTarget::Clear(ID3D11DeviceContext* ctx, const float rgba[4])
{
    ctx->ClearRenderTargetView(m_rtv.Get(), rgba);
    if (m_dsv)
        ctx->ClearDepthStencilView(
            m_dsv.Get(), D3D11_CLEAR_DEPTH | D3D11_CLEAR_STENCIL, 1.0f, 0);
}

void RenderTarget::Resize(ID3D11Device* device, UINT w, UINT h)
{
    if (w == m_width && h == m_height) return; // nothing to do
    m_width = w; m_height = h;

    // Release the old GPU resources before recreating at the new size.
    m_rtv.Reset(); m_srv.Reset(); m_colorTex.Reset();
    m_dsv.Reset(); m_depthTex.Reset();
    CreateResources(device);
}

Using it in the frame — and surviving resize

With the class in place, a two-pass frame reads almost like prose. Pass 1 binds the target and draws every mesh exactly as before; the only change is where the pixels land. Pass 2 (which we flesh out next lesson) restores the back buffer as the output and hands the target's SRV to a fullscreen shader.

void Renderer::RenderFrame()
{
    const float clear[4] = { 0.05f, 0.05f, 0.08f, 1.0f };

    // ---- Pass 1: draw the scene INTO the off-screen target ----
    m_sceneTarget.Bind(m_context.Get());
    m_sceneTarget.Clear(m_context.Get(), clear);

    for (Mesh* mesh : m_scene)          // same draw code as always
        mesh->Draw(m_context.Get());    // just a different destination

    // ---- Pass 2: read the scene texture, write to the screen ----
    // Switch output back to the back buffer FIRST (unbinds the RTV),
    // so the target is safe to sample. See the hazard rule above.
    ID3D11RenderTargetView* backRtvs[] = { m_backBufferRTV.Get() };
    m_context->OMSetRenderTargets(1, backRtvs, nullptr);

    ID3D11ShaderResourceView* srvs[] = { m_sceneTarget.GetSRV() };
    m_context->PSSetShaderResources(0, 1, srvs);

    DrawFullscreenPass();               // next lesson: the fullscreen quad

    m_swapChain->Present(1, 0);
}
Recreate your targets on resize — the classic post-FX bug

Off-screen targets are not resized for you. When the user drags the window bigger, the swap chain resizes (you already handle that), but a 1280×720 render target keeps its old size — so your full-screen scene renders into a small texture, then gets stretched and blurry, or worse, your viewport spills past the target's edge. Every time you handle a resize event, call Resize() on every render target you own, passing the new client size. It is the same value you pass to the swap chain's ResizeBuffers.

One more subtlety hiding in Bind(): the viewport. Direct3D's viewport is independent of the render target — it does not auto-resize to fit. If you bound the target but left the window-sized viewport active, you would render the scene at the wrong scale. That is why Bind() always sets a viewport matching the target's own dimensions. Match the target size, the viewport, and the resize handler, and the whole thing stays crisp at any window size.

macOS — render targets in Metal

Metal expresses the same idea with slightly different nouns. A render target is an MTL::Texture created with two usage flags — RenderTarget so it can be a color attachment, and ShaderRead so a later shader can sample it. You attach it to a MTL::RenderPassDescriptor for pass 1, and bind it as a fragment texture for pass 2.

// ---- Create the off-screen color target ----
MTL::TextureDescriptor* desc =
    MTL::TextureDescriptor::texture2DDescriptor(
        MTL::PixelFormatRGBA8Unorm, width, height, /*mipmapped*/ false);
desc->setUsage(MTL::TextureUsageRenderTarget |
               MTL::TextureUsageShaderRead);   // written, then sampled
desc->setStorageMode(MTL::StorageModePrivate); // GPU-only, fastest

MTL::Texture* sceneTarget = device->newTexture(desc);

// ---- Pass 1: render into it via a render pass descriptor ----
MTL::RenderPassDescriptor* pass = MTL::RenderPassDescriptor::alloc()->init();
auto* color = pass->colorAttachments()->object(0);
color->setTexture(sceneTarget);
color->setLoadAction(MTL::LoadActionClear);
color->setClearColor(MTL::ClearColor(0.05, 0.05, 0.08, 1.0));
color->setStoreAction(MTL::StoreActionStore); // KEEP the result to read later

MTL::RenderCommandEncoder* enc = cmdBuffer->renderCommandEncoder(pass);
// ... draw the scene ...
enc->endEncoding();                            // finishes pass 1

// ---- Pass 2: a new encoder samples the stored texture ----
MTL::RenderCommandEncoder* post = cmdBuffer->renderCommandEncoder(screenPass);
post->setFragmentTexture(sceneTarget, 0);
// ... draw the fullscreen quad ...
post->endEncoding();

Two details carry the lesson's rules across to Metal. First, setStoreAction(StoreActionStore) tells Metal to keep the texture's contents after pass 1 — the default for a transient attachment can be don't care, which would throw the scene away before you sample it. Second, the read/write hazard is handled for you by ending the first encoder before starting the second: Metal tracks the dependency between two encoders on the same command buffer and orders them, the same guarantee you get in D3D11 by unbinding the RTV before binding the SRV. An HLSL Texture2D at register(t0) lines up with the fragment texture at index 0 here, so the sampling shader is nearly identical on both backends.

Recap
  • Post-processing, shadow maps and reflections all start the same way: render the scene into a texture you own, then read that texture in a later pass instead of drawing straight to the screen.
  • One texture carries two views — a render target view (RTV) to write into and a shader resource view (SRV) to sample from — so the color texture must be created with D3D11_BIND_RENDER_TARGET | D3D11_BIND_SHADER_RESOURCE.
  • A 3D off-screen pass also needs a matching depth target at the same resolution; here it is depth-only, so it needs just a DSV.
  • You cannot sample a texture while it is bound as output. Unbind the RTV (switch output back to the back buffer) before binding the SRV, and ping-pong between two targets when an effect must read and write the same image.
  • The RenderTarget class owns the textures and views and exposes Bind(), Clear(), GetSRV() and Resize(); Bind() must also set a viewport matching the target's size.
  • Recreate every render target whenever the window resizes — forgetting is the classic bug that leaves your scene stretched and blurry. In Metal, use RenderTarget | ShaderRead usage and StoreActionStore, and end each encoder to order the passes.
