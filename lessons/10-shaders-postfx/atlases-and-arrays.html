Texture Atlases & Texture Arrays

Our textured cube binds one image and draws once — easy. But a world made of thousands of blocks, each wanting its own face texture, cannot bind a new texture between every draw: each rebind is a state change and a fresh draw call, and that overhead crushes the frame rate long before the GPU is busy. This lesson shows the two ways engines pack many textures into one bindable resource so a whole scene collapses into a handful of draws. Both feed directly into the voxel engine we build in module 11.

What you'll learn
  • Why per-texture binding does not scale, and how batching many textures into one resource fixes it.
  • How to pack tiles into a texture atlas and compute the UV rectangle for any tile index.
  • Why atlases bleed at tile edges with linear filtering and mipmaps — and the three standard fixes.
  • How a Texture2DArray stacks same-size layers you index per-vertex, with clean mips and no bleeding.
  • How to build a texture array in Direct3D 11 and Metal, and sample both atlas and array in HLSL/MSL.

The problem: one bind per texture does not scale

In the last lessons a draw looked like this: set the shader resource view, set the sampler, then DrawIndexed. That is fine for one cube. Now imagine a Minecraft-style world where grass, dirt, stone, wood and sand all need different images. If each block type carries its own texture, the renderer must interrupt itself between block types:

// One bind + one draw PER texture. Death by a thousand state changes.
for (const BlockBatch& batch : batches)
{
    context->PSSetShaderResources(0, 1, &batch.srv); // state change
    context->DrawIndexed(batch.indexCount, batch.startIndex, 0); // draw
}

Each PSSetShaderResources forces the driver to validate and re-bind, and each texture switch splits what could have been one big draw into many small ones. A chunk touching forty block types costs forty draws instead of one. The fix is to stop switching: bind one resource that holds all the tiles, and let every vertex pick the tile it wants from inside it. Two resources can do that — a texture atlas and a texture array.

Approach A: a texture atlas

An atlas is one big texture that packs many small tiles side by side, usually on a square N × N grid. Nothing new happens on the GPU — it is still a single Texture2D sampled with ordinary UVs. All the cleverness is in the UVs: instead of spanning the whole 0..1 range, each face's UVs address only the sub-rectangle where its tile lives.

Computing a tile's UV rectangle

Number the tiles row-major from the top-left: tile 0 is top-left, tile N-1 ends the first row, and so on. For a tile index i on an N × N grid, one tile spans tileSize = 1/N of the texture in each axis. The column is i % N, the row is i / N (integer division), and the tile's top-left corner is at:

struct UVRect { float u0, v0, u1, v1; };

// N x N tiles packed row-major, top-left origin (v grows downward).
UVRect TileUV(int index, int N)
{
    float tileSize = 1.0f / static_cast<float>(N);
    int   col = index % N;
    int   row = index / N;

    float u0 = col * tileSize;
    float v0 = row * tileSize;
    // The tile spans [u0, u0 + tileSize] x [v0, v0 + tileSize].
    return { u0, v0, u0 + tileSize, v0 + tileSize };
}

When you build a mesh, each face's four corners take the corners of that rectangle: (u0,v0), (u1,v0), (u0,v1), (u1,v1). The pixel shader is unchanged from the textured-cube lesson — it still just samples a Texture2D with the interpolated UV:

Texture2D    gAtlas   : register(t0);
SamplerState gSampler : register(s0);

float4 PSMain(PSInput input) : SV_TARGET
{
    // UVs were pre-baked into the atlas sub-rectangle on the CPU.
    return gAtlas.Sample(gSampler, input.uv);
}

The catch: edge bleeding

Atlases look simple until you turn on the two things you learned in the previous lesson: linear filtering and mipmaps. A linear sampler reads a 2×2 block of texels and blends them. Near a tile's border, that block straddles the boundary and pulls in texels from the neighboring tile — so a sliver of dirt appears along the edge of your grass. Mipmaps make it far worse: at mip level k the tile has shrunk by a factor of 2k, so a single coarse-mip texel already averages pixels from several tiles at once.

Bleeding gets worse as things shrink

The seams that are invisible up close appear as flickering colored lines when a textured surface is far away or steeply angled — exactly when higher mip levels kick in. This is the classic reason a naive atlas looks broken on distant terrain, and why plain WRAP addressing can never tile a single atlas tile: wrapping re-enters the atlas at the opposite edge, landing in some other tile entirely.

There are three standard mitigations, and real engines usually combine them:

FixWhat you doCost / limit
Half-texel insetShrink each tile's UV rect inward by half a texel so the linear 2×2 tap never crosses the border.Cheap; only protects mip level 0.
Gutters / paddingLeave a border of duplicated edge pixels around each tile, so bleeding samples the copy, not the neighbor.Wastes atlas space; needs a wider gutter for each extra mip level.
Limit mip levelsCap MaxLOD on the sampler so the GPU never uses mips coarse enough to blend across tiles.Loses the aliasing benefit of small mips in the distance.

The half-texel inset is a one-line change to TileUV, given the atlas's pixel size:

// Inset by half a texel so bilinear taps stay inside the tile.
UVRect TileUVInset(int index, int N, int atlasPixels)
{
    UVRect r = TileUV(index, N);
    float half = 0.5f / static_cast<float>(atlasPixels); // atlas is square
    return { r.u0 + half, r.v0 + half, r.u1 - half, r.v1 - half };
}

Notice none of these fixes is free, and none fully survives an aggressive mip chain. That accumulating friction is exactly why the second approach exists.

Approach B: a texture array

A Texture2DArray is a stack of independent same-size layers living in one resource. You still bind one thing, but instead of carving one image into rectangles, you select a whole layer by an integer slice index. The sampling coordinate gains a third component:

Texture2DArray gAtlas   : register(t0);
SamplerState   gSampler : register(s0);

float4 PSMain(PSInput input) : SV_TARGET
{
    // float3(u, v, layer): the third component is the array slice.
    return gAtlas.Sample(gSampler, float3(input.uv, input.layer));
}

The consequences are all upside for tiled surfaces. Because each slice is a separate image, filtering and mipmaps operate within a slice and never reach a neighbor — no edge bleeding, clean mip chains, no gutters. And because a slice fills the full 0..1 UV space, you can tile it: pass UVs of 0..w with WRAP addressing and a face repeats its texture w times, which an atlas simply cannot do.

The layer coordinate is an index, not a UV

The third component of the sample coordinate is the array slice number (0, 1, 2, …), not a normalized 0..1 value, and it is not interpolated between slices — the GPU rounds to the nearest layer. Because it must be constant across a triangle, carry it as a per-vertex integer and mark it nointerpolation so the rasterizer passes it through untouched.

To route a per-vertex slice index through the pipeline, add a uint field to the vertex and a matching input element. In Direct3D 11 that element uses DXGI_FORMAT_R32_UINT; the varying is flagged so it is delivered whole:

struct VSInput
{
    float3 position : POSITION;
    float2 uv       : TEXCOORD0;
    uint   layer    : TEXCOORD1; // which slice this vertex reads
};

struct PSInput
{
    float4 position : SV_POSITION;
    float2 uv       : TEXCOORD0;
    nointerpolation uint layer : TEXCOORD1; // do not interpolate the index
};

Building the texture array

The concept is identical on every backend — one resource, N same-size slices — but the resource-creation call is platform-specific. We show Direct3D 11 (Windows) first, then the Metal (macOS) equivalent; the tile decoding that produces the pixel buffers stays shared engine code.

Direct3D 11

An array texture is an ordinary ID3D11Texture2D whose ArraySize is greater than one; every slice shares the same width, height, format and mip count. You can fill it two ways. The tidiest is to hand CreateTexture2D one D3D11_SUBRESOURCE_DATA per slice up front:

// tiles: N images, each tileW x tileH RGBA, already decoded in CPU memory.
ComPtr<ID3D11ShaderResourceView> CreateTileArray(
    ID3D11Device* device,
    const std::vector<std::vector<uint32_t>>& tiles,
    UINT tileW, UINT tileH)
{
    const UINT layerCount = static_cast<UINT>(tiles.size());

    D3D11_TEXTURE2D_DESC desc = {};
    desc.Width            = tileW;
    desc.Height           = tileH;
    desc.MipLevels        = 1;               // 1 = no mips (see note below)
    desc.ArraySize        = layerCount;      // <-- the stack of slices
    desc.Format           = DXGI_FORMAT_R8G8B8A8_UNORM_SRGB;
    desc.SampleDesc.Count = 1;
    desc.Usage            = D3D11_USAGE_IMMUTABLE;
    desc.BindFlags        = D3D11_BIND_SHADER_RESOURCE;

    // One initial-data entry per slice. With MipLevels=1 the subresource
    // index equals the slice index.
    std::vector<D3D11_SUBRESOURCE_DATA> init(layerCount);
    for (UINT i = 0; i < layerCount; ++i)
    {
        init[i].pSysMem          = tiles[i].data();
        init[i].SysMemPitch      = tileW * sizeof(uint32_t); // bytes per row
        init[i].SysMemSlicePitch = 0;                        // 2D: unused
    }

    ComPtr<ID3D11Texture2D> tex;
    if (FAILED(device->CreateTexture2D(&desc, init.data(), &tex)))
        return nullptr;

    D3D11_SHADER_RESOURCE_VIEW_DESC srvDesc = {};
    srvDesc.Format                    = desc.Format;
    srvDesc.ViewDimension             = D3D11_SRV_DIMENSION_TEXTURE2DARRAY;
    srvDesc.Texture2DArray.MipLevels  = desc.MipLevels;
    srvDesc.Texture2DArray.ArraySize  = layerCount; // view all slices
    srvDesc.Texture2DArray.FirstArraySlice = 0;

    ComPtr<ID3D11ShaderResourceView> srv;
    if (FAILED(device->CreateShaderResourceView(tex.Get(), &srvDesc, &srv)))
        return nullptr;

    return srv; // bind with PSSetShaderResources exactly as before
}

The alternative is to create the texture empty (pass nullptr for the initial data, and drop IMMUTABLE for D3D11_USAGE_DEFAULT) and upload each slice afterward with UpdateSubresource. This is handy when tiles are streamed in or generated at runtime. The key is addressing the right subresource: an array texture numbers its subresources as mip + slice * mipLevels, and D3D11CalcSubresource computes it for you.

// Upload one slice into an already-created DEFAULT-usage array texture.
for (UINT i = 0; i < layerCount; ++i)
{
    UINT sub = D3D11CalcSubresource(/*mip*/ 0, /*slice*/ i, desc.MipLevels);
    context->UpdateSubresource(
        tex.Get(), sub, nullptr,
        tiles[i].data(), tileW * sizeof(uint32_t), 0);
}
Want mips? Generate them per slice

Set MipLevels = 0 to allocate a full chain, add D3D11_BIND_RENDER_TARGET and MiscFlags = D3D11_RESOURCE_MISC_GENERATE_MIPS, then call context->GenerateMips(srv.Get()) after uploading. Because each slice mips independently, the result stays clean at every level — no gutters, no capped MaxLOD. That freedom from mip babysitting is the array's biggest practical win over an atlas.

The Metal equivalent

On macOS the same concept is a MTLTextureType2DArray. Set arrayLength on the descriptor, then upload each slice with replaceRegion, passing the slice index explicitly:

MTL::TextureDescriptor* desc = MTL::TextureDescriptor::alloc()->init();
desc->setTextureType(MTL::TextureType2DArray);
desc->setPixelFormat(MTL::PixelFormatRGBA8Unorm_sRGB);
desc->setWidth(tileW);
desc->setHeight(tileH);
desc->setArrayLength(layerCount);          // <-- the stack of slices
desc->setUsage(MTL::TextureUsageShaderRead);

MTL::Texture* tex = device->newTexture(desc);
desc->release();

MTL::Region region = MTL::Region::Make2D(0, 0, tileW, tileH);
for (NS::UInteger i = 0; i < layerCount; ++i)
{
    tex->replaceRegion(
        region, /*level*/ 0, /*slice*/ i,      // <-- per-slice upload
        tiles[i].data(),
        /*bytesPerRow*/ tileW * 4,
        /*bytesPerImage*/ 0);                  // 0 is fine for a 2D slice
}
// Bind: encoder->setFragmentTexture(tex, 0);

In Metal Shading Language the array is a texture2d_array<float>, and its sample takes the slice as a separate argument rather than a packed third coordinate:

fragment float4 fragMain(
    VSOut in [[stage_in]],
    texture2d_array<float> atlas [[texture(0)]],
    sampler samp [[sampler(0)]])
{
    return atlas.sample(samp, in.uv, in.layer); // uv, then slice index
}

Atlas vs. array: choosing

Both bind once and draw many; they differ in what they cost you at the edges and in the distance. This table is the summary worth remembering:

ConcernAtlas (Texture2D)Array (Texture2DArray)
Edge bleedingYes — neighbors bleed at tile borders; needs inset/gutters.None — slices are fully independent.
Mipmap friendlinessFragile — coarse mips blend across tiles.Clean — each slice mips on its own.
UV tiling (repeat > 1)Impossible — WRAP re-enters the atlas.Trivial — UV 0..w wraps inside the slice.
Tile sizeAny packing, but mixed sizes waste space.Every slice is one fixed size.
Hardware limitOne texture, up to 16384×16384 (D3D11 FL 11).Up to 2048 slices (D3D11).
Memory / setupOne image, tightest packing, simplest upload.One resource, N slices, a little more overhead.
What the pros ship

Classic Minecraft used a single atlas — terrain.png, a 16×16 grid of 16-pixel tiles — and fought edge bleeding with inset UVs and capped mips for years. Modern voxel engines almost universally switched to a texture array: block textures are the same size anyway, tiling and mips come for free, and per-face slice indices pack neatly into the mesh. We will build MiniEngine's block textures as an array in Texturing Voxels with an Atlas in module 11.

The rule of thumb: reach for an atlas when tiles vary in size and never tile or rely on distant mips (UI sprites, decals, glyph pages). Reach for a texture array whenever tiles are uniform in size and you want repeat-tiling and clean mips — which describes almost every voxel and terrain material. Everything you built here binds through the same PSSetShaderResources call you already know; only the resource type and the sampling coordinate change.

Recap
  • Binding a different texture per object costs a state change and a draw call each time; batching many textures into one resource is what makes thousands of blocks affordable.
  • An atlas packs tiles into one Texture2D; for tile i on an N×N grid, tileSize = 1/N, u0 = (i % N) * tileSize, v0 = (i / N) * tileSize, spanning [u0, u0+tileSize].
  • Atlases bleed at tile edges under linear filtering and mipmaps; mitigate with a half-texel UV inset, gutter padding, or a capped MaxLOD — none fully free.
  • A Texture2DArray stacks same-size layers sampled with tex.Sample(smp, float3(u, v, layer)); the integer layer selects a slice, so there is no bleeding, mips are clean, and UVs can tile 0..w.
  • Build an array in D3D11 by setting ArraySize and uploading one D3D11_SUBRESOURCE_DATA or UpdateSubresource per slice; in Metal use MTLTextureType2DArray with replaceRegion:slice:.
  • Prefer an array for uniform, tiling, mip-friendly block textures — the choice we carry into the voxel engine in module 11.
