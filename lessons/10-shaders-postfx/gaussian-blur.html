Separable Gaussian Blur

Almost every soft, expensive-looking effect in a modern renderer is a blur wearing a costume: bloom, soft shadows, depth-of-field, screen-space glow. A blur replaces each pixel with a weighted average of its neighbors, and the Gaussian weighting — a bell curve — looks the most natural. Done naïvely a wide blur is brutally slow, but the Gaussian hides a trick that turns one costly 2D pass into two cheap 1D passes. That trick is this lesson, and it is the direct engine for the bloom lesson that follows.

  • The two-pass HLSL (and Metal) fullscreen blur, with the per-pass texelSize that trips everyone up.
  • The linear-sampling optimization that roughly halves your tap count — and why blurring a downsampled target makes wide blurs almost free.

A blur is a weighted average

Look at any pixel in a blurred image and it is simply a mix of the pixels that surrounded it in the original. A box blur weights every neighbor equally — cheap, but it produces harsh, boxy artifacts and visible edges. A Gaussian blur weights nearby pixels heavily and distant ones lightly, following a bell curve, so the falloff is smooth and the result looks physically soft. The set of weights is called the kernel, and the number of pixels it reaches on each side is the radius.

The continuous 2D Gaussian is

G(x, y) = exp( -(x*x + y*y) / (2 * sigma*sigma) )

where sigma (the standard deviation) controls how wide the blur spreads. A larger sigma means a flatter bell and a softer, wider blur. In practice you pick a radius of about 3 * sigma: beyond three standard deviations the weights are so tiny they contribute essentially nothing, so sampling further is wasted work.

Here is the problem. If your kernel is k pixels wide (a radius of r = (k-1)/2 on each side), the naïve approach samples every texel in a k×k square around each pixel — that is k*k texture fetches per output pixel. Texture sampling is the single most expensive thing a post-process shader does, and this cost grows with the square of the radius. A modest 33-wide blur would cost 1,089 samples per pixel. At 1080p that is over two billion fetches for one blur. We need to do better.

Separability: the whole trick

Watch what happens when we factor the 2D Gaussian. Because exp(a + b) = exp(a) * exp(b):

G(x, y) = exp( -(x*x + y*y) / (2*sigma*sigma) )
        = exp( -x*x / (2*sigma*sigma) ) * exp( -y*y / (2*sigma*sigma) )
        = G(x) * G(y)

The 2D kernel is the product of two identical 1D kernels — one along x, one along y. A kernel that factors this way is called separable, and it changes everything. Write the blurred value at pixel p as the full 2D sum, then pull the x-independent term out of the inner loop:

Blurred(p) = sum over x, y of  Image(p + (x, y)) * G(x) * G(y)

           = sum over y of  G(y) * [ sum over x of  Image(p + (x, y)) * G(x) ]
                                     \_______________ horizontal 1D blur ______/

The bracketed inner sum is exactly a horizontal blur using the 1D kernel G(x). The outer sum then takes that horizontally-blurred image and applies a vertical blur with the same 1D kernel G(y). So blurring horizontally into a temporary target, then blurring that result vertically, produces the identical pixels as the full 2D kernel — not an approximation, the same numbers. And the cost collapses: each 1D pass does k samples, so the total is k + k = 2k samples per pixel instead of k*k.

Why the saving grows with radius

Naïve cost is O(k²) but separable cost is O(k). The wider the blur, the bigger the win — which is exactly backwards from your intuition about "expensive" effects. This is why real engines can afford enormous, buttery bloom: the underlying blur scales linearly, not quadratically, in radius.

Computing the 1D weights

Both passes share one 1D kernel, so we compute it once on the CPU. We only store one side plus the center — index 0 is the center weight, index i is the weight applied at both +i and -i, since the Gaussian is symmetric. The raw exp() values do not sum to 1, so we normalize; if we skipped that step the image would get brighter or darker instead of just softer.

#include <vector>
#include <cmath>

// Normalized 1D Gaussian weights for offsets 0..radius.
// weights[0]  = center weight (applied once)
// weights[i]  = weight applied at BOTH +i and -i (symmetric)
std::vector<float> GaussianWeights1D(int radius, float sigma)
{
    std::vector<float> weights(radius + 1);

    float sum = 0.0f;
    for (int i = 0; i <= radius; ++i)
    {
        // exp(-i^2 / (2*sigma^2)). The 1/(sqrt(2*pi)*sigma) scale factor
        // is a constant that cancels in the normalization, so we drop it.
        weights[i] = std::exp(-(float)(i * i) / (2.0f * sigma * sigma));

        // The center is counted once; every side offset is counted twice
        // because it is sampled at both +i and -i.
        sum += (i == 0) ? weights[i] : 2.0f * weights[i];
    }

    // Divide by the total so the FULL symmetric kernel sums to exactly 1.
    for (float& w : weights)
        w /= sum;

    return weights;
}
Normalize over the whole kernel, not one side

Because index i is used twice (at +i and -i), the normalization sum must count side weights twice — that is the 2.0f * weights[i] above. A very common bug is dividing by weights[0] + weights[1] + … as if it were one-sided; the kernel then sums to less than 1 and every blurred frame comes out darker than the source.

A reasonable pairing is sigma = radius / 3.0f, so the radius reaches out to about three standard deviations. Upload the resulting weights (and the radius) to a constant buffer for the shader to read.

The two-pass shader

You already have a fullscreen post-process pass from the earlier lesson: a shader that runs once per output pixel with an interpolated uv, sampling an input texture. We run it twice. Pass one reads the scene target and writes a horizontally-blurred image into a temporary target; pass two reads that temporary and writes the final vertically-blurred result. The only per-pass difference is the axis we step along.

The step size is texelSize = 1.0 / float2(width, height) — the width of one texel in UV space. Moving i texels sideways means adding texelSize.x * i to u; moving i texels down means adding texelSize.y * i to v.

Texture2D    gSource : register(t0);
SamplerState gClamp  : register(s0);   // linear filtering, CLAMP addressing

cbuffer BlurCB : register(b0)
{
    float2 gTexelSize;   // 1.0 / source (width, height) for THIS pass
    int    gRadius;      // samples to each side
    int    _pad;
    // HLSL pads each array element out to a full float4 (16 bytes),
    // so we store one weight per slot in .x.
    float4 gWeight[9];   // gWeight[i].x = weight at offset i
};

struct PSInput { float4 pos : SV_POSITION; float2 uv : TEXCOORD0; };

// ---- Horizontal pass: step along x ----
float4 PSHorizontal(PSInput input) : SV_TARGET
{
    float3 color = gSource.Sample(gClamp, input.uv).rgb * gWeight[0].x;

    for (int i = 1; i <= gRadius; ++i)
    {
        float2 off = float2(gTexelSize.x * i, 0.0);
        color += gSource.Sample(gClamp, input.uv + off).rgb * gWeight[i].x;
        color += gSource.Sample(gClamp, input.uv - off).rgb * gWeight[i].x;
    }
    return float4(color, 1.0);
}

// ---- Vertical pass: identical, but step along y ----
float4 PSVertical(PSInput input) : SV_TARGET
{
    float3 color = gSource.Sample(gClamp, input.uv).rgb * gWeight[0].x;

    for (int i = 1; i <= gRadius; ++i)
    {
        float2 off = float2(0.0, gTexelSize.y * i);
        color += gSource.Sample(gClamp, input.uv + off).rgb * gWeight[i].x;
        color += gSource.Sample(gClamp, input.uv - off).rgb * gWeight[i].x;
    }
    return float4(color, 1.0);
}
Clamp the sampler, and set texelSize per pass

Two pitfalls that produce subtle, maddening artifacts. First, use a clamp sampler (D3D11_TEXTURE_ADDRESS_CLAMP / MTLSamplerAddressModeClampToEdge): a wrap sampler makes taps that fall off one edge pull in pixels from the opposite edge, smearing bright light across the screen. Second, gTexelSize must match the resolution of the target being sampled in this pass. If pass one wrote into a half-resolution temporary, pass two's texelSize must be 1.0 / halfRes, not the full-res value — otherwise your vertical blur is the wrong width.

Constant-buffer arrays pad to 16 bytes

An HLSL float array in a constant buffer does not pack tightly — each element is aligned to a full float4. If you declare float gWeight[9] and upload nine tightly-packed floats from C++, the shader reads them from the wrong offsets. Either declare float4 gWeight[9] and index .x (as above), or write your weights into the .x lane of a float4 array on the CPU side. Metal's constant float* buffers pack tightly, so this is a D3D/HLSL-specific gotcha.

Halving the taps with linear sampling

We can do better than 2k. A bilinear (linear-filtered) sampler, given a UV that lands between two texel centers, returns a blend of both texels in a single fetch. If we position one tap at the right fractional offset between two neighboring texels, that one fetch does the work of two — so we can collapse each pair of side taps into a single sample.

Take two adjacent texels at offsets 1 and 2 with Gaussian weights w1 and w2. Place a single tap at

offset         = (w1 * 1 + w2 * 2) / (w1 + w2)   // weighted midpoint
combinedWeight =  w1 + w2

Why this is exact: a bilinear fetch at fractional offset p (with 1 ≤ p ≤ 2) returns (2 - p)*texel1 + (p - 1)*texel2. Multiply that by combinedWeight = w1 + w2 and you need combinedWeight*(2 - p) = w1 and combinedWeight*(p - 1) = w2. Solving gives exactly the offset above — and adding the two equations confirms combinedWeight = w1 + w2. The center texel has no partner, so it keeps its own dedicated tap. Fold the discrete weights into linear taps once, on the CPU:

struct LinearTaps
{
    float              center;    // weight of the untouched center texel
    std::vector<float> offsets;   // fractional texels from center (one side)
    std::vector<float> weights;   // combined weight for that tap
};

// Fold a discrete symmetric kernel into bilinear taps.
// discrete[0..radius] are the NORMALIZED weights from GaussianWeights1D.
LinearTaps BuildLinearTaps(const std::vector<float>& discrete)
{
    const int radius = (int)discrete.size() - 1;
    LinearTaps taps;
    taps.center = discrete[0];

    // Pair up side offsets: (1,2), (3,4), (5,6), ...
    for (int i = 1; i <= radius; i += 2)
    {
        float w1 = discrete[i];
        float w2 = (i + 1 <= radius) ? discrete[i + 1] : 0.0f;
        float combined = w1 + w2;

        // Weighted midpoint between texel i and texel i+1.
        // If w2 == 0 (odd radius, lone last tap) this reduces to i.
        float offset = (w1 * (float)i + w2 * (float)(i + 1)) / combined;

        taps.offsets.push_back(offset);
        taps.weights.push_back(combined);
    }
    return taps;
}

The shader loop is now shorter — it walks the precomputed taps and uses their fractional offsets directly:

// gCenter, gOffset[i], gTapWeight[i], gTapCount uploaded from LinearTaps.
float4 PSHorizontalLinear(PSInput input) : SV_TARGET
{
    float3 color = gSource.Sample(gClamp, input.uv).rgb * gCenter;

    for (int i = 0; i < gTapCount; ++i)
    {
        float2 off = float2(gTexelSize.x * gOffset[i].x, 0.0);
        color += gSource.Sample(gClamp, input.uv + off).rgb * gTapWeight[i].x;
        color += gSource.Sample(gClamp, input.uv - off).rgb * gTapWeight[i].x;
    }
    return float4(color, 1.0);
}

A radius-8 kernel needed 8 discrete side taps; folded into pairs it needs just 4. Per pass that is 1 center + 2×4 = 9 fetches instead of 17 — a little under half, with mathematically identical output because the bilinear split reconstructs the original weights exactly. The equivalence holds as long as the sampler is truly linear and the source grid aligns with your sample positions, which it does for a same-resolution fullscreen pass.

Blur where it is cheap: downsample first

The final lever is resolution. A blur radius measured in texels covers twice as much of the screen if the target is half-resolution. So the standard move is to downsample the source into a half- or quarter-resolution target first, blur that, and only then combine it back. A blur that would be enormous at full resolution becomes cheap: fewer pixels to shade and a smaller radius for the same visual spread. Because the blurred result is soft by definition, the lost sharpness from the downsample is invisible — you were about to smear it anyway.

Big, cheap blurs live at low resolution

Never run a wide blur at full resolution — it is the single most common post-process performance mistake. Downsample first. At quarter resolution you shade one sixteenth of the pixels, and a modest radius there spreads as far as a huge radius would at full res. The bloom lesson leans on exactly this: it blurs a small, downsampled bright-pass target and gets a wide, soft glow for almost nothing.

macOS — the same passes in Metal

The structure is entirely backend-neutral: two render passes into intermediate textures, driven by the same 1D weights. Only the resource plumbing changes. In Metal your fullscreen fragment shader samples a texture2d<float> with a sampler configured for ClampToEdge addressing and linear filtering, and reads texelSize plus the weights from a bound buffer.

fragment float4 blurHorizontal(VOut in [[stage_in]],
                               texture2d<float> src   [[texture(0)]],
                               sampler         samp  [[sampler(0)]],
                               constant BlurCB& cb   [[buffer(0)]])
{
    float3 color = src.sample(samp, in.uv).rgb * cb.weight[0];
    for (int i = 1; i <= cb.radius; ++i)
    {
        float2 off = float2(cb.texelSize.x * i, 0.0);
        color += src.sample(samp, in.uv + off).rgb * cb.weight[i];
        color += src.sample(samp, in.uv - off).rgb * cb.weight[i];
    }
    return float4(color, 1.0);
}

Note that Metal's constant buffers pack a plain float array tightly, so the HLSL float4-per-element workaround is not needed here — you can upload the weights as a straight float array. Everything else — the weight computation, the linear-tap folding, the two-pass ordering, the clamp sampler, the downsampled target — is shared engine code that both backends drive identically.

Recap
  • A blur is a weighted average of neighbors; Gaussian weights follow the bell curve G(x,y) = exp(-(x²+y²)/(2σ²)) for a smooth, natural falloff. A naïve k×k kernel costs k² samples per pixel.
  • The Gaussian is separable: G(x,y) = G(x)·G(y), so a horizontal 1D pass followed by a vertical 1D pass gives the identical result for only 2k samples. The saving grows with radius.
  • Compute one normalized 1D kernel on the CPU; normalize over the whole symmetric kernel (side weights counted twice) or the image will darken.
  • Run the same fullscreen shader twice, stepping along texelSize.x then texelSize.y. Use a clamp sampler and set texelSize to match the resolution sampled in each pass.
  • Linear sampling folds each pair of taps into one bilinear fetch — offset = (w1·1 + w2·2)/(w1+w2), weight = w1+w2 — roughly halving the tap count exactly.
  • Blur a downsampled target for wide, cheap blurs; never run a wide blur at full resolution. This is the exact engine the next lesson uses for bloom.
