HDR Color & Tone Mapping

Every color our engine has produced so far has been trapped between black and white. That is fine for a flat cube, but it makes a lamp look the same brightness as the wall behind it, and it leaves nothing for effects like bloom to feed on. In this lesson we let light go past white by rendering into a floating-point target, then we learn to squeeze that unbounded range back down to the 0–1 your monitor can actually show. This one change is what separates a toy renderer from a real one.

What you'll learn
  • Why an 8-bit target clamps light at 1.0, and why lamps, the sun, and bloom need values above it.
  • How to render your scene into a floating-point target (R16G16B16A16_FLOAT / RGBA16Float) so a pixel can hold 8.0.
  • What tone mapping is: a curve that compresses [0, ∞) into the [0, 1] a display can show.
  • How exposure scales the scene before mapping, and the Reinhard and ACES filmic curves.
  • The correct end-of-frame order — HDR scene → exposure → tone-map → gamma/sRGB — and the double-correction pitfall.

Why white is not the brightest thing in the world

Our back buffer, like most, stores each channel in 8 bits: an integer from 0 to 255, which the hardware reads as 0.0 to 1.0. That format is called LDR — low dynamic range. The problem is that 1.0 means "fully white," and there is no way to say "brighter than white." But the real world is full of things that are: a bare light bulb, a specular highlight glinting off metal, glowing lava, the sun. Photographers measure the ratio between the dimmest and brightest light in a scene in the millions to one. An 8-bit channel offers a ratio of 255 to 1, and everything above the top simply clamps to white.

Clamping loses information you cannot get back. A pixel that should be 8.0 and one that should be 1.2 both become 1.0, so a flashlight and a sheet of paper end up the exact same color. Worse, the effects we build next — bloom, which makes bright things bleed light into their surroundings — need to know how much brighter than white a pixel is. If the value was already clamped, the glow has nothing to work from. The fix is to stop clamping while we render, and only clamp at the very end, deliberately.

LDR vs. HDR

HDR — high dynamic range — just means colors are allowed to exceed 1.0. It is not a monitor feature here; even on an ordinary screen, rendering in HDR and tone mapping down produces visibly better images than clamping early. HDR display output is a separate, later concern.

A floating-point render target

To hold values above 1.0 we render the scene into an off-screen floating-point texture instead of directly into the back buffer. You already know how to draw into a texture from the render-targets lesson; the only change is the pixel format. On Windows we ask Direct3D 11 for DXGI_FORMAT_R16G16B16A16_FLOAT — a 16-bit half-float per channel, which comfortably stores numbers like 8.0 or 60.0. On macOS the Metal equivalent is MTLPixelFormatRGBA16Float.

// A full-resolution HDR scene target: 16-bit float per channel.
// It is both a render target (we draw into it) and a shader resource
// (the tone-map pass samples it), so it needs both bind flags.
D3D11_TEXTURE2D_DESC desc = {};
desc.Width            = backBufferWidth;
desc.Height           = backBufferHeight;
desc.MipLevels        = 1;
desc.ArraySize        = 1;
desc.Format           = DXGI_FORMAT_R16G16B16A16_FLOAT; // the key line
desc.SampleDesc.Count = 1;
desc.Usage            = D3D11_USAGE_DEFAULT;
desc.BindFlags        = D3D11_BIND_RENDER_TARGET | D3D11_BIND_SHADER_RESOURCE;

ComPtr<ID3D11Texture2D> hdrTex;
device->CreateTexture2D(&desc, nullptr, &hdrTex);

// A render-target view to draw the scene into...
ComPtr<ID3D11RenderTargetView> hdrRTV;
device->CreateRenderTargetView(hdrTex.Get(), nullptr, &hdrRTV);

// ...and a shader-resource view so the tone-map pass can read it back.
ComPtr<ID3D11ShaderResourceView> hdrSRV;
device->CreateShaderResourceView(hdrTex.Get(), nullptr, &hdrSRV);

Each frame you now bind hdrRTV (with your usual depth buffer), clear it, and render every object exactly as before. Nothing about your meshes, materials, or lighting shaders changes — they were always outputting linear color; they just used to be clamped on write, and now they are not. When the scene is fully drawn, hdrTex holds the raw, unbounded image, and we switch to a fullscreen pass to bring it back to earth.

Half-float is plenty — and cheaper

You may see engines use DXGI_FORMAT_R11G11B10_FLOAT for the HDR target. It packs three positive floats into 32 bits — half the memory and bandwidth of RGBA16F — at the cost of a little precision and no alpha channel. For a first HDR pipeline, RGBA16F is the safe, obvious choice; reach for R11G11B10 later when bandwidth matters.

Tone mapping: fitting infinity onto a screen

The HDR texture can contain any positive value, but the display can only show 0..1. Tone mapping is the function that bridges the two: it takes a color in [0, ∞) and returns one in [0, 1], gently compressing the bright end so highlights roll off smoothly instead of clipping to a flat white blob. A good tone-map curve keeps dark and midtone colors nearly unchanged, then bends over as inputs grow large, so a value of 8.0 still reads as clearly brighter than 2.0 even though both end up near the top of the range.

The simplest curve that does this is Reinhard. For a single channel it is just:

// Reinhard: divide by (self + 1). Maps [0, inf) -> [0, 1).
float3 Reinhard(float3 c)
{
    return c / (c + 1.0);
}

Feed it 0 and you get 0; feed it 1 and you get 0.5; feed it a huge number and it approaches 1 but never reaches it. It is cheap and never clips. Its weakness is that it desaturates and dims highlights — bright colors drift toward a washed-out grey. That is why film and games usually prefer a curve with more character, which we will get to after one more ingredient.

Keep all lighting in linear HDR

Do every bit of lighting, blending, and post-processing math in linear HDR, and tone-map exactly once, at the very end. Tone mapping is not a color you compute with — it is the final translation from "what the scene emits" to "what the panel can display." Mix tone-mapped and un-tone-mapped values and your math stops being physically meaningful.

Exposure: choosing what counts as bright

Before we map, we get to decide how bright the scene should feel — the same thing a camera's exposure setting controls. Exposure is a single multiplier applied to the color before tone mapping. Multiply by a number below 1 to darken a blindingly bright scene (a desert at noon), or above 1 to lift a dim one (a candle-lit room) until its detail rises off the floor:

// Scale linear HDR color by an exposure factor, then tone-map.
float3 exposed = hdrColor * exposure;   // exposure > 0
float3 mapped  = ACESFilm(exposed);     // curve defined below

Because exposure runs before the curve, changing it slides the whole scene along the tone-map curve — the bright end rolls off more or less aggressively without you touching the curve itself. A fixed exposure is a fine start; real engines compute it automatically each frame from the average scene brightness (auto-exposure or "eye adaptation"), which is a natural extension once this pipeline is in place. We will pass exposure in through a constant buffer so it can be tuned live.

Two curves: Reinhard and ACES

The curve most modern games reach for is an approximation of the ACES filmic tone-map, published by Krzysztof Narkowicz. It is a single rational function with five hand-fitted constants, and it produces punchy, film-like highlights with far less desaturation than Reinhard. Here it is exactly, as an HLSL function:

// ACES filmic approximation (Narkowicz 2015).
// Input: linear HDR color. Output: display-ready color in [0, 1].
float3 ACESFilm(float3 x)
{
    float a = 2.51;
    float b = 0.03;
    float c = 2.43;
    float d = 0.59;
    float e = 0.14;
    return saturate((x * (a * x + b)) / (x * (c * x + d) + e));
}

The saturate clamps the result to [0, 1] as a final safety net — the rational part already lands almost exactly in range, but tiny overshoots at the extremes are trimmed. The figure below plots both curves so you can see how they behave across an input range of 0 to 4.

Notice that neither curve ever touches 1.0: no matter how bright the input, pure white is only approached, never reached, so highlights keep their shape instead of blowing out into a featureless slab. ACES climbs faster and higher, which is why it looks brighter and more saturated at a matching exposure.

The tone-map pass in HLSL

Everything comes together in a single fullscreen pass. You already built the fullscreen-triangle vertex shader in the previous lesson; it hands us an interpolated UV covering the whole screen. Our pixel shader samples the HDR texture at that UV, applies exposure, tone-maps, converts to sRGB, and writes the final color to the back buffer.

// ---- Pixel shader: HDR scene -> display ----
Texture2D    gHDRScene : register(t0); // the RGBA16F scene target
SamplerState gSampler  : register(s0); // linear-clamp is fine here

cbuffer ToneMapParams : register(b0)
{
    float  gExposure;   // e.g. 1.0; tune live
    float3 _padding;    // pad to a 16-byte boundary
};

struct PSInput
{
    float4 position : SV_POSITION;
    float2 uv       : TEXCOORD0;
};

float3 ACESFilm(float3 x)
{
    float a = 2.51, b = 0.03, c = 2.43, d = 0.59, e = 0.14;
    return saturate((x * (a * x + b)) / (x * (c * x + d) + e));
}

float4 PSMain(PSInput input) : SV_TARGET
{
    // 1. Read the linear HDR color (may be far above 1.0).
    float3 hdr = gHDRScene.Sample(gSampler, input.uv).rgb;

    // 2. Apply exposure.
    hdr *= gExposure;

    // 3. Tone-map: HDR -> [0, 1].
    float3 ldr = ACESFilm(hdr);

    // 4. Linear -> sRGB. ONLY if the back buffer is NOT an _SRGB format.
    ldr = pow(ldr, 1.0 / 2.2);

    return float4(ldr, 1.0);
}

On the C++ side the pass is tiny: unbind the HDR texture as a render target, bind the back buffer instead, bind hdrSRV as the pixel-shader input, upload the exposure constant, and draw the fullscreen triangle. No depth buffer, no scene geometry.

// The HDR scene is fully rendered into hdrTex at this point.
struct ToneMapParams { float exposure; float pad[3]; };
ToneMapParams params{ 1.0f, {0, 0, 0} };
context->UpdateSubresource(toneMapCB.Get(), 0, nullptr, &params, 0, 0);

// Draw to the swap-chain back buffer now, not the HDR target.
context->OMSetRenderTargets(1, backBufferRTV.GetAddressOf(), nullptr);

ID3D11ShaderResourceView* srvs[] = { hdrSRV.Get() };
context->PSSetShaderResources(0, 1, srvs);
context->PSSetSamplers(0, 1, linearClamp.GetAddressOf());
context->PSSetConstantBuffers(0, 1, toneMapCB.GetAddressOf());

context->VSSetShader(fullscreenVS.Get(), nullptr, 0);
context->PSSetShader(toneMapPS.Get(), nullptr, 0);
context->Draw(3, 0); // one fullscreen triangle covers the screen
Unbind before you rebind

The HDR texture cannot be a render target and a shader input at the same time. Direct3D's debug layer will warn and silently null out one binding if you forget. The line OMSetRenderTargets(1, backBufferRTV..., nullptr) above releases hdrRTV, which is exactly what lets PSSetShaderResources read hdrSRV on the next line.

Gamma, sRGB, and not doing it twice

There is one last translation. Your monitor does not display linear light — it applies a roughly 2.2-power response curve, so a linear value must be encoded into sRGB gamma before it is shown, or the image comes out too dark. The cheap approximation is the pow(color, 1.0 / 2.2) from the shader. The true sRGB transfer function is a piecewise curve (a small linear segment near black, then a 2.4-power segment), and it is worth using when you need color accuracy, but 1/2.2 is visually close and a single instruction.

The catch is that the hardware can do this conversion for you. If you create the back buffer with an _SRGB format — DXGI_FORMAT_R8G8B8A8_UNORM_SRGB on Windows, an sRGB pixel format on Metal — then the GPU encodes linear→sRGB automatically on write. In that case your shader must output linear color and skip the pow entirely.

Do not tone-map twice, and do not double-gamma

Two classic wash-out bugs. First, tone-map exactly once — if a later post effect also applies a curve, your image goes flat and grey. Second, apply gamma either with pow(color, 1/2.2) in the shader or by using an _SRGB back buffer, never both. Doing both encodes sRGB twice and the whole frame turns pale and milky. Pick one, and note in your renderer which one you chose.

To keep the order straight, here is the full end-of-frame sequence. Bloom, which you will add in two lessons, slots in right after the scene is drawn and operates entirely in this HDR space — before tone mapping, because it needs those above-1.0 values to know what should glow:

#StepSpace
1Render the scene into the RGBA16F targetlinear HDR
2(Later) extract bright areas & add bloomlinear HDR
3Multiply by exposurelinear HDR
4Tone-map (ACES / Reinhard) to [0, 1]linear LDR
5Encode linear → sRGB (pow or _SRGB target)sRGB display

macOS — the same pipeline in Metal

The structure is identical; only the type names change. Create the HDR target with MTLPixelFormatRGBA16Float and a texture usage of MTLTextureUsageRenderTarget | MTLTextureUsageShaderRead. Render the scene into it, then run a fullscreen pass that samples it and writes the drawable. The tone-map math is pure arithmetic, so the ACES function ports to Metal Shading Language almost character-for-character:

// Metal Shading Language: same ACES constants, same result.
float3 ACESFilm(float3 x)
{
    float a = 2.51, b = 0.03, c = 2.43, d = 0.59, e = 0.14;
    return saturate((x * (a * x + b)) / (x * (c * x + d) + e));
}

fragment float4 toneMapPS(VSOut in [[stage_in]],
                          texture2d<float> hdrScene [[texture(0)]],
                          sampler samp              [[sampler(0)]],
                          constant float& exposure  [[buffer(0)]])
{
    float3 hdr = hdrScene.sample(samp, in.uv).rgb * exposure;
    float3 ldr = ACESFilm(hdr);
    ldr = pow(ldr, 1.0 / 2.2); // skip if the drawable is an sRGB format
    return float4(ldr, 1.0);
}

As on Windows, if the Metal drawable uses an sRGB pixel format the hardware handles the gamma encode — so drop the pow to avoid double correction. Keep the tone-map function and the pipeline order in shared, platform-neutral code; only the texture creation and pass setup live in the backend.

Recap
  • An 8-bit LDR target clamps color at 1.0, throwing away everything brighter than white and starving effects like bloom.
  • Render the scene into a floating-point target — DXGI_FORMAT_R16G16B16A16_FLOAT on Windows, MTLPixelFormatRGBA16Float on macOS — so a pixel can hold values like 8.0.
  • Tone mapping compresses [0, ∞) into [0, 1]. Reinhard is c / (c + 1); ACES is a five-constant filmic curve with punchier highlights.
  • Exposure is a single multiplier applied before the curve, sliding the whole scene along it — start with a fixed value, add auto-exposure later.
  • The order is fixed: HDR scene → (bloom) → exposure → tone-map → gamma/sRGB, all done in one fullscreen pass at the end.
  • Tone-map once, and apply gamma either with pow(color, 1/2.2) or an _SRGB back buffer — never both, or the frame washes out.
