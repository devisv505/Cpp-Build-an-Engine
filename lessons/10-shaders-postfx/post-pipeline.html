A Composable Post Stack & Engine Integration

You have written each effect on its own: an off-screen target, a fullscreen pass, tone mapping, a separable blur, and finally bloom. One at a time they are demos. Wired together by hand they are a tangle of intermediate textures nobody can reorder. In this lesson we turn them into engine architecture: a PostProcessStack that owns two ping-pong targets, runs an ordered list of effects, and hands the finished image to the swap chain. Any scene our engine renders now gets the whole chain for free.

What you'll learn
  • The ping-pong pattern: two full-resolution targets, each pass reads one and writes the other.
  • A tiny PostEffect interface — Apply(input, output) plus an enabled flag.
  • A PostProcessStack that owns the ping-pong targets and runs an ordered effect list.
  • Two cheap new effects — a radial vignette and FXAA — plus where distance fog fits.
  • How the Renderer renders the scene into an HDR target and calls the stack each frame, recreating every target on resize.

Why a stack, not a pile of if-statements

A first attempt at chaining effects usually lives inside EndFrame: bind target B, run bloom reading target A; bind target C, run tone map reading B; and on it goes. It works until you want to turn bloom off, or run FXAA before vignette instead of after, or profile the cost of each pass. Then every change means re-threading textures by hand and hoping you did not read and write the same one.

The engine answer is to make an effect a thing — an object with one job — and make the ordering data. Each effect reads a texture and writes a texture; it does not care what ran before it or comes after. A container owns the plumbing (the intermediate targets) and walks the list. Reordering becomes reordering a std::vector; toggling becomes flipping a bool.

The ping-pong pattern

An effect cannot read and write the same texture in one draw — the GPU is streaming pixels out of it while you ask for pixels back. So we keep two full-resolution targets, call them A and B, and bounce between them. Read A, write B. Read B, write A. Read A, write B. Two buffers are all you ever need, no matter how many effects you stack, because a pass only ever touches the frame before it.

The very first read is the scene's HDR image. The very last write is the swap-chain back buffer, so the finished picture lands straight on screen with no extra copy. Everything in between ping-pongs.

Two buffers is the whole trick

Beginners sometimes allocate one target per effect. You never need more than two, because a pass reads only the immediately previous result. The ping-pong scheme also guarantees the input and output are different textures on every pass, which is exactly what the GPU requires.

A PostEffect interface

Every effect boils down to the same shape: take a readable image, produce a written image. That is one virtual method. We add an enabled flag so the stack can skip a pass without anyone rewiring targets, and a name for logging and tooling. 

#pragma once

class ShaderResource; // a readable GPU texture (an SRV on D3D11, MTL::Texture on Metal)
class RenderTarget;   // a writable GPU texture you can also sample back afterwards

// One step of the post pipeline: read `input`, write the result into `output`.
class PostEffect {
public:
    virtual ~PostEffect() = default;

    // A fullscreen pass: sample `input`, draw the effect into `output`.
    virtual void Apply(ShaderResource* input, RenderTarget* output) = 0;

    virtual const char* Name() const = 0;

    bool enabled = true;   // the A/B and profiling switch
};

ShaderResource and RenderTarget are the engine-level wrappers from the render targets lesson: a RenderTarget owns a texture with both a render-target view (to write) and a shader-resource view (to read), and gives you back a ShaderResource* through AsShaderResource(). The interface names no D3D11 or Metal type, so the same effect list compiles on both backends.

The PostProcessStack

The stack owns the two ping-pong targets and the ordered list. Add appends an effect; list order is pipeline order. Process takes the scene's HDR image and the back buffer, and threads every enabled effect between them. 

#pragma once
#include <memory>
#include <vector>
#include "PostEffect.h"
#include "RenderTarget.h"
#include "CopyEffect.h"   // a one-line passthrough that just returns the sampled color

class Renderer;

class PostProcessStack {
public:
    void Initialize(Renderer* renderer, int width, int height);
    void Resize(int width, int height);

    // Effects run in the order you add them.
    void Add(std::unique_ptr<PostEffect> effect) {
        m_effects.push_back(std::move(effect));
    }

    // Run every enabled effect: scene HDR in, back buffer out.
    void Process(ShaderResource* sceneHDR, RenderTarget* backBuffer);

private:
    Renderer*    m_renderer = nullptr;
    RenderTarget m_targets[2];        // the two full-res ping-pong buffers (A, B)
    int          m_width  = 0;
    int          m_height = 0;

    std::vector<std::unique_ptr<PostEffect>> m_effects;
    CopyEffect   m_blit;              // used when nothing is enabled
};

Resize is where the targets are born and reborn. Both are full-resolution and HDR-capable. Initialize just remembers the renderer and calls Resize once for the initial size. 

#include "PostProcessStack.h"
#include "Renderer.h"

void PostProcessStack::Initialize(Renderer* renderer, int width, int height) {
    m_renderer = renderer;
    Resize(width, height);           // build both targets at the starting size
}

void PostProcessStack::Resize(int width, int height) {
    m_width  = width;
    m_height = height;

    // Both ping-pong targets are full-resolution HDR floats. Values become LDR
    // once tone mapping runs, but they keep living in a float texture until the
    // very last pass writes the 8-bit back buffer.
    for (RenderTarget& rt : m_targets)
        rt.Recreate(m_renderer, width, height, TextureFormat::RGBA16F);
}

Now the heart of it. Collect the enabled effects, then walk them: the last one writes the back buffer, all the others ping-pong. Notice w ^= 1 — a one-bit flip between index 0 and 1 — and that the next input is always the target we just wrote. 

void PostProcessStack::Process(ShaderResource* sceneHDR, RenderTarget* backBuffer) {
    // Collect the effects switched on this frame.
    std::vector<PostEffect*> active;
    for (auto& e : m_effects)
        if (e->enabled) active.push_back(e.get());

    // Nothing on? Copy the scene straight to the back buffer and bail.
    if (active.empty()) {
        m_blit.Apply(sceneHDR, backBuffer);
        return;
    }

    ShaderResource* input = sceneHDR;   // the first read is the scene
    int             w     = 0;          // index of the next target to write (A)

    for (size_t i = 0; i < active.size(); ++i) {
        const bool last = (i + 1 == active.size());

        // The last effect writes the swap-chain back buffer; the rest
        // ping-pong between targets A and B.
        RenderTarget* output = last ? backBuffer : &m_targets[w];
        active[i]->Apply(input, output);

        if (!last) {
            input = m_targets[w].AsShaderResource(); // read what we just wrote
            w ^= 1;                                  // flip A <-> B for next time
        }
    }
}

Trace it with bloom, tone map, FXAA, vignette all on. Bloom reads the scene, writes A. Tone map reads A, writes B. FXAA reads B, writes A. Vignette is last: it reads A and writes the back buffer. Four passes, two targets, and no pass ever reads the texture it is writing.

An effect may own its own targets

The stack manages only the two full-resolution ping-pong buffers. Bloom needs more than that — a chain of half-, quarter-, eighth-resolution targets for its downsample-blur-upsample. Those live inside BloomEffect. From the stack's point of view bloom is still one HDR-in, HDR-out pass; it just does extra work behind its own Apply.

What an Apply actually does

Every effect's Apply follows the same recipe you built in the fullscreen-pass lesson: bind the output as the render target, bind the input as texture t0, set the shared fullscreen vertex shader plus this effect's pixel shader, and draw three vertices. Here is the vignette effect in full on the Direct3D 11 backend so the shape is concrete. 

void VignetteEffect::Apply(ShaderResource* input, RenderTarget* output) {
    ID3D11DeviceContext* ctx = m_renderer->Context();

    // 1) Draw into `output`. A fullscreen pass needs no depth buffer.
    output->BindAsTarget(ctx, /*withDepth*/ false);

    // 2) Feed the previous image in as texture t0, with a clamp sampler on s0.
    ID3D11ShaderResourceView* srv = input->SRV();
    ctx->PSSetShaderResources(0, 1, &srv);
    ctx->PSSetSamplers(0, 1, m_sampler.GetAddressOf());

    // 3) Our parameters (inner, outer, strength) go in constant buffer b0.
    UpdateConstantBuffer(ctx, m_params);
    ctx->PSSetConstantBuffers(0, 1, m_cb.GetAddressOf());

    // 4) Bind the shared fullscreen VS + this effect's PS, then draw the triangle.
    m_renderer->BindFullscreenPipeline(m_pixelShader.Get());
    ctx->Draw(3, 0);

    // 5) Unbind t0 so the NEXT pass may bind this texture as its render target.
    ID3D11ShaderResourceView* none = nullptr;
    ctx->PSSetShaderResources(0, 1, &none);
}
Unbind the input before it becomes an output

Step 5 is not optional. Because A and B swap roles, the texture you sampled this pass is the texture you write next pass. If it is still bound as t0 when it becomes a render target, D3D11 silently unbinds it and floods the debug output with hazard warnings; Metal's validation layer errors outright. One nullptr at the end of each Apply keeps the ping-pong clean.

Two cheap effects: vignette and FXAA

Vignette

A vignette darkens the image toward the corners — a subtle frame that pulls the eye to the center. It is one line of real work: measure how far a pixel is from the screen center and fade it out past a radius. HLSL first (the primary dialect for our Windows backend). 

Texture2D    SceneTex : register(t0);
SamplerState Samp     : register(s0);

cbuffer VignetteCB : register(b0)
{
    float uInner;    // radius where darkening begins   (~0.35)
    float uOuter;    // radius that is fully dark        (~0.75)
    float uStrength; // 0 = off, 1 = full vignette
    float _pad;      // round the cbuffer up to 16 bytes
};

float4 main(float2 uv : TEXCOORD0) : SV_Target
{
    float3 col  = SceneTex.Sample(Samp, uv).rgb;

    // 0 at the center, about 0.707 out at the corners.
    float dist  = length(uv - 0.5);

    // smoothstep(outer, inner, dist): 1 in the middle, easing to 0 past the edge.
    float mask  = smoothstep(uOuter, uInner, dist);

    // Blend by strength so the effect stays data-driven and toggleable.
    col *= lerp(1.0, mask, uStrength);
    return float4(col, 1.0);
}

Because uInner < uOuter, the reversed smoothstep(uOuter, uInner, dist) returns 1 near the center and falls smoothly to 0 at the edges — exactly the mask we want. An even cheaper approximation drops the smoothstep entirely: col *= 1.0 - dot(d, d) * uStrength where d = uv - 0.5, a parabolic falloff that costs a multiply-add. In Metal Shading Language the body is identical — smoothstep, length, and mix (Metal's name for lerp) all exist; only the entry-point signature changes, taking a texture2d<float> and a sampler as parameters.

FXAA

FXAA (Fast Approximate Anti-Aliasing) smooths jagged edges as a post pass, with no extra geometry samples. It runs on the tone-mapped LDR image on purpose: its edge test is based on perceived brightness (luma), so it wants gamma-encoded color, not linear HDR. The idea is small — find edges by local luma contrast, then blur along the edge to soften the stair-step. Here is the gist; the production version is Timothy Lottes' FXAA 3.11, which you drop straight into this shader slot. 

Texture2D    SceneTex : register(t0);
SamplerState Samp     : register(s0);

cbuffer FXAAConfig : register(b0)
{
    float2 uInvScreen; // 1.0 / (width, height) -- one texel in UV space
    float2 _pad;
};

static const float EDGE_THRESHOLD = 0.125;  // relative contrast that counts as an edge
static const float EDGE_MIN       = 0.0312; // absolute floor, so dark noise is ignored

float Luma(float3 c) { return dot(c, float3(0.299, 0.587, 0.114)); }

float4 main(float2 uv : TEXCOORD0) : SV_Target
{
    float2 t = uInvScreen;

    // Luma of this pixel and its four neighbours.
    float lC = Luma(SceneTex.Sample(Samp, uv).rgb);
    float lN = Luma(SceneTex.Sample(Samp, uv + float2( 0.0, -t.y)).rgb);
    float lS = Luma(SceneTex.Sample(Samp, uv + float2( 0.0,  t.y)).rgb);
    float lE = Luma(SceneTex.Sample(Samp, uv + float2( t.x,  0.0)).rgb);
    float lW = Luma(SceneTex.Sample(Samp, uv + float2(-t.x,  0.0)).rgb);

    float lMin = min(lC, min(min(lN, lS), min(lE, lW)));
    float lMax = max(lC, max(max(lN, lS), max(lE, lW)));
    float contrast = lMax - lMin;

    // Flat area: no edge here, leave the pixel exactly as it is.
    if (contrast < max(EDGE_MIN, lMax * EDGE_THRESHOLD))
        return SceneTex.Sample(Samp, uv);

    // Edge found. Estimate the luma gradient, then blur perpendicular to it
    // (that is, ALONG the edge). Full FXAA does a longer directional search;
    // this two-tap blend is the essence of it.
    float2 grad = float2(lE - lW, lS - lN);
    float2 dir  = normalize(float2(-grad.y, grad.x) + 1e-5);
    float3 a = SceneTex.Sample(Samp, uv - dir * t).rgb;
    float3 b = SceneTex.Sample(Samp, uv + dir * t).rgb;
    return float4(0.5 * (a + b), 1.0);
}
Keep every effect toggleable

The enabled flag is not just a convenience — it is your profiling and art-direction tool. Flip bloom off and read the frame time to learn its exact cost. A/B FXAA on and off to judge whether the shimmer is gone. Wire the flags to keys or an ImGui panel and tune the look live: m_post.Find("Bloom")->enabled = false;.

Where distance fog fits

Depth-based distance fog can go in either camp. The cheapest home is inside your scene shaders: a fragment already knows its own depth, so you lerp its color toward a fog color by view-space distance right there. As a post pass it is still doable — you sample the depth target you already keep for the depth buffer, reconstruct linear distance from the stored depth, and blend. Note that a fog PostEffect then needs a second bound texture (the depth SRV) beyond its input, which is the one case where an effect reads more than the previous color image. Bind it as t1 in that effect's Apply and the stack is none the wiser.

Wiring it into the Renderer

Two changes make the whole thing live. First, BeginFrame stops binding the back buffer and binds the HDR scene target instead — the scene now renders into float. Second, EndFrame hands that target to the stack, which paints the back buffer. The game code between them does not change at all. 

#include <SDL3/SDL.h>

// --- new members in Renderer.h ---
//   RenderTarget      m_sceneHDR;    // the scene renders here, in float HDR
//   RenderTarget      m_backBuffer;  // wraps the swap chain's back buffer (LDR)
//   PostProcessStack  m_post;
//   bool              m_vsync = true;

bool Renderer::Initialize(SDL_Window* window, bool vsync) {
    // ... create device, swap chain, wrap m_backBuffer as before ...
    m_vsync = vsync;
    int w = 0;
    int h = 0;
    if (!SDL_GetWindowSizeInPixels(window, &w, &h) ||
        w <= 0 || h <= 0) {
        return false;
    }

    m_sceneHDR.Recreate(this, w, h, TextureFormat::RGBA16F);  // HDR + a depth buffer
    m_post.Initialize(this, w, h);

    // Order is the pipeline. Bloom stays HDR; tone map crosses to LDR;
    // FXAA and vignette clean up the LDR image.
    m_post.Add(std::make_unique<BloomEffect>(this, /*threshold*/ 1.0f, /*intensity*/ 0.6f));
    m_post.Add(std::make_unique<ToneMapEffect>(this, /*exposure*/ 1.2f));
    m_post.Add(std::make_unique<FXAAEffect>(this));
    m_post.Add(std::make_unique<VignetteEffect>(this, /*inner*/ 0.35f, /*outer*/ 0.75f, /*strength*/ 0.6f));
    return true;
}

void Renderer::BeginFrame(const std::array<float, 4>& clear) {
    // The scene draws into the HDR target now, NOT the back buffer.
    m_sceneHDR.BindAsTarget(m_context.Get(), /*withDepth*/ true);
    m_sceneHDR.Clear(m_context.Get(), clear.data());
}

void Renderer::EndFrame() {
    // Hand the finished HDR scene to the stack; it writes the back buffer.
    m_post.Process(m_sceneHDR.AsShaderResource(), &m_backBuffer);
    m_swapChain->Present(m_vsync ? 1 : 0, 0);
}

void Renderer::Resize(int w, int h) {
    m_backBuffer.Reset();
    m_swapChain->ResizeBuffers(0, w, h, DXGI_FORMAT_UNKNOWN, 0);
    m_backBuffer.WrapSwapChain(m_device.Get(), m_swapChain.Get()); // rebuild the RTV

    m_sceneHDR.Recreate(this, w, h, TextureFormat::RGBA16F);  // HDR scene target
    m_post.Resize(w, h);                                      // both ping-pong targets
}

SDL_GetWindowSizeInPixels continues the same high-DPI rule established by the Window and Renderer lessons. The post-processing targets, depth target, viewport, and swap-chain back buffer must all use the same physical pixel dimensions.

On resize, recreate everything — and keep formats straight

A single missed target is a stale-size bug: the swap chain grows, one intermediate stays small, and the image smears. Resize must rebuild the back buffer, the HDR scene target, and both ping-pong targets. And mind the formats — the scene and ping-pong targets are RGBA16F so HDR values survive until tone mapping; only the back buffer is 8-bit UNORM. Tone-mapped LDR values simply ride along in the float ping-pong targets until the final pass writes those 8 bits. Feed HDR bloom an 8-bit buffer by mistake and every bright pixel clips to white before it can glow.

Order is the whole design

The list order encodes real dependencies, and it is worth saying why this one is right:

PassSpaceWhy here
BloomHDRGlow comes from truly bright values; it must run before tone mapping compresses them.
Tone mapHDR → LDRThe one conversion from unbounded float to a displayable 0–1 range.
FXAALDREdge detection reads perceptual luma, so it needs the tone-mapped image.
VignetteLDRA final artistic darkening on the finished picture.

Swap two rows and you can see the design in the result: tone-map before bloom and the glow never blooms; FXAA before tone-map and it anti-aliases the wrong brightness. The stack lets you try exactly that in seconds — reorder the Add calls — which is the point of making order into data.

That is the module. You began with a flat textured cube and finished with a composable HDR pipeline that any scene inherits automatically. In the next module you turn this engine into a Minecraft-style voxel sandbox — and the first time you place a glowstone block, the very stack you built here catches its brightness and blooms it, no new plumbing required.

Recap
  • Post effects ping-pong between just two full-resolution targets: read A write B, read B write A; the final pass writes the swap-chain back buffer.
  • A PostEffect is one virtual Apply(input, output) plus an enabled flag — reordering and toggling become data, not rewiring.
  • PostProcessStack owns the two targets and an ordered list; Process threads every enabled effect and guarantees input never equals output.
  • Unbind each effect's input texture after drawing, or it will still be bound when the ping-pong makes it the next output.
  • Vignette is a reversed smoothstep radial falloff; FXAA is luma edge-detect-then-blur on the LDR image (drop in reference FXAA 3.11 for production).
  • BeginFrame binds the HDR scene target; EndFrame calls Process; Resize recreates every target and keeps HDR float until after tone mapping.
