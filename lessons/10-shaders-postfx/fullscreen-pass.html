The Fullscreen Post-Process Pass

In the last lesson we stopped drawing to the screen and drew the whole scene into a texture instead. That texture is now just data — an image we can read and transform. This lesson teaches the one mechanic that turns that image into any effect you can imagine: draw geometry that covers the entire screen, and run a pixel shader over every pixel of the scene. Grayscale, invert, tone mapping, blur, bloom — they are all this same pass with a different pixel shader. Get this right once and every effect that follows is just math.

What you'll learn
  • What a fullscreen pass is and why it reads one texture and writes another.
  • The single-triangle trick — why one oversized triangle beats a two-triangle quad.
  • How to generate that triangle in HLSL from SV_VertexID with no vertex buffer bound.
  • A pass-through pixel shader plus grayscale, invert and tint variations.
  • A reusable FullscreenPass in our engine that any effect can drive with Draw(3, 0).

What a fullscreen pass actually is

A normal draw takes 3D geometry, transforms it by a camera matrix, and shades whatever fragments the triangles cover. A fullscreen pass throws all of that away. There is no model, no camera, no lighting — just a shape that blankets the viewport so the rasterizer generates exactly one fragment per output pixel. In the pixel shader we sample the scene texture from the previous lesson at this pixel's coordinate, do some math, and write the result. Input is one texture (bound as a shader resource view); output is another texture or the swap-chain back buffer (bound as a render target).

Every post-processing effect is this shape:

read  sceneSRV  (the rendered scene, from the previous lesson)
  for every output pixel:
    color = sceneSRV.Sample(clampSampler, uv)   // this pixel's scene color
    color = transform(color)                     // the effect lives here
write outputRTV (back buffer, or the next texture in the chain)

Because the transform runs per pixel with the full scene color already available, the pass is where tone mapping decides how to compress HDR, where blur averages neighbours, and where bloom adds glow. First we need the geometry that covers the screen.

One triangle, not two

The instinct is to draw a screen-sized quad: two triangles forming a rectangle from (-1,-1) to (1,1) in clip space. That works, but the professional trick is to draw a single triangle so large it overhangs the screen on all sides. The GPU clips the parts that fall outside the viewport, and what remains covers every pixel exactly once.

Why one triangle beats a quad

A quad has a diagonal seam where its two triangles meet. The GPU rasterizes in small quads of pixels (2×2), and along that seam pixels get shaded twice or fall into partial helper lanes — wasted work and, with derivative-sensitive effects, a faint visible line. A single triangle has no interior edge crossing the screen, so every screen pixel is covered by one continuous surface with perfectly linear UV interpolation. It is a touch faster and strictly cleaner.

Generating the triangle from the vertex id

Here is the elegant part: we do not upload any vertices. The vertex shader is invoked three times, and each invocation receives its index in SV_VertexID (0, 1, 2). From that single integer we compute both the clip-space position and the texture coordinate with a couple of bit operations. No vertex buffer, no input layout, no memory traffic.

struct VSOut
{
    float4 position : SV_POSITION; // clip-space position, required
    float2 uv       : TEXCOORD0;   // where to sample the scene texture
};

// Invoked 3 times. id is 0, 1, 2. No vertex buffer is bound.
VSOut FullscreenVS(uint id : SV_VertexID)
{
    VSOut o;

    // id 0 -> (0,0),  id 1 -> (2,0),  id 2 -> (0,2)
    float2 uv = float2((id << 1) & 2, id & 2);
    o.uv = uv;

    // Map uv in [0,2] to clip space in [-1,3], flipping V so that
    // uv (0,0) lands at the TOP-LEFT of the screen (D3D texture origin).
    o.position = float4(uv * float2(2.0, -2.0) + float2(-1.0, 1.0), 0.0, 1.0);
    return o;
}

Walk the three invocations through the math and you get the vertices in the figure:

The three clip positions (-1,1), (3,1), (-1,-3) form a right triangle whose legs stretch to twice the screen size, so it fully encloses the [-1,1] square. The UVs run from (0,0) to (2,0) and (0,2); after the rasterizer interpolates and clipping trims the overhang, the four visible screen corners end up with exactly (0,0), (1,0), (0,1), (1,1) — a perfect 0..1 UV map over the scene texture.

No buffers, no layout

All three positions come out of SV_VertexID, so there is nothing to allocate, upload, or describe with an input layout. Bind a null vertex buffer, set TRIANGLELIST topology, and issue Draw(3, 0). This shader is compiled once and shared by every post-process effect in the engine — only the pixel shader changes.

The pixel shader: pass-through, then effects

With the geometry handled, the pixel shader is where the effect lives. The simplest one just samples the scene texture at the interpolated UV and returns it unchanged — a copy pass. We declare the scene texture at t0 and the sampler at s0, matching the C++ binding slots.

Texture2D    sceneTex : register(t0); // the rendered scene from last lesson
SamplerState smp      : register(s0); // a CLAMP sampler (see warning below)

// Pass-through: copy the scene one-to-one.
float4 PSCopy(VSOut input) : SV_TARGET
{
    return sceneTex.Sample(smp, input.uv);
}

Because the color of every pixel is now in your hands, an effect is just a function of that color. Here are three that prove the point — each is one line of change from the copy pass.

// Grayscale via perceptual luminance (Rec. 601 weights).
float4 PSGrayscale(VSOut input) : SV_TARGET
{
    float4 c = sceneTex.Sample(smp, input.uv);
    float g = dot(c.rgb, float3(0.299, 0.587, 0.114));
    return float4(g, g, g, c.a);
}

// Photographic negative: flip each channel.
float4 PSInvert(VSOut input) : SV_TARGET
{
    float4 c = sceneTex.Sample(smp, input.uv);
    return float4(1.0 - c.rgb, c.a);
}

// Warm tint: scale channels toward amber.
float4 PSTint(VSOut input) : SV_TARGET
{
    float4 c = sceneTex.Sample(smp, input.uv);
    return float4(c.rgb * float3(1.10, 0.92, 0.72), c.a);
}

The grayscale weights are the standard luminance coefficients — the eye is far more sensitive to green than to blue, so a naive (r+g+b)/3 looks muddy by comparison. Everything harder that follows — tone mapping curves, blur kernels, bloom composites — is the same skeleton: sample, compute, return.

Use a CLAMP sampler, and mind the V flip

Bind a sampler with ADDRESS_CLAMP, not WRAP. Blur and other neighbour-reading effects sample slightly outside 0..1 at the screen edges; wrapping would pull in color from the opposite side of the image, clamping repeats the edge pixel instead. Also note the * float2(2,-2) + float2(-1,1) in the vertex shader: clip-space Y points up while a Direct3D texture's V axis points down (V=0 is the top row). The -2 flips V so the scene is not rendered upside down. Get this wrong and everything is mirrored vertically.

The draw call

On Direct3D 11 the whole pass is a handful of state changes and a three-vertex draw. Notice there is no input layout, no vertex buffer, and no index buffer.

// Output: write to the back buffer (or the next texture in the chain).
context->OMSetRenderTargets(1, backBufferRTV.GetAddressOf(), nullptr);

// The viewport must match the OUTPUT size, not the scene texture size.
D3D11_VIEWPORT vp = {};
vp.Width = (FLOAT)outputWidth; vp.Height = (FLOAT)outputHeight;
vp.MaxDepth = 1.0f;
context->RSSetViewports(1, &vp);

// No geometry to fetch: null vertex buffer, no input layout, list topology.
context->IASetInputLayout(nullptr);
context->IASetVertexBuffers(0, 0, nullptr, nullptr, nullptr);
context->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

// Shared fullscreen VS + the effect's pixel shader.
context->VSSetShader(fullscreenVS.Get(), nullptr, 0);
context->PSSetShader(effectPS.Get(),     nullptr, 0);

// Input: the scene texture as a shader resource, plus a clamp sampler.
ID3D11ShaderResourceView* srvs[]  = { sceneSRV.Get() };
ID3D11SamplerState*       samps[] = { clampSampler.Get() };
context->PSSetShaderResources(0, 1, srvs);
context->PSSetSamplers(0, 1, samps);

// Three vertices generated entirely from SV_VertexID.
context->Draw(3, 0);
Never read and write the same texture

A texture cannot be bound as an input SRV and an output render target at the same time. Direct3D silently unbinds one and the debug layer prints a warning; the sampled result is undefined. That is exactly why the previous lesson rendered into its own texture — so this pass can read that texture and write somewhere else. When you chain effects, you ping-pong between two intermediate textures: read A / write B, then read B / write A. We will lean on that in the blur and bloom lessons.

A reusable FullscreenPass in the engine

Every effect repeats the same dance — set the output target, bind an input SRV and sampler, set the shared VS and an effect PS, draw three vertices, then detach the input. That is a perfect thing to wrap once. Our engine gets a small FullscreenPass that owns the shared vertex shader and exposes a single Draw.

#include <d3d11.h>
#include <wrl/client.h>

using Microsoft::WRL::ComPtr;

// Owns the shared triangle-generating vertex shader. One per renderer.
class FullscreenPass
{
public:
    // Compile the fullscreen VS once at startup from its bytecode.
    void Init(ID3D11Device* device, const void* vsBytecode, size_t vsSize)
    {
        device->CreateVertexShader(vsBytecode, vsSize, nullptr, &m_vs);
    }

    // Run one effect: read `input`, write `output`, with `effectPS`.
    void Draw(ID3D11DeviceContext* ctx,
              ID3D11ShaderResourceView* input,
              ID3D11RenderTargetView*   output,
              ID3D11PixelShader*        effectPS,
              ID3D11SamplerState*       sampler)
    {
        ctx->OMSetRenderTargets(1, &output, nullptr);

        ctx->IASetInputLayout(nullptr);
        ctx->IASetVertexBuffers(0, 0, nullptr, nullptr, nullptr);
        ctx->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

        ctx->VSSetShader(m_vs.Get(), nullptr, 0);
        ctx->PSSetShader(effectPS,   nullptr, 0);
        ctx->PSSetShaderResources(0, 1, &input);
        ctx->PSSetSamplers(0, 1, &sampler);

        ctx->Draw(3, 0);

        // Detach the input so this texture can be an OUTPUT next pass.
        ID3D11ShaderResourceView* nullSRV[] = { nullptr };
        ctx->PSSetShaderResources(0, 1, nullSRV);
    }

private:
    ComPtr<ID3D11VertexShader> m_vs; // shared across all effects
};

The whole point is that swapping an effect is now a one-argument change. Applying grayscale to the rendered scene and presenting it becomes three self-explanatory lines:

// ... earlier: rendered the scene into sceneTexture (sceneRTV / sceneSRV) ...

// Post-process: read the scene, write the back buffer, using grayscale.
fullscreen.Draw(context, sceneSRV.Get(), backBufferRTV.Get(),
                grayscalePS.Get(), clampSampler.Get());

swapChain->Present(1, 0);

Point output at another intermediate texture instead of the back buffer and you can stack passes: scene → blur → bloom composite → tone map → screen. Each link is one FullscreenPass::Draw with a different pixel shader and the two textures swapped. That is the entire architecture of the post-processing stack we build across the rest of this module.

macOS — the same pass in Metal

Metal expresses the identical idea. The vertex function takes [[vertex_id]] and generates the triangle with the same bit math — Metal's clip space and texture origin match Direct3D's, so the flip is the same. You bind no vertex buffer and call drawPrimitives for three vertices.

#include <metal_stdlib>
using namespace metal;

struct VSOut {
    float4 position [[position]];
    float2 uv;
};

vertex VSOut fullscreenVS(uint id [[vertex_id]])
{
    VSOut o;
    float2 uv = float2((id << 1) & 2, id & 2);
    o.uv = uv;
    o.position = float4(uv * float2(2.0, -2.0) + float2(-1.0, 1.0), 0.0, 1.0);
    return o;
}

fragment float4 grayscaleFS(VSOut in [[stage_in]],
                            texture2d<float> sceneTex [[texture(0)]],
                            sampler smp              [[sampler(0)]])
{
    float4 c = sceneTex.sample(smp, in.uv);
    float g = dot(c.rgb, float3(0.299, 0.587, 0.114));
    return float4(g, g, g, c.a);
}
// output is an MTL::RenderPassDescriptor pointing at the destination texture.
MTL::RenderCommandEncoder* enc = cmd->renderCommandEncoder(output);
enc->setRenderPipelineState(grayscalePSO); // fullscreenVS + grayscaleFS
enc->setFragmentTexture(sceneTexture, 0);
enc->setFragmentSamplerState(clampSampler, 0);

// No vertex buffer: three vertices, generated from [[vertex_id]].
enc->drawPrimitives(MTL::PrimitiveTypeTriangle,
                    NS::UInteger(0), NS::UInteger(3));
enc->endEncoding();

Keep the two shaders byte-for-byte equivalent in HLSL and MSL and the effect looks the same on both backends. The engine-level FullscreenPass stays platform-neutral; only the resource binding and draw call differ behind the renderer abstraction we built in module 08.

Recap
  • A fullscreen pass reads the scene texture as an input SRV and writes a filtered result to an output render target — one fragment per pixel.
  • Draw a single oversized triangle with clip verts (-1,1), (3,1), (-1,-3), not a two-triangle quad: no diagonal seam, cleaner and slightly faster.
  • Generate all three positions and UVs from SV_VertexID with float2((id<<1)&2, id&2) — no vertex or index buffers, no input layout.
  • The * float2(2,-2) + float2(-1,1) maps uv to clip space and flips V, because a D3D texture's V axis points down while clip-space Y points up.
  • Issue the pass with TRIANGLELIST topology, a null vertex buffer, a CLAMP sampler, and context->Draw(3, 0); on Metal, drawPrimitives with vertex count 3 and [[vertex_id]].
  • Wrap it in a reusable FullscreenPass so tone mapping, blur and bloom are each just a different pixel shader on the same three-vertex draw.
