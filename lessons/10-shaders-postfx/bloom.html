Bloom: Making Bright Things Glow

Look at a bright light through a real camera, or squint at the sun, and its edges bleed a soft halo into everything nearby. That scatter — in the lens, in your eye — is what tells your brain "this is really bright," brighter than a screen can actually display. Bloom fakes that halo. In this lesson we assemble the flagship post effect of our engine: extract the bright parts of the HDR scene, blur them across a chain of shrinking render targets, then add that glow back on top — all before tone mapping. Get this right and glowing lava will pour light across your voxel world in Module 11.

What you'll learn
  • Why bloom only works on linear HDR color and must run before tone mapping.
  • How a bright pass keeps only the energy above a luminance threshold, with an optional soft knee.
  • How a downsample chain of mip targets reaches a huge blur radius for almost no cost.
  • The modern progressive upsample-and-add combine (a 9-tap tent filter) that yields a smooth, band-free glow.
  • The full C++ orchestration: build the mip chain, loop down then up, and composite the result.

What bloom actually simulates

A monitor tops out at white — value (1, 1, 1). But the HDR scene we built in the tone-mapping lesson stores light with no such ceiling: a lamp filament or a patch of lava can sit at (20, 14, 3), far outside displayable range. Tone mapping later squeezes that back into 0..1, but on the way there we have a brief window where "very bright" is still an honest number. Bloom exploits that window. We copy out the pixels whose light exceeds a threshold, smear them wide, and blend the smear back over the scene. The eye reads the resulting halo as intensity the display could never show directly.

The whole effect is one pipeline with four stages. We threshold the scene, downsample it several times into progressively smaller targets, upsample back up while adding each level to the next, and finally composite the glow onto the original scene right before the tone map. Everything here runs as fullscreen passes — the same "draw one big triangle and sample a texture" technique from the fullscreen-pass lesson — and every target uses the DXGI_FORMAT_R16G16B16A16_FLOAT HDR format we chose for the scene. Nothing is clamped until the very end.

Stage 1 — the bright pass

The first pass reads the HDR scene and throws away everything that is not bright. "Bright" means high luminance — the perceptual weighting of the three channels, using the Rec. 709 coefficients. We subtract the threshold from the luminance and rescale the color to match, which keeps the pixel's hue intact while dimming it by a fixed amount. Anything at or below the threshold falls to black and contributes no glow.

Texture2D    SceneHDR   : register(t0);
SamplerState LinearClamp : register(s0);

cbuffer BloomParams : register(b0)
{
    float  threshold; // luminance above which pixels bloom (HDR, e.g. 1.0)
    float  knee;      // soft-knee width; 0 = a hard cut
    float2 _pad;
};

float Luminance(float3 c)
{
    return dot(c, float3(0.2126, 0.7152, 0.0722)); // Rec. 709 luma
}

float4 PSBrightPass(float2 uv : TEXCOORD0) : SV_TARGET
{
    float3 c = SceneHDR.Sample(LinearClamp, uv).rgb;
    float  l = Luminance(c);

    // Hard threshold: keep only the energy above 'threshold'.
    float contribution = max(l - threshold, 0.0);

    // Optional soft knee: fade bloom in over a small band so a pixel just
    // past the threshold does not pop in abruptly between frames.
    float soft = clamp(l - threshold + knee, 0.0, 2.0 * knee);
    soft = (soft * soft) / (4.0 * knee + 1e-4);
    contribution = max(contribution, soft);

    // Rescale the colour so its new luminance equals 'contribution',
    // preserving hue. 1e-4 guards the divide when the pixel is black.
    c *= contribution / max(l, 1e-4);
    return float4(c, 1.0);
}

The soft knee is worth a moment. With knee = 0 the two soft lines collapse to zero and you get a hard cutoff: a pixel is either in or out. That produces a visible flickering edge as objects drift across the threshold. The quadratic knee ramps the contribution up smoothly across a band of width 2 * knee around the threshold, so the transition is gentle. This is the same soft-threshold curve Unity's post stack and Jorge Jimenez's Call of Duty presentation use.

The bright pass must read HDR

If you feed this shader an already tone-mapped, 0..1 image, almost nothing will be above a threshold of 1.0 and your bloom will be empty — or, if you drop the threshold to compensate, you will bloom the entire picture into a washed-out fog. The bright pass reads the linear HDR scene, before the tone map. That is the whole reason bloom lives in this stage of the frame.

Stage 2 — downsampling into a mip chain

A wide, soft glow needs a wide blur. Blurring at full resolution with a radius of 100 pixels would mean hundreds of taps per pixel — far too slow. The trick is that halving the resolution doubles the effective reach of any small filter, essentially for free. So instead of one enormous blur we build a chain of shrinking targets: mip 0 at half the screen resolution, mip 1 at a quarter, and so on for five or six levels. Each downsample step also blurs, so by the time we reach the smallest target its content has been smeared across the equivalent of a very large radius on the original image.

Each step uses a 13-tap filter — a set of overlapping 2×2 boxes sampled around the destination texel. The taps and weights are arranged so the total is energy-preserving (they sum to exactly 1.0) and so the filter is wide enough to avoid missing detail as the image shrinks. This is the downsample from Jimenez's "Next Generation Post Processing in Call of Duty: Advanced Warfare," and it has become the industry-standard kernel.

Texture2D    SrcTex      : register(t0);
SamplerState LinearClamp : register(s0);

cbuffer DownParams : register(b0)
{
    float2 srcTexelSize; // 1.0 / source resolution
    float2 _pad;
};

float4 PSDownsample(float2 uv : TEXCOORD0) : SV_TARGET
{
    float dx = srcTexelSize.x;
    float dy = srcTexelSize.y;

    // 13 taps: a 4x4 neighbourhood of the source, as overlapping boxes.
    //   a . b . c
    //   . j . k .
    //   d . e . f      e = the centre texel
    //   . l . m .
    //   g . h . i
    float3 a = SrcTex.Sample(LinearClamp, uv + float2(-2*dx,  2*dy)).rgb;
    float3 b = SrcTex.Sample(LinearClamp, uv + float2(  0.0,  2*dy)).rgb;
    float3 c = SrcTex.Sample(LinearClamp, uv + float2( 2*dx,  2*dy)).rgb;

    float3 d = SrcTex.Sample(LinearClamp, uv + float2(-2*dx,  0.0)).rgb;
    float3 e = SrcTex.Sample(LinearClamp, uv                      ).rgb;
    float3 f = SrcTex.Sample(LinearClamp, uv + float2( 2*dx,  0.0)).rgb;

    float3 g = SrcTex.Sample(LinearClamp, uv + float2(-2*dx, -2*dy)).rgb;
    float3 h = SrcTex.Sample(LinearClamp, uv + float2(  0.0, -2*dy)).rgb;
    float3 i = SrcTex.Sample(LinearClamp, uv + float2( 2*dx, -2*dy)).rgb;

    float3 j = SrcTex.Sample(LinearClamp, uv + float2(-dx,  dy)).rgb;
    float3 k = SrcTex.Sample(LinearClamp, uv + float2( dx,  dy)).rgb;
    float3 l = SrcTex.Sample(LinearClamp, uv + float2(-dx, -dy)).rgb;
    float3 m = SrcTex.Sample(LinearClamp, uv + float2( dx, -dy)).rgb;

    // Weights sum to 1.0:  0.125 + 4*0.03125 + 4*0.0625 + 4*0.125 = 1.0
    float3 colour  = e * 0.125;
    colour += (a + c + g + i) * 0.03125;
    colour += (b + d + f + h) * 0.0625;
    colour += (j + k + l + m) * 0.125;
    return float4(colour, 1.0);
}

The srcTexelSize constant is 1.0 / sourceResolution — the texel size of the texture we are reading, not the one we are writing. We update it before each downsample step because every level of the chain is a different size. A linear-filtering, clamp-addressed sampler is exactly what you want here: clamp keeps edge taps from wrapping garbage in from the far side of the image.

Kill fireflies on the first downsample

A single pixel at value 200 next to neighbours at 2 will dominate any average and flicker wildly as the camera moves — a "firefly." For the first downsample only (bright pass into mip 0, or mip 0 into mip 1), weight each tap by 1 / (1 + luma) — the "Karis average." It computes a luminance-weighted mean that tames outliers before they enter the chain, at the cost of one extra dot product per tap. The remaining levels use the plain weights above.

Stage 3 — the upsample-and-add combine

Now we walk back up the chain. Starting at the smallest mip, we upsample it and add it onto the next-larger mip — which still holds its own downsampled content from stage 2. Then we take that combined level, upsample, and add it onto the one above, and so on until we are back at mip 0. Because each level contributes its own blur radius and they accumulate, the final image is a sum of many blur scales at once. That superposition is what makes the modern glow so smooth and free of the concentric "banding" rings that a single fixed blur produces.

The upsample uses a small 9-tap tent filter (a 3×3 kernel weighted [1 2 1; 2 4 2; 1 2 1] / 16). Its samples are spread by a small radius in UV space, so the spread of the glow stays constant on screen regardless of each mip's pixel size.

Texture2D    SrcTex      : register(t0);
SamplerState LinearClamp : register(s0);

cbuffer UpParams : register(b0)
{
    float  filterRadius; // small UV-space offset, e.g. 0.005
    float3 _pad;
};

float4 PSUpsample(float2 uv : TEXCOORD0) : SV_TARGET
{
    float x = filterRadius;
    float y = filterRadius;

    //   a b c
    //   d e f      e = the centre texel
    //   g h i
    float3 a = SrcTex.Sample(LinearClamp, uv + float2(-x,  y)).rgb;
    float3 b = SrcTex.Sample(LinearClamp, uv + float2( 0,  y)).rgb;
    float3 c = SrcTex.Sample(LinearClamp, uv + float2( x,  y)).rgb;

    float3 d = SrcTex.Sample(LinearClamp, uv + float2(-x,  0)).rgb;
    float3 e = SrcTex.Sample(LinearClamp, uv                 ).rgb;
    float3 f = SrcTex.Sample(LinearClamp, uv + float2( x,  0)).rgb;

    float3 g = SrcTex.Sample(LinearClamp, uv + float2(-x, -y)).rgb;
    float3 h = SrcTex.Sample(LinearClamp, uv + float2( 0, -y)).rgb;
    float3 i = SrcTex.Sample(LinearClamp, uv + float2( x, -y)).rgb;

    // 3x3 tent filter, weights sum to 16.
    float3 colour  = e * 4.0;
    colour += (b + d + f + h) * 2.0;
    colour += (a + c + g + i) * 1.0;
    colour *= 1.0 / 16.0;
    return float4(colour, 1.0);
}

Notice this shader does not read the destination and add — the addition happens through the blend hardware. We render each upsample pass with an additive blend state (source and destination factors both ONE), so the tent-filtered result is summed onto whatever is already in the target. That is the whole "add" in "upsample-and-add."

One chain, traversed twice

There is no second set of "upsample" textures. The down pass and the up pass reuse the same mip targets. During the up loop, when we additively render into mip i-1, that target still contains the blur it received on the way down — and we blend the upsampled mip i on top. After the loop finishes, mip 0 holds the complete, multi-scale glow. That is why the diagram shows a single stack with a down arrow and an up arrow, not two columns.

If you prefer, you can swap this combine for the separable Gaussian from the previous lesson: blur each mip horizontally then vertically before adding it up the chain. It is a few more passes and needs a ping-pong buffer per level, but it is a valid, slightly softer alternative. The tent upsample is cheaper and, in practice, indistinguishable once composited.

Stage 4 — orchestrating it in C++

The engine side owns the mip chain and drives the two loops. First, at init we allocate the targets. Each is an HDR texture with both render-target and shader-resource bind flags, halving in size down the chain.

struct BloomMip
{
    ComPtr<ID3D11Texture2D>          tex;
    ComPtr<ID3D11RenderTargetView>   rtv;
    ComPtr<ID3D11ShaderResourceView> srv;
    UINT width  = 0;
    UINT height = 0;
};

void BloomPass::Init(ID3D11Device* device, UINT screenW, UINT screenH, int mipCount)
{
    mips_.clear();
    UINT w = screenW / 2;   // bloom lives at half resolution — plenty for a glow
    UINT h = screenH / 2;

    for (int i = 0; i < mipCount; ++i)
    {
        BloomMip mip;
        mip.width  = (w > 1) ? w : 1;
        mip.height = (h > 1) ? h : 1;

        D3D11_TEXTURE2D_DESC td = {};
        td.Width            = mip.width;
        td.Height           = mip.height;
        td.MipLevels        = 1;
        td.ArraySize        = 1;
        td.Format           = DXGI_FORMAT_R16G16B16A16_FLOAT; // HDR, like the scene
        td.SampleDesc.Count = 1;
        td.Usage            = D3D11_USAGE_DEFAULT;
        td.BindFlags        = D3D11_BIND_RENDER_TARGET | D3D11_BIND_SHADER_RESOURCE;
        device->CreateTexture2D(&td, nullptr, &mip.tex);

        device->CreateRenderTargetView  (mip.tex.Get(), nullptr, &mip.rtv);
        device->CreateShaderResourceView(mip.tex.Get(), nullptr, &mip.srv);

        mips_.push_back(mip);
        w = (w > 1) ? w / 2 : 1;
        h = (h > 1) ? h / 2 : 1;
    }
}

The additive blend state we need for the up loop is created once at startup:

// dst = src * ONE + dst * ONE   (the "add" in upsample-and-add)
D3D11_BLEND_DESC bd = {};
bd.RenderTarget[0].BlendEnable    = TRUE;
bd.RenderTarget[0].SrcBlend       = D3D11_BLEND_ONE;
bd.RenderTarget[0].DestBlend      = D3D11_BLEND_ONE;
bd.RenderTarget[0].BlendOp        = D3D11_BLEND_OP_ADD;
bd.RenderTarget[0].SrcBlendAlpha  = D3D11_BLEND_ONE;
bd.RenderTarget[0].DestBlendAlpha = D3D11_BLEND_ONE;
bd.RenderTarget[0].BlendOpAlpha   = D3D11_BLEND_OP_ADD;
bd.RenderTarget[0].RenderTargetWriteMask = D3D11_COLOR_WRITE_ENABLE_ALL;
device->CreateBlendState(&bd, &additiveBlend_);

The render method is the heart of the effect. It threshold-copies the scene into mip 0, walks down the chain with opaque blending, then walks back up with additive blending. Because a texture can never be bound as a shader input and a render target at the same time, we release slot 0 after every pass with a tiny helper.

static void UnbindSRV(ID3D11DeviceContext* ctx)
{
    ID3D11ShaderResourceView* nullSRV = nullptr;
    ctx->PSSetShaderResources(0, 1, &nullSRV);
}

void BloomPass::Render(ID3D11DeviceContext* ctx, ID3D11ShaderResourceView* hdrScene)
{
    ctx->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);
    ctx->VSSetShader(fullscreenVS_.Get(), nullptr, 0);
    ctx->PSSetSamplers(0, 1, linearClamp_.GetAddressOf());

    // --- 1. Bright pass: threshold the HDR scene into the largest mip ---
    ctx->OMSetBlendState(opaqueBlend_.Get(), nullptr, 0xffffffff);
    SetViewport(ctx, mips_[0].width, mips_[0].height);
    ctx->OMSetRenderTargets(1, mips_[0].rtv.GetAddressOf(), nullptr);
    ctx->PSSetShader(brightPassPS_.Get(), nullptr, 0);
    ctx->PSSetShaderResources(0, 1, &hdrScene);
    DrawFullscreen(ctx);
    UnbindSRV(ctx);

    // --- 2. Downsample: mip0 -> mip1 -> ... -> mip(N-1) ---
    ctx->PSSetShader(downsamplePS_.Get(), nullptr, 0);
    for (size_t i = 1; i < mips_.size(); ++i)
    {
        const BloomMip& src = mips_[i - 1];
        const BloomMip& dst = mips_[i];

        UpdateTexelSize(ctx, src.width, src.height); // cb0: 1 / src resolution
        SetViewport(ctx, dst.width, dst.height);
        ctx->OMSetRenderTargets(1, dst.rtv.GetAddressOf(), nullptr);
        ctx->PSSetShaderResources(0, 1, src.srv.GetAddressOf());
        DrawFullscreen(ctx);
        UnbindSRV(ctx);
    }

    // --- 3. Upsample + add: mip(N-1) -> ... -> mip0, additively ---
    ctx->OMSetBlendState(additiveBlend_.Get(), nullptr, 0xffffffff);
    ctx->PSSetShader(upsamplePS_.Get(), nullptr, 0);
    for (size_t i = mips_.size() - 1; i > 0; --i)
    {
        const BloomMip& src = mips_[i];       // smaller, already blurred
        const BloomMip& dst = mips_[i - 1];   // larger, keeps its own content

        UpdateFilterRadius(ctx, filterRadius_); // cb0: UV-space tent radius
        SetViewport(ctx, dst.width, dst.height);
        ctx->OMSetRenderTargets(1, dst.rtv.GetAddressOf(), nullptr);
        ctx->PSSetShaderResources(0, 1, src.srv.GetAddressOf());
        DrawFullscreen(ctx);                 // blend ADDs onto dst
        UnbindSRV(ctx);
    }

    ctx->OMSetBlendState(opaqueBlend_.Get(), nullptr, 0xffffffff);
}
// The finished glow now lives in mips_[0].srv.

After Render returns, mips_[0].srv is the bloom texture. Note that no ping-pong buffer is needed anywhere: the down loop always reads mip i-1 and writes a different mip i, and the up loop reads mip i and writes a different mip i-1. The mip chain is the scratch storage. (The separable-Gaussian variant is the one that needs a same-size ping-pong pair per level.)

Stage 5 — compositing before the tone map

The last step folds the glow back into the frame. We sample the original HDR scene and the half-res bloom texture (a linear sampler upscales it smoothly), add them with a small bloomStrength weight, and only then run the tone map from the HDR lesson. Adding in linear HDR is what makes the glow bleed into shadows physically instead of looking like a flat overlay.

Texture2D    SceneHDR   : register(t0);
Texture2D    BloomTex   : register(t1);
SamplerState LinearClamp : register(s0);

cbuffer Composite : register(b0)
{
    float  bloomStrength; // ~0.04 - 0.1
    float3 _pad;
};

float3 ACESFilm(float3 x)  // the operator from the HDR lesson
{
    const float a = 2.51, b = 0.03, c = 2.43, d = 0.59, e = 0.14;
    return saturate((x * (a * x + b)) / (x * (c * x + d) + e));
}

float4 PSComposite(float2 uv : TEXCOORD0) : SV_TARGET
{
    float3 hdr   = SceneHDR.Sample(LinearClamp, uv).rgb;
    float3 bloom = BloomTex.Sample(LinearClamp, uv).rgb;

    hdr += bloomStrength * bloom;   // add the glow in LINEAR HDR

    float3 mapped = ACESFilm(hdr);  // now squeeze into displayable range
    // ... apply the sRGB / gamma encode here, as in the HDR lesson ...
    return float4(mapped, 1.0);
}
Do bloom in linear, composite before tone mapping

Every stage here operates on linear HDR values, and the glow is added before ACESFilm runs. That ordering is the difference between light that looks like it belongs in the scene and a fake white smudge on top of your image. If you bloom after the tone map, the values are already clamped to 0..1, the halo double-brightens midtones, and the whole thing reads as a cheap filter.

Tuning the look

Four knobs shape the final effect. Set them per scene — a moody night level wants a tight, subtle glow; a neon arcade wants it loud.

ParameterTypical valueVisual effect
threshold1.0 (HDR)Which pixels glow. Higher = only the very brightest bloom; too low blooms the whole image into fog.
knee0.5Softness of the threshold. Fades bloom in over a band so objects near the cutoff don't flicker.
mip count5–6Blur reach. Each extra level roughly doubles the glow radius; more levels = wider, softer halo.
bloomStrength0.04–0.1How visible the glow is at composite. Turn up for stylized neon, down for grounded realism.
filterRadius0.005 (UV)Spread of the tent upsample. Larger = softer and more spread; too large starts to look smeary.
Two mistakes that ruin bloom

A threshold that is too low blooms ordinary midtones, not just highlights — the entire image hazes over and loses contrast. Keep the threshold at or above your "white" reference and let genuinely bright things poke through. Never bloom after the tone map. Once color is clamped to 0..1 there is no HDR energy left to extract; you will bloom everything equally and get a washed-out overlay. Bloom belongs strictly in linear HDR, before the tone curve.

macOS — the same pipeline in Metal

The structure ports directly. Allocate the mip chain as MTL::Texture objects with MTL::PixelFormatRGBA16Float and usage ShaderRead | RenderTarget. Each pass is its own MTLRenderCommandEncoder with a single color attachment set to the destination mip; the down passes use storeAction = Store and no blending, while the up passes enable additive blending on the attachment:

// Additive blend for the upsample passes, set on the pipeline descriptor.
auto* att = pipelineDesc->colorAttachments()->object(0);
att->setPixelFormat(MTL::PixelFormatRGBA16Float);
att->setBlendingEnabled(true);
att->setSourceRGBBlendFactor(MTL::BlendFactorOne);
att->setDestinationRGBBlendFactor(MTL::BlendFactorOne);
att->setRgbBlendOperation(MTL::BlendOperationAdd);

The three fragment functions are line-for-line equivalents of the HLSL: an HLSL Texture2D/SamplerState pair becomes an MSL texture2d<float>/sampler, Sample becomes .sample, and the threshold, 13-tap and tent math are identical. If you author in GLSL for a Vulkan backend, the same kernels translate with texture() lookups and a vec3 accumulator. All of the orchestration — mip sizes, loop order, parameters — stays in shared engine C++; only resource creation and encoder setup live in each backend.

That is a complete, production-shaped bloom. In Module 11 this is exactly what will make lava, glowstone and glowing ore read as light sources: give those materials emissive color above 1.0 so they land in the HDR target, and this pipeline turns them into pools of glow across the voxel world. Recap
  • Bloom fakes light scatter around very bright areas; it runs on linear HDR color and must be composited before tone mapping.
  • The bright pass keeps only luminance above a threshold, rescaling color to preserve hue, with an optional soft knee to avoid flicker.
  • A downsample chain of 5–6 half-then-quarter-res targets, each blurred with a 13-tap energy-preserving filter, reaches a huge blur radius cheaply.
  • The upsample-and-add pass walks back up the chain, additively blending a 9-tap tent filter so many blur scales sum into a smooth, band-free glow.
  • The C++ orchestration allocates one mip chain, loops down (opaque) then up (additive) with no ping-pong, and leaves the finished bloom in mip 0.
  • Composite with scene + bloomStrength * bloom, then tone map. Keep the threshold high enough that only highlights glow, and never bloom after the tone curve.
