Anatomy of a C++ Project

You can already build, run and step through a program in the debugger. Now we'll open the hood: the CMake project, its build targets, source and header files, generated build directories, include paths, and linked libraries. This structure is identical in CLion on Windows and macOS.

What you'll learn
  • The difference between a CMake project and a build target
  • Why we split code into source (.cpp) and header (.h) files
  • Where CLion and CMake put Debug and Release build outputs
  • How target_include_directories and target_link_libraries feed the build

The project and its targets

When you created your first program, CMake defined a project and one executable target. These concepts do different jobs.

ConceptWhat it isExample
project(...)Names the overall CMake project and its languages.project(MiniEngine LANGUAGES CXX)
TargetOne output built from a source list.An executable, static library, or test program.

A tiny app has one project and one target, so the distinction is easy to miss. Our engine will have several targets: an Engine static library, a Game executable, and perhaps a Tests executable. One CMake project, several named recipes.

CMakeLists.txt is portable text

CMakeLists.txt belongs in version control and describes the build for every supported operating system. CLion's generated cmake-build-* directories do not belong in version control.

Source files vs header files

Inside a project your C++ code lives in two kinds of files. This split feels odd at first, but it is one of the most important ideas in C++.

  • Header files (.h or .hpp) contain declarations — the promises. "There is a function called Add that takes two ints and returns an int." They describe what exists.
  • Source files (.cpp) contain definitions — the actual code. They describe how it works.

When another file wants to use your function, it #includes the header to learn the promise, then the linker connects that promise to the real code later. Let's see it concretely with three files that will feel like a mini version of an engine component.

// Header: declarations only. This is the "promise".
#pragma once  // ensures this header is included at most once per .cpp

// We promise a function named Add exists somewhere.
int Add(int a, int b);

// Source: the definition. This is the "how".
#include "MathUtils.h"  // pull in our own promise so it must match

int Add(int a, int b)
{
    return a + b;  // the real implementation lives here
}

#include <iostream>
#include "MathUtils.h"  // learn about Add without seeing its body

int main()
{
    int sum = Add(2, 3);
    std::cout << "2 + 3 = " << sum << '\n';
    return 0;
}

Notice that main.cpp never sees the body of Add — it only sees the header. Each .cpp becomes an object file, and the linker combines them into one executable. Add all source files to the target so CMake knows they belong together:

add_executable(Game
    main.cpp
    MathUtils.cpp
    MathUtils.h
)
Always start headers with #pragma once

It stops the same declarations from being pulled in twice, which is a classic source of confusing "already defined" errors. Every header we write for the engine will begin with this line.

Where build output lands

CLion keeps generated build files outside your source tree in one directory per CMake profile:

  • cmake-build-debug — debug symbols, minimal optimization, easy stepping.
  • cmake-build-release — optimized output for performance testing and shipping.

A portable CMake layout looks like this:

MiniEngine/
  CMakeLists.txt           <- portable build recipe
  src/
    main.cpp
    MathUtils.cpp
  include/
    MathUtils.h
  cmake-build-debug/       <- generated; do not commit
    Game                   <- macOS executable
    Game.exe               <- Windows executable name
    CMakeFiles/            <- object files and metadata
  cmake-build-release/     <- separate optimized build

The same source produces different binaries per profile and operating system. If stepping behaves strangely, confirm the Debug profile is active. Add cmake-build-*/ and .idea/ to .gitignore; both are generated locally.

Do not copy build folders between systems

A build directory caches absolute paths, compiler details, and an architecture. Delete or reset it after moving a project between Windows and macOS; CMake will generate the correct native files again.

Include directories and linker inputs

So far our headers sat right next to our source, so #include "MathUtils.h" just worked. But libraries like DirectX live elsewhere on your system. Two project settings tell the build where to look, and they map cleanly onto the two stages of the pipeline.

  • target_include_directories adds folders the compiler searches for headers.
  • target_link_libraries lists libraries or framework targets the linker must combine with your code.

Our renderer chooses libraries by platform. Windows links Direct3D; macOS links Apple frameworks. The shared target sees only the backend interface:

target_include_directories(Engine PUBLIC include)

if (WIN32)
    target_sources(Engine PRIVATE src/graphics/D3D11Renderer.cpp)
    target_link_libraries(Engine PRIVATE d3d11 dxgi d3dcompiler)
elseif (APPLE)
    target_sources(Engine PRIVATE src/graphics/MetalRenderer.cpp)
    target_link_libraries(Engine PRIVATE
        "-framework Foundation"
        "-framework Metal"
        "-framework QuartzCore")
endif()
Undefined reference means a missing source or library

If compilation succeeds but linking fails, check that the implementation file appears in target_sources and its external dependency appears in target_link_libraries.

Putting it together

One CMake project gathers targets. Each target compiles source files, uses headers found through include paths, and links the resulting object files with libraries into a native output. CLion reads that same model on Windows and macOS.

Recap
  • A CMake project groups one or more executable, library, and test targets.
  • Headers (.h) declare what exists; source files (.cpp) define how it works — start every header with #pragma once.
  • Generated output belongs in cmake-build-debug or cmake-build-release, never in version control.
  • target_include_directories feeds the compiler; target_link_libraries feeds the linker.
  • CMake's WIN32 and APPLE conditions select the correct native backend.
