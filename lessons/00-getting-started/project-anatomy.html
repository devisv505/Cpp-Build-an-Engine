Anatomy of a C++ Project

You can already build, run and step through a program in the debugger. But what actually is the thing you pressed play on? In this lesson we open the hood: solutions, projects, source and header files, the folders your .exe lands in, and the include and linker settings that will eventually let us pull DirectX into our engine. Understanding this layout now means you will never feel lost when a project grows to hundreds of files.

What you'll learn
  • The difference between a solution (.sln) and a project (.vcxproj)
  • Why we split code into source (.cpp) and header (.h) files
  • Where the compiler puts your .exe — the Debug/Release x64 output folders
  • What include directories and linker inputs are, and how they map to the compile-and-link pipeline

The solution and the project

When you created your first program, Visual Studio actually made two different containers. It is easy to blur them together, but they do very different jobs.

FileWhat it isThink of it as
.slnA solution — a workspace that groups one or more projects together.A folder of related work.
.vcxprojA project — one build target, such as a single .exe or a library.A recipe that produces one output.

A tiny app has one solution containing one project, so people often say "project" and "solution" interchangeably. But the distinction matters as soon as things grow. Our engine will eventually be a solution with several projects: an Engine project (a reusable library of components), a Game project (the actual .exe that uses the engine), and maybe a Tests project. One solution, many recipes — just like Unity separates its engine runtime from your game code.

Both files are just text

The .sln and .vcxproj files are XML text describing settings, not compiled code. This is exactly why they belong in version control — but the build output folders do not.

Source files vs header files

Inside a project your C++ code lives in two kinds of files. This split feels odd at first, but it is one of the most important ideas in C++.

  • Header files (.h or .hpp) contain declarations — the promises. "There is a function called Add that takes two ints and returns an int." They describe what exists.
  • Source files (.cpp) contain definitions — the actual code. They describe how it works.

When another file wants to use your function, it #includes the header to learn the promise, then the linker connects that promise to the real code later. Let's see it concretely with three files that will feel like a mini version of an engine component.

// Header: declarations only. This is the "promise".
#pragma once  // ensures this header is included at most once per .cpp

// We promise a function named Add exists somewhere.
int Add(int a, int b);

// Source: the definition. This is the "how".
#include "MathUtils.h"  // pull in our own promise so it must match

int Add(int a, int b)
{
    return a + b;  // the real implementation lives here
}

#include <iostream>
#include "MathUtils.h"  // learn about Add without seeing its body

int main()
{
    int sum = Add(2, 3);
    std::cout << "2 + 3 = " << sum << '\n';
    return 0;
}

Notice that main.cpp never sees the body of Add — it only sees the header. Each .cpp is compiled on its own into an object file (.obj), and then the linker stitches them into one .exe. That is the compile then link pipeline from the previous lesson, now with a reason to exist: split code compiles faster and stays organized.

Always start headers with #pragma once

It stops the same declarations from being pulled in twice, which is a classic source of confusing "already defined" errors. Every header we write for the engine will begin with this line.

Where your .exe actually lands

When you build, the compiler does not scatter files next to your source. It writes them into output folders named after your configuration and platform. Two settings drive this:

  • Configuration — Debug (extra checks, no optimization, easy to step through) or Release (optimized, fast, what you ship).
  • Platform — x64 (64-bit, what we always use) or the old x86.

Combine them and you get a folder path. A typical Visual Studio layout looks like this:

MyEngine\                 <- solution folder
  MyEngine.sln
  Game\                    <- one project
    Game.vcxproj
    main.cpp
    MathUtils.h
    MathUtils.cpp
  x64\
    Debug\
      Game.exe             <- your program lands here
      Game.pdb             <- debug symbols for breakpoints
    Release\
      Game.exe             <- optimized build lands here
  Game\x64\Debug\
      main.obj             <- intermediate object files
      MathUtils.obj

So the same code produces different executables depending on the configuration you selected in the toolbar. If you set a breakpoint and nothing stops, the number-one culprit is being in Release instead of Debug. And because these folders are regenerated on every build, you never commit them to git — a .gitignore excludes x64/, Debug/ and Release/.

Match your platform to your libraries

A 64-bit x64 program can only link against 64-bit libraries. Later, when we add DirectX, mixing an x86 build with 64-bit libs produces cryptic linker errors. Keep everything on x64 and you sidestep the whole class of problem.

Include directories and linker inputs

So far our headers sat right next to our source, so #include "MathUtils.h" just worked. But libraries like DirectX live elsewhere on your system. Two project settings tell the build where to look, and they map cleanly onto the two stages of the pipeline.

  • Include Directories — extra folders the compiler searches when it hits an #include <...>. This is how it finds a header it did not write itself.
  • Additional Dependencies (linker inputs) — the list of .lib files the linker pulls in to resolve promises whose bodies live in a prebuilt library.

DirectX 11 ships with the Windows SDK, so its headers and libraries are already on your machine — you just have to name them. When we reach the graphics lessons, wiring up Direct3D is literally these two settings. In modern C++ you can even declare the linker input right in your source with a pragma, which keeps the dependency next to the code that needs it:

#include <d3d11.h>  // Direct3D 11 header, found via include directories

// Tell the linker to pull in these libraries.
// Equivalent to adding them under Linker > Input > Additional Dependencies.
#pragma comment(lib, "d3d11.lib")   // core Direct3D 11
#pragma comment(lib, "dxgi.lib")    // DXGI, for swap chains

// A pointer to the D3D11 device we will create in a later lesson.
ID3D11Device* g_device = nullptr;   // just a declaration for now

Do not worry about what ID3D11Device does yet — the point is that <d3d11.h> is found because of an include directory, and d3d11.lib is linked because it is a linker input. Same two settings you would use for any library, DirectX is nothing special here.

"Unresolved external symbol" means a missing linker input

If the compiler is happy but the build fails at the link step with unresolved external symbol, you almost always forgot to add the matching .lib. The header gave the promise; without the library the linker cannot find the body. Remember which stage each setting serves and these errors stop being scary.

Putting it together

Zoom out and the whole layout is one story: a solution gathers projects; each project compiles its source files (guided by headers and include directories) into object files, then links them together with any library inputs to produce an .exe in a configuration-specific output folder. Every piece you have touched — build, run, debug — sits somewhere on that pipeline. From here on, when a project grows, you will know exactly what each file and setting is for.

Recap
  • A solution (.sln) groups one or more projects (.vcxproj); our engine will use several projects in one solution.
  • Headers (.h) declare what exists; source files (.cpp) define how it works — start every header with #pragma once.
  • Your .exe lands in a configuration/platform output folder like x64\Debug\; these folders are regenerated and kept out of version control.
  • Include directories feed the compiler (finding headers); linker inputs (.lib files) feed the linker — this is exactly how we will attach d3d11.lib for DirectX.
  • An "unresolved external symbol" error is a link-stage problem, usually a missing library input.
