Installing CLion & the C++ Tools

In the last lesson you learned what a compiler and linker do. Now we'll install one consistent development environment for the whole course: CLion. CLion manages the editor, CMake projects, builds, and debugging on both Windows and macOS; the compiler underneath changes with the operating system.

What you'll learn
  • What CLion, CMake, a compiler, and a debugger each contribute
  • How to install CLion on Windows or macOS
  • How to configure bundled MinGW on Windows or Apple Clang on macOS
  • How to verify the toolchain and repair common setup errors

One IDE, two native toolchains

CLion is the interface you use everywhere, while each operating system supplies the native platform layer. CMake describes the targets and libraries in text files, so the same project can choose the correct implementation at configure time.

Step 1 — Install CLion

Download CLion from the official CLion page. JetBrains recommends the Toolbox App because it installs updates and lets you roll back versions. The standalone installer works too.

  • Windows: run the downloaded installer and keep the default components.
  • macOS: choose the Apple silicon download on an M-series Mac or the Intel download on an Intel Mac, open the .dmg, and drag CLion to Applications.

Launch CLion, accept the privacy policy, choose the non-commercial license for this course, and reach the Welcome screen.

Windows — use CLion's bundled MinGW

Current CLion installations bundle a complete MinGW-w64 environment on Windows. It includes GCC, GDB, CMake, Ninja, Windows headers, and the import libraries needed by our Windows targets, so no separate IDE or compiler suite is required.

Check the Windows toolchain

Open Settings → Build, Execution, Deployment → Toolchains. Select or add a MinGW toolchain and choose the bundled environment. CLion should show green checks for:

ToolExpected Windows choice
C compilerBundled gcc.exe
C++ compilerBundled g++.exe
DebuggerBundled GDB
CMakeBundled
Build toolBundled Ninja

Click Apply. Later, CMake will link the Windows graphics libraries by name:

if (WIN32)
    target_link_libraries(MiniEngine PRIVATE d3d11 dxgi d3dcompiler)
endif()

This is the portable CMake replacement for IDE-specific linker settings. The dependency stays in source control and is applied automatically on every Windows machine.

macOS — install Apple Clang

CLion supplies the IDE, CMake, Ninja, and LLDB, but Apple supplies the compiler and macOS SDK. Open Terminal and install the Command Line Tools:

xcode-select --install

Accept the system dialog and wait for installation. If Terminal says the tools are already installed, continue. Full Xcode also contains this toolchain, so you do not need both packages.

Verify Apple Clang

xcode-select -p
xcrun --find clang++
clang++ --version

The commands should print an active developer directory, a compiler path, and a line beginning with Apple clang version.

Check the macOS toolchain in CLion

Open Settings → Build, Execution, Deployment → Toolchains and select the default System toolchain. It should show:

ToolExpected macOS choice
C compilerApple Clang (/usr/bin/clang)
C++ compilerApple Clang++ (/usr/bin/clang++)
DebuggerBundled LLDB
CMakeBundled
Build toolBundled Ninja

Click Apply once every entry has a green check.

Step 2 — verify everything with a CMake project

From the CLion Welcome screen, select New Project → C++ Executable. Name it ToolchainCheck, select C++20, and click Create. Replace main.cpp with:

#include <iostream>

int main()
{
    std::cout << "Toolchain OK\n";
    return 0;
}

Wait for CMake to finish loading, then click the green Run triangle beside main or in the top toolbar. Seeing the following output proves the editor, CMake, compiler, linker, and executable all agree:

Toolchain OK

Process finished with exit code 0

Possible errors and how to fix them

macOS — fatal error: 'iostream' file not found

Clang is present, but the selected developer directory points at an incomplete or outdated header set. If full Xcode is installed in Applications, select its complete toolchain:

sudo xcode-select --switch /Applications/Xcode.app/Contents/Developer
xcode-select -p
xcrun clang++ --version

Restart CLion and select Tools → CMake → Reset Cache and Reload Project. If you do not have full Xcode, install any Command Line Tools update under System Settings → General → Software Update. If no repair is offered, install Xcode and use the switch command above.

The project builds, but headers and standard-library names stay red

If the executable runs with exit code 0, but <iostream> or std::cout is red and std:: has no completion suggestions, the compiler works but CLion's code model has stale paths.

  1. Select Tools → CMake → Reset Cache and Reload Project.
  2. Wait for CMake loading and indexing to finish.
  3. Recheck the active toolchain and its green checks under Settings.
  4. As a final fallback, select File → Invalidate Caches → Invalidate and Restart.

Windows — compiler or debugger not detected

Return to Toolchains, select the bundled MinGW environment rather than an empty System toolchain, and click Apply. Reset and reload the CMake project afterward so its cached compiler paths are regenerated.

How the two graphics paths fit together

Windows lessons use Direct3D 11; macOS lessons show the equivalent Metal objects and commands. The engine architecture, scene, math, resources, gameplay, and networking remain shared C++. Only the thin window and renderer backends differ.

Recap
  • Use CLion and CMake for every course project on both operating systems.
  • Windows uses CLion's bundled MinGW-w64 compiler and GDB debugger.
  • macOS uses Apple Clang from the Command Line Tools and CLion's bundled LLDB.
  • Green toolchain checks plus Toolchain OK and exit code 0 confirm the full build pipeline.
  • Reset the CMake cache after changing toolchains so editor analysis and builds use the same paths.
