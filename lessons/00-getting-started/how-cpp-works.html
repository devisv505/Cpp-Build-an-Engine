How C++ Works: Compiler, Linker & Binary

In the last lesson we talked about why C++ is the language of game engines. Now let's look at how it actually runs. C++ doesn't execute your source files directly — a small assembly line of tools transforms your human-readable code into a native binary the CPU can run. Understanding that pipeline now will save you hours of confusion later, especially when our engine grows to dozens of files and the build starts talking back to you.

What you'll learn
  • The difference between compiled and interpreted languages, and where C++ sits.
  • The three stages of a C++ build: preprocessing, compilation, and linking.
  • What a translation unit and an object file actually are.
  • Why the linker is where so many beginner errors show up — and how to read them.

Compiled vs. interpreted

Broadly, programming languages reach the CPU in one of two ways. An interpreted language like Python ships your source code and runs it through another program — the interpreter — that reads and executes it line by line, every time. A compiled language like C++ translates your entire program ahead of time into raw machine instructions for a specific CPU and operating system. The result is a standalone executable that the machine runs directly, with no translator sitting in the middle.

That ahead-of-time translation is exactly why C++ is fast enough to render thousands of triangles per frame. There is no interpreter overhead at runtime — by the time your game is running, every instruction has already been turned into native code. The trade-off is that you must build the program before you can run it, and the build itself has a few moving parts.

A useful mental model

Think of the compiler as a translator who converts each chapter of a book separately, and the linker as the editor who binds all the translated chapters into one finished volume. Neither can produce the book alone.

The three stages of a build

When you click Build or Run in CLion, CMake asks the selected toolchain to perform three stages for every .cpp file. Let's walk through them with a tiny program.

#include <iostream>   // pulls in the declaration of std::cout

int add(int a, int b);  // a promise: "this function exists somewhere"

int main()
{
    int sum = add(2, 3);                      // we call it here...
    std::cout << "2 + 3 = " << sum << '\n';
    return 0;                                 // 0 means "success"
}
// The actual body of add() lives in a separate file.
int add(int a, int b)
{
    return a + b;   // ...and it's defined here
}

1. Preprocessing

Before real compilation begins, the preprocessor runs a simple text-substitution pass over your file. It handles every line that starts with #. When it sees #include <iostream>, it literally pastes the entire contents of that header into your file. It also expands macros and strips comments. The output is a single, self-contained blob of code called a translation unit — your .cpp plus everything it included.

One .cpp = one translation unit

Each source file is compiled on its own, knowing nothing about the others. That independence is powerful — it lets big projects compile files in parallel — but it's also why we need a linker to reconnect them at the end.

2. Compilation

The compiler takes one translation unit and translates it into machine code, producing an object file (.obj on Windows, .o elsewhere). This file contains the CPU instructions for that unit, but it is not yet runnable. Crucially, the compiler only needs a declaration — a promise that a function exists — to accept a call to it. In main.cpp above, the line int add(int a, int b); is that promise. The compiler happily emits a call to add without ever seeing its body, leaving a labelled "fill this in later" slot in the object file.

This is also the stage where syntax errors and type errors are caught. If you forget a semicolon or pass a string where an int was expected, the compiler stops here and points at the line.

3. Linking

Now we have two object files — main.o and math.o (some Windows toolchains use .obj) — each with holes where they reference things defined elsewhere. The linker collects every object file and library, matches each unresolved reference to its definition, and stitches them into one native executable. When it finds add in math.o, it connects the call in main.o.

Seeing the stages by hand

CLion and CMake normally drive this pipeline, but it helps to see the raw commands once. Windows uses the bundled MinGW g++; macOS uses clang++. The generic c++ command below means the active C++ compiler on either platform:

# Compile each translation unit without linking.
c++ -std=c++20 -c main.cpp
c++ -std=c++20 -c math.cpp

# Link the object files into one program.
c++ main.o math.o -o program

# macOS: ./program
# Windows: .\program.exe

CMake records the same recipe in a portable form:

cmake_minimum_required(VERSION 3.20)
project(BuildPipeline LANGUAGES CXX)
set(CMAKE_CXX_STANDARD 20)
add_executable(program main.cpp math.cpp)

Reading a linker error

Here's the payoff for understanding all this. Suppose you call add but forget to include math.cpp in the build. The compiler is perfectly happy — it saw the declaration, so main.cpp compiles fine. The linker is the one that fails, because it can never find a definition to fill the slot:

undefined reference to `add(int, int)'
linker command failed with exit code 1
Compiler error vs. linker error

An undefined reference means the code compiled but the linker couldn't find a definition — usually a missing .cpp in add_executable or an unlinked library. A syntax or type diagnostic is a compile error. Knowing which tool complained tells you where to look.

These two failure modes map directly onto the stages you just learned:

StageTurns…Into…Typical error

Compilea translation unitan object file (.o/.obj)syntax or type error
Linkall object files + librariesthe executableundefined reference

Why this matters for the engine

Our engine won't be one giant file — it will be dozens of translation units for transforms, rendering, assets, and platform backends. CMake links d3d11 on Windows and the Metal frameworks on macOS with target_link_libraries. An undefined reference to a graphics symbol means the code compiled but the matching platform library was not linked.

Recap
  • C++ is compiled ahead of time into native machine code, unlike interpreted languages that run source through an interpreter at runtime.
  • A build has three stages: the preprocessor expands #includes and macros, the compiler turns each translation unit into an object file, and the linker fuses object files and libraries into an executable.
  • A translation unit is one .cpp plus everything it includes; each is compiled independently, which is why the linker exists.
  • Compile errors mean a source file could not become an object file; linker errors mean a definition or library could not be found.
