Building, Running & Debugging

In the last lesson you compiled and ran your very first program. Now let's turn the crank properly: understand what actually happens when you press the green arrow, learn the difference between a Debug and a Release build, and pick up the single most important skill for engine work — using the debugger to freeze a running program and inspect it from the inside. Once your engine starts moving objects around in 3D, the debugger is how you'll find out why that object teleported to the wrong place.

What you'll learn
  • What the Debug and Release configurations do, and why we build for the x64 platform.
  • The difference between Start Debugging (F5) and Start Without Debugging (Ctrl + F5).
  • How to set a breakpoint and step through code with F10 and F11.
  • How to inspect variables with the Autos and Watch windows, and read the Call Stack.

Configurations and platforms

Look at the toolbar across the top of Visual Studio. You'll see two dropdowns sitting next to the green run button. The first is the configuration (usually Debug or Release), and the second is the platform (x64 or x86). Together they decide how your code is compiled.

ConfigurationOptimizationsDebug infoUse it for
DebugOffFullWriting and fixing code day to day.
ReleaseOn (fast)MinimalShipping, benchmarking, measuring real performance.

In a Debug build the compiler keeps a faithful, one-to-one map between your source lines and the machine code, and it does not rearrange anything. That is exactly what lets the debugger stop on line 12 and show you the real value of every variable. A Release build turns the optimizer loose: it inlines functions, reorders instructions, and deletes variables it can compute at compile time. Your engine will run several times faster in Release — but stepping through it feels like reading a book with random pages torn out, because the code no longer matches your source line-for-line.

For the platform, always choose x64. That means a 64-bit program, which is the modern default and what DirectX 11 game code targets. The older x86 (32-bit) setting caps your addressable memory at roughly 4 GB and has no upside for us. Set the dropdowns to Debug / x64 now and leave them there for most of this course.

A build config is not a code change

Switching from Debug to Release does not edit a single line of your source. It only changes the compiler's instructions. The same main.cpp produces two different executables.

Two ways to run: F5 vs Ctrl+F5

There are two commands to launch your program, and the difference matters more than beginners expect.

  • Start Debugging — F5. Runs your program attached to the debugger. Breakpoints work, you can pause and inspect, and if the program crashes the debugger catches it and shows you the exact line.
  • Start Without Debugging — Ctrl + F5. Runs your program on its own, with no debugger attached. It's faster to launch and, for console programs, Visual Studio helpfully keeps the window open with a "Press any key to close…" prompt at the end.

Use Ctrl + F5 when you just want to see the output. Use F5 the moment something misbehaves and you want to look inside. There's also Build Solution (Ctrl + Shift + B), which compiles without running — handy for checking that everything still compiles after a big edit.

Breakpoints do nothing under Ctrl+F5

If you set a breakpoint and it never triggers, you probably launched with Ctrl + F5. Breakpoints only work when the debugger is attached, so use F5.

Setting a breakpoint

A breakpoint tells the debugger: "when execution reaches this line, freeze everything so I can look around." You set one by clicking in the grey margin to the left of a line number, or by placing your cursor on the line and pressing F9. A red dot appears. Let's use a small program with a deliberate bug so there's something to investigate.

#include <iostream>
#include <vector>

// Adds up the numbers in a list and returns the average.
double average(const std::vector<int>& values)
{
    int sum = 0;
    for (int i = 0; i <= values.size(); ++i)  // BUG: <= walks one past the end
    {
        sum += values[i];
    }
    return static_cast<double>(sum) / values.size();
}

int main()
{
    std::vector<int> scores { 10, 20, 30, 40 };

    double result = average(scores);          // set a breakpoint on this line

    std::cout << "Average: " << result << '\n';
    return 0;
}

Set a breakpoint on the double result = average(scores); line and press F5. The program builds, launches, and stops on that line before running it. A yellow arrow in the margin marks the next statement to execute. Nothing after that arrow has happened yet — the program is frozen, waiting for you.

Stepping through code

Once you're stopped, you drive the program forward one piece at a time. The two commands you'll use constantly:

  • Step Over — F10. Executes the current line completely and stops on the next one. If the line calls a function, that whole function runs without you having to walk through it.
  • Step Into — F11. If the current line calls a function you wrote, this dives inside it and stops on its first line. Use it to follow a call down into the code.

There's also Step Out (Shift + F11), which runs the rest of the current function and pops you back to whoever called it, and Continue (F5 again), which resumes at full speed until the next breakpoint.

With the arrow on our breakpoint, press F11 to step into average. You land on its first line. Now keep pressing F10 to walk through the loop one iteration at a time and watch what happens to i and sum.

Step Over first, Step Into when curious

Most of the time you want F10 — you trust std::cout and don't need to walk through the standard library. Reach for F11 only when you suspect the bug lives inside a function you wrote.

Inspecting variables: Autos and Watch

While execution is paused, two windows show you the program's live state (both are under Debug › Windows if you don't see them):

  • The Autos window automatically lists the variables used on or near the current line. You don't configure it — it just follows along.
  • The Watch window is yours to fill. Type any variable name or even an expression like values.size() or sum + 1 and it re-evaluates every time you step. You can also hover your mouse over a variable in the editor to see its value in a tooltip.

Step through our loop and watch i. On the last iteration i becomes 4, but scores only has valid indices 0 through 3. The condition i <= values.size() let the loop run one step too far. In the Watch window values.size() reads 4, so values[4] reads memory that doesn't belong to the vector — undefined behavior. The fix is to use < instead of <=:

for (int i = 0; i < values.size(); ++i)  // stop before size(), indices 0..3
{
    sum += values[i];
}

The debugger didn't fix the bug — you did — but it showed you the exact iteration where reality diverged from what you expected. That's the whole game: make the invisible visible.

Reading the Call Stack

The Call Stack window answers the question "how did I get here?" It lists the chain of function calls that led to the current line, most recent on top. When we're stopped inside average, the stack looks like this:

Double-click any frame to jump to that function's code and see its local variables in Autos — without losing your place. This becomes invaluable in engine work, where a single frame update might call Scene::Update, which calls each component's Update, which calls into your math library. When something crashes deep in that chain, the call stack is the trail of breadcrumbs back to the line in your code that started it.

Why this matters for the engine

An engine runs thousands of times per second and you can't std::cout your way through all of it. When a mesh renders black or a transform explodes to NaN, you'll set a breakpoint, freeze one frame, and read the exact values feeding your DirectX draw call. Getting fluent with the debugger now pays off on every lesson that follows.

Recap
  • Debug builds are unoptimized and fully debuggable; Release builds are fast but hard to step through. Build for the x64 platform.
  • F5 runs with the debugger (breakpoints work); Ctrl + F5 runs without it and keeps the console window open.
  • Click the margin or press F9 to set a breakpoint; use F10 to step over and F11 to step into your functions.
  • Read live values in the Autos and Watch windows, and use the Call Stack to trace how execution reached the current line.
