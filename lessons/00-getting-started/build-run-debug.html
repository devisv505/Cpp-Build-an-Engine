Building, Running & Debugging

In the last lesson you compiled and ran your very first program. Now let's turn the crank properly: understand what actually happens when you press the green arrow, learn the difference between a Debug and a Release build, and pick up the single most important skill for engine work — using the debugger to freeze a running program and inspect it from the inside. Once your engine starts moving objects around in 3D, the debugger is how you'll find out why that object teleported to the wrong place.

What you'll learn
  • What CLion's Debug and Release CMake profiles do.
  • The difference between the Run and Debug actions.
  • How to set a breakpoint and use Step Over, Step Into, and Step Out.
  • How to inspect variables, watches, and stack frames.

CMake profiles

CLion stores build settings in CMake profiles. Open Settings → Build, Execution, Deployment → CMake to see the default Debug profile and add a Release profile. The selected profile tells CMake which compiler flags and output directory to use.

ConfigurationOptimizationsDebug infoUse it for
DebugOffFullWriting and fixing code day to day.
ReleaseOn (fast)MinimalShipping, benchmarking, measuring real performance.

In a Debug build the compiler keeps a faithful, one-to-one map between your source lines and the machine code, and it does not rearrange anything. That is exactly what lets the debugger stop on line 12 and show you the real value of every variable. A Release build turns the optimizer loose: it inlines functions, reorders instructions, and deletes variables it can compute at compile time. Your engine will run several times faster in Release — but stepping through it feels like reading a book with random pages torn out, because the code no longer matches your source line-for-line.

The toolchain chooses the native 64-bit architecture automatically: x86_64 on most Windows PCs, arm64 on Apple silicon, or x86_64 on Intel Macs. Keep the Debug profile selected while learning; CMake and CLion will place its files in a directory such as cmake-build-debug.

A build config is not a code change

Switching from Debug to Release does not edit a single line of your source. It only changes the compiler's instructions. The same main.cpp produces two different executables.

Two ways to launch: Run vs Debug

There are two commands to launch your program, and the difference matters more than beginners expect.

  • Debug — click the bug icon. CLion launches the selected CMake target with GDB on Windows or LLDB on macOS. Breakpoints work, you can pause and inspect, and crashes stop on the responsible line.
  • Run — click the green triangle. CLion launches the target normally without attaching a debugger. Use this when you only need the result or console output.

Use Run when you just want output. Use Debug when something misbehaves. Build → Build Project compiles without launching, which is useful after a large edit. CLion displays the current shortcuts beside these menu actions; they can differ between the Windows and macOS keymaps.

Breakpoints need Debug

If a breakpoint never triggers, check that you clicked the bug-shaped Debug action rather than the Run triangle and that the Debug CMake profile is active.

Setting a breakpoint

A breakpoint tells the debugger: "when execution reaches this line, freeze everything so I can look around." Click the gutter beside a line number; a red dot appears. Let's use a small program with a deliberate bug so there's something to investigate.

#include <iostream>
#include <vector>

// Adds up the numbers in a list and returns the average.
double average(const std::vector<int>& values)
{
    int sum = 0;
    for (int i = 0; i <= values.size(); ++i)  // BUG: <= walks one past the end
    {
        sum += values[i];
    }
    return static_cast<double>(sum) / values.size();
}

int main()
{
    std::vector<int> scores { 10, 20, 30, 40 };

    double result = average(scores);          // set a breakpoint on this line

    std::cout << "Average: " << result << '\n';
    return 0;
}

Set a breakpoint on the double result = average(scores); line and click Debug. The program builds, launches, and stops before that line runs. An arrow in the gutter marks the next statement to execute.

Stepping through code

Once you're stopped, you drive the program forward one piece at a time. The two commands you'll use constantly:

  • Step Over executes the current line and stops on the next one. A called function runs without opening its internal lines.
  • Step Into enters a function call and stops on its first line. Use it for functions you wrote.

Step Out finishes the current function and returns to its caller. Resume Program continues at full speed until the next breakpoint. These controls appear in the Debug tool window.

Step into average, then use Step Over to walk through the loop one iteration at a time and watch i and sum.

Step Over first, Step Into when curious

Most of the time you trust library calls and Step Over them. Step Into only when you suspect the bug lives inside a function you wrote.

Inspecting variables and watches

While execution is paused, two windows show you the program's live state (both are under Debug › Windows if you don't see them):

  • The Variables pane lists locals visible in the selected stack frame.
  • The Watches pane evaluates expressions such as values.size() or sum + 1 every time you step. You can also hover over a variable in the editor.

Step through our loop and watch i. On the last iteration i becomes 4, but scores only has valid indices 0 through 3. The condition i <= values.size() ran one step too far. Add values.size() as a watch to see why. The fix is < instead of <=:

for (int i = 0; i < values.size(); ++i)  // stop before size(), indices 0..3
{
    sum += values[i];
}

The debugger didn't fix the bug — you did — but it showed you the exact iteration where reality diverged from what you expected. That's the whole game: make the invisible visible.

Reading stack frames

The Frames pane answers "how did I get here?" It lists the call chain that led to the current line, most recent on top. When stopped inside average, it looks like this:

Select any frame to jump to that function and see its local variables. This becomes invaluable when an engine update passes through scenes, components, and math helpers. The stack is the trail back to the line in your code that started the chain.

Why this matters for the engine

An engine runs thousands of times per second and you can't std::cout your way through all of it. When a mesh renders black or a transform explodes to NaN, you'll set a breakpoint, freeze one frame, and read the exact values feeding your DirectX draw call. Getting fluent with the debugger now pays off on every lesson that follows.

Recap
  • Debug profiles preserve debug information; Release profiles enable optimization.
  • Run launches normally; Debug attaches GDB on Windows or LLDB on macOS so breakpoints work.
  • Click the gutter to set a breakpoint, then use Step Over, Step Into, Step Out, and Resume.
  • Read live state in Variables and Watches, and use Frames to trace how execution reached the current line.
