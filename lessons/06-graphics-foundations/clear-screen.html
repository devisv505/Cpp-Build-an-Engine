Render Targets: Clearing the Screen

Lesson 39 connected the GLFW window to Direct3D 11 on Windows or Metal on macOS. The renderer can now reach the GPU and presentation system, but it has not submitted a frame. In this lesson you will add the same final method on both platforms—RenderFrame()—then clear the window to a solid color and present it from the existing GLFW loop.

The frame has four jobs

The API calls differ, but the frame sequence is the same. Metal acquires a fresh drawable each frame, while D3D11 keeps a render-target view for the current swap-chain back buffer.

Frame jobDirect3D 11Metal
Obtain the color targetUse the existing renderTargetViewCall layer->nextDrawable()
Describe the clearPass a color to ClearRenderTargetViewSet the render-pass attachment's load action and clear color
Submit workCommands go through the immediate contextEnd the encoder and commit a command buffer
Show the frameCall swapChain->PresentPresent the drawable through the command buffer

Windows path — clear the D3D11 back buffer

Direct3D cannot write directly to an untyped texture resource. The renderTargetView created in Lesson 39 tells the output-merger stage to treat the swap chain's back buffer as a writable color target.

1. Replace the D3D11 renderer header

Replace the complete header from Lesson 39 with this version. In particular, replace the inline no-op Resize; do not keep it and add a second function with the same signature.

#pragma once

#include <d3d11.h>
#include <dxgi.h>
#include <wrl/client.h>

struct GLFWwindow;

class Renderer {
public:
    bool Initialize(GLFWwindow* window);
    void Resize(GLFWwindow* window);
    void RenderFrame();
    void Shutdown();

private:
    Microsoft::WRL::ComPtr<ID3D11Device> device;
    Microsoft::WRL::ComPtr<ID3D11DeviceContext> context;
    Microsoft::WRL::ComPtr<IDXGISwapChain> swapChain;
    Microsoft::WRL::ComPtr<ID3D11RenderTargetView> renderTargetView;

    int framebufferWidth = 0;
    int framebufferHeight = 0;
};

2. Resize the swap-chain target when needed

The main loop calls Resize(window) every frame, but this method returns immediately when the pixel dimensions have not changed. When they do change, release the old view before calling ResizeBuffers, then create a new view for the replacement back buffer.

void Renderer::Resize(GLFWwindow* window) {
    int newWidth = 0;
    int newHeight = 0;
    glfwGetFramebufferSize(window, &newWidth, &newHeight);

    // A minimized window can have a zero-sized framebuffer.
    if (newWidth == 0 || newHeight == 0) return;

    // Lesson 39 created its swap chain with width/height 0, which made DXGI
    // match the initial client size. Record that size without rebuilding it.
    if (framebufferWidth == 0 && framebufferHeight == 0) {
        framebufferWidth = newWidth;
        framebufferHeight = newHeight;
        return;
    }

    if (newWidth == framebufferWidth && newHeight == framebufferHeight) return;
    if (!swapChain.Get() || !device.Get() || !context.Get()) return;

    // The pipeline and our ComPtr must release the old back buffer first.
    context->OMSetRenderTargets(0, nullptr, nullptr);
    renderTargetView.Reset();

    HRESULT hr = swapChain->ResizeBuffers(
        0, // keep the existing buffer count
        static_cast<UINT>(newWidth),
        static_cast<UINT>(newHeight),
        DXGI_FORMAT_UNKNOWN, // keep the existing format
        0);
    if (FAILED(hr)) return;

    Microsoft::WRL::ComPtr<ID3D11Texture2D> backBuffer;
    hr = swapChain->GetBuffer(0, IID_PPV_ARGS(backBuffer.GetAddressOf()));
    if (FAILED(hr)) return;

    hr = device->CreateRenderTargetView(
        backBuffer.Get(), nullptr, renderTargetView.GetAddressOf());
    if (FAILED(hr)) return;

    framebufferWidth = newWidth;
    framebufferHeight = newHeight;
}
Release the old view before ResizeBuffers

ResizeBuffers fails when a render-target view or a currently bound pipeline slot still holds the old back buffer. Unbind it from the context, reset the ComPtr, resize, and only then create the replacement view.

3. Bind, clear and present

The viewport maps normalized coordinates to pixels. The clear itself does not need a viewport, but setting it now prepares the pipeline for the triangle in Lesson 41.

void Renderer::RenderFrame() {
    if (!renderTargetView.Get()) return;
    if (framebufferWidth == 0 || framebufferHeight == 0) return;

    // 1. Tell the output-merger where fragment colors should be written.
    ID3D11RenderTargetView* targets[] = { renderTargetView.Get() };
    context->OMSetRenderTargets(1, targets, nullptr);

    // 2. Map normalized coordinates to the complete back buffer.
    D3D11_VIEWPORT viewport{};
    viewport.TopLeftX = 0.0f;
    viewport.TopLeftY = 0.0f;
    viewport.Width = static_cast<float>(framebufferWidth);
    viewport.Height = static_cast<float>(framebufferHeight);
    viewport.MinDepth = 0.0f;
    viewport.MaxDepth = 1.0f;
    context->RSSetViewports(1, &viewport);

    // 3. Replace every pixel in the color target.
    const float clearColor[4] = { 0.10f, 0.18f, 0.30f, 1.0f };
    context->ClearRenderTargetView(renderTargetView.Get(), clearColor);

    // Future draw calls go here.

    // 4. Wait for the next display refresh and show the back buffer.
    HRESULT hr = swapChain->Present(1, 0);
    if (FAILED(hr)) {
        // A production renderer would log or recover from a lost device here.
    }
}

Present(1, 0) uses a sync interval of one, commonly called VSync. It waits for the next display refresh before presenting, which prevents tearing. A sync interval of zero presents immediately but may tear.

macOS path — clear a Metal drawable

Metal does not keep one permanent back-buffer view. The CAMetalLayer from Lesson 39 lends the renderer a drawable for the current frame. Its texture becomes color attachment zero of a render pass.

1. Add RenderFrame to the Metal header

Replace the complete header with this version so the public interface matches the D3D11 renderer:

#pragma once

struct GLFWwindow;
namespace MTL { class Device; class CommandQueue; }
namespace CA { class MetalLayer; }

class Renderer {
public:
    bool Initialize(GLFWwindow* window);
    void Resize(GLFWwindow* window);
    void RenderFrame();
    void Shutdown();

private:
    MTL::Device* device = nullptr;
    MTL::CommandQueue* commandQueue = nullptr;
    CA::MetalLayer* layer = nullptr;
};

2. Encode the clear in MetalRenderer.mm

Add the following method to the same MetalRenderer.mm file used in Lesson 39. That file is already listed in CMake; do not create an unlisted MetalRenderer.cpp.

void Renderer::RenderFrame() {
    // Metal creates autoreleased temporary objects while encoding a frame.
    NS::AutoreleasePool* pool = NS::AutoreleasePool::alloc()->init();

    if (!layer || !commandQueue) {
        pool->release();
        return;
    }

    // 1. Ask the display layer for this frame's presentable texture.
    CA::MetalDrawable* drawable = layer->nextDrawable();
    if (!drawable) { // normal while the window is minimized
        pool->release();
        return;
    }

    // 2. Describe what should happen to color attachment zero.
    MTL::RenderPassDescriptor* pass =
        MTL::RenderPassDescriptor::renderPassDescriptor();
    MTL::RenderPassColorAttachmentDescriptor* color =
        pass->colorAttachments()->object(0);

    color->setTexture(drawable->texture());
    color->setLoadAction(MTL::LoadActionClear);
    color->setStoreAction(MTL::StoreActionStore);
    color->setClearColor(MTL::ClearColor(0.10, 0.18, 0.30, 1.0));

    // 3. Encode an otherwise empty render pass; its load action performs
    //    the clear before any future draw calls.
    MTL::CommandBuffer* commands = commandQueue->commandBuffer();
    if (!commands) {
        pool->release();
        return;
    }

    MTL::RenderCommandEncoder* encoder =
        commands->renderCommandEncoder(pass);
    if (!encoder) {
        pool->release();
        return;
    }

    encoder->endEncoding();

    // 4. Present after the GPU finishes this command buffer.
    commands->presentDrawable(drawable);
    commands->commit();

    pool->release();
}
Why Metal does not set a viewport yet

The render pass's LoadActionClear clears the entire attachment before drawing starts, independent of the viewport. We will set a Metal viewport when Lesson 41 adds geometry. The drawable texture already has the Retina-aware size assigned to CAMetalLayer.drawableSize by Resize(window).

Use the same GLFW loop on both platforms

Return to main.cpp from Lesson 39 and replace its loop with this version. GLFW processes native Windows or Cocoa events internally; there is no separate Win32 loop in the Windows path.

while (!glfwWindowShouldClose(window)) {
    glfwPollEvents();

    // D3D11 recreates its back-buffer view only when needed.
    // Metal keeps CAMetalLayer.drawableSize in framebuffer pixels.
    renderer.Resize(window);

    // This call has the same name on Windows and macOS.
    renderer.RenderFrame();
}

What success looks like

Build and run from CLion. On either Windows or macOS, the whole drawable area should remain a steady dark blue while you move or resize the window. Closing the window should exit with code 0.

Recap
  • Both backends implement Initialize, Resize, RenderFrame and Shutdown.
  • D3D11 binds a persistent render-target view, sets a viewport, clears it and presents the swap chain.
  • Metal acquires a drawable, attaches its texture to a render pass, uses LoadActionClear, then commits and presents it.
  • The D3D11 view must be released and recreated after ResizeBuffers; Metal updates CAMetalLayer.drawableSize.
  • The application keeps one cross-platform GLFW loop and calls the same renderer methods on both systems.
