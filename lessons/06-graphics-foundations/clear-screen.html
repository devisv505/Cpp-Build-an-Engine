Render Targets: Clearing the Screen

In the previous lesson we brought Direct3D 11 to life: a device, a context, and a swap chain all wired up and ready. But the window still shows whatever garbage was in memory. In this lesson we take the first real step toward drawing — we tell the GPU where to draw, how big the drawable area is, and then fill it with a color every single frame. This clear-and-present loop is the beating heart your engine's renderer will be built around.

What you'll learn
  • What a render target view is and why the output-merger stage needs one.
  • How to bind the back buffer with OMSetRenderTargets.
  • How to describe and set the viewport using D3D11_VIEWPORT and RSSetViewports.
  • How to clear the screen with ClearRenderTargetView and show it with Present.
  • How to structure a minimal, reusable per-frame render function.

The output-merger needs a target

The GPU pipeline ends at the output-merger stage (abbreviated OM). This is the stage that decides where finished pixels land. It cannot write pixels into thin air — it needs a concrete destination, and that destination is a render target view (ID3D11RenderTargetView). Think of the render target view as a "writable lens" onto a texture. In our case that texture is the swap chain's back buffer: the off-screen image we draw into before flipping it to the monitor.

You already created this render target view at the end of the init lesson. As a quick refresher, that code grabbed the back buffer texture out of the swap chain and asked the device to make a view of it: 

// Grab the swap chain's back buffer (index 0) as a texture.
Microsoft::WRL::ComPtr<ID3D11Texture2D> backBuffer;
HRESULT hr = m_swapChain->GetBuffer(
    0, __uuidof(ID3D11Texture2D),
    reinterpret_cast<void**>(backBuffer.GetAddressOf()));
if (FAILED(hr)) { /* handle error */ }

// Create a render target view of that back buffer.
hr = m_device->CreateRenderTargetView(
    backBuffer.Get(),
    nullptr,                      // null = default view description
    m_renderTargetView.GetAddressOf());
if (FAILED(hr)) { /* handle error */ }

From here on we assume m_device, m_context, m_swapChain, and m_renderTargetView are stored members of your Renderer class, each held in a Microsoft::WRL::ComPtr smart pointer so their lifetimes are managed for you.

View vs. resource

The back buffer is the resource (the actual pixels in GPU memory). The render target view is a lightweight interpretation of that resource for the pipeline. One resource can have several views — this separation is a recurring theme in D3D11 and we will lean on it heavily later for textures and depth buffers.

Binding the render target

Having a render target view is not enough — we must bind it so the output-merger knows to use it. That is what OMSetRenderTargets does. It takes an array of render target views plus an optional depth-stencil view (we have no depth buffer yet, so we pass nullptr): 

// Bind our single render target. No depth buffer for now.
ID3D11RenderTargetView* rtvs[] = { m_renderTargetView.Get() };
m_context->OMSetRenderTargets(
    1,          // number of render targets
    rtvs,       // the array of RTVs
    nullptr);   // depth-stencil view (added in a later lesson)
Pass a raw pointer array, not a ComPtr array

OMSetRenderTargets wants a ID3D11RenderTargetView* const*. Call .Get() on your ComPtr to hand over the raw pointer. Passing the ComPtr object directly will not compile, and taking its address gives the GPU the wrong thing entirely.

Setting the viewport

The viewport tells the rasterizer how to map the GPU's normalized coordinate space onto actual pixels in the render target. If you skip it, D3D11 uses an empty viewport and nothing shows up — a classic "why is my screen black?" trap. A viewport is described by a plain struct, D3D11_VIEWPORT, which you fill in and hand to RSSetViewports (the RS is the rasterizer stage): 

// Describe the drawable region: the full window client area.
D3D11_VIEWPORT viewport = {};
viewport.TopLeftX = 0.0f;             // left edge, in pixels
viewport.TopLeftY = 0.0f;             // top edge, in pixels
viewport.Width    = static_cast<float>(m_width);   // window width
viewport.Height   = static_cast<float>(m_height);  // window height
viewport.MinDepth = 0.0f;             // near plane of the depth range
viewport.MaxDepth = 1.0f;             // far plane of the depth range

m_context->RSSetViewports(1, &viewport);

Here m_width and m_height are the client-area dimensions you already used when creating the swap chain. Note that Width and Height are floats, not integers — a common source of confused compiler warnings. MinDepth and MaxDepth almost always stay at 0.0 and 1.0; they only matter once we add depth testing for 3D geometry.

TopLeftX / TopLeftYPixel offset of the viewport's top-left corner0, 0

Bind once, or every frame?

For a fixed-size window you can set the render target and viewport once, right after initialization. But binding them at the top of each frame costs almost nothing and saves you from subtle bugs — especially once you add render-to-texture passes that swap targets mid-frame. We'll bind per frame here for clarity.

Clear, then present

Now the fun part. ClearRenderTargetView floods the entire bound target with one RGBA color. The color is a 4-element float array where each channel ranges from 0.0 to 1.0 — so {0, 0, 0, 1} is opaque black and {1, 1, 1, 1} is white. After clearing, Present flips the finished back buffer to the screen. 

void Renderer::RenderFrame()
{
    // 1. Bind the render target for this frame.
    ID3D11RenderTargetView* rtvs[] = { m_renderTargetView.Get() };
    m_context->OMSetRenderTargets(1, rtvs, nullptr);

    // 2. Set the viewport to cover the whole window.
    D3D11_VIEWPORT viewport = {};
    viewport.TopLeftX = 0.0f;
    viewport.TopLeftY = 0.0f;
    viewport.Width    = static_cast<float>(m_width);
    viewport.Height   = static_cast<float>(m_height);
    viewport.MinDepth = 0.0f;
    viewport.MaxDepth = 1.0f;
    m_context->RSSetViewports(1, &viewport);

    // 3. Clear the whole target to a calm engine-blue (RGBA, 0..1).
    const float clearColor[4] = { 0.10f, 0.18f, 0.30f, 1.0f };
    m_context->ClearRenderTargetView(m_renderTargetView.Get(), clearColor);

    // (Later lessons draw triangles and meshes right here.)

    // 4. Flip the back buffer to the screen.
    //    First arg = vsync interval: 1 waits for one refresh (no tearing).
    m_swapChain->Present(1, 0);
}

The two arguments to Present are worth understanding. The first is the sync interval: passing 1 asks the GPU to wait for the next vertical blank before flipping, which caps your frame rate to the monitor's refresh and eliminates tearing. Passing 0 presents immediately (uncapped, but you may see tearing). The second argument is a flags field; 0 means "present normally."

Wiring it into the message loop

RenderFrame does nothing until something calls it repeatedly. In a Win32 app that "something" is your main message loop. We drain any pending window messages, then render a frame, over and over: 

// Main loop: pump messages, then render one frame each iteration.
MSG msg = {};
while (msg.message != WM_QUIT)
{
    // Handle window events without blocking.
    if (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }
    else
    {
        // No messages waiting — draw a frame.
        renderer.RenderFrame();
    }
}

Build and run with Ctrl + F5. If everything is wired correctly, your window is now a solid, steady slab of that engine-blue color. It may not look like much, but you are running a real render loop: clearing and presenting sixty (or more) times a second.

Still black? Check these first

A black or flickering window almost always means one of: you forgot RSSetViewports (empty viewport), the render target view was never created or is nullptr, or you are clearing but never calling Present. Verify each step ran and check the debug output window — with the debug device layer enabled from the init lesson, D3D11 prints a clear explanation when something is wrong.

Where this fits in the engine

This clear-and-present pair is the outer shell of your engine's render pipeline. As we grow the engine, RenderFrame becomes the place where the Renderer walks every object's Transform and Mesh components and issues draw calls — but the bookends stay the same: bind the target and viewport, clear, draw, present. Unity's Camera.clearFlags and Unreal's scene color clear are doing exactly what you just wrote, only with more layers on top. You now own the foundation they are built on.

macOS — clear a Metal drawable

Metal describes the clear operation on a render-pass color attachment, then commits the encoded command buffer:

CA::MetalDrawable* drawable = layer->nextDrawable();
if (drawable == nullptr) return;

MTL::RenderPassDescriptor* pass =
    MTL::RenderPassDescriptor::renderPassDescriptor();
auto* color = pass->colorAttachments()->object(0);
color->setTexture(drawable->texture());
color->setLoadAction(MTL::LoadActionClear);
color->setStoreAction(MTL::StoreActionStore);
color->setClearColor(MTL::ClearColor(0.05, 0.08, 0.12, 1.0));

MTL::CommandBuffer* commands = commandQueue->commandBuffer();
MTL::RenderCommandEncoder* encoder =
    commands->renderCommandEncoder(pass);
encoder->endEncoding();
commands->presentDrawable(drawable);
commands->commit();

A Direct3D render-target view and a Metal color attachment both identify the texture receiving fragment output. Presenting the swap chain and presenting the Metal drawable serve the same final purpose.

Recap
  • The output-merger writes pixels into a render target view; bind it each frame with OMSetRenderTargets.
  • The viewport maps the pipeline's output to pixels — fill a D3D11_VIEWPORT and call RSSetViewports, remembering Width/Height are floats.
  • ClearRenderTargetView fills the target with an RGBA color where each channel is 0.0–1.0.
  • swapChain->Present(1, 0) flips the finished frame to the screen and syncs to the monitor's refresh.
  • Bind → clear → (draw) → present is the render loop your whole engine will grow from.
