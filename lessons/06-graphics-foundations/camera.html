A Movable Camera

Until now our triangle and textured quad have sat frozen in front of a fixed viewpoint. A camera changes that: it lets us move through the scene and look around. In this lesson we build a small Camera struct that rebuilds its view matrix every frame from keyboard and mouse input, then hands the view and projection matrices to a constant buffer. This is exactly the shape our future Camera component will take inside the engine.

What you'll learn
  • Why a camera is really just two matrices: view and projection.
  • How to store camera state as position plus yaw and pitch.
  • How to rebuild the view matrix each frame with XMMatrixLookToLH.
  • How to move with WASD and look with the mouse, scaled by delta time.
  • How to upload view + projection into a constant buffer for the vertex shader.

A camera is two matrices

Rendering in 3D is a chain of coordinate spaces. A vertex starts in local space (relative to its own model), gets moved into the world by a world matrix, then the camera transforms it twice more:

  • View matrix — moves the whole world so the camera sits at the origin looking down +Z. Moving the camera "right" is the same as sliding the world left; that inversion is baked into this matrix.
  • Projection matrix — applies perspective, so distant objects shrink and everything fits inside the clip volume the GPU expects.

The projection rarely changes (only when the window resizes or you tweak the field of view). The view matrix, on the other hand, is rebuilt every single frame from the camera's current position and orientation. That rebuild is the heart of this lesson.

Storing camera state

There are two common ways to describe where a camera looks. You can store a target point and aim at it (great for orbiting a model), or you can store yaw and pitch angles and derive a forward direction (great for first-person movement). We'll use the angle approach.

  • Yaw is rotation around the up (Y) axis — turning left and right.
  • Pitch is rotation around the right (X) axis — looking up and down.
#include <DirectXMath.h>
using namespace DirectX;

// A minimal first-person camera. All angles are in radians.
struct Camera
{
    XMFLOAT3 position = { 0.0f, 0.0f, -5.0f }; // world-space eye position
    float yaw   = 0.0f;   // turn left / right (around Y)
    float pitch = 0.0f;   // look up / down   (around X)

    float moveSpeed = 5.0f;      // world units per second
    float lookSpeed = 0.0025f;   // radians per pixel of mouse movement

    // Rebuild the view matrix from the current position and angles.
    XMMATRIX View() const
    {
        // Rotate the default forward (+Z) and up (+Y) by pitch and yaw.
        XMMATRIX rot     = XMMatrixRotationRollPitchYaw(pitch, yaw, 0.0f);
        XMVECTOR forward = XMVector3TransformNormal(XMVectorSet(0, 0, 1, 0), rot);
        XMVECTOR up      = XMVector3TransformNormal(XMVectorSet(0, 1, 0, 0), rot);
        XMVECTOR eye     = XMLoadFloat3(&position);

        // LookTo takes a DIRECTION, not a target point.
        return XMMatrixLookToLH(eye, forward, up);
    }

    // The projection only depends on the window's aspect ratio here.
    XMMATRIX Projection(float aspect) const
    {
        float fovY = XMConvertToRadians(60.0f); // vertical field of view
        return XMMatrixPerspectiveFovLH(fovY, aspect, 0.1f, 1000.0f);
    }
};
LookTo vs LookAt

XMMatrixLookToLH wants a direction to look along, which is perfect when you already have a forward vector. If you'd rather aim at a specific point, use XMMatrixLookAtLH(eye, target, up) instead — same result, different inputs.

Updating from input each frame

Every frame we do two things: apply the mouse movement to yaw and pitch, then apply the keyboard to slide the position along the camera's own forward and right vectors. Assume you already have a tiny input snapshot gathered from your window's message loop: 

// Whatever your platform layer fills in each frame.
struct Input
{
    bool  keyW = false, keyS = false, keyA = false, keyD = false;
    float mouseDeltaX = 0.0f;  // pixels moved since last frame
    float mouseDeltaY = 0.0f;
};

The update is the important part. Notice how dt (delta time, in seconds) keeps movement speed independent of frame rate:

void Update(Camera& cam, const Input& in, float dt)
{
    // --- Mouse look ---
    cam.yaw   += in.mouseDeltaX * cam.lookSpeed;
    cam.pitch += in.mouseDeltaY * cam.lookSpeed;

    // Clamp pitch just short of straight up / down so the view can't flip.
    const float limit = XMConvertToRadians(89.0f);
    if (cam.pitch >  limit) cam.pitch =  limit;
    if (cam.pitch < -limit) cam.pitch = -limit;

    // --- Build the movement basis from the current orientation ---
    XMMATRIX rot     = XMMatrixRotationRollPitchYaw(cam.pitch, cam.yaw, 0.0f);
    XMVECTOR forward = XMVector3TransformNormal(XMVectorSet(0, 0, 1, 0), rot);
    XMVECTOR right   = XMVector3TransformNormal(XMVectorSet(1, 0, 0, 0), rot);

    // --- Gather WASD into a single direction ---
    XMVECTOR move = XMVectorZero();
    if (in.keyW) move = XMVectorAdd(move, forward);
    if (in.keyS) move = XMVectorSubtract(move, forward);
    if (in.keyD) move = XMVectorAdd(move, right);
    if (in.keyA) move = XMVectorSubtract(move, right);

    // Normalize so diagonal movement isn't faster, then scale by speed * dt.
    if (!XMVector3Equal(move, XMVectorZero()))
    {
        move = XMVector3Normalize(move);
        XMVECTOR pos = XMLoadFloat3(&cam.position);
        pos = XMVectorAdd(pos, XMVectorScale(move, cam.moveSpeed * dt));
        XMStoreFloat3(&cam.position, pos);
    }
}
Why multiply by dt?

If a machine renders at 200 FPS and another at 60 FPS, adding a fixed step per frame would make one player fly and the other crawl. Multiplying by dt (roughly 0.016 s at 60 FPS) turns moveSpeed into a true "5 units per second," identical on both machines.

Normalizing the combined move vector is a small but real detail. Without it, holding W + D would move you √2 ≈ 1.41× faster than a single key, because the two unit vectors add to a longer one.

Feeding the shader

The vertex shader needs both matrices, so we pack them into a constant buffer. This mirrors the per-frame constant buffer we set up when we first drew geometry — the camera simply fills it in. 

// CPU-side layout must match the HLSL cbuffer exactly.
struct FrameConstants
{
    XMFLOAT4X4 view;
    XMFLOAT4X4 projection;
};

void UploadCamera(ID3D11DeviceContext* ctx,
                  ID3D11Buffer* frameCB,
                  const Camera& cam,
                  float aspect)
{
    FrameConstants cb;
    // Transpose: DirectXMath is row-major, HLSL expects column-major.
    XMStoreFloat4x4(&cb.view,       XMMatrixTranspose(cam.View()));
    XMStoreFloat4x4(&cb.projection, XMMatrixTranspose(cam.Projection(aspect)));

    // Copy into the GPU buffer through a mapped pointer.
    D3D11_MAPPED_SUBRESOURCE mapped;
    ctx->Map(frameCB, 0, D3D11_MAP_WRITE_DISCARD, 0, &mapped);
    memcpy(mapped.pData, &cb, sizeof(cb));
    ctx->Unmap(frameCB, 0);

    // Bind at slot b0 for the vertex shader.
    ctx->VSSetConstantBuffers(0, 1, &frameCB);
}
Don't forget the transpose

DirectXMath stores matrices row-major, but HLSL reads float4x4 as column-major by default. If you skip XMMatrixTranspose, your geometry will vanish or smear across the screen. Transposing on the CPU and using mul(vector, matrix) in the shader is the standard, matching pair.

The matching vertex shader multiplies each vertex by world, then view, then projection:

cbuffer FrameConstants : register(b0)
{
    float4x4 view;
    float4x4 projection;
};

cbuffer ObjectConstants : register(b1)
{
    float4x4 world;   // comes from the object's Transform
};

struct VSIn  { float3 pos : POSITION; float2 uv : TEXCOORD0; };
struct VSOut { float4 pos : SV_POSITION; float2 uv : TEXCOORD0; };

VSOut main(VSIn input)
{
    VSOut o;
    float4 worldPos = mul(float4(input.pos, 1.0f), world);
    float4 viewPos  = mul(worldPos, view);
    o.pos = mul(viewPos, projection);
    o.uv  = input.uv;
    return o;
}

Wiring it into the frame loop

Putting the pieces together, each frame follows the same rhythm:

  1. Gather input into an Input snapshot.
  2. Call Update(camera, input, dt) to move and rotate.
  3. Call UploadCamera(...) with the current window aspect ratio.
  4. Draw your objects — they now appear from the camera's point of view.

Compute dt from a high-resolution timer such as QueryPerformanceCounter, taking the difference between this frame's timestamp and the previous one. Recompute aspect as width / height whenever the window resizes so the projection never stretches your scene.

Toward a Camera component

Look at the shape of this code: a small bundle of data (position, yaw, pitch) plus an Update(..., dt) function that runs every frame. That is precisely the contract every engine component will follow. When we reach the component system, this struct becomes a Camera component attached to an entity, and its position will come from a shared Transform instead of a raw XMFLOAT3.

macOS — camera data for Metal

The camera's position, yaw, pitch, field of view, and near/far planes are platform-neutral. On macOS, construct the view/projection values with the engine math layer or Apple simd and upload a simd_float4x4 to the Metal uniform buffer.

#if defined(_WIN32)
    backend.UploadCamera(ToDirectXMatrix(camera.ViewProjection()));
#elif defined(__APPLE__)
    backend.UploadCamera(ToSimdMatrix(camera.ViewProjection()));
#endif

Keep input and camera movement shared through GLFW. Only matrix layout, clip-space convention, and the final upload type belong to the renderer backend.

Recap
  • A camera is two matrices: a view matrix rebuilt each frame and a projection matrix that only changes on resize.
  • We store position, yaw, and pitch, then derive forward, right, and up with XMMatrixRotationRollPitchYaw and XMVector3TransformNormal.
  • XMMatrixLookToLH builds the view matrix from an eye, a direction, and an up vector.
  • Mouse deltas drive yaw and pitch (clamp pitch near ±89°); WASD slide the position, scaled by moveSpeed * dt and normalized so diagonals aren't faster.
  • Transpose both matrices before copying them into the constant buffer, then bind at b0.
  • This data-plus-Update(dt) shape foreshadows the engine's Camera component.
