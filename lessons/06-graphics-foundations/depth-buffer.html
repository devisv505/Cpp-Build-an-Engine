The Depth Buffer & Culling

Last lesson you drew a cube with an index buffer — but if you rotated it, the far side probably bled through the near side, giving a scrambled, see-through mess. That is because the GPU has no idea which triangles are closer to the camera. In this lesson we give it that knowledge with a depth buffer, and we tell it to skip triangles facing away from us with back-face culling. Both are core pieces every real engine — Unity, Unreal, or the one you are building — relies on for every frame.

What you'll learn
  • Why triangles overwrite each other without depth testing, and what the depth buffer stores.
  • How to create a depth-stencil ID3D11Texture2D and a matching depth-stencil view (DSV).
  • How to bind the DSV in OMSetRenderTargets and clear it every frame.
  • How a depth-stencil state and a rasterizer state control depth testing and back-face culling.

The problem: painter's chaos

By default, the GPU draws triangles in the order you submit them. The last triangle to touch a pixel wins — it simply overwrites whatever was there. This is sometimes called the "painter's algorithm", but it only works if you sort everything back-to-front by hand, which is impossible for a rotating solid where the ordering changes every frame.

The fix is a second image, the same size as your screen, that stores a depth value (0.0 near, 1.0 far) for every pixel. Before the GPU writes a pixel's color, it compares the new pixel's depth against the stored value. If the new pixel is farther away, it is discarded. This per-pixel test is called the depth test, and the image it reads and writes is the depth buffer (also called the z-buffer).

Creating the depth-stencil texture

The depth buffer is just a texture with a special format. We use DXGI_FORMAT_D24_UNORM_S8_UINT: 24 bits of depth and 8 bits of stencil packed together. We will not use the stencil part in this lesson, but the format is the standard, widely supported choice. The key detail is the bind flag D3D11_BIND_DEPTH_STENCIL, which tells Direct3D this texture will be attached to the output-merger as a depth target.

// device        : ID3D11Device*        (created earlier with the swap chain)
// width, height : your back-buffer size (match the swap chain exactly!)

// 1) Describe the depth-stencil texture.
D3D11_TEXTURE2D_DESC depthDesc = {};
depthDesc.Width              = width;
depthDesc.Height             = height;
depthDesc.MipLevels          = 1;                              // no mip chain for a depth target
depthDesc.ArraySize          = 1;
depthDesc.Format             = DXGI_FORMAT_D24_UNORM_S8_UINT;  // 24-bit depth + 8-bit stencil
depthDesc.SampleDesc.Count   = 1;                              // must match the render target's MSAA
depthDesc.SampleDesc.Quality = 0;
depthDesc.Usage              = D3D11_USAGE_DEFAULT;
depthDesc.BindFlags          = D3D11_BIND_DEPTH_STENCIL;       // the crucial flag
depthDesc.CPUAccessFlags     = 0;
depthDesc.MiscFlags          = 0;

Microsoft::WRL::ComPtr<ID3D11Texture2D> depthTexture;
HRESULT hr = device->CreateTexture2D(&depthDesc, nullptr, &depthTexture);
if (FAILED(hr)) { /* handle error */ return false; }

// 2) Create a depth-stencil VIEW so the pipeline can bind the texture.
D3D11_DEPTH_STENCIL_VIEW_DESC dsvDesc = {};
dsvDesc.Format             = DXGI_FORMAT_D24_UNORM_S8_UINT;
dsvDesc.ViewDimension      = D3D11_DSV_DIMENSION_TEXTURE2D;    // 2D, no MSAA
dsvDesc.Texture2D.MipSlice = 0;

Microsoft::WRL::ComPtr<ID3D11DepthStencilView> depthStencilView;
hr = device->CreateDepthStencilView(depthTexture.Get(), &dsvDesc, &depthStencilView);
if (FAILED(hr)) { /* handle error */ return false; }
Sizes and sample counts must match

The depth texture's Width, Height, and SampleDesc must exactly match your render target (the swap chain back buffer). A mismatch is one of the most common causes of a blank window and a cryptic debug-layer error. When you handle window resizing later, you will recreate both together.

Binding and clearing the depth buffer

A resource does nothing until it is bound to the pipeline. The depth-stencil view is bound as the third argument to OMSetRenderTargets, alongside your color render target. Previously you likely passed nullptr there; now you pass the real DSV.

// context : ID3D11DeviceContext*
// renderTargetView : ID3D11RenderTargetView* for the back buffer

// Bind color + depth together every frame.
ID3D11RenderTargetView* rtvs[] = { renderTargetView.Get() };
context->OMSetRenderTargets(1, rtvs, depthStencilView.Get());

Just like the color buffer, the depth buffer holds stale values from the previous frame. You must clear it at the start of every frame, resetting depth to 1.0f (the farthest value) so the first triangle at any pixel always passes the test. Use ClearDepthStencilView.

void Renderer::BeginFrame()
{
    const float clearColor[4] = { 0.06f, 0.07f, 0.10f, 1.0f };
    context->ClearRenderTargetView(renderTargetView.Get(), clearColor);

    // Reset depth to 1.0 (far) and stencil to 0, every frame.
    context->ClearDepthStencilView(
        depthStencilView.Get(),
        D3D11_CLEAR_DEPTH | D3D11_CLEAR_STENCIL, // clear both planes
        1.0f,                                    // depth clear value (far)
        0);                                      // stencil clear value

    // Re-bind targets after clearing (safe habit).
    ID3D11RenderTargetView* rtvs[] = { renderTargetView.Get() };
    context->OMSetRenderTargets(1, rtvs, depthStencilView.Get());
}
Forgetting to clear depth

If you clear the color buffer but not the depth buffer, frame two tests against frame one's depth values. The result is triangles that mysteriously vanish or flicker as the object rotates. If your cube looks "eaten away", check that ClearDepthStencilView runs every frame.

Depth-stencil state: turning the test on

Direct3D bundles the depth-test rules into an immutable ID3D11DepthStencilState object. The default behavior (when you bind nullptr) already enables depth testing with a "less" comparison, which is what we want. But creating the state explicitly makes your intent clear and lets you tweak it later.

D3D11_DEPTH_STENCIL_DESC dsDesc = {};
dsDesc.DepthEnable    = TRUE;                          // do the depth test
dsDesc.DepthWriteMask = D3D11_DEPTH_WRITE_MASK_ALL;    // write passing depths back
dsDesc.DepthFunc      = D3D11_COMPARISON_LESS;         // keep the nearer pixel
dsDesc.StencilEnable  = FALSE;                         // stencil off for now

Microsoft::WRL::ComPtr<ID3D11DepthStencilState> depthState;
device->CreateDepthStencilState(&dsDesc, &depthState);

// Bind once per frame before drawing (stencil ref = 0):
context->OMSetDepthStencilState(depthState.Get(), 0);
FieldWhat it controls
DepthEnableWhether the depth comparison happens at all.
DepthWriteMaskWhether passing pixels update the stored depth. Off for transparent passes.
DepthFuncThe comparison. LESS keeps nearer pixels for a standard 0-near / 1-far setup.

Back-face culling with a rasterizer state

A closed solid like a cube has two triangles for every visible one: the outer face pointing toward you, and the inner face pointing away. Drawing the inner faces is wasted work and can cause artifacts. Back-face culling tells the rasterizer to discard triangles whose winding order shows they face away from the camera. This is configured on an ID3D11RasterizerState.

The GPU decides "front" versus "back" from the order the vertices appear on screen. With FrontCounterClockwise = FALSE, a triangle whose vertices wind clockwise on screen is treated as front-facing. Make sure your cube's indices follow a consistent winding — the standard cube from the previous lesson uses clockwise front faces.

D3D11_RASTERIZER_DESC rsDesc = {};
rsDesc.FillMode              = D3D11_FILL_SOLID;
rsDesc.CullMode              = D3D11_CULL_BACK;   // skip away-facing triangles
rsDesc.FrontCounterClockwise = FALSE;            // clockwise = front face
rsDesc.DepthClipEnable       = TRUE;             // clip to the near/far planes

Microsoft::WRL::ComPtr<ID3D11RasterizerState> rasterState;
device->CreateRasterizerState(&rsDesc, &rasterState);

context->RSSetState(rasterState.Get());
Cube looks inside-out or empty?

If enabling culling makes your cube disappear, its triangles wind the opposite way. Either flip your index order or set CullMode = D3D11_CULL_FRONT. While debugging geometry, D3D11_CULL_NONE draws everything so you can see the shape first, then re-enable culling once winding is correct.

Putting it together in the frame loop

Here is how the per-frame calls line up. States are bound once, then you clear, then you draw. Everything below assumes the view/projection matrices and the indexed cube from the previous lesson are already set up.

void Renderer::RenderCube()
{
    BeginFrame();  // clears color + depth, binds RTV + DSV

    // Pipeline state for solid 3D geometry.
    context->OMSetDepthStencilState(depthState.Get(), 0);
    context->RSSetState(rasterState.Get());

    // (Input layout, shaders, constant buffers, vertex + index buffers
    //  were bound as in the previous lesson.)
    context->DrawIndexed(36, 0, 0);  // 12 triangles * 3 = 36 indices for a cube

    swapChain->Present(1, 0);         // vsync on
}
Why this matters for the engine

Every renderable in our future engine will submit its geometry through a path like this. The depth-stencil state and rasterizer state will become part of a reusable material or render state object, so opaque objects share one configuration and, later, transparent objects use another (depth test on, depth write off). Getting the mental model right now pays off when we build the MeshRenderer component.

Compile and run. Rotate the cube and it should now look like a solid object: near faces cleanly cover far ones, and the interior triangles you never see are culled away. You have just added two of the most fundamental pieces of any 3D renderer.

macOS — depth testing and culling in Metal

MTL::DepthStencilDescriptor* depth =
    MTL::DepthStencilDescriptor::alloc()->init();
depth->setDepthCompareFunction(MTL::CompareFunctionLess);
depth->setDepthWriteEnabled(true);

MTL::DepthStencilState* depthState =
    device->newDepthStencilState(depth);

encoder->setDepthStencilState(depthState);
encoder->setCullMode(MTL::CullModeBack);
encoder->setFrontFacingWinding(MTL::WindingCounterClockwise);

The render pass also needs a depth attachment texture matching the drawable dimensions. As with Direct3D, clear depth to 1.0, recreate it on resize, and keep winding conventions consistent with the projection matrix.

Recap
  • Without a depth buffer, triangles overwrite each other by draw order, producing a see-through mess.
  • Create an ID3D11Texture2D with DXGI_FORMAT_D24_UNORM_S8_UINT and D3D11_BIND_DEPTH_STENCIL, then a ID3D11DepthStencilView over it.
  • Pass the DSV as the third argument to OMSetRenderTargets, and call ClearDepthStencilView to reset depth to 1.0f every frame.
  • A depth-stencil state (DepthFunc = LESS) enables the depth test; a rasterizer state with CullMode = D3D11_CULL_BACK skips away-facing triangles.
  • The depth texture must match the render target's size and sample count, or the frame will not render.
