Your First Triangle

Last lesson we cleared the back buffer to a solid color — proof that our swap chain and render target are wired up correctly. Now we put actual geometry on screen. A single triangle is the "Hello, World" of GPU rendering, and every model your engine will ever draw is ultimately just a pile of triangles. In this lesson you'll describe three vertices in C++, upload them to GPU memory as a vertex buffer, and ask the device to draw them.

What you'll learn
  • How to define a Vertex struct with position and color using DirectXMath types
  • How to create a GPU vertex buffer with D3D11_BUFFER_DESC and CreateBuffer
  • How to bind the buffer and set the primitive topology on the Input Assembler
  • How to issue a Draw call to render the triangle
  • Why we still need shaders and an input layout (coming in the next two lessons)

The GPU pipeline, from the top

When you tell Direct3D to draw, your data flows through a series of stages called the graphics pipeline. The very first stage is the Input Assembler (IA). Its job is to read raw vertices out of a buffer and group them into primitives — points, lines, or triangles. Those primitives then travel to the vertex shader, the rasterizer, and finally the pixel shader, which decides the color of each pixel on screen.

Today we focus on feeding the Input Assembler. That means answering three questions for the GPU: What is a vertex? Where is the vertex data? And how should those vertices be assembled into shapes?

Defining a vertex

A vertex is just a small bundle of data attached to one corner of our geometry. For our first triangle each vertex carries two attributes: a position in 3D space and a color. Direct3D ships with a math library, DirectXMath, that gives us tightly-packed types perfect for this. We use XMFLOAT3 (three floats) for position and XMFLOAT4 (four floats — red, green, blue, alpha) for color.

#include <DirectXMath.h>
using namespace DirectX;

// One corner of our geometry. The GPU will read these
// fields in the exact order we declare them here.
struct Vertex
{
    XMFLOAT3 position;  // x, y, z
    XMFLOAT4 color;     // r, g, b, a
};

// Three vertices = one triangle. Positions are given in
// clip space for now: x and y run from -1 (left/bottom)
// to +1 (right/top), z is 0. We'll replace this with real
// world-space coordinates once the engine has a camera.
static const Vertex kVertices[] =
{
    { XMFLOAT3( 0.0f,  0.5f, 0.0f), XMFLOAT4(1.0f, 0.0f, 0.0f, 1.0f) }, // top    - red
    { XMFLOAT3( 0.5f, -0.5f, 0.0f), XMFLOAT4(0.0f, 1.0f, 0.0f, 1.0f) }, // right  - green
    { XMFLOAT3(-0.5f, -0.5f, 0.0f), XMFLOAT4(0.0f, 0.0f, 1.0f, 1.0f) }, // left   - blue
};
Winding order matters

Direct3D's default is to treat triangles whose vertices wind clockwise as front-facing, and to cull (skip) back-facing ones. Our three vertices are listed clockwise, so the triangle faces the camera. If your triangle mysteriously vanishes, a reversed winding order is a common culprit.

Creating the vertex buffer

The array above lives in system RAM, but the GPU can't reach it there. We need to copy it into a resource the GPU owns — a buffer. Creating any Direct3D resource follows the same two-part pattern: fill out a description struct that says what you want, optionally supply the initial data, then call a Create… method on the device.

Here device is the ID3D11Device* you created back when you set up the swap chain. Buffer creation happens once, at load time — not every frame.

// device       : ID3D11Device*        (created with the swap chain)
// context      : ID3D11DeviceContext* (records our draw commands)
// vertexBuffer : Microsoft::WRL::ComPtr<ID3D11Buffer>

// 1) Describe the buffer we want the GPU to allocate.
D3D11_BUFFER_DESC bufferDesc = {};
bufferDesc.ByteWidth      = sizeof(kVertices);          // total size in bytes
bufferDesc.Usage          = D3D11_USAGE_DEFAULT;        // GPU reads it, we won't update it often
bufferDesc.BindFlags      = D3D11_BIND_VERTEX_BUFFER;   // this is vertex data
bufferDesc.CPUAccessFlags = 0;                          // CPU needs no direct access
bufferDesc.MiscFlags      = 0;
bufferDesc.StructureByteStride = 0;

// 2) Point the GPU at our initial vertex data.
D3D11_SUBRESOURCE_DATA initData = {};
initData.pSysMem = kVertices;                           // source array in system memory

// 3) Ask the device to allocate the buffer and copy the data in.
HRESULT hr = device->CreateBuffer(&bufferDesc, &initData, &vertexBuffer);
if (FAILED(hr))
{
    // In the engine we'll route this through a proper error handler.
    return hr;
}
Always check your HRESULTs

Nearly every Direct3D call returns an HRESULT. Use the FAILED() / SUCCEEDED() macros rather than comparing to S_OK directly. Silent failures here lead to a black screen that's painful to debug — checking early saves hours.

Why ComPtr?

Notice vertexBuffer is a Microsoft::WRL::ComPtr<ID3D11Buffer> rather than a raw pointer. Direct3D objects are reference-counted COM objects. ComPtr is a smart pointer (like std::unique_ptr, but for COM) that calls Release() for you when it goes out of scope. Using it everywhere means we won't leak GPU resources as the engine grows — exactly the kind of RAII discipline we established in the earlier C++ modules.

Binding and drawing

With the buffer created, the remaining work happens every frame on the device context. We tell the Input Assembler which buffer to read, how big each vertex is (its stride), where to start (its offset), and how to group vertices into primitives. Then we call Draw.

// --- Per-frame: after clearing the render target ---

// How many bytes to advance per vertex, and where to start.
UINT stride = sizeof(Vertex);
UINT offset = 0;

// Bind our buffer to input slot 0 of the Input Assembler.
context->IASetVertexBuffers(
    0,                          // start slot
    1,                          // number of buffers
    vertexBuffer.GetAddressOf(),// the buffer(s)
    &stride,                    // bytes per vertex
    &offset);                   // starting byte

// Tell the IA to assemble every 3 vertices into one triangle.
context->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

// Draw 3 vertices, starting at vertex 0.
context->Draw(3, 0);

// (swap chain Present() happens after this, as before)

The two arguments to Draw are the vertex count and the index of the first vertex. Because we're using a triangle list, the GPU consumes vertices in groups of three: vertices 0, 1, 2 form our one triangle. Draw six vertices and you'd get two separate triangles; that's how we'll eventually build a quad, and later, whole meshes.

Why nothing shows up yet

If you compile and run with only the code above, you'll still see just your cleared background. That's expected — the pipeline has two more pieces that are literally required before a single pixel can be shaded:

  • A vertex shader and pixel shader. The GPU has no built-in idea of what to do with a vertex; you must supply small programs written in HLSL. We'll write these in the next lesson, Shaders & HLSL.
  • An input layout (ID3D11InputLayout). This is the contract that tells the GPU how the bytes in each Vertex map to shader inputs — "the first 12 bytes are a POSITION, the next 16 are a COLOR." We'll build it in the lesson after that, once we have a compiled shader to validate it against.
Don't expect the triangle this lesson

Binding a vertex buffer and calling Draw without shaders bound is valid code, but it renders nothing. This is by design — we're building the pieces in a logical order. Your triangle will appear at the end of the input-layout lesson, when every stage of the Input Assembler and shader pipeline is finally connected.

How this fits the engine

This lowly triangle is the seed of your renderer. In a component-based engine, a MeshRenderer component will own a vertex buffer exactly like this one, created from mesh data loaded off disk. The stride/offset/topology dance you just wrote becomes a reusable DrawMesh() routine that the render system calls for every visible object each frame. Master the flow here — describe, create, bind, draw — and you've learned the pattern that every GPU resource in the engine follows.

macOS — draw the triangle with Metal

Create a shared buffer for the same vertex array, bind a compiled render pipeline, and encode a three-vertex draw:

MTL::Buffer* vertexBuffer = device->newBuffer(
    vertices, sizeof(vertices), MTL::ResourceStorageModeShared);

encoder->setRenderPipelineState(pipelineState);
encoder->setVertexBuffer(vertexBuffer, 0, 0);
encoder->drawPrimitives(MTL::PrimitiveTypeTriangle, 0, 3);

drawPrimitives is Metal's direct equivalent of a non-indexed Draw. Both APIs read vertices from slot 0, run the vertex function three times, rasterize one triangle, and run the fragment/pixel function.

Recap
  • A vertex is a bundle of per-corner data; ours holds an XMFLOAT3 position and an XMFLOAT4 color.
  • GPU resources are created with a description struct plus optional initial data, then a Create… call — here D3D11_BUFFER_DESC + D3D11_SUBRESOURCE_DATA + CreateBuffer.
  • Each frame you IASetVertexBuffers with the correct stride and offset, set the topology to TRIANGLELIST, and call Draw(3, 0).
  • Use ComPtr for COM objects and always check HRESULTs with FAILED().
  • The triangle won't appear until we add a vertex/pixel shader and an input layout in the next two lessons.
