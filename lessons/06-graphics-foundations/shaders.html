Shaders: HLSL & Metal Shading Language

Last lesson you drew a triangle, but the GPU can't do anything with your vertices until you tell it how. That "how" lives in two tiny programs: a vertex shader and a pixel shader, written in HLSL. In this lesson we'll write both for our colored triangle, compile them, and wire the resulting bytecode into the Direct3D 11 pipeline — the same shader flow your engine's material system will use later.

What you'll learn
  • What a vertex shader and a pixel shader each do, and where they sit in the pipeline.
  • How HLSL semantics like POSITION, COLOR, SV_POSITION and SV_TARGET connect stages together.
  • How to compile HLSL at runtime with D3DCompileFromFile (or offline with fxc).
  • How to create shader objects with CreateVertexShader / CreatePixelShader and bind them with VSSetShader / PSSetShader.

The two shaders in the pipeline

When you issue a draw call, the GPU runs your vertices through a fixed sequence of stages. Two of those stages are programmable and you must supply them:

  • The vertex shader runs once per vertex. Its job is to output a position in clip space — the coordinate system the rasterizer expects. It can also pass extra data (like a color) down to the next stage.
  • The pixel shader (also called the fragment shader) runs once per pixel that the triangle covers. Its job is to return the final color for that pixel.

Between them sits the fixed-function rasterizer, which turns the triangle into pixels and smoothly interpolates the per-vertex data across the surface. That interpolation is why a triangle with a red, green and blue corner blends into a rainbow in the middle — you never wrote code for the in-between pixels; the hardware did it for you.

HLSL structs and semantics

HLSL (High-Level Shading Language) looks a lot like C. Data moves between stages through structs, and every field carries a semantic — an uppercase tag that tells the GPU what a field means so it can be matched up between stages.

There are two kinds of semantics. Plain semantics like POSITION and COLOR are names you choose to link your vertex data to the shader input. Semantics beginning with SV_ are system-value semantics with special meaning to the hardware:

SemanticWhereMeaning
POSITIONVS inputThe raw vertex position coming from your vertex buffer.
COLORVS input / outputA user-defined color you attach to each vertex.
SV_POSITIONVS outputThe final clip-space position the rasterizer requires.
SV_TARGETPS outputThe color written to render target 0 (your back buffer).
SV_POSITION is mandatory

A vertex shader must output a float4 tagged SV_POSITION. It's in clip space, not screen pixels — for our flat 2D triangle we can pass coordinates straight through with w = 1. Once we add a camera and a Transform component, this is exactly where the view-projection matrix multiply will happen.

Writing the shaders

We'll keep both shaders in one .hlsl file with two entry points, VSMain and PSMain. The vertex shader receives a position and a color, forwards the color untouched, and promotes the 3-component position to a float4. The pixel shader simply returns whatever color the rasterizer handed it.

// Data coming in from the vertex buffer (matches our input layout).
struct VSInput
{
    float3 position : POSITION;   // one point of the triangle
    float4 color    : COLOR;      // its RGBA color
};

// Data passed from the vertex shader to the pixel shader.
struct PSInput
{
    float4 position : SV_POSITION; // clip-space position (required)
    float4 color    : COLOR;       // interpolated across the triangle
};

// --- Vertex shader: runs once per vertex ---
PSInput VSMain(VSInput input)
{
    PSInput output;
    // Promote the 3D position to a 4D clip-space position (w = 1).
    output.position = float4(input.position, 1.0f);
    output.color    = input.color;   // hand the color down the pipeline
    return output;
}

// --- Pixel shader: runs once per covered pixel ---
float4 PSMain(PSInput input) : SV_TARGET
{
    // Return the interpolated color as this pixel's final color.
    return input.color;
}
Names must match on both ends

The fields in VSInput and their semantics (POSITION, COLOR) must line up with the input layout you'll define next lesson. Likewise, the PSInput the pixel shader reads must match what the vertex shader writes. Semantics are the glue — mismatches produce a black screen or a compile error, not a crash.

Compiling HLSL to bytecode

The GPU doesn't run HLSL text directly — it runs compiled bytecode. You have two options:

Option A — compile at runtime with D3DCompileFromFile

Great while learning: edit the shader, re-run, see the change. It lives in d3dcompiler.h, so link against d3dcompiler.lib.

#include <d3d11.h>
#include <d3dcompiler.h>
#include <wrl/client.h>   // Microsoft::WRL::ComPtr

#pragma comment(lib, "d3dcompiler.lib")

using Microsoft::WRL::ComPtr;

// Compiles one entry point from an .hlsl file into a blob of bytecode.
HRESULT CompileShader(const wchar_t* path, const char* entry,
                      const char* target, ComPtr<ID3DBlob>& outBlob)
{
    UINT flags = D3DCOMPILE_ENABLE_STRICTNESS;
#if defined(_DEBUG)
    // Keep symbols and skip optimization so debugging shaders is sane.
    flags |= D3DCOMPILE_DEBUG | D3DCOMPILE_SKIP_OPTIMIZATION;
#endif

    ComPtr<ID3DBlob> errors;
    HRESULT hr = D3DCompileFromFile(
        path,            // L"Triangle.hlsl"
        nullptr,         // no #defines
        D3D_COMPILE_STANDARD_FILE_INCLUDE, // resolve #include normally
        entry,           // "VSMain" or "PSMain"
        target,          // "vs_5_0" or "ps_5_0"
        flags, 0,
        &outBlob, &errors);

    if (FAILED(hr) && errors)
        OutputDebugStringA((const char*)errors->GetBufferPointer());

    return hr;
}
Always read the error blob

When a compile fails, D3DCompileFromFile fills the errors blob with the exact line and reason. Printing it (as above) turns a mysterious FAILED(hr) into a readable message like error X3018: invalid subscript. Never ignore it.

Option B — compile offline with fxc

For shipping, compile ahead of time with the fxc tool from the Windows SDK and load the resulting .cso file at runtime. No compiler dependency in your release build:

# Vertex shader -> Triangle_VS.cso
fxc /T vs_5_0 /E VSMain /Fo Triangle_VS.cso Triangle.hlsl
# Pixel shader  -> Triangle_PS.cso
fxc /T ps_5_0 /E PSMain /Fo Triangle_PS.cso Triangle.hlsl

Either way you end up with a block of bytecode. With D3DCompileFromFile it's an ID3DBlob; with fxc it's a file you read into a std::vector<char>. Both feed the same Create*Shader calls next.

Creating and binding the shaders

Now we turn bytecode into GPU shader objects with CreateVertexShader and CreatePixelShader, then bind them each frame with VSSetShader and PSSetShader. Assume device is your ID3D11Device and context your ID3D11DeviceContext from the previous lessons.

ComPtr<ID3D11VertexShader> vertexShader;
ComPtr<ID3D11PixelShader>  pixelShader;
ComPtr<ID3DBlob> vsBlob, psBlob;

// 1) Compile both entry points from the same file.
HRESULT hr = CompileShader(L"Triangle.hlsl", "VSMain", "vs_5_0", vsBlob);
if (FAILED(hr)) return hr;
hr = CompileShader(L"Triangle.hlsl", "PSMain", "ps_5_0", psBlob);
if (FAILED(hr)) return hr;

// 2) Create the GPU shader objects from the bytecode.
hr = device->CreateVertexShader(
        vsBlob->GetBufferPointer(), vsBlob->GetBufferSize(),
        nullptr, &vertexShader);
if (FAILED(hr)) return hr;

hr = device->CreatePixelShader(
        psBlob->GetBufferPointer(), psBlob->GetBufferSize(),
        nullptr, &pixelShader);
if (FAILED(hr)) return hr;

// 3) Bind them to the pipeline (do this before Draw each frame).
context->VSSetShader(vertexShader.Get(), nullptr, 0);
context->PSSetShader(pixelShader.Get(),  nullptr, 0);

// ... set input layout + vertex buffer (next lesson), then:
// context->Draw(3, 0);

Notice we hang onto vsBlob beyond this point. That's deliberate: next lesson, CreateInputLayout needs the vertex-shader bytecode to validate that your vertex format matches the shader's VSInput. That's the direct bridge into Input Layouts & Vertex Formats.

Why ComPtr?

Microsoft::WRL::ComPtr is a smart pointer for COM objects. It calls Release() automatically when it goes out of scope, so you don't leak Direct3D resources. In our engine every renderer resource — shaders, buffers, textures — will be held in a ComPtr for exactly this reason.

Bind these two shaders, hook up the triangle's vertex buffer, and your rainbow triangle appears — this time driven entirely by code you wrote, not fixed hardware behavior. That's the whole point: shaders are where your look-and-feel lives, and every material in your engine will be a pair of programs just like these.

macOS — the same shaders in Metal Shading Language

Metal Shading Language (MSL) uses C++-like syntax and explicit stage attributes. This pair performs the same work as the HLSL vertex and pixel shaders:

#include <metal_stdlib>
using namespace metal;

struct Vertex { float3 position; float3 color; };
struct VertexOut { float4 position [[position]]; float3 color; };

vertex VertexOut vertexMain(const device Vertex* vertices [[buffer(0)]],
                            uint id [[vertex_id]])
{
    VertexOut out;
    out.position = float4(vertices[id].position, 1.0);
    out.color = vertices[id].color;
    return out;
}

fragment float4 fragmentMain(VertexOut in [[stage_in]])
{
    return float4(in.color, 1.0);
}

Compile .metal files into a .metallib as a CMake build step, then load vertexMain and fragmentMain into an MTL::RenderPipelineDescriptor. HLSL semantics such as SV_POSITION map to MSL attributes such as [[position]].

Recap
  • The vertex shader runs per vertex and must output an SV_POSITION in clip space; the pixel shader runs per pixel and outputs SV_TARGET.
  • Semantics (POSITION, COLOR, SV_POSITION, SV_TARGET) glue the stages together, and the rasterizer interpolates per-vertex data in between.
  • Compile HLSL to bytecode at runtime with D3DCompileFromFile (read the error blob!) or offline with fxc.
  • Create objects with CreateVertexShader / CreatePixelShader, then bind them with VSSetShader / PSSetShader before drawing.
  • Keep the vertex-shader blob around — the next lesson's input layout needs it.
