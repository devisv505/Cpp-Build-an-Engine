Initializing the Graphics Backend: Direct3D 11 & Metal

The window from Module 5 is only an empty operating-system surface. This lesson connects it to the GPU. Follow the Direct3D 11 path on Windows or the Metal path on macOS; both create the same engine concepts even though the API names and presentation models differ.

What you'll learn
  • Which GPU, command-submission, presentation and color-target objects a renderer needs
  • How GLFW exposes an HWND on Windows and an NSView on macOS
  • How to initialize a complete Direct3D 11 backend on Windows
  • How to initialize a Metal device, command queue and CAMetalLayer on macOS
  • Why macOS bridge code uses Objective-C++ and how metal-cpp objects are released
Choose one platform path

Windows students build the Direct3D 11 sections. macOS students build the Metal sections. Read the shared concepts and checkpoint on either platform; you do not need a Windows machine to complete the Metal version.

One renderer, two API vocabularies

The engine needs the same four jobs on both platforms. Direct3D exposes a swap chain and a persistent render-target view. Metal exposes a display layer that lends the renderer a drawable texture for each frame.

Engine jobDirect3D 11 (Windows)Metal (macOS)
Create GPU resourcesID3D11DeviceMTL::Device
Submit commandsID3D11DeviceContextMTL::CommandQueue, then one command buffer per frame
Present to the windowIDXGISwapChainCA::MetalLayer and CA::MetalDrawable
Receive rendered pixelsID3D11RenderTargetViewThe current drawable's MTL::Texture
Manage lifetimeMicrosoft::WRL::ComPtrmetal-cpp retain()/release()

There is no one-to-one Metal equivalent of an immediate D3D11 context or DXGI swap chain. The responsibilities map across, not necessarily the individual classes. Keep that distinction in mind when you later hide both backends behind one renderer interface.

Project files and CMake

Keep the platform bridge out of main.cpp. The common program includes one platform renderer, while CMake compiles only the files needed by the current operating system.

│   ├── main.cpp
│   ├── RendererD3D11.h
│   ├── RendererD3D11.cpp

│   └── MetalCpp.cpp

On macOS, download or clone Apple's official metal-cpp repository and place its three header folders under external/metal-cpp. The .mm extension tells Clang to compile the Cocoa bridge as Objective-C++: a file that can use both C++ and AppKit types such as NSView.

After changing CMake, use Tools → CMake → Reset Cache and Reload Project in CLion. This is especially important after adding the OBJCXX language or changing metal-cpp's include path.

The common program

Both platform headers define the same small Renderer class interface, so the application boot code stays identical.

#include <GLFW/glfw3.h>
#include <iostream>

#if defined(_WIN32)
    #include "RendererD3D11.h"
#elif defined(__APPLE__)
    #include "MetalRenderer.h"
    #include <Foundation/Foundation.hpp>
#endif

int RunEngine() {
    if (!glfwInit()) {
        std::cerr << "GLFW initialization failed\n";
        return 1;
    }

    // Direct3D and Metal create their own presentation surfaces.
    glfwWindowHint(GLFW_CLIENT_API, GLFW_NO_API);
    GLFWwindow* window = glfwCreateWindow(1280, 720, "MiniEngine", nullptr, nullptr);
    if (!window) {
        glfwTerminate();
        return 1;
    }

    Renderer renderer;
    if (!renderer.Initialize(window)) {
        std::cerr << "Graphics backend initialization failed\n";
        glfwDestroyWindow(window);
        glfwTerminate();
        return 1;
    }

    std::cout << "Graphics backend initialized\n";
    while (!glfwWindowShouldClose(window)) {
        glfwPollEvents();
        renderer.Resize(window); // harmless when the size has not changed
        // RenderFrame() arrives in Lesson 40.
    }

    renderer.Shutdown();        // release graphics objects before the window
    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}

int main() {
#if defined(__APPLE__)
    // Cocoa and Metal create autoreleased objects during startup and shutdown.
    NS::AutoreleasePool* appPool = NS::AutoreleasePool::alloc()->init();
#endif

    const int exitCode = RunEngine();

#if defined(__APPLE__)
    appPool->release();
#endif
    return exitCode;
}
GLFW_NO_API is required

Without this hint, GLFW creates an OpenGL context. Direct3D 11 and Metal do not use that context; they connect their own presentation objects to the native window.

Windows path — Direct3D 11

On Windows, GLFW owns the Win32 window and gives us its HWND. Direct3D uses that handle as the output window for a swap chain.

#pragma once

#include <d3d11.h>
#include <dxgi.h>
#include <wrl/client.h>

struct GLFWwindow;

class Renderer {
public:
    bool Initialize(GLFWwindow* window);
    void Resize(GLFWwindow*) {} // Lesson 40 replaces this inline no-op.
    void Shutdown();

private:
    Microsoft::WRL::ComPtr<ID3D11Device> device;
    Microsoft::WRL::ComPtr<ID3D11DeviceContext> context;
    Microsoft::WRL::ComPtr<IDXGISwapChain> swapChain;
    Microsoft::WRL::ComPtr<ID3D11RenderTargetView> renderTargetView;
};
#include "RendererD3D11.h"

#define GLFW_EXPOSE_NATIVE_WIN32
#include <GLFW/glfw3.h>
#include <GLFW/glfw3native.h>

bool Renderer::Initialize(GLFWwindow* window) {
    HWND hwnd = glfwGetWin32Window(window);
    if (!hwnd) return false;

    DXGI_SWAP_CHAIN_DESC desc{}; // zero every unused field
    desc.BufferDesc.Width = 0;   // match the window's client size
    desc.BufferDesc.Height = 0;
    desc.BufferDesc.Format = DXGI_FORMAT_R8G8B8A8_UNORM;
    desc.SampleDesc.Count = 1;   // no multisampling yet
    desc.BufferUsage = DXGI_USAGE_RENDER_TARGET_OUTPUT;
    desc.BufferCount = 2;
    desc.OutputWindow = hwnd;
    desc.Windowed = TRUE;
    desc.SwapEffect = DXGI_SWAP_EFFECT_DISCARD;

    const D3D_FEATURE_LEVEL requested[] = {
        D3D_FEATURE_LEVEL_11_0,
        D3D_FEATURE_LEVEL_10_1,
        D3D_FEATURE_LEVEL_10_0
    };
    D3D_FEATURE_LEVEL obtained{};

    UINT flags = 0;
#if defined(_DEBUG)
    flags |= D3D11_CREATE_DEVICE_DEBUG;
#endif

    HRESULT hr = D3D11CreateDeviceAndSwapChain(
        nullptr, D3D_DRIVER_TYPE_HARDWARE, nullptr, flags,
        requested, ARRAYSIZE(requested), D3D11_SDK_VERSION,
        &desc, swapChain.GetAddressOf(), device.GetAddressOf(),
        &obtained, context.GetAddressOf());
    if (FAILED(hr)) return false;

    // The swap chain owns the image; the view lets D3D render into it.
    Microsoft::WRL::ComPtr<ID3D11Texture2D> backBuffer;
    hr = swapChain->GetBuffer(0, IID_PPV_ARGS(backBuffer.GetAddressOf()));
    if (FAILED(hr)) return false;

    hr = device->CreateRenderTargetView(
        backBuffer.Get(), nullptr, renderTargetView.GetAddressOf());
    return SUCCEEDED(hr);
}

void Renderer::Shutdown() {
    renderTargetView.Reset();
    swapChain.Reset();
    context.Reset();
    device.Reset();
}

ComPtr calls the COM object's Release() method automatically. The back-buffer pointer can therefore leave scope after the render-target view is created; the swap chain and view keep the underlying texture alive.

macOS path — Metal with metal-cpp

Metal does not create a DXGI-style swap chain. Instead, a CAMetalLayer is attached to GLFW's Cocoa view. Each frame, the layer supplies a drawable: a texture that can be rendered and then presented.

1. Define the metal-cpp implementation once

metal-cpp is header-only, but one translation unit must generate its private implementation symbols. Put the three macros in this file and nowhere else.

#define NS_PRIVATE_IMPLEMENTATION
#define CA_PRIVATE_IMPLEMENTATION
#define MTL_PRIVATE_IMPLEMENTATION

#include <Foundation/Foundation.hpp>
#include <Metal/Metal.hpp>
#include <QuartzCore/QuartzCore.hpp>

2. Store the long-lived Metal objects

#pragma once

struct GLFWwindow;
namespace MTL { class Device; class CommandQueue; }
namespace CA { class MetalLayer; }

class Renderer {
public:
    bool Initialize(GLFWwindow* window);
    void Resize(GLFWwindow* window);
    // Lesson 40 adds: void RenderFrame();
    void Shutdown();

private:
    MTL::Device* device = nullptr;
    MTL::CommandQueue* commandQueue = nullptr;
    CA::MetalLayer* layer = nullptr;
};

These names are deliberate. In Lesson 40, RenderFrame() will use the same layer and commandQueue members to acquire, encode and present a frame.

3. Bridge GLFW to Cocoa and attach the Metal layer

#include "MetalRenderer.h"

#define GLFW_EXPOSE_NATIVE_COCOA
#include <GLFW/glfw3.h>
#include <GLFW/glfw3native.h>

#import <Cocoa/Cocoa.h>
#import <QuartzCore/CAMetalLayer.h>

#include <Foundation/Foundation.hpp>
#include <Metal/Metal.hpp>
#include <QuartzCore/QuartzCore.hpp>

bool Renderer::Initialize(GLFWwindow* window) {
    // Create selects the system's default GPU and returns an owned object.
    device = MTL::CreateSystemDefaultDevice();
    if (!device) return false;

    // The queue creates the command buffers submitted in future frames.
    commandQueue = device->newCommandQueue();
    if (!commandQueue) {
        Shutdown();
        return false;
    }

    // GLFW 3.4 exposes the NSView inside its Cocoa window.
    NSView* view = glfwGetCocoaView(window);
    if (!view) {
        Shutdown();
        return false;
    }

    // alloc/init gives Renderer one owned reference to the displayed layer.
    CAMetalLayer* cocoaLayer = [[CAMetalLayer alloc] init];
    if (!cocoaLayer) {
        Shutdown();
        return false;
    }
    layer = reinterpret_cast<CA::MetalLayer*>(cocoaLayer);
    layer->setDevice(device);
    layer->setPixelFormat(MTL::PixelFormatBGRA8Unorm);
    layer->setFramebufferOnly(true);

    cocoaLayer.opaque = YES;
    view.wantsLayer = YES;
    view.layer = cocoaLayer;

    Resize(window);
    return true;
}

void Renderer::Resize(GLFWwindow* window) {
    if (!layer) return;

    NSView* view = glfwGetCocoaView(window);
    if (!view) return;

    // Keep the Core Animation layer over the complete Cocoa view.
    CAMetalLayer* cocoaLayer = reinterpret_cast<CAMetalLayer*>(layer);
    cocoaLayer.frame = view.bounds;

    // A Metal texture is sized in physical pixels, not logical window points.
    int pixelWidth = 0;
    int pixelHeight = 0;
    glfwGetFramebufferSize(window, &pixelWidth, &pixelHeight);
    layer->setDrawableSize(CGSizeMake(pixelWidth, pixelHeight));
}

void Renderer::Shutdown() {
    // Release in reverse order of creation. Null checks make this safe after
    // a partially failed Initialize(). Call Shutdown() before destroying GLFW.
    if (layer) {
        layer->release();
        layer = nullptr;
    }
    if (commandQueue) {
        commandQueue->release();
        commandQueue = nullptr;
    }
    if (device) {
        device->release();
        device = nullptr;
    }
}
Why framebuffer size matters on a Retina display

A 1280 × 720 GLFW window may occupy 2560 × 1440 physical pixels on a Retina screen. glfwGetWindowSize reports logical screen coordinates; glfwGetFramebufferSize reports the pixel dimensions required by CAMetalLayer.drawableSize. Using the wrong one makes the image soft, incorrectly scaled or only partially filled.

4. Understand metal-cpp ownership

metal-cpp follows Cocoa's ownership rules; it is not managed by C++ smart pointers or Objective-C ARC here.

  • MTL::CreateSystemDefaultDevice() returns an object the renderer owns.
  • newCommandQueue() contains new, so the renderer owns the returned queue.
  • [[CAMetalLayer alloc] init] contains alloc, so the renderer owns the returned layer.
  • Every owned object receives one matching release() in Shutdown().

The application-level pool in main() catches Cocoa objects created during startup and shutdown. Beginning in Lesson 40, Metal also creates many short-lived objects every frame. Put a smaller autorelease pool around each frame's Metal work so those temporary objects are reclaimed promptly instead of accumulating until the program exits:

NS::AutoreleasePool* pool = NS::AutoreleasePool::alloc()->init();

// Acquire the drawable, encode commands and present it here.

pool->release();

What success looks like

Build and run from CLion. The console should print Graphics backend initialized, and the window should remain open until you close it. Its contents may still be blank, black or unchanged — that is expected. Initialization gives the renderer somewhere to draw; Lesson 40 sends the first clear command.

ProblemLikely cause and fix
Metal/Metal.hpp file not foundMake sure external/metal-cpp/Metal/Metal.hpp exists, then reload CMake. Point target_include_directories at the directory that contains the Metal, Foundation and QuartzCore folders.
Undefined metal-cpp private symbolsAdd MetalCpp.cpp to the target and define each *_PRIVATE_IMPLEMENTATION macro exactly once.
NSView or Objective-C syntax errorsThe bridge file must be named MetalRenderer.mm, and CMake must enable OBJCXX.
glfwGetCocoaView is missingUse GLFW 3.4 and define GLFW_EXPOSE_NATIVE_COCOA before including glfw3native.h.
Metal drawable is later nullptrCheck that the layer is attached, the window is visible and drawableSize is not zero. A minimized window can legitimately have no drawable.
D3D creation fails only in DebugInstall the Windows Graphics Tools optional feature, or temporarily remove D3D11_CREATE_DEVICE_DEBUG.
Recap
  • Both backends need GPU-resource creation, command submission, window presentation and a writable color texture.
  • Windows connects GLFW's HWND to an IDXGISwapChain, then creates a render-target view for its back buffer.
  • macOS connects GLFW's NSView to a CAMetalLayer, which provides a drawable texture for each frame.
  • The Cocoa bridge is Objective-C++ (.mm); the rest of the Metal API is used through Apple's metal-cpp C++ headers.
  • Use framebuffer pixels for drawableSize, especially on Retina displays.
  • ComPtr manages D3D objects; metal-cpp objects follow retain/release ownership rules.
