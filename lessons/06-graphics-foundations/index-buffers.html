Index Buffers & Drawing a Cube

Last lesson you built a Model-View-Projection matrix and pushed a single triangle into 3D space. A cube has flat faces made of triangles too — but if you naively list every triangle you end up repeating the same corners over and over. In this lesson you'll learn indexed drawing: define each corner of the cube exactly once, then reference those corners by number. This is the same trick every real engine uses to keep mesh data small, and it's how our future Mesh component will store geometry.

What you'll learn
  • Why indexed drawing lets you reuse vertices instead of duplicating them.
  • How to define a cube as 8 unique vertices and a list of 36 indices (12 triangles).
  • How to create an index buffer with D3D11_BIND_INDEX_BUFFER and bind it with IASetIndexBuffer.
  • How to issue a DrawIndexed call and combine it with your MVP so a full cube renders.

Why bother with indices?

A cube has 6 faces. Each face is a flat square, and the GPU only draws triangles, so every face is made of 2 triangles. That's 12 triangles total, and each triangle needs 3 vertices — 36 vertices if we spell them all out. But a cube only has 8 physical corners. Each corner is shared by three faces, so in a plain vertex list we'd store the same corner position many times over.

Indexed drawing splits geometry into two lists. The vertex buffer holds each unique corner once. The index buffer holds a list of small integers that say "draw a triangle using corners 0, 1 and 2", "now 0, 2 and 3", and so on. The GPU walks the index list and looks up the real vertex data as it goes.

ApproachVertices storedBytes (pos = 12B each)
Non-indexed (36 verts)36432 bytes
Indexed (8 verts + 36 indices)8 verts + 36 × 2B96 + 72 = 168 bytes

For a bare cube the saving is modest, but the ratio only grows with real meshes. A character model might share each vertex among six or more triangles; indexing can cut its memory roughly in half and, just as importantly, lets the GPU's post-transform cache skip re-processing a vertex it already handled.

Rule of thumb

If a vertex is used by more than one triangle — which is almost always true for solid meshes — use an index buffer. When we write our model loader later, it will build both buffers automatically.

Defining the cube

Let's place the cube's 8 corners around the origin, each axis running from -1 to +1. We'll give each corner a colour too, so the faces are easy to tell apart. This is the same Vertex layout you've been using — a position plus a colour. 

#include <DirectXMath.h>
#include <array>

using namespace DirectX;

struct Vertex
{
    XMFLOAT3 position;  // x, y, z
    XMFLOAT4 color;     // r, g, b, a
};

// 8 unique corners of the cube, each from -1 to +1.
const std::array<Vertex, 8> g_cubeVertices =
{{
    { XMFLOAT3(-1, -1, -1), XMFLOAT4(0, 0, 0, 1) },  // 0
    { XMFLOAT3( 1, -1, -1), XMFLOAT4(1, 0, 0, 1) },  // 1
    { XMFLOAT3( 1,  1, -1), XMFLOAT4(1, 1, 0, 1) },  // 2
    { XMFLOAT3(-1,  1, -1), XMFLOAT4(0, 1, 0, 1) },  // 3
    { XMFLOAT3(-1, -1,  1), XMFLOAT4(0, 0, 1, 1) },  // 4
    { XMFLOAT3( 1, -1,  1), XMFLOAT4(1, 0, 1, 1) },  // 5
    { XMFLOAT3( 1,  1,  1), XMFLOAT4(1, 1, 1, 1) },  // 6
    { XMFLOAT3(-1,  1,  1), XMFLOAT4(0, 1, 1, 1) },  // 7
}};

Now the index list. Each row below is one face made of two triangles (six indices). The winding order — the order the three corners are listed — matters: with the default rasterizer state, Direct3D treats clockwise triangles as front-facing. We'll wind every face so it faces outward, which pays off next lesson when we enable backface culling. 

// 36 indices = 12 triangles = 6 faces.
// R16_UINT lets each index be a 16-bit unsigned short.
const std::array<unsigned short, 36> g_cubeIndices =
{{
    // front face (z = -1)
    0, 2, 1,   0, 3, 2,
    // back face  (z = +1)
    4, 5, 6,   4, 6, 7,
    // left face  (x = -1)
    4, 7, 3,   4, 3, 0,
    // right face (x = +1)
    1, 2, 6,   1, 6, 5,
    // top face   (y = +1)
    3, 7, 6,   3, 6, 2,
    // bottom face(y = -1)
    4, 0, 1,   4, 1, 5,
}};
16-bit vs 32-bit indices

An unsigned short (16 bits) can address up to 65,535 vertices — plenty for a cube and most small meshes. If a mesh has more vertices than that, switch to unsigned int and use DXGI_FORMAT_R32_UINT when you bind the buffer. The format you pick must match the integer size of your index array.

Creating the buffers

Creating an index buffer is almost identical to creating a vertex buffer — you fill a D3D11_BUFFER_DESC and a D3D11_SUBRESOURCE_DATA, then call CreateBuffer. The one thing that changes is the bind flag: D3D11_BIND_INDEX_BUFFER instead of D3D11_BIND_VERTEX_BUFFER. 

#include <d3d11.h>
#include <wrl/client.h>   // Microsoft::WRL::ComPtr

using Microsoft::WRL::ComPtr;

ComPtr<ID3D11Buffer> m_vertexBuffer;
ComPtr<ID3D11Buffer> m_indexBuffer;

void CreateCubeBuffers(ID3D11Device* device)
{
    // --- Vertex buffer (the 8 unique corners) ---
    D3D11_BUFFER_DESC vbDesc = {};
    vbDesc.ByteWidth = sizeof(g_cubeVertices);           // 8 * sizeof(Vertex)
    vbDesc.Usage     = D3D11_USAGE_IMMUTABLE;            // never changes after upload
    vbDesc.BindFlags = D3D11_BIND_VERTEX_BUFFER;

    D3D11_SUBRESOURCE_DATA vbData = {};
    vbData.pSysMem = g_cubeVertices.data();

    HRESULT hr = device->CreateBuffer(&vbDesc, &vbData, &m_vertexBuffer);
    if (FAILED(hr)) { /* handle error */ }

    // --- Index buffer (the 36 indices) ---
    D3D11_BUFFER_DESC ibDesc = {};
    ibDesc.ByteWidth = sizeof(g_cubeIndices);            // 36 * sizeof(unsigned short)
    ibDesc.Usage     = D3D11_USAGE_IMMUTABLE;
    ibDesc.BindFlags = D3D11_BIND_INDEX_BUFFER;          // the key difference

    D3D11_SUBRESOURCE_DATA ibData = {};
    ibData.pSysMem = g_cubeIndices.data();

    hr = device->CreateBuffer(&ibDesc, &ibData, &m_indexBuffer);
    if (FAILED(hr)) { /* handle error */ }
}
ByteWidth must be the whole buffer

ByteWidth is the size of the entire buffer in bytes, not the size of one element. Using sizeof(g_cubeIndices) on a std::array is correct because the array stores all elements inline. If you ever switch to a std::vector, use indices.size() * sizeof(unsigned short) instead — sizeof(vec) would give you the size of the vector object, not its contents.

Binding and drawing

At draw time you bind both buffers to the Input Assembler stage, then call DrawIndexed. Note the format argument to IASetIndexBuffer: DXGI_FORMAT_R16_UINT because our indices are 16-bit. This is where the MVP matrix from the previous lesson comes back in — we update the constant buffer, then let the vertex shader transform each corner. 

void DrawCube(ID3D11DeviceContext* context)
{
    // 1. Bind the vertex buffer.
    UINT stride = sizeof(Vertex);
    UINT offset = 0;
    context->IASetVertexBuffers(0, 1, m_vertexBuffer.GetAddressOf(), &stride, &offset);

    // 2. Bind the index buffer. Format must match the index size (16-bit here).
    context->IASetIndexBuffer(m_indexBuffer.Get(), DXGI_FORMAT_R16_UINT, 0);

    // 3. Each three indices form one triangle.
    context->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

    // 4. Bind shaders, input layout and the MVP constant buffer (from last lesson).
    context->VSSetShader(m_vertexShader.Get(), nullptr, 0);
    context->PSSetShader(m_pixelShader.Get(),  nullptr, 0);
    context->IASetInputLayout(m_inputLayout.Get());
    context->VSSetConstantBuffers(0, 1, m_mvpBuffer.GetAddressOf());

    // 5. Draw! 36 indices, starting at index 0, with no base vertex offset.
    context->DrawIndexed(36, 0, 0);
}

The three arguments to DrawIndexed are: how many indices to read (36), where in the index buffer to start (0), and a value added to every index before the vertex lookup (0). That last argument, the base vertex location, is handy when several meshes share one big buffer — but for a lone cube it stays zero.

Where the MVP fits

Nothing about indexing changes your shader. Each of the 8 corners still flows through the same vertex shader once (the GPU caches the result), gets multiplied by the combined world-view-projection matrix, and lands on screen. Here's the vertex shader from last lesson, unchanged, for reference: 

cbuffer MVPBuffer : register(b0)
{
    matrix worldViewProj;   // model * view * projection
};

struct VSInput
{
    float3 position : POSITION;
    float4 color    : COLOR;
};

struct VSOutput
{
    float4 position : SV_POSITION;
    float4 color    : COLOR;
};

VSOutput main(VSInput input)
{
    VSOutput output;
    // Transform the corner from model space into clip space.
    output.position = mul(float4(input.position, 1.0f), worldViewProj);
    output.color    = input.color;
    return output;
}

Update the constant buffer with your rotating world matrix each frame, call DrawCube, present — and a spinning, multi-coloured cube appears. If the back faces look wrong or the cube seems inside-out, don't worry: that's exactly the problem the depth buffer and culling lesson solves next.

Don't call Draw with an index buffer bound

Once you're indexing, use DrawIndexed, not Draw. Draw ignores the index buffer entirely and reads vertices in sequence, so it would try to draw only 8 vertices as raw triangles and produce garbage. Matching the draw call to the buffers you bound is a common early mistake.

macOS — indexed drawing in Metal

MTL::Buffer* indexBuffer = device->newBuffer(
    indices, sizeof(indices), MTL::ResourceStorageModeShared);

encoder->drawIndexedPrimitives(
    MTL::PrimitiveTypeTriangle,
    indexCount,
    MTL::IndexTypeUInt32,
    indexBuffer,
    0);

This is Metal's equivalent of binding an index buffer and calling DrawIndexed. Mesh data and index ordering stay shared; only the backend draw command changes.

Recap
  • Indexed drawing stores each unique vertex once and references it by a small integer, saving memory and GPU work.
  • A cube is 8 vertices and 36 indices (12 triangles / 6 faces); winding each face outward prepares us for culling.
  • Create an index buffer just like a vertex buffer but with D3D11_BIND_INDEX_BUFFER, and size ByteWidth to the whole array.
  • Bind it with IASetIndexBuffer (matching R16_UINT/R32_UINT to your index size) and draw with DrawIndexed(36, 0, 0).
  • The MVP constant buffer and shaders from the previous lesson stay exactly the same — indexing only changes how vertices are fetched.
