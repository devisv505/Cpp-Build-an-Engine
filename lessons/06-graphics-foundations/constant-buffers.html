Constant Buffers

In the last lesson we described the shape of each vertex with an input layout. But a vertex shader almost always needs more than just vertex data — it needs to know where the camera is, how the object is transformed, what color to tint things this frame. Constant buffers are how the CPU hands that per-frame and per-object data to your shaders. They are the pipe our engine's Transform and Camera components will eventually push their matrices through, so getting them right now pays off for the rest of the series.

Why constant buffers exist

Vertex data flows into the pipeline one vertex at a time through the input assembler. But some values are the same for every vertex in a draw call: the current world matrix, the view and projection matrices, a tint color, the elapsed time. It would be wasteful — and impossible with the fixed vertex stream — to duplicate those across every vertex. Instead we place them in a constant buffer: a small, GPU-resident chunk of memory that stays "constant" across a draw call and is readable by the shader as a named block of variables.

The word "constant" refers to its lifetime within a draw, not forever. You are free — and expected — to change the contents between draws and between frames. A typical engine keeps a few of them: one updated once per frame (view/projection, time, light direction) and one updated once per object (its world matrix, its material color).

Registers, not addresses

Shaders don't see pointers. A constant buffer is bound to a numbered slot called a register — b0, b1, and so on (the b is for "buffer"). Your HLSL declares which register it reads from, and your C++ binds a buffer to that same slot. Match the numbers and data flows; mismatch them and the shader reads garbage.

The 16-byte alignment rule

This is the single most common source of "my shader receives the wrong numbers" bugs, so let's meet it head-on. HLSL packs constant buffer data into four-component vectors of 16 bytes (think of each row as a float4). The rule: a single field is never allowed to straddle a 16-byte boundary. If a field would cross the boundary, it gets pushed to the start of the next 16-byte slot, leaving padding behind.

A float4x4 matrix is four rows of 16 bytes = 64 bytes, perfectly aligned. The trouble starts with small trailing fields. Consider a buffer holding a matrix, a float3 color, and a single float:

FieldSizeOffsetNotes
float4x4 world64 bytes0Four aligned rows
float3 color12 bytes64Fits in the first 12 bytes of a 16-byte slot
float time4 bytes76Slots into the gap after color — no padding needed here

Here color (12 bytes) and time (4 bytes) together fill exactly one 16-byte row, so the total is a tidy 80 bytes. But if time came before a float2, or if you had a lone float3 at the end, you would need explicit padding. The safest habit: keep your C++ struct field order identical to the HLSL, and pad to a multiple of 16 bytes.

The total size must be a multiple of 16

ID3D11Device::CreateBuffer will fail (and the debug layer will shout at you) if a constant buffer's ByteWidth is not a multiple of 16. When in doubt, add a padding field or wrap your struct in a size check like static_assert(sizeof(T) % 16 == 0).

Mirroring an HLSL cbuffer in C++

Let's define both sides for a per-object buffer. First the HLSL. Notice the register(b0) — that ties this block to slot 0.

// A constant buffer the vertex shader reads from register b0.
cbuffer PerObject : register(b0)
{
    float4x4 World;   // 64 bytes: object-to-world transform
    float4x4 ViewProj;// 64 bytes: combined view * projection
    float4   Color;   // 16 bytes: tint applied in the pixel shader
};

struct VSInput
{
    float3 position : POSITION;
    float3 normal   : NORMAL;
};

struct VSOutput
{
    float4 position : SV_POSITION;
    float4 color    : COLOR;
};

VSOutput main(VSInput input)
{
    VSOutput output;
    // Transform to world space, then to clip space.
    float4 worldPos = mul(float4(input.position, 1.0f), World);
    output.position = mul(worldPos, ViewProj);
    output.color    = Color;
    return output;
}

Now the matching C++ struct. The field order, the types, and the total size all line up with the HLSL. We use DirectXMath types, which we introduced when talking about vertex formats.

#include <DirectXMath.h>

// Mirror of the HLSL 'cbuffer PerObject : register(b0)'.
// Field order and sizes MUST match the shader.
struct PerObject
{
    DirectX::XMFLOAT4X4 world;    // 64 bytes
    DirectX::XMFLOAT4X4 viewProj; // 64 bytes
    DirectX::XMFLOAT4   color;    // 16 bytes
};

// 144 bytes total, and 144 % 16 == 0. Catch mistakes at compile time.
static_assert(sizeof(PerObject) % 16 == 0,
    "Constant buffer size must be a multiple of 16 bytes");
Row-major vs. column-major

HLSL treats matrices as column-major by default, while DirectXMath stores them row-major. The cleanest fix is to XMMatrixTranspose your matrices before uploading, or annotate the HLSL with row_major. We'll settle on transposing in the next lesson when we build real view and projection matrices — for now just know the two worlds order their rows differently.

Creating the buffer

You create a constant buffer like any other Direct3D resource, but with the D3D11_BIND_CONSTANT_BUFFER bind flag. There are two common usage patterns, and picking the right one matters for performance.

Dynamic + Map / Unmap

For data that changes every frame, use D3D11_USAGE_DYNAMIC with CPU write access. You update it by mapping the buffer (getting a CPU-visible pointer), copying your struct in, and unmapping. This is the go-to path for per-frame and per-object buffers.

#include <d3d11.h>
#include <wrl/client.h>   // Microsoft::WRL::ComPtr
using Microsoft::WRL::ComPtr;

ComPtr<ID3D11Buffer> g_perObjectCB;

// Create a dynamic constant buffer sized for our struct.
bool CreateConstantBuffer(ID3D11Device* device)
{
    D3D11_BUFFER_DESC desc = {};
    desc.ByteWidth      = sizeof(PerObject);          // multiple of 16
    desc.Usage          = D3D11_USAGE_DYNAMIC;        // updated by the CPU
    desc.BindFlags      = D3D11_BIND_CONSTANT_BUFFER; // this is the key flag
    desc.CPUAccessFlags = D3D11_CPU_ACCESS_WRITE;     // needed for Map()

    HRESULT hr = device->CreateBuffer(&desc, nullptr, &g_perObjectCB);
    return SUCCEEDED(hr);
}

And here is the per-frame update. D3D11_MAP_WRITE_DISCARD tells the driver "I'm replacing the whole contents, don't stall waiting for the GPU to finish reading the old data" — the ideal hint for a buffer you rewrite each frame.

void UpdatePerObject(ID3D11DeviceContext* ctx,
                     const DirectX::XMMATRIX& world,
                     const DirectX::XMMATRIX& viewProj,
                     const DirectX::XMFLOAT4& color)
{
    // Fill our CPU-side struct, transposing for HLSL's column-major layout.
    PerObject data;
    DirectX::XMStoreFloat4x4(&data.world,    DirectX::XMMatrixTranspose(world));
    DirectX::XMStoreFloat4x4(&data.viewProj, DirectX::XMMatrixTranspose(viewProj));
    data.color = color;

    // Map, copy, unmap.
    D3D11_MAPPED_SUBRESOURCE mapped = {};
    HRESULT hr = ctx->Map(g_perObjectCB.Get(), 0,
                          D3D11_MAP_WRITE_DISCARD, 0, &mapped);
    if (SUCCEEDED(hr))
    {
        memcpy(mapped.pData, &data, sizeof(PerObject));
        ctx->Unmap(g_perObjectCB.Get(), 0);
    }
}

Default + UpdateSubresource

For data that changes rarely — or when you prefer a one-line update — use D3D11_USAGE_DEFAULT and call UpdateSubresource. No mapping, no CPU access flag. It's simpler, but for buffers you touch every single frame the dynamic/discard path is usually friendlier to the GPU.

// Alternative: a DEFAULT-usage buffer updated with UpdateSubresource.
// desc.Usage = D3D11_USAGE_DEFAULT; desc.CPUAccessFlags = 0;

void UpdatePerObjectDefault(ID3D11DeviceContext* ctx, const PerObject& data)
{
    // Copies straight from system memory into the GPU resource.
    ctx->UpdateSubresource(g_perObjectCB.Get(), 0, nullptr,
                           &data, 0, 0);
}
Don't mix and match

A DYNAMIC buffer must be updated with Map/Unmap and cannot use UpdateSubresource; a DEFAULT buffer is the reverse. Choose one usage per buffer up front. Our engine will standardize on dynamic buffers for anything that changes per frame.

Binding and drawing

Creating and filling the buffer isn't enough — you must bind it to the register the shader expects. VSSetConstantBuffers makes it visible to the vertex shader; PSSetConstantBuffers does the same for the pixel shader. The first argument is the start slot (0 for b0).

void RenderFrame(ID3D11DeviceContext* ctx,
                 const DirectX::XMMATRIX& world,
                 const DirectX::XMMATRIX& viewProj)
{
    // 1) Update the CPU data and push it to the GPU this frame.
    DirectX::XMFLOAT4 orange(1.0f, 0.5f, 0.1f, 1.0f);
    UpdatePerObject(ctx, world, viewProj, orange);

    // 2) Bind the buffer to slot b0 for the vertex shader.
    ID3D11Buffer* cbs[] = { g_perObjectCB.Get() };
    ctx->VSSetConstantBuffers(0, 1, cbs);

    // If the pixel shader also reads it (e.g. for Color), bind there too.
    ctx->PSSetConstantBuffers(0, 1, cbs);

    // 3) Draw. The shader now sees World, ViewProj, and Color.
    ctx->DrawIndexed(/*indexCount*/ 36, 0, 0);
}

Put together, your per-frame loop reads: update the buffer, bind it, draw — repeated for each object with its own world matrix. Because we used WRITE_DISCARD, calling UpdatePerObject many times per frame (once per object) is cheap and correct.

Forgetting to bind is a silent failure

If you update the buffer but never call VSSetConstantBuffers, the shader reads whatever was last bound to that slot — often all zeros, giving you a black or invisible object with no error. Always enable the D3D11 debug layer (D3D11_CREATE_DEVICE_DEBUG) while developing; it flags an unbound or wrongly-sized buffer immediately.

How this fits the engine

You've just built the exact mechanism our engine will use to render moving, transformed objects. When we add a Transform component, its world matrix will be stored into a PerObject buffer like this one; when we add a Camera, its view and projection will fill a per-frame buffer. The renderer will loop over every object, update the per-object buffer, bind it, and draw — precisely the pattern above. Next lesson we finally build those world, view, and projection matrices properly, so the numbers we're shoveling into ViewProj actually put things on screen where you expect.

Recap
  • Constant buffers pass per-frame and per-object data (matrices, colors, time) to shaders; they stay constant across a draw call but you change them between draws.
  • HLSL packs data into 16-byte rows — no field may cross a boundary, and a buffer's total size must be a multiple of 16. Keep your C++ struct's field order identical to the HLSL cbuffer.
  • Create with D3D11_BIND_CONSTANT_BUFFER: use DYNAMIC + Map/Unmap (with WRITE_DISCARD) for per-frame data, or DEFAULT + UpdateSubresource for rarely-changing data.
  • Bind to a register with VSSetConstantBuffers / PSSetConstantBuffers, matching the register(bN) in your shader — forgetting to bind fails silently.
  • Transpose matrices before upload to bridge DirectXMath's row-major storage and HLSL's column-major default.
