Initializing Direct3D 11

Last lesson we walked the graphics pipeline conceptually: vertices in, pixels out. Now we make it real. In this lesson you will create the three core objects every Direct3D 11 program owns — the device, the device context, and the swap chain — and connect them to the window you built earlier. This is the boot sequence of our engine's renderer, and every future lesson builds directly on the objects you create here.

What you'll learn
  • What the device, device context and swap chain each do, and why you need all three
  • How to fill in a DXGI_SWAP_CHAIN_DESC field by field
  • How to call D3D11CreateDeviceAndSwapChain and request feature levels
  • How to get the back buffer and wrap it in a render target view
  • How to keep everything alive safely with Microsoft::WRL::ComPtr

The three objects you always create

Direct3D 11 splits its work across a few COM objects. When you are starting out, three of them matter:

ObjectInterfaceWhat it is for
DeviceID3D11DeviceCreates resources — textures, buffers, shaders, views. Think of it as the factory.
Device contextID3D11DeviceContextIssues commands — binds resources and draws. Think of it as the worker that uses what the factory made.
Swap chainIDXGISwapChainOwns the images ("buffers") shown on screen and flips them so the window updates.

A swap chain holds two (or more) images. You draw into the hidden back buffer while the monitor displays the front buffer; calling Present swaps them. That double buffering is what stops the viewer from seeing a half-drawn frame.

DXGI vs D3D

Notice the swap chain interface starts with IDXGI, not ID3D11. DXGI (DirectX Graphics Infrastructure) is the lower layer that manages adapters, outputs and presentation. Direct3D handles the actual rendering. The two cooperate, which is why one function call can create objects from both.

Linking and headers

Before any of this compiles, the linker needs the Direct3D import library. Add d3d11.lib to your linker inputs, or drop a pragma near the top of your file. We also lean on Microsoft::WRL::ComPtr, a smart pointer for COM objects that calls Release() automatically when it goes out of scope — the same idea as std::unique_ptr, but for reference-counted COM interfaces.

#include <d3d11.h>
#include <dxgi.h>
#include <wrl/client.h>   // Microsoft::WRL::ComPtr

#pragma comment(lib, "d3d11.lib")   // or add d3d11.lib to the linker settings

// A short alias keeps the code readable.
using Microsoft::WRL::ComPtr;
Let ComPtr own lifetimes

Every raw ID3D11* pointer must eventually be released exactly once. Managing that by hand is a classic source of leaks and crashes. Store these objects in ComPtr members of your renderer class and you never call Release() yourself — destruction is automatic and in the right order.

Describing the swap chain

Instead of a dozen function arguments, Direct3D takes a filled-in description struct. DXGI_SWAP_CHAIN_DESC nests two smaller structs: BufferDesc (the back buffer's size and pixel format) and SampleDesc (multisampling). Let's fill it in for a plain, windowed setup.

// hwnd is the window handle you created in the Win32 lesson.
DXGI_SWAP_CHAIN_DESC scd = {};                     // zero everything first

// --- The back buffer image ---
scd.BufferDesc.Width  = 0;                          // 0 = match the window's client area
scd.BufferDesc.Height = 0;
scd.BufferDesc.Format = DXGI_FORMAT_R8G8B8A8_UNORM; // 8 bits each for R,G,B,A
scd.BufferDesc.RefreshRate.Numerator   = 0;         // 0/0 = let DXGI pick the refresh rate
scd.BufferDesc.RefreshRate.Denominator = 0;

// --- Multisampling (anti-aliasing). 1 sample = off. ---
scd.SampleDesc.Count   = 1;
scd.SampleDesc.Quality = 0;

// --- How the buffer is used and how many there are ---
scd.BufferUsage  = DXGI_USAGE_RENDER_TARGET_OUTPUT; // we render into it
scd.BufferCount  = 2;                               // front + back = double buffering
scd.OutputWindow = hwnd;                            // the window to present into
scd.Windowed     = TRUE;                            // start windowed, never exclusive fullscreen here
scd.SwapEffect   = DXGI_SWAP_EFFECT_DISCARD;        // simplest present model

A few of these deserve a note. Setting Width and Height to 0 tells DXGI to look up the window's client size for you — handy so the numbers can never disagree with the window. BufferCount = 2 gives us one front and one back buffer. And OutputWindow = hwnd is the crucial link between Direct3D and the actual window on screen.

Always zero-initialize the struct

The = {} that value-initializes scd is not optional. These descriptors have fields we are not touching (Flags, ScanlineOrdering, and so on). Leave them holding garbage stack values and creation may fail with a cryptic E_INVALIDARG, or behave unpredictably. Zero first, then set what you care about.

Feature levels

A feature level is Direct3D's way of naming a fixed set of GPU capabilities, independent of the driver version. You hand in an array ordered from most to least desired, and creation picks the highest one the GPU actually supports, reporting it back to you.

// Ordered best-first. Creation returns the highest one this GPU supports.
const D3D_FEATURE_LEVEL requested[] = {
    D3D_FEATURE_LEVEL_11_1,
    D3D_FEATURE_LEVEL_11_0,
    D3D_FEATURE_LEVEL_10_1,
    D3D_FEATURE_LEVEL_10_0,
};
D3D_FEATURE_LEVEL obtained = {};   // filled in by the create call

Creating everything at once

With the descriptor and feature levels ready, one call builds the device, the context and the swap chain together. We store each result in a ComPtr, and pass their addresses with & (ComPtr exposes the raw double-pointer these out-parameters need).

ComPtr<ID3D11Device>        device;
ComPtr<ID3D11DeviceContext> context;
ComPtr<IDXGISwapChain>      swapChain;

UINT flags = 0;
#if defined(_DEBUG)
flags |= D3D11_CREATE_DEVICE_DEBUG;   // extra validation messages while developing
#endif

HRESULT hr = D3D11CreateDeviceAndSwapChain(
    nullptr,                      // default adapter (the primary GPU)
    D3D_DRIVER_TYPE_HARDWARE,     // use the real GPU, not a software renderer
    nullptr,                      // no software module (only for D3D_DRIVER_TYPE_SOFTWARE)
    flags,                        // creation flags (debug layer in debug builds)
    requested,                    // our feature-level wish list
    ARRAYSIZE(requested),         // how many entries it has
    D3D11_SDK_VERSION,            // always this constant
    &scd,                         // the swap chain descriptor from above
    &swapChain,                   // out: IDXGISwapChain
    &device,                      // out: ID3D11Device
    &obtained,                    // out: the feature level we actually got
    &context);                    // out: ID3D11DeviceContext

if (FAILED(hr)) {
    // In a real engine, log hr and bail out cleanly.
    return false;
}
Check every HRESULT

Almost every Direct3D function returns an HRESULT. Use the SUCCEEDED(hr) and FAILED(hr) macros — a negative code means failure. The debug layer, enabled by D3D11_CREATE_DEVICE_DEBUG, prints a detailed reason to the Visual Studio Output window, which turns baffling failures into readable messages. It requires the Graphics Tools optional feature installed on Windows.

From back buffer to render target view

The swap chain owns the back buffer texture, but you cannot draw into a raw texture directly. Direct3D wants a view — a small object that says "treat this resource as a render target." So we ask the swap chain for buffer index 0 (the back buffer), then create an ID3D11RenderTargetView from it.

ComPtr<ID3D11Texture2D>        backBuffer;
ComPtr<ID3D11RenderTargetView> renderTargetView;

// GetBuffer(0, ...) hands us the back buffer as a texture.
hr = swapChain->GetBuffer(0, IID_PPV_ARGS(&backBuffer));
if (FAILED(hr)) return false;

// Wrap it in a view we can bind as an output.
hr = device->CreateRenderTargetView(
    backBuffer.Get(),   // the resource
    nullptr,            // nullptr = describe the view like the resource itself
    &renderTargetView);
if (FAILED(hr)) return false;

// backBuffer's ComPtr can go out of scope now; the view keeps the
// texture alive through COM reference counting.

IID_PPV_ARGS is a small helper that fills in both the interface ID and the output pointer for you, so the requested type and the pointer can never get out of sync. The .Get() method returns the raw pointer that the C-style API expects, without transferring ownership.

That render target view is the object we bind to the output-merger stage before drawing. In the next lesson we will hand it to the context and use it to clear the screen to a solid color — your first visible sign of life from the GPU.

Where this lives in the engine

Wrap these five members — device, context, swap chain, back buffer and render target view — in a Renderer class, and put this whole sequence in an Initialize(HWND hwnd) method. Because every member is a ComPtr, the destructor cleans up automatically. That class becomes the foundation the rest of our engine renders through.

Recap
  • Every D3D11 app owns a device (creates resources), a device context (issues commands) and a swap chain (presents to the window).
  • Fill a zero-initialized DXGI_SWAP_CHAIN_DESC: set the format, sample count, DXGI_USAGE_RENDER_TARGET_OUTPUT, BufferCount, OutputWindow = hwnd and Windowed = TRUE.
  • D3D11CreateDeviceAndSwapChain builds all three at once and reports the feature level you actually got; always check the returned HRESULT.
  • Call GetBuffer(0, ...) for the back buffer, then CreateRenderTargetView to get a view you can render into.
  • Link d3d11.lib and store every object in a Microsoft::WRL::ComPtr so lifetimes are managed for you.
