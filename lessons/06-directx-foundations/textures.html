Textures & Sampling

So far our triangles have been painted with flat vertex colors. Real engines wrap geometry in images — brick walls, character skins, UI icons. In this lesson we teach the GPU to read pixels from a texture and paint them onto our mesh. This is the exact machinery Unity and Unreal use for every material you have ever seen, and it is the foundation for the material system we will build later.

What you'll learn
  • What UV texture coordinates are and how to add them to your Vertex.
  • How to create an ID3D11Texture2D and wrap it in an ID3D11ShaderResourceView.
  • How a sampler state controls filtering and address (wrap/clamp) modes.
  • How to sample a texture in HLSL with tex.Sample(samp, uv).
  • How to bind both to the pixel shader with PSSetShaderResources and PSSetSamplers.

UV coordinates: a map from mesh to image

A texture is just a 2D grid of pixels. To decide which pixel lands on which part of a triangle, every vertex carries a texture coordinate — a 2D point usually written (u, v). By convention (0, 0) is the top-left corner of the image and (1, 1) is the bottom-right, regardless of the texture's actual resolution. The GPU interpolates these UVs across the triangle, exactly the way it interpolated colors in earlier lessons, so every pixel gets its own coordinate into the image.

Let's extend the vertex layout we built for our mesh. We add a two-float UV field and describe it to Direct3D with a new input element.

#include <DirectXMath.h>

struct Vertex
{
    DirectX::XMFLOAT3 position; // x, y, z
    DirectX::XMFLOAT2 uv;       // u, v  (new!)
};

// Describe the layout to the Input Assembler.
// The offsets must match the struct exactly.
D3D11_INPUT_ELEMENT_DESC layout[] =
{
    { "POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT, 0, 0,
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
    // 12 bytes = the three position floats come first.
    { "TEXCOORD", 0, DXGI_FORMAT_R32G32_FLOAT,    0, 12,
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
};
Offsets must match the struct

The 12 in the TEXCOORD element is the byte offset of uv — three 4-byte floats of position. If your struct and layout disagree, the GPU reads garbage and your texture appears scrambled. You can pass D3D11_APPEND_ALIGNED_ELEMENT instead of a hard-coded offset to let Direct3D compute it for you.

Creating the texture

A texture on the GPU is an ID3D11Texture2D. In a shipping engine you would load its pixels from a file — the WIC API (via DirectXTK's CreateWICTextureFromFile) decodes PNG/JPG, and a DDS loader (CreateDDSTextureFromFile) reads pre-compressed textures with mipmaps. Both helpers hand you the texture and its view in one call. To keep this lesson dependency-free and to make the data crystal clear, we will instead generate a tiny procedural checkerboard in memory and upload it ourselves.

#include <d3d11.h>
#include <vector>
#include <cstdint>
#include <wrl/client.h>

using Microsoft::WRL::ComPtr;

// Build an 8x8 RGBA checkerboard entirely in CPU memory.
static std::vector<uint32_t> MakeCheckerboard()
{
    const int size = 8;
    std::vector<uint32_t> pixels(size * size);
    for (int y = 0; y < size; ++y)
        for (int x = 0; x < size; ++x)
        {
            bool dark = ((x + y) & 1) != 0;
            // 0xAABBGGRR byte order for DXGI_FORMAT_R8G8B8A8_UNORM.
            pixels[y * size + x] = dark ? 0xFF404040u : 0xFFFFFFFFu;
        }
    return pixels;
}

// Returns a shader resource view ready to bind to the pixel shader.
ComPtr<ID3D11ShaderResourceView> CreateCheckerboardSRV(ID3D11Device* device)
{
    const UINT size = 8;
    std::vector<uint32_t> pixels = MakeCheckerboard();

    D3D11_TEXTURE2D_DESC desc = {};
    desc.Width          = size;
    desc.Height         = size;
    desc.MipLevels      = 1;                 // one level, no mipmaps
    desc.ArraySize      = 1;
    desc.Format         = DXGI_FORMAT_R8G8B8A8_UNORM;
    desc.SampleDesc.Count = 1;               // no MSAA on the texture
    desc.Usage          = D3D11_USAGE_IMMUTABLE;
    desc.BindFlags      = D3D11_BIND_SHADER_RESOURCE;

    // Point the initial data at our pixel buffer.
    D3D11_SUBRESOURCE_DATA init = {};
    init.pSysMem     = pixels.data();
    init.SysMemPitch = size * sizeof(uint32_t); // bytes per row

    ComPtr<ID3D11Texture2D> texture;
    HRESULT hr = device->CreateTexture2D(&desc, &init, &texture);
    if (FAILED(hr)) return nullptr;

    // A shader resource view is the handle shaders actually read through.
    D3D11_SHADER_RESOURCE_VIEW_DESC srvDesc = {};
    srvDesc.Format              = desc.Format;
    srvDesc.ViewDimension       = D3D11_SRV_DIMENSION_TEXTURE2D;
    srvDesc.Texture2D.MipLevels = 1;

    ComPtr<ID3D11ShaderResourceView> srv;
    hr = device->CreateShaderResourceView(texture.Get(), &srvDesc, &srv);
    if (FAILED(hr)) return nullptr;

    return srv;
}
Texture vs. view

The ID3D11Texture2D owns the pixel memory; the ID3D11ShaderResourceView (SRV) is the read-only handle a shader binds to. This split lets the same resource be viewed different ways — as a render target, a depth buffer, or a shader input. Shaders always read through a view, never the resource directly.

SysMemPitch is the number of bytes in one row of the source image. For our 8-pixel-wide RGBA texture that is 8 * 4 = 32 bytes. Getting the pitch wrong is a classic cause of skewed or diagonally-smeared textures.

The sampler: how pixels are read

The texture holds the data, but how the GPU reads it is a separate object: an ID3D11SamplerState. It answers two questions. First, filtering: when a UV lands between texels, do we snap to the nearest one (crisp, blocky) or blend neighbors (smooth)? Second, addressing: what happens when a UV goes outside the 0..1 range — wrap (tile), clamp, or mirror?

SettingValue we useEffect
FilterMIN_MAG_MIP_LINEARSmoothly blends texels when magnified or minified.
AddressU / VWRAPUVs beyond 1.0 tile the image — great for repeating brick.
Alternative filterMIN_MAG_MIP_POINTNearest-texel; keeps our checkerboard crisp and pixel-art sharp.
// Create a sampler that blends texels and tiles out-of-range UVs.
ComPtr<ID3D11SamplerState> CreateSampler(ID3D11Device* device)
{
    D3D11_SAMPLER_DESC desc = {};
    desc.Filter   = D3D11_FILTER_MIN_MAG_MIP_LINEAR; // smooth blending
    desc.AddressU = D3D11_TEXTURE_ADDRESS_WRAP;
    desc.AddressV = D3D11_TEXTURE_ADDRESS_WRAP;
    desc.AddressW = D3D11_TEXTURE_ADDRESS_WRAP;
    desc.ComparisonFunc = D3D11_COMPARISON_NEVER;
    desc.MinLOD   = 0;
    desc.MaxLOD   = D3D11_FLOAT32_MAX;

    ComPtr<ID3D11SamplerState> sampler;
    HRESULT hr = device->CreateSamplerState(&desc, &sampler);
    return SUCCEEDED(hr) ? sampler : nullptr;
}
Samplers are reusable

You typically create a handful of samplers at startup — one linear-wrap, one point-clamp, maybe an anisotropic one — and share them across every draw call. They are cheap, immutable state objects. Our engine's material system will let each material pick which shared sampler it wants.

Sampling in HLSL

Now the shader side. The pixel shader declares two new global objects — a Texture2D and a SamplerState — each bound to a register. The vertex shader simply passes the interpolated UV straight through.

// ---- Vertex shader ----
struct VSInput
{
    float3 position : POSITION;
    float2 uv       : TEXCOORD0;
};

struct PSInput
{
    float4 position : SV_POSITION; // clip-space, required
    float2 uv       : TEXCOORD0;   // interpolated across the triangle
};

PSInput VSMain(VSInput input)
{
    PSInput output;
    output.position = float4(input.position, 1.0f);
    output.uv       = input.uv; // hand the UV to the pixel shader
    return output;
}

// ---- Pixel shader ----
Texture2D    gTexture : register(t0); // matches SRV slot 0
SamplerState gSampler : register(s0); // matches sampler slot 0

float4 PSMain(PSInput input) : SV_TARGET
{
    // Read the color at this pixel's UV using our sampling rules.
    return gTexture.Sample(gSampler, input.uv);
}

The register letters matter: t0 is texture (SRV) slot 0, and s0 is sampler slot 0. These must line up with the C++ slots you bind to. In the next lesson we will multiply the position by a camera matrix here, but the texturing logic stays identical.

Binding before the draw call

Each frame, before DrawIndexed, we tell the pixel shader stage which SRV and which sampler to use. The PS prefix means "pixel shader stage"; there are matching VS/CS versions for other stages.

// context is an ID3D11DeviceContext*, created once at startup.
ID3D11ShaderResourceView* srvs[]     = { checkerSRV.Get() };
ID3D11SamplerState*        samplers[] = { sampler.Get() };

// Bind to slots starting at 0 -> matches register(t0) / register(s0).
context->PSSetShaderResources(0, 1, srvs);
context->PSSetSamplers(0, 1, samplers);

// ... set vertex/index buffers, shaders, input layout ...
context->DrawIndexed(indexCount, 0, 0);
Black geometry? Check these first

If your mesh renders solid black, the usual culprits are: you forgot to call PSSetShaderResources/PSSetSamplers, the register slots in HLSL don't match the C++ binding slots, or the UVs in your vertex buffer are all zero. Turn on the D3D11 debug layer — it prints a warning when a shader samples an unbound texture.

That is the complete pipeline: UVs travel from the vertex buffer, through the rasterizer, into the pixel shader, where a sampler reads the bound texture and returns a color. Swap the checkerboard for a WIC-loaded PNG and you have a real textured material — the same building block every asset in a modern game is built on.

Recap
  • UV coordinates (0,0)–(1,1) map each vertex to a spot in the image; add a XMFLOAT2 uv to your vertex and a TEXCOORD input element.
  • An ID3D11Texture2D holds the pixels; you read it through an ID3D11ShaderResourceView. Load real images with WIC or a DDS loader, or upload data yourself via D3D11_SUBRESOURCE_DATA.
  • An ID3D11SamplerState controls filtering (point vs. linear) and addressing (wrap/clamp/mirror), and is shared across draws.
  • In HLSL, declare Texture2D at register(t0) and SamplerState at register(s0), then call tex.Sample(samp, uv).
  • Bind them each frame with PSSetShaderResources and PSSetSamplers, matching the register slots.
