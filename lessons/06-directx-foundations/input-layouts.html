Input Layouts & Vertex Formats

In the last lesson you wrote your first HLSL shaders and compiled them to bytecode. But the GPU still has no idea what the bytes in your vertex buffer mean. An input layout is the contract that maps raw vertex bytes to the inputs your vertex shader expects. Get it right and your triangle appears; get it wrong and you get garbage, a blank screen, or a crash. This is one of the last pieces we need before our engine can draw its first real geometry.

What you'll learn
  • Why the GPU needs an input layout to interpret raw vertex bytes.
  • How to describe each field with a D3D11_INPUT_ELEMENT_DESC and pick the right DXGI_FORMAT.
  • How byte offsets work, and why D3D11_APPEND_ALIGNED_ELEMENT saves you from arithmetic mistakes.
  • How to create the layout from your vertex shader bytecode and bind it with IASetInputLayout.

The problem: bytes have no meaning

A vertex buffer on the GPU is just a flat block of memory. Suppose each vertex carries a 3D position and an RGBA color. In C++ we model that with a plain struct:

#include <DirectXMath.h>

// One vertex: a position in space, plus a color.
struct Vertex
{
    DirectX::XMFLOAT3 position;  // 3 floats = 12 bytes
    DirectX::XMFLOAT4 color;     // 4 floats = 16 bytes
};                               // total stride = 28 bytes

When you upload an array of these into a vertex buffer, the GPU sees only 28 bytes per vertex repeated over and over. It does not know that the first 12 bytes are a position or that the next 16 are a color. The input layout is how we tell it.

On the shader side, your vertex shader declares what it wants using semantics — the string tags you met last lesson:

struct VSInput
{
    float3 position : POSITION;   // wants a POSITION input
    float4 color    : COLOR;      // wants a COLOR input
};

struct VSOutput
{
    float4 position : SV_POSITION;
    float4 color    : COLOR;
};

VSOutput main(VSInput input)
{
    VSOutput output;
    output.position = float4(input.position, 1.0f);
    output.color    = input.color;
    return output;
}

The input layout is the bridge between these two worlds: it says "the bytes labelled POSITION live at offset 0 and are three 32-bit floats; the bytes labelled COLOR live at offset 12 and are four 32-bit floats."

Semantics are the matching key

The layout and the shader are matched by semantic name (like POSITION), not by order. The names must agree exactly, though they are case-insensitive.

Describing each field: D3D11_INPUT_ELEMENT_DESC

You describe the layout as an array of D3D11_INPUT_ELEMENT_DESC — one entry per field in your Vertex struct. Here is the full struct with what each member means:

FieldMeaning
SemanticNameMust match the shader semantic, e.g. "POSITION".
SemanticIndexDistinguishes duplicates like TEXCOORD0 vs TEXCOORD1. Usually 0.
FormatThe DXGI_FORMAT describing how many components and their bit size/type.
InputSlotWhich bound vertex buffer this field comes from. Usually 0.
AlignedByteOffsetByte offset of this field within one vertex. Or D3D11_APPEND_ALIGNED_ELEMENT.
InputSlotClassPER_VERTEX_DATA for normal geometry (vs per-instance data).
InstanceDataStepRate0 for per-vertex data.

Choosing the right DXGI_FORMAT

The Format tells the GPU how many components a field has and how each is stored. It has nothing to do with color specifically — R, G, B, A are just names for "channel 1, 2, 3, 4". A three-float position and a four-float color map like this:

C++ typeSemanticDXGI_FORMATBytes
XMFLOAT3POSITIONDXGI_FORMAT_R32G32B32_FLOAT12
XMFLOAT4COLORDXGI_FORMAT_R32G32B32A32_FLOAT16
XMFLOAT2TEXCOORDDXGI_FORMAT_R32G32_FLOAT8

Read the format name literally: R32G32B32_FLOAT means "three channels, each a 32-bit float." That is exactly an XMFLOAT3.

Offsets, by hand or automatically

The AlignedByteOffset is where the field starts inside one vertex. For our struct, position starts at byte 0 and color starts at byte 12. You can write those numbers yourself — but the moment you reorder or insert a field, every following offset must change, and a wrong number silently misreads memory. Instead, use the special constant D3D11_APPEND_ALIGNED_ELEMENT, which tells Direct3D to compute the offset by appending directly after the previous element:

// Describe the layout: one entry per field in Vertex.
// D3D11_APPEND_ALIGNED_ELEMENT auto-computes the byte offset.
D3D11_INPUT_ELEMENT_DESC layoutDesc[] =
{
    { "POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT,    0, 0,
      D3D11_INPUT_PER_VERTEX_DATA, 0 },

    { "COLOR",    0, DXGI_FORMAT_R32G32B32A32_FLOAT, 0,
      D3D11_APPEND_ALIGNED_ELEMENT,
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
};

The first element uses offset 0 explicitly. The second uses D3D11_APPEND_ALIGNED_ELEMENT, so Direct3D fills in 12 for us. When we later add normals or texture coordinates to our engine's vertex format, we just insert an entry and the offsets stay correct automatically.

Creating the layout from shader bytecode

Here is the part people often find surprising: CreateInputLayout needs your compiled vertex shader bytecode. Why? Because Direct3D validates your layout against the shader's actual input signature at creation time. If your layout promises a COLOR but the shader never declares one, or the formats are incompatible, creation fails right there instead of drawing garbage later.

Assume from last lesson you have the VS blob (vsBlob) from D3DCompileFromFile or a loaded .cso file, and an ID3D11Device. We use a Microsoft::WRL::ComPtr to manage the COM object's lifetime, as we set up earlier in this module:

#include <d3d11.h>
#include <wrl/client.h>
using Microsoft::WRL::ComPtr;

ComPtr<ID3D11InputLayout> inputLayout;

HRESULT hr = device->CreateInputLayout(
    layoutDesc,                       // our element descriptions
    ARRAYSIZE(layoutDesc),            // how many elements (2 here)
    vsBlob->GetBufferPointer(),       // the compiled VS bytecode...
    vsBlob->GetBufferSize(),          // ...and its size
    inputLayout.GetAddressOf());      // receives the created layout

if (FAILED(hr))
{
    // Almost always a mismatch between layoutDesc and the shader's inputs.
    // Check semantic names and formats against your VSInput struct.
    return hr;
}
Use the right shader's bytecode

CreateInputLayout must be given the vertex shader blob — not the pixel shader. The layout is validated against the vertex shader's input signature only. Passing the wrong blob is a common cause of a failing HRESULT.

Binding it before you draw

Creating the layout once at startup is not enough — you must bind it to the Input Assembler (IA) stage before each draw, alongside the vertex buffer and primitive topology. The layout, the buffer's stride, and the topology all work together to feed vertices to your shader:

// Per-frame draw setup on the immediate context.
UINT stride = sizeof(Vertex);   // 28 bytes — MUST match the layout
UINT offset = 0;

context->IASetInputLayout(inputLayout.Get());
context->IASetVertexBuffers(0, 1, vertexBuffer.GetAddressOf(),
                            &stride, &offset);
context->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

// Bind the shaders we compiled last lesson.
context->VSSetShader(vertexShader.Get(), nullptr, 0);
context->PSSetShader(pixelShader.Get(), nullptr, 0);

// Draw our 3 vertices as one triangle.
context->Draw(3, 0);
Stride must equal your struct size

The stride you pass to IASetVertexBuffers tells the IA how many bytes to step per vertex. Always use sizeof(Vertex) so it can never drift out of sync with your struct. A wrong stride reads each vertex from the wrong place and warps your geometry.

Why this matters for our engine

Right now we have a single hard-coded vertex format. But a real engine draws meshes with positions, normals, texture coordinates, and more. Every one of those formats needs a matching input layout, and each layout is tied to a shader signature. When we build our Mesh and Material components later, the input layout will be created and cached next to the shader it belongs to — so a mesh always knows exactly how to describe its own vertices to the GPU. The pattern you learned here scales directly to that design.

A blank screen is usually a mismatch

If your triangle does not appear, suspect the layout first. Enable the D3D11 debug layer and read the output window — it will tell you plainly when a semantic name, format, or stride disagrees with your shader. Debugging silently is far harder than reading its warnings.

Recap
  • A vertex buffer is just raw bytes; the input layout tells the GPU how to interpret each vertex and connect it to your shader.
  • You describe each field with a D3D11_INPUT_ELEMENT_DESC, matching by semantic name and choosing a DXGI_FORMAT like R32G32B32_FLOAT for a position or R32G32B32A32_FLOAT for a color.
  • Use D3D11_APPEND_ALIGNED_ELEMENT so byte offsets are computed automatically and stay correct when the format changes.
  • CreateInputLayout validates the layout against your compiled vertex shader bytecode, catching mismatches early.
  • Bind with IASetInputLayout and pass sizeof(Vertex) as the stride before you Draw. Next up: feeding transforms into shaders with constant buffers.
