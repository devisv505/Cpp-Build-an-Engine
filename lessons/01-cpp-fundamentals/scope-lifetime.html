Scope, Lifetime & the Stack

You already know how to write functions and pass values into them. But where can a variable be used, and how long does it stay alive? Answering that is the difference between confident C++ and mysterious crashes. In this lesson we map out scope, the call stack, and automatic lifetime — the same rules that will govern every local variable in our engine's update loop.

What you'll learn
  • The difference between local, global, and block scope
  • How the call stack stores each function call in its own stack frame
  • What automatic storage duration means for a variable's lifetime
  • How variable shadowing works and why it can bite you
  • Why we'll eventually need the heap for data that must outlive its scope

What is scope?

Scope is the region of your source code where a name is visible and usable. If you try to use a variable outside its scope, the compiler simply doesn't know what you mean — the name doesn't exist there. C++ decides scope by looking at the pair of curly braces { } that surround a declaration.

There are three kinds you'll meet constantly:

  • Local scope — a variable declared inside a function. It exists only within that function.
  • Block scope — a variable declared inside any { } block (an if, a for loop, or a bare block). It exists only inside that block.
  • Global scope — a variable declared outside every function. It's visible everywhere in the file after its declaration.
#include <iostream>

int g_frameCount = 0;   // global scope: visible to every function below

void tick() {
    int local = 5;      // local scope: only exists inside tick()
    g_frameCount++;     // fine — the global is visible here

    if (local > 3) {
        int bonus = 10; // block scope: only exists inside this if-block
        std::cout << local + bonus << "\n";
    }
    // std::cout << bonus; // ERROR: 'bonus' is out of scope here
}

int main() {
    tick();
    // std::cout << local; // ERROR: 'local' belongs to tick(), not main()
    std::cout << g_frameCount << "\n";
    return 0;
}
Avoid globals when you can

Globals like g_frameCount are visible everywhere, which sounds convenient but makes code hard to reason about — any function can change them at any time. In our engine we'll prefer passing data explicitly or storing it in objects. Reach for a global only for genuinely shared, rarely-changing state.

Block scope in detail

Every pair of braces creates a fresh scope. This includes loops, where a variable declared in the loop header lives only for the duration of the loop:

for (int i = 0; i < 3; ++i) {
    int square = i * i;   // 'square' is created fresh each iteration
    std::cout << square << " ";
}
// both 'i' and 'square' are gone here — the loop's scope ended

You can even open a bare block just to limit a variable's reach. This is a good habit: the tighter a variable's scope, the fewer places it can be misused, and the easier your code is to read.

The call stack and stack frames

When your program calls a function, C++ needs somewhere to keep that function's local variables. It uses a region of memory called the stack. Each function call gets its own slice of the stack — a stack frame — holding its parameters and local variables.

Frames stack up like a pile of plates. When main calls update, and update calls length, three frames are live at once. As each function returns, its frame is popped off the top, and its locals vanish instantly. This last-on, first-off behaviour is exactly why it's called a stack.

#include <iostream>
#include <cmath>

// A tiny helper — imagine this is a method on our future Vector2 type.
float length(float x, float y) {
    float sum = x * x + y * y;   // 'sum' lives in length()'s frame
    return std::sqrt(sum);
}

void update(float dt) {
    float speed = length(3.0f, 4.0f);  // pushes a length() frame, then pops it
    std::cout << "speed=" << speed << " dt=" << dt << "\n";
}

int main() {
    update(0.016f);   // ~one frame at 60 FPS
    return 0;         // main()'s frame is the last to pop
}
The stack is fast and finite

Pushing and popping frames is extremely cheap — it's just moving a pointer. That speed is why locals are the default. But the stack is small (often a megabyte or so), which is why very deep recursion or huge local arrays can cause a stack overflow. Big or long-lived data belongs on the heap instead.

Automatic storage duration and lifetime

Lifetime is when a variable exists at runtime, as opposed to scope, which is where its name is visible in the code. Local variables have automatic storage duration: they are automatically created when execution reaches their declaration, and automatically destroyed when their enclosing block ends. You don't manage this by hand — hence "automatic."

This automatic cleanup is one of C++'s best features. When an object with a destructor goes out of scope, its destructor runs right then, releasing whatever it held. We'll lean on this heavily later for smart pointers and resource-owning classes.

Never return the address of a local

Because a local dies when its function returns, returning a pointer or reference to it hands the caller a dangling reference to memory that no longer belongs to anyone. Reading it is undefined behaviour — it may look fine, then corrupt data later.

int* makeBad() {
    int value = 42;
    return &value;   // BUG: 'value' is destroyed the moment we return
}                    // the returned pointer is now dangling

int makeGood() {
    int value = 42;
    return value;    // fine — we return a COPY of the value, not its address
}

Variable shadowing

When an inner scope declares a variable with the same name as one in an outer scope, the inner name shadows the outer one. Inside the inner block, the name refers to the inner variable; the outer one is hidden but not destroyed — it comes back into view once the inner scope ends.

#include <iostream>

int main() {
    int health = 100;
    std::cout << health << "\n";   // 100

    {
        int health = 30;          // shadows the outer 'health'
        std::cout << health << "\n"; // 30 — the inner one
    }

    std::cout << health << "\n";   // 100 again — outer is back in view
    return 0;
}
Turn on shadowing warnings

Accidental shadowing is a classic source of "why isn't my value changing?" bugs. Enable -Wshadow with bundled MinGW on Windows or Apple Clang on macOS so the compiler flags it. In our engine we'll keep member variables and locals clearly named — a common convention is an m_ prefix for members — to avoid ambiguity entirely.

Looking ahead: the heap

Automatic, stack-based storage is perfect for short-lived values, but it has a hard limit: a local cannot outlive the function that created it. Yet a real engine is full of data that must persist across many frames — the list of every entity in a scene, textures loaded from disk, meshes uploaded to the GPU. That data needs a different home: the heap, where memory lives until we explicitly (or via smart pointers) release it.

We'll dive into the heap, new/delete, and smart pointers in the memory module. For now, the key mental model is this: stack = automatic and scope-bound; heap = manual and long-lived. Knowing which one a piece of data belongs on is a skill you'll use in every system we build.

Recap
  • Scope is where a name is visible; it's defined by { } braces — local, block, or global.
  • Each function call gets a stack frame on the call stack, holding its parameters and locals; frames pop off when the function returns.
  • Locals have automatic storage duration — created at their declaration, destroyed when their block ends.
  • Returning the address of a local produces a dangling pointer — never do it.
  • An inner variable can shadow an outer one of the same name; enable compiler warnings to catch accidental cases.
  • Data that must outlive its scope belongs on the heap, which we'll cover in the memory module.
