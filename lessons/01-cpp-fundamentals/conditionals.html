Decisions: if, else & switch

So far your programs have run straight from top to bottom, doing exactly the same thing every time. Real software — and especially a game engine — has to react: is the player alive, is a key pressed, did the ray hit anything? In this lesson you'll teach C++ to choose between different paths using if, else if, else, the switch statement, and the compact ternary operator. These branches are the tiny gates that later drive our game loop, input handling, and component logic.

What you'll learn
  • Write if / else if / else chains driven by boolean conditions
  • Combine and nest conditions cleanly, and read them the way the compiler does
  • Use switch / case / break / default — and dodge the classic fallthrough bug
  • Reach for the ternary ?: operator when a full if is overkill

Boolean conditions: the fuel for every branch

Every decision in C++ boils down to a single bool — a value that is either true or false. You already met the comparison and logical operators in the previous lesson; now we put them to work. Expressions like hp <= 0, score > best, or isGrounded && jumpPressed all evaluate to a bool, and that's exactly what a branch needs.

The simplest branch is a lone if. If the condition is true, the block in braces runs; otherwise it's skipped entirely.

#include <iostream>

int main() {
    int playerHealth = 30;

    // The condition is a bool expression. Only runs when it's true.
    if (playerHealth < 50) {
        std::cout << "Warning: health is low!\n";
    }

    std::cout << "Health = " << playerHealth << "\n";
    return 0;
}
Always use braces

C++ lets you skip the { } when a branch has a single statement, but don't. Braces prevent a whole family of bugs (like the infamous "goto fail" security hole) and keep diffs clean when you add a second line later. In this course, every branch gets braces.

Choosing between paths: else and else if

An if on its own answers a yes/no question. Add else to handle the "no" case, and chain else if to test several conditions in order. C++ checks them top to bottom and runs the first block whose condition is true — then skips the rest.

#include <iostream>

int main() {
    int hp = 62;

    if (hp <= 0) {
        std::cout << "Status: dead\n";
    } else if (hp < 25) {
        std::cout << "Status: critical\n";
    } else if (hp < 60) {
        std::cout << "Status: hurt\n";
    } else {
        std::cout << "Status: healthy\n";   // hp is 60 or more
    }

    return 0;
}

Order matters. Because C++ stops at the first match, you go from the most specific or most urgent case to the most general. If you flipped the checks so hp < 60 came first, a health of 10 would print "hurt" and never reach the "critical" branch. The final else is your catch-all: it runs when nothing above matched.

= is not ==

Writing if (hp = 0) assigns 0 to hp and then tests the result (0, which is false). You almost always meant ==. Modern compilers warn about this — turn warnings on and treat them as errors.

Nesting and combining conditions

You can place an if inside another if to express "this, and then within that, that". Often, though, a nested check reads more clearly when you flatten it with the logical operators && (and) and || (or).

bool isGrounded   = true;
bool jumpPressed  = true;
int  stamina      = 15;

// Nested version
if (isGrounded) {
    if (jumpPressed && stamina > 10) {
        // perform jump
    }
}

// Flatter, equivalent version — usually easier to read
if (isGrounded && jumpPressed && stamina > 10) {
    // perform jump
}

Both forms behave the same here. Reach for nesting when the inner and outer conditions need genuinely different else handling; otherwise, prefer the flat combined condition. One more thing worth knowing: && and || are short-circuit operators. In a && b, if a is false, C++ never even evaluates b. We'll lean on that later to safely write things like if (ptr != nullptr && ptr->IsActive()).

switch: branching on one value

When you're comparing a single integer-like value against many fixed constants, a long else if chain gets noisy. The switch statement expresses that intent directly. It's perfect for enum-driven state — exactly the kind of thing an engine has plenty of, such as a game state or an input action.

#include <iostream>

// An enum gives our states readable names instead of magic numbers.
enum class GameState { MainMenu, Playing, Paused, GameOver };

void update(GameState state) {
    switch (state) {
        case GameState::MainMenu:
            std::cout << "Showing the main menu\n";
            break;                              // stop here — don't fall through

        case GameState::Playing:
            std::cout << "Simulating the world\n";
            break;

        case GameState::Paused:
            std::cout << "Game paused\n";
            break;

        default:                                // anything we didn't list
            std::cout << "Game over screen\n";
            break;
    }
}

int main() {
    update(GameState::Playing);
    return 0;
}

The value in switch (state) is compared against each case label. When one matches, execution jumps there and runs until it hits a break. The default label catches any value that no case handled — think of it as the switch version of a trailing else. A switch only works on integral or enum types (and char), not on float or std::string.

Forgetting break = fallthrough

If you omit break, execution "falls through" into the next case and keeps running. Leaving out the break after Playing above would print both "Simulating the world" and "Game paused". This is a real, silent bug source — make break a reflex on every case.

Intentional fallthrough

Occasionally you want several cases to share one block — for example, treating two keys the same way. Stacking labels with no code between them is legal and clear, and it isn't the dangerous kind of fallthrough:

switch (key) {
    case 'w':
    case 'W':                 // both 'w' and 'W' run the same code
        moveForward();
        break;

    case 's':
    case 'S':
        moveBackward();
        break;
}
Say what you mean

If you deliberately fall through after running some code, mark it with the C++17 attribute [[fallthrough]]; on its own line. It documents intent for readers and silences compiler warnings, so accidental fallthrough still gets flagged.

if vs. switch: which to use

Use if / else if when…Use switch when…
Conditions involve ranges (hp < 25) or comparisonsYou test one value against fixed constants
You combine several variables with && / ||The value is an int, char, or enum
Only two or three outcomesThere are many discrete cases (states, key codes)

The ternary operator: a decision as a value

Sometimes an if just picks one of two values to store or print. The ternary operator condition ? a : b does that in a single expression: it evaluates to a when the condition is true, otherwise b.

#include <iostream>
#include <string>

int main() {
    int hp = 0;

    // Reads as: "hp > 0 ? then 'Alive' : otherwise 'Dead'"
    std::string label = (hp > 0) ? "Alive" : "Dead";
    std::cout << "Player is " << label << "\n";

    // Handy inline, too — clamp a value without a temporary variable.
    int lives = 3;
    std::cout << "Lives: " << (lives > 0 ? lives : 0) << "\n";
    return 0;
}

Keep ternaries short and readable — they shine for simple either/or choices. If either branch needs several statements or the logic starts to nest, go back to a full if. Clarity beats cleverness, especially in engine code you'll be re-reading for months.

Where this goes next

Branches are how our engine will decide whether an entity is active before updating it, whether a collision actually happened, or which render path to take. Combined with the loops in the next lesson, they form the beating heart of the game loop.

Recap
  • Every branch is driven by a bool condition; if / else if / else run the first block that's true and skip the rest, so order matters.
  • Always use braces, and remember == compares while = assigns.
  • Combine conditions with short-circuiting && and ||; flatten nested ifs when it reads more clearly.
  • switch is ideal for one value against many constant cases — but forgetting break causes silent fallthrough.
  • The ternary ?: picks between two values in a single expression; keep it simple.
