Constants & Console I/O

In the previous lesson you gave data a home with variables and fundamental types. Now we make two upgrades: values that are promised never to change (const and constexpr), and a way to talk to the outside world through the console. Together these let us build our first genuinely interactive program — the same input/output loop your future engine will use to load settings and report what it is doing.

What you'll learn
  • Declare values that cannot change with const, and compile-time constants with constexpr
  • Recognise the common kinds of literals and what type each one has
  • Print text and values to the screen with std::cout
  • Read a number and a line of text from the user with std::cin and std::getline
  • Choose between std::endl and "\n" and know why it matters

Values that never change: const

A regular variable can be reassigned at any time. Often that is exactly what you do not want. If a value represents a fixed fact — the number of milliseconds in a second, the maximum number of lights your renderer supports — you should say so in the code. The keyword const makes a variable read-only after it is initialised.

#include <iostream>

int main() {
    const int maxLights = 8;   // read-only from here on
    std::cout << "Max lights: " << maxLights << "\n";

    // maxLights = 16;  // ERROR: assignment of read-only variable
    return 0;
}

The commented line would refuse to compile. That is a feature, not an annoyance: the compiler now guards your intent. Anyone reading the code — including you in six months — knows this value is a fixed rule, and any accidental attempt to change it is caught immediately instead of becoming a hard-to-find bug.

Const by default

A good habit: reach for const first, and only drop it when you actually need to change the value. Code full of clearly-constant values is far easier to reason about.

Compile-time constants: constexpr

A const value must never change once set, but it might be computed while the program runs. Sometimes we want more: a value the compiler can figure out before the program even runs. That is what constexpr guarantees — a constant known at compile time.

constexpr int targetFps   = 60;              // known while compiling
constexpr double frameTime = 1.0 / targetFps; // computed at compile time too

// Because it is a compile-time constant, it can size a fixed array:
int frameHistory[targetFps]; // 60 slots, decided by the compiler

Every constexpr is also implicitly const. The extra promise is when the value exists. This matters because some things — like the size of a plain fixed array, shown above — must be known at compile time. Later, when we design our engine, values such as the maximum number of components per entity will be constexpr so the compiler can lay out memory efficiently.

Literals: values written directly in code

A literal is a value you type straight into the source. Every literal has a type, and small suffixes let you steer that type.

  • 42 — an int literal.
  • 3.14 — a double literal; write 3.14f for a float.
  • 60u — an unsigned int; 10L is a long.
  • 'A' — a single char (single quotes).
  • "hello" — a string literal (double quotes).
  • true / false — bool literals.

You can also make long numbers readable with digit separators, and write values in other bases:

int   big   = 1'000'000;   // apostrophes are ignored, just for readability
int   hex   = 0xFF;        // 255, written in hexadecimal
int   binary = 0b1010;     // 10, written in binary
float gravity = 9.81f;     // the f suffix keeps it a float

Talking to the screen: std::cout

You have already met std::cout, the standard output stream. The << operator "sends" values into it, and you can chain as many as you like. Anything printable — text, numbers, the value of a variable — flows in left to right.

#include <iostream>

int main() {
    constexpr int fps = 60;
    std::cout << "Rendering at " << fps << " frames per second.\n";
    return 0;
}

std::endl vs "\n"

You will see two ways to end a line. Both move output to a new line, but they are not identical:

  • "\n" is simply a newline character in the text.
  • std::endl writes a newline and then flushes the buffer — it forces everything queued to be written out immediately.

Flushing has a cost. In a loop that runs thousands of times per second — like a render loop — flushing on every line will slow you down noticeably. Prefer "\n" for ordinary output, and save std::endl for the rare moment you truly need the text on screen right now (for example, just before a long pause).

Rule of thumb

Use "\n" almost everywhere. Only reach for std::endl when you have a specific reason to force an immediate flush.

Reading from the user: std::cin

Input mirrors output. The stream is std::cin, and the >> operator "extracts" a value from it into a variable. When you use >>, whitespace (spaces, tabs, newlines) separates one value from the next, so reading a number is simple:

int fps = 0;
std::cout << "Target FPS: ";
std::cin >> fps;   // waits for the user to type a number and press Enter

Reading whole words works the same way. But a name or a sentence contains spaces, and >> would stop at the first space. To capture a full line — spaces included — use std::getline with a std::string:

#include <string>

std::string engineName;
std::getline(std::cin, engineName); // reads everything up to Enter
The leftover newline trap

After std::cin >> number, the Enter key you pressed is still sitting in the input buffer. A following std::getline will read that empty leftover instead of waiting for you. The fix is to discard it first with std::cin.ignore(...), shown in the program below.

Putting it together: an interactive setup

Here is a complete, compilable program that greets the user, reads a name (which may contain spaces) and a target FPS, and echoes a small summary. It uses everything from this lesson: constexpr, literals, both streams, getline, and the newline discard.

#include <iostream>
#include <string>
#include <limits>

int main() {
    constexpr int   defaultFps = 60;          // compile-time constant
    constexpr double msPerSec  = 1000.0;      // for the frame-time math

    std::cout << "=== Engine Setup ===\n";

    // Read a full line, spaces allowed.
    std::cout << "Name your engine: ";
    std::string engineName;
    std::getline(std::cin, engineName);

    // Read a single number with the extraction operator.
    std::cout << "Target FPS (default " << defaultFps << "): ";
    int targetFps = 0;
    std::cin >> targetFps;

    // Guard against silly input using a plain const decision.
    const bool valid = (targetFps > 0);
    if (!valid) {
        targetFps = defaultFps;
        std::cout << "That wasn't a positive number - using the default.\n";
    }

    const double frameTimeMs = msPerSec / targetFps; // budget per frame

    std::cout << "\nEngine  : " << engineName << "\n";
    std::cout << "Running : " << targetFps << " FPS\n";
    std::cout << "Budget  : " << frameTimeMs << " ms per frame\n";
    return 0;
}

Compile and run it, then type an engine name and a number. At 60 FPS the program reports about 16.6667 ms per frame — the exact time budget a real render loop has to do all its work. That number will become very familiar as we build the engine, because staying inside that budget is what "60 FPS" really means.

Notice the order

We call getline before the numeric >>. Doing it this way sidesteps the leftover-newline trap entirely. When you need a number first and then a line, add std::cin.ignore(std::numeric_limits<std::streamsize>::max(), '\n'); between them — that is why we included <limits>.

Recap
  • const makes a value read-only after initialisation; constexpr additionally requires it to be known at compile time and is implicitly const.
  • Literals are values written in code (42, 3.14f, 'A', "text", true); suffixes and prefixes like f, u, 0x steer their type.
  • std::cout << ... prints values; std::cin >> ... reads whitespace-separated values into variables.
  • Use std::getline(std::cin, str) to read a whole line including spaces, and watch out for the leftover newline after >>.
  • Prefer "\n" over std::endl — endl also flushes, which is slower and rarely needed.
