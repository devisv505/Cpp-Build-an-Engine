Water, Transparency & Glow

Your world already renders — stone, dirt, grass, all crisply opaque. But water, glass and leaves break every assumption the opaque path makes: they let you see what sits behind them, so they cannot simply overwrite those pixels. And the glowing blocks — lava, glowstone, ore — need to do the opposite of hiding: they need to pour light into the scene. This lesson closes MiniEngine's voxel rendering story by adding a second, alpha-blended pass for see-through blocks, and by wiring bright blocks straight into the bloom pipeline you built back in Module 10.

What you'll learn
  • Why opaque and transparent geometry need different draw rules, and how to split each chunk into two meshes by the block's transparent flag.
  • How to build the alpha-blend and depth-write-off render state on Direct3D 11, and the Metal pipeline/depth equivalents.
  • How to order the frame: all opaque first, then transparent sorted far-to-near at chunk granularity — and why that order is non-negotiable.
  • How to give water a tint and a watertight animated wobble with a sum of sines in the vertex shader.
  • How to make lava, glowstone and ore emit HDR color above 1.0 so Module 10's bloom makes them glow for free.

Two kinds of geometry, two sets of rules

Everything you have drawn so far is opaque: each pixel a block covers is that block, full stop. Opaque geometry is wonderfully forgiving. Because it writes depth and every fragment fully replaces whatever was behind it, you can draw opaque blocks in any order at all — the depth buffer sorts them per-pixel for you, keeping the nearest surface and discarding the rest.

Transparent geometry throws that guarantee away. A pixel of water is part water and part whatever is behind it, mixed by the water's alpha. That mixing — alpha blending — reads the color already in the framebuffer, so what is behind the water must already have been drawn. And it must be composited in the right order, because "60% blue over stone, then 60% blue over that" is a different result than doing it the other way around. Three rules fall straight out of this:

Opaque passTransparent pass
Blendingoff — fragment replaces the pixelon — src·α + dst·(1−α)
Depth testonon
Depth writeonoff
Draw orderany (depth buffer sorts)far → near
Whenfirstafter all opaque

The subtle one is depth write off. Transparent surfaces still test against the depth buffer — water behind a solid wall must be hidden by that wall. But a transparent fragment must not write its own depth, because it did not replace what is behind it; it let it show through. If it wrote depth, a nearer pane of glass would stamp its distance into the buffer and then wrongly occlude a farther transparent surface you should still be able to see through it.

Opaque first, and depth-write OFF for transparent — or nothing works

These two are the load-bearing rules of the whole lesson. Draw transparent geometry before the opaque behind it and there is nothing in the framebuffer to blend against — the water blends with the sky. Leave depth-write enabled on the transparent pass and your translucent surfaces stamp depth and then occlude each other (and the scene behind them) as if they were solid, so you get hard black or clipped edges instead of see-through blue. If transparency ever looks "solid but tinted," check these two first.

Splitting the chunk into two meshes

Because the two categories are drawn with different state, they cannot share a draw call. The clean fix is to have the mesher emit two meshes per chunk: an opaque mesh and a transparent mesh. This is a tiny extension of the mesher from the earlier lessons — the geometry it builds per face is byte-for-byte identical, we only choose which buffer the face lands in, based on the block's transparent flag from the registry. 

#include "BlockQueries.h"   // IsTransparent, FaceVisible, FaceTexture, LightEmission
#include "Vertex.h"
#include <vector>
#include <cstdint>

// One CPU-side mesh: a vertex list and an index list.
struct MeshBuffers {
    std::vector<Vertex>   verts;
    std::vector<uint32_t> indices;
};

// A chunk now produces TWO meshes: everything opaque, and everything
// see-through. They are uploaded separately and drawn in different passes.
struct ChunkMeshData {
    MeshBuffers opaque;       // stone, dirt, grass, ...   (depth write ON)
    MeshBuffers transparent;  // water, glass, leaves      (alpha-blended)
};

void BuildChunkMesh(const Chunk& chunk, ChunkMeshData& out) {
    for (int z = 0; z < Chunk::SIZE; ++z)
    for (int y = 0; y < Chunk::SIZE; ++y)
    for (int x = 0; x < Chunk::SIZE; ++x) {
        BlockId here = chunk.At(x, y, z);
        if (here == BLOCK_AIR) continue;

        // The whole split, in one line: route this block's faces to the
        // matching buffer set. Same geometry — only the destination differs.
        MeshBuffers& dst = IsTransparent(here) ? out.transparent : out.opaque;

        for (int f = 0; f < 6; ++f) {
            BlockId neighbor = chunk.NeighborAt(x, y, z, f);

            // Exactly the visibility rule from Hidden Face Removal, reused
            // unchanged. FaceVisible = IsTransparent(neighbor) && neighbor != self,
            // so the 'neighbor != self' term still culls water-water and
            // glass-glass interior faces. A water face pressed against solid
            // stone is left to the stone's own opaque face — nothing is drawn
            // twice, and no two coplanar faces z-fight.
            if (!FaceVisible(here, neighbor)) continue;

            EmitFace(dst, x, y, z, f, here);   // the greedy/naive emit from earlier
        }
    }
}

That single ternary is the entire structural change. Everything downstream stays the same: two MeshBuffers in, two Mesh objects out, each uploaded once. Note that reusing FaceVisible for both meshes is exactly what we want — it culls the interior faces between two water cells (you never see the seam inside a lake) while still keeping the water surface that meets air.

Leaves want alpha-test, not alpha-blend

Our two-way split treats every transparent block the same, but there are really two flavors of see-through. Translucent surfaces (water, glass) mix smoothly with what is behind them and must be blended and sorted. Cutout surfaces (leaves, grass, saplings) are fully opaque where the texture is opaque and fully gone where it is not — a hard clip() on alpha with no blending. Cutout geometry writes depth and needs no sorting, so a fuller engine keeps a third mesh for it and draws it in the opaque pass with an alpha-tested shader. Water and glass are the only blocks that truly need the blended pass below; leaves are along for the ride only to keep this lesson to two meshes.

The emit step packs one extra thing now. We already pack a texture layer, ambient occlusion and baked light into a single uint32_t per vertex (from Texturing Voxels with an Atlas). We add two fields the shaders below need: an emissive strength for glow, and a one-bit wave flag for water. There is plenty of room left in the word. 

// Extends the packed vertex from "Texturing Voxels with an Atlas".
//   bits  0..15  texture-array layer
//   bits 16..17  ambient occlusion (0..3)
//   bits 18..21  baked light level (0..15)
//   bits 22..25  emissive strength (0..15, 0 = not emissive)   <-- new
//   bit  26      wave (1 = water surface, animate it)          <-- new
inline uint32_t packVertex(uint16_t layer, uint8_t ao, uint8_t light,
                           uint8_t emissive, bool wave) {
    return  uint32_t(layer)
         | (uint32_t(ao       & 0x3u) << 16)
         | (uint32_t(light    & 0xFu) << 18)
         | (uint32_t(emissive & 0xFu) << 22)
         | (uint32_t(wave ? 1u : 0u)  << 26);
}

// Inside EmitFace, the two new fields come straight from the block type:
//   const BlockType& bt = Blocks().Get(here);
//   uint8_t emissive = bt.emissive ? bt.lightLevel : 0;  // reuse the 0..15 level
//   bool    wave     = (here == BLOCK_WATER);
//   uint32_t data    = packVertex(layer, ao, light, emissive, wave);

The blend and depth state

Transparency is just two pieces of pipeline state: a blend state that performs the "over" composite, and a depth-stencil state that tests but does not write. On Direct3D 11 you create both once at startup and bind them for the transparent pass. 

// --- created once at startup ---

// OPAQUE depth: test AND write. (You already have this from the depth lesson.)
D3D11_DEPTH_STENCIL_DESC opaqueDS = {};
opaqueDS.DepthEnable    = TRUE;
opaqueDS.DepthWriteMask = D3D11_DEPTH_WRITE_MASK_ALL;    // writes depth
opaqueDS.DepthFunc      = D3D11_COMPARISON_LESS;
device->CreateDepthStencilState(&opaqueDS, &m_depthWrite);

// TRANSPARENT depth: test stays ON, but depth WRITE is OFF.
D3D11_DEPTH_STENCIL_DESC blendDS = {};
blendDS.DepthEnable    = TRUE;
blendDS.DepthWriteMask = D3D11_DEPTH_WRITE_MASK_ZERO;    // <-- test yes, WRITE no
blendDS.DepthFunc      = D3D11_COMPARISON_LESS_EQUAL;
device->CreateDepthStencilState(&blendDS, &m_depthTestNoWrite);

// TRANSPARENT blend: standard straight-alpha "over".
D3D11_BLEND_DESC bd = {};
bd.RenderTarget[0].BlendEnable    = TRUE;
bd.RenderTarget[0].SrcBlend       = D3D11_BLEND_SRC_ALPHA;      // out = src.rgb * src.a
bd.RenderTarget[0].DestBlend      = D3D11_BLEND_INV_SRC_ALPHA;  //     + dst.rgb * (1 - src.a)
bd.RenderTarget[0].BlendOp        = D3D11_BLEND_OP_ADD;
bd.RenderTarget[0].SrcBlendAlpha  = D3D11_BLEND_ONE;
bd.RenderTarget[0].DestBlendAlpha = D3D11_BLEND_INV_SRC_ALPHA;
bd.RenderTarget[0].BlendOpAlpha   = D3D11_BLEND_OP_ADD;
bd.RenderTarget[0].RenderTargetWriteMask = D3D11_COLOR_WRITE_ENABLE_ALL;
device->CreateBlendState(&bd, &m_alphaBlend);

On Metal the split is a little different: blending is baked into the render pipeline state object, not a separate bindable, so you compile a second pipeline for transparent geometry with blending switched on. Depth-write-off, however, is still a standalone MTLDepthStencilState you set per pass, just like on D3D11. 

// Metal: blending lives on the transparent PIPELINE's color attachment.
auto* ca = pipelineDesc->colorAttachments()->object(0);
ca->setBlendingEnabled(true);
ca->setSourceRGBBlendFactor(MTL::BlendFactorSourceAlpha);
ca->setDestinationRGBBlendFactor(MTL::BlendFactorOneMinusSourceAlpha);
ca->setRgbBlendOperation(MTL::BlendOperationAdd);
ca->setSourceAlphaBlendFactor(MTL::BlendFactorOne);
ca->setDestinationAlphaBlendFactor(MTL::BlendFactorOneMinusSourceAlpha);
ca->setAlphaBlendOperation(MTL::BlendOperationAdd);

// Depth WRITE off is a separate depth-stencil state, set per render pass:
auto* dsd = MTL::DepthStencilDescriptor::alloc()->init();
dsd->setDepthCompareFunction(MTL::CompareFunctionLessEqual);
dsd->setDepthWriteEnabled(false);   // test on, write off
m_depthTestNoWrite = device->newDepthStencilState(dsd);

Ordering the frame

Now the render loop puts it together. We already have a list of visible chunks from Frustum Culling & Draw-Call Batching. Pass one draws all of their opaque meshes in whatever order they come. Then we sort the same chunks far-to-near and draw their transparent meshes with the blended state. 

// visibleChunks: already frustum-culled this frame. cam = camera world position.

// --- PASS 1: OPAQUE ------------------------------------------------------
// Depth write ON, no blending. Any draw order is fine; the depth buffer
// resolves overlaps per pixel.
ctx->OMSetBlendState(nullptr, nullptr, 0xffffffff);          // blending off
ctx->OMSetDepthStencilState(m_depthWrite.Get(), 0);
for (const Chunk* c : visibleChunks)
    c->mesh.opaque.Draw(ctx);

// --- sort transparent chunks FAR -> NEAR --------------------------------
// Nearer water must blend on top of farther water. Per-face sorting is
// impractical in a voxel world, so we sort at chunk granularity and accept
// the small error (see the note below).
std::sort(visibleChunks.begin(), visibleChunks.end(),
    [&](const Chunk* a, const Chunk* b) {
        return DistanceSq(a->CenterWorld(), cam)
             > DistanceSq(b->CenterWorld(), cam);   // farthest first
    });

// --- PASS 2: TRANSPARENT -------------------------------------------------
// Blend ON, depth test ON, depth WRITE off.
ctx->OMSetBlendState(m_alphaBlend.Get(), nullptr, 0xffffffff);
ctx->OMSetDepthStencilState(m_depthTestNoWrite.Get(), 0);
for (const Chunk* c : visibleChunks)
    if (c->mesh.transparent.IndexCount() > 0)      // skip chunks with no water/glass
        c->mesh.transparent.Draw(ctx);

Both passes render into the same HDR sceneTarget (the RGBA16Float texture from Module 10), because everything you draw still has to flow through tone mapping at the end. That target is also what makes the glow in the next section possible.

Chunk-granularity sorting is an accepted approximation

Truly correct alpha blending would sort every transparent triangle back-to-front every frame — hopeless when a lake is thousands of quads. Sorting whole chunks is cheap (a few hundred entries) and looks right almost always, because a chunk's transparent surfaces are usually a single flat water plane with no self-overlap. The error only shows up in pathological cases — two panes of glass in the same chunk seen edge-on — and players never notice. Disabling depth-write is what keeps even that error graceful: surfaces blend in a slightly wrong order rather than punching holes in each other.

Water: a tint and a wobble

Water is drawn with its own shader in the transparent pass. Two things make it read as water: a semi-transparent blue, and a gentle animated surface. The color is easy — sample the water tile and output an alpha below 1.0 so the blend state lets the scene behind show through. The motion is a vertex-shader trick: nudge each surface vertex up and down by a sum of sines of time.

The one detail that keeps the surface from tearing apart: the wave must be a function of world (x, z) and time only — never of the vertex's own identity. Then any two vertices that share an (x, z) — across faces, across blocks, across chunk borders — receive the identical offset and stay welded together. The whole sheet ripples as one continuous surface with no cracks. 

cbuffer PerFrame : register(b0) { float4x4 gViewProj; float gTime; float3 _pad0; };
cbuffer PerChunk : register(b1) { float3 gChunkOrigin; float _pad1; };

Texture2DArray gBlocks : register(t0);
SamplerState   gSamp   : register(s0);

struct VSIn  { float3 pos : POSITION; float2 uv : TEXCOORD0; uint data : TEXCOORD1; };
struct VSOut { float4 pos : SV_POSITION; float2 uv : TEXCOORD0;
               nointerpolation float layer : TEXCOORD1; float shade : TEXCOORD2; };

// Gentle sum-of-sines ripple. Depends ONLY on world (x,z) and time, so
// vertices that share an (x,z) always get the same offset -> watertight.
float WaveHeight(float2 worldXZ, float t) {
    return 0.06 * sin(worldXZ.x * 0.7 + t * 1.5)
         + 0.04 * sin(worldXZ.y * 0.9 + t * 1.1);
}

VSOut VSWater(VSIn i) {
    float3 world = i.pos + gChunkOrigin;                 // chunk-local -> world

    // Bit 26 (the 'wave' flag) gates the animation, so glass and leaves that
    // share this transparent mesh stay perfectly still.
    float wave = (float)((i.data >> 26) & 0x1u);
    world.y += wave * WaveHeight(world.xz, gTime);

    VSOut o;
    o.pos   = mul(float4(world, 1.0), gViewProj);
    o.uv    = i.uv;
    o.layer = (float)(i.data & 0xFFFFu);
    o.shade = (float)((i.data >> 18) & 0xFu) / 15.0;      // baked light, as before
    return o;
}

float4 PSWater(VSOut i) : SV_TARGET {
    float4 tex = gBlocks.Sample(gSamp, float3(i.uv, i.layer));
    float3 col = tex.rgb * i.shade;
    // Alpha < 1 is what the blend state reads as SRC_ALPHA. ~0.6 keeps the
    // water clearly blue while letting most of the scene behind through.
    return float4(col, 0.6);
}
Never tie the wave to a per-vertex value

If WaveHeight reads anything unique to the vertex — a vertex index, a random per-quad seed, the chunk-local position instead of the world position — then two surface quads that meet at a shared edge compute different offsets and a visible crack opens along the seam, flickering as the water moves. Chunk borders are where this bites hardest, because the two sides are meshed independently. Feed the wave world coordinates and the seam is invisible by construction.

The MTLSL version is a direct translation: the same WaveHeight, reading time from a uniform and the packed data from a vertex attribute, with [[flat]] on the layer output instead of nointerpolation. If you want to go further, the completed HDR sceneTarget is already a sampleable image — a screen-space refraction pass can sample it at a UV nudged by the water's normal to bend whatever is underwater, exactly the render-target-as-input trick from Module 10. That is an optional extension; the tint-and-wobble above is already convincing.

Glow: driving bloom from emissive blocks

Now the payoff the bloom lesson promised. Lava, glowstone and glowing ore should not just be bright textures — they should bleed light into the pixels around them. You already built the machine for that in Module 10: bloom extracts everything in the HDR scene above a luminance threshold, blurs it wide, and adds it back. All this lesson has to do is make emissive blocks write color above 1.0 into the scene target. The bright pass does the rest.

The opaque block pixel shader already computes a lit color from the texture and the baked AO·light shade. We add one term on top: an emissive contribution that is not dimmed by shade — glowstone glows just as hard in a pitch-black cave — and is free to exceed 1.0. 

// Opaque block pixel shader. Writes into the HDR sceneTarget (RGBA16F),
// BEFORE tone mapping, so values above 1.0 are preserved for bloom.
Texture2DArray gBlocks : register(t0);
SamplerState   gSamp   : register(s0);

struct PSIn {
    float4 pos      : SV_POSITION;
    float2 uv       : TEXCOORD0;
    nointerpolation float layer    : TEXCOORD1;
    float  shade    : TEXCOORD2;                 // ao * light, in [0,1]
    nointerpolation float emissive : TEXCOORD3;  // 0..15 unpacked from the vertex
};

// Peak HDR multiplier for a fully-emissive block (light level 15).
static const float kEmissiveScale = 6.0;

float4 PSBlock(PSIn i) : SV_TARGET {
    float4 base = gBlocks.Sample(gSamp, float3(i.uv, i.layer));

    // Reflected light: dimmed by AO and baked light, stays within [0,1].
    float3 lit = base.rgb * i.shade;

    // Emissive light ADDS on top and is NOT dimmed. Scaled past 1.0 it lands
    // as genuine HDR in the scene target -- exactly what the bright pass wants.
    float  strength = (i.emissive / 15.0) * kEmissiveScale;   // glowstone(15) -> 6.0
    float3 hdr = lit + base.rgb * strength;

    return float4(hdr, 1.0);
}

That is the whole connection. A glowstone block has emissive = 15, so its pixels leave this shader at roughly 6× their texture color — far above the display's ceiling of 1.0. Tone mapping will later squeeze that back into range, but bloom runs first, sees those bright pixels, and smears a soft halo of light around them. Dial kEmissiveScale per material if you like — a lava block wants to be brighter than an ore vein — but the mechanism never changes.

Reuse Module 10's post stack verbatim — the glow is already built

Do not write a new "block glow" effect. The exact PostProcessStack you assembled in Module 10 — bright pass, separable blur down the mip chain, composite, then tone map — already turns any above-1.0 pixel into a halo. Emissive-in-HDR is the only new ingredient: the moment your lava writes > 1.0 into the scene target, it blooms with zero extra rendering code. This is the concrete reason the whole engine keeps its lighting in linear HDR and tone-maps exactly once, at the very end.

Where does the shader's emissive input come from? The mesher packed it: bits 22–25 of the vertex data, set to bt.lightLevel for emissive blocks and 0 otherwise. The vertex shader unpacks those four bits with (data >> 22) & 0xF and forwards them as a nointerpolation float — flat across the quad, since every corner of a glowstone face is equally bright. No new buffers, no new draw calls; the glow rides along in the geometry you were already emitting.

Recap
  • Split each chunk into an opaque mesh and a transparent mesh by the block's transparent flag; both reuse FaceVisible, so water-water and glass-glass interior faces still vanish.
  • Draw all opaque geometry first (depth test + write, any order), then the transparent mesh with alpha blending, depth test on, and depth write off.
  • Sort transparent chunks far-to-near; per-face sorting is impractical in a voxel world, so chunk granularity is the accepted approximation and depth-write-off keeps its error graceful.
  • The blend is straight-alpha "over": SrcBlend = SRC_ALPHA, DestBlend = INV_SRC_ALPHA on D3D11, the matching factors on the Metal pipeline; depth-write-off is a separate depth-stencil / MTLDepthStencilState.
  • Water gets alpha near 0.6 and a sum-of-sines wobble that depends only on world (x, z) and time, keeping the surface watertight across chunk borders; refraction via the scene target is an optional extension.
  • Emissive blocks add unclamped base × strength into the HDR sceneTarget; Module 10's bright pass catches everything above 1.0, so lava and glowstone bloom with no new rendering code.
