Texturing Voxels with an Atlas

Your greedy mesher now emits big, merged quads, but every one of them is still blank geometry. In this lesson we give each face the right tile — dirt on the sides, grass on top, stone all the way around — by looking the tile up from the block registry and feeding it to the texturing machinery you built in Module 10. Along the way we hit a real fork in the road: a texture atlas and a texture array both work for a single cube, but only one of them survives greedy meshing. We will build the array path and wire it straight into the mesher and the shader.

What you'll learn
  • How to store per-face tiles in the block registry and look one up with faceTexture(block, face).
  • Why a texture atlas fights greedy meshing, and why a texture array is the right tool for voxels.
  • How to pack a tile layer, ambient occlusion and light into a single uint32_t per vertex.
  • How the mesher chooses a layer per face and writes tiling UVs that run 0..w across a merged quad.
  • The HLSL (and Metal) that samples a Texture2DArray and multiplies by the baked per-vertex shade.

Where the tile comes from: the block registry

In the registry lesson we described each block with a data-oriented BlockType. Texturing adds one field: which tile each of the six faces should use. A stone block uses the same tile everywhere, but grass is the classic case — a green top, grassy-dirt sides, and a plain dirt bottom. So the natural shape is a six-entry array of tile indices, one per face direction.

enum class Face : uint8_t { PosX, NegX, PosY, NegY, PosZ, NegZ };
// PosY is the top face, NegY the bottom.

struct BlockType
{
    const char* name;
    bool        solid;        // from the registry lesson
    uint16_t    faceTex[6];   // texture-array layer per face
};

// Every face the same tile (stone, dirt, sand, ...).
inline void setAllFaces(BlockType& b, uint16_t layer)
{
    for (uint16_t& f : b.faceTex) f = layer;
}

// A "column" block: distinct top / side / bottom (grass, logs, ...).
inline void setColumn(BlockType& b, uint16_t top, uint16_t side, uint16_t bottom)
{
    b.faceTex[(int)Face::PosY] = top;
    b.faceTex[(int)Face::NegY] = bottom;
    b.faceTex[(int)Face::PosX] = b.faceTex[(int)Face::NegX] =
    b.faceTex[(int)Face::PosZ] = b.faceTex[(int)Face::NegZ] = side;
}

// The one call the mesher makes.
inline uint16_t faceTexture(const BlockRegistry& reg, BlockId id, Face face)
{
    return reg.get(id).faceTex[(int)face];
}

The tile indices are the layers of the texture array you assembled in Module 10's "Texture Atlases & Texture Arrays". Give each tile a name so registration reads like English:

// Layer indices into the block texture array (built at load time).
enum : uint16_t {
    TEX_STONE = 0, TEX_DIRT, TEX_GRASS_TOP, TEX_GRASS_SIDE, TEX_SAND
};

void RegisterBlocks(BlockRegistry& reg)
{
    BlockType stone { "stone", /*solid*/true };
    setAllFaces(stone, TEX_STONE);
    reg.define(BLOCK_STONE, stone);

    BlockType grass { "grass", /*solid*/true };
    setColumn(grass, TEX_GRASS_TOP, TEX_GRASS_SIDE, TEX_DIRT);
    reg.define(BLOCK_GRASS, grass);
    // ... dirt, sand, and the rest ...
}

Two ways to put a tile on a face

Module 10 gave you two ways to fit many small tiles into one GPU resource. Both can texture a single, unmerged cube. Only one plays nicely with the merged quads greedy meshing produces.

Approach A — the atlas

An atlas packs every tile into one large Texture2D. Each tile lives in a sub-rectangle, and you texture a face by pointing its UVs at that sub-rect — say (0.0, 0.0)–(0.0625, 0.0625) for the first tile in a 16×16 grid. That is fine for a 1×1 face. But a greedy quad might cover a run of, say, three tiles, and now you want the tile to repeat three times across it. The sampler cannot do that: WRAP wraps the whole atlas back to (0,0), not the sub-rect, so instead of three grass tiles you get one grass tile, then whatever tile sits to its right in the atlas. Your only escapes are to stop merging across repeats — throwing away most of greedy meshing's win on textured surfaces — or to fake the repeat in the shader with frac(), which breaks mipmap selection at every tile seam.

Atlas mip bleeding

Even for single cubes, atlases bleed. When the GPU drops to a lower mip level, each mip texel is an average of a 2×2 block from the level above — and near a tile edge that block straddles into the neighbouring tile. The result is a faint fringe of the wrong colour around every tile as the camera pulls back. You fight it with gutters and clamped UVs, but it never fully goes away. This edge bleeding is the main reason we prefer arrays for voxels.

Approach B — the texture array (recommended)

A texture array stores each tile as its own independent slice of a Texture2DArray: same width, same height, its own full mip chain. You select a tile with an integer layer index and sample with arr.Sample(smp, float3(u, v, layer)). Because each slice is a whole texture in its own right, 0..1 UVs cover exactly one tile, a WRAP sampler tiles that slice cleanly for as many repeats as you like, and mips are averaged within the slice — so there is no neighbour to bleed from. UVs of 0..w across a merged quad just work.

We build the array path. It keeps the mesher simple, keeps greedy meshing intact, and gives us the crisp, seam-free look voxel worlds are known for.

One layer per vertex, packed tight

To sample the array in the shader, each vertex needs to carry its tile's layer. It will soon also need ambient occlusion (the next lesson) and light (a later one). Rather than grow the vertex with three separate fields, we pack all three into a single uint32_t — voxel meshes have millions of vertices, and every byte you shave off is bandwidth saved and cache hits gained.

struct Vertex
{
    float    px, py, pz;   // position within the chunk
    float    u, v;         // tile-space UV: 0..w and 0..h
    uint32_t data;         // packed: layer | ao<<16 | light<<18
};

// layer: 16 bits (0..65535)   ao: 2 bits (0..3)   light: 4 bits (0..15)
inline uint32_t packVertex(uint16_t layer, uint8_t ao, uint8_t light)
{
    return  uint32_t(layer)
         | (uint32_t(ao    & 0x3)  << 16)
         | (uint32_t(light & 0xF)  << 18);
}

The input layout describes those three fields to the Input Assembler. The packed integer goes across as a single R32_UINT so the shader can unpack it — never as a float, which would let the rasterizer interpolate the bits into nonsense.

D3D11_INPUT_ELEMENT_DESC layout[] =
{
    { "POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT, 0,  0,
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
    { "TEXCOORD", 0, DXGI_FORMAT_R32G32_FLOAT,    0, 12,   // after 3 floats
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
    { "TEXCOORD", 1, DXGI_FORMAT_R32_UINT,        0, 20,   // after 5 floats
      D3D11_INPUT_PER_VERTEX_DATA, 0 },
};
// 12 (pos) + 8 (uv) + 4 (data) = 24 bytes per vertex.
Room to grow

16 bits of layer is 65,536 possible tiles — far more than D3D11's 2048-slice array limit, so you can shrink it to 11 bits later and hand the spare bits to a second light channel. For now the generous split keeps the code readable; the AO and light fields sit at bits 16 and 18 waiting for the next two lessons to fill them.

Choosing the layer in the mesher

Greedy meshing scans one face direction at a time and builds a 2D mask of which cells have a visible face. That mask is exactly where the tile is decided: when a face is visible, we record the layer faceTexture returns for that block and direction. We also stub in full ambient occlusion and full light so the vertices look correct today and the later lessons only have to change two numbers.

struct MaskCell
{
    bool     solid = false;   // is there a visible face here?
    uint16_t layer = 0;       // texture-array layer for this face
    uint8_t  ao    = 3;       // full — the AO lesson fills this in
    uint8_t  light = 15;      // full — the lighting lesson fills this in
};

// Build one mask cell while scanning a slice (from the greedy lesson).
MaskCell faceCell(const BlockRegistry& reg,
                  BlockId here, BlockId neighbour, Face face)
{
    // Visible only if 'here' is solid and its neighbour is not.
    if (!reg.get(here).solid || reg.get(neighbour).solid)
        return {};                       // empty: no face here
    MaskCell c;
    c.solid = true;
    c.layer = faceTexture(reg, here, face);
    return c;
}

Two cells may only be merged into one quad when they would produce the same pixels — same tile, and (once the later lessons land) the same AO and light. That test belongs in the mask comparison the greedy sweep already runs.

Put the layer in the mask key

Add the tile layer to the equality test your greedy sweep uses to decide whether two cells can merge. Then a run of dirt never fuses with a run of stone, and a merge can never straddle two different textures. This is the same guardrail that will later stop AO and light gradients from being flattened across a merge.

bool canMerge(const MaskCell& a, const MaskCell& b)
{
    return a.solid && b.solid
        && a.layer == b.layer     // never merge across textures
        && a.ao    == b.ao
        && a.light == b.light;
}

// Emit one merged quad, w x h tiles, for a given face.
// p0..p3 are its corners in CCW order (from the greedy lesson).
void emitQuad(std::vector<Vertex>& verts, std::vector<uint32_t>& indices,
              const Float3& p0, const Float3& p1,
              const Float3& p2, const Float3& p3,
              int w, int h, const MaskCell& cell)
{
    uint32_t packed = packVertex(cell.layer, cell.ao, cell.light);
    uint32_t base   = (uint32_t)verts.size();

    // UVs run 0..w and 0..h so a WRAP sampler tiles the slice.
    verts.push_back({ p0.x, p0.y, p0.z, 0.0f,      0.0f,     packed });
    verts.push_back({ p1.x, p1.y, p1.z, (float)w,  0.0f,     packed });
    verts.push_back({ p2.x, p2.y, p2.z, (float)w,  (float)h, packed });
    verts.push_back({ p3.x, p3.y, p3.z, 0.0f,      (float)h, packed });

    indices.insert(indices.end(),
        { base+0, base+1, base+2,  base+0, base+2, base+3 });
}

The layer is written identically on all four vertices — a quad only ever shows one tile — while the UVs stretch to w and h so the slice repeats once per tile the quad covers. That single detail is what makes the array path merge-friendly where the atlas could not.

Sampling the array in the shader

The vertex shader unpacks the integer and forwards three things: the tile-space UV, the layer (flat — it must not be interpolated), and a combined shade term. Because all four corners share a layer, we mark it nointerpolation so the hardware takes it verbatim instead of blending it.

cbuffer PerObject : register(b0) { float4x4 gMVP; }

struct VSInput
{
    float3 position : POSITION;
    float2 uv       : TEXCOORD0;
    uint   data     : TEXCOORD1;   // packed layer | ao | light
};

struct PSInput
{
    float4 position : SV_POSITION;
    float2 uv       : TEXCOORD0;
    nointerpolation float layer : TEXCOORD1;
    float shade     : TEXCOORD2;   // AO * light, interpolated across the quad
};

PSInput VSMain(VSInput input)
{
    PSInput o;
    o.position = mul(gMVP, float4(input.position, 1.0));
    o.uv       = input.uv;
    o.layer    = (float)( input.data & 0xFFFF);
    float ao    = (float)((input.data >> 16) & 0x3 ) / 3.0;   // 0..1
    float light = (float)((input.data >> 18) & 0xF ) / 15.0;  // 0..1
    o.shade    = ao * light;       // full = 1.0 for now
    return o;
}

Texture2DArray gBlocks  : register(t0);
SamplerState   gSampler : register(s0);

float4 PSMain(PSInput input) : SV_TARGET
{
    // Third coord is the array slice; the hardware rounds it to an int.
    float4 texel = gBlocks.Sample(gSampler, float3(input.uv, input.layer));
    texel.rgb *= input.shade;      // bake in AO * light
    return texel;
}

The shade term is deliberately a hook. Today ambient occlusion and light are both maxed out, so shade is 1.0 and the texture shows at full brightness. The next lesson writes real AO into bits 16–17 and the corners of your quads start to darken; the lighting lesson does the same for bits 18–21. Neither has to touch this shader — they just change the numbers the mesher packs.

Point filtering plus mipmaps for the pixel look

For the crisp, blocky aesthetic, sample with point magnification so up-close texels stay sharp, but keep the mip chain so distant blocks do not shimmer — D3D11_FILTER_MIN_MAG_POINT_MIP_LINEAR with ADDRESS_WRAP is the sweet spot we settled on in Module 10's "Textures In Depth: Filtering, Mipmaps & sRGB". Point sampling also sidesteps the edge-blend a linear filter would introduce at every tile seam inside a tiled quad.

The same shader in Metal

On macOS the layer is a separate uint argument to sample, not part of the coordinate. The unpacking is identical; only the sampling call differs.

fragment float4 fragmentMain(
    VOut in [[stage_in]],
    texture2d_array<float> blocks [[texture(0)]],
    sampler smp [[sampler(0)]])
{
    // Metal takes the array slice as its own uint argument.
    float4 texel = blocks.sample(smp, in.uv, uint(in.layer));
    return float4(texel.rgb * in.shade, texel.a);
}

The vertex-side unpack (data & 0xFFFF, the shifts, the ao * light product) is plain integer math and stays byte-for-byte the same between HLSL and MSL. Building the array resource and generating its per-slice mips is backend work — ID3D11Device::CreateTexture2D with ArraySize = tileCount on Windows, an MTL::TextureDescriptor of type texture2DArray on macOS — both of which you set up in Module 10.

A worked example: the grass block

Walk a single grass block through the whole path. When the greedy sweep reaches its top face (Face::PosY), faceTexture returns TEX_GRASS_TOP; the mask cell carries layer 2, and the emitted quad's four vertices all pack layer 2. The pixel shader samples slice 2 — green grass. The four sides resolve to TEX_GRASS_SIDE (layer 3), so they show the grass-over-dirt strip regardless of which of the four horizontal directions you are looking at. The bottom (Face::NegY) resolves to TEX_DIRT (layer 1) — plain dirt, exactly as in real Minecraft. Six faces, three different slices, one Texture2DArray, zero atlas bookkeeping. And because sides all share layer 3, a whole hillside of grass merges into a handful of quads that still texture correctly.

Transparent tiles come later

Water, glass and leaves also live as slices in this array, and the mesher looks up their layers the same way. What is different is blending — the alpha channel of those tiles, the sorting, and the separate draw pass they need. We leave that to the transparency lesson; here every tile is treated as opaque, which is correct for the solid blocks that make up the vast majority of a world.

Recap
  • Store a six-entry faceTex array in BlockType and read it with faceTexture(block, face); setColumn gives grass its distinct top, sides and bottom.
  • An atlas cannot tile a sub-rect across a merged quad and bleeds at tile edges under mipmapping; a texture array selects a whole slice by integer layer and tiles cleanly, so it is the right choice for voxels.
  • Pack the tile layer, AO and light into one uint32_t per vertex and hand it to the Input Assembler as an R32_UINT, never a float.
  • Decide the layer in the greedy mask, add it to the merge key so quads never span two textures, write it on all four vertices, and let UVs run 0..w so a WRAP sampler tiles the slice.
  • Sample gBlocks.Sample(smp, float3(u, v, layer)) in HLSL (a separate uint slice argument in Metal) and multiply by the baked shade so the AO and lighting lessons plug straight in.
  • Keep point magnification with mipmaps and wrap addressing for the crisp, seam-free pixel look.
