Blocks, IDs & a Data-Oriented Registry

The chunk from the last lesson is a flat array of voxels, and there can be tens of millions of them on screen at once. The single most important decision in a voxel engine is what one voxel actually is. Get it wrong — a fat, polymorphic object per cell — and you drown in memory and cache misses before you draw a single triangle. Get it right — a tiny integer id that indexes a shared table of block types — and millions of voxels cost almost nothing. This lesson builds that table: a data-oriented block registry that the mesher, the lighting system, and the culler will all hammer.

What you'll learn
  • Why a polymorphic Block object per voxel is a performance disaster, and how the flyweight pattern fixes it.
  • How to store each voxel as a small integer id (uint8_t / uint16_t) that indexes a shared table.
  • How to design a BlockType descriptor: solidity, transparency, light emission, and per-face texture layers.
  • How to build a contiguous BlockRegistry with Register() and a hot Get(id), keyed by an enum with AIR = 0.
  • How to write inline IsSolid / IsTransparent helpers and pack an id plus a few bits of state into a 16-bit voxel.

The trap: one object per voxel

The instinct from earlier modules is to reach for a class hierarchy: a base Block with virtual bool IsSolid(), then Stone, Dirt, Grass overriding it. It reads beautifully. It is also catastrophic here, and it is worth understanding exactly why.

Do the arithmetic. A single 32×32×32 chunk holds 32,768 voxels. A modest view distance is a few hundred chunks loaded at once — call it tens of millions of voxels. Now suppose each voxel is a Block* pointing at a heap-allocated polymorphic object:

RepresentationBytes per voxelPer 32³ chunkCost to query
Block* to a heap object (vptr + fields + alloc overhead)~56~1.8 MBpointer chase + vtable indirection
Inline value struct with virtuals (32 B)321 MBvtable indirection
uint16_t id into a registry264 KBone indexed load
uint8_t id into a registry132 KBone indexed load

The pointer version is roughly 28× the memory of a 2-byte id, and every single query — and the mesher asks “is my neighbor solid?” six times per voxel — pays for a pointer dereference into scattered heap memory followed by a virtual call. That is death by cache miss. The CPU spends its life waiting on RAM instead of building geometry.

Never use per-voxel polymorphism or per-voxel heap objects

No new Block per cell, no virtual methods called per voxel, no std::shared_ptr<Block> arrays. There are only a few dozen kinds of block but tens of millions of instances. The moment you allocate or dispatch per instance, meshing and lighting fall off a performance cliff you cannot climb back up.

The flyweight: an id that indexes a shared table

The fix is the flyweight pattern. Notice that every stone block is identical to every other stone block — same solidity, same texture, same everything. So we store that shared data once, in a BlockType descriptor, and give each type a small integer id. A voxel is then just that id. Thousands of stone voxels all carry the number 1 and all index the same single descriptor.

The BlockType descriptor

A BlockType is a plain aggregate of the facts every consumer needs: a debug name, a few boolean flags, a light level, and one texture per face. There is exactly one of these per kind of block — never one per voxel. We index textures per face so grass can wear a grassy top, a dirt bottom, and a grass-side band around the middle. Those indices are layers into the block texture array we built back in module 10 (a Texture2DArray on Direct3D 11, an array MTLTexture on Metal). 

#include <array>
#include <string>
#include <cstdint>
#include <utility>

// The six faces of a cube, indexed by which axis-direction they point.
enum Face : uint8_t {
    FACE_NEG_X = 0, // west
    FACE_POS_X,     // east
    FACE_NEG_Y,     // bottom
    FACE_POS_Y,     // top
    FACE_NEG_Z,     // north
    FACE_POS_Z,     // south
    FACE_COUNT      // == 6
};

// A *shared* description of one kind of block. Exactly one exists per
// block type in the whole world; a voxel just stores this type's id.
struct BlockType {
    std::string name;                 // "grass" — for tools and debugging
    bool    solid       = true;       // has collision / blocks movement
    bool    transparent = false;      // lets light & sight pass (glass, leaves, water)
    bool    emissive    = false;      // gives off light of its own
    uint8_t lightLevel  = 0;          // 0..15, brightness if emissive (glowstone = 15)

    // One texture-array layer index per face (see module 10's texture array).
    std::array<uint16_t, FACE_COUNT> faceTex{};

    // Paint every face with the same tile.
    static BlockType Uniform(std::string n, uint16_t tile) {
        BlockType b;
        b.name = std::move(n);
        b.faceTex.fill(tile);
        return b;
    }

    // A "log-style" block: distinct top, sides, and bottom.
    static BlockType Column(std::string n, uint16_t top, uint16_t side, uint16_t bottom) {
        BlockType b;
        b.name = std::move(n);
        b.faceTex[FACE_POS_Y] = top;
        b.faceTex[FACE_NEG_Y] = bottom;
        b.faceTex[FACE_NEG_X] = b.faceTex[FACE_POS_X] =
        b.faceTex[FACE_NEG_Z] = b.faceTex[FACE_POS_Z] = side;
        return b;
    }
};
solid and transparent are independent

They answer different questions. solid is for physics — can you walk through it? transparent is for rendering and lighting — does it hide the neighbor's face and block light? Leaves are solid but transparent (you stand on them, yet see through). Water is not solid but transparent (you swim through, and see through). Glass is solid and transparent. Keeping them as two flags lets every combination fall out naturally.

A contiguous BlockRegistry

The registry is the flyweight pool: a single std::vector<BlockType> where the index is the id. Register() appends a type and returns the id it landed at. Get() is the hot path — it is called millions of times per meshing pass — so it does a bare operator[] with no bounds check. The ids it receives come from our own chunk data, so there is nothing to validate at runtime. 

#include <vector>
#include <cassert>

using BlockId = uint16_t; // a voxel is, at its simplest, just one of these

// Well-known ids. AIR = 0 is deliberate and load-bearing: a zeroed chunk
// (default-constructed, or memset to 0) is entirely air, for free.
enum : BlockId {
    BLOCK_AIR = 0,
    BLOCK_STONE,
    BLOCK_DIRT,
    BLOCK_GRASS,
    BLOCK_SAND,
    BLOCK_WOOD,
    BLOCK_LEAVES,
    BLOCK_WATER,
    BLOCK_GLASS,
    BLOCK_GLOWSTONE,
};

class BlockRegistry {
public:
    // Append a type and hand back the id it now occupies.
    BlockId Register(BlockType type) {
        BlockId id = static_cast<BlockId>(m_types.size());
        m_types.push_back(std::move(type));
        return id;
    }

    // The hot lookup: one indexed load into a contiguous array.
    // No bounds check — ids come from our own data and this is called
    // millions of times per meshing/lighting pass.
    const BlockType& Get(BlockId id) const {
        assert(id < m_types.size()); // debug-only guard, compiled out in release
        return m_types[id];
    }

    size_t Count() const { return m_types.size(); }

private:
    std::vector<BlockType> m_types; // contiguous, cache-friendly
};

// One registry for the whole engine.
inline BlockRegistry& Blocks() {
    static BlockRegistry registry;
    return registry;
}
Keep the registry a plain contiguous array

Resist the urge to key it by std::unordered_map<std::string, BlockType>. The mesher and light propagation index it by integer, in tight loops, and a std::vector gives them a predictable, cache-friendly load — the descriptor is often already in cache from the last voxel. Look types up by name only once at load time; look them up by id forever after.

Registering the default blocks

Registration order must line up with the enum, because the id is nothing more than the position in the vector. We register air first so it lands at 0, then the rest in enum order. The texture constants are layer indices into module 10's block atlas array. 

// Layer indices into the block texture array from module 10.
enum : uint16_t {
    TEX_STONE = 0, TEX_DIRT, TEX_GRASS_TOP, TEX_GRASS_SIDE, TEX_SAND,
    TEX_WOOD_TOP, TEX_WOOD_SIDE, TEX_LEAVES, TEX_WATER, TEX_GLASS, TEX_GLOWSTONE
};

void RegisterDefaultBlocks() {
    BlockRegistry& r = Blocks();

    // The ORDER here MUST match the enum: each Register() returns the next id.
    BlockType air = BlockType::Uniform("air", 0);
    air.solid = false;
    air.transparent = true;               // air never hides a neighbor's face
    r.Register(std::move(air));                              // -> BLOCK_AIR (0)

    r.Register(BlockType::Uniform("stone", TEX_STONE));      // -> BLOCK_STONE (1)
    r.Register(BlockType::Uniform("dirt",  TEX_DIRT));       // -> BLOCK_DIRT  (2)
    r.Register(BlockType::Column ("grass", TEX_GRASS_TOP,
                                   TEX_GRASS_SIDE, TEX_DIRT));// -> BLOCK_GRASS (3)
    r.Register(BlockType::Uniform("sand",  TEX_SAND));       // -> BLOCK_SAND  (4)
    r.Register(BlockType::Column ("wood", TEX_WOOD_TOP,
                                   TEX_WOOD_SIDE, TEX_WOOD_TOP));// -> BLOCK_WOOD (5)

    BlockType leaves = BlockType::Uniform("leaves", TEX_LEAVES);
    leaves.transparent = true;            // solid for collision, but see-through
    r.Register(std::move(leaves));                           // -> BLOCK_LEAVES (6)

    BlockType water = BlockType::Uniform("water", TEX_WATER);
    water.solid = false;                  // you swim through it
    water.transparent = true;
    r.Register(std::move(water));                            // -> BLOCK_WATER (7)

    BlockType glass = BlockType::Uniform("glass", TEX_GLASS);
    glass.transparent = true;             // solid, but light and sight pass
    r.Register(std::move(glass));                            // -> BLOCK_GLASS (8)

    BlockType glow = BlockType::Uniform("glowstone", TEX_GLOWSTONE);
    glow.emissive = true;
    glow.lightLevel = 15;                 // maximum brightness
    r.Register(std::move(glow));                             // -> BLOCK_GLOWSTONE (9)
}
Registration order is a contract

Because the id is the vector index, inserting a new block in the middle of this function silently renumbers everything after it — and every chunk you ever saved to disk now decodes into the wrong blocks. Add new blocks only at the end, and keep the enum and this function in the same order. When you reach world serialization, you will map ids through a stable name table for exactly this reason.

Hot-path helpers

The mesher and the light propagation code should never spell out Blocks().Get(id).solid by hand — it is noisy and easy to get subtly wrong. Wrap the common questions in inline free functions. They are a single indexed load plus a field read, so with inlining the compiler folds them straight into the caller's inner loop; there is no function-call overhead at all. 

// Tiny, inlined into the mesher's and lighting's innermost loops.
inline bool IsSolid(BlockId id)       { return Blocks().Get(id).solid; }
inline bool IsTransparent(BlockId id) { return Blocks().Get(id).transparent; }

inline uint8_t LightEmission(BlockId id) {
    const BlockType& b = Blocks().Get(id);
    return b.emissive ? b.lightLevel : 0;
}

// The texture layer to use for one face of a block — fed to the mesher's UVs.
inline uint16_t FaceTexture(BlockId id, Face face) {
    return Blocks().Get(id).faceTex[face];
}

// The mesher's core question: draw my face toward 'neighbor'?
// Draw it only when the neighbor doesn't hide it. The 'neighbor != self'
// term merges the interior faces between two of the same transparent block
// (two water cells, two glass cells) so we don't emit hidden geometry.
inline bool FaceVisible(BlockId self, BlockId neighbor) {
    return IsTransparent(neighbor) && neighbor != self;
}

That is the whole interface the next lessons lean on. The naïve mesher asks FaceVisible six times per voxel and reads FaceTexture for the faces it keeps; flood-fill lighting uses IsTransparent to decide where light spreads and LightEmission to seed the bright cells; physics and picking use IsSolid. One small table, consumed everywhere.

Packing an id and a little state into 16 bits

A raw id is enough to say “this is stone.” But some blocks need a scrap of per-voxel state: a log's orientation, a water cell's fill level, a rotated stair. Rather than grow the voxel, we can carve a 16-bit word into a 12-bit id and 4 bits of state. Twelve bits gives 4096 distinct block types — plenty — and four bits gives sixteen states per block. 

// A stored voxel: a block id plus a little state, in one 16-bit word.
//   bits 0..11  -> block id   (up to 4096 types)
//   bits 12..15 -> state      (16 values: rotation, fill level, ...)
using PackedVoxel = uint16_t;

constexpr uint16_t VOXEL_ID_BITS  = 12;
constexpr uint16_t VOXEL_ID_MASK  = (1u << VOXEL_ID_BITS) - 1; // 0x0FFF

inline PackedVoxel PackVoxel(BlockId id, uint8_t state) {
    return static_cast<PackedVoxel>(
        (id & VOXEL_ID_MASK) |
        (static_cast<uint16_t>(state & 0xF) << VOXEL_ID_BITS));
}

inline BlockId VoxelId(PackedVoxel v)    { return static_cast<BlockId>(v & VOXEL_ID_MASK); }
inline uint8_t VoxelState(PackedVoxel v) { return static_cast<uint8_t>(v >> VOXEL_ID_BITS); }

Crucially, air is still zero: PackVoxel(BLOCK_AIR, 0) == 0, so a zeroed chunk stays all air even with packing. All the hot helpers keep working — the mesher calls VoxelId(v) first and hands the plain id to IsSolid and friends. Whether a chunk stores raw BlockIds or PackedVoxels is a decision the chunk makes; the registry never sees the state bits.

Two bytes per voxel still adds up — palette compression is coming

Even at 2 bytes, a loaded world of a few hundred chunks is well over 100 MB of raw voxels, most of it air or stone repeated endlessly. A later lesson introduces palette compression: a chunk stores a tiny per-chunk list of the block types it actually contains, then indexes it with just the bits it needs — often 4 bits or fewer per voxel. The global registry you built here is what those per-chunk palettes point back into.

Wiring it up

The registry is entirely platform-neutral — it holds no GPU handles, so there is no Direct3D 11 or Metal split here. You populate it once at startup, before any world loads, and from then on it is a read-only lookup that every subsystem shares. 

// --- startup, once, before loading any chunks ---
RegisterDefaultBlocks();

// --- anywhere in the hot paths, forever after ---
PackedVoxel v = chunk.At(x, y, z);
BlockId id    = VoxelId(v);

if (IsSolid(id)) {
    // collision, picking, ...
}
if (FaceVisible(id, neighborId)) {
    uint16_t layer = FaceTexture(id, FACE_POS_Y);
    // ... emit a top face using texture 'layer'
}
Recap
  • Storing a polymorphic object per voxel wastes tens of times the memory and pays a cache miss and vtable call on every query — never do it.
  • The flyweight pattern makes each voxel a tiny integer id (uint8_t/uint16_t) that indexes a shared table of BlockType descriptors.
  • A BlockType carries a name, independent solid and transparent flags, emissive/lightLevel, and a texture layer per face.
  • The BlockRegistry is a contiguous std::vector<BlockType>; the id is the index, AIR = 0 so a zeroed chunk is free air, and registration order is a contract.
  • Inline IsSolid / IsTransparent / LightEmission / FaceTexture helpers fold into the mesher and lighting loops with zero overhead.
  • Packing 12 bits of id and 4 bits of state into a 16-bit voxel adds per-block metadata for free, and foreshadows the palette compression a later lesson uses to shrink storage further.
