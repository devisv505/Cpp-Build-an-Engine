Infinite Worlds: Chunk Streaming & LOD

Your engine can now generate a chunk of terrain, light it, and turn it into a textured mesh — but only a fixed patch of it. Walk far enough and you either run off the edge of the world or you keep every chunk you have ever visited resident until memory fills up. This lesson makes the world feel endless on a small, fixed budget: keep only the chunks near the player in memory, stream new ones in as they walk, drop the ones they leave behind, and draw distant terrain at lower detail. The work is almost entirely bookkeeping — a map of chunks and a few rings around the camera — and getting those rings right is what separates a world that glides from one that stutters every time you cross a chunk border.

What you'll learn
  • Store resident chunks in a std::unordered_map keyed by chunk coordinate, with a fast custom hash — and why those lookups sit on the hot path.
  • Each frame, find the player's chunk and update the load, keep, and unload rings around it.
  • Use hysteresis — an unload radius larger than the load radius — so pacing a border never thrashes chunks in and out.
  • Fill the world toward the camera by loading nearest-first, and cap the work with a per-frame budget.
  • Coarsen distant chunks with level of detail, and hide the seams between LOD levels with skirts.

A window that follows the player

The core idea is a moving window. Pick a render distance R measured in chunks. At any moment the only chunks that need to exist are the ones within R of the player's current chunk; everything else can be generated later or freed now. As the player moves, the window slides with them: chunks entering it are streamed in, chunks leaving it are unloaded. The player is always at the centre of a bubble of loaded world that travels with them, so from the inside the world has no edge.

Streaming happens in the horizontal plane — you walk around, you rarely fly straight up — so we key the window on the (x, z) column and let each column own a vertical stack of chunk sections (more on that at the end). The reason we cannot simply crank R up to the horizon is that the count of chunks grows with the square of the radius:

82894,624~36 MB
161,08917,424~136 MB
324,22567,600~528 MB

Doubling R roughly quadruples the chunk count, and the mesh and lighting data for each chunk roughly doubles the memory in the last column again. (A circular window trims about 21% off the square counts, which we will take for free below.) That quadratic wall is exactly why the two halves of this lesson exist: stream so you only pay for what is near, and use LOD so the far rings cost a fraction of a full chunk each.

Keying chunks by their coordinate

The resident world is a dictionary from a chunk coordinate to a chunk. A std::unordered_map is the natural fit — average O(1) insert, erase, and lookup — but it needs two things for a struct key: an equality operator, and a hash function. Here is the 2D column coordinate and a hash worth using: 

#include <cstdint>
#include <cstddef>

// The horizontal address of a column of chunks. y is handled by the vertical stack.
struct ColumnCoord {
    int x, z;
    bool operator==(const ColumnCoord& o) const noexcept {
        return x == o.x && z == o.z;
    }
};

struct ColumnCoordHash {
    std::size_t operator()(const ColumnCoord& c) const noexcept {
        // Pack the two 32-bit coords into one 64-bit value. This part is a perfect,
        // collision-free map as long as each coord fits in 32 bits.
        std::uint64_t h = (static_cast<std::uint64_t>(static_cast<std::uint32_t>(c.x)) << 32)
                        |  static_cast<std::uint64_t>(static_cast<std::uint32_t>(c.z));
        // Then finalize with a splitmix64 avalanche so adjacent coords scatter widely.
        h ^= h >> 30; h *= 0xBF58476D1CE4E5B9ull;
        h ^= h >> 27; h *= 0x94D049BB133111EBull;
        h ^= h >> 31;
        return static_cast<std::size_t>(h);
    }
};

Why bother mixing if the packing is already collision-free? Because the map does not hash straight to a slot — it takes hash % bucket_count, and bucket counts are usually powers of two, which throws away the high bits. Without the avalanche step, the packed x (living entirely in the high 32 bits) would never influence which bucket a chunk lands in, so an entire row of the world would collide into a handful of buckets. The splitmix rounds fold every input bit down into the low bits, so neighbouring chunks — the ones you look up together while meshing a border — spread out instead of piling up. You would then declare the map as std::unordered_map<ColumnCoord, std::unique_ptr<Column>, ColumnCoordHash>.

Map lookups are on the hot path

Reading a block across a chunk border — which the meshing and lighting passes do thousands of times per chunk — turns into a hash, a modulo, and a bucket walk. That is far more expensive than the multiply-add of an in-chunk index. Never call GetBlock(worldX, worldY, worldZ) in a tight loop; instead resolve the six neighbour chunk pointers once before the loop and index directly. Caching the current chunk pointer between accesses is the single biggest win in any voxel hot path.

Load, keep and unload rings

The naive rule — "load everything within R, unload everything past R" — has a nasty failure mode. Stand exactly on a chunk boundary and take one step back and forth: a column flickers from just-inside to just-outside R every frame, so you generate it, free it, generate it again — a stall on every step. The fix is hysteresis: make the unload radius a little larger than the load radius, leaving a neutral keep band between them where chunks are neither created nor destroyed.

Every distance test in the streaming loop compares a chunk's offset from the centre against a radius. We never actually need the distance — only whether it is inside a ring — so we compare squared distances and skip the square root entirely: 

// Is column c inside a ring of the given radius (in chunks) around the centre?
static inline bool WithinRadius(ColumnCoord c, ColumnCoord center, int radius) {
    int dx = c.x - center.x;
    int dz = c.z - center.z;
    return dx * dx + dz * dz <= radius * radius;   // integer math, no sqrt
}
Compare squared distances

Because a < b iff a² < b² for non-negative values, you can do every ring test, LOD selection, and nearest-first sort on dx*dx + dz*dz in pure integer arithmetic. No floats, no sqrt, and no rounding surprises on the exact boundary — which is precisely where a mis-rounded distance would cause the flicker hysteresis is meant to prevent.

Unloading a neighbour a chunk still needs leaves holes

Hidden-face removal and flood-fill lighting both read one voxel past the chunk border to decide whether a boundary face is visible and how bright it is. If you free a chunk while a still-resident neighbour has already been meshed against it, that neighbour's border now reads as air: you get missing faces (holes you can see the sky through) or a dark seam of wrong lighting. This is why the unload radius must sit comfortably outside the load radius, and why unloading a chunk should mark its resident neighbours dirty so their borders re-mesh — or, cheaper, keep a one-voxel border copy so the mesh does not depend on the neighbour staying alive.

Nearest-first, and a per-frame budget

When the player teleports or spins the camera, dozens of columns can enter the load ring at once. Meshing them all in a single frame would freeze the game for a beat. Two rules keep it smooth. First, process the nearest missing column first so the world fills in toward the camera — the player sees the ground under their feet appear immediately and the far horizon fill in over the next second, which reads as natural. Second, budget: load at most N columns per frame and let the rest wait. The window is a set, not a queue, so nothing is lost by deferring — next frame the still-missing columns are simply still missing, and the nearest of them go next.

To do both, we collect the missing coordinates, sort them by squared distance, and take the first N. That sort is the reason we gather into a std::vector before touching the map.

The ChunkManager

Here is the whole thing assembled. Update runs once per frame with the player's world position and does three passes: unload what has drifted out, find what is missing, and load the nearest few within budget. 

#include <unordered_map>
#include <vector>
#include <memory>
#include <algorithm>
#include <cmath>

class ChunkManager {
public:
    // Tunables, all in chunks. unloadRadius > loadRadius gives us the hysteresis band.
    int loadRadius       = 12;
    int unloadRadius     = 14;
    int maxLoadsPerFrame = 4;

    void Update(const Vector3& playerWorldPos);

private:
    // One resident column: a vertical stack of chunk sections plus its built mesh + state.
    struct Column {
        Mesh mesh;
        bool meshed = false;
        int  lod    = 0;
        // ... unique_ptr<Chunk> sections[SECTIONS] ...
    };

    std::unordered_map<ColumnCoord, std::unique_ptr<Column>, ColumnCoordHash> columns_;

    // Player world position -> the chunk column they stand in. The shift is the
    // floor-divide-by-CHUNK_SIZE trick from the coordinates lesson (correct for negatives).
    ColumnCoord CenterColumn(const Vector3& p) const {
        int bx = static_cast<int>(std::floor(p.x));
        int bz = static_cast<int>(std::floor(p.z));
        return { bx >> CHUNK_BITS, bz >> CHUNK_BITS };
    }

    std::unique_ptr<Column> LoadColumn(ColumnCoord c);   // generate (or read from disk) + mesh
};
void ChunkManager::Update(const Vector3& playerWorldPos) {
    const ColumnCoord center = CenterColumn(playerWorldPos);

    // 1) UNLOAD: free every column that has drifted beyond the (larger) unload radius.
    const int unloadSq = unloadRadius * unloadRadius;
    for (auto it = columns_.begin(); it != columns_.end(); ) {
        const int dx = it->first.x - center.x;
        const int dz = it->first.z - center.z;
        if (dx * dx + dz * dz > unloadSq) {
            // (Save lesson: if this column was edited, write it to disk here first.)
            it = columns_.erase(it);          // unique_ptr frees voxel data + GPU mesh
        } else {
            ++it;
        }
    }

    // 2) FIND MISSING: every column inside the (smaller) load radius that isn't resident.
    const int loadSq = loadRadius * loadRadius;
    std::vector<ColumnCoord> missing;
    for (int dz = -loadRadius; dz <= loadRadius; ++dz) {
        for (int dx = -loadRadius; dx <= loadRadius; ++dx) {
            if (dx * dx + dz * dz > loadSq) continue;      // circular window
            ColumnCoord c { center.x + dx, center.z + dz };
            if (columns_.find(c) == columns_.end())
                missing.push_back(c);
        }
    }

    // 3) PRIORITIZE: nearest-first, so the world fills toward the camera.
    std::sort(missing.begin(), missing.end(),
        [&](const ColumnCoord& a, const ColumnCoord& b) {
            const int da = (a.x - center.x) * (a.x - center.x) + (a.z - center.z) * (a.z - center.z);
            const int db = (b.x - center.x) * (b.x - center.x) + (b.z - center.z) * (b.z - center.z);
            return da < db;
        });

    // 4) BUDGET: load at most maxLoadsPerFrame this frame; the rest wait for later frames.
    const int budget = std::min<int>(maxLoadsPerFrame, static_cast<int>(missing.size()));
    for (int i = 0; i < budget; ++i) {
        columns_.emplace(missing[i], LoadColumn(missing[i]));
    }
}

Notice how naturally the hysteresis falls out: the unload pass tests against unloadRadius and the load pass against the smaller loadRadius, so a column sitting in the keep band satisfies neither test and is simply left alone. There is no explicit "keep" list — the gap between the two radii is the keep zone.

The smooth-streaming recipe

Hysteresis margin (no thrashing at borders) + nearest-first (the world fills toward you) + a per-frame budget (no single-frame stall) together give you streaming the player never notices. The one thing this version still does wrong is that LoadColumn generates and meshes on the main thread, so a big budget still hitches. That is exactly the problem the next lesson solves by handing LoadColumn's work to a job system running on background threads — then you can raise the budget without touching the frame time.

Level of detail: coarser chunks far away

A chunk sixty blocks away and a chunk six hundred blocks away are drawn at wildly different sizes on screen, yet both cost a full mesh. Level of detail fixes that: pick a coarseness for each column from its distance to the player, so the far rings are cheap. Because we already compute a squared distance for streaming, LOD selection is a couple of comparisons: 

// Choose a detail level for a column from its chunk-distance to the player.
int LodForColumn(ColumnCoord c, ColumnCoord center) {
    int dx = c.x - center.x, dz = c.z - center.z;
    int distSq = dx * dx + dz * dz;
    if (distSq <  6 * 6)  return 0;   // full detail near the camera
    if (distSq < 12 * 12) return 1;   // half resolution
    return 2;                         // quarter resolution
}

There are a few ways to make a mesh coarser. You can run the same greedy mesher on a down-sampled copy of the voxels — merge each 2×2×2 group into one block and mesh the resulting 8³ grid at double voxel scale; you can merge several neighbouring chunks into one coarse mesh to cut draw calls; or you can simply mesh at half the vertex density. Down-sampling is the most direct and composes with everything you have already built: 

// Build a coarse voxel buffer: each 2x2x2 group of source voxels becomes one.
// The result is (CHUNK_SIZE/2)^3 voxels; the mesher emits its faces at 2x scale.
std::vector<BlockId> Downsample2x(const Chunk& src) {
    const int N = CHUNK_SIZE / 2;                       // 8 for a 16^3 chunk
    std::vector<BlockId> coarse(N * N * N, 0);
    for (int z = 0; z < N; ++z)
    for (int y = 0; y < N; ++y)
    for (int x = 0; x < N; ++x) {
        BlockId chosen = 0;                             // pick any solid block in the group
        for (int o = 0; o < 8 && chosen == 0; ++o) {
            int ox = o & 1, oy = (o >> 1) & 1, oz = (o >> 2) & 1;
            chosen = src.GetBlock(x * 2 + ox, y * 2 + oy, z * 2 + oz);
        }
        coarse[x + N * (y + N * z)] = chosen;           // 0 stays air only if all 8 were air
    }
    return coarse;
}

The catch with any LOD scheme is cracks. Where a full-detail chunk meets a half-resolution neighbour, their surface vertices no longer line up along the shared edge: the fine side has a vertex the coarse side skipped, so the two surfaces leave a thin gap you can see the sky (or the void) through. The standard, cheap fix is a skirt — a vertical wall of geometry dropped straight down from the chunk's border edge. It does not try to match the neighbour; it just hangs below the surface far enough that whatever gap opens up is covered by wall instead of a hole: 

// Cover LOD cracks: drop a vertical wall along a chunk-border edge loop.
// 'depth' must exceed the largest surface-height gap between adjacent LOD levels.
void AddSkirt(Mesh& mesh, const std::vector<Vector3>& edge, float depth) {
    for (std::size_t i = 0; i + 1 < edge.size(); ++i) {
        Vector3 a = edge[i];
        Vector3 b = edge[i + 1];
        Vector3 aDown { a.x, a.y - depth, a.z };
        Vector3 bDown { b.x, b.y - depth, b.z };
        mesh.AddQuad(a, b, bDown, aDown);      // one downward-facing wall segment
    }
}

Skirts are almost free — a thin ribbon of quads around each chunk's four sides — and because they are textured like the block wall they hide behind, the player never notices them. There are fancier fixes (stitching the two edges together vertex-by-vertex, or clamping the fine edge to the coarse one), but for a Minecraft-style world skirts are the pragmatic default, and they hide streaming pop-in at the same time.

Scaling up: vertical stacks and sparse octrees

Two structural notes to close on. First, vertical stacking: because the world has real height, a column is not one chunk but a stack of chunk sections — a 256-block-tall world is 16 sections of 16³ each. Streaming stays 2D (you stream columns, not individual sections), but generation, meshing, and the memory table above all multiply by the number of sections. Sections that are entirely air can be left un-allocated, so a column above open sky costs almost nothing.

Second, the flat unordered_map is perfect for the modest render distances a player-following window needs, but it scales linearly with the number of resident chunks. For very large view distances — flight sims, planet-scale terrain, or an editor that wants to see for kilometres — the usual move is a sparse voxel octree: a tree that stores empty regions as a single node and refines only where there is detail, which naturally encodes LOD (a shallower node is a coarser sample) and keeps far-away emptiness nearly free.

Don't reach for the octree yet

A sparse octree is more code and trickier to edit and re-mesh incrementally than a flat map of chunks. For a render distance of 8–32 chunks — the range a Minecraft-style game actually uses — the unordered_map window in this lesson is both simpler and faster. Treat the octree as the tool you graduate to when your view distance outgrows what a flat map of resident chunks can hold, not the one you start with.

Recap
  • An infinite world is a moving window of chunks: keep the ones within render distance R of the player resident and free the rest. Chunk count grows with R², so streaming plus LOD is not optional at large distances.
  • Store resident chunks in a std::unordered_map keyed by a ColumnCoord, with a pack-then-avalanche hash so adjacent chunks spread across buckets. Cross-border lookups are on the hot path — cache neighbour pointers instead of hashing per access.
  • Hysteresis — an unload radius larger than the load radius — leaves a keep band that stops a player pacing a border from thrashing chunks in and out. The gap between the two radii is the keep zone; no explicit list needed.
  • Load nearest-first so the world fills toward the camera, and cap it with a per-frame budget so a teleport or fast turn never stalls a frame. Do every ring/LOD/sort test on squared integer distances.
  • LOD meshes distant columns coarser (down-sample 2×2×2, merge chunks, or halve density); the cracks between LOD levels are hidden by a cheap vertical skirt dropped along each chunk border.
  • Beware unloading a chunk a still-resident neighbour was meshed against — it leaves holes and dark seams; re-mesh dirty borders or keep a one-voxel border copy. The synchronous LoadColumn here is what the next lesson moves onto a job system.
