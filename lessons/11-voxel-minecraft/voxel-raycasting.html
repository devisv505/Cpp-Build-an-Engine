Block Picking with Voxel Raycasting (DDA)

Point the camera at a block and left-click: it breaks. Right-click the space in front of it: a block appears. Both actions rest on one question — which voxel is the player looking at? The answer is a ray cast from the camera into the grid, and there is exactly one correct way to walk a ray through a voxel world: the Amanatides & Woo traversal, a grid DDA that visits every cell the ray crosses, in order, exactly once. In this lesson we implement it in full, recover the hit block, the face normal, and the empty cell to build into — then wire break and place into the job system so edits relight and re-mesh the affected chunks.

What you'll learn

  • The Amanatides & Woo grid DDA precisely: step, tDelta and tMax, and how to initialise each from the ray's fractional start.
  • The full RaycastVoxel returning the hit block, the empty cell to place into, the face normal, and the distance.
  • How to handle the sharp edges: a zero direction component, and starting inside a solid block.
  • Wiring left-click break and right-click place to relight and re-mesh the edited chunk — and its neighbours when the edit lands on a border.

Why not just march the ray?

The obvious idea is to sample the ray at fixed intervals: step a small amount along the direction, floor the position to a voxel, and check if it is solid. It looks like this — and it is a trap. 

// Tempting, and wrong: march the ray in fixed steps.
glm::vec3 dir = glm::normalize(lookDir);
for (float t = 0.0f; t < maxDistance; t += 0.1f) {
    glm::vec3  p    = origin + dir * t;
    glm::ivec3 cell = glm::ivec3(glm::floor(p));
    if (world.isSolid(cell)) {
        // We're somewhere *inside* this block. But which face did we enter
        // through? We can't say — so we can't place a block against it. And
        // did the last step tunnel straight through a thin cell we skipped?
        break;
    }
}

Both failure modes are structural, not tuning problems. Too large a step and the ray tunnels: it jumps over a one-voxel-thick wall entirely, sampling air on both sides. Too small a step and you test the same cell dozens of times — pure wasted work — and you still land at some arbitrary point inside a block, so you cannot recover the face you crossed to get there. But placing a block needs exactly that face: the new block goes into the empty cell on the near side of it. No fixed step can give you an exact, ordered, once-each walk of the cells a ray passes through. A grid DDA can.

The algorithm has a name and a paper

This is the traversal from John Amanatides and Andrew Woo, "A Fast Voxel Traversal Algorithm for Ray Tracing" (Eurographics, 1987). It is a 3D digital differential analyser: it walks the exact sequence of voxels a ray intersects in front-to-back order, and its cost is O(number of cells crossed) — independent of how far apart the solid blocks are. Every voxel engine on the planet uses it for block picking.

The idea: cross one boundary at a time

A voxel at integer coordinates (x, y, z) occupies the unit cube from (x, y, z) to (x+1, y+1, z+1). As the ray moves forward it is boxed inside its current voxel by three pairs of planes — the cell walls perpendicular to each axis. To leave the cell it must cross one of those walls. The whole algorithm is: of the next wall on each axis, cross the nearest one, move into the neighbouring voxel across it, and repeat. Because we always advance to the closest boundary, we can never skip a cell and never revisit one.

We track distance along the ray with the parameter t, where a point on the ray is origin + t · dir. Three quantities drive the loop, one per axis:

QuantityMeaningValue (per axis)
stepWhich direction we walk cells on this axis.+1 if dir > 0, -1 if dir < 0
tDeltaHow much t advances to cross one full cell on this axis.abs(1 / dir)
tMaxThe value of t at which the ray reaches the first cell boundary on this axis.from the fractional start (below)

tDelta is the reciprocal of the direction's magnitude on that axis because a bigger component means the ray sweeps across a cell's width in less parameter distance. tMax is the only value that depends on where inside the start cell the ray begins. If we are stepping in the positive direction, the next boundary is at floor(o) + 1, so the distance to it is (floor(o) + 1 − o) / dir. If we are stepping negative, the next boundary is the lower wall at floor(o), giving (o − floor(o)) / (−dir). Both expressions are non-negative, as a distance must be.

Each loop iteration picks the axis with the smallest tMax — that is the wall the ray reaches next — steps the voxel one cell along that axis, then pushes that axis's tMax forward by its tDelta so it now points at the following boundary on the same axis. The face we just crossed has an outward normal of −step on that axis, and the empty cell we came from is the neighbour on the normal side.

Setting up the traversal

Let's fix the return type first. A ray query answers four things at once: whether it hit, the solid block it struck, the empty cell to build into, and the face normal — plus the distance, handy for a reach limit or a highlight. 

#include <glm/glm.hpp>

struct VoxelHit {
    bool       hit        = false;   // did the ray reach a solid block?
    glm::ivec3 block      = {0,0,0}; // the solid voxel it struck (break this)
    glm::ivec3 placeBlock = {0,0,0}; // the empty voxel just before it (place here)
    glm::ivec3 normal     = {0,0,0}; // outward normal of the face we entered
    float      distance   = 0.0f;    // t along the ray at the hit
};

VoxelHit RaycastVoxel(const World& world, glm::vec3 origin,
                      glm::vec3 dir, float maxDistance);

The per-axis setup is a short loop. We floor the origin to find the starting voxel, then compute step, tDelta, and the initial tMax for each of the three axes. The one branch that matters is dir[a] == 0: a ray parallel to an axis never crosses that axis's walls, so we set both tDelta and tMax to infinity, which guarantees that axis is never the smallest and is never chosen. 

#include "VoxelRaycast.h"
#include <cmath>
#include <limits>

VoxelHit RaycastVoxel(const World& world, glm::vec3 origin,
                      glm::vec3 dir, float maxDistance) {
    const float kInf = std::numeric_limits<float>::infinity();

    // The voxel the ray starts in.
    glm::ivec3 voxel = glm::ivec3(glm::floor(origin));

    glm::ivec3 step;    // +1 / -1 / 0 per axis
    glm::vec3  tDelta;  // t to cross one whole cell
    glm::vec3  tMax;    // t to the first boundary

    for (int a = 0; a < 3; ++a) {
        if (dir[a] > 0.0f) {
            step[a]   = 1;
            tDelta[a] = 1.0f / dir[a];
            tMax[a]   = (std::floor(origin[a]) + 1.0f - origin[a]) / dir[a];
        } else if (dir[a] < 0.0f) {
            step[a]   = -1;
            tDelta[a] = 1.0f / -dir[a];
            tMax[a]   = (origin[a] - std::floor(origin[a])) / -dir[a];
        } else {
            step[a]   = 0;
            tDelta[a] = kInf;   // never crosses a wall on this axis...
            tMax[a]   = kInf;   // ...so it is never the smallest tMax
        }
    }
    // ... loop below ...
A zero direction component must be set to infinity explicitly

Do not try to fold the zero case into the general formula. A shortcut like tMax[a] = (origin[a] - std::floor(origin[a])) * tDelta[a] looks tidy, but when dir[a] is exactly 0, tDelta[a] is +inf and the fractional part can be 0 — and 0 * inf is NaN. A NaN in tMax poisons every comparison (all of <, >, == return false), the wrong axis gets chosen, and the traversal walks off in a straight line forever. Branch on the sign and assign infinity by hand, as above.

The traversal loop

Now the heart of it. Each iteration finds the axis with the smallest tMax with a pair of comparisons, checks that we have not exceeded the ray's reach, then steps that axis. Crucially the block test happens after stepping — we test the voxel we just entered. The face normal is −step on the stepped axis, and the empty cell we came from is voxel + normal, which is exactly the previous voxel. 

    // ... continued from the setup above ...

    glm::ivec3 normal = {0,0,0};
    while (true) {
        // Which boundary does the ray reach next? Pick the smallest tMax.
        int axis;
        if (tMax.x < tMax.y) axis = (tMax.x < tMax.z) ? 0 : 2;
        else                 axis = (tMax.y < tMax.z) ? 1 : 2;

        // If even the nearest boundary is past our reach, we hit nothing.
        if (tMax[axis] > maxDistance) break;

        // Cross that boundary: advance one cell along this axis.
        voxel[axis] += step[axis];

        // The face we entered through points back the way we came.
        normal = {0,0,0};
        normal[axis] = -step[axis];

        float dist   = tMax[axis];    // distance to the boundary we just crossed
        tMax[axis]  += tDelta[axis];  // advance to the NEXT boundary on this axis

        // Test the voxel we just stepped into.
        if (world.isSolid(voxel)) {
            VoxelHit h;
            h.hit        = true;
            h.block      = voxel;            // break this block
            h.placeBlock = voxel + normal;   // the empty cell we came from
            h.normal     = normal;
            h.distance   = dist;
            return h;
        }
    }
    return VoxelHit{};   // reached maxDistance without hitting anything
}

That is the entire algorithm — no floating-point accumulation drift, no re-testing a cell, no skipped cells. It visits precisely the voxels the ray passes through, front to back, and stops at the first solid one. Because placeBlock is voxel + normal, it is guaranteed to be the empty cell the ray occupied one step earlier, which is exactly where a placed block should land: flush against the face the player is aiming at.

We step before we test — so the start voxel is never reported

The loop advances a cell and then checks solidity, which means the voxel the camera is standing in is never returned as a hit. For block picking that is what you want: you should not be able to break the block your own head is inside. But if you reuse this routine for collision or a light probe that starts inside geometry, add a single test of the start voxel before the loop — if (world.isSolid(voxel)) { /* already embedded */ } — because the traversal will otherwise walk right out of it and report the first solid block it meets on the way.

Reading the world in world space

RaycastVoxel leans on one method: World::isSolid(glm::ivec3), taking an absolute world voxel coordinate. That is the same accessor the ambient-occlusion mesher used, and it decomposes into the coordinate helpers from Voxels, Chunks & World Coordinates plus the registry query from the block-registry lesson. It must read across chunk boundaries transparently — the ray does not care where one chunk ends and the next begins. 

// One block by absolute world coordinate. Unloaded space reads as air.
BlockId World::getBlock(glm::ivec3 wp) const {
    ChunkCoord   cc = WorldToChunk(wp.x, wp.y, wp.z);
    const Chunk* c  = getChunk(cc);        // hash-map lookup; null if not loaded
    if (!c) return BLOCK_AIR;
    LocalCoord lc = WorldToLocal(wp.x, wp.y, wp.z);
    return c->GetBlock(lc.x, lc.y, lc.z);
}

// Does the ray stop here? Ask the registry, not the raw id.
bool World::isSolid(glm::ivec3 wp) const {
    return IsSolid(getBlock(wp));          // Blocks().Get(id).solid, inlined
}
One traversal, many uses

This exact walk powers far more than block picking. Swap the "is it solid?" test for "is it opaque?" and you have line-of-sight for AI or a sunlight check. March it toward a light and sum the blocks in the way for cheap light sampling. Run it along a movement vector and you have a swept collision query against the grid. It is exact and its cost is only the number of cells crossed, so it stays fast even over long distances — the same reason it is the standard voxel ray-tracing primitive.

Breaking and placing blocks

With the hit in hand, the interaction is short. Cast the ray from the camera's world position along its look direction each frame the player clicks, then act on block or placeBlock. Normalise the direction so maxDistance and distance come out in world units — a reach of 6.0f then means six blocks. 

// Called when the player clicks. camera/input come from modules 07–08.
void UpdateBlockInteraction(World& world, const Camera& camera,
                            const Input& input, BlockId heldBlock) {
    glm::vec3 origin = camera.WorldPosition();          // from the Transform
    glm::vec3 dir    = glm::normalize(camera.Forward()); // look direction

    VoxelHit hit = RaycastVoxel(world, origin, dir, /*maxDistance=*/ 6.0f);
    if (!hit.hit) return;                                // aiming at the sky

    // (Draw a wireframe box at hit.block here for the selection outline.)

    if (input.MousePressed(MouseButton::Left)) {
        world.editBlock(hit.block, BLOCK_AIR);           // break: turn to air
    } else if (input.MousePressed(MouseButton::Right)) {
        world.editBlock(hit.placeBlock, heldBlock);      // place into empty cell
    }
}

The real work is in editBlock. Writing the block into its chunk is one line; the rest is keeping the rendered world consistent. A changed voxel invalidates its chunk's lighting (light no longer flows the same way) and its mesh (a face appeared or vanished). And if the edited voxel sits on a chunk face, the neighbouring chunk meshed against it — its border faces were culled or kept based on this voxel — so that neighbour must be rebuilt as well. Forget the neighbour and you get a hole or a stray face straddling the chunk seam. 

void World::editBlock(glm::ivec3 wp, BlockId id) {
    ChunkCoord cc    = WorldToChunk(wp.x, wp.y, wp.z);
    Chunk*     chunk = getChunk(cc);
    if (!chunk) return;                    // don't edit space that isn't loaded

    LocalCoord lc = WorldToLocal(wp.x, wp.y, wp.z);
    chunk->SetBlock(lc.x, lc.y, lc.z, id);

    // The edited chunk always needs a relight + remesh.
    markChunkDirty(cc);

    // A voxel on a chunk face changes a border block the NEIGHBOUR meshes
    // against, so rebuild that neighbour too. Check each of the six faces.
    if (lc.x == 0)              markChunkDirty({cc.x - 1, cc.y, cc.z});
    if (lc.x == CHUNK_SIZE - 1) markChunkDirty({cc.x + 1, cc.y, cc.z});
    if (lc.y == 0)              markChunkDirty({cc.x, cc.y - 1, cc.z});
    if (lc.y == CHUNK_SIZE - 1) markChunkDirty({cc.x, cc.y + 1, cc.z});
    if (lc.z == 0)              markChunkDirty({cc.x, cc.y, cc.z - 1});
    if (lc.z == CHUNK_SIZE - 1) markChunkDirty({cc.x, cc.y, cc.z + 1});
}

markChunkDirty is where this meets the job system from the previous lesson. Relighting and re-meshing are expensive and must not stall the frame, so we hand them to a worker thread. The order is fixed: relight first, mesh second, because the mesher bakes light and ambient-occlusion values into the vertices. Only the final GPU upload comes back to the render thread — creating buffers is a Direct3D 11 immediate-context (or Metal encoder) operation and cannot happen off it. 

void World::markChunkDirty(ChunkCoord cc) {
    if (!getChunk(cc)) return;             // neighbour off the edge of the world

    // Worker thread: the flood-fill lighting and greedy mesher from the last
    // few lessons, both pure CPU work on chunk data.
    m_jobs.submit([this, cc] {
        relightChunk(cc);                          // flood-fill lighting lesson
        ChunkMeshData mesh = buildChunkMesh(*this, cc);  // greedy mesh + AO

        // GPU upload must run on the render thread (D3D11 / Metal), so post
        // the finished CPU buffers back and let it create the vertex/index
        // buffers next frame.
        m_mainThread.post([this, cc, mesh = std::move(mesh)]() mutable {
            uploadChunkMesh(cc, std::move(mesh));
        });
    });
}
Border edits ripple further than six faces if you bake AO

The six-face check above is exactly right for hidden-face removal, where a chunk's mesh depends only on its face-adjacent neighbours. But ambient occlusion samples diagonal neighbours, which can live in an edge- or corner-adjacent chunk. If you bake AO, an edit near a chunk corner can change up to seven neighbours' meshes. The simple, robust fix is to dirty every chunk touched by the edited voxel's 3×3×3 neighbourhood; the cost is a few extra remeshes on rare corner edits, and the job system absorbs them.

Putting it together

The full loop is now: every frame, if the player clicked, cast one ray with RaycastVoxel; on a hit, break block or place at placeBlock through editBlock; that marks the chunk (and any bordering neighbour) dirty; the job system relights and re-meshes on a worker; and the render thread picks up the new buffers next frame. One exact traversal, and a clean path from a mouse click all the way to updated geometry on screen — with the heavy work kept off the main thread so the world never hitches when you dig.

Recap
  • Fixed-step ray marching is unusable for picking: it tunnels through thin cells, re-tests cells, and can't recover the entry face needed to place a block.
  • The Amanatides & Woo grid DDA (1987) walks every voxel a ray crosses, in order, exactly once, at a cost of O(cells crossed).
  • Per axis: step = sign(dir), tDelta = abs(1/dir), and tMax is the distance to the first boundary from the fractional start; each step crosses the axis with the smallest tMax.
  • On a hit, normal = −step on the stepped axis and placeBlock = block + normal — the empty cell the ray came from, exactly where a placed block belongs.
  • Guard dir[a] == 0 by setting tDelta and tMax to infinity (never 0 * inf, which is NaN), and remember the start voxel is never reported since we step before testing.
  • Break and place both relight and re-mesh the edited chunk through the job system — and its neighbours whenever the edit lands on a chunk border.
