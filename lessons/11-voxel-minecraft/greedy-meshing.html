Greedy Meshing

Hidden face removal already threw away every buried face, and our chunks render far fewer triangles than the naïve mesher. But a flat surface still costs us one quad per block: a plain 16×16 stone wall is 256 identical quads sitting edge to edge. In this lesson we implement greedy meshing, Mikola Lysenko's classic algorithm, which fuses adjacent coplanar faces that share the same attributes into a single large rectangle. That wall drops from 256 quads to exactly one, and MiniEngine's chunks get dramatically cheaper to draw.

What you'll learn
  • Why hidden face removal still leaves flat terrain paying one quad per block.
  • How the greedy sweep walks the three axes and builds a 2D face mask per slice.
  • What belongs in the merge key — block id, texture layer, AO, light and face direction — and why leaving any of it out causes seams.
  • The exact greedy consume: grow width, then height, emit one quad, zero the rectangle.
  • How to compute a merged quad's corners, winding and tiling UVs, and why tiling needs a texture array, not an atlas.

One quad per block is too many

After the previous lesson, a chunk only emits faces that touch air — no buried geometry survives. That is a huge saving over the naïve mesher, but it does nothing for large flat areas, which are exactly what Minecraft-style worlds are made of: ground, walls, cliff faces, the ceiling of a cave. Every one of those blocks still contributes its own little quad, even though its neighbours are identical.

The insight is simple. If two neighbouring faces lie in the same plane, point the same way, and look exactly the same — same block, same texture, same lighting — then a player cannot tell whether they are two quads or one. So we merge them. Do that across a whole flat region and thousands of tiny faces collapse into a handful of big ones.

The idea: merge coplanar faces

Greedy meshing works one slice at a time. Pick an axis — say X — and imagine sweeping a plane through the chunk perpendicular to it. At each position of that plane you have a flat 2D grid of the faces that cross it. That grid is a mask: each cell either holds a face or is empty. The whole trick is turning that 2D mask into as few rectangles as possible.

The greedy rule is: scan the mask cell by cell; at the first face you meet, extend a rectangle as far right as the faces keep matching, then extend it downward as long as every cell of that full-width row still matches; emit one quad for the rectangle and erase those cells so they are never reused. Repeat until the mask is empty. A cell only merges with its neighbour when their keys are identical, so a single odd block splits the sweep into more pieces — but flat regions collapse almost completely.

The merge key: what has to match

Everything hinges on when two faces are allowed to merge. Get this wrong and the mesh is smaller but visibly broken — textures on the wrong block, lighting that jumps across a seam. So we define a FaceKey that captures every property that must be identical before two faces can fuse. Two faces merge only when their keys are equal. 

// Everything that must match for two faces to merge into one quad.
// Two adjacent faces fuse only when their FaceKeys are equal.
struct FaceKey {
    BlockId  block = BlockId::Air; // which block owns the face
    uint16_t layer = 0;            // texture-array slice for THIS face
    uint8_t  light = 0;            // baked light level (0..15)
    uint8_t  ao    = 0;            // 4 corners packed, 2 bits each
    int8_t   dir   = 0;            // +1 = face points along +axis, -1 = -axis, 0 = empty

    bool empty() const { return dir == 0; }

    bool operator==(const FaceKey& o) const {
        return block == o.block && layer == o.layer &&
               light == o.light && ao == o.ao && dir == o.dir;
    }
    bool operator!=(const FaceKey& o) const { return !(*this == o); }
};
Never merge across different AO, light, or texture

It is tempting to key only on the block id. Don't. If you merge two faces with different ambient occlusion, the darkening at a corner smears across the whole big quad and you get a lighting seam. Merge across different light levels and a shadow boundary vanishes. Merge across different texture layers and the wrong tile appears. Every one of those must sit in the key. This is precisely why the Voxel Ambient Occlusion and Flood-Fill Lighting lessons feed their per-face values back into this struct — the key is the contract that keeps the merged mesh looking identical to the unmerged one.

Building the key for one face reads the block's per-face texture layer from the registry we built in the block registry lesson, plus the light and AO sampled from the empty neighbour on the other side of the face. For now those baked values default to zero (fully lit, no occlusion); the lighting and AO lessons replace the zeros with real samples. Because they already live in the key, no code here changes when they do — faces with different values simply stop merging. 

// Build the merge key for the face of `block` at (x,y,z), on axis `d`
// (0=X,1=Y,2=Z) facing direction `dir` (+1 or -1).
FaceKey makeFaceKey(const Chunk& chunk, int x, int y, int z, int d, int dir) {
    BlockId block = chunk.blockAt(x, y, z);

    FaceKey key;
    key.block = block;
    key.dir   = static_cast<int8_t>(dir);

    // Face index: +X,-X,+Y,-Y,+Z,-Z -> 0..5. The texture layer differs per
    // face (grass top vs. dirt side vs. dirt bottom). See module 10.
    int face  = d * 2 + (dir > 0 ? 0 : 1);
    key.layer = BlockRegistry::get(block).faceLayer[face];

    // Baked in the AO / lighting lessons; zero = fully lit, no occlusion.
    key.light = 0;
    key.ao    = 0;
    return key;
}

Sweeping the three axes

A face can face along any of the three axes, so we run the sweep three times, once for each axis d ∈ {0, 1, 2}. The other two axes become the in-plane directions u = (d+1)%3 and v = (d+2)%3. With that choice the unit vectors line up neatly: e_u × e_v = e_d, which we will lean on later to get winding right without guesswork.

For a chunk of SIZE voxels along an axis there are SIZE + 1 boundary planes, so we sweep x[d] from -1 to SIZE - 1. At each plane we compare the voxel just below it (a, at x[d]) with the voxel just above it (b, at x[d]+1). If exactly one is solid there is a visible face; if both are solid the face is hidden; if both are empty there is nothing. The face's plane sits at coordinate x[d] + 1.

Why start at -1?

Starting the sweep at x[d] = -1 puts the first plane below the chunk, so the bottom faces of the lowest slice are captured, and ending at SIZE - 1 captures the top faces of the highest slice. Both facing directions fall out of one loop: when a is the solid one we get a +d face; when b is, a -d face. Because dir is in the key, the two directions never merge into each other even though they share the mask.

One honest caveat: chunk.blockAt must return the correct neighbour block at the chunk border, not air. If it returns air outside the chunk, two adjacent chunks each render the shared boundary and you get a doubled, z-fighting wall between them. The hidden face removal lesson set this up by sampling neighbouring chunks; greedy meshing relies on that same sampling.

Consuming the mask greedily

Here is the whole sweep. For each slice we first fill the mask, then greedily peel rectangles out of it. The mask is a flat array indexed n = i + j·SIZE, with i running along u (stride 1) and j along v (stride SIZE) — so "extend right" is n + w and "the row below" is n + h·SIZE. 

Mesh greedyMesh(const Chunk& chunk) {
    std::vector<VoxelVertex> verts;
    std::vector<uint32_t>    indices;

    const int S = Chunk::SIZE;
    std::vector<FaceKey> mask(S * S);   // reused scratch, one key per slice cell

    for (int d = 0; d < 3; ++d) {
        const int u = (d + 1) % 3;      // first in-plane axis
        const int v = (d + 2) % 3;      // second in-plane axis

        int x[3] = {0, 0, 0};           // the voxel on the -d side of the plane
        int q[3] = {0, 0, 0};           // unit step along d
        q[d] = 1;

        // SIZE + 1 boundary planes: x[d] from -1 up to SIZE - 1.
        for (x[d] = -1; x[d] < S; ++x[d]) {

            // ---- 1. Build the mask for this slice ----
            int n = 0;
            for (x[v] = 0; x[v] < S; ++x[v]) {
                for (x[u] = 0; x[u] < S; ++x[u], ++n) {
                    BlockId a = chunk.blockAt(x[0],        x[1],        x[2]);
                    BlockId b = chunk.blockAt(x[0]+q[0],   x[1]+q[1],   x[2]+q[2]);
                    bool aSolid = isOpaque(a);
                    bool bSolid = isOpaque(b);

                    if (aSolid == bSolid) {
                        mask[n] = FaceKey{};                 // hidden or empty: no face
                    } else if (aSolid) {
                        // a solid, b air -> +d face owned by a
                        mask[n] = makeFaceKey(chunk, x[0], x[1], x[2], d, +1);
                    } else {
                        // b solid, a air -> -d face owned by b
                        mask[n] = makeFaceKey(chunk, x[0]+q[0], x[1]+q[1], x[2]+q[2], d, -1);
                    }
                }
            }

            // ---- 2. Greedily carve the mask into quads ----
            n = 0;
            for (int j = 0; j < S; ++j) {
                for (int i = 0; i < S; /* i advances by w below */) {
                    if (mask[n].empty()) { ++i; ++n; continue; }

                    const FaceKey key = mask[n];

                    // Grow width along +u while the key keeps matching.
                    int w = 1;
                    while (i + w < S && mask[n + w] == key) ++w;

                    // Grow height along +v while an ENTIRE w-wide row matches.
                    int h = 1;
                    bool rowMatches = true;
                    while (j + h < S && rowMatches) {
                        for (int k = 0; k < w; ++k) {
                            if (mask[n + k + h * S] != key) { rowMatches = false; break; }
                        }
                        if (rowMatches) ++h;
                    }

                    // Emit one quad for the w x h rectangle.
                    int base[3];
                    base[d] = x[d] + 1;   // the plane's coordinate along d
                    base[u] = i;
                    base[v] = j;
                    emitQuad(verts, indices, d, u, v, base, w, h, key);

                    // Erase the consumed rectangle so it is never reused.
                    for (int l = 0; l < h; ++l)
                        for (int k = 0; k < w; ++k)
                            mask[n + k + l * S] = FaceKey{};

                    i += w;
                    n += w;
                }
            }
        }
    }
    return Mesh(std::move(verts), std::move(indices));
}

Two details make this correct. First, n stays in lock-step with i + j·SIZE because every row advances n by exactly SIZE — one per empty cell, w per emitted quad. Second, we only ever look right and down, and we erase cells the moment we consume them, so a cell that has already been claimed reads back as empty and can never join a later rectangle. That erase step is what keeps the same face from being emitted twice.

Flat is where greedy shines

Greedy meshing pays off most exactly where naïve meshing hurts most: broad flat surfaces — floors, walls, the tops of chunks, cliff faces. There it turns hundreds of quads into a handful. On jagged, noisy geometry the win is smaller, but it never makes the mesh worse — a lone unmergeable face just becomes a 1×1 quad, exactly what you had before.

Emitting the quad: corners, winding and UVs

A rectangle in the mask is described by its base corner (i, j) on the slice plane, plus its width w along u and height h along v. Turning that into four 3D corners is just adding w along u and h along v to the base. The four corners are base, base + w·e_u, base + w·e_u + h·e_v, and base + h·e_v.

Winding decides which side is the front face, and it must match the convention MiniEngine's cube factory uses: order the corners so that cross(c1 - c0, c2 - c0) points outward. Because we chose u and v so that e_u × e_v = e_d, the order [base, +u, +u+v, +v] gives an outward normal of +d. For a -d face we simply reverse the middle two, [base, +v, +u+v, +u], flipping the normal to -d. No trial and error.

The UVs are what make merging pay off visually. Instead of the usual 0..1 range, a merged quad's UVs run 0..w across the width and 0..h across the height. With a wrapping sampler that tiles the block's texture w times one way and h times the other, so a 16-wide merged floor shows sixteen tiles, not one stretched one. 

struct VoxelVertex {
    DirectX::XMFLOAT3 pos;
    DirectX::XMFLOAT3 normal;
    DirectX::XMFLOAT2 uv;    // runs 0..w and 0..h across the merged quad
    uint32_t          layer; // texture-array slice for this face
};

void emitQuad(std::vector<VoxelVertex>& verts, std::vector<uint32_t>& indices,
              int d, int u, int v, const int base[3], int w, int h,
              const FaceKey& key) {
    // Base corner and the two in-plane edge vectors.
    float p[3]  = { (float)base[0], (float)base[1], (float)base[2] };
    float du[3] = {0, 0, 0}; du[u] = (float)w;   // edge along u
    float dv[3] = {0, 0, 0}; dv[v] = (float)h;   // edge along v

    DirectX::XMFLOAT3 c00 = { p[0],                 p[1],                 p[2]                 };
    DirectX::XMFLOAT3 cW0 = { p[0]+du[0],           p[1]+du[1],           p[2]+du[2]           };
    DirectX::XMFLOAT3 cWH = { p[0]+du[0]+dv[0],     p[1]+du[1]+dv[1],     p[2]+du[2]+dv[2]     };
    DirectX::XMFLOAT3 c0H = { p[0]+dv[0],           p[1]+dv[1],           p[2]+dv[2]           };

    DirectX::XMFLOAT3 nrm = {0, 0, 0};
    if      (d == 0) nrm.x = (float)key.dir;
    else if (d == 1) nrm.y = (float)key.dir;
    else             nrm.z = (float)key.dir;

    const float fw = (float)w, fh = (float)h;
    uint32_t start = (uint32_t)verts.size();

    if (key.dir > 0) {
        // +d face: cross(c1-c0, c2-c0) points +d (front). UVs 0..w, 0..h.
        verts.push_back({ c00, nrm, {0.f, 0.f}, key.layer });
        verts.push_back({ cW0, nrm, {fw,  0.f}, key.layer });
        verts.push_back({ cWH, nrm, {fw,  fh }, key.layer });
        verts.push_back({ c0H, nrm, {0.f, fh }, key.layer });
    } else {
        // -d face: reverse winding so the front points -d.
        verts.push_back({ c00, nrm, {0.f, 0.f}, key.layer });
        verts.push_back({ c0H, nrm, {0.f, fh }, key.layer });
        verts.push_back({ cWH, nrm, {fw,  fh }, key.layer });
        verts.push_back({ cW0, nrm, {fw,  0.f}, key.layer });
    }

    // Two triangles, same 0,1,2 / 0,2,3 pattern as our cube factory.
    indices.insert(indices.end(), {
        start + 0, start + 1, start + 2,
        start + 0, start + 2, start + 3
    });
}
An atlas cannot tile — use a texture array

UVs of 0..w only tile correctly if the sampler can wrap a single tile. A texture atlas packs many tiles into one image, so a UV of 1.3 lands on the neighbouring tile, not a repeat of the same one — the merged quad shows a smear of other blocks. The fix is the texture array from module 10: each block face is its own full-image layer, and a wrap sampler tiles that layer cleanly. Greedy meshing is the reason we built texture arrays there.

Sampling a merged quad in the shader

The pixel shader samples a Texture2DArray with the per-vertex layer selecting the slice and the tiling UVs selecting the point within it. The layer is a flat integer, so it must not be interpolated across the triangle — mark it nointerpolation. The wrap sampler does the tiling for free; no frac() is needed. 

Texture2DArray gBlocks : register(t0);
SamplerState   gWrap   : register(s0); // AddressU = AddressV = WRAP

struct PSIn {
    float4 pos   : SV_POSITION;
    float3 nrm   : NORMAL;
    float2 uv    : TEXCOORD0;              // runs 0..w and 0..h
    nointerpolation uint layer : TEXCOORD1; // texture-array slice
};

float4 main(PSIn i) : SV_Target {
    // A WRAP sampler tiles the layer automatically across the merged quad.
    float3 albedo = gBlocks.Sample(gWrap, float3(i.uv, (float)i.layer)).rgb;
    return float4(albedo, 1.0);
}

The Metal backend is the same idea in Metal Shading Language: declare a texture2d_array<float>, create the sampler with s_address::repeat and t_address::repeat, and sample with the slice as the third argument — gBlocks.sample(gWrap, in.uv, in.layer). The wrap addressing is what tiles the merged UVs on both platforms; make sure your module 10 sampler was created with wrap, not clamp, addressing.

How much does it save?

Consider a 16×16×16 chunk whose bottom eight layers are solid stone and the top eight are air — a flat filled floor, the most common shape in a voxel world. Here is the quad count at each stage of the module, assuming the exposed surfaces share a uniform texture and lighting:

StageQuadsWhat it does
Naïve meshing12,288Every block emits all 6 faces (2,048 blocks × 6).
Hidden face removal1,024Only the box's surface faces survive: top + bottom + four sides.
Greedy meshing6Each flat face of the box merges into a single quad.

Real terrain never merges quite this perfectly — varied blocks, and the per-corner ambient occlusion we add in the next lessons, both split the mask into more rectangles. But the shape of the win holds: greedy meshing routinely cuts a chunk's quad count by a large factor, which means fewer vertices to transform, less bandwidth, and more headroom for draw distance. It is the single highest-leverage optimisation in a voxel renderer, and every property that could break it is safely encoded in the merge key.

Recap
  • Hidden face removal still costs one quad per block on flat surfaces; greedy meshing merges coplanar same-attribute faces into large rectangles.
  • The sweep runs once per axis d, building a 2D mask for each slice from x[d] = -1 to SIZE - 1 so both facing directions are captured.
  • A FaceKey — block, texture layer, AO, light and direction — decides when faces may merge; leaving any field out causes texture or lighting seams, which is why the AO and lighting lessons feed back into it.
  • Greedy consume: grow width along u, grow height along v while whole rows match, emit one quad, then erase the rectangle so it is never reused.
  • Merged UVs run 0..w and 0..h and tile with a wrap sampler, which requires a texture array (module 10), never an atlas; winding follows e_u × e_v = e_d, reversed for -d faces.
  • On flat terrain the savings are enormous — a 16³ half-filled chunk drops from 12,288 quads to 6.
