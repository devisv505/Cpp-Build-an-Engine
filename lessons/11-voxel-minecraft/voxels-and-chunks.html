Voxels, Chunks & World Coordinates

Welcome to module 11, where MiniEngine grows a Minecraft-style voxel world. Before we can mesh a single block or generate any terrain, we need the one thing everything else stands on: a data model for the world. This lesson is all about that model — what a voxel is, why an infinite world has to be sliced into fixed-size chunks, and the small pile of integer math that lets you turn "the block at world position (−37, 64, 12)" into an exact byte inside an exact chunk. Get this layer right and the rest of the module is a pleasure; get it subtly wrong and you will chase the same bug every voxel developer meets on day one.

What you'll learn
  • What a voxel is, and why the world is conceptually one enormous 3D grid of block ids.
  • Why a single giant array is impossible, and how fixed-size chunks solve it.
  • How to store a chunk as one flat, cache-friendly array and index it with x + SIZE*(y + SIZE*z).
  • The three coordinate spaces — world, chunk, and local — and how to convert between them.
  • The classic negative-coordinate bug, its correct floorDiv/floorMod fix, and the power-of-two shift/mask shortcut.

What is a voxel?

A voxel is a "volume pixel": a pixel is a colored square on a 2D grid, and a voxel is a block sitting on a regular 3D grid. In a Minecraft-style world every position (x, y, z) in that grid holds exactly one block — stone, dirt, air, water — and each block type is identified by a small integer, its block id. By convention we reserve id 0 for air, the empty block, so a freshly created world is simply a grid full of zeros.

So conceptually the entire world is one gigantic 3D array of block ids: world[x][y][z]. That mental model is exactly right, and it is worth holding onto — most of what we build in this module is just an efficient way to store, stream, and draw that array. The catch is the word "gigantic."

Why not one giant array?

The trouble is that the world is effectively unbounded. A player can walk in any horizontal direction indefinitely, and we would like to generate fresh terrain as they go. There is no finite world[X][Y][Z] we could allocate up front that is both big enough to never run out and small enough to fit in memory. Even a merely "large" fixed world runs into three walls at once:

  • Memory. A modest 4096 × 256 × 4096 world at two bytes per block is over eight gigabytes — most of it air the player will never see.
  • Streaming. You want to load the terrain around the player and unload what is far away. One monolithic array is all-or-nothing; you cannot page a corner of it in from disk.
  • Saving. When the player edits one block, you want to write a small amount back to disk, not re-serialize the whole planet.

The fix is the idea the rest of this module is built on: stop thinking of the world as one array and start thinking of it as a grid of small, independent boxes.

Chunks: slicing the world into boxes

A chunk is a fixed-size cube of voxels — a self-contained tile of the world that can be generated, loaded, saved, meshed, and freed on its own. The world becomes a sparse grid of chunks: we keep the handful near the player resident in memory and forget the rest. Walk east, and new chunks stream in ahead of you while chunks behind you are written to disk and released.

We will use a chunk that is 16 × 16 × 16 voxels: 

#include <cstdint>

// A block type is a small integer id. 0 is reserved for "air".
using BlockId = uint16_t;

// Edge length of a chunk, in voxels. A power of two on purpose.
static constexpr int CHUNK_SIZE = 16;

// Total voxels in one chunk: 16 * 16 * 16 = 4096.
static constexpr int CHUNK_VOLUME = CHUNK_SIZE * CHUNK_SIZE * CHUNK_SIZE;

Why 16? It is a sweet spot. It is small enough that one chunk is a tiny 4096-voxel unit of work — cheap to generate and quick to re-mesh when the player edits a block — yet large enough that the world is not drowning in chunk bookkeeping. Crucially it is a power of two, which (as we will see) turns the world-to-chunk coordinate math into single-instruction bit operations. A cubic chunk also keeps our code symmetric: the same CHUNK_SIZE governs all three axes, so there is only one number to reason about.

How Minecraft does it

Minecraft's chunks are 16 × 16 columns that are tall — historically 256 blocks high, now more — but internally each column is sliced into 16 × 16 × 16 "sections." So under the hood it is cubic chunks too. We use cubic 16³ chunks directly because it keeps the math uniform on all three axes and the sections idea generalizes cleanly if you ever want a taller world.

One flat array, not three dimensions

It is tempting to store a chunk as a real 3D array — BlockId blocks[16][16][16], or worse, a std::vector of vectors of vectors. Don't. A vector<vector<vector<...>>> scatters 16 × 16 = 256 little allocations all over the heap, and every GetBlock chases three pointers through memory that the CPU cache hates.

Instead we store the chunk as one flat, contiguous array of 4096 block ids and do the 3D-to-1D mapping ourselves with a little arithmetic: 

// Map local (x, y, z), each in [0, CHUNK_SIZE), to a flat index in [0, CHUNK_VOLUME).
static inline int Index(int x, int y, int z) {
    return x + CHUNK_SIZE * (y + CHUNK_SIZE * z);
}

One allocation, perfect contiguity, and one multiply-add to reach any block. Expanding the formula shows the memory layout: index = x + 16*y + 256*z. So x is the fastest-changing axis (neighbouring x values are neighbouring bytes), y strides by 16, and z strides by 256. That choice of axis order matters when you iterate: to walk the array in memory order — which is what the cache wants — put x in the innermost loop and z in the outermost: 

// Cache-friendly: consecutive iterations touch consecutive memory.
for (int z = 0; z < CHUNK_SIZE; ++z)
    for (int y = 0; y < CHUNK_SIZE; ++y)
        for (int x = 0; x < CHUNK_SIZE; ++x) {
            BlockId id = blocks[Index(x, y, z)]; // steps through memory in order
            // ...
        }
8 KB per chunk

4096 voxels × 2 bytes (uint16_t) is exactly 8 KB of block data per chunk — small enough to hold thousands of them resident. If your game never needs more than 256 block types you can switch BlockId to uint8_t and halve that to 4 KB. Start with uint16_t; you will thank yourself the first time you add the 257th block.

Three coordinate spaces

Once the world is a grid of chunks, a block has an address in three different frames of reference, and you will constantly convert between them:

SpaceWhat it namesRangeExample for a block
WorldThe block's absolute position in the whole world.any integer(−37, 64, 12)
ChunkWhich chunk that block lives in.any integer(−3, 4, 0)
LocalWhere inside that chunk the block sits.0 … CHUNK_SIZE−1(11, 0, 12)

World coordinates are global signed integers. Chunk coordinates say which box; they are also signed, because the world extends in the negative direction too. Local coordinates are always in the range 0 … 15 — they are the index we feed to Index(x, y, z). The whole game is computing chunk and local from world.

The negative-coordinate trap

Going from a world coordinate to a chunk coordinate is "divide by 16," and from world to local is "the remainder." For positive coordinates that is exactly integer / and %, and everything looks fine — until the player walks west of the origin into negative coordinates. This is the single most common first voxel bug, so let us meet it head-on.

C++ integer division truncates toward zero, and % follows suit, so −1 % 16 is −1, not 15. Watch what that does: 

int worldX = -1;                     // one block west of the origin
int chunkX = worldX / CHUNK_SIZE;    // == 0   -- WRONG, it should be -1
int localX = worldX % CHUNK_SIZE;    // == -1  -- WRONG, and it is negative

BlockId id = chunk.blocks[localX];   // blocks[-1]: out-of-bounds read, undefined behavior

Block −1 should live in chunk −1 at local index 15 — the last slot of the chunk just west of the origin. Instead truncating division reports chunk 0 and a negative local index, which indexes off the front of the array. The symptom in-game is a seam of corrupted or crashing blocks along every axis where the coordinate crosses zero.

What we actually want is floor division (round toward negative infinity) and a matching modulo that always returns a non-negative result. Here is the table that makes the difference obvious:

The fixes are two small helpers. Both nudge the truncated result by one when the operands have different signs and the division was not exact: 

// Integer division that rounds toward negative infinity (floor), not toward zero.
static inline int floorDiv(int a, int b) {
    int q = a / b;
    if ((a % b != 0) && ((a < 0) != (b < 0))) --q;
    return q;
}

// Modulo whose result has the sign of b (here always non-negative, since b = CHUNK_SIZE > 0).
static inline int floorMod(int a, int b) {
    int r = a % b;
    if (r != 0 && ((r < 0) != (b < 0))) r += b;
    return r;
}
Never use plain / and % for world coordinates

The very first time your world extends west or south of the origin, plain division silently puts blocks in the wrong chunk and hands you a negative local index. It usually does not crash immediately — it corrupts a thin seam of blocks near x = 0 or z = 0 that looks like a bizarre rendering glitch. Always route world-to-chunk math through floorDiv/floorMod (or the bit trick below), and it simply never happens.

The power-of-two shortcut

Here is the payoff for choosing CHUNK_SIZE = 16. Because 16 is 2⁴, floor division by 16 is exactly an arithmetic right shift by 4, and floor modulo by 16 is exactly a bitwise AND with 15. And — this is the elegant part — both bit operations already do the right thing for negative numbers in two's-complement, which C++20 guarantees for signed integers. 

static constexpr int CHUNK_BITS = 4;               // log2(CHUNK_SIZE)
static constexpr int CHUNK_MASK = CHUNK_SIZE - 1;  // 15 == 0b1111

// For any int w (positive or negative, C++20 two's-complement):
//   floorDiv(w, 16)  ==  w >> CHUNK_BITS
//   floorMod(w, 16)  ==  w &  CHUNK_MASK

Check it against the tricky rows of the table. For w = −1, the bit pattern is all ones: −1 & 15 keeps the low four bits, giving 15 (correct local), and −1 >> 4 is still −1 (correct chunk). For w = −17, −17 & 15 is 15 and −17 >> 4 is −2 — again exactly the floor answers. The mask and shift are branchless and single-instruction, which is why essentially every voxel engine uses a power-of-two chunk size.

Keep floorDiv around anyway

The shift/mask trick only works when CHUNK_SIZE is a power of two. Keep the general floorDiv/floorMod helpers in the codebase for any math that is not chunk-sized — and so that if you ever experiment with a non-power-of-two size, the world does not quietly break.

The Chunk struct

Now we can assemble the pieces. A Chunk is little more than that flat 8 KB array wrapped in GetBlock/SetBlock accessors that hide the index math. Note that these accessors take local coordinates in 0 … 15 and trust the caller — bounds checking every access would be wasteful in the meshing loops of later lessons. In debug builds an assert catches mistakes; in release it compiles away. 

#include <vector>
#include <cassert>
#include "VoxelTypes.h"

struct Chunk {
    // One flat, contiguous array of 4096 block ids (8 KB), all air (0) to start.
    std::vector<BlockId> blocks = std::vector<BlockId>(CHUNK_VOLUME, 0);

    // Local (x, y, z), each in [0, CHUNK_SIZE), -> flat index.
    static inline int Index(int x, int y, int z) {
        return x + CHUNK_SIZE * (y + CHUNK_SIZE * z);
    }

    // Read/write a block by LOCAL coordinates. Caller guarantees 0 <= x,y,z < CHUNK_SIZE.
    BlockId GetBlock(int x, int y, int z) const {
        assert(x >= 0 && x < CHUNK_SIZE && y >= 0 && y < CHUNK_SIZE && z >= 0 && z < CHUNK_SIZE);
        return blocks[Index(x, y, z)];
    }

    void SetBlock(int x, int y, int z, BlockId id) {
        assert(x >= 0 && x < CHUNK_SIZE && y >= 0 && y < CHUNK_SIZE && z >= 0 && z < CHUNK_SIZE);
        blocks[Index(x, y, z)] = id;
    }
};

The two remaining pieces are free functions that convert a world coordinate into "which chunk" and "where inside it." We express them per-axis and bundle the results in tiny structs: 

struct ChunkCoord { int x, y, z; };  // which chunk
struct LocalCoord { int x, y, z; };  // position within that chunk, 0..CHUNK_SIZE-1

// Which chunk does this world position belong to?
inline ChunkCoord WorldToChunk(int wx, int wy, int wz) {
    return { wx >> CHUNK_BITS, wy >> CHUNK_BITS, wz >> CHUNK_BITS };
}

// Where inside that chunk does it sit? Always in [0, CHUNK_SIZE).
inline LocalCoord WorldToLocal(int wx, int wy, int wz) {
    return { wx & CHUNK_MASK, wy & CHUNK_MASK, wz & CHUNK_MASK };
}

And the inverse — reconstructing a world coordinate from a chunk plus a local offset — is just the definition run backwards, world = chunk × CHUNK_SIZE + local: 

inline int LocalToWorld(int chunkCoord, int localCoord) {
    return chunkCoord * CHUNK_SIZE + localCoord;
}

That is the whole coordinate toolkit. To read the block at world (−37, 64, 12) you now call WorldToChunk to find the chunk, look that chunk up in your world's chunk map, call WorldToLocal for the offset, and hand those local coordinates to GetBlock. The chunk map itself — how you store and look up chunks by their ChunkCoord — we will build shortly; here the point is that every access decomposes cleanly into these conversions.

What we are deliberately leaving out

Right now a block is just a number. What that number means — is it solid, is it transparent, what textures does it wear — is the job of a block registry, which is the very next lesson. And turning these voxels into the triangles MiniEngine actually draws is meshing, a few lessons after that. This lesson intentionally stops at the data model: the grid, the box, and the math that connects them.

Recap
  • A voxel is a block on a regular 3D grid; the world is conceptually one huge 3D array of BlockIds, with id 0 meaning air.
  • One giant array is impossible (unbounded, can't stream, can't save incrementally), so the world is split into fixed-size chunks — we use 16×16×16 = 4096 voxels.
  • Store a chunk as one flat, contiguous array indexed by x + CHUNK_SIZE*(y + CHUNK_SIZE*z); that layout makes x the innermost, cache-friendliest loop axis. At 2 bytes each, a chunk is 8 KB.
  • Every block has a world, chunk, and local (0…15) address, and you convert world → chunk/local constantly.
  • Plain / and % are wrong for negative coordinates — use floorDiv/floorMod, or for a power-of-two size the equivalent w >> 4 and w & 15.
  • The Chunk struct plus WorldToChunk/WorldToLocal is the entire data model the rest of module 11 builds on — block properties and meshing come next.
