A Job System: Meshing Off the Main Thread

MiniEngine can now stream an infinite world. But fly across it quickly and you feel the world stutter — a little hitch every time a new chunk crosses the horizon. That hitch is generating, lighting, and greedy-meshing a chunk, all happening on the main thread while the frame waits. The fix is not to make that work cheaper; it is to move it off the frame entirely. In this lesson we build a small thread-pool job system: pure CPU work runs on a pool of worker threads, and only the one thing that must stay on the render thread — the GPU upload — comes back to it.

What you'll learn
  • Why generating, lighting, and meshing a chunk on the main thread produces visible frame hitches.
  • How to build a compact thread pool with std::thread, std::mutex, and std::condition_variable running on hardware_concurrency() − 1 workers.
  • Why every GPU call must stay on one thread, and how to hand finished geometry back as a plain POD result.
  • How to make meshing thread-safe by snapshotting a padded, read-only volume on the main thread so workers never touch a live chunk.
  • How to cancel stale work with a per-chunk version id, and dodge the classic data race when the player edits a chunk mid-mesh.

The hitch you can feel

The chunk-streaming lesson gave every new chunk a pipeline: generate its terrain from noise, run flood-fill lighting, then greedy-mesh it into triangles. Each of those steps costs real time — call it a fraction of a millisecond to generate, most of a millisecond to light, and a couple of milliseconds to mesh a busy chunk. At a 16 ms frame budget (60 FPS), meshing even one chunk inline eats a tenth of the frame; meshing three in a burst as you round a hill blows straight through it and the camera visibly jerks. 

// The tempting, wrong approach: do it all inline on the main thread.
void World::Update() {
    for (ChunkCoord c : m_chunksNeedingMesh) {
        Chunk& chunk = m_chunks[c];
        GenerateTerrain(chunk, c);       // ~0.3 ms
        ComputeLighting(chunk, c);       // ~0.8 ms
        Mesh mesh = GreedyMesh(chunk);   // ~2 ms   <-- and this blocks the frame
        mesh.Upload(m_device);           // the frame cannot present until we get here
    }
}

The streaming lesson papered over this with a per-frame budget: only mesh, say, one chunk per frame so the spike is small. That helps, but it is a compromise on both ends — the world loads in slowly because you are rationing work, and a single heavy chunk still stutters. The real answer is that generation, lighting, and meshing are pure CPU computation with no graphics API in them at all. Nothing about turning voxels into a vertex array needs the render thread. So we run it somewhere else.

The plan: CPU on workers, GPU on one thread

Split the chunk pipeline in two along a bright line. Everything that is arithmetic over voxel data — generation, lighting, meshing — becomes a job that a worker thread runs. The job produces a plain array of vertices and indices in ordinary memory. Only the final step, uploading that array into a GPU buffer, comes back to the render thread.

That last part is not a style choice; it is a hard requirement of the graphics APIs. On Windows, Direct3D 11's immediate context — the ID3D11DeviceContext you call CreateBuffer-adjacent work and DrawIndexed on — is not safe to touch from multiple threads at once. On macOS, Metal is more forgiving about creating resources, but binding and encoding into the live command stream must stay coordinated on the render thread anyway. The portable rule that keeps both backends simple and correct is the same: all GPU calls happen on one thread.

The immediate context is single-threaded

Direct3D 11 does offer deferred contexts for recording commands off-thread, and you can create some resources on a background device thread — but it is fiddly and the win for a voxel engine is small. Uploading a vertex buffer is microseconds; the expensive part was always the meshing, and that is exactly what we moved. Keep the render thread as the sole owner of the device context, and you will never chase a threading crash inside the driver.

A thread pool

A thread pool is startlingly small. You spawn a fixed set of worker threads once, and each one loops forever: sleep until there is work, take one job, run it, repeat. The only tricky parts are the hand-off (a queue guarded by a mutex) and the sleeping (a condition variable so idle workers burn no CPU). Here is the whole thing. 

#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <functional>
#include <vector>

class ThreadPool {
public:
    explicit ThreadPool(unsigned workerCount) {
        for (unsigned i = 0; i < workerCount; ++i)
            m_workers.emplace_back([this] { WorkerLoop(); });
    }

    ~ThreadPool() { Shutdown(); }

    // A thread pool owns OS threads; never copy it.
    ThreadPool(const ThreadPool&)            = delete;
    ThreadPool& operator=(const ThreadPool&) = delete;

    // Called from the main thread. Hand a unit of work to the pool.
    void Enqueue(std::function<void()> job) {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            m_jobs.push(std::move(job));
        }
        m_cv.notify_one();               // wake exactly one sleeping worker
    }

    void Shutdown() {
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            if (m_stopping) return;
            m_stopping = true;
        }
        m_cv.notify_all();               // wake everyone so they can exit
        for (std::thread& t : m_workers)
            if (t.joinable()) t.join();  // wait for each worker to finish cleanly
        m_workers.clear();
    }

private:
    void WorkerLoop() {
        for (;;) {
            std::function<void()> job;
            {
                std::unique_lock<std::mutex> lock(m_mutex);
                // Sleep until there is a job to do, or we are shutting down.
                m_cv.wait(lock, [this] { return m_stopping || !m_jobs.empty(); });
                if (m_stopping && m_jobs.empty()) return;
                job = std::move(m_jobs.front());
                m_jobs.pop();
            }
            job();                       // run OUTSIDE the lock so workers run in parallel
        }
    }

    std::vector<std::thread>             m_workers;
    std::queue<std::function<void()>>    m_jobs;
    std::mutex                          m_mutex;
    std::condition_variable             m_cv;
    bool                                m_stopping = false;
};

Three details carry the whole design. The worker copies the job out of the queue and then releases the lock before calling it — if it ran the job while holding m_mutex, the other workers could not pop their own jobs and your "pool" would run one job at a time. The condition variable's predicate (m_stopping || !m_jobs.empty()) guards against spurious wake-ups, which real condition variables do produce. And Shutdown sets the flag, wakes everyone, and joins — no detached threads, no leaks, no work half-done.

Size the pool to leave the render thread a core to itself: 

inline unsigned WorkerCount() {
    unsigned hw = std::thread::hardware_concurrency(); // may return 0 on odd platforms
    return hw > 1 ? hw - 1 : 1;                         // reserve one core for rendering
}
A lock-free queue is the upgrade, not the starting point

Under heavy load, a single mutex on the job queue can become a contention point — every worker fights for the same lock to pop. The mature fix is a lock-free multi-producer / multi-consumer (MPMC) queue such as moodycamel::ConcurrentQueue, which lets workers pop without blocking each other. Reach for it only when a profiler says the mutex is hot. For a voxel engine whose jobs each run for milliseconds, the lock is held for nanoseconds and this mutex version is entirely fine — ship it first.

The chunk job and its POD result

Now define what actually crosses the thread boundary. Going in, the worker needs a read-only copy of the voxel data. Coming out, it produces a plain struct holding the vertex and index arrays plus enough identity to route them. Crucially, nothing in either direction is a GPU handle or a pointer into the live world — just owned, self-contained data. 

// The result a worker hands back. Plain data only: no ID3D11* / MTL* handles,
// no pointers into the live world. It owns its vectors outright.
struct ChunkMeshResult {
    ChunkCoord            coord;    // which chunk this mesh is for
    uint32_t              version;  // the chunk's generation id when the job was queued
    std::vector<Vertex>   vertices; // CPU-side geometry, ready for Mesh::Upload
    std::vector<uint32_t> indices;
};

Meshing a chunk needs more than the chunk's own 16³ voxels: greedy meshing and hidden-face removal have to peek one block into each of the six neighbours to decide whether a boundary face is exposed. So the input snapshot is a padded volume — 18³ blocks, the chunk plus a one-voxel border. We build it on the main thread by reading from the live world, so the worker never dereferences a chunk that another thread might be changing. 

// The chunk (16³) plus a 1-voxel border from its neighbours = 18³ blocks.
struct PaddedVolume {
    static constexpr int N = CHUNK_SIZE + 2;                 // 18
    std::vector<BlockId> blocks = std::vector<BlockId>(N * N * N, 0);

    static int Index(int x, int y, int z) { return x + N * (y + N * z); }
    BlockId At(int x, int y, int z) const { return blocks[Index(x, y, z)]; } // x,y,z in [0, 18)
};

// MAIN THREAD: copy the chunk and its neighbour border into a private buffer.
// world.GetBlock() takes WORLD coords and returns air for anything not loaded.
PaddedVolume SnapshotPadded(const World& world, ChunkCoord c) {
    PaddedVolume vol;
    for (int z = -1; z <= CHUNK_SIZE; ++z)
        for (int y = -1; y <= CHUNK_SIZE; ++y)
            for (int x = -1; x <= CHUNK_SIZE; ++x) {
                int wx = LocalToWorld(c.x, x);              // crossing into a neighbour
                int wy = LocalToWorld(c.y, y);              // is just a bigger local index
                int wz = LocalToWorld(c.z, z);
                vol.blocks[PaddedVolume::Index(x + 1, y + 1, z + 1)] =
                    world.GetBlock(wx, wy, wz);
            }
    return vol;
}
Only POD crosses the thread boundary

The snapshot is the entire safety argument in one move. Copying 18³ = 5832 block ids is about 11 KB — a memcpy that costs a handful of microseconds, next to the milliseconds meshing will take. Once the worker holds that private copy, it shares nothing mutable with any other thread, so no lock is needed inside the meshing loop at all. Pass plain data across the boundary and the hardest bugs in threaded code simply cannot occur.

Terrain generation is even easier to hand off. As you built it in the terrain lesson, a chunk's blocks are a deterministic function of the world seed and the chunk coordinate — same seed, same coord, same terrain, every time. That means a generation job needs no inputs at all and shares no mutable state; it produces the chunk's blocks from pure arithmetic. Meshing is the step that depends on neighbours, so in the streaming pipeline you generate first, and only queue a meshing job for a chunk once it and its six neighbours have all been generated.

Requesting work and draining results

Wrap the pool in a small system that owns the results queue too. RequestMesh runs on the main thread: it snapshots the padded volume, captures the chunk's current version, and enqueues a job that meshes the snapshot and pushes a ChunkMeshResult. Note the second mutex — results have their own lock so that draining them never contends with enqueuing jobs. 

#include <algorithm>
#include <iterator>

class ChunkMeshSystem {
public:
    // MAIN THREAD. Snapshot everything the job needs, then queue it.
    void RequestMesh(const World& world, ChunkCoord c) {
        PaddedVolume vol = SnapshotPadded(world, c);   // private copy: no shared state
        uint32_t     version = world.VersionOf(c);     // captured now, checked on the way back

        m_pool.Enqueue([this, c, version, vol = std::move(vol)]() mutable {
            ChunkMeshResult result;
            result.coord   = c;
            result.version = version;
            GreedyMesh(vol, result.vertices, result.indices); // pure CPU, read-only input

            std::lock_guard<std::mutex> lock(m_resultsMutex);
            m_results.push_back(std::move(result));           // hand the POD back
        });
    }

    // MAIN THREAD, once per frame. Upload at most `budget` finished meshes.
    void IntegrateResults(World& world, ID3D11Device* device, int budget) {
        std::vector<ChunkMeshResult> ready;
        {
            std::lock_guard<std::mutex> lock(m_resultsMutex);
            int n = std::min<int>(budget, (int)m_results.size());
            ready.insert(ready.end(),
                         std::make_move_iterator(m_results.begin()),
                         std::make_move_iterator(m_results.begin() + n));
            m_results.erase(m_results.begin(), m_results.begin() + n);
        }
        for (ChunkMeshResult& r : ready) {
            // Cancellation: drop results the world has moved past (see below).
            if (!world.IsLoaded(r.coord) || world.VersionOf(r.coord) != r.version)
                continue;

            Mesh mesh(std::move(r.vertices), std::move(r.indices));
            mesh.Upload(device);                        // the ONLY GPU call, on the main thread
            world.SetChunkMesh(r.coord, std::move(mesh));
        }
    }

private:
    ThreadPool                   m_pool{ WorkerCount() };
    std::mutex                   m_resultsMutex;
    std::vector<ChunkMeshResult> m_results;   // written by workers, drained by main thread
};

The budget parameter is where the streaming lesson's per-frame limit lives now — but it throttles something far cheaper. Before, the budget rationed meshing, several milliseconds apiece; now it rations only uploads, microseconds apiece. You can integrate a dozen chunks in a frame without a hitch, so the world fills in fast while the render thread stays smooth. Call it once per frame: 

void Engine::Update() {
    m_world.StreamAround(m_camera.Position());      // queues RequestMesh for new chunks
    m_meshSystem.IntegrateResults(m_world, m_device, /*budget=*/8);
    // ... render the world's chunk meshes as usual ...
}
Metal is the same story with different names

On macOS, swap the D3D11 device for your id<MTLDevice> and let Mesh::Upload call newBufferWithBytes:length:options:. The ChunkMeshResult, the pool, the snapshot, and the drain loop are all platform-neutral — they are just C++ over plain arrays. Only the one Upload call at the end changes per backend, and it still runs on the render thread.

Cancelling stale work

Between the moment you queue a job and the moment its result arrives, the world can move on. The player flies away and the chunk unloads. The player breaks a block and the chunk needs re-meshing. Either way, the vertices the worker is busy producing are already obsolete. Uploading them would waste GPU memory or, worse, show a mesh that no longer matches the blocks.

The fix is a monotonically increasing version (a generation id) per chunk. Anything that changes a chunk's contents bumps it; the value is captured when a job is queued and checked when its result comes home. A mismatch means "the world changed under this job" — discard it. 

struct Chunk {
    std::vector<BlockId> blocks = std::vector<BlockId>(CHUNK_VOLUME, 0);
    uint32_t            version = 0;   // bump on any edit; loading a fresh chunk starts a new line

    void SetBlock(int x, int y, int z, BlockId id) {
        blocks[Index(x, y, z)] = id;
        ++version;                     // this chunk's outstanding mesh jobs are now stale
    }
};

You saw the check already in IntegrateResults: world.VersionOf(r.coord) != r.version throws the stale result away, and because the edit also re-queued a fresh RequestMesh with a new snapshot, the correct mesh is already on its way. Editing a block near a chunk boundary bumps both the edited chunk and the neighbour whose border changed, so both re-mesh.

The classic voxel data race: editing a chunk a worker is reading

The bug that bites every threaded voxel engine is this: the player places a block, the main thread writes into a live Chunk, and at the same instant a worker is iterating that very chunk to mesh it. One thread writes while another reads the same memory with no synchronization — undefined behaviour, and in practice a torn mesh, a garbage vertex, or a crash that reproduces once a week. Our design forecloses it by construction: workers never read a live chunk. They mesh a private snapshot taken on the main thread, and the version check discards whatever was in flight when the edit landed. If you ever skip the snapshot to "save a copy," you must instead lock the chunk for the entire mesh or you re-open this exact race — copying is the cheaper and safer trade.

Shutdown is the last loose end, and the pool already handles it: when the engine exits, ThreadPool's destructor sets the stopping flag, wakes the workers, and joins them. Any results still sitting in m_results are simply never integrated — they hold only plain vectors, which free themselves. No thread outlives the engine, and no GPU resource leaks, because the only resources are made on the thread that is tearing everything down.

Recap
  • Generating, lighting, and meshing a chunk is milliseconds of pure CPU work; doing it inline on the main thread causes visible frame hitches, so we move it onto worker threads.
  • A thread pool is a fixed set of workers looping on a mutex-plus-condition-variable job queue; size it to hardware_concurrency() − 1 to leave the render thread a core.
  • Every GPU call stays on the render thread — Direct3D 11's immediate context is single-threaded and Metal's command stream must stay coordinated — so only Mesh::Upload runs there.
  • Jobs exchange plain POD: a padded, read-only PaddedVolume snapshot goes in, a ChunkMeshResult of vertices and indices comes out — no GPU handles, no pointers into the live world.
  • Snapshotting on the main thread means workers share nothing mutable, which forecloses the classic voxel data race of editing a chunk while a worker meshes it.
  • A per-chunk version id cancels stale work: edits and unloads bump it, and results whose version no longer matches are discarded on the way back — the per-frame budget now throttles only cheap uploads.
