Voxel Ambient Occlusion

Right now our voxel world is correct but oddly flat — every face is lit the same, so a deep pit and an open field read identically. Ambient occlusion (AO) is the trick that fixes this for almost no cost: darken the vertices tucked into crevices where nearby blocks would smother the ambient light. It is the single cheapest change that makes a blocky world suddenly look solid, and Minecraft's famous soft corner shadows are exactly this. We'll compute it per vertex from the neighboring voxels, then chase down the one nasty artifact — a diagonal seam — that trips up every first implementation.

What you'll learn
  • Why AO works as cheap, baked, fake global illumination for a voxel world.
  • The exact three-neighbor rule that turns a vertex's surroundings into an AO level 0–3.
  • The concrete neighbor offsets to sample for all four vertices of every one of the six faces.
  • How to store AO in a vertex — as a float, or packed with light and texture layer into a uint32_t.
  • The diagonal-seam artifact, and the flip-the-triangulation fix that makes interpolation symmetric.

Fake global illumination, one vertex at a time

Real ambient occlusion asks, for each point on a surface, "how much of the sky can this point actually see?" A point at the bottom of a narrow slot sees very little sky, so it receives little ambient light and looks dark. Computing that properly means casting rays over the hemisphere — far too expensive for a world of millions of blocks. But voxels give us a shortcut: the geometry around any face corner is just a handful of neighboring blocks, either solid or empty. We can approximate "how enclosed is this corner?" by simply counting the solid neighbors that hug it.

Because a voxel mesh already has four vertices per face, we bake one AO value into each vertex at mesh time and let the rasterizer interpolate it across the face for free. No extra draw calls, no textures, no runtime cost beyond a multiply in the pixel shader. It is entirely a meshing-time computation — which is why it lives here, right alongside greedy meshing.

The three neighbors and the AO rule

Focus on a single face and a single one of its four corners. That corner vertex sits where three neighboring voxels meet on the outer side of the face — the side the face looks out toward. Two of them share an edge with the empty cell the face occupies (call them side1 and side2); the third sits diagonally across the corner (corner). These are the only three blocks that can cast shadow onto that vertex.

The mapping from those three booleans to an AO level is delightfully small. This is the exact rule from Mikola Lysenko's classic write-up "Ambient occlusion for Minecraft-like worlds," and every voxel engine uses some form of it: 

// side1, side2 = the two edge-adjacent neighbors on the face's outer side.
// corner       = the diagonal neighbor across the same corner.
// Returns an occlusion level: 0 = darkest (buried), 3 = fully open.
int vertexAO(bool side1, bool side2, bool corner) {
    if (side1 && side2)
        return 0;                          // both edges blocked -> fully occluded
    return 3 - (side1 + side2 + corner);   // otherwise count the blockers
}
Why the early return matters

If both edge neighbors (side1 and side2) are solid, the diagonal can't be seen at all — the corner is sealed into an inside edge. The rule short-circuits to 0 so a fully tucked corner is the darkest possible, regardless of what the diagonal holds. Without that special case a sealed corner would read as level 1 and look subtly wrong.

The four levels are then mapped to brightness. A simple, good-looking ramp is {0.4, 0.6, 0.8, 1.0}: level 0 keeps 40% of the light, level 3 keeps all of it. You can use a smoother curve if you prefer softer valleys, but the linear ramp is what most voxel games ship. 

// Level 0..3 -> brightness multiplier.
constexpr float kAOBrightness[4] = { 0.4f, 0.6f, 0.8f, 1.0f };

Which neighbors, for which vertex?

The whole scheme is only correct if we sample the right three cells for each vertex. The clean way to keep this straight for all six faces at once is to describe each face with three integer axis vectors: the outward normal N, and two in-plane axes u and v. Every neighbor we sample lives in the layer one step out along the normal — that is, offset from the block position by N — and then displaced along u and/or v.

FaceNormal Nu axisv axis
+X( 1, 0, 0)(0, 1, 0)(0, 0, 1)
−X(−1, 0, 0)(0, 1, 0)(0, 0, 1)
+Y(0, 1, 0)(1, 0, 0)(0, 0, 1)
−Y(0, −1, 0)(1, 0, 0)(0, 0, 1)
+Z(0, 0, 1)(1, 0, 0)(0, 1, 0)
−Z(0, 0, −1)(1, 0, 0)(0, 1, 0)

Each of the four vertices corresponds to a sign pair (su, sv), each sign being −1 or +1, walking the corners in order (−,−) (+,−) (+,+) (−,+). For a vertex with signs (su, sv):

  • side1 is the block at b + N + su·u
  • side2 is the block at b + N + sv·v
  • corner is the block at b + N + su·u + sv·v

Let's pin this down with one face fully worked. Take the top (+Y) face of the block at (x, y, z). Its normal steps us up to the layer y + 1, and u = +X, v = +Z. The four vertices sample these exact cells:

Vertex(su, sv)side1side2corner
0(−1, −1)(x−1, y+1, z)(x, y+1, z−1)(x−1, y+1, z−1)
1(+1, −1)(x+1, y+1, z)(x, y+1, z−1)(x+1, y+1, z−1)
2(+1, +1)(x+1, y+1, z)(x, y+1, z+1)(x+1, y+1, z+1)
3(−1, +1)(x−1, y+1, z)(x, y+1, z+1)(x−1, y+1, z+1)

The other five faces are the same picture rotated: swap in that face's N, u, and v and the same (su, sv) loop produces the right cells. Here is the whole thing as one function: 

struct FaceAO { int level[4]; };

// b = block being meshed; N, u, v = this face's axis vectors (from the table).
// isSolid() must return true for opaque blocks and read across chunk borders.
FaceAO computeFaceAO(const World& world, glm::ivec3 b,
                     glm::ivec3 N, glm::ivec3 u, glm::ivec3 v) {
    auto solid = [&](glm::ivec3 p) { return world.isSolid(p); };

    // Corner order: (-,-), (+,-), (+,+), (-,+) -- must match the mesher's quad.
    static const int su[4] = { -1, +1, +1, -1 };
    static const int sv[4] = { -1, -1, +1, +1 };

    FaceAO ao;
    for (int i = 0; i < 4; ++i) {
        bool side1  = solid(b + N + su[i] * u);
        bool side2  = solid(b + N + sv[i] * v);
        bool corner = solid(b + N + su[i] * u + sv[i] * v);
        ao.level[i] = vertexAO(side1, side2, corner);
    }
    return ao;
}
Sample across chunk borders

A block on a chunk's edge has neighbors that live in the next chunk. If isSolid() only reads the local chunk and treats out-of-bounds as empty, you'll get bright AO seams running along every chunk boundary. Give the mesher access to a small neighborhood (the chunk plus its six neighbors, or a padded copy) so corner sampling never falls off the edge.

Storing AO in the vertex

The simplest option is a plain float on the vertex — resolve the level to a brightness at mesh time and write it straight in: 

struct Vertex {
    glm::vec3 position;
    glm::vec2 uv;
    float     ao;   // 0.4 .. 1.0, baked at mesh time
};

A voxel engine, though, wants tiny vertices — you're pushing millions of them. AO is only two bits (level 0–3), so it packs naturally alongside the texture-atlas layer and the block/sky light from the lighting lesson. Keep the raw level in the pack and expand it to brightness in the shader: 

// Layout: [ texLayer:16 | light:8 | ao:2 | spare:6 ]  in one 32-bit word.
uint32_t packVertexData(uint16_t texLayer, uint8_t light, uint8_t aoLevel) {
    return (uint32_t(texLayer) << 16)
         | (uint32_t(light)    <<  8)
         | (uint32_t(aoLevel & 0x3u) << 6);
}

The diagonal-seam artifact

Here is the subtle bug that catches everyone. A face is a quad, but the GPU only draws triangles, so the quad is split into two. The rasterizer interpolates each vertex value — including AO — linearly across each triangle independently. When all four AO values agree, or when the two diagonal corners happen to match, this is fine. But when the two corners on one diagonal are dark and the two on the other are bright, the two triangulations of the quad give genuinely different shading, and the default split picks the ugly one: a hard diagonal crease cuts across the face where the two triangles meet.

The fix is to choose the diagonal so that the interpolation stays symmetric. Label the four corner levels a[0], a[1], a[2], a[3] going around the quad. Diagonal 0–2 connects a[0] and a[2]; diagonal 1–3 connects a[1] and a[3]. Split along the diagonal whose two corners are darker. Concretely: if a[0] + a[2] > a[1] + a[3], flip to the 1–3 diagonal; otherwise keep the default 0–2 split. 

// Append one face: 4 vertices, then 6 indices with the anisotropy-aware split.
void emitFace(Mesh& mesh, const std::array<Vertex, 4>& corners, const FaceAO& ao) {
    uint32_t base = static_cast<uint32_t>(mesh.vertices.size());

    for (int i = 0; i < 4; ++i) {
        Vertex v = corners[i];
        v.ao = kAOBrightness[ao.level[i]];   // bake brightness onto the vertex
        mesh.vertices.push_back(v);
    }

    auto& idx = mesh.indices;
    if (ao.level[0] + ao.level[2] > ao.level[1] + ao.level[3]) {
        // Flip: split along the 1-3 diagonal so the dark corners share an edge.
        idx.insert(idx.end(), { base + 0, base + 1, base + 3,
                                base + 1, base + 2, base + 3 });
    } else {
        // Default: split along the 0-2 diagonal.
        idx.insert(idx.end(), { base + 0, base + 1, base + 2,
                                base + 0, base + 2, base + 3 });
    }
}

Both branches wind the same direction, so back-face culling still behaves. The only thing that changes is which two triangles tile the quad — and that is precisely enough to make the AO gradient run straight through the dark corners instead of skewing across the face.

Skip the flip and you'll see it everywhere

Without this branch, every face that meets a corner of geometry — the base of a tree, the inside of a doorway, the top of a step — gets a visible diagonal line where the shading kinks. It's the classic "why do my blocks have a scar across them?" bug. The flip is three lines and it removes the artifact entirely.

Multiplying AO into the light

On the GPU, AO is just one more multiplier on the surface brightness. In the pixel shader, sample the atlas, then fold AO into the lighting. This is HLSL (Direct3D 11, our Windows backend); the Metal shading language version is line-for-line the same idea in a fragment function. 

struct VSOutput {
    float4 pos   : SV_Position;
    float2 uv    : TEXCOORD0;
    float  ao    : TEXCOORD1;   // 0.4 .. 1.0
    float  light : TEXCOORD2;   // 0 .. 1 from flood-fill lighting
};

float4 PSMain(VSOutput i) : SV_Target {
    float3 albedo = gAtlas.Sample(gSampler, i.uv).rgb;
    float  shade  = i.ao * i.light;      // AO scales the incoming light
    return float4(albedo * shade, 1.0);
}

If you packed AO as a level in a uint, unpack and expand it in the vertex shader before it reaches the interpolators, so the pixel shader still sees a ready-to-use brightness: 

uint  aoLevel = (packed >> 6) & 0x3u;         // 0..3
float ao      = 0.4 + (float(aoLevel) / 3.0) * 0.6;  // -> 0.4, 0.6, 0.8, 1.0

In Metal, the fragment function does the same multiply — half shade = in.ao * in.light; — reading the interpolated stage-in values the vertex function wrote.

Combine AO and scene light by multiply — or min

Multiplying AO into the block/sky light (ao * light) reads as soft contact shadow and is the usual choice. Some engines instead take min(ao, light), which keeps already-dark corners from double-darkening in dim areas. Both look good; multiply is warmer, min is flatter. Pick one and keep it consistent so lighting stays predictable across the world.

AO and greedy meshing

Back in the greedy meshing lesson we merged runs of identical faces into big quads using a per-face mask. AO breaks a naive merge: two side-by-side faces can share a block type, a normal, a texture layer, and a light level, yet still have different corner shadows because one sits next to a ledge and the other doesn't. Merge them and the AO of the whole strip snaps to one face's values — smeared, wrong shading across the merged quad.

The fix is to make AO part of the mask key. Two faces may only merge when their four AO levels match as well. On a big flat wall away from other geometry every face is {3, 3, 3, 3}, so those all still merge into one large quad — exactly what we want. Near a corner the AO varies from cell to cell, so the run naturally breaks into smaller quads that each carry correct shading. 

// A cell in the greedy mask. AO now belongs in the key, so faces only
// merge when their corner shadows also agree.
struct MaskCell {
    uint16_t block   = 0;              // block / texture layer
    bool     visible = false;
    uint8_t  ao[4]   = { 3, 3, 3, 3 }; // per-corner AO levels

    bool operator==(const MaskCell& o) const {
        return block == o.block && visible == o.visible
            && ao[0] == o.ao[0] && ao[1] == o.ao[1]
            && ao[2] == o.ao[2] && ao[3] == o.ao[3];
    }
};
Forget AO in the key and greed wins over correctness

If the mask key ignores AO, the greedy pass happily fuses faces with mismatched shadows and you get blocky bands of wrong brightness marching across otherwise-uniform surfaces. Requiring identical AO to merge is conservative — it makes a few more quads near edges — but it is correct, and flat regions (the vast majority of any world) still collapse to single quads.

That's the entire feature: one tiny rule turns three booleans into a level, a brightness ramp turns the level into a multiplier, a diagonal flip keeps the interpolation honest, and one extra field in the greedy key keeps the merger honest too. Cheap, baked, and the difference between a world that looks like flat cardboard and one that looks carved.

Recap
  • Voxel AO fakes global illumination by counting the solid neighbors that hug each face corner — a meshing-time computation with near-zero runtime cost.
  • For each vertex, sample three neighbors on the face's outer side (two edges, one diagonal) and apply vertexAO: both edges solid returns 0, else 3 - (side1 + side2 + corner).
  • Describe each face by axis vectors N, u, v; the four vertices use sign pairs (su, sv) to pick side1, side2, and corner — and neighbor sampling must read across chunk borders.
  • Store the level as a brightness float, or pack the 2-bit level with light and texture layer into a uint32_t and expand it in the shader.
  • Fix the diagonal seam by flipping the triangulation when a[0] + a[2] > a[1] + a[3], so the split runs through the dark corners and interpolation stays symmetric.
  • AO must be part of the greedy-meshing mask key: faces only merge when their four AO levels match, or you smear shadows across merged quads.
