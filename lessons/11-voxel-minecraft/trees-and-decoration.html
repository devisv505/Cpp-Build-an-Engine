Trees, Grass & World Decoration

Base terrain gives you the ground — hills, stone, a dirt shoulder, a grass cap, caves scooped underneath. It does not give you a world you want to stand in. That comes from the decoration pass: the stage that scatters tall grass and flowers across the surface, grows trees, and hides ore veins in the rock. Decoration is really a lesson in placing structures deterministically — and in the one genuinely hard problem that follows, which is what to do when a tree straddles the seam between two chunks.

What you'll learn
  • Where the decoration/feature pass sits in the pipeline, and why it must run before lighting and meshing
  • How to seed a fast, deterministic per-chunk PRNG from the world seed so features are reproducible, thread-safe and save-friendly
  • How to scatter per-column flora and grow a small oak from a trunk-and-canopy pattern
  • The cross-chunk problem, and the staged-generation state machine (with a neighbor-readiness gate) that kills half-trees, duplicates and seams
  • How to place ore veins with a random walk, and let glowing ores feed the Module 10 bloom pipeline

Where decoration fits in the pipeline

The previous lesson's GenerateChunk produced base terrain: it capped every column with a grass block over a few dirt blocks over stone, filled the low air with water, and carved caves with 3D noise. Decoration adds the two things terrain deliberately left out — what sits on top of the surface (flora, trees) and what hides inside the rock (ores). It never invents ground; it stamps features onto ground that already exists.

The order of the four passes is not a suggestion — later passes read what earlier passes wrote:

#PassWhat it does
1Base terrainHeightmap + biome palette + caves — the previous lesson
2DecorationFlora, trees and ore veins stamped onto the terrain — this lesson
3Flood-fill lightingPropagate skylight and block light — a later lesson
4Greedy meshingTurn the final blocks into triangles — from the meshing lessons
Decoration must run before lighting and meshing

Leaves are semi-transparent and attenuate skylight, so a tree changes the baked light of everything beneath its canopy. If you light or mesh a chunk first and grow the tree afterwards, the light values and the vertex data are already wrong — you would have to re-light and re-mesh. Grow the world first, then light it, then turn it into triangles. Get the order backwards and every feature you add forces a redo.

Flag your decoration blocks correctly

Meshing and lighting only behave if the registry describes each new block honestly. Blocks::Leaves is solid but transparent (you stand on it, yet light and sight pass, so interior faces are culled and skylight leaks through). Blocks::TallGrass and Blocks::Flower are not solid and transparent — decoration you walk straight through. Set those solid/transparent flags on the BlockType when you register the ids, exactly as in the block-registry lesson, or the mesher will wall off your forests and the light will stop at a flower.

One structural choice makes the rest of this lesson simple: we run decoration once per chunk column — the vertical stack of the cubic 16³ chunks you already built, addressed together by their horizontal (x, z). A column is just those sub-chunks seen as one tall unit; a world-aware SetBlock(wx, wy, wz, id) drops each block into whichever 16³ slice owns that y. Keying decoration to the column means a tree that grows tall simply spills into the sub-chunk above it — same column, no coordination — so the only cross-chunk case we must worry about is horizontal spread into the neighboring column. That is exactly the shape Minecraft's own generator takes.

Deterministic placement: seed from world coordinates

Every feature we place is a random decision — does a tree go here? how tall? where do the flowers land? — but it must be reproducible random, not rand(). We seed a small pseudo-random number generator per column, from the world seed folded together with the column coordinate. The same (x, z, worldSeed) then produces the same stream of numbers forever.

#pragma once
#include <cstdint>

// A tiny, dependency-free PRNG for worldgen. This is splitmix32: one 32-bit
// word of state, excellent distribution, and trivial to reproduce anywhere.
struct Rng {
    uint32_t state;

    uint32_t next() {                        // raw 32 random bits
        uint32_t z = (state += 0x9E3779B9u);
        z = (z ^ (z >> 16)) * 0x21F0AAADu;
        z = (z ^ (z >> 15)) * 0x735A2D97u;
        return z ^ (z >> 15);
    }

    float nextFloat() {                      // uniform in [0, 1)
        return (next() >> 8) * (1.0f / 16777216.0f);   // 24 mantissa bits
    }

    int nextInt(int n) {                     // uniform in [0, n)
        return int(next() % uint32_t(n));
    }
};

// Fold the column coordinate and the world seed into one 32-bit seed.
// Same (cx, cz, worldSeed) -> same stream, every run, on every thread.
inline uint32_t hashChunk(int cx, int cz, uint32_t worldSeed) {
    uint32_t h = worldSeed;
    h ^= uint32_t(cx) * 0x8DA6B343u;
    h ^= uint32_t(cz) * 0xD8163841u;
    h  = (h ^ (h >> 15)) * 0x2C1B3C6Du;
    return h ^ (h >> 13);
}

Determinism is not a nicety here; three separate systems depend on it. It is reproducible, so a bug you see in a chunk is there again next launch. It is thread-safe by construction — a column's PRNG reads nothing but its own coordinate, so the job system can decorate many columns in parallel with no shared state and no locks. And it is save-friendly: because a column decorates identically every time, you never write a single generated tree to disk. You persist only the blocks the player changed and regenerate everything else from the seed.

Seed from world coordinates, never from a global counter

Seed each column's Rng from hashChunk(x, z, worldSeed) and draw features from it in a fixed order (flora, then trees, then ores). Do that and placement is deterministic, thread-safe and free to save — a chunk you have never visited generates exactly like one you have. The moment you reach for a shared rand(), a frame counter, or wall-clock time, you lose all three at once.

Grass and flowers: the easy per-column case

Flora is the gentle introduction because it never leaves its column. For each column in the chunk we find the surface — the topmost solid voxel — and, with some probability, drop a single plant block into the air directly above it. We assume the world exposes SurfaceY(wx, wz), which scans a world column down from the top and returns the y of the highest solid block (or -1 if that column is not ready yet), and a PlaceIfAir helper that only writes when the target voxel is currently air.

void WorldGen::ScatterFlora(ChunkManager& world, ChunkCoord coord, Rng& rng) {
    for (int lx = 0; lx < CHUNK_SIZE; ++lx)
    for (int lz = 0; lz < CHUNK_SIZE; ++lz) {
        const int wx = LocalToWorld(coord.x, lx);   // column -> world coords
        const int wz = LocalToWorld(coord.z, lz);

        const int sy = world.SurfaceY(wx, wz);       // top solid voxel, or -1
        if (sy < 0) continue;
        const BlockId ground = world.GetBlock(wx, sy, wz);

        // Plains: mostly tall grass, the occasional flower -- on grass only.
        if (ground == Blocks::Grass && rng.nextFloat() < 0.22f) {
            BlockId plant = (rng.nextFloat() < 0.12f) ? Blocks::Flower
                                                      : Blocks::TallGrass;
            world.PlaceIfAir(wx, sy + 1, wz, plant); // sits in the air above
        }
        // Desert: a rare dead bush -- on sand only.
        else if (ground == Blocks::Sand && rng.nextFloat() < 0.05f) {
            world.PlaceIfAir(wx, sy + 1, wz, Blocks::DeadBush);
        }
    }
}

Notice we did not sample the biome noise to decide plains-versus-desert: the surface block already encodes it. The terrain pass caps deserts with Blocks::Sand and plains with Blocks::Grass, so branching on the ground block is biome gating for free. When you want finer control — cactus columns only deep in a desert, a "flower forest" that is dense with blossoms — sample the same temperature/humidity fields from the terrain lesson and switch on those. Either way, flora writes a single block straight up into air it can see, so nothing here can ever reach into a neighbor.

Trees: choosing where trunks go

Trees are where placement gets interesting, because two trunks packed onto adjacent blocks look terrible. We want candidates spread out with some minimum spacing. There are two standard ways to get that.

Method A — random attempts with rejection

Make a handful of candidate positions per column and reject any that lands too close to a trunk we have already placed. This is a cheap, one-column stand-in for Poisson-disk sampling: propose, measure, keep or discard.

#include <vector>

void WorldGen::PlaceTrees(ChunkManager& world, ChunkCoord coord, Rng& rng) {
    constexpr int   ATTEMPTS    = 6;    // candidate trunks per column
    constexpr int   MIN_SPACE   = 5;    // keep trunks this far apart (blocks)
    constexpr float TREE_CHANCE = 0.5f; // how many attempts actually try

    struct P { int x, z; };
    std::vector<P> placed;               // trunks placed by THIS column's pass

    for (int i = 0; i < ATTEMPTS; ++i) {
        if (rng.nextFloat() > TREE_CHANCE) continue;

        const int wx = LocalToWorld(coord.x, rng.nextInt(CHUNK_SIZE));
        const int wz = LocalToWorld(coord.z, rng.nextInt(CHUNK_SIZE));

        // Reject if this candidate crowds a trunk we already placed.
        bool tooClose = false;
        for (const P& t : placed) {
            const int dx = t.x - wx, dz = t.z - wz;
            if (dx * dx + dz * dz < MIN_SPACE * MIN_SPACE) { tooClose = true; break; }
        }
        if (tooClose) continue;

        const int groundY = world.SurfaceY(wx, wz);
        if (groundY < 0) continue;
        if (world.GetBlock(wx, groundY, wz) != Blocks::Grass) continue; // grass only

        PlaceTree(world, wx, groundY, wz, rng);
        placed.push_back({ wx, wz });
    }
}

This has one honest limitation: placed only knows about trunks from this column's pass, so a tree near the border can still land within MIN_SPACE of a tree the neighbor column placed. It is usually invisible, but if you want spacing that respects borders, reach for method B.

Method B — a forest-density noise

Sample a second, very low-frequency noise field and let it gate both where forests appear and how dense they are. Because the field is continuous across chunk borders, spacing and clumping stay coherent from one column to the next with no bookkeeping:

// Inside the attempt loop, after picking (wx, wz):
float density = m_forest.GetNoise(wx * FOREST_FREQ, wz * FOREST_FREQ); // -1..1
float d01     = density * 0.5f + 0.5f;         // remap to 0..1
if (rng.nextFloat() > d01) continue;           // denser noise -> more trees

Most engines combine the two: density noise decides the broad clearings and thickets, the rejection test keeps individual trunks from touching. Whichever you use, the outcome is a set of trunk positions — and now we have to actually build a tree at each one.

Growing one tree

A tree is a trunk of Blocks::Wood with a canopy of Blocks::Leaves on top. We will build the classic small oak: a trunk four to six blocks tall, two wide leaf layers around the upper trunk, a tighter layer at the top, and a small plus-shaped cap. The trunk height comes from the PRNG, so trees vary; the canopy is placed with PlaceIfAir so leaves settle into open air and never eat into a hillside behind them.

#include <cstdlib>   // std::abs

void WorldGen::PlaceTree(ChunkManager& world, int wx, int groundY, int wz, Rng& rng) {
    const int trunkH = 4 + rng.nextInt(3);       // 4, 5, or 6 logs
    const int topY   = groundY + trunkH;         // y of the highest log

    // 1) Trunk: a column of Wood standing on the ground block.
    for (int y = groundY + 1; y <= topY; ++y)
        world.SetBlock(wx, y, wz, Blocks::Wood);

    // 2) Two wide 5x5 leaf layers around the upper trunk, corners trimmed.
    for (int y = topY - 2; y <= topY - 1; ++y)
        for (int dx = -2; dx <= 2; ++dx)
            for (int dz = -2; dz <= 2; ++dz) {
                if (std::abs(dx) == 2 && std::abs(dz) == 2) continue; // round it off
                world.PlaceIfAir(wx + dx, y, wz + dz, Blocks::Leaves);
            }

    // 3) A tighter 3x3 layer at the top log.
    for (int dx = -1; dx <= 1; ++dx)
        for (int dz = -1; dz <= 1; ++dz)
            world.PlaceIfAir(wx + dx, topY, wz + dz, Blocks::Leaves);

    // 4) A plus-shaped cap one block above the trunk.
    world.PlaceIfAir(wx,     topY + 1, wz,     Blocks::Leaves);
    world.PlaceIfAir(wx + 1, topY + 1, wz,     Blocks::Leaves);
    world.PlaceIfAir(wx - 1, topY + 1, wz,     Blocks::Leaves);
    world.PlaceIfAir(wx,     topY + 1, wz + 1, Blocks::Leaves);
    world.PlaceIfAir(wx,     topY + 1, wz - 1, Blocks::Leaves);
}

Two details carry weight. Leaves go down with PlaceIfAir, never SetBlock, so a canopy that brushes against a cliff or another tree fills only the open cells and leaves solid terrain untouched. And look hard at wx + dx: when a trunk sits within two blocks of the column edge, some of those leaf writes land in the next column. That single line is the whole cross-chunk problem, so let us face it directly.

The cross-chunk problem

A tree anchored near a border reaches across it. Its canopy — sometimes even the trunk's overhang — wants to write blocks into a neighbor column that may not have been generated yet. Two naive reactions both fail:

  • Write only into your own column, drop the rest. The leaves that fell outside are discarded, and you get an ugly half-tree sliced flat along the chunk seam.
  • Write blindly into the neighbor anyway. If that neighbor is later generated from scratch, your leaves are overwritten and lost — or, if the neighbor also grows its own copy of the same tree, you get duplicates and doubled trunks straddling the line.

The fix: staged generation with a neighbor gate

Give every chunk a generation stage and let it advance through the stages in order. The rule that makes cross-chunk writes safe is a single gate on one edge of that state machine: a chunk may be decorated only once all of its horizontal neighbors have reached at least TerrainReady. Once the neighbors have their block arrays filled, a feature is free to write into them.

Now trace why this eliminates all three artifacts at once. Placement is seeded from world coordinates, and each tree is owned by exactly one column — the one its trunk sits in. So:

  • No half-trees. The gate guarantees the neighbor's block array exists before A decorates, so A's canopy writes land in B instead of being dropped.
  • No duplicates. A tree is stamped only by the pass of the column that owns its trunk. B's own decoration pass grows B's trees, never A's — the anchor is unambiguous, so nobody grows the same tree twice.
  • No seams. Because the writes are deterministic and the ownership is fixed, A-into-B and B-into-A compose the same way regardless of which column decorates first.

The "regardless of order" part leans on the write discipline from earlier: leaves use PlaceIfAir and ores replace Blocks::Stone only, so features fill air or swap rock but never clobber a neighbor's finished feature. Owning each structure by a single anchor plus gating decoration behind neighbor readiness is the whole solution.

The gate is not optional

Decorate a chunk before its neighbors are TerrainReady — or worse, write features into a neighbor that has not been generated — and you are back to half-trees where canopies get clipped, duplicated trunks where two columns claim the same tree, and visible seams where features stop dead at a border. The neighbor-readiness gate is the one line of defense against all of them; never let a chunk skip it.

Alternative: a deferred write queue

Staged generation is the clean, recommended answer, but there is a lighter-weight option worth knowing. When a feature writes a block that falls outside a generated chunk, do not drop it — push the edit into a per-target mailbox. When that target chunk is later generated, it drains its mailbox before it does anything else:

struct BlockEdit { int wx, wy, wz; BlockId id; };
std::unordered_map<ColumnKey, std::vector<BlockEdit>> pending;

// A feature block that lands in a not-yet-generated neighbor:
if (Chunk* c = world.GetChunk(cx, cz))
    c->SetBlock(lx, ly, lz, id);              // neighbor exists -> write now
else
    pending[{ cx, cz }].push_back({ wx, wy, wz, id });   // stash for later

// When that neighbor is finally generated, apply its queued edits first:
for (const BlockEdit& e : pending[{ cx, cz }])
    world.SetBlock(e.wx, e.wy, e.wz, e.id);

It works, and it avoids the strict ordering the gate imposes, but it trades that for a persistent side table you must carefully save and restore — those pending edits are real world state until they are applied. Staged generation keeps everything as a pure function of the seed, which is why we prefer it.

Ores: seeding veins underground

Ores are decoration too — they just hide in the rock instead of sitting on the surface. There are two good approaches. The first is a 3D density threshold within a depth band: sample noise3D(x, y, z), and where it crosses a threshold inside the right range of y, replace stone with the ore — coal shallow and common, iron in the middle, diamond deep and rare. The second, which we will code, grows deterministic veins: pick a start cell from the PRNG and random-walk a few steps, converting Blocks::Stone to the ore as you go. The walk gives the connected, blobby clusters real ore looks like.

void WorldGen::PlaceOres(ChunkManager& world, ChunkCoord coord, Rng& rng) {
    struct Vein { BlockId ore; int tries, minY, maxY, length; };
    static const Vein kVeins[] = {
        { Blocks::CoalOre,    18, 16, 96, 7 },   // common, shallow to high
        { Blocks::IronOre,    10,  8, 56, 5 },   // less common, mid depth
        { Blocks::DiamondOre,  3,  2, 16, 4 },   // rare, deep
    };

    for (const Vein& v : kVeins)
    for (int i = 0; i < v.tries; ++i) {
        // A start cell inside this column, within the ore's depth band.
        int wx = LocalToWorld(coord.x, rng.nextInt(CHUNK_SIZE));
        int wz = LocalToWorld(coord.z, rng.nextInt(CHUNK_SIZE));
        int wy = v.minY + rng.nextInt(v.maxY - v.minY);

        // Random-walk a short vein, replacing solid Stone only.
        for (int step = 0; step < v.length; ++step) {
            if (world.GetBlock(wx, wy, wz) == Blocks::Stone)
                world.SetBlock(wx, wy, wz, v.ore);
            switch (rng.nextInt(6)) {            // step to a face neighbor
                case 0: ++wx; break;  case 1: --wx; break;
                case 2: ++wy; break;  case 3: --wy; break;
                case 4: ++wz; break;  case 5: --wz; break;
            }
        }
    }
}

Replacing only stone matters: the walk can wander into a cave or an air pocket, and we do not want a floating vein hanging in a cavern. A vein near the column edge can also step across a border — the same cross-chunk case as trees, and it is covered by the same neighbor gate, so ores need no special handling.

Glowing ores feed the bloom pipeline

Register an emissive ore — think a glowstone-like ore, or a magma vein — with emissive = true and a non-zero lightLevel, exactly as the registry lesson set up Blocks::Glowstone. Its faces output emissive HDR color above 1.0, which is precisely what the Module 10 post-processing chain thresholds and blurs. Drop a few of these veins in the dark and the bloom pass makes them glow through the cave for free — decoration and your renderer talking to each other.

Tying it together: the WorldGen stages

Now assemble the pieces into the object the streaming system drives. We add a Stage enum and a Stage stage field to the Chunk from the first Module 11 lesson, and give WorldGen the terrain generator plus the decoration methods we just wrote. For decoration we address chunks by their horizontal column, so GetChunk(x, z) returns the column record and the world-aware SetBlock routes each write to the right vertical slice.

#pragma once
#include <cstdint>
#include "TerrainGenerator.h"
#include "ChunkManager.h"
#include "Rng.h"

// How far a chunk has progressed. The order is load-bearing: a stage may only
// advance once its predecessor -- and, for Decorated, its neighbors -- is done.
enum class Stage : uint8_t {
    Empty,          // allocated, no blocks yet
    TerrainReady,   // heightmap + caves written; safe for neighbors to touch
    Decorated,      // flora, trees and ores stamped in
    Meshed,         // vertex buffers built and uploaded
};

class WorldGen {
public:
    explicit WorldGen(uint32_t seed) : m_worldSeed(seed), m_terrain(seed) {}

    void Update(ChunkManager& world);   // drives every chunk through the stages

private:
    void Decorate    (ChunkManager& world, Chunk& chunk);
    void ScatterFlora(ChunkManager& world, ChunkCoord coord, Rng& rng);
    void PlaceTrees  (ChunkManager& world, ChunkCoord coord, Rng& rng);
    void PlaceOres   (ChunkManager& world, ChunkCoord coord, Rng& rng);
    void PlaceTree   (ChunkManager& world, int wx, int groundY, int wz, Rng& rng);
    bool CanDecorate (ChunkManager& world, ChunkCoord coord) const;

    uint32_t         m_worldSeed;
    TerrainGenerator m_terrain;
};
#include "WorldGen.h"

// The neighbor gate: a column may decorate only once all eight of its
// horizontal neighbors (the surrounding 3x3) are at least TerrainReady,
// so a feature is free to write blocks into any of them.
bool WorldGen::CanDecorate(ChunkManager& world, ChunkCoord c) const {
    for (int dz = -1; dz <= 1; ++dz)
    for (int dx = -1; dx <= 1; ++dx) {
        Chunk* n = world.GetChunk(c.x + dx, c.z + dz);
        if (!n || n->stage < Stage::TerrainReady)
            return false;
    }
    return true;
}

void WorldGen::Decorate(ChunkManager& world, Chunk& chunk) {
    // One PRNG per column, seeded from world coords: deterministic and
    // independent of any neighbor, so parallel workers never disagree.
    Rng rng{ hashChunk(chunk.coord.x, chunk.coord.z, m_worldSeed) };

    // Fixed draw order keeps the random stream reproducible run to run.
    ScatterFlora(world, chunk.coord, rng);
    PlaceTrees  (world, chunk.coord, rng);
    PlaceOres   (world, chunk.coord, rng);
}

void WorldGen::Update(ChunkManager& world) {
    // 1. Terrain: every Empty chunk becomes TerrainReady.
    for (Chunk* c : world.LoadedChunks())
        if (c->stage == Stage::Empty) {
            m_terrain.GenerateChunk(c->coord, *c);
            c->stage = Stage::TerrainReady;
        }

    // 2. Decoration: only where the neighbor gate is satisfied.
    for (Chunk* c : world.LoadedChunks())
        if (c->stage == Stage::TerrainReady && CanDecorate(world, c->coord)) {
            Decorate(world, *c);
            c->stage = Stage::Decorated;
        }

    // 3. Light, then mesh: their own lessons. They wait for the 3x3 neighbors
    //    to be Decorated so edge faces and baked skylight come out correct.
}

Read the flow once more. Terrain fills chunks independently; decoration waits at the gate until the neighborhood is ready and then stamps flora, trees and ores, drawing from a PRNG seeded purely from the column's world position; lighting and meshing follow, each waiting on their own neighborhood. Nothing consults a clock or a global — the same seed rebuilds the same forest, every launch, on every thread. That is what turns a heightmap into a place.

Recap
  • Decoration is a feature pass that runs after base terrain and before lighting and meshing — leaves attenuate skylight, so growing them late forces a re-light and re-mesh.
  • Seed a per-column PRNG from hashChunk(x, z, worldSeed) and draw features in a fixed order: placement becomes reproducible, thread-safe, and save-friendly (you persist only player edits, never generated trees).
  • Flora is a per-column write into the air above the surface; trees need spacing, from per-column rejection sampling and/or a low-frequency forest-density noise.
  • Build a tree as a Wood trunk plus a Leaves canopy placed with PlaceIfAir, so leaves fill open air and never eat terrain — but a canopy near a border reaches into the next column.
  • The cross-chunk problem is solved by staged generation: a chunk decorates only once its 3×3 neighbors are TerrainReady, and each feature is owned by one anchor column — no half-trees, no duplicates, no seams. A deferred write queue is the lighter alternative.
  • Ore veins random-walk through stone, replacing Blocks::Stone only; emissive ores output HDR color that the Module 10 bloom pipeline turns into a glow.
