Frustum Culling & Draw-Call Batching

Even after greedy meshing squeezes a chunk down to a few hundred triangles, an open world still holds hundreds of loaded chunks — and on any given frame the camera can see only a handful of them. Handing the GPU every chunk, most of them behind you or off to the side, burns vertex-shader and driver time on geometry that never touches a pixel. This lesson adds the cheapest, highest-leverage optimization in the whole renderer: build the camera's view volume as six planes, and before each chunk's draw call ask one quick question — could any part of this box be on screen? If the answer is no, skip it entirely.

What you'll learn
  • Why frustum culling still pays off after greedy meshing, and what the camera's six clip planes are.
  • How to represent a plane as a unit normal plus an offset, and exactly when normalization matters.
  • The Gribb-Hartmann method: pull all six planes straight from the view-projection matrix, minding row-major vs column-major and the D3D/Metal [0,1] vs OpenGL [-1,1] depth range.
  • The positive-vertex test that rejects a chunk's world-space AABB against a plane by touching one corner, not eight.
  • How to wire the test into the render loop and keep driver overhead tiny with one shared shader and one block texture array.

The camera only sees a box-shaped slice of the world

A perspective camera does not see everything — it sees a truncated pyramid called the view frustum. Six flat planes bound it: left and right pinch it in from the sides, bottom and top from above and below, and the near and far planes cap it front and back. Anything outside all six is invisible this frame. A chunk that sits behind you, or off past the right edge, or beyond your view distance, will be transformed by the vertex shader and then thrown away wholesale by the clipper — pure wasted submission. Frustum culling does that rejection up front, on the CPU, with a test far cheaper than a draw call.

A plane is a normal and an offset

Each of the six sides is an infinite flat plane, and we store it the standard way: a normal n = (a, b, c) and a scalar d, so the plane is the set of points where a·x + b·y + c·z + d = 0. We choose the sign of n so it points into the frustum. Then a world point p is on the inside of that plane exactly when Dot(n, p) + d >= 0. If n is a unit vector, that same expression is the signed distance from p to the plane, measured in world units — positive inside, negative outside. 

#include "Vector3.h"   // engine::Vector3, Dot, Length

namespace engine {

// Plane a*x + b*y + c*z + d = 0. After Normalize(), n is a UNIT normal that
// points INTO the frustum, so a world point p is inside when
// Dot(n, p) + d >= 0 — and that value is p's signed distance to the plane.
struct Plane {
    Vector3 n;            // (a, b, c)
    float   d = 0.0f;

    Plane() = default;
    Plane(float a, float b, float c, float dd) : n(a, b, c), d(dd) {}

    void Normalize() {
        float inv = 1.0f / n.Length();   // length of (a, b, c)
        n = n * inv;
        d *= inv;                        // scale d too, or distances are wrong
    }
};

// Axis-aligned bounding box in WORLD space: two opposite corners.
struct AABB {
    Vector3 min;
    Vector3 max;
};

} // namespace engine
Normalize for distance, not for the yes/no test

The raw planes come out un-normalized, and for a pure inside/outside decision that is actually fine — scaling a plane by a positive number never flips the sign of Dot(n, p) + d, and the positive-vertex test below only looks at the sign of each normal component. But the moment you want a metric distance — cull spheres by radius, fade LODs by range, add a little slop near the camera — an un-normalized plane hands you a meaningless number. Normalize once, up front, and never think about it again.

Extracting the six planes (Gribb-Hartmann)

The elegant part: you do not build the planes from the camera's position and angles. They fall straight out of the combined view-projection matrix — the very matrix from the Camera Component lesson that maps a world point to clip space. This is the Gribb-Hartmann method, and it works because clip-space itself is defined by inequalities on the six sides.

A world point p becomes a clip-space point clip = p · VP. After the perspective divide, a point is inside the frustum when its normalized coordinates satisfy -clip.w <= clip.x <= clip.w, -clip.w <= clip.y <= clip.w, and (for Direct3D and Metal) 0 <= clip.z <= clip.w. Rearrange each of those six inequalities into the form a·x + b·y + c·z + d >= 0 and you have read a plane directly off the matrix. For example the left plane, clip.x + clip.w >= 0, is column 0 plus column 3 of VP.

Rows or columns? It depends on your matrix convention

MiniEngine follows DirectXMath's row-vector convention: a point is a row and clip = point · VP, so clip.x is the point dotted with column 0 of VP. That is why our planes are column combinations. A library that puts the vector on the right (clip = VP · point, the OpenGL/GLM style) transforms points with the transpose, so the identical planes come from the matrix rows instead — that is the form the original Gribb-Hartmann paper prints. Same six planes, read along whichever axis your convention transforms points.

Here is the extraction for our engine. We take the raw combined matrix from Camera::GetViewProjection(), store it row-major so we can index individual elements, and write out the six column combinations. Note the near plane: DirectXMath's XMMatrixPerspectiveFovLH maps the near plane to clip.z = 0 (the Direct3D and Metal depth range is [0, 1]), so the near plane is simply clip.z >= 0 — column 2 alone. Under OpenGL's [-1, 1] range it would instead be clip.z + clip.w >= 0, column 2 plus column 3. 

#include <DirectXMath.h>
#include "Frustum.h"   // engine::Plane

using namespace DirectX;

// Extract the six frustum planes from a combined view-projection matrix.
// Pass the RAW row-vector matrix (Camera::GetViewProjection()) — the one
// BEFORE you transpose it for the constant buffer.
void ExtractFrustumPlanes(FXMMATRIX viewProj, engine::Plane out[6]) {
    XMFLOAT4X4 m;
    XMStoreFloat4x4(&m, viewProj);   // row-major; m._<row><col>, 1-indexed

    // Row-vector convention: clip.x/.y/.z/.w are the four COLUMNS of VP dotted
    // with the point. Column K = (m._1K, m._2K, m._3K, m._4K). Gribb-Hartmann
    // combines those columns into the six planes.

    // left:   clip.x + clip.w >= 0   ->  col0 + col3
    out[0] = engine::Plane(m._11 + m._14, m._21 + m._24, m._31 + m._34, m._41 + m._44);
    // right:  clip.w - clip.x >= 0   ->  col3 - col0
    out[1] = engine::Plane(m._14 - m._11, m._24 - m._21, m._34 - m._31, m._44 - m._41);
    // bottom: clip.y + clip.w >= 0   ->  col1 + col3
    out[2] = engine::Plane(m._12 + m._14, m._22 + m._24, m._32 + m._34, m._42 + m._44);
    // top:    clip.w - clip.y >= 0   ->  col3 - col1
    out[3] = engine::Plane(m._14 - m._12, m._24 - m._22, m._34 - m._32, m._44 - m._42);
    // near (D3D/Metal [0,1] depth): clip.z >= 0   ->  col2 alone
    out[4] = engine::Plane(m._13, m._23, m._33, m._43);
    // far:    clip.w - clip.z >= 0   ->  col3 - col2
    out[5] = engine::Plane(m._14 - m._13, m._24 - m._23, m._34 - m._33, m._44 - m._43);

    // Metric distances need unit normals — normalize every plane.
    for (int i = 0; i < 6; ++i)
        out[i].Normalize();
}
Don't drop the near and far planes, and don't feed the transposed matrix

It is tempting to extract only the four side planes — but then a chunk directly behind the camera passes all four and gets drawn every frame, and nothing bounds your view distance. Extract all six. Equally important: feed ExtractFrustumPlanes the raw GetViewProjection(), not the transposed copy you upload to the constant buffer (see the Camera Component lesson). Transposing turns the column combinations above into the wrong rows, and either everything culls or nothing does.

Testing a chunk with the positive-vertex trick

A chunk occupies an axis-aligned box in world space. The naive test would check all eight corners of that box against all six planes — 48 dot products per chunk. We can do far better. A box is entirely outside a plane only if every corner is on the plane's negative side, and the corner most likely to still be on the positive side is the one farthest along the plane's normal. That corner is the positive vertex (or p-vertex): for each axis, pick the box's max where the normal component is positive and its min where it is negative. If even the p-vertex is behind the plane, all eight corners are, and the whole box is out. One corner per plane, not eight. 

#include "Frustum.h"

// Returns false only if the box lies entirely outside the frustum. A box that
// is fully inside — or straddling a plane — returns true, so we keep it.
bool AABBInFrustum(const engine::AABB& box, const engine::Plane planes[6]) {
    using engine::Vector3;
    for (int i = 0; i < 6; ++i) {
        const engine::Plane& p = planes[i];

        // Positive vertex: the ONE corner farthest along this plane's normal.
        // max on each axis where the normal is positive, min where negative.
        Vector3 pv(
            p.n.x >= 0.0f ? box.max.x : box.min.x,
            p.n.y >= 0.0f ? box.max.y : box.min.y,
            p.n.z >= 0.0f ? box.max.z : box.min.z);

        // If even that farthest corner is behind the plane, all eight are —
        // the box is fully outside this plane and cannot be in the frustum.
        if (Dot(p.n, pv) + p.d < 0.0f)
            return false;
    }
    return true;   // inside, or intersecting at least one plane — draw it
}
A chunk that straddles a plane must stay in

This is a conservative test on purpose. We only reject a box when it is completely outside a single plane; a chunk that pokes through the frustum wall is kept and drawn, because part of it is visible. Never require the box to be fully inside all six planes — that would clip away everything at the screen edges. False positives (drawing a chunk that turns out fully off-screen) are cheap; false negatives (dropping a visible chunk) are holes in your world.

Wiring it into the render loop

The plumbing is small. Once per frame, extract the six planes from the current camera. Then, as you walk the loaded chunks, build each chunk's world-space AABB and call AABBInFrustum before its draw. A chunk's box is trivial: its origin is coord × CHUNK_SIZE (from Voxels, Chunks & World Coordinates) and it spans one CHUNK_SIZE cube from there. 

using namespace engine;

// A chunk's world-space bounding box: origin = coord * CHUNK_SIZE, size 16.
AABB ChunkWorldAABB(ChunkCoord c) {
    Vector3 mn(float(c.x * CHUNK_SIZE),
               float(c.y * CHUNK_SIZE),
               float(c.z * CHUNK_SIZE));
    Vector3 mx(mn.x + CHUNK_SIZE, mn.y + CHUNK_SIZE, mn.z + CHUNK_SIZE);
    return { mn, mx };
}

void RenderChunks(RenderContext& ctx, World& world, Camera* camera) {
    // Build the frustum once per frame from the RAW view-projection matrix.
    Plane frustum[6];
    ExtractFrustumPlanes(camera->GetViewProjection(), frustum);

    BindVoxelShader(ctx);        // ONE shader/material for every chunk
    BindBlockTextureArray(ctx);  // ONE Texture2DArray, bound once (module 10)

    int drawn = 0;
    for (auto& [coord, chunk] : world.LoadedChunks()) {
        if (!chunk.mesh.HasGeometry())
            continue;                                  // empty / all-air chunk

        if (!AABBInFrustum(ChunkWorldAABB(coord), frustum))
            continue;                                  // off-screen: skip it

        SetPerChunkOrigin(ctx, coord);                 // tiny constant update
        chunk.mesh.Draw(ctx);                          // one indexed draw call
        ++drawn;
    }
}

With a 16-chunk view distance you might have well over a thousand chunks loaded, yet a 70° camera sees only a wedge of them. Culling routinely drops the drawn count by 60–80%, and every skipped chunk is a draw call, a pile of vertices, and a chunk of index bandwidth you never spend. Because the test runs on a single AABB — three comparisons and one dot product per plane — it costs a rounding error next to the draw it prevents.

Draw-call batching: one shader, one texture array

Culling decides which chunks to draw; batching makes each draw as cheap as possible. The enemy is state change. Every time you swap a shader, a texture, a sampler, or a constant buffer, the driver has to validate and re-bind, and that CPU cost dwarfs the actual triangle work for a chunk. The voxel engine sidesteps almost all of it by construction: every chunk uses the same shader and the same material, and thanks to the Texture2DArray from Texture Atlases & Texture Arrays, every block texture lives in one resource bound a single time at the top of the frame.

That leaves exactly one thing that differs per chunk: its geometry. So a chunk draw is just "bind this vertex buffer, set one small per-chunk constant, DrawIndexed." The per-chunk constant is the chunk's world origin. We keep each chunk's vertices in chunk-local space (0..16) and let the vertex shader add the origin to reach world space — no per-chunk world matrix, no re-upload of geometry, just twelve bytes that change between draws. 

cbuffer PerFrame : register(b0) { float4x4 gViewProj; };          // set once/frame
cbuffer PerChunk : register(b1) { float3 gChunkOrigin; float _pad; }; // per chunk

Texture2DArray gBlocks  : register(t0);   // bound ONCE for every chunk
SamplerState   gSampler : register(s0);

VSOutput VSMain(VSInput vin) {
    VSOutput o;
    // Vertices are stored in chunk-local space (0..16). Add the per-chunk
    // origin to reach world space, then project. No per-chunk world matrix.
    float3 worldPos = vin.position + gChunkOrigin;
    o.clip  = mul(float4(worldPos, 1.0), gViewProj);  // gViewProj uploaded
    o.uv    = vin.uv;                                  // transposed, as in the
    o.layer = vin.layer;                               // Camera Component lesson
    return o;
}

The Metal path is the same shape: gViewProj and gChunkOrigin arrive as constant buffer arguments, the block textures are one texture2d_array bound once, and the vertex function adds the origin before projecting. In both backends the driver spends its time drawing, not re-validating state.

Cull first, then batch — they compound

Run the cheap AABB test before every draw so off-screen chunks never reach the driver at all, and give every chunk one shared shader plus the single block texture array so the draws that survive are almost free to submit. Frustum culling shrinks the number of draws; batching shrinks the cost of each. Together they take a "hundreds of chunks, hundreds of state changes" frame down to a tight loop of cheap indexed draws.

Beyond the frustum: occlusion culling

Frustum culling removes what is off-screen, but not what is on-screen and hidden. Stand in a cave or at the bottom of a ravine and dozens of chunks in front of you are fully behind solid rock — inside the frustum, invisible, still drawn. Removing those is occlusion culling, and it is the natural next step.

Two approaches are worth knowing. Hardware occlusion queries ask the GPU to render a chunk's bounding box (depth-only, cheap) and report how many pixels passed the depth test; if none did, the chunk is hidden and you skip its real draw next frame. The results lag a frame and need care to avoid stalls, so many voxel engines prefer a cheaper CPU heuristic: a chunk whose six neighbors are all completely solid opaque blocks cannot possibly be seen, so flag it "sealed" at mesh time and skip it outright. That single test alone erases the interiors of mountains. Either way, layer occlusion culling after frustum culling — reject the off-screen chunks first with the near-free AABB test, then spend more effort deciding which of the remaining, on-screen chunks are actually blocked.

Recap
  • The camera sees a six-plane frustum; any chunk fully outside a plane is invisible, so testing it before drawing saves a whole draw call and its geometry.
  • We store each plane as a normal n and offset d with the inside test Dot(n, p) + d >= 0; normalizing makes that value a metric distance (needed for distance work, optional for a pure yes/no).
  • Gribb-Hartmann reads all six planes from the view-projection matrix — column combinations in our row-vector DirectXMath convention (rows under OpenGL/GLM), with the near plane as clip.z >= 0 for D3D/Metal's [0,1] depth.
  • The positive-vertex test rejects a chunk's world-space AABB with one dot product per plane instead of eight, and it is conservative: straddling chunks are kept.
  • Feed the raw (un-transposed) GetViewProjection(), test in world space, and never omit the near/far planes — the three classic ways this silently breaks.
  • Batching makes the survivors cheap: one shared shader, one Texture2DArray bound once, and a per-chunk origin constant so chunks differ only by their vertex buffer — with occlusion culling as the next step beyond frustum.
