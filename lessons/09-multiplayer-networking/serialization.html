Serializing Game State

In the last lesson we opened a UDP socket and shot raw datagrams across the network. But a datagram is just a bag of bytes — the socket has no idea what a Transform or a player id is. In this lesson we build the translation layer: a tiny binary protocol plus a ByteWriter and ByteReader that turn engine state into bytes and back. Get this right and two MiniEngine instances can finally agree on where everything is.

What you'll learn
  • Why you must never memcpy a raw struct across machines
  • How to design a simple message protocol: a type byte followed by a payload
  • Building a ByteWriter/ByteReader over a std::vector<uint8_t>
  • Serializing a Transform and player id into a PlayerStatePacket
  • Keeping integers and floats portable with htonl/ntohl and bit-casting

Why not just send the struct?

It is tempting. You have a PlayerState struct in memory already, so why not point sendto at it and blast the raw bytes? Because the bytes on one machine do not mean the same thing on another. Two problems bite you:

  • Struct padding and alignment. The compiler inserts invisible gaps between fields so each one lands on a convenient address. A uint8_t followed by a float often has 3 padding bytes between them. Different compilers, flags, or target architectures pad differently, so sizeof and the field offsets can disagree between your two builds.
  • Endianness. The order in which a multi-byte integer is stored in memory (little-endian vs big-endian) is CPU-dependent. The value 0x01020304 is four bytes, but which byte comes first differs. Send it raw and the other machine may read 0x04030201.

Serialization fixes both by defining an explicit, byte-for-byte layout that both sides obey — independent of how any one compiler happens to lay out memory.

Never do this

sendto(sock, (const char*)&state, sizeof(state), 0, ...) works perfectly on your machine and fails the moment a friend on a different CPU or compiler joins. It is the single most common networking bug in hobby engines.

A tiny binary protocol

A protocol is just an agreement about what the bytes mean. Ours starts every message with a single type byte so the receiver knows how to interpret the rest. We define the types with an enum backed by uint8_t:

#include <cstdint>

// Every datagram starts with one of these as its first byte.
enum class MsgType : uint8_t {
    Hello        = 1,  // a client joining
    PlayerState  = 2,  // a Transform update for one player
    Goodbye      = 3,  // a client leaving
};

Fixed-size vs variable messages

Our PlayerState is fixed-size: the receiver always reads exactly 33 payload bytes. That is simplest and ideal for real-time transform updates. Some messages are variable-size — a chat string or a list of spawned objects — and those need a length field (say a uint16_t count) before the repeated data so the reader knows when to stop. We will lean on fixed-size packets here and mention the length-prefix trick where it matters.

ByteWriter and ByteReader

Rather than hand-pack every message, we build two small helpers. The ByteWriter appends values to a growing std::vector<uint8_t>; the ByteReader walks a received buffer, tracking an offset. Both convert integers to network byte order (big-endian) with htonl/htons so the wire format is identical everywhere.

Floats need one extra trick. There is no htonf, so we reinterpret the float's 32 bits as a uint32_t, byte-swap that, and send it. On the way back we reverse the steps. We use std::memcpy for the bit reinterpret — it is the only way to type-pun in C++ without undefined behavior (a plain cast through a pointer breaks strict aliasing).

#include <cstdint>
#include <cstring>   // std::memcpy
#include <vector>
#include <stdexcept>

#ifdef _WIN32
  #include <winsock2.h>   // htonl, htons, ntohl, ntohs
#else
  #include <arpa/inet.h>
#endif

class ByteWriter {
public:
    void writeU8(uint8_t v) {
        buf_.push_back(v);
    }
    void writeU16(uint16_t v) {
        uint16_t n = htons(v);              // host -> network order
        appendBytes(&n, sizeof(n));
    }
    void writeU32(uint32_t v) {
        uint32_t n = htonl(v);
        appendBytes(&n, sizeof(n));
    }
    void writeFloat(float f) {
        uint32_t bits;
        std::memcpy(&bits, &f, sizeof(bits)); // reinterpret the 32 bits, no UB
        writeU32(bits);                       // reuse the portable u32 path
    }

    const std::vector<uint8_t>& bytes() const { return buf_; }

private:
    void appendBytes(const void* p, size_t n) {
        const auto* b = static_cast<const uint8_t*>(p);
        buf_.insert(buf_.end(), b, b + n);
    }
    std::vector<uint8_t> buf_;
};

class ByteReader {
public:
    ByteReader(const uint8_t* data, size_t size)
        : data_(data), size_(size) {}

    uint8_t readU8() {
        need(1);
        return data_[offset_++];
    }
    uint16_t readU16() {
        need(2);
        uint16_t n;
        std::memcpy(&n, data_ + offset_, sizeof(n));
        offset_ += sizeof(n);
        return ntohs(n);                     // network -> host order
    }
    uint32_t readU32() {
        need(4);
        uint32_t n;
        std::memcpy(&n, data_ + offset_, sizeof(n));
        offset_ += sizeof(n);
        return ntohl(n);
    }
    float readFloat() {
        uint32_t bits = readU32();
        float f;
        std::memcpy(&f, &bits, sizeof(f));
        return f;
    }

private:
    void need(size_t n) {
        if (offset_ + n > size_)
            throw std::runtime_error("ByteReader: buffer underrun");
    }
    const uint8_t* data_;
    size_t size_;
    size_t offset_ = 0;
};
Always bounds-check the reader

A UDP datagram can be truncated, malformed, or forged. The need() guard turns "read past the end" — which would otherwise be a silent memory bug — into a clean exception you can catch and drop the packet. Never trust the length of data that came off a socket.

Serializing a Transform and a PlayerStatePacket

Recall the Transform component from the engine lessons: a position and a rotation quaternion (we ignore scale here since it rarely changes for a networked player). We add free functions to serialize it, then wrap it in a PlayerStatePacket that carries the owning player's id.

#include "ByteBuffer.h"
#include "Protocol.h"

// Matches the engine's Transform (from the component-system lessons).
struct Vec3 { float x, y, z; };
struct Quat { float x, y, z, w; };

struct Transform {
    Vec3 position;
    Quat rotation;
};

struct PlayerStatePacket {
    uint32_t  playerId;
    Transform transform;

    // Write: type byte, then the payload, into a fresh byte vector.
    std::vector<uint8_t> Serialize() const {
        ByteWriter w;
        w.writeU8(static_cast<uint8_t>(MsgType::PlayerState));

        w.writeU32(playerId);

        w.writeFloat(transform.position.x);
        w.writeFloat(transform.position.y);
        w.writeFloat(transform.position.z);

        w.writeFloat(transform.rotation.x);
        w.writeFloat(transform.rotation.y);
        w.writeFloat(transform.rotation.z);
        w.writeFloat(transform.rotation.w);

        return w.bytes();  // 1 + 4 + 28 = 33 bytes
    }

    // Read: the caller has already peeked the type byte and dispatched here,
    // so 'r' is positioned just after it.
    static PlayerStatePacket Deserialize(ByteReader& r) {
        PlayerStatePacket p;
        p.playerId = r.readU32();

        p.transform.position.x = r.readFloat();
        p.transform.position.y = r.readFloat();
        p.transform.position.z = r.readFloat();

        p.transform.rotation.x = r.readFloat();
        p.transform.rotation.y = r.readFloat();
        p.transform.rotation.z = r.readFloat();
        p.transform.rotation.w = r.readFloat();

        return p;
    }
};

Notice how Serialize and Deserialize mirror each other exactly: same fields, same order. That symmetry is the whole discipline of serialization. If you ever add a field, add it to both in the same position, or the two sides silently disagree.

Keep the type byte out of the payload

Serialize writes the MsgType, but Deserialize does not read it — the dispatcher does that first to decide which packet type to build. Splitting the responsibility keeps each Deserialize function focused on just its own payload.

Sending it between MiniEngine instances

Now we connect it to the UDP socket from the previous lesson. On the sending side, MiniEngine grabs a player's Transform, serializes a packet, and calls sendto. On the receiving side it reads the datagram, peeks the type byte, dispatches, and applies the state to the matching remote player.

// --- Sending: called each network tick for the local player ---
void SendPlayerState(SOCKET sock, const sockaddr_in& to,
                     uint32_t myId, const Transform& xf) {
    PlayerStatePacket pkt{ myId, xf };
    std::vector<uint8_t> bytes = pkt.Serialize();

    int sent = sendto(sock,
                      reinterpret_cast<const char*>(bytes.data()),
                      static_cast<int>(bytes.size()),
                      0,
                      reinterpret_cast<const sockaddr*>(&to),
                      sizeof(to));
    if (sent == SOCKET_ERROR) {
        // WSAGetLastError() gives the real reason on Windows.
        printf("sendto failed: %d\n", WSAGetLastError());
    }
}

// --- Receiving: called when recvfrom returns a datagram ---
void OnDatagram(const uint8_t* data, int len, Scene& scene) {
    if (len < 1) return;            // not even a type byte
    ByteReader r(data, static_cast<size_t>(len));

    try {
        auto type = static_cast<MsgType>(r.readU8());
        switch (type) {
            case MsgType::PlayerState: {
                PlayerStatePacket p = PlayerStatePacket::Deserialize(r);
                // Find the GameObject for this player and update its Transform.
                if (GameObject* go = scene.FindPlayer(p.playerId)) {
                    go->GetComponent<Transform>()->position = p.transform.position;
                    go->GetComponent<Transform>()->rotation = p.transform.rotation;
                }
                break;
            }
            default:
                // Unknown or unhandled type: ignore it safely.
                break;
        }
    } catch (const std::exception& e) {
        // A malformed packet threw in the reader. Drop it, keep running.
        printf("bad packet dropped: %s\n", e.what());
    }
}

That is the full round trip. The same 33 bytes that leave one MiniEngine arrive at another and become a moving object on screen — regardless of whether the two machines share a CPU architecture or a compiler. Because every integer went through htonl/ntohl and every float was bit-cast rather than blindly copied, the layout is nailed down and reproducible.

Serialization is not compression or encryption

Our 33-byte packet is honest but not tiny. Real engines shrink transform updates by quantizing floats (sending a rotation as 3 bytes of compressed quaternion, for example) and by sending only what changed. That is an optimization to layer on later — get the correct, portable version working first.

How versioning creeps in

The moment two people run slightly different builds, the two sides can disagree about the packet layout. A common guard is to send a small protocol version in the Hello message and refuse mismatched clients. You do not need it today, but keep it in mind: a serialized format is a contract, and contracts have versions.

Recap
  • Never memcpy/send raw structs: padding and endianness differ between machines, corrupting the data.
  • A protocol is an agreed byte layout — ours leads with a MsgType byte, then a type-specific payload.
  • ByteWriter/ByteReader centralize the packing: htonl/htons for integers, bit-cast via memcpy for floats.
  • Serialize and Deserialize must mirror each other field-for-field, in the same order.
  • Always bounds-check the reader and drop malformed datagrams — data off a socket is never trusted.
  • Fixed-size packets suit real-time transforms; variable data needs a length prefix.
