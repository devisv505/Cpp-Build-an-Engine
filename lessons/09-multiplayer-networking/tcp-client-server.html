A TCP Client & Server

In the last lesson you initialized Winsock and learned what a socket is. Now we make it real: two small programs that actually talk to each other. We'll build a TCP echo server that repeats back whatever it receives, and a client that connects, sends a greeting, and prints the reply. This is the "hello world" of networking, and it's the foundation the MiniEngine networking layer will grow from.

What you'll learn
  • The server call sequence: socket → bind → listen → accept → recv/send.
  • The client call sequence: socket → connect → send → recv.
  • How to fill a sockaddr_in with a port and an address using inet_pton.
  • Why accept and recv block, and how to check every call for errors with WSAGetLastError.
  • Why TCP is a byte stream — message boundaries are not preserved, so you must frame your own messages.

The shape of a TCP conversation

TCP is connection-oriented. Before any data flows, the two sides perform a handshake and establish a reliable, ordered channel. One side plays the server: it picks a port, waits, and answers whoever knocks. The other side plays the client: it reaches out to a specific address and port. Once connected, the link is symmetric — either side can send and recv.

The server

The server's job is to own a port and hand back a fresh socket for each client that connects. Read the five steps below, then we'll walk through the code.

  1. socket() — create a listening socket.
  2. bind() — claim a local address and port so the OS routes incoming connections to us.
  3. listen() — mark the socket as passive; it will now queue incoming connections.
  4. accept() — block until a client connects, then return a new socket dedicated to that client.
  5. recv() / send() — talk over the client socket, then closesocket().
// server.cpp — a minimal TCP echo server (Winsock2)
#include <winsock2.h>
#include <ws2tcpip.h>   // for inet_ntop
#include <cstdio>
#include <string>

#pragma comment(lib, "ws2_32.lib")   // link the Winsock library

constexpr unsigned short PORT = 27015;

int main() {
    WSADATA wsa;
    if (WSAStartup(MAKEWORD(2, 2), &wsa) != 0) {
        printf("WSAStartup failed: %d\n", WSAGetLastError());
        return 1;
    }

    // 1) Create a TCP socket (IPv4, stream, TCP).
    SOCKET listenSock = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP);
    if (listenSock == INVALID_SOCKET) {
        printf("socket failed: %d\n", WSAGetLastError());
        WSACleanup();
        return 1;
    }

    // 2) Describe the local address to bind to.
    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;   // accept on any local interface
    addr.sin_port = htons(PORT);         // host-to-network byte order!

    if (bind(listenSock, (sockaddr*)&addr, sizeof(addr)) == SOCKET_ERROR) {
        printf("bind failed: %d\n", WSAGetLastError());
        closesocket(listenSock);
        WSACleanup();
        return 1;
    }

    // 3) Start listening. SOMAXCONN = a reasonable backlog queue length.
    if (listen(listenSock, SOMAXCONN) == SOCKET_ERROR) {
        printf("listen failed: %d\n", WSAGetLastError());
        closesocket(listenSock);
        WSACleanup();
        return 1;
    }
    printf("Listening on port %d ...\n", PORT);

    // 4) Block until a client connects. accept() returns a NEW socket
    //    just for that client; listenSock keeps listening.
    sockaddr_in clientAddr{};
    int clientLen = sizeof(clientAddr);
    SOCKET clientSock = accept(listenSock, (sockaddr*)&clientAddr, &clientLen);
    if (clientSock == INVALID_SOCKET) {
        printf("accept failed: %d\n", WSAGetLastError());
        closesocket(listenSock);
        WSACleanup();
        return 1;
    }

    char ip[INET_ADDRSTRLEN]{};
    inet_ntop(AF_INET, &clientAddr.sin_addr, ip, sizeof(ip));
    printf("Client connected from %s\n", ip);

    // 5) Echo loop: read a chunk, send the same bytes back.
    char buf[512];
    for (;;) {
        int n = recv(clientSock, buf, sizeof(buf), 0);
        if (n > 0) {
            printf("Received %d bytes, echoing back.\n", n);
            int sent = 0;
            while (sent < n) {                 // send() may not send it all at once
                int m = send(clientSock, buf + sent, n - sent, 0);
                if (m == SOCKET_ERROR) {
                    printf("send failed: %d\n", WSAGetLastError());
                    break;
                }
                sent += m;
            }
        } else if (n == 0) {
            printf("Client closed the connection.\n");   // graceful shutdown
            break;
        } else {
            printf("recv failed: %d\n", WSAGetLastError());
            break;
        }
    }

    closesocket(clientSock);
    closesocket(listenSock);
    WSACleanup();
    return 0;
}
Two different sockets

A common beginner mistake is calling recv on the listening socket. Don't. listenSock only accepts connections; the socket returned by accept() — here clientSock — is the one you actually read from and write to.

Byte order: why htons?

Different CPUs store multi-byte integers in different orders. The network expects "big-endian" (network byte order), so we convert the port with htons (host-to-network short). INADDR_ANY is the special address 0.0.0.0, meaning "bind to every network interface on this machine" — convenient for a test server.

The client

The client is shorter. It has no bind, no listen, no accept. It creates a socket and calls connect(), pointing at the server's address. We use inet_pton (presentation-to-network) to turn the human-readable string "127.0.0.1" — the loopback address that means "this same machine" — into the binary form sockaddr_in wants.

// client.cpp — a minimal TCP echo client (Winsock2)
#include <winsock2.h>
#include <ws2tcpip.h>   // for inet_pton
#include <cstdio>
#include <cstring>

#pragma comment(lib, "ws2_32.lib")

constexpr unsigned short PORT = 27015;

int main() {
    WSADATA wsa;
    if (WSAStartup(MAKEWORD(2, 2), &wsa) != 0) {
        printf("WSAStartup failed: %d\n", WSAGetLastError());
        return 1;
    }

    // 1) Create a TCP socket.
    SOCKET sock = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP);
    if (sock == INVALID_SOCKET) {
        printf("socket failed: %d\n", WSAGetLastError());
        WSACleanup();
        return 1;
    }

    // 2) Describe the server we want to reach.
    sockaddr_in server{};
    server.sin_family = AF_INET;
    server.sin_port = htons(PORT);
    if (inet_pton(AF_INET, "127.0.0.1", &server.sin_addr) != 1) {
        printf("inet_pton failed: %d\n", WSAGetLastError());
        closesocket(sock);
        WSACleanup();
        return 1;
    }

    // 3) Connect — this performs the TCP handshake with the server.
    if (connect(sock, (sockaddr*)&server, sizeof(server)) == SOCKET_ERROR) {
        printf("connect failed: %d\n", WSAGetLastError());
        closesocket(sock);
        WSACleanup();
        return 1;
    }
    printf("Connected to server.\n");

    // 4) Send a message.
    const char* msg = "Hello, MiniEngine!";
    int len = (int)strlen(msg);
    if (send(sock, msg, len, 0) == SOCKET_ERROR) {
        printf("send failed: %d\n", WSAGetLastError());
        closesocket(sock);
        WSACleanup();
        return 1;
    }

    // 5) Read the echo back.
    char buf[512]{};
    int n = recv(sock, buf, sizeof(buf) - 1, 0);
    if (n > 0) {
        buf[n] = '\0';                 // recv does NOT null-terminate for us
        printf("Server echoed: %s\n", buf);
    } else if (n == 0) {
        printf("Server closed the connection.\n");
    } else {
        printf("recv failed: %d\n", WSAGetLastError());
    }

    closesocket(sock);
    WSACleanup();
    return 0;
}
Run the server first

Start server.exe in one terminal, then client.exe in another. If you launch the client alone, connect fails with error 10061 (connection refused) because nothing is listening on the port yet.

Reading the return values

Every Winsock call can fail, and the whole discipline of network code is checking that it didn't. Two sentinel values do most of the work, and WSAGetLastError tells you why something failed. Keep this table handy.

CallFailure looks likeWhat to do
socketreturns INVALID_SOCKETprint WSAGetLastError(), bail out
bind / listen / connect / sendreturns SOCKET_ERRORprint WSAGetLastError(), clean up
acceptreturns INVALID_SOCKETprint error, keep the listener alive if you can
recv0 = peer closed, SOCKET_ERROR = errorstop the loop; a positive value is the byte count
Blocking calls freeze the thread

By default accept and recv are blocking: the thread simply stops there until data arrives. That's fine for a demo, but you cannot call them on your engine's main thread — a stalled recv would freeze rendering. Real engines run networking on a separate thread or switch sockets to non-blocking mode. We'll return to this when we wire networking into MiniEngine's loop.

The big pitfall: TCP is a stream, not messages

Here is the single most important thing to internalize. When the client calls send("Hello, MiniEngine!"), it feels like you sent one message. You did not. TCP is a continuous byte stream with no built-in concept of where one message ends and the next begins.

That means a single recv might return only "Hello, Mini" and the rest arrives on the next call. Or, if the client sends two messages quickly, one recv might return both glued together: "Hello, MiniEngine!Ping". This is often called the "sticky packet" problem. Our echo server sidesteps it because it blindly bounces back whatever bytes it got — but any real protocol must recover message boundaries itself.

The standard fix is length-prefix framing: before each message, send its length as a fixed-size integer. The reader first reads exactly those length bytes, then reads exactly that many payload bytes.

// Send a length-prefixed message: [4-byte big-endian length][payload].
bool sendFramed(SOCKET s, const char* data, uint32_t len) {
    uint32_t netLen = htonl(len);                 // length in network byte order
    if (!sendAll(s, (const char*)&netLen, 4)) return false;
    return sendAll(s, data, len);
}

// Read exactly 'need' bytes, looping because recv may return partial data.
bool recvAll(SOCKET s, char* out, int need) {
    int got = 0;
    while (got < need) {
        int n = recv(s, out + got, need - got, 0);
        if (n <= 0) return false;                  // 0 = closed, <0 = error
        got += n;
    }
    return true;
}

// Receive one framed message into 'buffer'.
bool recvFramed(SOCKET s, std::vector<char>& buffer) {
    uint32_t netLen = 0;
    if (!recvAll(s, (char*)&netLen, 4)) return false;
    uint32_t len = ntohl(netLen);                 // back to host byte order
    buffer.resize(len);
    return recvAll(s, buffer.data(), (int)len);
}
Never assume one send equals one recv

Code that calls recv once and treats the result as a whole message will work on your machine, in your tests, over loopback — and then corrupt data the moment it runs over a real network. Always frame your messages and always loop your reads until you have every byte you expect. In production you'd also cap len so a malicious peer can't ask you to allocate gigabytes.

How this feeds MiniEngine

Everything here is a building block for the engine's networking component. A future NetworkComponent will own a socket much like clientSock, run its recv loop on a worker thread, and hand decoded messages to gameplay code through a queue. The framing helpers above become the wire format for the snapshots and input packets our objects will exchange. TCP's reliability is great for lobby chat, matchmaking, and loading state — but you'll notice its blocking, ordered delivery is a poor fit for fast-moving position updates. That tension is exactly why the next lesson reaches for UDP.

Recap
  • A TCP server follows socket → bind → listen → accept → recv/send; accept returns a brand-new socket per client.
  • A TCP client is socket → connect → send/recv; use inet_pton to parse "127.0.0.1" and htons for the port.
  • Check every call: INVALID_SOCKET and SOCKET_ERROR are the sentinels, and WSAGetLastError() explains the failure.
  • accept and recv block the thread — keep them off your render loop.
  • TCP is a byte stream, so message boundaries are not preserved; frame your messages (e.g. length-prefix) and loop until you've read every byte.
