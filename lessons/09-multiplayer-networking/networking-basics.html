Networking Fundamentals for Games

Up to now, MiniEngine has been a single-player world: one machine, one simulation, one player. In this module we teach it to talk over the internet so two or more players can share the same world in real time. Before we touch a single socket, though, we need a mental model of how networked games actually work — what a connection is, why fast games avoid the "reliable" protocol, and who gets to decide the truth. This lesson is that map.

What you'll learn
  • What a network connection is: IP addresses, ports, and packets (datagrams)
  • Client-server vs peer-to-peer, and why the authoritative server model stops cheating
  • TCP vs UDP — reliability and ordering vs low latency, and why action games choose UDP
  • The metrics that define a connection's quality: latency, bandwidth, packet loss, and jitter
  • The core game networking loop that we will build MiniEngine's multiplayer around

What is a network connection?

When two programs on different machines talk to each other, they do it by sending small chunks of data across the internet. Three ideas make that possible: addresses, ports, and packets.

  • IP address — the "street address" of a machine on a network. IPv4 addresses look like 203.0.113.42; the newer IPv6 looks like 2001:db8::1. Your router also gives your PC a private address such as 192.168.1.20.
  • Port — a numbered "mailbox" on that machine, from 0 to 65535. One computer runs many programs at once, so the port tells the operating system which program a message belongs to. Your game server might listen on port 7777; a web server usually uses 443.
  • Packet (datagram) — data doesn't travel as one long stream; it's chopped into small, individually addressed envelopes. Each packet carries a header (who sent it, where it's going) and a payload (your actual game data). The network delivers them one hop at a time, and they may arrive out of order, late, or not at all.

An address plus a port is called an endpoint, and a message is aimed at one: "send these bytes to 203.0.113.42 port 7777." That single sentence is the foundation of everything else in this module.

A useful analogy

Think of the IP address as an apartment building, the port as a specific mailbox inside it, and a packet as a postcard. Postcards are cheap and fast, but the postal service never promises they'll arrive, or arrive in the order you mailed them. Hold that thought — it explains UDP.

Topologies: who talks to whom?

Once machines can send packets, you have to decide how they're wired together. Two arrangements dominate games.

Peer-to-peer (P2P)

Every player connects directly to every other player. It's cheap — there's no server to rent — and it can feel snappy because messages take the shortest path. But it scales badly (with 8 players each machine maintains 7 connections), it exposes players' IP addresses to each other, and, worst of all, nobody is in charge. If two peers disagree about where a rocket landed, whose version wins?

Client-server

One special machine, the server, sits in the middle. Every client talks only to the server, never directly to other clients. The server relays and coordinates. This is what we'll build. It scales cleanly, hides players from each other, and — crucially — gives us one place to make decisions.

The authoritative server

Here is the single most important idea in the whole module. In an authoritative server model, the server owns the one true copy of the game world. Clients do not tell the server what happened — they only send their inputs ("I'm pressing forward, I fired at time t"). The server runs the real simulation, decides the outcome, and sends the results back. Clients are, in effect, fancy displays plus input devices.

Why go to this trouble? Cheating. If a client could say "I have 999 health" or "my bullet hit," a hacker would simply lie. When the server is the sole authority, a modified client can send fake input, but it cannot invent a game state — the server won't accept "I teleported across the map" because the server, not the client, runs the movement code.

Never trust the client

"Never trust the client" is the golden rule of multiplayer. Anything a client sends can be forged. Every gameplay-critical decision — damage, pickups, collisions, scoring — must be validated or computed on the server. Later lessons on client-side prediction exist precisely because the authoritative model introduces delay we have to hide.

TCP vs UDP: the two ways to send packets

On top of IP, the operating system offers two transport protocols. Choosing between them is the defining decision of game netcode.

TCPUDP
ConnectionConnection-based (handshake first)Connectionless (just send)
ReliabilityGuaranteed delivery, auto-resendNone — packets may be lost
OrderingStrictly in orderMay arrive out of order
OverheadHigher (acks, buffering)Very low
Latency behaviorCan stall (head-of-line blocking)Consistently low
Typical useWeb, chat, downloads, turn-basedAction games, voice, video

TCP is the careful courier: it establishes a connection, numbers every byte, resends anything lost, and delivers everything in perfect order. That sounds ideal — until you hit head-of-line blocking. If packet #5 goes missing, TCP holds packets #6, #7, #8 hostage in a buffer until #5 is re-sent and arrives. In a shooter, that means the whole game freezes for a moment to redeliver a position update you no longer care about, because a newer one already exists. TCP would rather be late than out of order, and games would rather be current than complete.

UDP is the postcard service: fire a datagram at an endpoint and hope. No handshake, no resending, no ordering, no guarantees — but also no stalling. A lost packet is simply gone, and the next fresh update rolls in a fraction of a second later. For a fast-paced game, a slightly newer snapshot is worth far more than a perfectly reconstructed old one.

So fast-paced multiplayer games use UDP and build their own lightweight reliability on top — but only for the messages that truly need it. Position snapshots can be dropped freely (a newer one is coming). But "you picked up the flag" or "you died" must arrive, so the game adds sequence numbers, acknowledgements, and selective resends for just those events. You get UDP's low latency and TCP-like guarantees exactly where you want them — and nowhere you don't. That custom layer is what people mean by a game's "netcode," and we'll build a small version of it later in this module.

Turn-based games can just use TCP

If MiniEngine were powering a card game or a strategy title where a half-second delay is invisible, TCP would be the pragmatic choice — reliability for free, no custom protocol to write. UDP earns its complexity only when every 10 ms matters. Match the tool to the game.

Measuring a connection: the four metrics

You'll see these four numbers everywhere in networking, and they decide how good multiplayer feels.

  • Latency / ping / round-trip time (RTT) — how long a packet takes to travel there and back, in milliseconds. "Ping 40" means 40 ms round trip. Under ~50 ms feels instant; above ~150 ms feels sluggish. It's bounded by the speed of light and the number of hops, so you can hide it but never eliminate it.
  • Bandwidth — how much data the link can carry per second (e.g. 10 Mbps). Games send small messages frequently, so they're rarely bandwidth-starved, but sending full world state to everyone 60 times a second adds up fast.
  • Packet loss — the percentage of packets that vanish in transit. Even a healthy connection loses 1–2%. Our reliability layer exists to survive this.
  • Jitter — the variation in latency. A steady 60 ms ping is smooth; a ping bouncing between 20 and 120 ms is jittery and makes other players teleport and stutter. We smooth jitter with buffering and interpolation.

The core game networking loop

Put the pieces together and you get the heartbeat of every authoritative multiplayer game. It repeats many times per second, forever:

  1. Client samples input and sends it to the server ("forward held, mouse aimed here, fired").
  2. Server receives inputs from all clients and runs one step of the real simulation — movement, physics, damage, scoring.
  3. Server sends the new state back to every client (positions, health, events).
  4. Clients render the state they received, so everyone sees a consistent world.

Here's that loop as pseudocode, mapped onto MiniEngine's existing Update structure. It's illustrative, not the real socket code (that starts next lesson), but it shows exactly where networking slots into the engine you already have.

// Illustrative sketch of the authoritative loop.
// Real UDP socket calls (Winsock2 on Windows, BSD sockets on macOS) arrive next.

struct InputCommand {
    uint32_t sequence;   // lets us detect loss / order
    float    moveX, moveZ;
    bool     firePressed;
};

// ---- Runs on each CLIENT, every frame ----
void Client::Update(float dt) {
    InputCommand cmd = SampleLocalInput();   // read keyboard / mouse
    cmd.sequence = m_nextSeq++;
    m_socket.SendTo(m_serverEndpoint, Serialize(cmd));   // step 1

    // step 4: apply the latest world snapshot the server sent us
    while (WorldSnapshot snap; m_socket.Receive(snap)) {
        ApplySnapshotToScene(snap);          // move GameObjects to authoritative positions
    }
}

// ---- Runs on the SERVER, at a fixed tick rate (e.g. 60 Hz) ----
void Server::Tick(float dt) {
    // step 2: drain inputs from every connected client
    for (ClientConn& c : m_clients) {
        InputCommand cmd;
        while (c.socket.Receive(cmd)) {
            c.pendingInput = cmd;            // keep only the freshest input
        }
    }

    m_world.Simulate(dt);                    // the ONE true simulation

    // step 3: broadcast the resulting state to everyone
    WorldSnapshot snap = m_world.Capture();
    for (ClientConn& c : m_clients) {
        c.socket.SendTo(c.endpoint, Serialize(snap));
    }
}

Notice the asymmetry: clients send intentions, the server sends facts. That single design choice is what makes the game fair and cheat-resistant. It also creates a problem — the round trip means the client sees the result of its own input a little late. Solving that gracefully (prediction, interpolation, lag compensation) is what the rest of this module builds, one layer at a time, on top of MiniEngine's GameObject–Component system.

Recap
  • A connection is IP address + port; data travels as packets (datagrams) that can arrive late, out of order, or not at all.
  • Client-server beats peer-to-peer for scale and control; an authoritative server owns the truth so clients can send input but never forge state — the antidote to cheating.
  • TCP is reliable and ordered but suffers head-of-line blocking; UDP is unreliable and unordered but low-latency, so action games use UDP and add their own selective reliability.
  • Judge a link by latency, bandwidth, packet loss, and jitter — latency is the one you can hide but never remove.
  • The core loop: client sends input → server simulates → server sends state → client renders. Everything else in this module refines that loop.
