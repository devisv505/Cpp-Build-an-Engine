Sockets in C++ with Winsock

Last lesson you learned why games talk over the network — packets, latency, UDP versus TCP. Now we get our hands on the actual plumbing. On Windows, every network connection starts with the Winsock2 library, and by the end of this lesson you will have a small, reusable helper that boots the socket subsystem, opens a UDP socket, and cleans up after itself. This is the foundation of the NetworkSystem we will bolt onto MiniEngine.

What you'll learn
  • Which headers and libraries Winsock needs, and how to link ws2_32.lib.
  • Starting the sockets subsystem with WSAStartup and shutting it down with WSACleanup.
  • Creating a UDP socket and validating it against INVALID_SOCKET.
  • Filling in a sockaddr_in address, including byte order and htons.
  • Wrapping the whole lifecycle in a clean RAII-style helper for the engine.

Headers and linking

Winsock2 lives in two headers. <winsock2.h> gives you the core socket API, and <ws2tcpip.h> adds the newer TCP/IP helpers such as inet_pton. Order matters: always include <winsock2.h> before the old <windows.h>, otherwise Windows pulls in the ancient Winsock 1 header and you get a wall of redefinition errors.

You also have to link against the Winsock library, ws2_32.lib. The quickest way is a #pragma right in your source; alternatively, add ws2_32.lib to your linker's input list in the project settings (or your CMake target_link_libraries).

// Include Winsock2 BEFORE windows.h to avoid the old winsock.h clash.
#include <winsock2.h>   // core socket API: socket, sendto, recvfrom, ...
#include <ws2tcpip.h>   // inet_pton, getaddrinfo, and friends

// Tell the linker to pull in the Winsock library.
// (Or add ws2_32.lib to your linker inputs instead of this pragma.)
#pragma comment(lib, "ws2_32.lib")
The pragma is MSVC-only

The #pragma comment(lib, ...) trick works with the Microsoft compiler. If you ever build with MinGW or on another platform, drop the pragma and link the library through your build system instead.

Starting and stopping the subsystem

Unlike Linux or macOS, Windows requires you to initialize the sockets subsystem before you touch it. That is what WSAStartup does: it loads the Winsock DLL and negotiates a version. We ask for version 2.2 with the MAKEWORD(2, 2) macro, and it fills in a WSADATA struct with details about the implementation.

WSAStartup is unusual among Winsock calls — it returns the error code directly (0 means success) rather than through WSAGetLastError. When you are finished, every successful WSAStartup should be matched by a WSACleanup to release the library.

Call WSAStartup first — always

You must call WSAStartup before any other socket function. If you call socket(), sendto(), or even inet_pton() first, it fails with WSANOTINITIALISED (error 10093). In the engine, do the startup once when NetworkSystem comes online, well before any gameplay code sends a packet.

Creating a socket

A socket is just a handle — an integer-like value the OS gives you to represent one endpoint of a connection. In Winsock its type is SOCKET. You create one with the socket() function, which takes three arguments:

ArgumentUDP valueMeaning
Address familyAF_INETIPv4 addressing (use AF_INET6 for IPv6).
Socket typeSOCK_DGRAMDatagrams — connectionless messages. Use SOCK_STREAM for TCP.
ProtocolIPPROTO_UDPUDP. Pair SOCK_STREAM with IPPROTO_TCP for TCP.

For a fast-paced action game we want UDP — connectionless, low-overhead datagrams where a dropped packet is cheaper than a stalled connection. If socket() fails it returns the sentinel value INVALID_SOCKET, and you fetch the reason with WSAGetLastError. Always check for it.

The address structure

To send to or bind on a specific host and port, you describe the endpoint with a sockaddr_in struct. Three fields matter for IPv4:

  • sin_family — set to AF_INET, matching the socket.
  • sin_port — the port number, in network byte order.
  • sin_addr — the IP address, also in network byte order.

Network protocols agree on big-endian ("network") byte order, but x86 CPUs are little-endian ("host") order. So you cannot just assign a raw port number — you convert it with htons (host-to-network-short) for the 16-bit port. For the address, the modern helper inet_pton ("presentation to network") parses a text IP like "127.0.0.1" straight into sin_addr in the correct order.

Putting it together: a UDP socket helper

Let's wrap the whole lifecycle in a small class. The constructor does WSAStartup plus socket(); the destructor does closesocket() plus WSACleanup. This RAII pattern means the socket is released automatically when the object goes out of scope — the same discipline we used for GPU resources back in the DirectX lessons.

#include <winsock2.h>
#include <ws2tcpip.h>
#include <cstdint>
#include <cstring>   // std::strlen
#include <iostream>
#include <stdexcept>
#include <string>

#pragma comment(lib, "ws2_32.lib")

// A minimal RAII wrapper around a UDP socket.
// The eventual NetworkSystem will own one of these.
class UdpSocket {
public:
    UdpSocket() {
        // 1) Boot the Winsock subsystem (version 2.2).
        WSADATA wsaData;
        int startup = WSAStartup(MAKEWORD(2, 2), &wsaData);
        if (startup != 0) {
            // WSAStartup reports its error code directly, not via WSAGetLastError.
            throw std::runtime_error("WSAStartup failed: " + std::to_string(startup));
        }
        m_wsaReady = true;

        // 2) Create a UDP socket. For TCP you'd use SOCK_STREAM / IPPROTO_TCP.
        m_socket = socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP);
        if (m_socket == INVALID_SOCKET) {
            int err = WSAGetLastError();
            WSACleanup();            // undo the successful startup before bailing
            m_wsaReady = false;
            throw std::runtime_error("socket() failed: " + std::to_string(err));
        }
    }

    ~UdpSocket() {
        // Release resources in the reverse order we acquired them.
        if (m_socket != INVALID_SOCKET) {
            closesocket(m_socket);   // note: closesocket, not close()
        }
        if (m_wsaReady) {
            WSACleanup();
        }
    }

    // Sockets own an OS handle: forbid copies, allow moves.
    UdpSocket(const UdpSocket&) = delete;
    UdpSocket& operator=(const UdpSocket&) = delete;

    // Build a sockaddr_in for a text IP + port, ready for sendto/bind.
    static sockaddr_in MakeAddress(const char* ip, uint16_t port) {
        sockaddr_in addr{};                 // value-init: all fields zeroed
        addr.sin_family = AF_INET;          // IPv4, matching the socket
        addr.sin_port   = htons(port);      // host -> network byte order
        // inet_pton parses "127.0.0.1" straight into sin_addr (network order).
        if (inet_pton(AF_INET, ip, &addr.sin_addr) != 1) {
            throw std::runtime_error("inet_pton: bad address");
        }
        return addr;
    }

    SOCKET handle() const { return m_socket; }

private:
    SOCKET m_socket = INVALID_SOCKET;
    bool   m_wsaReady = false;
};

int main() {
    try {
        UdpSocket sock;                                   // startup + socket()
        sockaddr_in server = UdpSocket::MakeAddress("127.0.0.1", 7777);

        const char* msg = "hello from MiniEngine";
        int sent = sendto(sock.handle(), msg, static_cast<int>(std::strlen(msg)),
                          0, reinterpret_cast<sockaddr*>(&server), sizeof(server));
        if (sent == SOCKET_ERROR) {
            std::cerr << "sendto failed: " << WSAGetLastError() << "\n";
        } else {
            std::cout << "sent " << sent << " bytes\n";
        }
    } catch (const std::exception& e) {
        std::cerr << "network error: " << e.what() << "\n";
        return 1;
    }
    // sock's destructor runs here: closesocket() then WSACleanup().
    return 0;
}
One startup per subsystem

You do not need a WSAStartup per socket. In the real engine, NetworkSystem calls it once during Init() and WSACleanup once during Shutdown(), then hands out sockets as needed. Winsock refcounts startup/cleanup pairs, so keeping them balanced is what matters.

The same code on other platforms

Winsock is Microsoft's take on the BSD sockets API that Linux and macOS also use, so most of this code is portable. The differences are small:

ConceptWindows (Winsock2)Linux / macOS (BSD)
InitializeWSAStartup / WSACleanupNothing — no init needed
Handle typeSOCKETint (a file descriptor)
Failure sentinelINVALID_SOCKET-1
Closeclosesocket(s)close(s)
Last errorWSAGetLastError()errno

The core calls — socket, bind, sendto, recvfrom, htons, inet_pton, sockaddr_in — are identical. A handful of #ifdef _WIN32 guards around the platform-specific bits is all it takes to make the engine's networking layer cross-platform later.

Recap
  • Include <winsock2.h> and <ws2tcpip.h> (before <windows.h>) and link ws2_32.lib.
  • Call WSAStartup(MAKEWORD(2,2), &wsaData) before any other socket call, and match it with WSACleanup.
  • Create a UDP socket with socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP) and check it against INVALID_SOCKET.
  • Fill a sockaddr_in with AF_INET, htons(port), and an address via inet_pton — mind the byte order.
  • Release with closesocket(); wrap the lifecycle in an RAII helper so the engine cleans up automatically.
