UDP Datagrams for Real-Time

In the last lesson we built a TCP client and server: a tidy, reliable, ordered stream of bytes. That is perfect for a chat message or a login handshake, but it is the wrong tool for a fast-moving game world. Here we switch to UDP — small, connectionless, fire-and-forget datagrams — and wire a non-blocking receive loop into MiniEngine so the network never freezes a frame.

What you'll learn
  • How connectionless messaging works with sendto() and recvfrom() — no listen, accept, or connect
  • Why lost, duplicated, and reordered packets are acceptable for constantly-resent game state
  • How UDP sidesteps TCP head-of-line blocking
  • How to make a socket non-blocking with ioctlsocket(sock, FIONBIO, &mode) and drain all pending packets each frame
  • Where you would layer your own sequence numbers and acks for the parts that must be reliable

Streams versus datagrams

TCP gives you a stream: an unbroken river of bytes that arrive in order, with no gaps, exactly once. To get that guarantee, TCP retransmits anything that goes missing and holds newer data back until the missing piece shows up. That holding-back is called head-of-line blocking, and it is a disaster for a shooter or a racing game: if one position update is lost, every update behind it waits in line, so your character freezes and then teleports when the backlog clears.

UDP takes the opposite stance. Each message is a self-contained datagram — you hand the OS a block of bytes and a destination address, and it sends exactly one packet. There is no connection, no ordering, no retransmission, and no acknowledgement. A datagram may arrive, arrive twice, arrive late, or never arrive at all. That sounds terrifying until you realize what game state actually looks like: you are sending the player's position sixty times a second. If packet 118 is lost, packet 119 is already on its way with fresher data. Waiting for the old one would be worse than useless.

Rule of thumb

If the newest message makes the previous one obsolete, use UDP. If every message matters and none can be skipped — a chat line, an inventory purchase, a "match started" event — you want reliability, either from TCP or from a reliability layer you build on top of UDP.

Creating a UDP socket

The Winsock boilerplate is the same as before — WSAStartup at launch, WSACleanup at exit, and you must link ws2_32.lib. The only change at socket-creation time is the type: SOCK_DGRAM with IPPROTO_UDP instead of SOCK_STREAM. Because there is no connection, the sender does not even bind to a port; it just names a destination on every send.

// Link with ws2_32.lib
#include <winsock2.h>
#include <ws2tcpip.h>   // inet_pton
#include <cstdint>
#include <cstdio>

#pragma comment(lib, "ws2_32.lib")

int main()
{
    WSADATA wsa;
    if (WSAStartup(MAKEWORD(2, 2), &wsa) != 0) {
        printf("WSAStartup failed: %d\n", WSAGetLastError());
        return 1;
    }

    // SOCK_DGRAM + IPPROTO_UDP = a UDP socket. No connect() needed.
    SOCKET sock = socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP);
    if (sock == INVALID_SOCKET) {
        printf("socket failed: %d\n", WSAGetLastError());
        WSACleanup();
        return 1;
    }

    // Describe WHERE each datagram goes. We fill this once and reuse it.
    sockaddr_in dest{};
    dest.sin_family = AF_INET;
    dest.sin_port   = htons(9000);              // host -> network byte order
    inet_pton(AF_INET, "127.0.0.1", &dest.sin_addr);

    // A tiny fake "player state" payload (see the next lesson for real serialization).
    struct PlayerState { float x, y, z; };
    PlayerState state{ 1.0f, 0.0f, -3.5f };

    // sendto() is one datagram. The address is passed EVERY call.
    int sent = sendto(sock,
                      reinterpret_cast<const char*>(&state),
                      sizeof(state),
                      0,
                      reinterpret_cast<sockaddr*>(&dest),
                      sizeof(dest));

    if (sent == SOCKET_ERROR)
        printf("sendto failed: %d\n", WSAGetLastError());
    else
        printf("sent %d bytes to 127.0.0.1:9000\n", sent);

    closesocket(sock);
    WSACleanup();
    return 0;
}
No connect, but you may

You can call connect() on a UDP socket. It does not open a connection — it just remembers a default destination so you can use plain send()/recv() and get error replies (like ICMP "port unreachable"). For a game talking to many peers, passing the address on each sendto() is more flexible, so that is what we do here.

The receiver: bind, then recvfrom

The side that listens for datagrams does need a fixed port so senders know where to aim. That is the only reason it calls bind() — there is still no listen or accept. recvfrom() then hands you both the payload and the address it came from, which is exactly what a server needs to reply to the right client.

#include <winsock2.h>
#include <ws2tcpip.h>
#include <cstdio>

#pragma comment(lib, "ws2_32.lib")

int main()
{
    WSADATA wsa;
    WSAStartup(MAKEWORD(2, 2), &wsa);

    SOCKET sock = socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP);

    // Bind to a known port so clients can send to us.
    sockaddr_in local{};
    local.sin_family      = AF_INET;
    local.sin_port        = htons(9000);
    local.sin_addr.s_addr = htonl(INADDR_ANY);  // accept on any local NIC

    if (bind(sock, reinterpret_cast<sockaddr*>(&local), sizeof(local)) == SOCKET_ERROR) {
        printf("bind failed: %d\n", WSAGetLastError());
        closesocket(sock);
        WSACleanup();
        return 1;
    }

    struct PlayerState { float x, y, z; };
    PlayerState state{};

    sockaddr_in from{};
    int fromLen = sizeof(from);

    // This call BLOCKS until a datagram arrives. Fine for a demo, fatal in a game loop.
    int n = recvfrom(sock,
                    reinterpret_cast<char*>(&state),
                    sizeof(state),
                    0,
                    reinterpret_cast<sockaddr*>(&from),
                    &fromLen);

    if (n == SOCKET_ERROR) {
        printf("recvfrom failed: %d\n", WSAGetLastError());
    } else {
        char ip[INET_ADDRSTRLEN];
        inet_ntop(AF_INET, &from.sin_addr, ip, sizeof(ip));
        printf("got %d bytes from %s:%d -> pos(%.2f, %.2f, %.2f)\n",
               n, ip, ntohs(from.sin_port), state.x, state.y, state.z);
    }

    closesocket(sock);
    WSACleanup();
    return 0;
}
One send, one receive

A datagram is atomic: one sendto() becomes exactly one recvfrom(). Unlike TCP, messages never merge or split, so you never have to reassemble a stream. But if your buffer is smaller than the datagram, the extra bytes are discarded and you get WSAEMSGSIZE. Always size the buffer for your largest expected packet, and keep packets well under ~1200 bytes to avoid IP fragmentation.

Non-blocking: don't freeze the frame

The receiver above has a fatal flaw for a real-time engine: recvfrom() blocks. If no packet has arrived, your thread parks inside that call, the game loop stops, and the whole engine stutters. We cannot afford that — MiniEngine's loop has to update transforms, run components, and render a DirectX frame roughly every 16 milliseconds whether or not the network has news.

The fix is to put the socket into non-blocking mode with ioctlsocket and the FIONBIO command. After that, recvfrom() returns immediately: if a datagram is waiting you get it, and if not you get SOCKET_ERROR with WSAGetLastError() equal to WSAEWOULDBLOCK. That error code is not a failure — it simply means "nothing yet, ask again later."

// Call once, right after creating and binding the socket.
u_long mode = 1;                       // 1 = non-blocking, 0 = blocking
if (ioctlsocket(sock, FIONBIO, &mode) == SOCKET_ERROR) {
    printf("ioctlsocket failed: %d\n", WSAGetLastError());
}

Draining the queue each frame

Between two frames, several datagrams may have piled up in the OS receive buffer. If you read only one per frame, you fall behind and add latency. The pattern is a drain loop: keep calling recvfrom() until it reports WSAEWOULDBLOCK, which means the queue is empty for now. Below is that loop wrapped in a component method — the kind of thing you would call from your engine's Update(), right alongside your other components.

#include <winsock2.h>
#include <ws2tcpip.h>
#include <vector>
#include <cstdint>

// A component that owns a non-blocking UDP socket and pumps it once per frame.
class NetworkComponent {
public:
    struct Packet {
        sockaddr_in from{};
        std::vector<uint8_t> data;
    };

    // Called every frame from the engine loop. Never blocks.
    void Receive(std::vector<Packet>& out) {
        uint8_t buffer[1200];          // one datagram's worth of bytes

        // Drain EVERY pending datagram so we never lag behind the network.
        for (;;) {
            sockaddr_in from{};
            int fromLen = sizeof(from);

            int n = recvfrom(m_sock,
                            reinterpret_cast<char*>(buffer),
                            sizeof(buffer),
                            0,
                            reinterpret_cast<sockaddr*>(&from),
                            &fromLen);

            if (n == SOCKET_ERROR) {
                int err = WSAGetLastError();
                if (err == WSAEWOULDBLOCK)
                    break;             // queue is empty for this frame — done
                if (err == WSAEMSGSIZE)
                    continue;          // datagram bigger than buffer: skip it
                break;                 // a real error: stop draining this frame
            }

            if (n == 0)
                continue;              // empty datagram, ignore

            // Hand the payload up for the game to interpret next lesson.
            Packet p;
            p.from = from;
            p.data.assign(buffer, buffer + n);
            out.push_back(std::move(p));
        }
    }

    // Send is unchanged: one datagram, addressed every time.
    void Send(const sockaddr_in& dest, const uint8_t* data, int len) {
        sendto(m_sock,
               reinterpret_cast<const char*>(data),
               len,
               0,
               reinterpret_cast<const sockaddr*>(&dest),
               sizeof(dest));
    }

private:
    SOCKET m_sock = INVALID_SOCKET;    // created, bound, and set non-blocking at init
};

Notice how naturally this drops into the engine you already built. NetworkComponent is just another component hanging off a GameObject; its Receive() runs in the update phase, fills a list of packets, and the rest of your systems consume them — no threads, no blocking, no special cases in the render code.

select() is the alternative

ioctlsocket with FIONBIO is the simplest approach and scales fine for a handful of sockets. If you ever manage many sockets at once, select() lets you ask "which of these have data ready?" in a single call, so you avoid polling each one. Both solve the same problem: never let the network block your frame.

What UDP does not give you

UDP hands you raw datagrams and nothing else — no ordering, no delivery guarantee, no duplicate detection. For pure state you resend every frame, that is exactly what you want. But some messages genuinely must arrive: "you picked up the flag", "the round ended", "you took 40 damage". You cannot just resend those and hope.

The professional answer is to build a thin reliability layer on top of UDP. In practice that means stamping each packet with a sequence number, having the receiver send back acks for the sequence numbers it saw, and resending anything that was not acked within a timeout. You also use the sequence number to drop old, reordered packets and to detect duplicates. This is how libraries like ENet and the netcode in commercial engines work — reliable-when-you-need-it, unreliable-when-you-don't, all over one UDP socket.

Never trust the wire

A datagram can come from anyone and contain anything — a malformed or malicious packet is a classic attack vector. Always validate n (the byte count) before reading fields, never index past what actually arrived, and treat every incoming byte as untrusted input. We will parse packets safely in the next lesson.

We will not build the full reliability layer yet — first we need a clean way to turn a PlayerState into bytes and back. That is exactly what the next lesson, on serializing game state, is about. The sequence numbers and acks slot in right after.

Recap
  • UDP is connectionless: create a SOCK_DGRAM socket and name the destination on every sendto() — no listen, accept, or connect required.
  • Each datagram is independent and may be lost, duplicated, or reordered; that is fine for game state you resend many times a second, and it avoids TCP's head-of-line blocking.
  • The listening side calls bind() on a known port, then recvfrom(), which also reports the sender's address.
  • Put the socket in non-blocking mode with ioctlsocket(sock, FIONBIO, &mode) and drain all pending packets each frame, treating WSAEWOULDBLOCK as "no data yet."
  • For messages that must arrive, layer your own sequence numbers and acks on top of UDP — coming up after we tackle serialization.
