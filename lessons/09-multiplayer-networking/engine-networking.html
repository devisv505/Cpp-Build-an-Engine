Networking MiniEngine

This is it — the finale. You already have a component-based engine running with Direct3D on Windows or Metal on macOS, with a GameObject/Component scene, a scripting-style Behaviour system, a UDP socket, packet serialization, and interpolation. In this last lesson we wire all of it together into a NetworkSystem subsystem so two copies of MiniEngine can see each other move in real time. Then we point you at where to go next and send you off.

What you'll learn

  • Serialize the local player's Transform and send it; receive and apply remote Transforms.

Where networking fits in the engine loop

Everything MiniEngine does happens inside one loop per frame. Networking doesn't replace any of that — it slots in at two precise moments. We send our local state right after we've read input (so we transmit the freshest intent), and we receive and apply remote state right after our own simulation has advanced, just before we draw. That ordering keeps local and remote objects consistent within a frame.

The NetworkSystem subsystem

A subsystem in MiniEngine is a plain object owned by the engine, with an Update that the loop calls. NetworkSystem owns the UDP socket from the portable sockets lessons and the address of the peer we're talking to. It exposes two methods instead of one Update, because Send and Receive happen at different points in the frame.

Recall from the client-server lesson that we send at a fixed network tick (say 20 Hz), not every render frame. So Send accumulates elapsed time and only transmits when the tick elapses. Receive drains the socket every frame — packets arrive whenever they arrive.

#pragma once
#include "SocketPlatform.h"
#include <cstdint>
#include <vector>
#include "Transform.h"      // your engine's Transform (position, rotation, scale)
#include "Scene.h"

// One 32-byte packet: a player id + a position + a yaw. Fixed layout, little-endian.
struct StatePacket {
    uint32_t playerId;   // who this update is about
    uint32_t tick;       // sender's tick number, for ordering/interpolation
    float    x, y, z;    // world position
    float    yaw;        // facing; add pitch/roll if you need them
};

class NetworkSystem {
public:
    // localId identifies us; peer is the address we send our state to.
    bool Start(uint16_t bindPort, const char* peerIp, uint16_t peerPort, uint32_t localId);
    void Stop();

    // Called right after input, before Scene.Update.
    void Send(const Transform& localPlayer, float dt);

    // Called after Scene.Update, before rendering. Applies remote state to the scene.
    void ReceiveAndApply(Scene& scene, float dt);

private:
    SocketRuntime m_runtime;
    SocketHandle  m_sock       = InvalidSocket;
    sockaddr_in m_peerAddr   {};
    uint32_t    m_localId    = 0;
    uint32_t    m_tick       = 0;
    float       m_sendAccum  = 0.0f;
    const float m_sendRate   = 1.0f / 20.0f; // 20 network ticks per second
};
Fixed layout on purpose

We use plain floats and uint32_ts in a flat struct so serialization is trivial. In the serialization lesson you learned this is fragile across machines with different endianness — for a LAN demo it's fine, but production code byte-swaps to a canonical order.

Starting the socket

This is the portable socket setup you already wrote, packaged as Start. We create a non-blocking UDP socket, bind it so we can receive, and remember the peer's address for sendto. Non-blocking matters: our game loop must never stall waiting on a packet.

#include "NetworkSystem.h"
#include <cstring>

bool NetworkSystem::Start(uint16_t bindPort, const char* peerIp,
                          uint16_t peerPort, uint32_t localId) {
    if (!m_runtime.Ready()) return false;

    m_sock = socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP);
    if (m_sock == InvalidSocket) return false;

    // Bind to our local port so recvfrom() works.
    sockaddr_in local{};
    local.sin_family = AF_INET;
    local.sin_addr.s_addr = INADDR_ANY;
    local.sin_port = htons(bindPort);
    if (bind(m_sock, (sockaddr*)&local, sizeof(local)) != 0) {
        return false; // inspect LastSocketError() to see why
    }

    // Non-blocking: recvfrom returns immediately if no data is waiting.
    if (!SetNonBlocking(m_sock)) return false;

    // Remember where to send our updates.
    m_peerAddr.sin_family = AF_INET;
    m_peerAddr.sin_port = htons(peerPort);
    inet_pton(AF_INET, peerIp, &m_peerAddr.sin_addr);

    m_localId = localId;
    return true;
}

void NetworkSystem::Stop() {
    if (m_sock != InvalidSocket) CloseSocket(m_sock);
    m_sock = InvalidSocket;
}

Send: serialize the local Transform

Send waits for the network tick, fills a StatePacket from the local player's Transform, and fires it off with sendto. Because our packet is a flat POD struct, "serialization" here is a memcpy into a byte buffer — exactly the simplest case from the serialization lesson.

void NetworkSystem::Send(const Transform& localPlayer, float dt) {
    m_sendAccum += dt;
    if (m_sendAccum < m_sendRate) return; // not time to send yet
    m_sendAccum -= m_sendRate;

    StatePacket pkt{};
    pkt.playerId = m_localId;
    pkt.tick     = m_tick++;
    pkt.x        = localPlayer.position.x;
    pkt.y        = localPlayer.position.y;
    pkt.z        = localPlayer.position.z;
    pkt.yaw      = localPlayer.rotation.y;   // Euler yaw for the demo

    // Serialize: copy the fixed-layout struct into a byte buffer.
    uint8_t buffer[sizeof(StatePacket)];
    std::memcpy(buffer, &pkt, sizeof(pkt));

    int sent = sendto(m_sock, (const char*)buffer, (int)sizeof(buffer), 0,
                      (sockaddr*)&m_peerAddr, sizeof(m_peerAddr));
    if (sent < 0) {
        // A dropped UDP send is normal-ish; log and move on. Never block the loop.
        // int error = LastSocketError();
    }
}

Receive: apply remote Transforms with interpolation

ReceiveAndApply drains every packet waiting in the socket. For each one, it finds (or needs) a GameObject that represents that remote player and hands the incoming position to that object's interpolation buffer. We don't snap the remote Transform to the packet — that would look jittery. Instead we push the new sample and let the interpolation from the previous lesson smoothly move the object between the last two known states.

void NetworkSystem::ReceiveAndApply(Scene& scene, float dt) {
    uint8_t buffer[sizeof(StatePacket)];
    sockaddr_in from{};
    SocketLength fromLen = sizeof(from);

    // Drain the socket: keep reading until there's nothing left this frame.
    while (true) {
        int n = recvfrom(m_sock, (char*)buffer, (int)sizeof(buffer), 0,
                         (sockaddr*)&from, &fromLen);
        if (n < 0) {
            // WouldBlock means "no more packets right now" on either platform.
            if (WouldBlock(LastSocketError())) break;
            break; // any other error: stop draining this frame
        }
        if (n != (int)sizeof(StatePacket)) continue; // ignore malformed sizes

        StatePacket pkt{};
        std::memcpy(&pkt, buffer, sizeof(pkt));
        if (pkt.playerId == m_localId) continue; // don't apply our own echo

        // Look up the GameObject that mirrors this remote player.
        GameObject* remote = scene.FindNetworked(pkt.playerId);
        if (!remote) remote = scene.SpawnNetworkedProxy(pkt.playerId);

        // Hand the sample to that object's NetworkBehaviour for interpolation.
        if (auto* nb = remote->GetComponent<NetworkBehaviour>()) {
            nb->PushState({ pkt.x, pkt.y, pkt.z }, pkt.yaw, pkt.tick);
        }
    }

    // Advance interpolation on every networked object this frame.
    for (GameObject* go : scene.NetworkedObjects()) {
        if (auto* nb = go->GetComponent<NetworkBehaviour>())
            nb->Interpolate(dt);
    }
}
Why drain in a loop

UDP can deliver several packets between frames. If you called recvfrom once per frame you'd fall behind and lag would grow without bound. The while loop empties the buffer every frame; a “would block” result is the normal signal that you're caught up.

Keeping it component-based: NetworkBehaviour

MiniEngine's whole philosophy — the one you share with Unity and Unreal — is that behavior lives in components. So the networked object shouldn't know about sockets at all. It just carries a NetworkBehaviour, a Behaviour whose job is to buffer remote samples and, each OnUpdate, drive its own GameObject's Transform toward the interpolated target. NetworkSystem feeds it data; the behaviour does the smoothing.

#pragma once
#include "Behaviour.h"
#include "Vec3.h"
#include <cstdint>

// A Behaviour that makes a GameObject follow a networked player's Transform.
class NetworkBehaviour : public Behaviour {
public:
    // NetworkSystem calls this when a fresh packet arrives.
    void PushState(const Vec3& pos, float yaw, uint32_t tick) {
        // Shift "current target" into "previous" and store the new sample.
        m_prevPos = m_targetPos;  m_prevYaw = m_targetYaw;
        m_targetPos = pos;        m_targetYaw = yaw;
        m_blend = 0.0f;           // restart the interpolation window
        m_lastTick = tick;
    }

    // Advance the blend factor toward the newest sample.
    void Interpolate(float dt) {
        // Reach the target over roughly one network interval (~50 ms at 20 Hz).
        m_blend = std::min(1.0f, m_blend + dt / m_interval);
    }

    // Standard Behaviour hook: runs inside Scene.Update via the component system.
    void OnUpdate(float /*dt*/) override {
        Transform& t = GetTransform();
        t.position   = Vec3::Lerp(m_prevPos, m_targetPos, m_blend);
        t.rotation.y = LerpAngle(m_prevYaw, m_targetYaw, m_blend);
    }

private:
    Vec3     m_prevPos{}, m_targetPos{};
    float    m_prevYaw = 0.0f, m_targetYaw = 0.0f;
    float    m_blend = 1.0f;          // 0 = at prev, 1 = at target
    float    m_interval = 0.05f;      // one 20 Hz tick
    uint32_t m_lastTick = 0;
};

Notice how clean this is: the remote player is just another GameObject with a mesh and a NetworkBehaviour. The renderer draws it exactly like anything else. That's the payoff of the component architecture you built earlier — networking became one more component, not a special case.

Wiring it into the engine loop

Here's the whole frame, with the two networking calls in place. Compare it to the diagram at the top — it's the same shape, now in code.

void Engine::RunFrame(float dt) {
    input.Poll();                          // 1. read keyboard/mouse

    // 2. Send our freshest local state (throttled to the network tick).
    net.Send(localPlayer->transform, dt);

    scene.Update(dt);                      // 3. advance all Behaviours + physics

    // 4. Apply remote state, then interpolate networked objects smoothly.
    net.ReceiveAndApply(scene, dt);

    renderer.DrawScene(scene);             // 5. selected graphics backend draws
    renderer.Present();                    // 6. present swap chain or Metal drawable
}
This is a sketch, not a netcode stack

The demo is deliberately minimal: no reliability, no acknowledgements, no client-side prediction of your own movement, no lag compensation. It's enough to see two engines mirror each other on a LAN — and enough to understand the architecture. Real games layer a lot on top, which is exactly what the next section is about.

A quick sanity check

Run two copies of MiniEngine on the same machine or LAN, each bound to a different local port and pointed at the other. Move your local player. You should see the remote proxy glide — not teleport — toward your position, about one tick behind. If it teleports, your interpolation window is too short; if it lags badly, you're probably not draining the socket every frame.

SymptomLikely cause
Remote object jitters / snapsInterpolation window too short, or applying packets directly to Transform
Lag grows over timerecvfrom not looped — draining only one packet per frame
Nothing receivedWrong port, firewall, or forgot to bind
Loop stalls / stuttersSocket left blocking — call the portable SetNonBlocking helper

Where to go from here

You've built the skeleton. Production networking is a deep field, and you do not want to write a reliable-UDP layer, congestion control, and encryption from scratch. Reach for a battle-tested library instead:

  • ENet — small, classic reliable-UDP library. The friendliest next step from what you built here.
  • Valve's GameNetworkingSockets — reliable UDP with encryption and congestion control, extracted from Steam.
  • RakNet / SLikeNet — long-standing full-featured stacks (object replication, RPCs, NAT support).
  • yojimbo — Glenn Fiedler's library focused on secure client-server game connections.
  • Steamworks — if you ship on Steam, its networking, matchmaking, and lobbies do a lot for you.

And the topics still ahead of you, once the basics feel comfortable:

  • NAT punchthrough — letting two players behind home routers connect without a dedicated server.
  • Matchmaking & lobbies — discovering peers and grouping players into sessions.
  • Security & anti-cheat — never trust the client; validate movement and actions server-side.
  • Delta compression — send only what changed since the last acknowledged state to slash bandwidth.
  • Client-side prediction & reconciliation — hide latency on your own player, not just remote ones.

Recap
  • NetworkSystem owns the UDP socket and runs on a fixed network tick, exposing separate Send and ReceiveAndApply steps.
