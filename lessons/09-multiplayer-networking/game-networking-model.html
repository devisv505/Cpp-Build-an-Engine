The Client-Server Game Model

Last lesson you learned to turn game state into bytes you can send across a wire. But sending bytes is the easy part — the hard part is making a game feel instant and smooth when every packet arrives 30, 80, or 150 milliseconds late. This lesson is the "why it feels good" chapter: the architecture that top competitive games use so that your shots land, your movement is buttery, and cheaters can't just edit memory to win. We'll stay mostly conceptual, with light code, so you understand the shape before we wire it into MiniEngine next lesson.

What you'll learn
  • Why the authoritative server model is the industry standard, and how the tick loop works.
  • How snapshot interpolation renders remote players smoothly despite discrete, laggy packets.
  • How client-side prediction makes your own movement feel zero-latency.
  • How server reconciliation corrects mispredictions by replaying unacknowledged inputs.
  • What lag compensation does so that "I clearly hit them" is actually true.

The authoritative server

The naive way to build multiplayer is peer-to-peer: every client tells every other client "here's where I am now." This falls apart fast. Whose version of reality is correct when two players disagree? And nothing stops a cheater from broadcasting "I have 999 health and I'm standing on your head." So virtually every serious online game uses an authoritative server: one machine that owns the one true copy of the game world.

The contract is simple and strict:

  • Clients never change the world directly. They only send their inputs — "I pressed W and clicked at this angle on tick 412."
  • The server simulates everything. It gathers inputs from all clients, steps the physics and game rules forward at a fixed rate, and produces the authoritative state.
  • The server broadcasts snapshots. At a fixed send rate, it ships each client a compact picture of the world (the serialized state from last lesson).

Because the server is the referee, a hacked client can lie about its inputs but not about the outcome — the server decides whether that shot connected, whether that door opened, whether you actually have the ammo. This is exactly the GameObject–Component world you already built in MiniEngine, except only one copy of it — the server's — is real. Clients run a view of it.

Tick rate vs. send rate

The server usually simulates at a high fixed rate (say 60 ticks/second) but sends snapshots less often (say 20/second) to save bandwidth. That gap — big, discrete jumps between snapshots — is the whole reason we need the smoothing tricks below.

The server tick loop

Conceptually the server runs a loop just like your engine's frame loop, but on a fixed timestep and driven by network input rather than a keyboard:

// Runs at a fixed tick rate on the authoritative server.
constexpr double kTickRate    = 60.0;             // simulation steps / second
constexpr double kDt          = 1.0 / kTickRate;  // fixed timestep (seconds)
constexpr int    kSnapshotEvery = 3;              // send every 3rd tick -> 20 Hz

uint32_t tick = 0;
while (serverRunning) {
    // 1. Drain all input packets that arrived since the last tick.
    for (ClientInput& in : ReceivePendingInputs()) {
        ApplyInputToPlayer(world, in.clientId, in);   // queue their intent
        acked[in.clientId] = in.sequence;             // remember last input we saw
    }

    // 2. Step the one true simulation forward by a fixed dt.
    world.Update(kDt);   // physics, movement, hit detection, game rules

    // 3. Periodically broadcast an authoritative snapshot to everyone.
    if (tick % kSnapshotEvery == 0) {
        Snapshot snap = Serialize(world, tick);       // from the previous lesson
        for (Client& c : clients)
            SendSnapshot(c, snap, acked[c.id]);        // include their last-acked input
    }

    ++tick;
    SleepUntilNextTick(kDt);   // keep a steady, wall-clock-accurate cadence
}

Notice the server tells each client which of their inputs it has processed (acked[c.id]). Hold that thought — reconciliation depends on it.

What latency breaks

If a client just displayed whatever the last snapshot said, the experience would be miserable in two specific ways:

  1. Remote players would teleport. Snapshots arrive ~20 times a second, but you render 60–144 times a second. Between packets you'd have nothing to show, so other players would stutter and jump.
  2. Your own player would feel like mud. If you had to press W, wait for the packet to reach the server, wait for it to simulate, and wait for the snapshot to come back before you moved, every step would lag by the full round-trip time. At 100 ms ping that's unplayable.

The three classic techniques below solve these one at a time. The mental trick that unlocks all of them: we render remote entities slightly in the past, and predict our own entity slightly in the future.

Snapshot interpolation (smooth remote players)

You cannot render a remote player at "now," because the freshest thing you know about them is the last snapshot — already slightly stale. So instead of chasing the newest snapshot and stuttering, you deliberately buffer snapshots and render an interpolation delay in the past (commonly ~100 ms). At any render moment you keep at least two snapshots that straddle your render time, and you blend between them.

Say snapshot A is stamped at t = 1.00 s and snapshot B at t = 1.05 s, and your render clock (already offset into the past) reads t = 1.03 s. You're 60% of the way from A to B, so you draw each remote entity 60% of the way between its A position and its B position. As long as your buffer stays fed, remote motion is perfectly smooth even though the data underneath is chunky and discrete.

#include <vector>
#include <algorithm>
#include "Vec3.h"   // your engine's math type from earlier lessons

struct EntityState { uint32_t id; Vec3 position; /* rotation, etc. */ };

struct Snapshot {
    double                     time;     // server timestamp for this snapshot
    std::vector<EntityState>    entities;
};

// A small ring/queue of the most recent snapshots, oldest first.
static std::vector<Snapshot> g_buffer;

// How far in the past we render remote entities (~2 snapshots worth).
constexpr double kInterpDelay = 0.100; // 100 ms

// Look up one entity's position in a snapshot; returns false if absent.
static bool FindEntity(const Snapshot& s, uint32_t id, Vec3& out) {
    for (const EntityState& e : s.entities)
        if (e.id == id) { out = e.position; return true; }
    return false;
}

// Interpolate a remote entity's position for the current render time.
Vec3 InterpolateRemote(uint32_t entityId, double renderNow) {
    // We want to draw the world as it was kInterpDelay seconds ago.
    const double renderTime = renderNow - kInterpDelay;

    // Find the two snapshots that straddle renderTime: from <= renderTime <= to.
    const Snapshot* from = nullptr;
    const Snapshot* to   = nullptr;
    for (size_t i = 0; i + 1 < g_buffer.size(); ++i) {
        if (g_buffer[i].time <= renderTime && renderTime <= g_buffer[i + 1].time) {
            from = &g_buffer[i];
            to   = &g_buffer[i + 1];
            break;
        }
    }

    Vec3 a, b;
    if (!from || !to) {
        // Not enough buffered data (or a hitch): fall back to newest known pose.
        if (!g_buffer.empty() && FindEntity(g_buffer.back(), entityId, b)) return b;
        return Vec3{0, 0, 0};
    }

    // Normalized blend factor in [0, 1] between the two snapshot times.
    const double span = to->time - from->time;
    double t = (span > 0.0) ? (renderTime - from->time) / span : 0.0;
    t = std::clamp(t, 0.0, 1.0);

    if (FindEntity(*from, entityId, a) && FindEntity(*to, entityId, b))
        return a + (b - a) * static_cast<float>(t);   // linear interpolation (lerp)

    return b; // entity just spawned and only exists in the newer snapshot
}
Why render in the past on purpose?

It feels backwards to delay everyone by 100 ms, but that buffer is what guarantees you (almost) always have a snapshot on both sides of your render time. Without it, a single late packet becomes a visible stutter. Delay a little, stay smooth a lot.

Client-side prediction (instant local movement)

Interpolation is great for other players, but you would never accept a 100 ms delay on your own character. So for the local player we do the opposite: predict. The instant you press W, the client applies that input to your local player immediately — using the exact same movement code the server runs — without waiting for confirmation. Your character moves the frame you press the key. It feels like there's no network at all.

Crucially, the client keeps a history of every input it has sent, each tagged with a sequence number, because it will need to re-apply them in a moment:

struct InputCommand {
    uint32_t sequence;   // ever-increasing id for this input
    float    dt;         // how long it covered
    Vec3     move;       // WASD direction, etc.
};

std::vector<InputCommand> g_pendingInputs;   // sent but not yet acked by server
Vec3                      g_localPosition;

void OnLocalInput(const InputCommand& cmd) {
    SendToServer(cmd);                 // (1) tell the authority what we did
    g_pendingInputs.push_back(cmd);    // (2) remember it for reconciliation
    // (3) apply it right now with the SAME sim code the server uses:
    g_localPosition = StepMovement(g_localPosition, cmd);
}

Prediction is correct most of the time, because the client and server run identical movement math. But sometimes they disagree — you got knocked back by something you couldn't see yet, or a packet was lost. When that happens, the authoritative snapshot must win. That's reconciliation.

Server reconciliation (correcting mispredictions)

When a snapshot arrives, it tells you the server's authoritative position for your player and the sequence number of the last input the server actually processed (that acked value from the tick loop). The client does three things:

  1. Snap to authority. Set the local player to the server's position — the truth as of that acked input.
  2. Discard confirmed inputs. Drop every pending input with a sequence ≤ the acked one; the server has already accounted for those.
  3. Replay the rest. Re-apply every input the server hasn't seen yet, on top of the authoritative position, using the same StepMovement. This brings your prediction back up to "now."
// Called when an authoritative snapshot for the local player arrives.
void Reconcile(const Vec3& serverPos, uint32_t serverAckedSeq) {
    // (1) Trust the server: rewind to the authoritative position.
    g_localPosition = serverPos;

    // (2) Forget inputs the server already applied.
    g_pendingInputs.erase(
        std::remove_if(g_pendingInputs.begin(), g_pendingInputs.end(),
            [&](const InputCommand& c){ return c.sequence <= serverAckedSeq; }),
        g_pendingInputs.end());

    // (3) Replay everything the server hasn't acknowledged yet.
    for (const InputCommand& c : g_pendingInputs)
        g_localPosition = StepMovement(g_localPosition, c);
    // Result: correct authoritative base + all our still-in-flight moves.
}

If your prediction was right, replaying produces the same position you already had, so nothing visibly changes — the common case is invisible. If you mispredicted, you'll see a small correction. Well-tuned games smooth that correction out over a few frames instead of snapping, so it reads as a gentle nudge rather than a jitter.

Prediction and server MUST share code

Reconciliation only works if StepMovement is deterministic and identical on client and server. If the two use even slightly different physics or a different fixed timestep, prediction and authority will constantly disagree and the player will rubber-band. Share the exact same simulation code between both builds.

A note on lag compensation

Interpolation and prediction still leave one honesty problem. Remember you're rendering enemies ~100 ms in the past, plus your own ping. So when you line up a shot and click, the enemy on your screen is where they were, not where the server thinks they are now. Without help, you'd have to lead every target and "obvious" hits would miss.

Lag compensation fixes this on the server. Each client's shot is stamped with the client's render time. When the server processes the shot, it rewinds every other player's hitboxes back to where they were at that timestamp — reconstructing the world exactly as the shooter saw it — checks the ray against that rewound state, then returns to the present. You shot at what you saw, and the server agrees. The mild downside is the classic "I ran behind a wall and still got hit," because from the shooter's slightly-past view, you hadn't yet.

Put together, these give the illusion every online game sells: instant controls, smooth opponents, fair hits, and one referee that can't be cheated. Recap
  • The authoritative server owns the one true world: clients send inputs, the server simulates at a fixed tick and broadcasts snapshots at a lower send rate.
  • Snapshot interpolation buffers the last 2+ snapshots and renders remote entities slightly in the past, blending between them for smooth motion.
  • Client-side prediction applies your own input immediately with the same sim code the server uses, so local controls feel instant.
  • Server reconciliation snaps to the authoritative state, drops acked inputs, and replays the unacknowledged ones to fix mispredictions invisibly.
  • Lag compensation rewinds other players' hitboxes to the shooter's view time so hits are fair — at the cost of the occasional "around the corner" death.
