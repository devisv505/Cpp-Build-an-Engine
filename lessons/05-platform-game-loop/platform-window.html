Creating a Window on Windows & macOS

Every game needs a native window and a drawable surface. Writing separate Win32 and Cocoa applications would bury that simple goal under two very different APIs, so our engine uses GLFW as a small cross-platform window layer. The same C++ entry point works in CLion on Windows and macOS, while each renderer can still retrieve the native handle it needs.

What you'll learn
  • How CMake fetches and links GLFW on both course platforms
  • How to initialize GLFW, create a no-API window, and clean it up
  • Why window size and framebuffer size can differ on Retina displays
  • How Direct3D 11 receives an HWND and Metal receives a Cocoa window/layer

Why GLFW?

Windows exposes Win32 handles and a message queue; macOS exposes Cocoa objects and an application event loop. GLFW wraps those differences behind a small C API designed for games and graphics tools. It does not choose our renderer: we ask for GLFW_NO_API, then connect Direct3D 11 on Windows or Metal on macOS ourselves.

Step 1 — add GLFW with CMake

Put the dependency in CMakeLists.txt, not in an IDE-specific settings panel. CMake downloads GLFW the first time it configures the project and reuses it afterward.

cmake_minimum_required(VERSION 3.20)
project(PlatformWindow LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

include(FetchContent)
FetchContent_Declare(
    glfw
    GIT_REPOSITORY https://github.com/glfw/glfw.git
    GIT_TAG 3.4
)
FetchContent_MakeAvailable(glfw)

add_executable(PlatformWindow main.cpp)
target_link_libraries(PlatformWindow PRIVATE glfw)

When CLion notices the edit, accept Reload CMake Project. The target and its dependency now work with bundled MinGW on Windows and Apple Clang on macOS.

Step 2 — create the window

#include <GLFW/glfw3.h>
#include <iostream>

int main()
{
    if (glfwInit() != GLFW_TRUE)
    {
        std::cerr << "GLFW initialization failed\n";
        return 1;
    }

    // We will attach Direct3D 11 or Metal later, so GLFW should not
    // create an OpenGL context for this window.
    glfwWindowHint(GLFW_CLIENT_API, GLFW_NO_API);
    glfwWindowHint(GLFW_RESIZABLE, GLFW_TRUE);

    GLFWwindow* window = glfwCreateWindow(
        1280, 720, "MiniEngine", nullptr, nullptr);

    if (window == nullptr)
    {
        std::cerr << "Window creation failed\n";
        glfwTerminate();
        return 1;
    }

    while (!glfwWindowShouldClose(window))
    {
        glfwPollEvents();
    }

    glfwDestroyWindow(window);
    glfwTerminate();
    return 0;
}

Run the CMake target in CLion. A resizable 1280×720 window should appear on either operating system. Closing it changes GLFW's close flag; the loop ends and releases resources in reverse order.

Initialization and shutdown must balance

Call glfwTerminate once after destroying all GLFW windows. This restores platform resources GLFW changed during initialization.

Logical size vs drawable pixels

On a standard-density Windows display, a 1280×720 window often has a similarly sized framebuffer. On a Retina Mac, one logical point can cover multiple physical pixels. Always ask for the framebuffer size when creating or resizing render targets:

int width = 0;
int height = 0;
glfwGetFramebufferSize(window, &width, &height);

// Use width and height for the Direct3D back buffer or Metal drawable.
A minimized window can report zero pixels

Pause render-target recreation while either framebuffer dimension is zero. Wait for a later resize event instead of asking a graphics API to allocate a zero-sized texture.

Handing the native surface to the renderer

Most engine code keeps GLFWwindow* opaque. Only a platform renderer asks GLFW for the native object:

The Windows renderer passes its native handle to DXGI. The macOS renderer attaches a CAMetalLayer to the Cocoa content view. That one seam keeps native types out of gameplay code.

Wrap ownership in a class

Our engine's Window class will own the pointer and expose platform-neutral operations such as PollEvents, ShouldClose, and GetFramebufferSize. The renderer receives the native handle only during backend initialization.

Recap
  • CMake fetches GLFW and links the same window target on Windows and macOS.
  • GLFW_NO_API leaves graphics ownership to Direct3D 11 or Metal.
  • Create with glfwCreateWindow, poll events, destroy the window, then terminate GLFW.
  • Use framebuffer dimensions—not logical window dimensions—for render targets, especially on Retina displays.
  • Retrieve native handles only inside the platform renderer backend.
