The Game Loop & Delta Time

A game is alive even when no input arrives: every frame it processes events, updates the world, and renders. We'll build that loop once with SDL3 and measure time with the standard C++ clock, giving Windows and macOS the same frame-rate-independent behavior.

What you'll learn
  • The three phases of a frame: input, Update(dt), and render
  • How std::chrono::steady_clock measures portable delta time
  • Why movement must be multiplied by dt
  • How variable and fixed timesteps serve different systems

The portable real-time loop

bool running = true;

while (running)
{
    input.BeginFrame();

    SDL_Event event;
    while (SDL_PollEvent(&event))
    {
        input.ProcessEvent(event);
        if (event.type == SDL_EVENT_QUIT ||
            event.type == SDL_EVENT_WINDOW_CLOSE_REQUESTED)
        {
            running = false;
        }
    }

    if (!running)
        break;

    const float dt = timer.Tick();
    Update(dt);
    Render();
}

Everything an engine does fits around those phases. SDL3 pumps the native Windows or Cocoa event system. The update advances gameplay by elapsed time. The selected renderer backend records Direct3D 11 commands on Windows or Metal commands on macOS.

Measure time with the standard library

std::chrono::steady_clock is monotonic: it never jumps backward if the wall clock changes. That makes it the correct portable clock for frame intervals.

#pragma once
#include <algorithm>
#include <chrono>

class Timer
{
public:
    Timer() : m_previous(Clock::now()) {}

    float Tick()
    {
        const auto now = Clock::now();
        const std::chrono::duration<float> elapsed = now - m_previous;
        m_previous = now;

        // A debugger pause or dragged window should not launch physics
        // several seconds into the future.
        return std::min(elapsed.count(), 0.25f);
    }

private:
    using Clock = std::chrono::steady_clock;
    Clock::time_point m_previous;
};

This one class replaces platform-specific performance-counter code. MinGW and Apple Clang both implement the standard clock with the appropriate high-resolution native source.

Seconds make engine math readable

duration<float> returns seconds, so a speed of 5.0f naturally means five units per second. Avoid mixing milliseconds and seconds between systems.

Frame-rate-independent movement

Moving a fixed amount per frame makes the game run faster on a faster computer:

// Wrong: five units per frame.
playerX += 5.0f;

Express velocity in units per second and multiply by the seconds elapsed:

void Update(float dt)
{
    constexpr float speed = 5.0f;
    const bool* keys = SDL_GetKeyboardState(nullptr);

    if (keys[SDL_SCANCODE_D])
        playerX += speed * dt;

    if (keys[SDL_SCANCODE_A])
        playerX -= speed * dt;
}

This direct SDL query is useful while learning the loop. Module 7 moves the scancode mapping into the engine's Input service so gameplay eventually asks for actions rather than library keys.

Variable vs fixed timestep

Rendering and most gameplay can use the actual variable dt. Physics often needs a fixed step so its results remain stable and reproducible:

constexpr float kFixedStep = 1.0f / 60.0f;
float accumulator = 0.0f;

bool running = true;
while (running)
{
    input.BeginFrame();

    SDL_Event event;
    while (SDL_PollEvent(&event))
    {
        input.ProcessEvent(event);
        if (event.type == SDL_EVENT_QUIT ||
            event.type == SDL_EVENT_WINDOW_CLOSE_REQUESTED)
        {
            running = false;
        }
    }

    if (!running)
        break;

    const float dt = timer.Tick();
    accumulator += dt;

    while (accumulator >= kFixedStep)
    {
        FixedUpdate(kFixedStep);
        accumulator -= kFixedStep;
    }

    Update(dt);
    Render();
}

The accumulator converts uneven real frames into uniform simulation steps. A slow frame may run several fixed updates; a fast frame may run none. Rendering still occurs once per outer loop.

Do not spin wastefully

Later, swap-chain presentation provides frame pacing: Direct3D can wait for vertical sync, and Metal schedules presentation through its drawable. Until rendering exists, this empty loop may consume a full CPU core. That is expected for the temporary window exercise; do not add arbitrary sleeps to the final renderer loop.

Recap
  • Every frame processes events, advances simulation, and renders.
  • std::chrono::steady_clock measures portable monotonic delta time.
  • Multiply velocities by dt so behavior does not depend on frame rate.
  • Clamp exceptional deltas caused by debugger pauses or window interactions.
  • Use variable time for rendering/gameplay and a fixed-step accumulator for physics.
