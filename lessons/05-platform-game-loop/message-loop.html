Events & Input with SDL3

Windows and macOS deliver input and window notifications through different native systems. SDL3 translates both into one event queue, so the engine can process keyboard, mouse, resize, and close events in ordinary cross-platform C++.

What you'll learn
  • How SDL_PollEvent drains queued operating-system events without blocking
  • When to consume an event and when to query current keyboard state
  • How to handle close, key, mouse, and drawable-pixel resize events
  • How to forward SDL events into engine-owned window and input systems

One event pump, two operating systems

SDL_PollEvent returns one queued event at a time. Call it repeatedly until the queue is empty, then update and render the frame:

bool running = true;

while (running)
{
    SDL_Event event;
    while (SDL_PollEvent(&event))
    {
        if (event.type == SDL_EVENT_QUIT ||
            event.type == SDL_EVENT_WINDOW_CLOSE_REQUESTED)
        {
            running = false;
        }
    }

    if (!running)
        break;

    Update();
    Render();
}

Polling is non-blocking: after the current events are drained, the game continues even if the player does nothing. SDL translates Win32 messages on Windows and Cocoa events on macOS before your code sees them.

Drain events on the main thread

SDL video and event functions must stay on the main thread. Worker threads can load assets or prepare jobs; the window event pump remains part of the main loop on both platforms.

Events for transitions

Use events when the transition itself matters: a key went down, a key came up, the mouse moved, or drawable pixels changed. SDL_Event is a tagged union, so first check event.type and then read the matching member.

while (SDL_PollEvent(&event))
{
    switch (event.type)
    {
    case SDL_EVENT_QUIT:
    case SDL_EVENT_WINDOW_CLOSE_REQUESTED:
        running = false;
        break;

    case SDL_EVENT_KEY_DOWN:
        // Ignore operating-system key repeat for one-shot actions.
        if (!event.key.repeat &&
            event.key.scancode == SDL_SCANCODE_ESCAPE)
        {
            running = false;
        }
        break;

    case SDL_EVENT_MOUSE_MOTION:
        camera.AddLookDelta(event.motion.xrel, event.motion.yrel);
        break;

    case SDL_EVENT_WINDOW_PIXEL_SIZE_CHANGED:
        renderer.Resize(event.window.data1, event.window.data2);
        break;

    default:
        break;
    }
}

Text input is opt-in

SDL3 does not emit SDL_EVENT_TEXT_INPUT by default. Call SDL_StartTextInput(window) when a console or text field gains focus, consume its UTF-8 text events, then call SDL_StopTextInput(window) when editing ends. Keyboard controls based on scancodes do not need text-input mode.

Query held keys after polling

For continuous movement, ask SDL for its current keyboard snapshot after processing the event queue. Scancodes describe physical key positions, which is usually what a game control needs:

const bool* keys = SDL_GetKeyboardState(nullptr);

if (keys[SDL_SCANCODE_W])
    MoveForward();

if (keys[SDL_SCANCODE_S])
    MoveBackward();

An event tells you that a state changed; the keyboard snapshot tells you the state now. A robust engine combines both rather than trying to use repeated key-down events for movement.

Forward events into engine objects

SDL3's queue removes the need to register a separate C callback for every operation. The application owns the loop and forwards each event to the systems that care about it:

void Application::PollEvents()
{
    SDL_Event event;
    while (SDL_PollEvent(&event))
    {
        m_input.ProcessEvent(event);

        if (event.type == SDL_EVENT_QUIT ||
            event.type == SDL_EVENT_WINDOW_CLOSE_REQUESTED)
        {
            m_running = false;
        }

        if (event.type == SDL_EVENT_WINDOW_PIXEL_SIZE_CHANGED)
        {
            m_pendingWidth = event.window.data1;
            m_pendingHeight = event.window.data2;
            m_windowResized = true;
        }
    }
}

With more than one window, compare event.window.windowID with SDL_GetWindowID(window) before applying a window event. A single-window game can keep the simpler version above.

Pressed, held, and released

Gameplay usually needs three answers for every engine key: pressed this frame, held, and released this frame. Clear the one-frame transition flags before polling, then let key events update the input service:

input.BeginFrame(); // clears pressed/released flags

SDL_Event event;
while (SDL_PollEvent(&event))
{
    input.ProcessEvent(event);

    if (event.type == SDL_EVENT_QUIT ||
        event.type == SDL_EVENT_WINDOW_CLOSE_REQUESTED)
    {
        running = false;
    }
}

if (!running)
    break;

if (input.IsKeyPressed(Key::Space))
    player.Jump();

if (input.IsKeyDown(Key::W))
    player.MoveForward(dt);

Only the input implementation maps SDL_SCANCODE_W to Key::W. A later action map can translate that engine key into an action such as MoveForward. Gameplay never depends on SDL, Win32, or macOS key constants.

Platform shortcut note

SDL reports modifier state in event.key.mod. Control is SDL_KMOD_CTRL; the Windows key or macOS Command key is SDL_KMOD_GUI. Editor-style shortcuts should use an action map so Windows can prefer Control while macOS prefers Command without changing the tool's behavior.

Recap
  • Drain SDL_PollEvent every frame on the main thread.
  • Events are best for transitions; SDL_GetKeyboardState is best for held controls.
  • Use SDL_EVENT_WINDOW_PIXEL_SIZE_CHANGED dimensions for graphics targets.
  • The application forwards SDL events into engine-owned Window and Input objects.
  • Gameplay reads engine keys and actions, so its logic is the same on Windows and macOS.
