Events & Input with GLFW

A game window must react to keys, mouse movement, resizing, focus changes, and closing. Windows and macOS deliver those events differently, but GLFW translates both native event systems into one set of callbacks and query functions. Our engine can therefore keep input state in ordinary C++.

What you'll learn
  • How glfwPollEvents pumps Win32 or Cocoa events without blocking
  • When to query key state and when to install a callback
  • How to handle keyboard, mouse, framebuffer resize, and close events
  • How a window user pointer routes C callbacks into a C++ object

One event pump, two operating systems

GUI systems are event-driven. The OS queues hardware and window events; the application must regularly process them. GLFW's non-blocking event pump performs the native work for the current system:

while (!glfwWindowShouldClose(window))
{
    glfwPollEvents(); // pumps Windows messages or Cocoa events

    Update();
    Render();
}

glfwPollEvents returns after processing currently queued events, so the game continues updating even when the player does nothing. glfwWaitEvents blocks until something happens and is better suited to paused tools than a real-time game.

Poll on the main thread

Window event processing must stay on the main thread on macOS, and doing the same on Windows keeps the platform layer consistent. Worker threads can load assets or prepare commands, but the window pump belongs in the main loop.

Polling held keys

For continuous actions such as movement, query whether a key is down during each update:

void UpdateInput(GLFWwindow* window)
{
    if (glfwGetKey(window, GLFW_KEY_W) == GLFW_PRESS)
        MoveForward();

    if (glfwGetKey(window, GLFW_KEY_ESCAPE) == GLFW_PRESS)
        glfwSetWindowShouldClose(window, GLFW_TRUE);
}

This code is unchanged between Windows and macOS. GLFW key constants describe physical keyboard keys; later, the engine's input service will translate them into actions such as MoveForward.

Callbacks for discrete events

Callbacks are better for transitions such as one press, one release, text entry, or a resize. GLFW invokes the callback while glfwPollEvents is running.

void OnKey(GLFWwindow* window, int key, int scancode,
           int action, int mods)
{
    if (key == GLFW_KEY_ESCAPE && action == GLFW_PRESS)
        glfwSetWindowShouldClose(window, GLFW_TRUE);
}

void OnMouseMove(GLFWwindow* window, double x, double y)
{
    // Store the latest cursor position for the input system.
}

void OnFramebufferResize(GLFWwindow* window, int width, int height)
{
    // Mark the Direct3D back buffer or Metal drawable for recreation.
    // Do not recreate while width or height is zero.
}

// After creating the window:
glfwSetKeyCallback(window, OnKey);
glfwSetCursorPosCallback(window, OnMouseMove);
glfwSetFramebufferSizeCallback(window, OnFramebufferResize);

Routing callbacks into a C++ class

GLFW callbacks are plain function pointers, so a non-static member function cannot be registered directly. Store this in the window's user pointer and forward through a captureless callback:

class Window
{
public:
    explicit Window(GLFWwindow* handle) : m_handle(handle)
    {
        glfwSetWindowUserPointer(m_handle, this);

        glfwSetFramebufferSizeCallback(
            m_handle,
            [](GLFWwindow* raw, int width, int height)
            {
                auto* self = static_cast<Window*>(
                    glfwGetWindowUserPointer(raw));
                self->OnFramebufferResize(width, height);
            });
    }

private:
    void OnFramebufferResize(int width, int height)
    {
        m_framebufferWidth = width;
        m_framebufferHeight = height;
        m_resized = width > 0 && height > 0;
    }

    GLFWwindow* m_handle = nullptr;
    int m_framebufferWidth = 0;
    int m_framebufferHeight = 0;
    bool m_resized = false;
};

This pattern keeps GLFW at the boundary while the rest of the engine works with a normal class.

macOS command-key note

GLFW exposes modifier flags such as GLFW_MOD_CONTROL and GLFW_MOD_SUPER. On macOS, Command maps to GLFW_MOD_SUPER; on Windows, the Windows key does. Bind editor-style shortcuts through an engine action map so each platform can use its conventional modifier without changing gameplay code.

Input state for an engine

A useful input system records three states per key: pressed this frame, held, and released this frame. At the start of each frame, clear the transition flags; callbacks set transitions while polling; queries read the stable state during updates.

input.BeginFrame();
glfwPollEvents();       // callbacks update the input object

if (input.WasPressed(Key::Jump))
    player.Jump();

input.EndFrame();
Recap
  • glfwPollEvents processes queued native events without blocking the game loop.
  • Poll key state for continuous controls; use callbacks for transitions, text, mouse, and resize events.
  • The framebuffer-size callback is the correct place to mark render targets for recreation.
  • A GLFW user pointer forwards C callbacks into a C++ Window or Input object.
  • The same event and input code runs in CLion on Windows and macOS.
