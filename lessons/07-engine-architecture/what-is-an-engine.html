What a Game Engine Actually Is

You have drawn textured geometry and flown a camera with Direct3D 11 on Windows or Metal on macOS. Now we zoom out: what reusable machine turns those demos into an engine?

What you'll learn
  • A concrete definition of a game engine as a set of reusable runtime systems
  • The difference between reusable engine code and game-specific code, and why that line matters
  • The layered view: platform → engine systems → game
  • What Unity and Unreal hand you for free — and what you gain by building your own

The one-sentence definition

A game engine is a collection of reusable runtime systems that handle the recurring, game-agnostic problems — drawing, input, timing, loading assets, managing a world of objects — so that the game on top can focus on rules and content. That is the whole idea. Everything else in this module is just detail about how those systems are organized and how they talk to each other.

Notice the word reusable. A one-off demo that hard-codes a single spinning cube is not an engine. The moment you can load a different model, bind a different texture, or spawn a different object without rewriting your renderer, you have crossed the line from "a game" into "an engine that runs a game."

Engine vs. game

Rule of thumb: if code would still be useful in a completely different game, it belongs in the engine. If it only makes sense for this game's rules (how the player scores, what the enemies do), it is game code. Keeping that boundary clean is the single most valuable habit in this whole module.

The systems inside an engine

"Engine" sounds like one big thing, but it is really a team of smaller specialists called subsystems. Each one owns a single concern and exposes a small, stable interface to the rest of the program. Here are the core ones almost every engine has — including yours, by the end of this course.

RenderingTurns scene data into pixels via the GPUD3D11 or Metal backend, shaders, buffers

You will not build all of these at once, and that is fine. The point is that each is a replaceable box with a defined job. Rendering does not care how input works; input does not care how audio works. That independence is what lets a large team — or a solo learner — grow an engine without every change rippling through the whole codebase.

The layered view

Engines are organized as layers, where each layer only depends on the ones below it. Think of it as a stack: the raw platform at the bottom, your reusable engine systems in the middle, and the specific game at the top.

  • Platform layer. GLFW, Direct3D 11 or Metal, the filesystem, and the clock. This is where Windows/macOS differences live.
  • Engine systems layer. Your reusable code that wraps the platform in friendlier, game-facing tools: a renderer instead of raw device calls, an input manager instead of scattered key reads. This is what you are building.
  • Game layer. The specific thing you are shipping — its rules, its levels, its characters. It leans entirely on the layer below and never talks to Direct3D directly.

The golden rule is dependencies point downward only. The game may call the engine; the engine must never call the game. When that direction is respected, you can drop the same engine under a racing game or a puzzle game without touching a line of engine code.

What this looks like in code

Here is a sketch of the shape we are heading toward. Do not worry about implementing every method yet — we build these across the next lessons. Focus on how the game (the main function) only ever speaks to engine systems, never to the platform.

#pragma once
#include <memory>

// Each subsystem is its own reusable box with a small interface.
class Renderer;      // wraps the selected graphics backend
class InputManager;  // wraps keyboard / mouse
class Clock;         // wraps the high-resolution timer
class World;         // holds the objects that exist

// The engine owns the systems and wires them together.
class Engine {
public:
    bool Initialize(void* windowHandle); // set up all systems
    void RunFrame();                      // one tick of the loop
    void Shutdown();

private:
    std::unique_ptr<Renderer>     renderer_;
    std::unique_ptr<InputManager> input_;
    std::unique_ptr<Clock>        clock_;
    std::unique_ptr<World>        world_;
};
#include "Engine.h"

// This is GAME code. Notice it never mentions a graphics API directly —
// it only asks the engine to do reusable things.
int main() {
    Engine engine;
    if (!engine.Initialize(/* window handle */ nullptr)) {
        return 1; // engine failed to start up
    }

    // The main loop: keep ticking the engine until the game asks to quit.
    bool running = true;
    while (running) {
        engine.RunFrame(); // update systems + render one frame
        // running = ...   // game rules decide when to stop
    }

    engine.Shutdown();
    return 0;
}

Read main.cpp again and note what is absent: no device, no swap chain, no message pump, no vertex buffers. All of that lives behind Engine. That absence is the whole point — the game author gets a clean, reusable machine and spends their energy on the game itself. We will flesh out RunFrame and the subsystems it drives in the next lesson.

Own your systems' lifetime

We store subsystems in std::unique_ptr so the engine clearly owns them, they get destroyed in a well-defined order, and we never leak. This is exactly the kind of modern-C++ ownership habit that keeps a growing engine from turning into a maze of raw new/delete.

What Unity and Unreal hand you

Commercial engines are simply this same layered idea, matured over decades and polished with tools. When you open Unity or Unreal, you are handed all of the systems above already built, plus an editor, an asset pipeline, a scripting layer, and a physics engine. You drag a model into a scene and it just renders, because their rendering, resource, and world systems are doing everything you saw in earlier lessons — behind a friendly interface.

That convenience is wonderful for shipping, but it hides the machinery. You can spend years in Unity and never learn why a draw call is expensive or how a transform becomes pixels. Building your own small engine is the opposite trade: slower to ship, but every layer becomes something you understand in your bones.

We are not out-building Unreal

The goal is understanding, not competing with a thousand-engineer product. Our engine will be small and focused. That is a feature: a small engine you fully understand teaches you more than a giant one you only rent. The concepts transfer directly the day you do open a commercial engine.

Why the boundary is worth the effort

Keeping engine code and game code cleanly separated is not just tidiness for its own sake. It buys you concrete things:

  1. Reuse. The same renderer runs your next project with zero changes.
  2. Testability. You can test the input system without launching a full game.
  3. Replaceability. The same game can use Direct3D on Windows and Metal on macOS because it depends on the renderer interface.
  4. Sanity. When something breaks, you know which box to open.

Every design decision in the coming lessons — the component-based world, the resource manager, the main loop — exists to protect that boundary. Keep the layered picture in your head and the rest of the module will feel like filling in a diagram you already understand.

Recap
  • A game engine is a set of reusable runtime systems — rendering, input, time, resources, world, later physics and audio — that solve game-agnostic problems.
  • Engine code is anything that would help a different game; game code is specific to one game's rules. Keeping them separate is the core habit.
  • Engines are layered: platform → engine systems → game, with dependencies pointing downward only.
  • Unity and Unreal hand you all these systems pre-built; building your own smaller version trades shipping speed for deep understanding.
