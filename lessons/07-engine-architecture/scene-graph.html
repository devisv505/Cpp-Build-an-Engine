Scenes, Transforms & Hierarchy

In the last lesson we broke objects apart into entities and components. Now we give those objects a place to live: a scene. A scene is a tree of GameObjects, each carrying a Transform, and the magic of that tree is that moving one object automatically moves everything attached to it. This is exactly how a turret rides on a tank, or a hand follows an arm, in every real engine.

What you'll learn
  • What a scene is: a tree (really a forest) of GameObjects.
  • How a Transform stores position, rotation and scale in local space.
  • The difference between a local transform and a world transform.
  • How to compute world = parent × local — and why matrix order matters in DirectXMath.
  • How one recursive traversal updates every object's world matrix each frame.

What is a scene?

A scene is the container for everything you can see or interact with in a level: the player, the props, the lights, the camera. Structurally it is a collection of GameObjects, but not a flat list — the objects form a hierarchy. Each GameObject may have one parent and any number of children, which makes the whole thing a tree. Because a scene can hold several unparented objects at the top level, it is technically a forest (many trees), and we call those top-level objects the roots.

Here is a small hierarchy for a tank. The player is the root; the turret is attached to the player; the barrel is attached to the turret.

The Transform component

Every GameObject owns a Transform: its position, rotation and scale. The key idea is that these values are stored in local space — relative to the object's parent, not to the world. A barrel positioned at (0, 0, 2) is two units in front of its turret, wherever that turret happens to be.

We store the raw values and provide a helper that bakes them into a single 4×4 matrix using DirectXMath, the same math library we will feed to our shaders.

#include <DirectXMath.h>
#include <vector>
#include <memory>
#include <string>

using namespace DirectX;

// A Transform stores position, rotation and scale in LOCAL space --
// that is, relative to its parent. The engine bakes it into a matrix.
struct Transform {
    XMFLOAT3 position { 0.0f, 0.0f, 0.0f };
    XMFLOAT3 rotation { 0.0f, 0.0f, 0.0f }; // Euler angles, in radians
    XMFLOAT3 scale    { 1.0f, 1.0f, 1.0f };

    // Build the local matrix in the order scale, then rotate, then translate.
    XMMATRIX LocalMatrix() const {
        XMMATRIX s = XMMatrixScaling(scale.x, scale.y, scale.z);
        XMMATRIX r = XMMatrixRotationRollPitchYaw(rotation.x, rotation.y, rotation.z);
        XMMATRIX t = XMMatrixTranslation(position.x, position.y, position.z);
        return s * r * t; // row-vector convention: applied left to right
    }
};

// A GameObject owns a Transform and a list of child GameObjects.
struct GameObject {
    std::string name;
    Transform   transform;

    GameObject* parent = nullptr;
    std::vector<std::unique_ptr<GameObject>> children;

    XMFLOAT4X4 world; // cached world matrix, refreshed every frame

    // Create a child, wire up the parent pointer, and hand back a handle.
    GameObject* AddChild(std::string childName) {
        auto child = std::make_unique<GameObject>();
        child->name   = std::move(childName);
        child->parent = this;
        children.push_back(std::move(child));
        return children.back().get();
    }
};

// A Scene is just the set of top-level objects (the roots of the forest).
struct Scene {
    std::vector<std::unique_ptr<GameObject>> roots;
};
Ownership by parent

Each node owns its children through std::unique_ptr, so deleting a GameObject cleanly destroys its whole subtree — no manual bookkeeping. The raw parent pointer is a non-owning back-reference and must never be a smart pointer, or you would create an ownership cycle.

Local space vs world space

There are two matrices to keep straight:

MatrixMeansWho stores it
LocalWhere the object sits relative to its parent.Computed on demand from the Transform.
WorldWhere the object sits in the whole scene.Cached in GameObject::world each frame.

The renderer only ever cares about the world matrix — that is what maps a mesh's vertices into the shared world before the camera looks at them. So the job of the scene graph each frame is to turn every local transform into a world transform by walking down the tree.

Computing the world matrix

An object's world transform is its parent's world transform combined with its own local transform. Conceptually, in the column-vector math you see in textbooks:

Multiplication order flips in DirectXMath

DirectXMath uses row vectors (v' = v · M), so a point is transformed by its local matrix first and its parent matrix second. That means the code multiplies local * parentWorld — the reverse of the parent × local you see in column-vector textbooks. Same result, opposite writing order. Getting this backwards is the classic "my child orbits the wrong way" bug.

Walking the tree each frame

Now the payoff. A single recursive function visits every node, passing each node's freshly computed world matrix down to its children as their parent. Roots start from the identity matrix, meaning "no parent."

// Walk one branch of the tree, turning local transforms into world matrices.
// parentWorld is the already-computed world matrix of this node's parent
// (or the identity matrix for a root object).
void UpdateWorldTransforms(GameObject& node, const XMMATRIX& parentWorld) {
    // 1. The object's transform relative to its parent.
    XMMATRIX local = node.transform.LocalMatrix();

    // 2. Combine with the parent. In DirectXMath's row-vector convention a
    //    point is moved by LOCAL first and by the PARENT second, so we write
    //    local * parentWorld. (In column-vector math this is parentWorld * local.)
    XMMATRIX world = XMMatrixMultiply(local, parentWorld);

    // 3. Store it so the renderer and any children can read it back.
    XMStoreFloat4x4(&node.world, world);

    // 4. Every child inherits THIS node's world matrix as its parent.
    for (auto& child : node.children) {
        UpdateWorldTransforms(*child, world);
    }
}

// Once per frame, refresh the whole scene starting from an identity matrix.
void UpdateScene(Scene& scene) {
    for (auto& root : scene.roots) {
        UpdateWorldTransforms(*root, XMMatrixIdentity());
    }
}
Parents before children — always

A child's world matrix depends on its parent's, so the parent must be updated first. Our top-down recursion guarantees that automatically. If you ever flatten the tree into an array for speed, sort it so every parent appears before its children, or the children will read a stale matrix for one frame.

Why hierarchy is powerful

Because every child derives its position from its parent, you get grouped movement for free. Watch what happens when we move only the player: the turret and barrel come along without a single extra line.

Scene scene;

// Add the top-level object (a root of the scene).
scene.roots.push_back(std::make_unique<GameObject>());
GameObject* player = scene.roots.back().get();
player->name = "Player";

// Build a small hierarchy: Player -> Turret -> Barrel.
GameObject* turret = player->AddChild("Turret");
GameObject* barrel = turret->AddChild("Barrel");

// The barrel sits 2 units in front of the turret, in the turret's LOCAL space.
barrel->transform.position = { 0.0f, 0.0f, 2.0f };

// Move ONLY the player. The turret and barrel are carried along for free.
player->transform.position.x += 5.0f;

UpdateScene(scene); // barrel's world matrix now reflects the 5-unit shift too

Rotating the turret works the same way: because the barrel's position is expressed in the turret's local space, spinning the turret swings the barrel around it. This is the entire reason engines use a scene graph instead of a flat list.

Handing the world matrix to the GPU

Once node.world is up to date, drawing a mesh means uploading that matrix into a constant buffer and letting the vertex shader move each vertex from local space into world space. Here is the HLSL side — note it multiplies vector then matrix, matching the row-vector convention we used in C++.

cbuffer PerObject : register(b0)
{
    matrix World;          // node.world, uploaded before each draw call
    matrix ViewProjection; // supplied by the camera
};

float4 main(float3 localPos : POSITION) : SV_POSITION
{
    // local space -> world space -> clip space
    float4 worldPos = mul(float4(localPos, 1.0f), World);
    return mul(worldPos, ViewProjection);
}

That is the whole loop of a scene graph: store transforms locally, walk the tree to compute world matrices, and feed each world matrix to the GPU. In later modules we will hang meshes, materials and cameras off these same GameObjects, and add a "dirty flag" so we only recompute branches that actually moved.

Recap
  • A scene is a tree (a forest) of GameObjects; the unparented ones are the roots.
  • A Transform stores position, rotation and scale in local space — relative to the parent.
  • The world matrix is what the renderer needs; it is the parent's world combined with the node's local matrix.
  • In DirectXMath's row-vector convention that combination is local * parentWorld, computed with XMMatrixMultiply.
  • A single top-down recursion updates every node each frame, so moving a parent moves all of its children automatically.
