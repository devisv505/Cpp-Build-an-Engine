Entity-Component-System

In the last lesson we met Unreal's Actor model, where each object owns its logic through virtual methods. Now we flip the whole idea inside-out. Entity-Component-System (ECS) is the architecture behind engines like Unity DOTS, Bevy, and countless high-performance simulations. It splits your game world into three plain, separate pieces — ids, data, and logic — and in doing so unlocks the kind of cache-friendly, massively-parallel performance that a classic object graph struggles to reach.

The three letters, one idea at a time

The whole pattern is in its name. Let's take each part in turn and keep it concrete.

Entities are just ids

An entity is not an object. It has no fields, no methods, no behaviour. It is simply a unique number — an id — that acts as a label. Entity 42 might be the player, entity 7 a bullet, entity 9001 a particle. On its own the id means nothing; its meaning comes entirely from which components are attached to it.

Components are plain data

A component is a small bundle of data with no logic inside it. A Position is just { x, y, z }. A Velocity is just { dx, dy, dz }. A Health is just an int. Components are typically plain structs — no virtual methods, no inheritance, no behaviour. They are pure state.

Systems are the logic

A system is a function that runs over every entity that has a particular set of components. A MovementSystem asks the world: "give me every entity that has both a Position and a Velocity," then loops over exactly those and updates the positions. The system is where all the behaviour lives, and it operates on data in bulk.

The mental model

Think of ECS as a spreadsheet. Each row is an entity. Each column is a component type. A system is a query that selects the rows where certain columns are filled in, then processes just those rows.

Why do it this way? Data-oriented design

ECS is the flagship example of data-oriented design: organise your program around how data is laid out in memory and how the CPU actually reads it, rather than around tidy real-world "objects." To see why that matters, we need to talk about the cache.

Your CPU is astonishingly fast, but main memory (RAM) is slow by comparison. To hide that gap, the CPU keeps small, very fast caches and pulls memory in chunks called cache lines (typically 64 bytes). When you read one byte, the hardware grabs the whole surrounding line. If the next thing you need is already in that line, it is nearly free. If it is somewhere else in RAM, you pay a cache miss — potentially hundreds of wasted cycles.

Now imagine 100,000 moving objects. In a classic engine each object is heap-allocated and scattered across memory, and each holds a lot of unrelated data (mesh pointers, names, flags). To update positions you chase pointers all over RAM, dragging in 64 bytes each time to use maybe 12. In ECS, all the Position components live packed together in one contiguous array, and all the Velocity components in another. The MovementSystem marches straight down those arrays. Almost every read is a cache hit.

Because systems process homogeneous arrays with no branching per object, this layout is also friendly to the compiler's auto-vectorization (SIMD) and trivially parallelizable across threads. That is the real source of ECS's reputation for speed.

A concrete example

Let's build the smallest ECS that shows the shape of the idea: two data-only components and one system. This is a teaching model — a real ECS library adds sparse sets or archetypes to make the queries fast — but every core idea is here.

#include <cstdint>
#include <vector>
#include <iostream>
#include <optional>

// An entity is nothing but an id.
using Entity = std::uint32_t;

// Components are plain data. No methods, no logic.
struct Position { float x = 0.0f, y = 0.0f, z = 0.0f; };
struct Velocity { float x = 0.0f, y = 0.0f, z = 0.0f; };

// A tiny world: parallel arrays indexed by entity id.
// std::optional lets an entity "not have" a component.
struct World {
    std::vector<std::optional<Position>> positions;
    std::vector<std::optional<Velocity>> velocities;

    Entity create() {
        Entity e = static_cast<Entity>(positions.size());
        positions.emplace_back();   // starts empty (nullopt)
        velocities.emplace_back();
        return e;
    }
};

// The MovementSystem: pure logic. It runs over every entity that
// has BOTH a Position and a Velocity, and integrates motion.
void MovementSystem(World& world, float dt) {
    for (std::size_t e = 0; e < world.positions.size(); ++e) {
        auto& pos = world.positions[e];
        auto& vel = world.velocities[e];
        if (pos && vel) {            // does this entity match the query?
            pos->x += vel->x * dt;  // p = p + v * dt
            pos->y += vel->y * dt;
            pos->z += vel->z * dt;
        }
    }
}

int main() {
    World world;

    Entity ball = world.create();
    world.positions[ball] = Position{0.0f, 0.0f, 0.0f};
    world.velocities[ball] = Velocity{1.0f, 2.0f, 0.0f};

    Entity wall = world.create();
    world.positions[wall] = Position{5.0f, 0.0f, 0.0f};
    // The wall has NO Velocity, so MovementSystem simply skips it.

    for (int frame = 0; frame < 3; ++frame) {
        MovementSystem(world, 0.5f); // dt = half a second
    }

    const Position& p = *world.positions[ball];
    std::cout << "ball is at (" << p.x << ", " << p.y << ", " << p.z << ")\n";
    // After 3 steps of dt=0.5: (1.5, 3.0, 0.0)
    return 0;
}

Notice how the MovementSystem never knew about "balls" or "walls." It only knew about the components. Give any entity a Position and a Velocity and it will move — a projectile, a leaf, a camera. Behaviour is composed by adding data, not by writing new classes. That is the heart of ECS.

Tag components

Because behaviour is driven by which components an entity has, empty structs are useful too. A struct Player{}; with no fields is a tag — a system can query for it to find "all players." You compose gameplay by mixing data components and tag components.

ECS vs. the GameObject-Component model

The classic model most beginners meet — Unity's MonoBehaviour, or the Actor model from the previous lesson — is often called OOP GameObject-Component. There, a GameObject owns a list of components, and each component carries both its data and its own Update() logic. It is object-oriented: state and behaviour live together.

ECS keeps the word "component" but changes its meaning completely: components hold data only, and all logic is pulled out into systems. Here is how the two compare.

When ECS wins

ECS shines when you have many, many similar entities and performance matters: bullet-hell shooters, RTS games with thousands of units, particle and physics simulations, large open worlds. The bigger and more uniform the crowd, the more the cache-friendly layout pays off.

When the OOP model wins

The GameObject model wins on clarity. For a few hundred richly-varied objects, each with unique behaviour, an Actor::Update() you can read top-to-bottom is far easier to reason about than logic scattered across a dozen systems and a query engine. Most gameplay code is not performance-bound; it is human-bound.

Don't cargo-cult ECS

A common beginner trap is reaching for a full ECS on day one because it is "what fast engines use." Real ECS libraries are complex, and a naive hand-rolled one (like our std::optional arrays above) is often slower than plain objects. Adopt ECS when a profiler tells you a hot loop over many entities is the bottleneck — not before.

What our engine will do

Our engine will use the classic GameObject-Component model, not ECS. That is a deliberate teaching choice. The GameObject model maps cleanly to how a beginner pictures a game world — "a thing that has a transform, a mesh, and some behaviour" — and it keeps the code readable while you are still learning C++, DirectX, and engine architecture all at once.

You are not giving up performance you need. A learning engine rendering a scene of dozens or hundreds of objects will be limited by the GPU and by DirectX draw calls long before component memory layout matters. And nothing here is wasted: in the next lesson we build the Transform and scene hierarchy on top of that GameObject, and the ideas you just learned — separating data from logic, thinking about memory layout — will make you a sharper engine programmer no matter which model you use.

Both, eventually

Mature engines are rarely purely one or the other. Unity ships its classic GameObject model and the ECS-based DOTS stack, and teams mix them. Learning the GameObject model first gives you the vocabulary to appreciate ECS when you meet it in the wild.

Recap
  • Entities are just ids; components are plain data with no logic; systems are functions that run over every entity holding a given set of components.
  • ECS is data-oriented design: packing components into contiguous arrays keeps the CPU cache full, which is why it scales to tens of thousands of entities and parallelizes naturally.
  • The MovementSystem example showed logic driven by which components an entity has — the wall stayed put simply because it had no Velocity.
  • The OOP GameObject-Component model bundles data and behaviour together: clearer and easier to debug, but slower at massive scale.
  • Our engine uses the GameObject model for clarity — the right call for a learning project where the GPU, not memory layout, is the real bottleneck.
