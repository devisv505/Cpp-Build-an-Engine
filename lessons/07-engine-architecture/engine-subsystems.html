Subsystems & the Main Loop

In the last lesson we defined what an engine actually is: a bundle of reusable systems that turn code and assets into an interactive experience. Now we give that idea a shape. An engine is a small set of subsystems — Renderer, Input, Time, ResourceManager, Scene — owned by a single Engine object, and driven by one central main loop that ticks every subsystem in order, frame after frame. Get this skeleton right and everything else we build for the rest of the course slots neatly into it.

What you'll learn
  • The five core subsystems every engine needs and what each one owns
  • Why a single Engine object should own and outlive all of them
  • The lifecycle: Initialize → loop(ProcessInput → Update → Render) → Shutdown
  • How the main loop drives each subsystem in a fixed order every frame
  • A C++ skeleton for the Engine class you'll flesh out over the coming modules

The core subsystems

A subsystem is a self-contained slice of the engine with one clear job. Keeping each concern separate means you can reason about, test, and replace one piece without disturbing the rest. We start with five that nearly every engine has — Unity and Unreal have the same responsibilities, just under different names.

SubsystemResponsibilityCalled each frame in
TimeMeasures how long the last frame took (delta time) and tracks total elapsed time.start of the loop
InputReads keyboard and mouse state so gameplay can ask "is W held?"ProcessInput
SceneHolds the objects in the world and advances their logic.Update
RendererWraps Direct3D 11 on Windows or Metal on macOS.Render
ResourceManagerLoads and caches meshes, textures and shaders so they load once, not per-frame.on demand (not every frame)

Notice that not every subsystem runs every frame. The ResourceManager mostly works during loading; the Renderer and Scene are the workhorses of the per-frame loop. That distinction is important — it tells us where to spend our performance budget later.

Delta time, in one sentence

Delta time is the number of seconds the previous frame took. Multiplying movement by it (position += speed * dt) keeps your game running at the same real-world speed whether the machine draws 30 or 300 frames per second.

One owner: the Engine object

If subsystems were scattered global variables, their creation order and lifetime would be a constant source of bugs. Instead we give them a single owner: an Engine (sometimes called Application) object created in main(). The Engine constructs each subsystem, hands them to each other where needed, runs the loop, and tears everything down in the right order when the program ends.

We express ownership with std::unique_ptr — the smart pointer you met earlier. Each subsystem is heap-allocated and owned by exactly one Engine; when the Engine is destroyed, its members are destroyed automatically in reverse order of declaration. No manual delete, no leaks.

The lifecycle

Every engine follows the same three-phase lifecycle. It is worth memorising, because it never changes no matter how big the engine grows:

  1. Initialize — create the GLFW window, initialize the selected graphics backend, load resources, and build the scene. Runs once.
  2. The loop — repeat ProcessInput → Update → Render until the user quits. Runs thousands of times per second.
  3. Shutdown — release GPU resources, close the window, free memory. Runs once.
Order matters — both ways

The Renderer needs a GLFW window before it creates a Direct3D device or Metal layer. Shut down in reverse so the GPU releases its surface before GLFW destroys the window.

Why input, update, render — in that order?

The order inside the loop is not arbitrary. Each stage feeds the next:

  • ProcessInput snapshots the keyboard and mouse so the whole frame sees one consistent input state.
  • Update advances game logic using that input and this frame's delta time — moving objects, running physics, ticking timers.
  • Render draws the world after it has been updated, so the picture on screen reflects the newest state.

Reading input after updating, or rendering before updating, would make objects lag one frame behind reality. Input → Update → Render keeps everything in sync.

The Engine class skeleton

Here is the shape of the Engine class we'll grow across the rest of the course. This is a sketch — the subsystem types are forward-declared and there's no real implementation yet — but the structure is exactly what we'll build on. Read it top to bottom and match it against the lifecycle above.

#pragma once
#include <memory>   // std::unique_ptr

// Forward declarations: we only need pointers here, not full definitions,
// so the real headers stay out of this file. Faster builds, fewer includes.
class Time;
class Input;
class Scene;
class Renderer;
class ResourceManager;

class Engine {
public:
    Engine();
    ~Engine();               // out-of-line so unique_ptr can see full types

    // Boots every subsystem. Returns false if any step fails.
    bool Initialize();

    // Runs the main loop until the user quits, then tears everything down.
    void Run();

private:
    // --- the three loop stages, called once per frame ---
    void ProcessInput();     // snapshot keyboard / mouse
    void Update(float dt);   // advance game logic by dt seconds
    void Render();           // draw through the selected backend

    void Shutdown();         // release subsystems in reverse order

    // The Engine OWNS every subsystem. Declaration order below is also
    // construction order; destruction happens in reverse automatically.
    std::unique_ptr<Time>            m_time;
    std::unique_ptr<Input>           m_input;
    std::unique_ptr<ResourceManager> m_resources;
    std::unique_ptr<Renderer>        m_renderer;
    std::unique_ptr<Scene>           m_scene;

    bool m_running = false;  // set true in Initialize, false to exit the loop
};

And the heart of it all — the loop itself. Notice how Run() reads like the lifecycle diagram written in code:

void Engine::Run() {
    m_running = true;

    // Keep looping until something asks us to stop.
    while (m_running) {
        // 1) Time first: how long did the previous frame take?
        float dt = m_time->Tick();   // e.g. 0.016f at ~60 FPS

        // 2) Read this frame's input as a single consistent snapshot.
        ProcessInput();

        // 3) Advance all game logic by dt seconds.
        Update(dt);

        // 4) Draw the freshly-updated world to the screen.
        Render();
    }

    // Loop exited: clean everything up in the right order.
    Shutdown();
}

void Engine::Update(float dt) {
    // The Scene ticks every object it owns, passing delta time down.
    m_scene->Update(dt);
}

void Engine::Render() {
    m_renderer->BeginFrame();     // clear the back buffer
    m_renderer->DrawScene(*m_scene);
    m_renderer->EndFrame();       // present the swap chain
}

Your main.cpp then becomes refreshingly small — it just creates the Engine, initializes it, and hands control to the loop:

#include "Engine.h"

int main() {
    Engine engine;                 // subsystems created here

    if (!engine.Initialize())      // window + graphics + resources
        return -1;                 // bail out if boot failed

    engine.Run();                  // loop until quit, then Shutdown()

    return 0;                      // Engine destructor frees everything
}
Why declare the destructor even though it does "nothing"?

When members are std::unique_ptr to forward-declared types, the compiler needs the full type to generate their destruction code. Declaring ~Engine() in the header and defining it in the .cpp (where the real headers are included) puts that generated code somewhere the types are complete. Leaving it out gives a confusing "incomplete type" error — a very common beginner trap.

Where this is heading

Right now the subsystems are empty shells, but the architecture is real. We will put D3D11 or Metal behind Renderer, a portable clock inside Time, and GLFW keyboard/mouse state inside Input. Next we introduce the GameObject–Component model.

Recap
  • An engine is a set of focused subsystems — Time, Input, Scene, Renderer, ResourceManager — each with one job.
  • A single Engine object owns every subsystem via std::unique_ptr, guaranteeing correct construction and destruction order.
  • The lifecycle is Initialize (once) → loop of ProcessInput → Update → Render (every frame) → Shutdown (once).
  • The loop order keeps input, logic, and the drawn image in sync; delta time from Time makes movement frame-rate independent.
  • The Engine skeleton and tiny main() we sketched are the scaffold we'll fill in for the rest of the course.
