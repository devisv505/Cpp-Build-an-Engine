The Actor Model (Unreal)

Last lesson we studied Unity's GameObject–Component split. Now we look at how Unreal Engine solves the same problem with a different vocabulary: Actors live in a level, and Components attach to them to supply behaviour, geometry, and a transform. Seeing two successful engines side by side reveals what is essential and what is just naming — and that insight will guide the architecture of our own C++ engine.

What you'll learn
  • What an Actor is, and how it differs from a bare GameObject.
  • The difference between an ActorComponent and a SceneComponent.
  • Why every visible Actor has a single root component that anchors its transform.
  • What UObject, reflection, and garbage collection give Unreal at a high level.
  • A direct equivalence table mapping Unreal concepts onto Unity's.

Actors: the things you place in a level

In Unreal, a level (a scene, in Unity terms) is a collection of Actors. An Actor is anything that can be placed into or spawned inside the world: a character, a light, a camera, a trigger volume, a sound emitter, or an invisible game-rules manager. The C++ base class is AActor — Unreal prefixes Actor classes with A by convention.

Here is the key mental shift from Unity. A Unity GameObject is a nearly empty container that always has a Transform. An Unreal Actor is a slightly richer object: it is itself a class you frequently subclass, and it does not carry a transform directly. Instead, an Actor is a container for components, and its transform comes from one special component it designates as the root. Actors that never appear in space (a scoreboard manager, say) simply have no root component and no position — that is perfectly legal.

Naming prefixes

Unreal's C++ prefixes encode the base type: A for Actors (AActor), U for other UObjects including components (UActorComponent), F for plain structs (FVector), and T for templates (TArray). They are conventions, not language features, but the engine's tooling relies on them.

Two families of component

Where Unity has one Component base class, Unreal splits components into two layers, and understanding the split is the heart of this lesson.

ActorComponent — behaviour without a place

UActorComponent is the base class for any component. It can tick every frame, respond to events, and hold state, but it has no position, rotation, or scale. It answers the question "what can this Actor do?" A health component, an inventory, an AI decision module, or a stats tracker are all good candidates: none of them need to exist at a point in space.

SceneComponent — a component with a transform

USceneComponent derives from UActorComponent and adds a transform: a location, rotation, and scale relative to its parent. Crucially, SceneComponents can be attached to one another, forming a tree inside a single Actor. A car Actor might have a body mesh as its root, four wheel meshes attached beneath it, and a camera attached above — move the root and the whole assembly moves together.

Renderable pieces go one level deeper still. UPrimitiveComponent derives from USceneComponent, and UStaticMeshComponent derives from UPrimitiveComponent — because to draw geometry you first need a place to draw it. This is exactly the layering we will build toward: a transform is a prerequisite for rendering, never the other way around.

The root component

If an Actor can hold a whole tree of SceneComponents, which one defines the Actor's own location? The answer is the root component (AActor::GetRootComponent()). Exactly one SceneComponent is designated as the root; every other SceneComponent attaches beneath it, directly or transitively. When you call SetActorLocation(), you are really moving the root, and the whole attached subtree follows because child transforms are relative to their parent.

This is the same idea as Unity's Transform hierarchy

Unity nests GameObjects under one another; Unreal nests SceneComponents inside a single Actor and can also attach whole Actors together. Both give you parent-relative transforms and a tree you move as a unit. The engine we build will use the same relative-transform math for our own Transform component.

Here is a compact, realistic AActor subclass that builds a small component tree in its constructor. This is idiomatic Unreal C++. 

// A collectible that shows a mesh and detects overlaps.
UCLASS()
class APickupActor : public AActor
{
    GENERATED_BODY()

public:
    APickupActor();

private:
    // The transform anchor for the whole Actor.
    UPROPERTY(VisibleAnywhere)
    USceneComponent* Root;

    // Visible geometry, attached under the root.
    UPROPERTY(VisibleAnywhere)
    UStaticMeshComponent* Mesh;

    // A trigger sphere, also under the root.
    UPROPERTY(VisibleAnywhere)
    USphereComponent* Trigger;
};
#include "PickupActor.h"

APickupActor::APickupActor()
{
    // Ask the engine to tick this Actor each frame.
    PrimaryActorTick.bCanEverTick = true;

    // 1) Create a bare SceneComponent and make it the root.
    Root = CreateDefaultSubobject<USceneComponent>(TEXT("Root"));
    SetRootComponent(Root);

    // 2) Create the mesh and attach it under the root.
    Mesh = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("Mesh"));
    Mesh->SetupAttachment(Root);

    // 3) Create the trigger sphere, also under the root.
    Trigger = CreateDefaultSubobject<USphereComponent>(TEXT("Trigger"));
    Trigger->SetupAttachment(Root);
    Trigger->SetSphereRadius(64.0f);
}

Notice how SetupAttachment(Root) wires each SceneComponent into the tree, and SetRootComponent(Root) chooses which node the Actor's world position tracks. Move this Actor and the mesh and trigger travel with it — no manual bookkeeping required.

Constructor-only helpers

CreateDefaultSubobject and SetupAttachment are meant for the constructor (edit-time defaults). To add or attach components at runtime you use NewObject plus RegisterComponent and AttachToComponent instead. Mixing them up is a classic beginner crash.

UObject, reflection, and garbage collection

Both AActor and UActorComponent ultimately derive from UObject, Unreal's universal base class. UObject is what gives Unreal capabilities that plain C++ does not have out of the box:

  • Reflection. Macros like UCLASS(), UPROPERTY(), and UFUNCTION() feed a build-time tool (the Unreal Header Tool) that generates metadata. That metadata powers the editor's property panels, Blueprint access, and save/load serialization — all things standard C++ has no notion of.
  • Garbage collection. Any UPROPERTY pointer to a UObject is tracked by Unreal's garbage collector. When nothing references an object anymore, the engine reclaims it automatically. This is why Unreal code uses raw pointers like UStaticMeshComponent* rather than the std::unique_ptr you would reach for in plain C++.
Forget UPROPERTY, lose your object

A pointer not marked UPROPERTY is invisible to the garbage collector. The object it points to can be collected out from under you, leaving a dangling pointer and a hard crash. This is the single most common source of "why did my component disappear?" bugs in Unreal.

We will not build a reflection system or a garbage collector for our own engine — those are enormous undertakings. But it is worth knowing they exist, because they explain why Unreal code looks the way it does, and they show the price a AAA engine pays for editor integration and safety.

Mapping Unreal onto Unity

Now the payoff. Despite the different names, the two engines line up almost point for point. Keep this table handy — it is the Rosetta Stone between the two ecosystems.

ConceptUnityUnreal
Scene / world containerSceneLevel (World)
Placeable objectGameObjectActor (AActor)
Behaviour with no transformMonoBehaviour / ComponentUActorComponent
Component with a transformTransform (built in)USceneComponent
The transform anchorAlways-present TransformRoot component
Nesting / hierarchyParent GameObjectsAttached SceneComponents / Actors

Memory managementC# garbage collectorUObject garbage collector

The one real structural difference: in Unity a GameObject always has exactly one Transform, so transform and identity are welded together. In Unreal the transform is a component you can add, omit, or nest several of. Unreal is a touch more flexible; Unity is a touch simpler. Both, however, converge on the same big idea we met last lesson — composition over inheritance. You do not build a "flying exploding pickup" by subclassing; you build it by attaching a movement component, a health component, and a mesh to one container.

Where this leaves our engine

Our C++ engine will borrow the best of both: an Entity that plays the role of Actor / GameObject, a Transform component that behaves like Unity's built-in transform or Unreal's root SceneComponent, and pluggable components for everything else. Next lesson pushes the idea one step further into a data-oriented layout — the Entity-Component-System.

Recap
  • An Actor (AActor) is Unreal's placeable object and a container for components; it has no transform of its own.
  • UActorComponent adds behaviour without a position; USceneComponent adds a transform and can be nested into a tree.
  • Exactly one SceneComponent is the root component — it defines the Actor's location, and its subtree moves with it.
  • UObject underpins everything, providing reflection (via UCLASS/UPROPERTY) and automatic garbage collection.
  • Unreal's Actor model and Unity's GameObject model map almost one-to-one and both embrace composition over inheritance.
