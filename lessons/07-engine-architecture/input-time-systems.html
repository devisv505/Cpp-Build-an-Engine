Input & Time as Engine Services

Your game loop is running and drawing to the screen, but right now nothing responds to the player. In this lesson we build two small but essential engine services: an Input class that turns raw Win32 messages into simple questions like “is W held down?”, and a Time class that tells every system how much time passed since the last frame. Getting these abstractions right early is what lets the rest of the engine stay clean and platform-independent.

What you'll learn
  • Why gameplay code should talk to services, not raw Win32 messages
  • How to record key and mouse state and expose IsKeyDown, IsKeyPressed and mouse delta
  • The “current vs. previous frame” trick that powers edge-detection
  • How a Time service computes DeltaTime, TotalTime and a frame count
  • Where these services plug into the game loop you built earlier

Why wrap the platform at all?

Windows delivers input through the message queue. Every key press arrives as a WM_KEYDOWN, every release as WM_KEYUP, and the mouse fires WM_MOUSEMOVE and friends. That is fine for collecting input, but it is a terrible thing to sprinkle throughout gameplay code. If a PlayerController reaches directly into Win32, it becomes impossible to test, impossible to port to another platform, and it can only ask questions at the exact moment a message arrives — not “is the key down right now, this frame?”

The fix is a classic engine pattern: hide the platform behind a service. Gameplay code asks the service simple, frame-oriented questions. The service is the only thing that knows Win32 exists. Unity does this with its Input class; Unreal does it with its input subsystem. We will do the same, just smaller.

Think in frames, not events

Gameplay usually wants “is jump held?” or “was jump pressed this frame?”. A service that snapshots state per frame answers both cleanly, without every system subscribing to raw messages.

Current vs. previous state

The single most useful idea here is keeping two snapshots of the keyboard: the state this frame and the state last frame. Comparing them lets us answer three different questions from the same data:

QuestionLogicUse case
IsKeyDowndown this framecontinuous movement (hold W to walk)
IsKeyPresseddown now, up last frameone-shot actions (jump, fire, toggle menu)
IsKeyReleasedup now, down last framecharge-and-release, letting go of a button

A Windows virtual-key code (like 'W' or VK_SPACE) is just a number from 0 to 255, so a fixed array of booleans is all the storage we need. Each frame we copy “current” into “previous”, then keep updating “current” from incoming messages.

The Input service

Here is a compact but complete sketch. The window procedure calls OnKeyDown/OnKeyUp as messages arrive; the game loop calls NewFrame() exactly once per frame, before pumping messages.

#pragma once
#include <array>
#include <cstdint>

// A tiny input service. Gameplay code queries it; only the window
// layer feeds it raw Win32 events. Nothing else touches the OS.
class Input {
public:
    // Called once per frame, before we process the message queue.
    // Snapshots "current" into "previous" so edge detection works.
    void NewFrame() {
        m_prevKeys = m_keys;          // copy the whole array
        m_mouseDeltaX = 0;            // deltas are per-frame
        m_mouseDeltaY = 0;
    }

    // --- Fed by the window procedure (WM_KEYDOWN / WM_KEYUP) ---
    void OnKeyDown(uint8_t vk) { m_keys[vk] = true;  }
    void OnKeyUp(uint8_t vk)   { m_keys[vk] = false; }

    // WM_MOUSEMOVE gives absolute client coords; we accumulate a delta.
    void OnMouseMove(int x, int y) {
        m_mouseDeltaX += x - m_mouseX;
        m_mouseDeltaY += y - m_mouseY;
        m_mouseX = x;
        m_mouseY = y;
    }
    void OnMouseButton(int button, bool down) {
        if (button >= 0 && button < 3) m_mouseButtons[button] = down;
    }

    // --- Queried by gameplay ---
    bool IsKeyDown(uint8_t vk)     const { return m_keys[vk]; }
    // Down now, but was up last frame = a fresh press this frame.
    bool IsKeyPressed(uint8_t vk)  const { return m_keys[vk] && !m_prevKeys[vk]; }
    bool IsKeyReleased(uint8_t vk) const { return !m_keys[vk] && m_prevKeys[vk]; }

    bool IsMouseButtonDown(int button) const {
        return (button >= 0 && button < 3) && m_mouseButtons[button];
    }
    int MouseDeltaX() const { return m_mouseDeltaX; }
    int MouseDeltaY() const { return m_mouseDeltaY; }

private:
    std::array<bool, 256> m_keys{};      // current frame
    std::array<bool, 256> m_prevKeys{};  // previous frame
    std::array<bool, 3>   m_mouseButtons{};
    int m_mouseX = 0, m_mouseY = 0;      // last known position
    int m_mouseDeltaX = 0, m_mouseDeltaY = 0;
};

Wiring it into the window procedure is a handful of cases. The WPARAM already carries the virtual-key code, and lParam packs the mouse position:

// g_input is our single Input instance owned by the engine.
LRESULT CALLBACK WndProc(HWND hwnd, UINT msg, WPARAM wParam, LPARAM lParam) {
    switch (msg) {
    case WM_KEYDOWN:   g_input.OnKeyDown(static_cast<uint8_t>(wParam)); return 0;
    case WM_KEYUP:     g_input.OnKeyUp(static_cast<uint8_t>(wParam));   return 0;
    case WM_MOUSEMOVE:
        g_input.OnMouseMove(GET_X_LPARAM(lParam), GET_Y_LPARAM(lParam));
        return 0;
    case WM_LBUTTONDOWN: g_input.OnMouseButton(0, true);  return 0;
    case WM_LBUTTONUP:   g_input.OnMouseButton(0, false); return 0;
    case WM_DESTROY:     PostQuitMessage(0);              return 0;
    }
    return DefWindowProc(hwnd, msg, wParam, lParam);
}
Call NewFrame at the right moment

Call NewFrame() before you pump messages each frame. If you call it after, you wipe out the “previous” snapshot that IsKeyPressed depends on, and one-shot actions will fire every frame the key is held. GET_X_LPARAM lives in <windowsx.h>.

The Time service

Almost every system needs to know how much real time elapsed since the last frame — that is delta time. Multiply movement by delta time and your game runs at the same speed whether the machine renders 60 or 240 frames per second. We measure it with <chrono>, the standard C++ high-resolution clock, so there is no Win32 in this file at all.

#pragma once
#include <chrono>
#include <cstdint>

// Owns the engine clock. Call Tick() once per frame; everyone else
// just reads DeltaTime()/TotalTime().
class Time {
public:
    Time() {
        auto now = Clock::now();
        m_start = now;
        m_lastFrame = now;
    }

    // Advance the clock by one frame.
    void Tick() {
        auto now = Clock::now();
        // Duration in seconds as a float — perfect for gameplay math.
        m_deltaTime = std::chrono::duration<float>(now - m_lastFrame).count();
        m_totalTime = std::chrono::duration<float>(now - m_start).count();
        m_lastFrame = now;
        ++m_frameCount;
    }

    float    DeltaTime()  const { return m_deltaTime; }  // seconds since last frame
    float    TotalTime()  const { return m_totalTime; }  // seconds since startup
    uint64_t FrameCount() const { return m_frameCount; }

private:
    using Clock = std::chrono::steady_clock; // monotonic: never jumps backwards
    Clock::time_point m_start;
    Clock::time_point m_lastFrame;
    float    m_deltaTime = 0.0f;
    float    m_totalTime = 0.0f;
    uint64_t m_frameCount = 0;
};
Why steady_clock?

We use steady_clock instead of system_clock because it is monotonic — it cannot be dragged backwards by the user changing the system time or by an NTP sync. Time going negative for a frame would send everything in your engine flying.

Plugging into the game loop

Remember the loop skeleton from the game-loop lesson? The two services slot straight in. Notice the order: reset input, drain messages (which feed input), advance time, then update and render using the fresh data.

Input g_input;
Time  g_time;

void RunFrame() {
    g_input.NewFrame();            // 1. snapshot previous, clear deltas

    MSG msg;
    while (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE)) {
        TranslateMessage(&msg);    // 2. WndProc feeds g_input here
        DispatchMessage(&msg);
    }

    g_time.Tick();                 // 3. compute delta time for this frame
    float dt = g_time.DeltaTime();

    // 4. Gameplay: clean, platform-free, testable.
    if (g_input.IsKeyPressed(VK_SPACE))
        Jump();                    // fires exactly once per press

    if (g_input.IsKeyDown('W'))
        MoveForward(5.0f * dt);    // frame-rate independent movement

    Render();
}

Look at how little the gameplay code knows. It never mentions WM_KEYDOWN, never touches chrono, never sees a Windows handle. When we build the component system in the next module, a Transform component can advance its position by velocity * Time::DeltaTime() and read Input without any idea that Windows is underneath. That is exactly the seam that would let us swap in a different platform layer later without rewriting a single line of game logic.

Don't scatter Win32 into gameplay

The moment a component calls GetAsyncKeyState or reads the message queue directly, you have punched a hole through the abstraction. Keep every OS call on the platform side of the services, or the whole benefit collapses.

Recap
  • Services hide platform details so gameplay code stays clean, testable and portable.
  • Input keeps a current and previous key snapshot; NewFrame() copies current into previous each frame before messages are pumped.
  • IsKeyDown is for held keys; IsKeyPressed/IsKeyReleased detect the edge using the previous snapshot. Mouse delta is accumulated per frame and reset in NewFrame().
  • Time uses std::chrono::steady_clock to expose DeltaTime, TotalTime and FrameCount; multiply motion by delta time for frame-rate independence.
  • In the loop: NewFrame() → pump messages → Tick() → update & render — gameplay only ever talks to the services.
