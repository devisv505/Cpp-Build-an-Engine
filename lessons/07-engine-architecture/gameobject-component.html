The GameObject–Component Model (Unity)

In the last lesson we built a main loop that ticks our subsystems every frame. But a loop that ticks nothing interesting is just a spinning fan. Now we need a way to describe the things in our world — the player, the camera, a crate, a light — and give them behavior. Unity solved this with an elegant idea: a GameObject is an almost-empty container, and everything it can do comes from Components you snap onto it. This is the exact model we'll build for our engine.

The problem with deep inheritance

A beginner's first instinct for "things in a game" is a class tree. You make a base Entity, then Character inherits from it, then Player and Enemy inherit from Character. It feels tidy — for about a week.

Then reality shows up. You want a crate that can be shot (needs health) but can't move. You want a security camera that renders to a screen but has no mesh of its own. You want a flying turret that shares behavior with both the stationary turret and the drone. Where do those go in a single tree? You end up with a tangle, copy-pasting logic between branches, or cramming everything into a bloated base class that every object drags around.

The core idea

Inheritance answers "what is this?" (a Player is a Character). Composition answers "what can this do?" (this object has a mesh, has a rigidbody, has health). Games change constantly, so "has-a" is far more flexible than "is-a".

Composition over inheritance

Unity's answer is to stop building tall class trees. Instead, every object in the scene is the same lightweight type — a GameObject — and you describe what it does by attaching small, focused Component pieces:

  • Transform — position, rotation, and scale. Every GameObject has exactly one, always.
  • MeshRenderer — draws a 3D mesh with a material.
  • Camera — defines a viewpoint the world is rendered from.
  • Rigidbody / Collider — physics and collision.
  • Your own scripts — a PlayerController, a Health, a Spinner.

A player is a GameObject with a Transform, a MeshRenderer, a Rigidbody, and a PlayerController script. A camera is a GameObject with just a Transform and a Camera component. A trigger volume is a GameObject with a Transform and a Collider — no mesh at all. Same container, different mix of parts. That crate-that-can-be-shot? Give it a Health component and you're done — no class hierarchy to fight.

The Transform is always there

There is one component every GameObject has and can never remove: the Transform. It answers a question every object in a 3D world needs answered — where am I, how am I oriented, and how big am I? Even an invisible spawn point needs a position. Because it's guaranteed, other components can safely rely on it: a MeshRenderer reads the Transform to know where to draw; a Camera reads it to know where to look from.

Transforms also form a hierarchy (parent–child). Parent a sword to a hand, move the hand, and the sword follows automatically because child positions are expressed relative to the parent. We'll lean on this heavily when we build scene graphs later.

A conceptual C++ sketch

Let's translate the idea into C++. We want a Component base class with lifecycle hooks, a GameObject that owns a list of components plus a guaranteed Transform, and a type-based lookup so we can ask "does this object have a Camera?" We'll use std::unique_ptr so each GameObject owns its components and cleans them up automatically.

#include <memory>
#include <vector>

// Every component derives from this. The virtual hooks are the lifecycle.
class GameObject; // forward declaration

class Component {
public:
    virtual ~Component() = default;

    // Called once, right after the component is attached.
    virtual void Awake() {}
    // Called once, before the first Update, after all Awakes are done.
    virtual void Start() {}
    // Called every frame; dt is the seconds since the last frame.
    virtual void Update(float dt) {}

    GameObject* gameObject = nullptr; // set by GameObject when attached
};

class GameObject {
public:
    // Create a component of type T, attach it, and return a raw pointer.
    // Args are forwarded to T's constructor.
    template <typename T, typename... Args>
    T* AddComponent(Args&&... args) {
        auto owned = std::make_unique<T>(std::forward<Args>(args)...);
        T* ptr = owned.get();
        ptr->gameObject = this;      // back-pointer so the component can find us
        components_.push_back(std::move(owned));
        ptr->Awake();                // Awake fires immediately on attach
        return ptr;
    }

    // Find the first attached component that IS-A T. Returns nullptr if none.
    template <typename T>
    T* GetComponent() {
        for (auto& c : components_) {
            // dynamic_cast succeeds only if the stored object really is a T.
            if (T* hit = dynamic_cast<T*>(c.get()))
                return hit;
        }
        return nullptr;
    }

    void StartAll() {
        for (auto& c : components_) c->Start();
    }

    void UpdateAll(float dt) {
        for (auto& c : components_) c->Update(dt);
    }

private:
    std::vector<std::unique_ptr<Component>> components_;
};

Notice how AddComponent<T>() and GetComponent<T>() mirror Unity's API almost exactly. The template parameter T is the component type you want, and the engine does the bookkeeping. Here's how a caller uses it:

// A simple gameplay script — just another Component.
class Spinner : public Component {
public:
    float speed = 90.0f; // degrees per second

    void Start() override {
        // Grab a sibling component up front so Update stays cheap.
        transform_ = gameObject->GetComponent<Transform>();
    }

    void Update(float dt) override {
        if (transform_)
            transform_->rotationY += speed * dt; // spin around Y
    }

private:
    Transform* transform_ = nullptr;
};

int main() {
    GameObject crate;
    crate.AddComponent<Transform>();     // every object needs a Transform
    crate.AddComponent<MeshRenderer>();  // give it something to draw
    crate.AddComponent<Spinner>();       // and some behavior

    crate.StartAll();                   // Start runs after all Awakes

    // The engine's main loop would call this once per frame.
    for (int frame = 0; frame < 3; ++frame) {
        crate.UpdateAll(1.0f / 60.0f);  // ~16.7 ms per frame
    }
    return 0;
}
Cache your lookups

Notice Spinner calls GetComponent once in Start and stores the pointer. GetComponent loops and does a dynamic_cast, so calling it every frame inside Update is wasteful. Grab it once, reuse it. Our real engine will later replace the linear search with a type-indexed map for speed.

The lifecycle: Awake, Start, Update

Those three virtual functions are the heartbeat of the model. Unity guarantees a specific order so components can depend on each other safely:

HookRunsUse it for
AwakeOnce, the moment the component is attached / the object loadsWiring up internal state that does not depend on other objects
StartOnce, after every object's Awake has finished, just before the first frameGrabbing references to other components/objects that must already exist
UpdateEvery frame, forever, while the object is activePer-frame logic: movement, input, animation, timers

Why split Awake from Start? Ordering. If object A's Start needs to find object B, you must be sure B has finished initializing. By running all Awakes first and then all Starts, the engine guarantees that by the time any Start runs, every object in the scene already exists and has done its own private setup. It removes a whole category of "was it created yet?" bugs.

Don't reach across objects in Awake

Inside Awake, other GameObjects may not be initialized yet — some might not even exist. Only touch your own data there. Anything that depends on another object belongs in Start, once the whole scene is guaranteed to be up.

This maps cleanly onto the main loop from the previous lesson: the loop calls UpdateAll(dt) on every active GameObject each frame, right between polling input and rendering. Later we'll add sibling passes like FixedUpdate (for physics) and LateUpdate (for cameras that must follow objects after they've moved), but Awake/Start/Update are the foundation.

Why we're building this model

This architecture is not a toy — it is the engine backbone. Transform, MeshRenderer, and Camera are ordinary Components; the renderer walks the scene and forwards draw data to Direct3D on Windows or Metal on macOS. Add a feature by writing one small component, not by rewriting a class tree.

In the next lesson we'll look at Unreal Engine's alternative — the Actor model — which makes different trade-offs. Seeing both will help you understand why we chose the Unity-style approach for what we're building.

Recap
  • Deep inheritance trees break down fast in games; composition ("has-a") scales far better than inheritance ("is-a").
  • A GameObject is a lightweight container that always has a Transform and gains behavior from attached Components (MeshRenderer, Camera, scripts).
  • AddComponent<T>() creates and owns a component; GetComponent<T>() finds one by its type at runtime — cache the result instead of calling it every frame.
  • The Awake → Start → Update lifecycle guarantees safe initialization order: all Awakes, then all Starts, then Update every frame.
  • This is the exact model our engine will build on, with each engine feature written as a small Component.
