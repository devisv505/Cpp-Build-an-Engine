Assets & Resource Management

A single texture can be tens of megabytes, and uploading it to the GPU takes real time. Yet in a scene, hundreds of objects might all use the same brick texture or the same cube mesh. Loading that file once per object would be catastrophic. In this lesson we build the part of the engine that loads each asset exactly once and hands out shared references, so a thousand walls quietly share one GPU resource — just like the asset databases inside Unity and Unreal.

What you'll learn
  • Why assets are expensive and should be loaded once and shared
  • How std::shared_ptr lets many objects co-own one resource safely
  • How RAII and ComPtr guarantee GPU memory is released automatically
  • How to build a small ResourceManager<T> that caches by path and avoids duplicate loads

Why assets need special treatment

Everything the renderer draws comes from an asset: a mesh (vertex and index buffers), a texture (pixels living in GPU memory), or a compiled shader. Two things make them different from the ordinary data we have used so far:

  • They are expensive. Reading a file from disk, decoding it, and uploading it to the GPU can take milliseconds each — an eternity when you have hundreds of them.
  • They are shared. The same crate model may appear 500 times in a level. All 500 instances want to point at one vertex buffer, not 500 identical copies wasting memory.

The rule that falls out of this is simple and is the heart of the whole lesson: load once, share everywhere, free when nobody is left. In the last lesson each object had a Transform describing where it is; an asset describes what it looks like. Many transforms can reference the same asset.

A useful mental model

Think of the ResourceManager as a library. You ask for a book by its title (the file path). If the library already has it, you get a reference to the copy on the shelf. If not, the library fetches it once, then everyone borrows that same copy.

Shared ownership with shared_ptr

When many objects use one asset, who owns it? If any single object owned it and got deleted, the others would be left pointing at freed memory. If nobody owned it, it would leak. The right tool is shared ownership: std::shared_ptr<T> keeps a reference count, and the resource is destroyed the instant the last owner lets go. That is exactly the "free when nobody is left" rule, enforced automatically.

So our manager will return std::shared_ptr<Mesh>, std::shared_ptr<Texture>, and so on. A renderable component just holds onto one of those pointers. Copy the component, copy the shared_ptr, and the count goes up — no data is duplicated.

Lifetime of GPU resources: RAII and ComPtr

A shared_ptr handles the C++ object, but the Windows payload lives on the GPU behind Direct3D interfaces like ID3D11Texture2D. Those are COM objects with their own reference count that you release by calling Release(). Forgetting that call leaks video memory. We never call it by hand — we wrap every Direct3D interface in Microsoft::WRL::ComPtr, which calls Release() for us when it goes out of scope. This is RAII: acquisition in the constructor, release in the destructor. 

#include <d3d11.h>
#include <wrl/client.h>
#include <string>

using Microsoft::WRL::ComPtr;

// One Texture owns its GPU resources. When the last shared_ptr<Texture>
// is released, ~Texture runs, each ComPtr calls Release(), and the
// GPU memory is freed automatically. We write no cleanup code at all.
class Texture {
public:
    // Does the expensive work: read the file and upload pixels to the GPU.
    void LoadFromFile(ID3D11Device* device, const std::string& path);

    // Shaders bind a texture through its shader-resource view.
    ID3D11ShaderResourceView* GetSRV() const { return m_srv.Get(); }

private:
    ComPtr<ID3D11Texture2D>          m_texture; // the pixels, in GPU memory
    ComPtr<ID3D11ShaderResourceView> m_srv;     // how shaders sample it
};
Two reference counts, one goal

The shared_ptr counts C++ owners of the Texture; the ComPtr counts COM references to the D3D interface. They live at different layers but cooperate: when the last shared_ptr dies, the ComPtr members are destroyed, dropping the D3D count to zero and releasing the video memory. Never mix a raw Release() call into this — let RAII do it.

The cache: a ResourceManager<T>

Now we tie it together. The manager keeps an std::unordered_map keyed by file path. When you ask for an asset, it checks the map. A hit returns the existing pointer; a miss loads the asset, stores it, and returns it. That single lookup is what turns 500 load requests into one actual load.

One subtlety decides how memory behaves: should the cache own the resource, or merely track it? If we store shared_ptr in the map, the cache is an owner, so nothing is ever freed while the manager lives. If we store a std::weak_ptr, the cache observes without keeping the asset alive — the resource frees as soon as the last real user releases it, and the manager reloads it on demand. The weak_ptr version is the one that scales, so that is what we build. 

#include <memory>
#include <string>
#include <unordered_map>

struct ID3D11Device; // fwd-declare: we only pass the pointer through

// Loads assets of type T once and shares them. T must provide a
// LoadFromFile(...) method (see Texture above). Templated so the same
// logic serves Mesh, Texture, Shader, ... with no duplication.
template <typename T>
class ResourceManager {
public:
    explicit ResourceManager(ID3D11Device* device) : m_device(device) {}

    // Return a shared handle to the asset at 'path', loading it only
    // if it is not already live in the cache.
    std::shared_ptr<T> Load(const std::string& path) {
        // 1. Have we loaded this path before, and is it still alive?
        auto it = m_cache.find(path);
        if (it != m_cache.end()) {
            if (std::shared_ptr<T> existing = it->second.lock()) {
                return existing;            // cache hit: share the copy
            }
        }

        // 2. Cache miss (or the old one expired): do the expensive load.
        std::shared_ptr<T> resource = std::make_shared<T>();
        resource->LoadFromFile(m_device, path);

        // 3. Track it with a weak reference so we do NOT keep it alive
        //    by ourselves, but can still hand it out again next time.
        m_cache[path] = resource;
        return resource;
    }

private:
    ID3D11Device* m_device = nullptr; // needed to create GPU resources
    std::unordered_map<std::string, std::weak_ptr<T>> m_cache;
};

Using it from the rest of the engine is pleasantly boring, which is the point: 

ResourceManager<Texture> textures(device);

// The file is read from disk and uploaded to the GPU exactly once here.
std::shared_ptr<Texture> brick = textures.Load("assets/brick.png");

// Every later request for the same path returns the SAME Texture.
// No disk read, no GPU upload — just a reference-count bump.
std::shared_ptr<Texture> sameBrick = textures.Load("assets/brick.png");

// Proof they are one object, shared by two owners:
// brick.get() == sameBrick.get(), and brick.use_count() == 2.

Thread-safety comes later

This sketch assumes assets are loaded from one thread (typically the main thread during level load). Real engines add a mutex around the map or a background loading queue. We keep it single-threaded for now so the ownership idea stays front and centre — the caching logic will not change when we make it concurrent.

How this fits the engine

A renderable component — the thing we will attach to a scene object next to its Transform — simply holds a std::shared_ptr<Mesh> and a std::shared_ptr<Texture> it got from the managers. Spawning 500 crates means 500 transforms but two shared assets. When a level unloads and those components disappear, the last references drop, the ComPtr destructors fire, and the GPU memory returns on its own. You now have the same "reference one asset, instantiate it everywhere" model that Unity's meshes and Unreal's static meshes are built on.

The same cache with Metal

On macOS, the cache and shared_ptr<Texture> ownership remain identical. Only the private payload changes from Direct3D interfaces to MTL::Texture. Keep that object inside the Metal implementation so callers still request a portable Texture by path.

Configuration is not a cached GPU asset

The YAML settings added in Module 08 are tiny CPU-side startup data. Load and validate them once into EngineConfig; do not route them through this mesh/texture cache or upload them to a graphics backend.

Recap
  • Assets (meshes, textures, shaders) are expensive to load and upload, so load each one once and share it.
  • std::shared_ptr<T> gives many objects safe co-ownership; the asset frees when the last owner releases it.
  • Use ComPtr for Direct3D on Windows and metal-cpp ownership helpers for Metal on macOS so RAII releases GPU resources.
  • A ResourceManager<T> caches by file path in an unordered_map, turning duplicate load requests into one real load.
  • Storing weak_ptr in the cache lets it track resources without keeping them alive, so unused assets free themselves.
