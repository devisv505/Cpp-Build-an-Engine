Dot & Cross Products

Last lesson you learned that a vector is just a bundle of numbers describing a direction and length in 3D. On their own, vectors are quiet. The moment they start talking to each other — through the dot product and the cross product — you get the two most important operations in all of real-time graphics. These two little formulas power lighting, camera direction, surface normals, and the coordinate frames every engine is built on. Get comfortable with them here and the rest of the math module will feel easy.

What you'll learn
  • How to compute the dot product and read its sign to tell facing from perpendicular from opposite.
  • How the dot product relates to the angle between vectors and to projection.
  • How diffuse lighting uses the classic N·L term.
  • How the cross product builds a perpendicular vector for surface normals and coordinate frames.
  • Why handedness matters, and how DirectX's left-handed convention affects your results.

The dot product

The dot product takes two vectors and returns a single number (a scalar). For vectors a and b the formula is beautifully simple — multiply matching components and add:

a · b = a.x*b.x + a.y*b.y + a.z*b.z

That number by itself looks unremarkable, but it hides a second identity that makes it powerful:

a · b = |a| * |b| * cos(θ)

Here |a| and |b| are the lengths (magnitudes) of the vectors and θ is the angle between them. So the dot product bundles together how long the vectors are and how aligned they are. If you first normalize both vectors (make them length 1, which you saw last lesson), the lengths drop out and you are left with just cos(θ) — a value from -1 to 1 that tells you the angle directly.

Reading the sign

Because cosine is positive for small angles and negative for large ones, the sign of the dot product is an instant direction test:

Dot productAngleMeaning
> 0less than 90°Vectors point the same general way (facing).
= 0exactly 90°Vectors are perpendicular (orthogonal).
< 0more than 90°Vectors point roughly opposite.

This is how a game checks "is the enemy in front of me?" — take the vector from you to the enemy, dot it with your forward vector, and if the result is positive the enemy is somewhere ahead.

A vector dotted with itself

Notice that a · a equals a.x² + a.y² + a.z², which is exactly the length squared. That is why we often use a · a to compare lengths without paying for a square root.

A worked example

Let a = (1, 0, 0) and b = (1, 1, 0). The dot product is 1*1 + 0*1 + 0*0 = 1. The length of a is 1 and the length of b is √2 ≈ 1.414. So cos(θ) = 1 / (1 * 1.414) ≈ 0.707, which means θ = 45°. That matches our intuition: b points diagonally up-right, exactly 45° off the x-axis.

Projection

The dot product also answers "how much of vector a lies along the direction of b?" If b is a unit vector, then a · b is the signed length of a's shadow onto b. Scale b by that number and you get the projection of a onto b. We will use this later to slide a character along a wall or to split velocity into "into the surface" and "along the surface" parts.

Dot product in code

Let's extend the small Vec3 type from last lesson with a dot product and a couple of helpers. This is real, compilable C++17.

#include <iostream>
#include <cmath>

struct Vec3 {
    float x = 0, y = 0, z = 0;
};

// Dot product: multiply matching components and sum.
float dot(const Vec3& a, const Vec3& b) {
    return a.x * b.x + a.y * b.y + a.z * b.z;
}

float length(const Vec3& v) {
    return std::sqrt(dot(v, v));   // |v| = sqrt(v · v)
}

// Return a unit-length copy (guard against divide-by-zero).
Vec3 normalize(const Vec3& v) {
    float len = length(v);
    if (len < 1e-6f) return {0, 0, 0};
    return { v.x / len, v.y / len, v.z / len };
}

// Angle between two vectors, in degrees.
float angleDeg(const Vec3& a, const Vec3& b) {
    float c = dot(normalize(a), normalize(b));
    // Clamp to avoid tiny floating-point overshoot past ±1.
    c = std::fmax(-1.0f, std::fmin(1.0f, c));
    return std::acos(c) * (180.0f / 3.14159265f);
}

int main() {
    Vec3 a{1, 0, 0};
    Vec3 b{1, 1, 0};

    std::cout << "dot   = " << dot(a, b)      << "\n"; // 1
    std::cout << "angle = " << angleDeg(a, b) << " deg\n"; // ~45
    return 0;
}
Clamp before acos

Floating-point rounding can make a normalized dot product come back as 1.0000001. Feeding that to std::acos returns NaN. Always clamp the value to the range [-1, 1] first, as the code above does.

Lighting: the N·L term

Here is where the dot product earns its keep. The most common lighting model, Lambert diffuse, says a surface is brightest when it faces the light head-on and dims to zero as it turns away. "Facing the light" is exactly an angle test — and we already know the tool for that.

Take N, the surface normal (a unit vector pointing straight out of the surface), and L, a unit vector pointing toward the light. The brightness is:

diffuse = max(0, N · L)

When the surface points right at the light, N · L = 1 (full brightness). At 90° it is 0 (the light grazes the surface). Past 90° the dot goes negative, meaning the surface is turned away — and the max(0, …) clamps it to darkness so we never get negative light. You will write this exact line in HLSL when we reach the shaders module. Here it is:

// Pixel shader snippet: classic Lambert diffuse lighting.
float3 N = normalize(input.worldNormal);   // surface normal
float3 L = normalize(lightPos - input.worldPos); // toward the light

float ndotl = saturate(dot(N, L));  // saturate = clamp to [0,1]
float3 color = albedo * lightColor * ndotl;

That single dot(N, L) is the heart of nearly every lighting shader ever written. When our engine later feeds normals into a shader, this is what makes surfaces look three-dimensional instead of flat.

The cross product

The dot product turns two vectors into a number. The cross product does something different: it turns two vectors into a third vector that is perpendicular to both of them. That property is exactly what we need to build surface normals and coordinate axes.

a × b = ( a.y*b.z - a.z*b.y,
          a.z*b.x - a.x*b.z,
          a.x*b.y - a.y*b.x )

The length of the result also carries meaning: |a × b| = |a| * |b| * sin(θ). It is largest when the vectors are perpendicular and zero when they are parallel (parallel vectors span no area, so there is no unique perpendicular).

Handedness

There are always two perpendicular directions — straight up or straight down. Which one the cross product picks is decided by handedness. With the standard formula and a right-handed system, point the fingers of your right hand along a and curl them toward b; your thumb points along a × b.

DirectX is left-handed

Direct3D conventionally uses a left-handed coordinate system (with +Z going into the screen), while much math you read online assumes right-handed. The cross-product formula itself doesn't change, but the direction you interpret as "outward" can flip. If your normals or triangle winding come out inside-out, handedness is the usual culprit — we'll pin down our engine's convention when we set up the camera.

Surface normals from a triangle

Every triangle in a mesh has three corner positions. To find the normal — the direction the face points — take two edges of the triangle and cross them. That is literally how mesh normals are generated.

#include <iostream>
#include <cmath>

struct Vec3 { float x = 0, y = 0, z = 0; };

Vec3 operator-(const Vec3& a, const Vec3& b) {
    return { a.x - b.x, a.y - b.y, a.z - b.z };
}

float dot(const Vec3& a, const Vec3& b) {
    return a.x*b.x + a.y*b.y + a.z*b.z;
}

// Cross product: result is perpendicular to both a and b.
Vec3 cross(const Vec3& a, const Vec3& b) {
    return {
        a.y * b.z - a.z * b.y,
        a.z * b.x - a.x * b.z,
        a.x * b.y - a.y * b.x
    };
}

Vec3 normalize(const Vec3& v) {
    float len = std::sqrt(dot(v, v));
    if (len < 1e-6f) return {0, 0, 0};
    return { v.x / len, v.y / len, v.z / len };
}

// Face normal of the triangle (p0, p1, p2).
Vec3 triangleNormal(const Vec3& p0, const Vec3& p1, const Vec3& p2) {
    Vec3 edge1 = p1 - p0;
    Vec3 edge2 = p2 - p0;
    return normalize(cross(edge1, edge2));
}

int main() {
    // A triangle lying flat in the XZ plane.
    Vec3 p0{0, 0, 0};
    Vec3 p1{1, 0, 0};
    Vec3 p2{0, 0, 1};

    Vec3 n = triangleNormal(p0, p1, p2);
    std::cout << "normal = ("
              << n.x << ", " << n.y << ", " << n.z << ")\n";
    // Expect (0, -1, 0): perpendicular to the flat triangle.
    return 0;
}

Run it and the normal comes out as (0, -1, 0) — straight down, perpendicular to a triangle that lies flat. Swap the order of the two edges (cross edge2 with edge1) and the normal flips to (0, 1, 0). That flip is handedness in action, and it is why the winding order of your triangle vertices matters when the engine decides which side of a face is "front".

Building a coordinate frame

The cross product's other superpower is constructing a full set of perpendicular axes from a single direction. Given a forward vector and a rough "up" hint, one cross product gives you a right vector, and a second cross gives you a clean up vector — three mutually perpendicular axes. This is exactly the recipe a camera or a Transform component uses to know its own orientation, and we'll build it directly on top of these operations in the next module.

Cross product is not commutative

Order matters: a × b = -(b × a). Swapping the operands reverses the result. Contrast this with the dot product, where a · b = b · a. Mixing up the order is one of the most common sources of inverted normals and mirrored coordinate frames — pick a convention and stick to it.

The same operations in Metal

Metal Shading Language provides the same dot, cross, normalize, and max functions as HLSL. The lighting equation is unchanged on macOS; only the shader language and the engine's chosen handedness affect how you write and interpret the values.

Recap
  • The dot product is a.x*b.x + a.y*b.y + a.z*b.z and also equals |a||b|cos(θ); its sign tells you facing (+), perpendicular (0), or opposite (−).
  • Normalizing first turns the dot product into cos(θ), giving you the angle and enabling projection.
  • Diffuse lighting is just max(0, N · L) — the single most important line in shading.
  • The cross product produces a vector perpendicular to both inputs, used for triangle normals and coordinate frames; its length is |a||b|sin(θ).
  • Handedness and operand order decide which way the result points — DirectX is left-handed, so watch your winding order and conventions.
