Vectors: the Language of 3D

In the last lesson you learned that every object in a 3D scene lives inside a coordinate space. But a coordinate on its own is just three numbers. The vector is the tool that turns those numbers into something you can move, aim, and measure. Nearly everything in graphics — a camera pointing at a target, a bullet's trajectory, the surface a light bounces off — is expressed as a vector, so this is the single most important piece of math in your engine.

What you'll learn
  • The two ways to read a vector: as a position and as a direction.
  • Adding, subtracting, and scaling vectors — and what each operation means geometrically.
  • How to measure a vector's length (magnitude) with a square root.
  • What normalization is and why unit vectors matter so much.
  • How to reuse your Vector3 class and preview DirectXMath's XMFLOAT3.

What a vector actually is

A 3D vector is just three numbers grouped together: an x, a y, and a z component. The magic is that the same three numbers can mean two different things depending on how you use them.

  • As a position, a vector says where something is. The point (3, 0, 5) is "3 units right, 0 up, 5 forward" from the origin.
  • As a direction (or displacement), a vector says which way and how far. The vector (0, 1, 0) means "straight up," and (0, 5, 0) means "straight up, five units of it."

This dual nature is why we can use one Vector3 type for the position of a cube, the direction a camera faces, and the velocity of a projectile. When we build our Transform component later, its position will be a vector, and so will the direction it moves each frame.

Adding and subtracting vectors

Vector addition works component by component: you add the x's, the y's, and the z's. Geometrically, adding means "start at the first vector, then walk along the second." If you are at position (2, 0, 3) and you move by the direction (0, 0, 4), you end up at (2, 0, 7).

Subtraction is the operation you will reach for constantly. The vector b - a is "the direction from a to b." To make a camera look at a target, you compute target - cameraPosition and that difference is exactly the direction the camera must face.

Remember: destination minus source

The direction "from A to B" is always B - A, not A - B. Get the order backwards and your object points in exactly the wrong direction — a very common first bug.

Scalar multiplication

Multiplying a vector by a single number — a scalar — scales its length while keeping its direction. Multiply (0, 1, 0) by 5 and you get (0, 5, 0): same "up" direction, five times as long. Multiply by a negative number and the vector flips to point the opposite way. This is how you turn a velocity into a per-frame step: velocity * deltaTime.

Reusing our Vector3 class

Back in the OOP module we built a small Vector3 struct with overloaded operators. Here it is again, now that the operations above have real geometric meaning. Notice how each operator is just the component-wise rule we described.

#pragma once
#include <cmath>

struct Vector3 {
    float x = 0.0f;
    float y = 0.0f;
    float z = 0.0f;

    // Add two vectors component by component.
    Vector3 operator+(const Vector3& rhs) const {
        return { x + rhs.x, y + rhs.y, z + rhs.z };
    }

    // Subtract: (b - a) is the direction from a to b.
    Vector3 operator-(const Vector3& rhs) const {
        return { x - rhs.x, y - rhs.y, z - rhs.z };
    }

    // Scale by a single number (scalar multiply).
    Vector3 operator*(float s) const {
        return { x * s, y * s, z * s };
    }

    // Length (magnitude): the Pythagorean theorem in 3D.
    float Length() const {
        return std::sqrt(x * x + y * y + z * z);
    }

    // Return a unit-length copy pointing the same way.
    Vector3 Normalized() const {
        float len = Length();
        if (len > 0.0f) {
            return { x / len, y / len, z / len };
        }
        return { 0.0f, 0.0f, 0.0f }; // guard against divide-by-zero
    }
};

Length: how long is a vector?

The magnitude (or length) of a vector answers "how far?" It is the straight-line distance from the tail of the arrow to its tip. In 3D it comes straight from the Pythagorean theorem, extended to three axes:

length = sqrt(x*x + y*y + z*z)

So the vector (3, 0, 4) has length sqrt(9 + 0 + 16) = sqrt(25) = 5. Once you can measure length, the distance between two points a and b is simply (b - a).Length() — the length of the vector between them. You'll use that for things like "is this enemy close enough to attack?"

sqrt is not free

The square root is a relatively expensive CPU operation. When you only need to compare distances (which is closer?), compare the squared lengths and skip the sqrt entirely — the ordering is identical.

Normalization: making a unit vector

A unit vector is a vector with a length of exactly 1. Normalizing means dividing a vector by its own length so it keeps its direction but shrinks (or grows) to length 1. That is exactly what Normalized() does above.

Why do we care so much? Because a unit vector represents pure direction with no magnitude baked in. Almost every direction in graphics is expected to be normalized:

  • Surface normals used in lighting must be unit length or your shading will be too bright or too dark.
  • The direction a camera or character faces is a unit vector, so you can multiply it by any speed you like.
  • Many math formulas (like the dot product in the next lesson) assume their inputs are normalized.
Never normalize a zero vector

A vector of length 0 has no direction, and dividing by 0 produces NaN values that silently corrupt every calculation downstream. Always guard with an if (len > 0.0f) check, exactly as the class above does.

Putting it together

Here is a short, complete program that uses every operation from this lesson. It computes the direction from a camera to a target and steps the camera toward it by a fixed speed.

#include <iostream>
#include "Vector3.h"

int main() {
    Vector3 cameraPos { 0.0f, 2.0f, 0.0f };
    Vector3 target    { 3.0f, 2.0f, 4.0f };

    // Direction from camera to target = destination - source.
    Vector3 toTarget = target - cameraPos;

    float distance = toTarget.Length();          // how far away is it?
    Vector3 dir    = toTarget.Normalized();       // pure direction, length 1

    std::cout << "Distance to target: " << distance << "\n";
    std::cout << "Facing direction:  ("
              << dir.x << ", " << dir.y << ", " << dir.z << ")\n";

    // Move the camera 1.5 units along that direction.
    float speed = 1.5f;
    cameraPos = cameraPos + dir * speed;          // add a scaled direction

    std::cout << "New camera position: ("
              << cameraPos.x << ", " << cameraPos.y << ", " << cameraPos.z << ")\n";
    return 0;
}

Run it and you'll see the distance is 5 (a 3-4 right triangle on the x/z plane), the facing direction is a unit vector, and the camera slides 1.5 units toward the target. Every line here is an operation you'll perform thousands of times per frame in a real engine.

A preview of DirectXMath

Our hand-written Vector3 is perfect for learning, but when we reach the DirectX modules we'll store vector data in the format the GPU and the DirectXMath library expect: XMFLOAT3. It is a plain three-float struct, deliberately simple so it maps directly to what a shader reads.

#include <DirectXMath.h>
using namespace DirectX;

// XMFLOAT3 is just storage: three floats, no math built in.
XMFLOAT3 positionStorage { 3.0f, 0.0f, 4.0f };

// Load it into an XMVECTOR for fast SIMD math, then store the result back.
XMVECTOR v      = XMLoadFloat3(&positionStorage);
XMVECTOR unit   = XMVector3Normalize(v);          // hardware-accelerated normalize
XMFLOAT3 result;
XMStoreFloat3(&result, unit);

The pattern is the same math you just wrote by hand — the library only adds a load/store step so it can use fast SIMD instructions under the hood. Because the concepts are identical, everything you learned here transfers directly. Don't worry about memorizing this now; it's just a signpost for where we're headed.

Storage vs. math types

DirectXMath splits these on purpose: XMFLOAT3 is compact storage, while XMVECTOR is the math type your CPU processes quickly. You load, compute, then store. Keep them straight and the API stops feeling mysterious.

A preview of Apple simd

On macOS, the SDK provides simd_float3 for compact vector math and functions such as simd_length and simd_normalize. Our engine's Vector3 wrapper will expose the same API regardless of whether DirectXMath or Apple simd performs the calculation.

#include <simd/simd.h>

simd_float3 position{1.0f, 2.0f, 3.0f};
simd_float3 direction = simd_normalize(position);
Recap
  • A vector is three components (x, y, z) that read as either a position or a direction.
  • Addition combines movements; subtraction (B - A) gives the direction from A to B; scalar multiply changes length without changing direction.
  • Length is sqrt(x*x + y*y + z*z); compare squared lengths to avoid the costly square root.
  • Normalizing divides by length to make a unit vector — pure direction — and you must guard against the zero vector.
  • Our Vector3 class already implements all of this, and it mirrors DirectXMath's XMFLOAT3 / XMVECTOR that we'll adopt later.
