Meshes & Loading Models

Every 3D object your engine draws — a crate, a character, a planet — is ultimately a bag of triangles. In this lesson we wrap that geometry in a clean Mesh class that owns its GPU buffers, knows how to upload itself, and knows how to draw itself. This is one of the first real engine building blocks, and it sits right underneath the MeshRenderer component we'll assemble a little later.

What you'll learn
  • What a vertex, a vertex buffer, and an index buffer actually are.
  • How to design a Mesh class that owns a vertex buffer, an index buffer, and an index count.
  • How to implement Upload(device) and Draw(context) with real Direct3D 11 calls.
  • How to build a cube procedurally with a small factory function.
  • Where real model files (OBJ, glTF) fit in, and how MeshRenderer will use all of this.

What is a mesh?

A mesh is a list of points in space (vertices) plus a recipe for connecting them into triangles (indices). Each vertex carries more than just a position: it also stores a normal (which way the surface faces, used for lighting) and a UV coordinate (how a texture wraps onto it). We'll describe a single vertex with a plain struct built from the math types from the previous lesson's math library work. 

#include <DirectXMath.h>

// One vertex of a mesh: where it is, which way it faces, and
// how a texture maps onto it. This is the exact data our shaders read.
struct Vertex {
    DirectX::XMFLOAT3 position; // model-space position (x, y, z)
    DirectX::XMFLOAT3 normal;   // surface direction, used for lighting
    DirectX::XMFLOAT2 uv;       // texture coordinates (u, v)
};

Why bother with a separate index buffer? Because vertices are expensive and shared. A cube corner is touched by three faces; without indices we'd store that corner three times. Indices let many triangles reuse the same vertex, so we store the geometry once and just reference it by number. A cube ends up as a handful of vertices and a short list of integers.

Designing the Mesh class

Our Mesh will hold the CPU-side data (a std::vector of vertices and one of indices) plus the two GPU buffers it creates. We use Microsoft::WRL::ComPtr — a smart pointer for COM objects — so the Direct3D buffers are released automatically when the mesh is destroyed. No manual Release(), no leaks. 

#include <d3d11.h>
#include <wrl/client.h>
#include <DirectXMath.h>
#include <vector>
#include <cstdint>

using Microsoft::WRL::ComPtr;

class Mesh {
public:
    // Build the mesh on the CPU. Nothing touches the GPU yet.
    Mesh(std::vector<Vertex> vertices, std::vector<uint32_t> indices)
        : m_vertices(std::move(vertices)), m_indices(std::move(indices)) {}

    // Create the GPU buffers. Call once, after we have a device.
    bool Upload(ID3D11Device* device);

    // Bind our buffers and issue the draw call for this frame.
    void Draw(ID3D11DeviceContext* context) const;

    uint32_t IndexCount() const { return m_indexCount; }

private:
    std::vector<Vertex>   m_vertices; // CPU copy of the geometry
    std::vector<uint32_t> m_indices;  // CPU copy of the triangle list

    ComPtr<ID3D11Buffer> m_vertexBuffer; // GPU vertex data
    ComPtr<ID3D11Buffer> m_indexBuffer;  // GPU index data
    uint32_t            m_indexCount = 0; // how many indices to draw
};
Split construction from upload

The constructor only fills vectors, so a mesh can be created before the graphics device even exists — handy for loading on a background thread. Upload() is the single point where the GPU gets involved.

Uploading to the GPU

To draw geometry, the data has to live in GPU memory as Direct3D buffers. We describe each buffer with a D3D11_BUFFER_DESC, point the driver at our vector's raw bytes with a D3D11_SUBRESOURCE_DATA, and call CreateBuffer. Because our cube never changes, we mark the buffers D3D11_USAGE_IMMUTABLE — the fastest option, since the GPU knows the contents are read-only. 

bool Mesh::Upload(ID3D11Device* device) {
    if (m_vertices.empty() || m_indices.empty())
        return false;

    // ---- Vertex buffer ----
    D3D11_BUFFER_DESC vbDesc = {};
    vbDesc.ByteWidth = static_cast<UINT>(m_vertices.size() * sizeof(Vertex));
    vbDesc.Usage     = D3D11_USAGE_IMMUTABLE;      // GPU-only, never changes
    vbDesc.BindFlags = D3D11_BIND_VERTEX_BUFFER;

    D3D11_SUBRESOURCE_DATA vbData = {};
    vbData.pSysMem = m_vertices.data();            // source bytes to copy

    HRESULT hr = device->CreateBuffer(&vbDesc, &vbData, &m_vertexBuffer);
    if (FAILED(hr)) return false;

    // ---- Index buffer ----
    D3D11_BUFFER_DESC ibDesc = {};
    ibDesc.ByteWidth = static_cast<UINT>(m_indices.size() * sizeof(uint32_t));
    ibDesc.Usage     = D3D11_USAGE_IMMUTABLE;
    ibDesc.BindFlags = D3D11_BIND_INDEX_BUFFER;

    D3D11_SUBRESOURCE_DATA ibData = {};
    ibData.pSysMem = m_indices.data();

    hr = device->CreateBuffer(&ibDesc, &ibData, &m_indexBuffer);
    if (FAILED(hr)) return false;

    m_indexCount = static_cast<uint32_t>(m_indices.size());
    return true; // buffers now live on the GPU
}
Always check your HRESULTs

Almost every Direct3D call returns an HRESULT. Ignoring it means a failed buffer creation shows up later as a mysterious blank screen. Wrapping the check in FAILED(hr) and bailing early saves hours of debugging.

Drawing the mesh

Drawing is a three-step ritual on the input assembler: bind the vertex buffer, bind the index buffer, and tell Direct3D that the indices form a triangle list. Then a single DrawIndexed submits every triangle. Note the stride — the size of one vertex — which tells the GPU how far to step between vertices. 

void Mesh::Draw(ID3D11DeviceContext* context) const {
    const UINT stride = sizeof(Vertex); // bytes per vertex
    const UINT offset = 0;

    ID3D11Buffer* vb = m_vertexBuffer.Get();
    context->IASetVertexBuffers(0, 1, &vb, &stride, &offset);

    // 32-bit indices match our uint32_t vector.
    context->IASetIndexBuffer(m_indexBuffer.Get(), DXGI_FORMAT_R32_UINT, 0);
    context->IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST);

    context->DrawIndexed(m_indexCount, 0, 0); // one call, every triangle
}

The mesh does not bind shaders, textures, or the camera — that is deliberate. A mesh only knows about geometry. Everything else is the caller's job, which keeps this class small and reusable. The next lesson's material and shader system will take care of the rest.

A cube factory

Rather than hand-authoring a model file just to see something on screen, we can generate simple shapes in code. A cube needs 24 vertices, not 8: each of the six faces gets its own four corners so every face can have a correct, flat normal and its own texture mapping. Sharing eight corners would smear the normals and break lighting. 

// Build a unit-ish cube centered on the origin, edge length = size.
Mesh CreateCube(float size = 1.0f) {
    const float h = size * 0.5f; // half-extent

    // 4 vertices per face so each face keeps its own normal & uv.
    std::vector<Vertex> vertices = {
        // +X face
        {{ h,-h,-h}, { 1, 0, 0}, {0,1}}, {{ h, h,-h}, { 1, 0, 0}, {0,0}},
        {{ h, h, h}, { 1, 0, 0}, {1,0}}, {{ h,-h, h}, { 1, 0, 0}, {1,1}},
        // -X face
        {{-h,-h, h}, {-1, 0, 0}, {0,1}}, {{-h, h, h}, {-1, 0, 0}, {0,0}},
        {{-h, h,-h}, {-1, 0, 0}, {1,0}}, {{-h,-h,-h}, {-1, 0, 0}, {1,1}},
        // +Y face
        {{-h, h,-h}, { 0, 1, 0}, {0,1}}, {{-h, h, h}, { 0, 1, 0}, {0,0}},
        {{ h, h, h}, { 0, 1, 0}, {1,0}}, {{ h, h,-h}, { 0, 1, 0}, {1,1}},
        // -Y face
        {{-h,-h, h}, { 0,-1, 0}, {0,1}}, {{-h,-h,-h}, { 0,-1, 0}, {0,0}},
        {{ h,-h,-h}, { 0,-1, 0}, {1,0}}, {{ h,-h, h}, { 0,-1, 0}, {1,1}},
        // +Z face
        {{ h,-h, h}, { 0, 0, 1}, {0,1}}, {{ h, h, h}, { 0, 0, 1}, {0,0}},
        {{-h, h, h}, { 0, 0, 1}, {1,0}}, {{-h,-h, h}, { 0, 0, 1}, {1,1}},
        // -Z face
        {{-h,-h,-h}, { 0, 0,-1}, {0,1}}, {{-h, h,-h}, { 0, 0,-1}, {0,0}},
        {{ h, h,-h}, { 0, 0,-1}, {1,0}}, {{ h,-h,-h}, { 0, 0,-1}, {1,1}},
    };

    // 6 faces * 2 triangles * 3 indices = 36 indices total.
    std::vector<uint32_t> indices;
    indices.reserve(36);
    for (uint32_t f = 0; f < 6; ++f) {
        const uint32_t base = f * 4;
        indices.insert(indices.end(), {
            base + 0, base + 1, base + 2, // first triangle
            base + 0, base + 2, base + 3  // second triangle
        });
    }

    return Mesh(std::move(vertices), std::move(indices));
}
Winding order decides what's a "front" face

The order the three indices of a triangle are listed determines whether it faces toward or away from the camera. With back-face culling on, if your cube looks inside-out or invisible, swap two indices per triangle to flip the winding. It's the most common first-cube surprise.

Putting it together

With a device and a device context already created during startup, using the mesh is short. Build it once, upload once, and draw it every frame after your shaders and camera constants are bound. 

// --- setup, once ---
Mesh cube = CreateCube(1.0f);
if (!cube.Upload(device.Get())) {
    // Log and handle the failure; don't draw an empty mesh.
}

// --- render loop, every frame ---
// (bind vertex/pixel shaders, input layout, and camera constants first)
cube.Draw(context.Get());

Uploading and drawing with Metal on macOS

The CPU-side Vertex and index vectors are shared. The Metal backend copies those bytes into MTLBuffer objects and draws the same indexed triangle list:

m_vertexBuffer = device->newBuffer(
    vertices.data(), vertices.size() * sizeof(Vertex), MTL::ResourceStorageModeShared);
m_indexBuffer = device->newBuffer(
    indices.data(), indices.size() * sizeof(uint32_t), MTL::ResourceStorageModeShared);

encoder->setVertexBuffer(m_vertexBuffer, 0, 0);
encoder->drawIndexedPrimitives(
    MTL::PrimitiveTypeTriangle,
    indices.size(),
    MTL::IndexTypeUInt32,
    m_indexBuffer,
    0);

Keep vertices, indices, bounds, and model loading in the shared Mesh. Put ID3D11Buffer and MTL::Buffer ownership in their respective GPU-mesh backends.

From cubes to real models

Procedural shapes are perfect for testing, but eventually you'll want art made in Blender or Maya. The good news: nothing about Mesh changes. Loading a model just means filling the same two vectors from a file instead of from a loop.

  • OBJ is a plain-text format — great to parse by hand as a learning exercise. You read positions, normals, and UVs, then assemble Vertex entries and indices.
  • glTF is the modern standard for real-time engines and maps cleanly onto GPU buffers.
  • Assimp (the Open Asset Import Library) reads dozens of formats and hands you vertex and index arrays ready to drop straight into CreateBuffer.

A real loader will return a std::vector<Mesh> — a single file often contains many sub-meshes, each with a different material. That's a natural bridge to the next lesson.

Foreshadowing the MeshRenderer component

A Mesh is pure geometry: it has no position in the world and no idea what color it should be. In a component-based engine — the same idea behind Unity's MeshRenderer or Unreal's static mesh component — a game object will own a Transform (from our math library) and a MeshRenderer component. That renderer will hold a pointer to a shared Mesh plus a material, and each frame it will feed the transform into the shader and call mesh->Draw(context). One mesh, many renderers, many positions — that's how you get a hundred crates on screen from a single set of GPU buffers.

Recap
  • A mesh is a list of vertices (position, normal, UV) plus indices that connect them into triangles.
  • The Mesh class owns a vertex buffer, an index buffer, and an index count, using ComPtr for automatic cleanup.
  • The backend uploads the shared vertex/index data: D3D11 buffers on Windows or MTLBuffer objects on macOS.
  • A cube needs 24 vertices (4 per face) and 36 indices; a factory function generates it in code — and winding order controls which side faces out.
  • Real models (OBJ, glTF, Assimp) fill the same two vectors, and a MeshRenderer component will pair a shared mesh with a Transform and material.
