A Material & Shader System

Last lesson we loaded real meshes into vertex and index buffers. But a mesh is just shape — it says nothing about colour, texture, or lighting. In every serious engine (Unity, Unreal, our own) the how it looks is a separate object from the what it is. In this lesson we build two small classes, Shader and Material, that give us exactly that separation and set us up for a real render loop.

What you'll learn
  • Why an engine keeps geometry separate from shading, and what a "material" really is.
  • How to wrap a compiled vertex shader, pixel shader, and input layout in one reusable Shader class.
  • How a Material stores parameters (a colour, a texture) and uploads them to a constant buffer.
  • How Bind() arms the whole pipeline before a mesh's draw call.

Why separate the mesh from how it's drawn

A cube mesh is a fixed list of positions, normals and texture coordinates. That same cube might appear a hundred times in a scene: one glowing red, one wrapped in a wooden crate texture, one tinted blue. If shading lived inside the mesh, we'd need a hundred copies of the same geometry. That's wasteful and it's the wrong mental model.

Instead we split responsibilities. A Shader is the program that runs on the GPU — a vertex shader plus a pixel shader plus the input layout that describes our vertex format. A Material picks a shader and fills in its knobs: a base colour, a texture, later on roughness or metalness. Many materials can share one shader; many objects can share one material. The mesh, meanwhile, knows nothing about any of this — it just supplies geometry.

The Shader class

Back in the shaders and constant-buffers lessons we compiled HLSL by hand and called VSSetShader / PSSetShader inline. Let's package that up. The class holds three COM objects and exposes a single Bind() that sets them all on the device context. We use Microsoft::WRL::ComPtr so the GPU resources release themselves automatically. 

#pragma once
#include <d3d11.h>
#include <d3dcompiler.h>
#include <wrl/client.h>          // Microsoft::WRL::ComPtr
#include <string>

using Microsoft::WRL::ComPtr;

// Bundles a compiled vertex shader, pixel shader and the input
// layout that describes our vertex format. One Bind() arms the pipeline.
class Shader
{
public:
    bool Load(ID3D11Device* device,
              const std::wstring& vsPath,
              const std::wstring& psPath,
              const D3D11_INPUT_ELEMENT_DESC* layout,
              UINT layoutCount);

    void Bind(ID3D11DeviceContext* context) const;

private:
    ComPtr<ID3D11VertexShader> m_vs;
    ComPtr<ID3D11PixelShader>  m_ps;
    ComPtr<ID3D11InputLayout>  m_inputLayout;
};

The implementation compiles both HLSL files, creates the shader objects, and — importantly — builds the input layout from the vertex shader's bytecode so Direct3D can verify our vertex format matches what the shader expects. 

#include "Shader.h"

// Helper: compile one .hlsl file to GPU bytecode.
static bool CompileFile(const std::wstring& path, const char* entry,
                        const char* target, ComPtr<ID3DBlob>& outBlob)
{
    ComPtr<ID3DBlob> errors;
    HRESULT hr = D3DCompileFromFile(
        path.c_str(), nullptr, D3D_COMPILE_STANDARD_FILE_INCLUDE,
        entry, target, D3DCOMPILE_ENABLE_STRICTNESS, 0,
        &outBlob, &errors);

    if (FAILED(hr)) {
        if (errors)   // the compiler writes human-readable messages here
            OutputDebugStringA((const char*)errors->GetBufferPointer());
        return false;
    }
    return true;
}

bool Shader::Load(ID3D11Device* device,
                  const std::wstring& vsPath,
                  const std::wstring& psPath,
                  const D3D11_INPUT_ELEMENT_DESC* layout,
                  UINT layoutCount)
{
    ComPtr<ID3DBlob> vsBlob, psBlob;
    if (!CompileFile(vsPath, "main", "vs_5_0", vsBlob)) return false;
    if (!CompileFile(psPath, "main", "ps_5_0", psBlob)) return false;

    // Create the shader objects from their compiled bytecode.
    if (FAILED(device->CreateVertexShader(
            vsBlob->GetBufferPointer(), vsBlob->GetBufferSize(),
            nullptr, &m_vs)))
        return false;

    if (FAILED(device->CreatePixelShader(
            psBlob->GetBufferPointer(), psBlob->GetBufferSize(),
            nullptr, &m_ps)))
        return false;

    // The input layout is validated against the vertex-shader signature.
    if (FAILED(device->CreateInputLayout(
            layout, layoutCount,
            vsBlob->GetBufferPointer(), vsBlob->GetBufferSize(),
            &m_inputLayout)))
        return false;

    return true;
}

void Shader::Bind(ID3D11DeviceContext* context) const
{
    context->IASetInputLayout(m_inputLayout.Get());
    context->VSSetShader(m_vs.Get(), nullptr, 0);
    context->PSSetShader(m_ps.Get(), nullptr, 0);
}
Compile once, bind many

Compiling HLSL is slow, so call Load() during start-up, never inside the render loop. Bind() is the cheap per-frame call — it just points the pipeline at objects that already exist.

The Material class

A material owns a shared pointer to a Shader plus the data that customises it. We keep a CPU-side struct, Constants, that mirrors a GPU constant buffer exactly, along with an optional texture (a shader-resource view, or SRV) and a sampler. Using std::shared_ptr<Shader> lets dozens of materials reference the same compiled program without owning or duplicating it. 

#pragma once
#include <d3d11.h>
#include <wrl/client.h>
#include <DirectXMath.h>
#include <memory>
#include "Shader.h"

using Microsoft::WRL::ComPtr;

// A Material = a Shader + the parameters that make one surface look
// different from another (a tint colour and an optional texture).
class Material
{
public:
    // Mirrored in a GPU constant buffer. Keep it 16-byte aligned to
    // match HLSL's packing rules (one float4 is exactly 16 bytes).
    struct Constants
    {
        DirectX::XMFLOAT4 baseColor{ 1.0f, 1.0f, 1.0f, 1.0f };
    };

    bool Create(ID3D11Device* device, std::shared_ptr<Shader> shader);

    void SetColor(const DirectX::XMFLOAT4& c)            { m_constants.baseColor = c; }
    void SetTexture(ComPtr<ID3D11ShaderResourceView> t)  { m_texture = t; }

    // Arm shader, upload params, bind texture + sampler. Call before Draw.
    void Bind(ID3D11DeviceContext* context);

private:
    std::shared_ptr<Shader>           m_shader;
    Constants                        m_constants;
    ComPtr<ID3D11Buffer>             m_cbuffer;
    ComPtr<ID3D11ShaderResourceView> m_texture;
    ComPtr<ID3D11ShaderResourceView> m_defaultTex;   // 1x1 white fallback
    ComPtr<ID3D11SamplerState>       m_sampler;
};

Create() allocates a dynamic constant buffer (so we can rewrite it every frame) and a sampler. Bind() is where the magic happens: it arms the shader, copies our parameters into the GPU buffer, and binds both the buffer and the texture before we draw. 

#include "Material.h"

bool Material::Create(ID3D11Device* device, std::shared_ptr<Shader> shader)
{
    m_shader = std::move(shader);

    // Dynamic constant buffer so we can rewrite it each frame with Map().
    D3D11_BUFFER_DESC cbd = {};
    cbd.ByteWidth      = sizeof(Constants);   // must be a multiple of 16
    cbd.Usage          = D3D11_USAGE_DYNAMIC;
    cbd.BindFlags      = D3D11_BIND_CONSTANT_BUFFER;
    cbd.CPUAccessFlags = D3D11_CPU_ACCESS_WRITE;
    if (FAILED(device->CreateBuffer(&cbd, nullptr, &m_cbuffer)))
        return false;

    // A linear, wrapping sampler shared by this material's textures.
    D3D11_SAMPLER_DESC sd = {};
    sd.Filter         = D3D11_FILTER_MIN_MAG_MIP_LINEAR;
    sd.AddressU       = D3D11_TEXTURE_ADDRESS_WRAP;
    sd.AddressV       = D3D11_TEXTURE_ADDRESS_WRAP;
    sd.AddressW       = D3D11_TEXTURE_ADDRESS_WRAP;
    sd.ComparisonFunc = D3D11_COMPARISON_NEVER;
    sd.MaxLOD         = D3D11_FLOAT32_MAX;
    if (FAILED(device->CreateSamplerState(&sd, &m_sampler)))
        return false;

    // A 1x1 opaque-white texture used when a material has no texture of its
    // own. Sampling white means texel * baseColor leaves the colour untouched.
    const UINT white = 0xFFFFFFFF;
    D3D11_TEXTURE2D_DESC td = {};
    td.Width            = td.Height = 1;
    td.MipLevels        = td.ArraySize = 1;
    td.Format           = DXGI_FORMAT_R8G8B8A8_UNORM;
    td.SampleDesc.Count = 1;
    td.Usage            = D3D11_USAGE_IMMUTABLE;
    td.BindFlags        = D3D11_BIND_SHADER_RESOURCE;
    D3D11_SUBRESOURCE_DATA srd = {};
    srd.pSysMem     = &white;
    srd.SysMemPitch = sizeof(white);
    ComPtr<ID3D11Texture2D> tex;
    if (FAILED(device->CreateTexture2D(&td, &srd, &tex)))
        return false;
    if (FAILED(device->CreateShaderResourceView(tex.Get(), nullptr, &m_defaultTex)))
        return false;

    return true;
}

void Material::Bind(ID3D11DeviceContext* context)
{
    // 1) Activate the pipeline program (VS + PS + input layout).
    m_shader->Bind(context);

    // 2) Push the latest CPU-side parameters into the GPU buffer.
    D3D11_MAPPED_SUBRESOURCE mapped;
    if (SUCCEEDED(context->Map(m_cbuffer.Get(), 0,
            D3D11_MAP_WRITE_DISCARD, 0, &mapped)))
    {
        memcpy(mapped.pData, &m_constants, sizeof(Constants));
        context->Unmap(m_cbuffer.Get(), 0);
    }

    // 3) Bind material params to slot b1 (b0 stays free for the per-object
    //    transform we'll add with the Transform component next lesson).
    ID3D11Buffer* cbs[] = { m_cbuffer.Get() };
    context->PSSetConstantBuffers(1, 1, cbs);

    // 4) Bind the texture + sampler. Fall back to the 1x1 white texture so an
    //    untextured material samples white and shows its baseColor alone.
    ID3D11ShaderResourceView* srvs[]  = { m_texture ? m_texture.Get() : m_defaultTex.Get() };
    ID3D11SamplerState*       samps[] = { m_sampler.Get() };
    context->PSSetShaderResources(0, 1, srvs);
    context->PSSetSamplers(0, 1, samps);
}
Constant buffers must be 16-byte aligned

Direct3D requires a constant buffer's size to be a multiple of 16 bytes. A single float4 is exactly 16, so we're fine here — but the moment you add a lone float, pad the struct up to the next multiple of 16 or your data will read back scrambled.

The matching pixel shader

The CPU struct and the HLSL cbuffer must agree on layout and register. We bind our material data to slot b1, sample the texture from t0, and multiply the two so the colour tints the texture. Sample a white pixel if you want the texture untouched, or use the colour alone when there's no texture bound. 

// Per-material parameters — matches Material::Constants on the CPU side.
cbuffer MaterialConstants : register(b1)
{
    float4 baseColor;
};

Texture2D    albedo : register(t0);
SamplerState samp   : register(s0);

struct PSInput
{
    float4 position : SV_POSITION;
    float2 uv       : TEXCOORD0;
};

float4 main(PSInput input) : SV_TARGET
{
    float4 texel = albedo.Sample(samp, input.uv);
    return texel * baseColor;    // tint the sampled texture
}
Register slots are a contract

The register(b1), register(t0) and register(s0) numbers in HLSL must line up with the slot numbers you pass to PSSetConstantBuffers, PSSetShaderResources and PSSetSamplers. We reserve b0 for transforms so materials never collide with per-object data.

Putting it together

Now the payoff. One shader backs two materials, and one mesh is drawn twice with completely different looks. Notice the mesh's Draw() is identical both times — it has no idea a material exists. 

// One shared shader can back any number of materials.
auto litShader = std::make_shared<Shader>();
litShader->Load(device, L"Lit_VS.hlsl", L"Lit_PS.hlsl",
                layout, _countof(layout));

// Two materials that shade the same geometry differently.
Material red, crate;
red.Create(device, litShader);
red.SetColor({ 1.0f, 0.2f, 0.2f, 1.0f });   // plain red tint

crate.Create(device, litShader);
crate.SetTexture(crateSRV);                  // textured, white tint

// The mesh knows nothing about colour or textures.
red.Bind(context);
cubeMesh.Draw(context);

crate.Bind(context);
cubeMesh.Draw(context);   // same mesh, a completely different look

This is the exact pattern Unity's Material and Unreal's material instances use under the hood: a shared program, per-instance parameters, bound just before the draw call. With shape and shading now cleanly separated, the last missing piece is where each object sits in the world — its position, rotation and scale. That's the Transform component, and it's next.

Materials with Metal on macOS

The shared Material still stores a color, texture reference, and shader reference. The Metal backend replaces the D3D objects with an MTL::RenderPipelineState, MTL::Texture, MTL::SamplerState, and parameter buffer.

struct MaterialConstants { float4 baseColor; };

fragment float4 fragmentMain(
    VertexOut in [[stage_in]],
    constant MaterialConstants& material [[buffer(1)]],
    texture2d<float> colorMap [[texture(0)]],
    sampler linearSampler [[sampler(0)]]) {
    return colorMap.sample(linearSampler, in.uv) * material.baseColor;
}

Recap
  • An engine separates geometry (the mesh) from appearance (shader + material) so one mesh can be drawn many ways.
  • A Shader bundles a vertex shader, pixel shader and input layout; its Bind() arms all three in one call.
  • A Material references a shared shader and stores parameters — a colour and an optional texture SRV.
  • Material Bind() maps parameters into a 16-byte-aligned constant buffer at slot b1, then binds the texture and sampler.
  • Resource slots must match in C++ and the shader: HLSL registers on Windows and Metal buffer/texture/sampler indices on macOS.
