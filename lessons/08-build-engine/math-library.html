Building a Small Math Library

Every engine runs on math: positions, rotations, and the matrices that turn a 3D model into pixels. Now we will hide the platform SIMD library behind friendly Vector3 and Matrix types. The public API stays identical while DirectXMath does the work on Windows and Apple simd does it on macOS.

What you'll learn
  • Why we wrap DirectXMath and Apple simd instead of exposing either library everywhere
  • How to build a header-only Vector3 with operators and helpers like Length and Normalize
  • How to build a Matrix type with factory helpers: Translation, Rotation, Scale, LookAt, and Perspective
  • How to combine them to build a world matrix — the exact thing our future Transform component needs

Why not just use DirectXMath directly?

DirectXMath is fast and correct, but it was designed for performance, not for comfort. It splits its types into two families: storage types like XMFLOAT3 that hold plain floats, and calculation types like XMVECTOR that live in SIMD registers. You constantly convert between them with functions like XMLoadFloat3 and XMStoreFloat3. That's great for a tight inner loop, but it makes everyday gameplay code noisy and hard to read.

Look at what adding two positions costs in raw DirectXMath:

// The verbose reality of raw DirectXMath for a simple add
XMFLOAT3 a{1, 2, 3};
XMFLOAT3 b{4, 5, 6};

XMVECTOR va = XMLoadFloat3(&a);   // load into SIMD register
XMVECTOR vb = XMLoadFloat3(&b);
XMVECTOR vr = XMVectorAdd(va, vb); // do the math

XMFLOAT3 result;
XMStoreFloat3(&result, vr);        // store back to memory

Five lines to add two vectors. Multiply that across an entire game and you drown in boilerplate. What we want is Vector3 result = a + b;. So we'll build a small wrapper layer that gives us that ergonomics while still using DirectXMath under the hood for the actual arithmetic.

The wrapper strategy

Store data as the simple XMFLOAT types (so a Vector3 is just three floats you can put in a component). Load into XMVECTOR only for the duration of a single operation, then store the result back. The conversions become an implementation detail your game code never sees.

A header-only Vector3

Because these types are small and used everywhere, we make the library header-only: everything lives in .h files with functions marked inline. There's no .cpp to compile separately, and the compiler can inline the tiny operations away completely. Here's a compact but complete Vector3.

#pragma once
#include <DirectXMath.h>

namespace engine {

// A friendly 3D vector. Stores plain floats so it can live inside
// components, yet delegates real math to DirectXMath.
struct Vector3 {
    float x = 0.0f;
    float y = 0.0f;
    float z = 0.0f;

    Vector3() = default;
    Vector3(float x, float y, float z) : x(x), y(y), z(z) {}

    // Convert to/from the DirectXMath SIMD vector. Kept private-ish
    // by convention: game code should not need these.
    DirectX::XMVECTOR Load() const {
        DirectX::XMFLOAT3 f{ x, y, z };
        return DirectX::XMLoadFloat3(&f);
    }
    static Vector3 Store(DirectX::XMVECTOR v) {
        DirectX::XMFLOAT3 f;
        DirectX::XMStoreFloat3(&f, v);
        return Vector3{ f.x, f.y, f.z };
    }

    float Length() const {
        return DirectX::XMVectorGetX(DirectX::XMVector3Length(Load()));
    }

    Vector3 Normalized() const {
        return Store(DirectX::XMVector3Normalize(Load()));
    }
};

// --- Operators: this is the whole point of the wrapper ---
inline Vector3 operator+(const Vector3& a, const Vector3& b) {
    return { a.x + b.x, a.y + b.y, a.z + b.z };
}
inline Vector3 operator-(const Vector3& a, const Vector3& b) {
    return { a.x - b.x, a.y - b.y, a.z - b.z };
}
inline Vector3 operator*(const Vector3& v, float s) {
    return { v.x * s, v.y * s, v.z * s };
}

// Dot and cross products delegate to DirectXMath.
inline float Dot(const Vector3& a, const Vector3& b) {
    return DirectX::XMVectorGetX(DirectX::XMVector3Dot(a.Load(), b.Load()));
}
inline Vector3 Cross(const Vector3& a, const Vector3& b) {
    return Vector3::Store(DirectX::XMVector3Cross(a.Load(), b.Load()));
}

} // namespace engine

Notice the simple operations like operator+ just add floats directly — no need to bounce through SIMD for three additions. We only reach for DirectXMath when the operation genuinely benefits, like XMVector3Normalize, which does a square root and division. This keeps things both fast and readable.

Left-handed, row-major

DirectXMath uses a left-handed coordinate system (the +Z axis points away from you into the screen) and row-major matrices, where you multiply as world = scale * rotation * translation. We'll stay consistent with those conventions throughout the engine so nothing gets mirrored or flipped.

A Matrix type with factory helpers

Now the matrix. A Matrix is a 4×4 grid of floats used to transform points. Rather than wrapping every operation, we give it a clean storage member and a set of static factory functions that build the common transforms an engine needs. This is the ergonomic sweet spot: Matrix::Translation(pos) reads far better than a bare XMMatrixTranslation call buried in game logic.

#pragma once
#include <DirectXMath.h>
#include "Vector3.h"

namespace engine {

// A 4x4 transform matrix. Stores an XMFLOAT4X4 (16 plain floats)
// and exposes factory helpers plus multiplication.
struct Matrix {
    DirectX::XMFLOAT4X4 m;

    Matrix() { DirectX::XMStoreFloat4x4(&m, DirectX::XMMatrixIdentity()); }
    explicit Matrix(DirectX::FXMMATRIX xm) { DirectX::XMStoreFloat4x4(&m, xm); }

    DirectX::XMMATRIX Load() const { return DirectX::XMLoadFloat4x4(&m); }

    // --- Factory helpers: the engine's public vocabulary ---
    static Matrix Identity() { return Matrix{}; }

    static Matrix Translation(const Vector3& t) {
        return Matrix{ DirectX::XMMatrixTranslation(t.x, t.y, t.z) };
    }

    static Matrix Scale(const Vector3& s) {
        return Matrix{ DirectX::XMMatrixScaling(s.x, s.y, s.z) };
    }

    // Rotation from Euler angles given in radians (pitch, yaw, roll).
    static Matrix Rotation(const Vector3& eulerRadians) {
        return Matrix{ DirectX::XMMatrixRotationRollPitchYaw(
            eulerRadians.x, eulerRadians.y, eulerRadians.z) };
    }

    // A view matrix looking from 'eye' toward 'target'.
    static Matrix LookAt(const Vector3& eye, const Vector3& target,
                         const Vector3& up) {
        return Matrix{ DirectX::XMMatrixLookAtLH(
            eye.Load(), target.Load(), up.Load()) };
    }

    // A perspective projection. fovY in radians.
    static Matrix Perspective(float fovY, float aspect,
                              float nearZ, float farZ) {
        return Matrix{ DirectX::XMMatrixPerspectiveFovLH(
            fovY, aspect, nearZ, farZ) };
    }
};

// Matrix multiply. Row-major order: 'a' is applied before 'b'.
inline Matrix operator*(const Matrix& a, const Matrix& b) {
    return Matrix{ DirectX::XMMatrixMultiply(a.Load(), b.Load()) };
}

// Transform a position (a point, so w = 1) by a matrix.
inline Vector3 TransformPoint(const Matrix& m, const Vector3& p) {
    return Vector3::Store(
        DirectX::XMVector3TransformCoord(p.Load(), m.Load()));
}

} // namespace engine

The explicit Matrix(FXMMATRIX) constructor lets our factories hand back an XMMATRIX result and have it stored automatically. The Load() method does the reverse for operations. Everything else — Translation, LookAt, Perspective — is a one-line, self-documenting wrapper around the correct DirectXMath call.

Radians, not degrees

Every DirectXMath angle is in radians. A 90-degree field of view is 90.0f * XM_PI / 180.0f, not 90.0f. Passing degrees is one of the most common beginner bugs — your camera ends up zoomed to an extreme or your objects spin far too fast. Consider adding a tiny ToRadians helper to catch yourself.

Putting it together: building a world matrix

Here's the payoff. Every object in a 3D scene has a world matrix that places it in the world: first scale it, then rotate it, then move it into position. With our library, that whole idea becomes three readable lines — and this is exactly the calculation our Transform component will do next lesson.

#include <cstdio>
#include "Matrix.h"

using namespace engine;

int main() {
    // An object's transform, expressed in friendly engine types.
    Vector3 position{ 5.0f, 0.0f, 10.0f };
    Vector3 rotation{ 0.0f, DirectX::XM_PIDIV4, 0.0f }; // 45 deg yaw
    Vector3 scale{ 2.0f, 2.0f, 2.0f };

    // Build the world matrix: scale, then rotate, then translate.
    Matrix world = Matrix::Scale(scale)
                 * Matrix::Rotation(rotation)
                 * Matrix::Translation(position);

    // Where does the object's local origin end up in world space?
    Vector3 worldOrigin = TransformPoint(world, Vector3{ 0, 0, 0 });
    std::printf("World origin: (%.2f, %.2f, %.2f)\n",
                worldOrigin.x, worldOrigin.y, worldOrigin.z);

    // A camera looking at the object, and a projection matrix.
    Matrix view = Matrix::LookAt(
        Vector3{ 0, 3, -8 },   // eye
        position,              // look at the object
        Vector3{ 0, 1, 0 });   // up

    float aspect = 1280.0f / 720.0f;
    Matrix proj = Matrix::Perspective(DirectX::XM_PIDIV4, aspect, 0.1f, 100.0f);

    // The combined transform a vertex shader will use.
    Matrix worldViewProj = world * view * proj;
    (void)worldViewProj; // ready to upload to a constant buffer

    return 0;
}

The game code above never once mentions XMVECTOR, XMLoadFloat3, or XMMatrixMultiply. It speaks in Vector3, Matrix, Scale, Rotation, and LookAt — a vocabulary any beginner can read. That worldViewProj matrix at the end is precisely what we'll pack into a constant buffer and send to a shader, keeping the boundary between "engine math" and "GPU details" crisp.

The Apple simd backend on macOS

Keep the same public Vector3 and Matrix API, but implement its calculations with the SIMD types included in Apple's SDK. For example:

#include <simd/simd.h>

simd_float3 ToNative(const Vector3& v) {
    return {v.x, v.y, v.z};
}

Vector3 Normalize(const Vector3& v) {
    simd_float3 result = simd_normalize(ToNative(v));
    return {result.x, result.y, result.z};
}

float Dot(const Vector3& a, const Vector3& b) {
    return simd_dot(ToNative(a), ToNative(b));
}

Vector3 Cross(const Vector3& a, const Vector3& b) {
    simd_float3 result = simd_cross(ToNative(a), ToNative(b));
    return {result.x, result.y, result.z};
}

Choose one convention

DirectXMath examples here use row vectors; Metal examples commonly use column vectors. Define the engine's handedness, matrix storage, multiplication order, and shader-upload transpose in one backend boundary. Do not mix conventions inside gameplay code.

Keep it thin

Resist the urge to reimplement matrix multiplication or normalization by hand "to understand it." DirectXMath is already tested, SIMD-accelerated, and correct. Our job is only to give it friendlier names. A thin wrapper is a feature, not a shortcut.

Recap
  • Raw DirectXMath is fast but verbose because it splits storage types (XMFLOAT3) from SIMD calculation types (XMVECTOR); a thin wrapper gives us clean, readable game code.
  • We built a header-only Vector3 that stores plain floats, provides +, -, *, and delegates heavier work like Normalize, Dot, and Cross to DirectXMath.
  • Our Matrix stores an XMFLOAT4X4 and exposes factory helpers — Translation, Rotation, Scale, LookAt, and Perspective — plus a * operator.
  • Combining Scale * Rotation * Translation yields a world matrix in three readable lines, with no raw XM types leaking into game code — the foundation for next lesson's Transform component.
  • The same engine API can use DirectXMath on Windows and Apple simd on macOS; backend conversion owns any transpose.
  • Remember the chosen conventions and keep angles in radians.
