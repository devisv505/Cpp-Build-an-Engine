Structuring the Engine Project

This is where the tutorial becomes a real project. We will build one reusable Engine library and one small Sandbox application with CLion and CMake. Shared gameplay code stays portable; only the graphics backend differs: Direct3D 11 on Windows and Metal on macOS.

What you'll learn
  • Split the engine into two CMake targets: a static library and an executable.
  • Keep shared code separate from Windows and macOS backends.
  • Configure platform sources and libraries with WIN32 and APPLE.
  • Use a portable precompiled header without leaking platform APIs across the engine.

Two targets, one goal

  • Engine — a static library (.lib on Windows, .a on macOS) containing reusable systems.
  • Sandbox — the executable that links Engine and contains the game-specific code.

The dependency points one way: Sandbox uses Engine; Engine never includes Sandbox. That makes it possible to build another game or tool on top of the same library.

The root CMake file

cmake_minimum_required(VERSION 3.24)
project(MyEngine LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

add_subdirectory(Engine)
add_subdirectory(Sandbox)

The Engine target

On macOS, add the same tested metal-cpp release used in Lesson 39 before configuring the new project:

git clone --depth 1 \
  --branch release/metal-cpp_macOS26_iOS26 \
  https://github.com/apple/metal-cpp.git external/metal-cpp
include(FetchContent)

# Build SDL3 into the project. A static build avoids copying SDL3.dll
# beside Sandbox on Windows.
set(SDL_SHARED OFF CACHE BOOL "" FORCE)
set(SDL_STATIC ON CACHE BOOL "" FORCE)
set(SDL_TESTS OFF CACHE BOOL "" FORCE)
set(SDL_EXAMPLES OFF CACHE BOOL "" FORCE)

FetchContent_Declare(SDL3
  GIT_REPOSITORY https://github.com/libsdl-org/SDL.git
  GIT_TAG release-3.4.12
  GIT_SHALLOW TRUE)
FetchContent_MakeAvailable(SDL3)

add_library(Engine STATIC
  src/Core/Application.cpp
  src/Platform/SDLWindow.cpp
  src/Renderer/RendererFactory.cpp)

target_include_directories(Engine PUBLIC src)
target_link_libraries(Engine PUBLIC SDL3::SDL3)

if(WIN32)
  target_sources(Engine PRIVATE src/Renderer/D3D11Renderer.cpp)
  target_link_libraries(Engine PRIVATE d3d11 dxgi d3dcompiler)
elseif(APPLE)
  if(NOT EXISTS "${PROJECT_SOURCE_DIR}/external/metal-cpp/Metal/Metal.hpp")
    message(FATAL_ERROR
      "Clone the pinned metal-cpp release into external/metal-cpp")
  endif()

  target_sources(Engine PRIVATE
    src/Renderer/MetalRenderer.cpp
    src/Renderer/MetalCpp.cpp)
  target_include_directories(Engine PRIVATE ${PROJECT_SOURCE_DIR}/external/metal-cpp)
  target_link_libraries(Engine PRIVATE
    "-framework Foundation"
    "-framework Metal"
    "-framework QuartzCore")
else()
  message(FATAL_ERROR "This course currently supports Windows and macOS")
endif()

target_precompile_headers(Engine PRIVATE src/pch.h)

SDL3::SDL3 is the same dependency on both platforms. The WIN32 and APPLE branches select only the native renderer backend: Direct3D 11 or Metal. Shared headers may forward-declare SDL_Window, but they must not expose ID3D11Device or MTL::Device.

Emit metal-cpp's private implementations in the one source file already listed by CMake:

#define NS_PRIVATE_IMPLEMENTATION
#define CA_PRIVATE_IMPLEMENTATION
#define MTL_PRIVATE_IMPLEMENTATION

#include <Foundation/Foundation.hpp>
#include <Metal/Metal.hpp>
#include <QuartzCore/QuartzCore.hpp>
Reload CMake after adding SDL3

In CLion, choose Tools → CMake → Reload CMake Project. The first configure downloads and builds SDL3, so it takes longer than later reloads. Link the CMake target SDL3::SDL3; do not add platform-specific SDL library paths by hand.

Keep the PCH portable

Put stable standard-library headers in pch.h. Include Direct3D headers only in Windows renderer files and Metal headers only in macOS renderer files. CMake creates and injects the PCH correctly for Apple Clang and the Windows toolchain.

The code we are working toward

#include <SDL3/SDL.h>
#include <SDL3/SDL_main.h> // include once, in the file that defines main
#include <Engine.h>
#include <exception>
#include <iostream>
#include <utility>

class SandboxApp : public Application {
public:
    explicit SandboxApp(EngineConfig config)
        : Application(std::move(config)) {}

    void OnUpdate(float deltaTime) override {
        // Portable game logic goes here.
    }
};

int main(int argc, char* argv[]) {
    try {
        SandboxApp app{LoadStartupConfig()};
        return app.Run();
    } catch (const std::exception& error) {
        std::cerr << "Startup failed: " << error.what() << '\n';
        return 1;
    }
}
Recap
  • Engine and Sandbox are CMake targets, so the same project opens in CLion on Windows and macOS.
  • SDL3 provides one portable window, event, and input layer; Direct3D 11 and Metal live behind separate renderer implementations.
  • Platform APIs stay out of shared public headers and the portable PCH.
  • Sandbox depends on Engine, never the reverse.
