Structuring the Engine Project

This is the lesson where the tutorial becomes a project. Everything you have learned so far — variables, functions, classes, the standard library, and the input and time services from the previous lesson — now gets a permanent home. We are going to lay out the skeleton of a real engine: a reusable Engine library and a small Sandbox game that links against it, exactly the split that Unity, Unreal, and every serious engine use.

What you'll learn
  • Why an engine is split into a static library plus a thin executable that uses it.
  • The folder layout we will fill for the rest of the module: Core, Math, Platform, Renderer, and Scene.
  • How to organize headers and set up a precompiled header to keep compile times sane.
  • The naming conventions we will follow so every file in the project looks like it was written by one person.

Two projects, one goal

A common beginner mistake is to pile every .cpp file into a single executable. That works for a demo, but it makes the "engine" and the "game" impossible to tell apart. Professional engines draw a hard line between the two:

  • Engine — a static library (.lib on Windows). It contains reusable systems: the game loop, the window, the renderer, the math types, the scene. It knows nothing about your specific game.
  • Sandbox — an executable (.exe) that links the Engine and does something concrete with it: opens a window, spins a cube, prints the frame time. This is where your game logic lives.

The payoff is enormous. When you later want a second game — or a tools application — you link the same Engine again instead of copy-pasting code. It also forces you to design clean public interfaces, because the Sandbox can only touch what the Engine exposes through its headers.

Five source folders, each a clear responsibility:

FolderResponsibilityDepends on
CoreGame loop, logging, assertions, shared base types.nothing
MathVectors, matrices, the Transform we will build components around.nothing
PlatformWindows-specific glue: creating a window, reading input, measuring time.Core
RendererThe Direct3D 11 device, swap chain, buffers, and shaders.Core, Math, Platform
SceneEntities and the components that hang off them (Transform, Mesh, …).Core, Math

Notice the dependencies point one way: Core and Math know nothing about rendering, while Renderer is allowed to use both. Keeping this direction clean is what stops an engine from turning into spaghetti.

Header organization and the umbrella header

Each subsystem exposes its public types through .h files. The Sandbox should not have to know the internal folder structure, so we provide a single convenience header that pulls the public pieces together:

#pragma once

// One header the game can include to get the public engine API.
// The Sandbox never reaches into Renderer/ or Platform/ directly.

#include "Core/Application.h"
#include "Core/Log.h"
#include "Math/Vector3.h"
#include "Math/Matrix4.h"
#include "Scene/Entity.h"
#include "Scene/Component.h"

Every header starts with #pragma once. It tells the compiler "include me at most once per translation unit," which prevents the redefinition errors you get when two files include the same header. It is a one-line, compiler-supported alternative to the old #ifndef include-guard dance, and every compiler we care about supports it.

Include what you use

Inside the Engine's own .cpp files, include the specific headers you need (#include "Math/Vector3.h"), not the umbrella Engine.h. The umbrella is a convenience for the consumer. Using it internally would create needless rebuild cascades — touch one header and everything recompiles.

The precompiled header

Big headers like <windows.h> and the DirectX headers are enormous and get parsed for every .cpp file that includes them. A precompiled header (PCH) parses those heavy, rarely-changing headers once and reuses the result, often cutting build times dramatically. We put the stable includes in pch.h:

#pragma once

// Heavy, stable headers that almost never change.
// Precompiled once, reused across the whole Engine library.

// C++ standard library
#include <cstdint>
#include <memory>
#include <string>
#include <vector>
#include <unordered_map>

// Windows + DirectX 11
#include <windows.h>
#include <d3d11.h>
#include <DirectXMath.h>

Then every .cpp in the Engine makes pch.h its very first line:

#include "pch.h"          // must be the first include
#include "Core/Application.h"

namespace Engine {

    Application::Application() {
        // The heavy DirectX/Windows headers above are already
        // parsed thanks to the PCH, so this compiles fast.
    }

} // namespace Engine
The PCH must come first

In Visual Studio, the PCH include line has to be the first non-comment line of the file, before any other #include. Anything above it is ignored by the compiler, which produces a confusing "unexpected end of file while looking for precompiled header" error. If you ever see that message, this is almost always the cause.

Naming conventions

Consistency is what makes a codebase feel professional. Pick a style now and never deviate. Here is the convention we will use for the rest of the module:

ThingStyleExample
Types (class/struct/enum)PascalCaseGraphicsDevice, Transform
Functions & methodsPascalCaseOnUpdate(), CreateBuffer()
Local variables & parameterscamelCasedeltaTime, vertexCount
Private member variablesm_ prefixm_device, m_running
Constants / enum valuesPascalCase or UPPERMaxEntities
NamespacePascalCaseEngine
Filesmatch the primary typeApplication.h holds Application

Everything the Engine exposes lives inside a single namespace Engine { ... }. That prevents name clashes with the standard library or DirectX and makes engine code obvious at the call site: Engine::Vector3 reads clearly from the Sandbox.

What the Sandbox will look like

To make the plan concrete, here is roughly the main.cpp we are working toward. It will not compile yet — the classes it names are the ones we build over the coming lessons — but it shows how little the game needs to know:

#include <Engine.h>   // the umbrella header from the Engine library

// Our game derives from the engine's Application base class
// and overrides the hooks it cares about.
class SandboxApp : public Engine::Application {
public:
    void OnUpdate(float deltaTime) override {
        // Game logic will go here. deltaTime comes from the
        // time service we built in the previous lesson.
    }
};

int main() {
    SandboxApp app;
    app.Run();   // hands control to the engine's game loop
    return 0;
}
This is the shape of every engine

A tiny game that subclasses an Application, overrides a couple of hooks, and calls Run() — that is the same pattern Unity's MonoBehaviour and Unreal's Actor callbacks are built on. Getting the project structure right today is what lets the rest of the module stay focused on interesting graphics work instead of plumbing.

Recap
  • An engine is split into a reusable static library (Engine) and a thin executable (Sandbox) that links it — the same architecture as Unity and Unreal.
  • The library is organized into Core, Math, Platform, Renderer, and Scene, with dependencies pointing one way (Core/Math know nothing about rendering).
  • Every header uses #pragma once; a single Engine.h umbrella header exposes the public API to the game.
  • A precompiled header (pch.h) parses heavy headers like <windows.h> and <d3d11.h> once, and must be the first include in each .cpp.
  • We follow one naming convention throughout: PascalCase types and methods, camelCase locals, m_ member prefixes, all inside namespace Engine.
