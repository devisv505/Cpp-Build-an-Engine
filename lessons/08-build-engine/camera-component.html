The Camera Component

Every engine needs an eye. In Unity you drop a Camera onto a GameObject; in Unreal it is an actor component. We are going to do exactly the same thing: our camera will be just another component that lives on a GameObject and reads its Transform. By the end of this lesson our renderer will be able to ask one object, "what does the world look like from here?", and turn that answer into the matrix every mesh needs to appear on screen.

What you'll learn
  • Why a camera is best modelled as a Component, not a special hard-coded object
  • How to build a view matrix from the owner GameObject's Transform
  • How to build a projection matrix from FOV, aspect ratio, near and far planes
  • How the Scene tracks an active camera and the renderer uses it for each object's MVP

A camera is just a component

In the previous lesson we built GameObject and the Component base class, and we gave every GameObject a Transform that describes where it sits in the world. A camera does not need any new machinery. It needs a position and orientation in the world (that is the Transform we already have) plus a few lens settings. So the camera is a small class that derives from Component and does one job: turn its owner's Transform into the matrices the GPU wants.

This is the same design Unity and Unreal use, and it is a big reason engines feel so flexible. Because the camera rides on a GameObject, moving the camera is the same as moving any object, parenting works for free, and you can have many cameras and simply choose which one is active.

The view matrix: where the camera is looking

A Transform can already produce a world matrix — the matrix that places an object into the scene. The view matrix is the mirror image of that idea. Instead of pushing an object out into the world, it pulls the whole world in front of a camera sitting at the origin, looking down +Z. If the camera's world matrix says "I am here, facing this way", then undoing that transform is exactly what moves everything into the camera's point of view.

That "undoing" is a matrix inverse. So the view matrix is simply the inverse of the camera's world matrix — no separate look-at bookkeeping required.

View = inverse of the camera's world matrix

This one identity is the whole trick. Because our camera lives on a GameObject with a Transform, we reuse the GetWorldMatrix() we already wrote last lesson and invert it. Move the GameObject and the view updates automatically.

The projection matrix: how the camera sees

The projection matrix decides the shape of the viewing volume — the frustum. For a perspective camera it comes from four familiar knobs:

ParameterMeaningTypical value
fovYVertical field of view (in radians) — how wide the lens is45°–60°
aspectBack-buffer width divided by height16 / 9
nearZClosest distance that is still drawn0.1
farZFarthest distance that is still drawn1000

DirectXMath gives us XMMatrixPerspectiveFovLH, which builds exactly this left-handed perspective matrix. Here is the whole Camera component — small, focused, and derived from Component just like everything else:

#include <DirectXMath.h>
using namespace DirectX;

// Camera derives from Component, exactly like Transform or MeshRenderer.
// It stores no position of its own — it reads the owner GameObject's Transform.
class Camera : public Component
{
public:
    float fovY   = XM_PIDIV4;      // vertical field of view (45 degrees)
    float aspect = 16.0f / 9.0f;   // back-buffer width / height
    float nearZ  = 0.1f;           // closest visible distance
    float farZ   = 1000.0f;        // farthest visible distance

    // The view matrix pulls the WORLD in front of a camera at the origin.
    // That is simply the inverse of the camera's world transform.
    XMMATRIX GetView() const
    {
        XMMATRIX world = owner->transform->GetWorldMatrix();
        return XMMatrixInverse(nullptr, world);
    }

    // A perspective projection built from the four classic lens knobs.
    XMMATRIX GetProjection() const
    {
        return XMMatrixPerspectiveFovLH(fovY, aspect, nearZ, farZ);
    }

    // The combined matrix the renderer wants for every object this frame.
    XMMATRIX GetViewProjection() const
    {
        return GetView() * GetProjection();
    }
};

Notice owner — that is the GameObject* back-pointer the Component base class gave every component last lesson. Through it the camera can reach the shared transform. That is the entire link between "a thing in the scene" and "a lens on the scene".

Recompute aspect on resize

The aspect ratio must match the render target. When the window resizes and you rebuild the swap chain's buffers, set camera->aspect = width / (float)height; so nothing looks stretched.

GetViewProjection(): one matrix for the pipeline

The renderer does not want view and projection separately — it wants them multiplied. Because DirectXMath uses row vectors, transforms compose left to right, so view * projection is the correct order. Combining them once per frame (not once per object) is a small but real optimisation: the view and projection never change while we walk the object list.

The scene tracks the active camera

A scene may hold several cameras — a main view, a minimap, a shadow view. The renderer needs to know which one to render through right now. We store a single non-owning pointer on the Scene: the GameObject owns the Camera; the Scene merely remembers which camera is active.

class Scene
{
public:
    std::vector<std::unique_ptr<GameObject>> objects;

    // Raw, NON-owning pointer. The GameObject owns the Camera via its
    // component list; the Scene only remembers which one is rendering.
    Camera* activeCamera = nullptr;

    void SetActiveCamera(Camera* cam) { activeCamera = cam; }
};
Never store an owning pointer here

activeCamera must be a plain Camera*, not a unique_ptr. The GameObject already owns the component. If two things tried to own it, you would get a double free when the scene tears down.

The renderer builds the per-object MVP

Now the payoff. Each frame the renderer grabs the active camera's view-projection once, then loops over the drawable objects. For each one it folds in that object's world matrix to produce the MVP (model-view-projection) matrix, and uploads it to a constant buffer the vertex shader reads.

struct CBPerObject
{
    XMFLOAT4X4 mvp;   // world * view * projection, ready for the GPU
};

void Renderer::DrawScene(Scene& scene)
{
    if (scene.activeCamera == nullptr)
        return;   // nothing to look through — skip the frame

    // Ask the active camera for view * projection ONCE this frame.
    XMMATRIX viewProj = scene.activeCamera->GetViewProjection();

    for (auto& obj : scene.objects)
    {
        MeshRenderer* mesh = obj->GetComponent<MeshRenderer>();
        if (mesh == nullptr)
            continue;   // this GameObject has nothing to draw

        // Per-object MVP: the object's world folded into view * projection.
        XMMATRIX mvp = obj->transform->GetWorldMatrix() * viewProj;

        // HLSL reads matrices column-major but DirectXMath is row-major,
        // so we transpose before copying into the constant buffer.
        CBPerObject cb;
        XMStoreFloat4x4(&cb.mvp, XMMatrixTranspose(mvp));
        m_context->UpdateSubresource(m_cbPerObject.Get(), 0, nullptr, &cb, 0, 0);

        mesh->Draw(m_context.Get());   // binds buffers and calls DrawIndexed
    }
}

And the matching vertex shader is tiny — it just multiplies each vertex by the MVP we uploaded:

cbuffer CBPerObject : register(b0)
{
    float4x4 gMVP;   // model-view-projection for this object
};

struct VSInput
{
    float3 posL : POSITION;   // vertex position in local / model space
};

float4 main(VSInput vin) : SV_POSITION
{
    // Row-vector convention: position on the left, matrix on the right.
    return mul(float4(vin.posL, 1.0f), gMVP);
}
Forgetting the transpose = a black screen

DirectXMath stores matrices row-major; an HLSL constant buffer interprets them column-major. If you skip XMMatrixTranspose, your MVP is effectively transposed and everything projects to nowhere. When a scene renders blank with no errors, check this first.

That is the complete data flow: a GameObject's Transform feeds the Camera, the Camera feeds the Scene's active-camera slot, and the renderer combines it with each object's world matrix to place every triangle on screen. In the next lesson we will wire this into the engine loop so it all runs, frame after frame.

The Metal camera path on macOS

The Camera component remains shared C++. The macOS backend converts the engine's view-projection matrix to simd_float4x4 and uploads it to an MTLBuffer. Metal Shading Language normally multiplies a column vector as mvp * position, so perform any row/column conversion once during upload.

struct FrameConstants {
    simd_float4x4 mvp;
};

FrameConstants constants{ ToMetalMatrix(world * viewProjection) };
[encoder setVertexBytes:&constants
                  length:sizeof(constants)
                 atIndex:0];
struct FrameConstants { float4x4 mvp; };

vertex float4 vertexMain(float3 position [[attribute(0)]],
                         constant FrameConstants& frame [[buffer(0)]]) {
    return frame.mvp * float4(position, 1.0);
}
Recap
  • The camera is just a Component on a GameObject, so it reuses the Transform we already built.
  • The view matrix is the inverse of the camera's world matrix; the projection matrix comes from FOV, aspect, near and far.
  • GetViewProjection() returns view * projection once per frame for the renderer to reuse.
  • The Scene keeps a non-owning activeCamera pointer; the GameObject remains the owner.
  • The renderer builds one per-object MVP and converts it at the backend boundary: HLSL on Windows or Metal Shading Language on macOS.
