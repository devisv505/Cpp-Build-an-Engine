Scripting with a MonoBehaviour-Style Class

You already have a Component base class and a GameObject that owns a list of components. That is the exact foundation Unity and Unreal are built on. In this lesson we add the piece that makes it feel like a real engine: a scripting layer. We will write a Behaviour class — our version of Unity's MonoBehaviour — so you can drop a script onto any object in the scene and give it its own logic.

What you'll learn
  • Why a Behaviour class sits on top of Component — and why it mirrors Unity's MonoBehaviour.
  • How to give scripts an OnStart() that runs once and an OnUpdate(float dt) that runs every frame.
  • How GameObject::Update drives the whole lifecycle with a single m_started flag.
  • How to write real scripts — a spinning RotatorBehaviour and a WASD PlayerController.
  • How to attach them with AddComponent<T>(), composition-style, with no extra wiring.

From Component to Behaviour

Your Component class already does a lot: it holds a GameObject* owner, exposes GetComponent<T>(), and has a virtual OnUpdate(float dt) that the GameObject calls each frame. So why add anything?

Because most scripts need two moments, not one. They need a place to set things up once — grab a reference, read a starting position, spawn something — and a place to run logic every frame. In Unity those are Start() and Update(). We are going to add the same pair as OnStart() and OnUpdate(), and we will call the class that owns them Behaviour. Everything else — the owner pointer, GetComponent, living inside a GameObject's component list — we inherit for free.

Behaviour is our MonoBehaviour

When you write class Player : public MonoBehaviour in Unity, you are deriving from a component that adds the Start/Update lifecycle. Our Behaviour plays the same role — a thin, script-friendly layer over the raw Component.

The Behaviour base class

Here is the whole layer. The trick is a single private bool m_started flag and a small driver method — Tick() — that the engine calls. Behaviour overrides Tick() to fire OnStart() exactly once, then OnUpdate() forever after. 

#include <memory>

class GameObject;   // forward declaration — defined in a previous lesson

// Component (from the previous lesson): every component knows its owner
// and gets a chance to run logic each frame via OnUpdate().
class Component {
public:
    virtual ~Component() = default;

    virtual void OnUpdate(float dt) {}

    // The engine calls Tick() every frame. By default it just forwards
    // to OnUpdate(); Behaviour overrides it to inject a one-time OnStart().
    virtual void Tick(float dt) { OnUpdate(dt); }

    GameObject* GetOwner() const { return m_owner; }

    template<typename T>
    T* GetComponent() const { return m_owner->GetComponent<T>(); }

private:
    friend class GameObject;      // GameObject sets m_owner when it attaches us
    GameObject* m_owner = nullptr;
};

// Behaviour is our MonoBehaviour: derive from it to write a script.
class Behaviour : public Component {
public:
    virtual void OnStart() {}      // called ONCE, right before the first update
    // OnUpdate(float dt) is inherited from Component — called every frame.

    void Tick(float dt) override {
        if (!m_started) {          // run OnStart exactly once...
            OnStart();
            m_started = true;
        }
        OnUpdate(dt);              // ...then OnUpdate every frame after that
    }

private:
    bool m_started = false;
};

That is the entire scripting engine. Notice we did not touch the GameObject much — we only added a virtual Tick() to Component so a Behaviour can wrap its own lifecycle around the plain update. A component that is not a Behaviour just uses the default Tick(), which forwards straight to OnUpdate().

Driving it from GameObject

The GameObject is unchanged from last lesson except that its per-frame loop now calls Tick() instead of OnUpdate() directly. Because Tick() is virtual, each component decides for itself what happens — plain components update, behaviours start-then-update. 

class GameObject {
public:
    // ... name, Transform, and the component list from the last lesson ...

    // Add a script (or any component) and get a pointer back to it.
    template<typename T, typename... Args>
    T* AddComponent(Args&&... args) {
        auto comp = std::make_unique<T>(std::forward<Args>(args)...);
        comp->m_owner = this;                 // GameObject is a friend of Component
        T* raw = comp.get();
        m_components.push_back(std::move(comp));
        return raw;
    }

    // Called once per frame by the scene. This is what powers Start/Update.
    void Update(float dt) {
        for (auto& c : m_components)
            c->Tick(dt);                      // behaviours fire OnStart then OnUpdate
    }

    Transform* GetTransform() const { return m_transform; }

private:
    Transform* m_transform = nullptr;
    std::vector<std::unique_ptr<Component>> m_components;
};

Your first script: RotatorBehaviour

Time to actually write a script. This one spins its object around the Y axis at a constant speed. It caches the Transform in OnStart() so it does not have to look it up 60 times a second, then nudges the rotation in OnUpdate(). 

// Spins its GameObject around the Y axis at a constant speed.
class RotatorBehaviour : public Behaviour {
public:
    float degreesPerSecond = 90.0f;   // a public "inspector" field, just like Unity

    void OnStart() override {
        // Cache the Transform once so we don't fetch it every frame.
        m_transform = GetComponent<Transform>();
    }

    void OnUpdate(float dt) override {
        if (!m_transform) return;                          // no Transform, nothing to spin
        m_transform->rotation.y += degreesPerSecond * dt;  // multiply by dt = frame-rate safe
    }

private:
    Transform* m_transform = nullptr;
};
Always multiply movement by dt

degreesPerSecond * dt means the cube rotates the same amount per second whether the game runs at 30 or 300 FPS. Forget the * dt and your object spins faster on faster machines — one of the most common beginner bugs.

Reading input: a PlayerController

A rotating cube is nice, but scripts get interesting when they react to the player. Here is a controller that moves its object across the XZ plane using WASD. It uses the GLFW window already owned by the engine, so the same code works in CLion on Windows and macOS. 

#include <GLFW/glfw3.h>

// Moves its GameObject across the XZ plane using WASD.
class PlayerController : public Behaviour {
public:
    float moveSpeed = 5.0f;   // world units per second

    void OnStart() override {
        m_transform = GetOwner()->GetTransform();
    }

    void OnUpdate(float dt) override {
        Vector3 dir{ 0.0f, 0.0f, 0.0f };

        GLFWwindow* window = Application::Get().GetWindow().NativeHandle();
        if (glfwGetKey(window, GLFW_KEY_W) == GLFW_PRESS) dir.z += 1.0f;
        if (glfwGetKey(window, GLFW_KEY_S) == GLFW_PRESS) dir.z -= 1.0f;
        if (glfwGetKey(window, GLFW_KEY_D) == GLFW_PRESS) dir.x += 1.0f;
        if (glfwGetKey(window, GLFW_KEY_A) == GLFW_PRESS) dir.x -= 1.0f;

        m_transform->position.x += dir.x * moveSpeed * dt;
        m_transform->position.z += dir.z * moveSpeed * dt;
    }

private:
    Transform* m_transform = nullptr;
};

Both scripts follow the same shape: reach for what they need in OnStart(), do work in OnUpdate(). That predictable rhythm is exactly why the Unity pattern scales to thousands of scripts — and why it will keep your MiniEngine sane too.

Don't call GetComponent in the constructor

When a script's constructor runs, its m_owner may not be set yet and sibling components may not be attached. Reaching for GetComponent<Transform>() too early gives you nullptr. OnStart() exists precisely so every reference is ready before you grab it — always cache there, not in the constructor.

Dropping scripts onto objects

Here is the payoff. Because a script is a component, attaching behaviour to any object is one call — AddComponent<T>() — with no scene-graph surgery, no registration, no switch statements. That is the composition promise from the last lesson, delivered. 

// Build a little scene and give objects behaviour by attaching scripts.
GameObject* cube = scene.CreateGameObject("SpinningCube");
cube->AddComponent<MeshRenderer>();                 // from the rendering lessons

RotatorBehaviour* spin = cube->AddComponent<RotatorBehaviour>();
spin->degreesPerSecond = 180.0f;                   // AddComponent returns a T*, so we configure it

GameObject* player = scene.CreateGameObject("Player");
player->AddComponent<PlayerController>();           // WASD movement, zero extra wiring

// ...and in the main loop, once per frame:
for (GameObject* go : scene.GetObjects())
    go->Update(dt);                                // every Behaviour ticks itself

Read that again and compare it to Unity: you drop a script onto an object and it comes alive. Want a second spinning object? AddComponent<RotatorBehaviour>() on any GameObject. Want the cube to also chase the player? Write a ChaseBehaviour and attach it alongside the rotator — they coexist because each is just another entry in the component list. No inheritance gymnastics, no god-class. This is the same architecture that lets Unity and Unreal scale to entire games.

Recap
  • Behaviour derives from Component and adds a scripting lifecycle — it is our MonoBehaviour.
  • A private bool m_started lets Tick() call OnStart() once, then OnUpdate(float dt) every frame.
  • GameObject::Update just loops its components calling Tick(); plain components forward to OnUpdate, behaviours run the full start-then-update flow.
  • Real scripts like RotatorBehaviour and PlayerController cache references in OnStart() and do frame work in OnUpdate() — always scaled by dt.
  • AddComponent<T>() drops a script onto any object, so you compose behaviour exactly like Unity.
