First, take a moment

Building an engine is hard. Most people who try to learn graphics programming stall out at "I got a triangle on screen and then got overwhelmed." You pushed past that. You understand the pieces — a window, a device, a swap chain, vertex and index buffers, a constant buffer feeding matrices to a shader, and a loop that ties it all together. More importantly, you understand why those pieces exist and how they talk to each other. That mental model is the thing that transfers to any engine, any API, any job.

Everything from here is an extension of what you already have. You are no longer learning to program; you are learning graphics and systems techniques and expressing them in tools you already own.

Build features, not a rewrite

The temptation now is to scrap everything and "do it properly." Resist it. Real engines grow feature by feature on top of a working core. Add one thing, get it on screen, commit, repeat. A messy engine that runs teaches you far more than a perfect architecture that never renders.

Lighting and normals — do this next

Right now your objects are lit by nothing, so they read as flat silhouettes. Lighting is the single biggest visual jump you can make, and it is mostly a shader change. The core idea: every vertex carries a normal — a direction the surface faces — and in the pixel shader you compare that normal against the direction to a light. Surfaces facing the light are bright; surfaces facing away are dark.

You will extend your vertex format to include a normal, then add a light direction to a constant buffer. The math for basic diffuse (Lambert) lighting is a single dot product:

// Pixel shader input already carries the interpolated world-space normal.
cbuffer LightBuffer : register(b1)
{
    float3 lightDir;   // direction the light travels, normalized
    float  padding;
    float4 lightColor;
}

float4 PSMain(PSInput input) : SV_TARGET
{
    // Normalize because interpolation across the triangle shortens it.
    float3 N = normalize(input.normalWS);

    // Light against the surface. We negate lightDir so it points
    // from the surface toward the light source.
    float  diffuse = saturate(dot(N, -lightDir));

    float3 baseColor = input.color.rgb;
    float3 lit       = baseColor * lightColor.rgb * diffuse;

    // A little ambient so shadowed areas aren't pure black.
    lit += baseColor * 0.05;

    return float4(lit, 1.0);
}

Once this works, a Phong material adds a specular highlight (a bright spot where the reflection of the light hits your eye), and from there PBR (physically based rendering) replaces the ad-hoc numbers with parameters that mean something real: metallic, roughness, albedo. That is your first proper Material component — data that lives on an entity and tells the renderer how its surface behaves.

Normals need care

When you scale an object non-uniformly, normals must be transformed by the inverse transpose of the world matrix, not the world matrix itself. Get this wrong and lighting looks subtly broken on stretched objects. It is a classic bug — knowing it exists saves you an afternoon.

Asset importing with assimp

Typing cube vertices by hand gets old fast. assimp (the Open Asset Import Library) reads dozens of model formats and hands you clean arrays of positions, normals, texture coordinates, and indices. Your job is to copy those arrays into the same vertex and index buffers you already know how to make. This is where your engine starts loading real content.

Shadow mapping

Shadows sound intimidating but the trick is elegant: render the scene from the light's point of view into an off-screen depth texture. Then, when shading normally, check whether each pixel is farther from the light than what that depth texture recorded. If it is, something is between it and the light — so it is in shadow. You already know how to render to a target and sample a texture, which is 80% of the work.

An editor with Dear ImGui

This is the feature that makes your engine feel like an engine. Dear ImGui is a lightweight immediate-mode UI library that drops into your render loop and lets you draw panels, sliders, and lists in a few lines. Wire it to your entity list and you have an Inspector: click an entity, see its Transform, drag a slider, watch it move. It pairs perfectly with the next item.

Scenes, serialization, and scripting

A scene is your list of entities and their components. You already know how to read and write YAML configuration, so YAML or JSON is a friendly first representation—but scene data needs its own schema, loader, validation, and version migration. Do not force dynamic world state through EngineConfig. Once you can save and load scenes, your editor becomes genuinely useful. Scripting—often via Lua or C#—then lets you write gameplay behavior without recompiling the whole engine, exactly the split you see between Unreal's C++ core and its Blueprints, or Unity's engine and your C# scripts.

Physics, audio, and going modern

For physics, do not write your own at first — integrate a mature library like Jolt or Bullet, feed it your transforms, and read the results back each frame. Audio can start with a simple cross-platform library that plays a sound at a position. When you want lower-level GPU control, continue with Direct3D 12 on Windows, deepen your Metal knowledge on macOS, or explore Vulkan where it is supported. The concepts map back to the renderer boundary you already built.

How to learn the hard parts

You will hit topics that no single tutorial explains perfectly. That is normal, and it is a skill in itself to work through them. A reliable process:

  1. Read the official docs first. The vendor API reference is the source of truth — it tells you exactly what each function expects and returns.
  2. Find one good book or course and follow it end to end rather than hopping between ten half-tutorials. Depth beats breadth here.
  3. Read other engines' source. Small open-source engines are a goldmine; seeing how someone else structured a renderer is worth a dozen articles.
  4. Ask a community. Active graphics and game-dev forums and chat servers are full of people who enjoy these problems. Post a minimal example and the exact error.
  5. Use API validation. Enable the Direct3D debug layer on Windows and Metal API Validation on macOS, then read every warning.
Beware tutorial-hopping

Jumping between sources that each use different conventions is the fastest way to get confused and quit. Pick one primary path per topic, finish it, and only then compare it against others. Consistency in what you read mirrors the consistency that made this course click.

Your next week

Momentum is fragile right after a big finish, so give yourself a small, concrete goal — not "rewrite the renderer," but something you can land in a few evenings:

  • Add a normal to your vertex struct and get basic diffuse lighting working on your existing cube.
  • Commit it. Then add a second light color and a slider for it — even without ImGui yet, just change a constant and recompile.
  • Write down three features from the roadmap that excite you most, and pick the top one as your next milestone.

That is how engines get built: not in one heroic sprint, but in dozens of small, shippable wins stacked on a core that already runs. You have that core now. Keep going — and enjoy the moment your world first responds to light. Thank you for building this with us, and good luck out there.
