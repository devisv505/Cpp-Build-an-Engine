Reading & Writing YAML Configuration

Window size, clear color, and VSync should not require recompiling the engine. In this lesson we replace those early constexpr experiments with a typed EngineConfig, load it from a human-readable YAML file, validate every value, and save user settings to the correct writable location on Windows and macOS.

What you'll learn
  • Choose YAML for small, human-edited engine settings—not every kind of serialization.
  • Add a pinned yaml-cpp dependency with CMake FetchContent.
  • Read maps, sequences, scalars, optional values, and useful parse errors.
  • Convert untrusted YAML into ordinary C++ structs and validate the result.
  • Find shipped and writable files portably with SDL3 on Windows and macOS.
  • Write YAML and verify a load → save → reload round trip.

Where YAML belongs

YAML is good when a developer or player may open the file and understand it. It is deliberately more verbose than a binary format. Use the format that matches the job:

DataGood formatWhy
Engine and user settingsYAMLSmall, named, readable, and easy to edit.
Future scenes or component prototypesYAML or JSONEditor-friendly structured data; add an explicit schema.
Per-frame network packetsCompact binary protocolBandwidth, deterministic layout, and strict bounds matter.
Large voxel regions or production save dataVersioned binary formatSize and loading speed matter more than hand editing.
Configuration is runtime data

The constants from Module 01 were useful C++ practice. Keep genuine compile-time facts as constexpr, but move values a user may change—title, resolution, VSync, colors, and asset roots—into runtime configuration.

Write the shipped defaults

Create config/default-engine.yaml at the project root. YAML has maps of named keys, sequences such as the four color values, and scalar strings, numbers, and booleans.

# This file ships with the engine. User changes are saved separately.
schema_version: 1

window:
  title: MiniEngine
  width: 1280
  height: 720
  resizable: true

renderer:
  vsync: true
  clear_color: [0.04, 0.08, 0.16, 1.0]
  • Indent with spaces, never tabs. Every child under window uses the same indentation.
  • Text usually needs no quotes, but quote a value containing YAML punctuation such as : or #.
  • schema_version is required. If the layout changes later, increment it and migrate or reject old files deliberately.
  • A missing optional key uses a safe C++ default; a present but malformed value is an error.

Add yaml-cpp to CMake

yaml-cpp is a C++ YAML parser and emitter. Extend Engine/CMakeLists.txt with the pinned 0.9.0 release so Windows and macOS build the same tested source. Put this dependency block after the existing include(FetchContent) and before add_library(Engine ...):

# Build yaml-cpp as part of MiniEngine. Pinning makes builds reproducible.
set(YAML_BUILD_SHARED_LIBS OFF CACHE BOOL "" FORCE)
set(YAML_CPP_BUILD_TOOLS OFF CACHE BOOL "" FORCE)
set(YAML_CPP_BUILD_TESTS OFF CACHE BOOL "" FORCE)
set(YAML_CPP_INSTALL OFF CACHE BOOL "" FORCE)

FetchContent_Declare(
  yaml-cpp
  GIT_REPOSITORY https://github.com/jbeder/yaml-cpp.git
  GIT_TAG yaml-cpp-0.9.0
  GIT_SHALLOW TRUE)
FetchContent_MakeAvailable(yaml-cpp)

Then keep the existing add_library(Engine ...) and add these lines after that target has been created:

target_sources(Engine PRIVATE src/Core/EngineConfig.cpp)
target_link_libraries(Engine PRIVATE yaml-cpp::yaml-cpp)

The link is PRIVATE because no public engine header will expose a YAML::Node. That keeps most of the project independent of the library and reduces recompilation.

Reload CMake in CLion

Choose Tools → CMake → Reload CMake Project. The first configure downloads and builds yaml-cpp. Include <yaml-cpp/yaml.h> in C++; do not add a header path or library file manually in CLion.

Copy defaults beside the executable

A relative path such as config/default-engine.yaml depends on CLion's working-directory setting. Avoid that hidden dependency. Replace the current two-line Sandbox/CMakeLists.txt with this version: it copies the default beside Sandbox after every build, then the engine finds it from the executable's base path.

add_executable(Sandbox src/main.cpp)
target_link_libraries(Sandbox PRIVATE Engine)

# Custom targets run even when Sandbox itself does not need relinking.
# copy_if_different avoids touching the destination unnecessarily.
add_custom_target(CopyEngineConfig ALL
  COMMAND ${CMAKE_COMMAND} -E make_directory
          "$<TARGET_FILE_DIR:Sandbox>/config"
  COMMAND ${CMAKE_COMMAND} -E copy_if_different
          "${PROJECT_SOURCE_DIR}/config/default-engine.yaml"
          "$<TARGET_FILE_DIR:Sandbox>/config/default-engine.yaml"
  DEPENDS "${PROJECT_SOURCE_DIR}/config/default-engine.yaml"
  VERBATIM)
add_dependencies(Sandbox CopyEngineConfig)

This works for CLion's cmake-build-debug, another build folder, and a Release build. If you later create a real macOS .app bundle, install this file as a bundle resource instead; the configuration API below does not otherwise change.

Expose typed C++, not YAML nodes

The rest of the engine should receive validated C++ values. Give every field a useful default so older files may omit new optional settings.

#pragma once
#include <array>
#include <filesystem>
#include <string>

struct WindowConfig {
    std::string title{"MiniEngine"};
    int width = 1280;
    int height = 720;
    bool resizable = true;
    bool operator==(const WindowConfig&) const = default;
};

struct RendererConfig {
    bool vsync = true;
    std::array<float, 4> clearColor{0.04f, 0.08f, 0.16f, 1.0f};
    bool operator==(const RendererConfig&) const = default;
};

struct EngineConfig {
    int schemaVersion = 1;
    WindowConfig window;
    RendererConfig renderer;
    bool operator==(const EngineConfig&) const = default;
};

EngineConfig LoadEngineConfig(const std::filesystem::path& path);
void SaveEngineConfig(const EngineConfig& config,
                      const std::filesystem::path& path);

std::filesystem::path DefaultEngineConfigPath();
std::filesystem::path UserEngineConfigPath();
EngineConfig LoadStartupConfig();

The naming boundary is intentional: YAML uses snake_case, while C++ members use the project's camelCase convention. The loader is where the two representations meet.

Read optional values without hiding mistakes

yaml-cpp can convert a node with node.as<T>(). Its as<T>(fallback) overload also returns the fallback for some bad conversions, which could silently turn width: banana into 1280. We want a default only when the key is absent:

#include "EngineConfig.h"
#include <SDL3/SDL.h>
#include <yaml-cpp/yaml.h>
#include <cmath>
#include <fstream>
#include <memory>
#include <stdexcept>
#include <string_view>
#include <utility>

namespace {
std::filesystem::path PathFromUtf8(std::string_view text) {
    // SDL returns UTF-8. A u8string lets std::filesystem convert it to
    // UTF-16 on Windows while preserving UTF-8 bytes on macOS.
    std::u8string utf8;
    utf8.reserve(text.size());
    for (unsigned char byte : text) {
        utf8.push_back(static_cast<char8_t>(byte));
    }
    return std::filesystem::path{utf8};
}

std::string PathForDisplay(const std::filesystem::path& path) {
    const std::u8string utf8 = path.generic_u8string();
    std::string text;
    text.reserve(utf8.size());
    for (char8_t byte : utf8) {
        text.push_back(static_cast<char>(byte));
    }
    return text;
}

template <typename T>
T ReadOptional(const YAML::Node& map, const char* key, T fallback) {
    const YAML::Node value = map[key];
    return value ? value.as<T>() : std::move(fallback);
}

void RequireMap(const YAML::Node& node, const char* name) {
    if (node && !node.IsMap()) {
        throw std::runtime_error(std::string{"'"} + name +
                                 "' must be a map of named values");
    }
}

void Validate(const EngineConfig& config) {
    if (config.schemaVersion != 1) {
        throw std::runtime_error("Unsupported config schema_version: " +
                                 std::to_string(config.schemaVersion));
    }
    if (config.window.title.empty()) {
        throw std::runtime_error("window.title cannot be empty");
    }
    if (config.window.width < 320 || config.window.width > 16384 ||
        config.window.height < 200 || config.window.height > 16384) {
        throw std::runtime_error("window dimensions are outside a sensible range");
    }
    for (float channel : config.renderer.clearColor) {
        if (!std::isfinite(channel) || channel < 0.0f || channel > 1.0f) {
            throw std::runtime_error("renderer.clear_color values must be in [0, 1]");
        }
    }
}
} // namespace

Load and validate the document

Use std::ifstream with a std::filesystem::path. That preserves native path handling on Windows instead of forcing the path through a narrow string merely to call YAML::LoadFile.

EngineConfig LoadEngineConfig(const std::filesystem::path& path) {
    std::ifstream input(path, std::ios::binary);
    if (!input) {
        throw std::runtime_error("Cannot open engine config: " +
                                 PathForDisplay(path));
    }

    try {
        const YAML::Node root = YAML::Load(input);
        if (!root.IsMap()) {
            throw std::runtime_error("The YAML document root must be a map");
        }

        EngineConfig config;
        const YAML::Node schema = root["schema_version"];
        if (!schema) {
            throw std::runtime_error("schema_version is required");
        }
        config.schemaVersion = schema.as<int>();

        const YAML::Node window = root["window"];
        RequireMap(window, "window");
        if (window) {
            config.window.title =
                ReadOptional(window, "title", config.window.title);
            config.window.width =
                ReadOptional(window, "width", config.window.width);
            config.window.height =
                ReadOptional(window, "height", config.window.height);
            config.window.resizable =
                ReadOptional(window, "resizable", config.window.resizable);
        }

        const YAML::Node renderer = root["renderer"];
        RequireMap(renderer, "renderer");
        if (renderer) {
            config.renderer.vsync =
                ReadOptional(renderer, "vsync", config.renderer.vsync);

            const YAML::Node color = renderer["clear_color"];
            if (color) {
                if (!color.IsSequence() || color.size() != 4) {
                    throw std::runtime_error(
                        "renderer.clear_color must contain exactly 4 numbers");
                }
                for (std::size_t i = 0; i < 4; ++i) {
                    config.renderer.clearColor[i] = color[i].as<float>();
                }
            }
        }

        Validate(config);
        return config;
    } catch (const YAML::Exception& error) {
        // Parser and conversion errors normally include line and column.
        throw std::runtime_error("Invalid YAML in " + PathForDisplay(path) +
                                 ": " + error.what());
    }
}

There are three distinct policies here:

  1. A missing optional key keeps its typed default.
  2. A present value of the wrong type produces a location-aware YAML error.
  3. A value with the right type but an unsafe range produces a clear validation error.
Do not silently recover from a broken file

Defaults are for absent optional keys, not typos. If a user writes width: wide, stop with a useful message. Silently booting with another value makes configuration bugs much harder to find.

Use the correct path on each operating system

MiniEngine has two configuration locations:

  • Shipped default: read-only, copied beside the executable, and found with SDL_GetBasePath().
  • User settings: a complete writable configuration stored in the operating system's per-user application-data location from SDL_GetPrefPath().
std::filesystem::path DefaultEngineConfigPath() {
    const char* base = SDL_GetBasePath(); // cached pointer: do not free it
    if (!base) {
        throw std::runtime_error(std::string{"SDL_GetBasePath failed: "} +
                                 SDL_GetError());
    }
    return PathFromUtf8(base) / "config" / "default-engine.yaml";
}

std::filesystem::path UserEngineConfigPath() {
    std::unique_ptr<char, decltype(&SDL_free)> raw{
        SDL_GetPrefPath("DevisV", "MiniEngine"), &SDL_free};
    if (!raw) {
        throw std::runtime_error(std::string{"SDL_GetPrefPath failed: "} +
                                 SDL_GetError());
    }

    // The custom deleter balances SDL's allocation even if path conversion throws.
    std::filesystem::path directory = PathFromUtf8(raw.get());
    return directory / "settings.yaml";
}

EngineConfig LoadStartupConfig() {
    const std::filesystem::path user = UserEngineConfigPath();
    if (std::filesystem::exists(user)) {
        return LoadEngineConfig(user);
    }
    return LoadEngineConfig(DefaultEngineConfigPath());
}

On Windows the user file normally lands under the user's application-data folder; on macOS it normally lands under ~/Library/Application Support. SDL chooses the platform convention and creates the application directory. SDL returns UTF-8 paths, so PathFromUtf8 also keeps non-ASCII user and folder names valid on Windows. Change DevisV and MiniEngine to the stable organization and application identifiers for your project.

Write validated YAML

Build a fresh YAML::Node, emit it, create the destination directory, and check both the emitter and file stream. Saving a new document preserves values—not the comments, key ordering, quoting, or exact formatting of the file that was loaded. That is why the commented shipped default remains separate.

void SaveEngineConfig(const EngineConfig& config,
                      const std::filesystem::path& path) {
    Validate(config);

    YAML::Node root;
    root["schema_version"] = config.schemaVersion;

    root["window"]["title"] = config.window.title;
    root["window"]["width"] = config.window.width;
    root["window"]["height"] = config.window.height;
    root["window"]["resizable"] = config.window.resizable;

    root["renderer"]["vsync"] = config.renderer.vsync;
    YAML::Node color;
    for (float channel : config.renderer.clearColor) {
        color.push_back(channel);
    }
    root["renderer"]["clear_color"] = color;

    YAML::Emitter emitter;
    if (!emitter.SetIndent(2)) {
        throw std::runtime_error("Could not configure YAML indentation");
    }
    emitter << root;
    if (!emitter.good()) {
        throw std::runtime_error(std::string{"Could not emit YAML: "} +
                                 emitter.GetLastError());
    }

    if (path.has_parent_path()) {
        std::filesystem::create_directories(path.parent_path());
    }

    std::ofstream output(path, std::ios::binary | std::ios::trunc);
    if (!output) {
        throw std::runtime_error("Cannot write engine config: " +
                                 PathForDisplay(path));
    }

    output << emitter.c_str() << '\n';
    output.flush();
    if (!output) {
        throw std::runtime_error("Writing engine config failed: " +
                                 PathForDisplay(path));
    }
}
Save only when settings change

The checkpoint below writes once to prove the round trip. In the finished engine, save after the user applies settings or during a clean shutdown—not in the frame loop and not unconditionally on every startup. For crash-safe production saves, write and flush a temporary file, then replace the old settings file; the exact atomic replacement call differs between Windows and macOS.

Prove the round trip

Temporarily use Sandbox as a small test. It loads the complete user settings file when present, otherwise the shipped default; writes the typed result to the user path; reloads it; and checks that the meaningful data survived.

#include <SDL3/SDL.h>
#include <SDL3/SDL_main.h>
#include "Core/EngineConfig.h"
#include <exception>
#include <iostream>

int main(int argc, char* argv[]) {
    try {
        const EngineConfig config = LoadStartupConfig();
        const std::filesystem::path userPath = UserEngineConfigPath();

        SaveEngineConfig(config, userPath);
        const EngineConfig reloaded = LoadEngineConfig(userPath);
        if (reloaded != config) {
            std::cerr << "YAML round-trip changed a setting\n";
            return 1;
        }

        std::cout << "Loaded " << config.window.title << ' '
                  << config.window.width << 'x'
                  << config.window.height << "\nSaved "
                  << userPath << '\n';
        return 0;
    } catch (const std::exception& error) {
        std::cerr << "Configuration error: " << error.what() << '\n';
        return 1;
    }
}
Loaded MiniEngine 1280x720
Saved "/the/platform-specific/user/folder/settings.yaml"

Process finished with exit code 0

Recap
  • YAML is appropriate for small human-readable settings; networking and large save data still need purpose-built formats.
  • yaml-cpp stays private to EngineConfig.cpp; the public API contains ordinary typed C++.
  • Missing optional keys use defaults, malformed values fail visibly, and typed values are range-checked.
  • CMake copies shipped defaults beside Sandbox; SDL3 finds that file and a separate writable user folder on both operating systems.
  • Writing constructs a new document, so values round-trip but original comments and formatting do not.
  • Load configuration once at startup and pass it to systems—never parse YAML inside the frame loop.
