The Application & Window Classes

Last lesson we carved the engine into folders and a build system. Now we start filling those folders with real code. In this lesson we wrap the raw Win32 window into a tidy Window class, then build an Application base class with an Initialize / Run / Shutdown lifecycle and virtual OnUpdate and OnRender hooks. This is the exact pattern Unity and Unreal use: the framework owns the loop, and your game just overrides a couple of methods.

What you'll learn
  • Encapsulate an HWND, its window class registration, and its message procedure inside a reusable Window class.
  • Route Win32's C-style WndProc callback back into a C++ object instance.
  • Design an Application base class with a clean lifecycle and virtual OnUpdate(dt) / OnRender hooks.
  • Write a non-blocking Run loop that pumps messages with PeekMessage and ticks every frame with a real delta time.

Why wrap Win32 at all?

In earlier lessons you called RegisterClassEx, CreateWindowEx, and ran a GetMessage loop directly in WinMain. That works, but it scatters global state and mixes platform plumbing with game logic. An engine needs a clean seam: the messy operating-system details live behind a class, and everything above it stays portable and readable.

Our target design is two small classes:

  • Window — owns one HWND, registers the window class, creates the window, and translates Win32 messages into simple state (like "should we close?").
  • Application — owns a Window, drives the frame loop, and exposes virtual methods your game subclass fills in.
Separation of concerns

The rule of thumb: the Window class knows about Win32; the Application class knows about frames. Your game code should know about neither. That boundary is what lets us swap in DirectX next lesson without touching game logic.

The Window class

The tricky part of Win32 is that the message procedure — WndProc — is a plain C function pointer. It has no this, so it cannot directly reach our object's members. The standard trick is to stash a pointer to our Window instance inside the OS window's user data with SetWindowLongPtr(GWLP_USERDATA), then read it back on every message.

#pragma once
#define WIN32_LEAN_AND_MEAN
#include <windows.h>
#include <string>

class Window {
public:
    // Creates and shows the window. Returns false on failure.
    bool Create(const std::wstring& title, int width, int height);
    void Destroy();

    // Pull all pending OS messages out of the queue.
    // Returns false when the user asked to quit (WM_QUIT / close).
    bool ProcessMessages();

    HWND Handle() const { return m_hwnd; }
    int  Width()  const { return m_width;  }
    int  Height() const { return m_height; }

private:
    // The real per-instance message handler.
    LRESULT HandleMessage(UINT msg, WPARAM wParam, LPARAM lParam);

    // Static thunk that Win32 calls; it forwards to HandleMessage.
    static LRESULT CALLBACK WndProc(HWND hwnd, UINT msg,
                                    WPARAM wParam, LPARAM lParam);

    HWND        m_hwnd  = nullptr;
    HINSTANCE   m_hinst = nullptr;
    int         m_width  = 0;
    int         m_height = 0;
    bool        m_shouldClose = false;

    static constexpr const wchar_t* kClassName = L"EngineWindowClass";
};

Now the implementation. Watch how the static WndProc recovers the this pointer and delegates to the member function.

#include "Window.h"

bool Window::Create(const std::wstring& title, int width, int height) {
    m_hinst  = GetModuleHandle(nullptr);
    m_width  = width;
    m_height = height;

    // Register the window class once.
    WNDCLASSEX wc = {};
    wc.cbSize        = sizeof(WNDCLASSEX);
    wc.style         = CS_HREDRAW | CS_VREDRAW;
    wc.lpfnWndProc   = &Window::WndProc;      // static thunk
    wc.hInstance     = m_hinst;
    wc.hCursor       = LoadCursor(nullptr, IDC_ARROW);
    wc.lpszClassName = kClassName;
    RegisterClassEx(&wc);                     // harmless if already registered

    // We want a client area of exactly width x height, so adjust for borders.
    RECT rc = { 0, 0, width, height };
    DWORD style = WS_OVERLAPPEDWINDOW;
    AdjustWindowRect(&rc, style, FALSE);

    // Pass "this" through the last CreateWindowEx parameter so the very
    // first messages can find our object (see WM_NCCREATE below).
    m_hwnd = CreateWindowEx(
        0, kClassName, title.c_str(), style,
        CW_USEDEFAULT, CW_USEDEFAULT,
        rc.right - rc.left, rc.bottom - rc.top,
        nullptr, nullptr, m_hinst, this);

    if (!m_hwnd) return false;

    ShowWindow(m_hwnd, SW_SHOW);
    UpdateWindow(m_hwnd);
    return true;
}

void Window::Destroy() {
    if (m_hwnd) {
        DestroyWindow(m_hwnd);
        m_hwnd = nullptr;
    }
}

// Drain the message queue without blocking. PeekMessage returns immediately
// when the queue is empty, so the caller can keep rendering.
bool Window::ProcessMessages() {
    MSG msg = {};
    while (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE)) {
        if (msg.message == WM_QUIT)
            m_shouldClose = true;
        TranslateMessage(&msg);
        DispatchMessage(&msg);   // routes to WndProc
    }
    return !m_shouldClose;
}

LRESULT Window::HandleMessage(UINT msg, WPARAM wParam, LPARAM lParam) {
    switch (msg) {
        case WM_CLOSE:
            m_shouldClose = true;
            return 0;
        case WM_DESTROY:
            PostQuitMessage(0);
            return 0;
        case WM_SIZE:
            m_width  = LOWORD(lParam);
            m_height = HIWORD(lParam);
            return 0;
    }
    return DefWindowProc(m_hwnd, msg, wParam, lParam);
}

LRESULT CALLBACK Window::WndProc(HWND hwnd, UINT msg,
                                 WPARAM wParam, LPARAM lParam) {
    // On the first creation message, store the "this" pointer we passed in.
    if (msg == WM_NCCREATE) {
        auto* cs   = reinterpret_cast<CREATESTRUCT*>(lParam);
        auto* self = static_cast<Window*>(cs->lpCreateParams);
        self->m_hwnd = hwnd;
        SetWindowLongPtr(hwnd, GWLP_USERDATA,
                         reinterpret_cast<LONG_PTR>(self));
    }

    // For every later message, fetch it back and forward to the instance.
    auto* self = reinterpret_cast<Window*>(
        GetWindowLongPtr(hwnd, GWLP_USERDATA));
    if (self)
        return self->HandleMessage(msg, wParam, lParam);

    return DefWindowProc(hwnd, msg, wParam, lParam);
}
Why WM_NCCREATE, not WM_CREATE?

WM_NCCREATE is the first message a window receives, arriving before WM_CREATE. Storing our pointer there means every subsequent message — including WM_CREATE itself — can find the object. Both messages carry the CREATESTRUCT with our lpCreateParams.

PeekMessage vs. GetMessage

A game must not block. GetMessage sleeps until a message arrives — fine for a text editor, fatal for a render loop. PeekMessage with PM_REMOVE returns immediately when the queue is empty, letting us fall through to rendering every frame regardless of user input.

The Application base class

With the window handled, the Application class becomes small and expressive. It owns the lifecycle and defines the virtual hooks your game overrides. OnUpdate receives a delta time (seconds since last frame) so movement is frame-rate independent — exactly what we'll need when a Transform component integrates velocity later.

#pragma once
#include "Window.h"
#include <string>

class Application {
public:
    Application(const std::wstring& title, int width, int height);
    virtual ~Application() = default;

    // The template method that runs the whole show.
    int Run();

protected:
    // Game subclasses override these. Base versions do nothing.
    virtual bool OnInitialize() { return true; }
    virtual void OnUpdate(float dt) { (void)dt; }
    virtual void OnRender() {}
    virtual void OnShutdown() {}

    Window m_window;

private:
    bool Initialize();
    void Shutdown();

    std::wstring m_title;
    int m_width;
    int m_height;
};
#include "Application.h"
#include <chrono>

Application::Application(const std::wstring& title, int width, int height)
    : m_title(title), m_width(width), m_height(height) {}

bool Application::Initialize() {
    if (!m_window.Create(m_title, m_width, m_height))
        return false;
    // Later: create the D3D11 device + swap chain here.
    return OnInitialize();          // let the game set itself up
}

void Application::Shutdown() {
    OnShutdown();                   // game cleans up first...
    m_window.Destroy();             // ...then we tear down the window
}

int Application::Run() {
    if (!Initialize())
        return -1;

    using clock = std::chrono::high_resolution_clock;
    auto previous = clock::now();

    // Main loop: pump messages, then update + render one frame.
    while (m_window.ProcessMessages()) {
        auto now = clock::now();
        std::chrono::duration<float> elapsed = now - previous;
        previous = now;
        float dt = elapsed.count();  // seconds since last frame

        OnUpdate(dt);
        OnRender();
    }

    Shutdown();
    return 0;
}

Notice the shape: Run is a template method. It fixes the order of operations — initialize, loop (update then render), shut down — while delegating the content of each step to virtual functions. The framework is in charge; the game fills in the blanks. This inversion of control is the beating heart of every engine.

Putting it together

Here is what a game subclass and its entry point look like. This is a complete, compilable program: it opens a window and ticks a frame loop. Right now OnRender does nothing visible — next lesson the renderer fills it in.

#include "Application.h"

// A concrete game just derives from Application and overrides hooks.
class Game : public Application {
public:
    Game() : Application(L"My Engine", 1280, 720) {}

protected:
    bool OnInitialize() override {
        // Load assets, create the scene, etc.
        return true;
    }

    void OnUpdate(float dt) override {
        m_time += dt;   // frame-rate independent: works at 30 or 300 FPS
        // Later: update Transform components, run systems...
    }

    void OnRender() override {
        // Later: clear the back buffer, draw the scene, present.
    }

private:
    float m_time = 0.0f;
};

// Windows GUI entry point. WinMain is where a desktop app begins.
int WINAPI wWinMain(HINSTANCE, HINSTANCE, LPWSTR, int) {
    Game game;
    return game.Run();
}
Look how little the game knows

Game never mentions HWND, PeekMessage, or WinMain mechanics. It just overrides three methods. That is the payoff of the wrapper: from here on, gameplay code reads like gameplay, not operating-system glue.

The two classes map cleanly onto the folders you created last lesson: Window.h/.cpp and Application.h/.cpp live under your core engine module, while main.cpp with the Game subclass sits in the app that links against it. Building on those, next lesson we give the Application a renderer so OnRender can finally draw something to the screen.

One entry point only

Use wWinMain (wide) or WinMain (narrow), never both, and make sure the linker subsystem is set to Windows, not Console. A mismatch produces the classic "unresolved external symbol WinMain" link error that trips up nearly everyone the first time.

Recap
  • The Window class owns one HWND, registers the class, and hides Win32 message plumbing behind Create, ProcessMessages, and Destroy.
  • A static WndProc stores the object's this pointer in GWLP_USERDATA during WM_NCCREATE, then forwards every later message to the instance's HandleMessage.
  • PeekMessage with PM_REMOVE keeps the loop non-blocking so we render every frame; GetMessage would stall waiting for input.
  • The Application base class fixes the Initialize / Run / Shutdown order and delegates behavior to virtual OnInitialize, OnUpdate(dt), OnRender, and OnShutdown hooks.
  • A game subclass overrides just those hooks — no Win32 in sight — which is the same inversion of control Unity and Unreal are built on.
