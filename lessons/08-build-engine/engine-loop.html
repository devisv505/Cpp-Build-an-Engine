Tying It Together: the Engine Loop

You have built every piece separately: a window, a Renderer, an Input system, a Time clock, GameObjects with Components, a Mesh, a Material, and a Camera. This is the lesson where they finally click together. We'll write the Application that owns all the subsystems and runs one clean frame loop — and by the end you'll have a spinning cube drawn by a real, if small, engine.

What you'll learn
  • How the Application owns the Renderer, Input, Time and the active Scene.
  • The anatomy of one frame: poll input, update the scene by delta time, then render.
  • How the Scene walks every GameObject and Component each frame.
  • How the Renderer finds the active Camera and draws each mesh with its world matrix.
  • A complete main() that ties it all together into a spinning cube.

The shape of a frame

Every game engine, from a hobby project to Unreal, is a loop that does the same three things over and over, thousands of times per second:

  1. Gather input — what did the player press since last frame?
  2. Update — advance the simulation by the elapsed time (dt, the delta time).
  3. Render — draw the current state of the world to the screen.

That's it. Everything else — physics, animation, AI, spinning cubes — hangs off those three steps. Our job now is to give each step a clear owner and a clear order.

Who owns what

The Application is the top of the tree. It owns the long-lived subsystems for the whole program's life, and it owns the Scene — a plain list of the GameObjects currently in the world. Nothing above the Application; everything below is reachable from it.

OwnerResponsibility
ApplicationCreates the window, owns all subsystems, runs the loop.
RendererOwns the D3D11 device/context/swap chain; draws meshes.
InputTracks which keys and mouse buttons are down this frame.
TimeMeasures delta time between frames.
SceneHolds the list of GameObjects; updates them all.

Recall from earlier lessons that a GameObject owns a list of Components, always including a Transform. A MeshRenderer is just another Component that holds a pointer to a shared Mesh and Material. The Camera is a Component too. That uniformity is the whole point of the design — the loop below never needs a special case for any object type.

The Scene: update everything

The Scene's job during update is simple: walk every GameObject, and for each one give every Component a chance to run its per-frame logic. Here is the Scene and the base Component contract.

#pragma once
#include <vector>
#include <memory>
#include "GameObject.h"

class Scene {
public:
    // Create a GameObject, own it, and return a raw pointer for convenience.
    GameObject* CreateObject(const std::string& name) {
        auto obj = std::make_unique<GameObject>(name);
        GameObject* ptr = obj.get();
        m_objects.push_back(std::move(obj));
        return ptr;
    }

    // Advance every object (and thus every component) by dt seconds.
    void Update(float dt) {
        for (auto& obj : m_objects)
            obj->Update(dt);
    }

    // Read-only access so the Renderer can iterate the world.
    const std::vector<std::unique_ptr<GameObject>>& Objects() const {
        return m_objects;
    }

private:
    std::vector<std::unique_ptr<GameObject>> m_objects;
};

Each GameObject::Update loops over its components and calls their virtual Update(dt). A component that has nothing to do (like a static Mesh) simply doesn't override it. A component that spins the cube overrides it and rotates its Transform:

#pragma once
#include "Component.h"
#include "Transform.h"

// A tiny gameplay component: rotate the owner around Y every frame.
class Spinner : public Component {
public:
    float speed = 1.0f; // radians per second

    void Update(float dt) override {
        // GetTransform() is provided by Component via its owning GameObject.
        Transform* t = GetTransform();
        t->rotation.y += speed * dt; // frame-rate independent thanks to dt
    }
};
Always multiply by dt

Notice speed * dt, not just speed. Multiplying movement and rotation by delta time makes your game run at the same real-world speed whether it renders at 60 or 300 frames per second. Forget it and your cube spins faster on a better PC.

The Renderer: draw everything

Rendering is where the Camera and the meshes meet. The Renderer opens the frame, works out the camera's view-projection matrix once, then draws each GameObject that has both a MeshRenderer and a Transform. For each one it uploads the object's world matrix and the shared view-projection into a constant buffer, binds the material, and issues the draw call.

// Per-frame constant buffer layout — must match the HLSL cbuffer exactly.
struct FrameConstants {
    DirectX::XMFLOAT4X4 world;          // this object's model-to-world
    DirectX::XMFLOAT4X4 viewProjection; // shared camera matrix
};

void Renderer::DrawScene(const Scene& scene, Camera* camera) {
    using namespace DirectX;

    // 1) Compute view-projection once for the whole frame.
    XMMATRIX view = camera->GetViewMatrix();
    XMMATRIX proj = camera->GetProjectionMatrix(AspectRatio());
    XMMATRIX viewProj = XMMatrixMultiply(view, proj);

    // 2) Draw every object that can be drawn.
    for (const auto& obj : scene.Objects()) {
        MeshRenderer* mr = obj->GetComponent<MeshRenderer>();
        if (!mr || !mr->mesh || !mr->material)
            continue; // no visual — skip (e.g. the camera object)

        Transform* t = obj->GetComponent<Transform>();

        // Fill and upload the per-object constants.
        FrameConstants cb{};
        XMStoreFloat4x4(&cb.world, XMMatrixTranspose(t->GetWorldMatrix()));
        XMStoreFloat4x4(&cb.viewProjection, XMMatrixTranspose(viewProj));
        m_context->UpdateSubresource(m_frameCB.Get(), 0, nullptr, &cb, 0, 0);

        // Bind material (shaders + input layout) and the mesh buffers, then draw.
        mr->material->Bind(m_context.Get());
        mr->mesh->Bind(m_context.Get());
        m_context->DrawIndexed(mr->mesh->IndexCount(), 0, 0);
    }
}
Why the transpose?

HLSL constant buffers expect column-major matrices by default, but DirectXMath is row-major. XMMatrixTranspose bridges the two. If your cube renders as a garbled mess, a missing transpose is the usual culprit.

The matching vertex shader multiplies each vertex by world then view-projection, exactly the two matrices we uploaded:

cbuffer FrameConstants : register(b0)
{
    float4x4 world;
    float4x4 viewProjection;
};

struct VSInput  { float3 pos : POSITION; float3 color : COLOR; };
struct VSOutput { float4 pos : SV_POSITION; float3 color : COLOR; };

VSOutput VSMain(VSInput input)
{
    VSOutput o;
    float4 worldPos = mul(float4(input.pos, 1.0), world);
    o.pos   = mul(worldPos, viewProjection); // world -> clip space
    o.color = input.color;
    return o;
}

float4 PSMain(VSOutput input) : SV_TARGET
{
    return float4(input.color, 1.0);
}

The Application: one loop to run them all

Now we assemble the owners. The Application holds the subsystems, builds a Scene with a cube and a camera, and runs the loop. The frame ordering is deliberate and worth memorising: input, then update, then render. Input must be fresh before update reads it, and update must finish before render draws the result.

#include "Application.h"
#include "Renderer.h"
#include "Input.h"
#include "Time.h"
#include "Scene.h"
#include "MeshRenderer.h"
#include "Camera.h"
#include "Spinner.h"

int Application::Run() {
    // --- Subsystems owned for the whole program's lifetime ---
    m_window.Create(L"My Engine", 1280, 720);
    m_renderer.Initialize(m_window.Handle(), 1280, 720);
    m_input.Initialize();
    m_time.Reset();

    // --- Build the scene ---
    Scene scene;

    // A camera object: Transform + Camera component.
    GameObject* camObj = scene.CreateObject("MainCamera");
    camObj->GetComponent<Transform>()->position = { 0.0f, 1.5f, -5.0f };
    Camera* camera = camObj->AddComponent<Camera>();
    camera->fovYRadians = DirectX::XMConvertToRadians(60.0f);

    // A cube object: Transform + MeshRenderer + Spinner.
    GameObject* cube = scene.CreateObject("Cube");
    MeshRenderer* mr = cube->AddComponent<MeshRenderer>();
    mr->mesh     = Mesh::CreateCube(m_renderer.Device());
    mr->material = Material::Create(m_renderer.Device(), L"cube.hlsl");
    cube->AddComponent<Spinner>()->speed = 1.2f;

    // --- The frame loop ---
    while (!m_window.ShouldClose()) {
        m_window.PumpMessages();          // OS events -> keep window alive
        float dt = m_time.Tick();         // seconds since last frame

        m_input.Update();                 // 1. snapshot keyboard/mouse
        scene.Update(dt);                 // 2. update every GameObject/Component

        m_renderer.BeginFrame(0.1f, 0.1f, 0.12f); // 3a. clear the back buffer
        m_renderer.DrawScene(scene, camera);      // 3b. draw the world
        m_renderer.EndFrame();                    // 3c. present the swap chain
    }
    return 0;
}
Order matters

If you render before you update, you draw the previous frame's positions — everything appears one frame stale. If you update before polling input, your controls feel laggy. Keep the sequence Input → Update → Render and both problems disappear.

Finding the active camera

Here we pass the camera pointer in directly because we know it. In a bigger engine you'd let the Scene find it: iterate the GameObjects, ask each for a Camera component, and use the first one flagged active. That keeps Run() free of hard-coded object knowledge and lets designers add or swap cameras in data. The principle is the same — the Renderer needs exactly one camera's view-projection per frame.

What you actually built

Step back and look at what that loop represents. You have a data-driven world (the Scene), uniform objects made of composable Components, a subsystem that advances simulation by real time, and a subsystem that turns world state into pixels — with a clean seam between them. That is the same architecture Unity calls GameObjects and MonoBehaviours, and Unreal calls Actors and Components. Yours is smaller, but it is genuinely the same idea, and every feature from here (lighting, multiple meshes, physics) is just another Component or another draw in the same loop.

Recap
  • The Application owns the Renderer, Input, Time and the active Scene for the whole program.
  • One frame is always Input.Update → Scene.Update(dt) → Render, in that order.
  • Scene.Update(dt) walks every GameObject, which updates each of its Components; multiply motion by dt for frame-rate independence.
  • The Renderer computes the camera's view-projection once, then draws each object using its world matrix uploaded to a constant buffer.
  • Transpose matrices going into HLSL, and match the cbuffer layout to the shader — the two most common bugs.
  • This small skeleton mirrors how Unity and Unreal are architected: composable Components running inside one central loop.
