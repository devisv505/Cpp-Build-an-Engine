GameObjects & Components

Last lesson we built a Transform that knows its position, rotation and scale, and can be parented to another transform. Now we assemble the heart of the engine: the GameObject. Just like Unity, a GameObject is mostly an empty shell — a container that owns a Transform and a list of Components that give it behaviour. This "composition over inheritance" design is what lets a designer build a spinning enemy, a camera or a glowing pickup from the same small set of reusable parts.

What you'll learn
  • Why real engines favour composition (components) over deep inheritance trees.
  • How to write a Component base class with virtual OnUpdate/OnRender hooks and an owner pointer.
  • How a GameObject owns its parts with std::vector<std::unique_ptr<Component>>.
  • How to write templated AddComponent<T>(args...) and GetComponent<T>() using perfect forwarding and dynamic_cast.

Composition, not a giant inheritance tree

A tempting first design is to make everything inherit from a base Entity: Player extends Entity, Enemy extends Entity, and so on. That falls apart fast. What about a flying, shooting enemy? A player that can also be a camera target? Deep inheritance forces every combination into one rigid line, and you quickly get "diamond" problems and copy-pasted code.

Unity and Unreal solve this with composition. A GameObject has behaviours rather than being a specific type. Want it to move? Add a MoverComponent. Want it to draw a mesh? Add a MeshRenderer. Want a camera? Add a CameraComponent (next lesson). The GameObject itself stays tiny and generic — the components carry the logic.

The mental model

A GameObject is a noun ("the goblin"); components are its verbs and adjectives ("moves", "renders", "is on fire"). You describe an object by the components you bolt onto it.

The Component base class

Every component shares a common interface so the GameObject can treat them uniformly. We give it two virtual hooks — OnUpdate(float dt) for per-frame logic and OnRender() for drawing — plus a back-pointer to the GameObject that owns it. That owner pointer is how a component reaches its own Transform or finds sibling components. 

#pragma once

class GameObject; // forward declaration — we only need a pointer here

// Base class for every behaviour that can live on a GameObject.
class Component {
public:
    virtual ~Component() = default; // virtual dtor: safe deletion through base pointer

    // Called once per frame with the frame time in seconds.
    virtual void OnUpdate(float dt) {}

    // Called during the render pass. Empty by default so
    // non-visual components (movers, scripts) can ignore it.
    virtual void OnRender() {}

    // The GameObject that owns this component. Set by AddComponent.
    GameObject* owner = nullptr;
};
Why the virtual destructor matters

We store components as unique_ptr<Component> but each one is really a derived type. When the vector is destroyed it deletes through the base pointer. Without virtual ~Component(), the derived destructor would be skipped — a classic memory bug. This is exactly the polymorphism rule from the earlier virtual-functions lesson.

The GameObject: a Transform plus a bag of components

Now the container itself. It owns exactly one Transform (every object has a place in the world) and a std::vector of components. We use std::unique_ptr so the GameObject has clear, single ownership: when the GameObject dies, its components die with it, and we never leak or double-free. 

#pragma once
#include <vector>
#include <memory>
#include <string>
#include <utility>      // std::forward, std::move
#include <type_traits> // std::is_base_of
#include "Transform.h"
#include "Component.h"

class GameObject {
public:
    explicit GameObject(std::string name = "GameObject")
        : m_name(std::move(name)) {}

    // Create a component of type T in place, wire up its owner,
    // and hand back a raw pointer so the caller can configure it.
    template <typename T, typename... Args>
    T* AddComponent(Args&&... args) {
        // Compile-time guard: T must actually be a Component.
        static_assert(std::is_base_of<Component, T>::value,
                      "T must derive from Component");

        auto comp = std::make_unique<T>(std::forward<Args>(args)...);
        comp->owner = this;          // let the component find us back
        T* raw = comp.get();          // borrow a non-owning pointer
        m_components.push_back(std::move(comp)); // vector now owns it
        return raw;
    }

    // Return the first component of type T, or nullptr if none.
    template <typename T>
    T* GetComponent() {
        for (const auto& c : m_components) {
            // dynamic_cast succeeds only if the real type is a T.
            if (T* hit = dynamic_cast<T*>(c.get()))
                return hit;
        }
        return nullptr;
    }

    // Per-frame tick: forward to every component.
    void Update(float dt) {
        for (const auto& c : m_components)
            c->OnUpdate(dt);
    }

    // Render pass: forward to every component.
    void Render() {
        for (const auto& c : m_components)
            c->OnRender();
    }

    Transform& GetTransform()       { return m_transform; }
    const std::string& GetName() const { return m_name; }

private:
    std::string m_name;
    Transform   m_transform; // every GameObject has exactly one
    std::vector<std::unique_ptr<Component>> m_components;
};

Reading the two templates

AddComponent<T>(args...) is a variadic template. The Args&&... parameters and std::forward are perfect forwarding: whatever arguments you pass are handed straight to T's constructor without extra copies. So AddComponent<MoverComponent>(5.0f) constructs a MoverComponent(5.0f) directly inside the vector.

GetComponent<T>() walks the list and uses dynamic_cast, which returns a valid pointer only when the stored object really is a T (or derived from it), and nullptr otherwise. That is exactly how Unity's GetComponent<Rigidbody>() behaves.

Never store the raw pointer past the GameObject's life

AddComponent and GetComponent return non-owning raw pointers — fine for reading and configuring right now, but the unique_ptr in the vector is the real owner. If the GameObject is destroyed, that pointer dangles. Never delete it yourself, and never keep it around longer than the GameObject.

A concrete component

Let's write a tiny MoverComponent that spins its owner every frame. Notice how it reaches the owner's Transform through the owner pointer — no globals, no casting gymnastics. 

#pragma once
#include "Component.h"
#include "GameObject.h"

// Rotates its owner around the Y axis at a fixed speed.
class MoverComponent : public Component {
public:
    explicit MoverComponent(float degreesPerSecond)
        : m_speed(degreesPerSecond) {}

    void OnUpdate(float dt) override {
        // Reach into the GameObject we belong to and nudge its rotation.
        Transform& t = owner->GetTransform();
        t.rotationY += m_speed * dt; // dt keeps motion frame-rate independent
    }

private:
    float m_speed;
};

Putting it together

Here is the whole pattern in action: build a GameObject, bolt components onto it, then run a simple update/render loop. This is the skeleton your engine's scene loop will grow from. 

#include <iostream>
#include "GameObject.h"
#include "MoverComponent.h"

int main() {
    GameObject enemy("Goblin");

    // Compose behaviour: this goblin spins at 90 degrees/second.
    enemy.AddComponent<MoverComponent>(90.0f);

    // Later we can ask for it back by type.
    if (MoverComponent* m = enemy.GetComponent<MoverComponent>())
        std::cout << enemy.GetName() << " has a mover.\n";

    // A fake game loop: 3 frames at ~60 FPS (dt = 1/60 second).
    const float dt = 1.0f / 60.0f;
    for (int frame = 0; frame < 3; ++frame) {
        enemy.Update(dt);   // ticks every component
        enemy.Render();     // draws every component

        float y = enemy.GetTransform().rotationY;
        std::cout << "Frame " << frame
                  << " rotationY = " << y << " degrees\n";
    }
    return 0;
}

Because dt is 1⁄60 of a second and the speed is 90 deg/s, each frame adds 1.5 degrees. Run it and you'll see the rotation climb 1.5, 3.0, 4.5 — motion that stays the same speed no matter the frame rate. When we render meshes with DirectX later, this same Transform becomes a world matrix fed to the vertex shader.

Update vs. Render — why two passes?

PassRunsJob
OnUpdate(dt)Once per frame, before drawingChange state: move, rotate, run gameplay logic, read input.
OnRender()Once per frame, after all updatesSubmit draw calls using the now-final state.

Splitting the two keeps rendering clean: everything finishes moving first, so no object draws itself halfway between two states. A camera can safely read every transform during render because updates are already done.

Don't try to copy a GameObject

std::unique_ptr is move-only, so a GameObject that holds a vector of them is non-copyable too. Writing GameObject a = b; will fail to compile — and that's a feature. Pass GameObjects by reference or store them in a scene container (we'll build that soon); never copy them around by value.

How this echoes Unity

If you've touched Unity, this will feel familiar: GameObject.AddComponent<T>(), GetComponent<T>(), a Transform that's always present, and MonoBehaviour's Update() callback. We've just built the C++ bones of that same idea. In the coming lessons we'll add a scene that owns many GameObjects, then a CameraComponent that produces the view and projection matrices DirectX needs.

Recap
  • Engines favour composition: a GameObject has components instead of inheriting one rigid type.
  • The Component base class defines virtual OnUpdate(dt)/OnRender() hooks and an owner back-pointer — with a virtual destructor for safe deletion.
  • A GameObject owns one Transform plus std::vector<std::unique_ptr<Component>>, giving clear single ownership.
  • AddComponent<T>(args...) uses variadic templates and perfect forwarding; GetComponent<T>() uses dynamic_cast to find a component by type.
  • Update and Render simply iterate the components — the same loop your scene and camera will build on next.
