Abstracting the Renderer

Over the last few lessons you learned how to create a Direct3D 11 device, build a swap chain, clear the screen, and add a depth buffer — one long, ceremony-heavy block of setup code. In this lesson we fold all of that into a single GraphicsDevice class. Instead of scattering raw DirectX calls through your game loop, you will call BeginFrame() and EndFrame(). This is exactly how real engines like Unity and Unreal hide the graphics API behind a clean surface.

What you'll learn
  • Why we wrap the raw D3D11 objects behind one Renderer class.
  • How Microsoft::WRL::ComPtr frees GPU resources automatically.
  • Building Initialize, BeginFrame, and EndFrame around the device, context, swap chain, RTV, and DSV.
  • Exposing the device and context so other systems can issue draw calls.
  • Handling window resize cleanly by recreating the render targets.

Why abstract the renderer at all?

Direct3D is powerful, but it is verbose. Creating a device and swap chain, fishing the back buffer out of that swap chain, wrapping it in a render target view, building a matching depth texture — that is dozens of lines before a single triangle appears. If we leave all of it in main(), every new feature makes that function longer and more fragile.

The fix is an abstraction: a class that owns the graphics objects and offers a small, friendly set of methods. Callers say "begin the frame, clear to this color" and "end the frame, show it on screen." They never touch a raw ID3D11DeviceContext pointer unless they truly need one. When you later add DirectX 12 support or a debug overlay, you change one class instead of your whole codebase.

A name that describes a role

We call the class GraphicsDevice because it represents "the thing that talks to the GPU." Some engines call it Renderer or RHI (render hardware interface). The name matters less than the boundary it draws.

ComPtr: no more manual Release()

Every DirectX object you create — the device, the swap chain, each view — is a COM object with an internal reference count. In older tutorials you must call Release() on each one in the right order, or you leak GPU memory. We are past that. Microsoft::WRL::ComPtr is a smart pointer for COM objects: when it goes out of scope it calls Release() for you, just like std::unique_ptr calls delete. Store every DirectX object in a ComPtr and destruction becomes automatic and correct.

ComPtr callWhat it gives you
.Get()The raw pointer, for passing into a D3D call.
.GetAddressOf()Address of the pointer, for a call that creates an object (an out-parameter).
.Reset()Releases the object now and sets the pointer to null.

The header

Here is the interface. Notice how small it is compared to the setup work it hides. The private members are all ComPtrs, so the class needs no destructor at all. 

#pragma once
#include <d3d11.h>
#include <wrl/client.h>   // Microsoft::WRL::ComPtr
#include <cstdint>

class GraphicsDevice {
public:
    // Create the device, context, swap chain and back-buffer targets.
    bool Initialize(HWND window, uint32_t width, uint32_t height);

    // Bind the render target and wipe it (plus depth) for a new frame.
    void BeginFrame(const float clearColor[4]);

    // Present the finished frame to the screen.
    void EndFrame();

    // Recreate the buffers after the window changes size.
    void Resize(uint32_t width, uint32_t height);

    // Systems that record draw calls (mesh renderer, UI) grab these.
    ID3D11Device*        Device()  const { return m_device.Get(); }
    ID3D11DeviceContext* Context() const { return m_context.Get(); }

private:
    bool CreateRenderTargets(uint32_t width, uint32_t height);

    template <typename T>
    using ComPtr = Microsoft::WRL::ComPtr<T>;

    ComPtr<ID3D11Device>           m_device;
    ComPtr<ID3D11DeviceContext>    m_context;
    ComPtr<IDXGISwapChain>         m_swapChain;
    ComPtr<ID3D11RenderTargetView> m_renderTargetView;   // the RTV
    ComPtr<ID3D11DepthStencilView> m_depthStencilView;   // the DSV
    D3D11_VIEWPORT                 m_viewport{};
};

Initializing the device

Initialize does the one-time setup. It fills out a swap-chain description and creates the device, context, and swap chain together with D3D11CreateDeviceAndSwapChain. In a debug build we add the debug device flag so DirectX prints helpful validation messages to the output window. 

#include "GraphicsDevice.h"

using Microsoft::WRL::ComPtr;

bool GraphicsDevice::Initialize(HWND window, uint32_t width, uint32_t height) {
    // Describe the swap chain: one back buffer sized to the window.
    DXGI_SWAP_CHAIN_DESC scd{};
    scd.BufferCount       = 1;
    scd.BufferDesc.Width  = width;
    scd.BufferDesc.Height = height;
    scd.BufferDesc.Format = DXGI_FORMAT_R8G8B8A8_UNORM;
    scd.BufferUsage       = DXGI_USAGE_RENDER_TARGET_OUTPUT;
    scd.OutputWindow      = window;
    scd.SampleDesc.Count  = 1;      // no MSAA for now
    scd.Windowed          = TRUE;

    UINT flags = 0;
#if defined(_DEBUG)
    flags |= D3D11_CREATE_DEVICE_DEBUG; // extra validation in debug builds
#endif

    // Create device + context + swap chain in a single call.
    HRESULT hr = D3D11CreateDeviceAndSwapChain(
        nullptr,                     // default adapter (GPU)
        D3D_DRIVER_TYPE_HARDWARE,
        nullptr,
        flags,
        nullptr, 0,                  // default feature levels
        D3D11_SDK_VERSION,
        &scd,
        m_swapChain.GetAddressOf(),
        m_device.GetAddressOf(),
        nullptr,                     // we don't need the chosen level
        m_context.GetAddressOf());

    if (FAILED(hr)) return false;
    return CreateRenderTargets(width, height);
}

Splitting the target creation into its own helper is deliberate — we will call it again from Resize. It grabs the swap chain's back buffer, wraps it in a render target view, builds a matching depth/stencil texture and view, and sets the viewport. 

bool GraphicsDevice::CreateRenderTargets(uint32_t width, uint32_t height) {
    // 1) Wrap the swap chain's back buffer in a render target view.
    ComPtr<ID3D11Texture2D> backBuffer;
    HRESULT hr = m_swapChain->GetBuffer(
        0, __uuidof(ID3D11Texture2D),
        reinterpret_cast<void**>(backBuffer.GetAddressOf()));
    if (FAILED(hr)) return false;

    hr = m_device->CreateRenderTargetView(
        backBuffer.Get(), nullptr, m_renderTargetView.GetAddressOf());
    if (FAILED(hr)) return false;

    // 2) Create a matching depth/stencil texture, then a view onto it.
    D3D11_TEXTURE2D_DESC depthDesc{};
    depthDesc.Width            = width;
    depthDesc.Height           = height;
    depthDesc.MipLevels        = 1;
    depthDesc.ArraySize        = 1;
    depthDesc.Format           = DXGI_FORMAT_D24_UNORM_S8_UINT;
    depthDesc.SampleDesc.Count = 1;
    depthDesc.Usage            = D3D11_USAGE_DEFAULT;
    depthDesc.BindFlags        = D3D11_BIND_DEPTH_STENCIL;

    ComPtr<ID3D11Texture2D> depthTexture;
    hr = m_device->CreateTexture2D(
        &depthDesc, nullptr, depthTexture.GetAddressOf());
    if (FAILED(hr)) return false;

    hr = m_device->CreateDepthStencilView(
        depthTexture.Get(), nullptr, m_depthStencilView.GetAddressOf());
    if (FAILED(hr)) return false;

    // 3) Viewport covers the whole window; depth runs 0 -> 1.
    m_viewport.Width    = static_cast<float>(width);
    m_viewport.Height   = static_cast<float>(height);
    m_viewport.MinDepth = 0.0f;
    m_viewport.MaxDepth = 1.0f;
    m_context->RSSetViewports(1, &m_viewport);
    return true;
}
Link against d3d11.lib

These calls live in the Direct3D 11 runtime, so tell the linker to include it. Add #pragma comment(lib, "d3d11.lib") near the top of the .cpp, or add d3d11.lib to your linker inputs. Forget it and you get unresolved-symbol errors, not a runtime crash.

Begin and end a frame

This is the payoff. All the per-frame boilerplate — binding the targets, clearing color, clearing depth, presenting — collapses into two methods. Every frame of your game loop calls BeginFrame, records draw calls, then calls EndFrame. 

void GraphicsDevice::BeginFrame(const float clearColor[4]) {
    // Bind our render target + depth buffer for this frame...
    m_context->OMSetRenderTargets(
        1, m_renderTargetView.GetAddressOf(), m_depthStencilView.Get());

    // ...then wipe both so nothing from the last frame lingers.
    m_context->ClearRenderTargetView(m_renderTargetView.Get(), clearColor);
    m_context->ClearDepthStencilView(
        m_depthStencilView.Get(),
        D3D11_CLEAR_DEPTH | D3D11_CLEAR_STENCIL, 1.0f, 0);

    m_context->RSSetViewports(1, &m_viewport);
}

void GraphicsDevice::EndFrame() {
    // Swap the finished back buffer onto the screen. 1 = wait for vsync.
    m_swapChain->Present(1, 0);
}

And Resize, so a maximized or dragged window does not stretch a stale image. We must release the old views first — the swap chain cannot resize its buffers while anything still references them. 

void GraphicsDevice::Resize(uint32_t width, uint32_t height) {
    if (!m_swapChain) return;

    // Unbind and release the old views before resizing the buffers.
    m_context->OMSetRenderTargets(0, nullptr, nullptr);
    m_renderTargetView.Reset();
    m_depthStencilView.Reset();

    m_swapChain->ResizeBuffers(0, width, height, DXGI_FORMAT_UNKNOWN, 0);
    CreateRenderTargets(width, height); // rebuild RTV, DSV, viewport
}
Order of release matters

If you skip the Reset() calls, ResizeBuffers fails because the render target view still holds a reference to the old back buffer. This is one place where understanding COM reference counting pays off directly.

Using it in the loop

Remember the Application and Window classes from the previous lesson? The renderer slots right in beside them. Look how readable the loop becomes — no DirectX in sight except the color we choose. 

GraphicsDevice gfx;
if (!gfx.Initialize(window.Handle(), 1280, 720)) {
    return -1; // couldn't reach the GPU
}

const float sky[4] = { 0.10f, 0.12f, 0.18f, 1.0f }; // RGBA

while (window.ProcessMessages()) {
    gfx.BeginFrame(sky);
    //   ... later: systems record draw calls via gfx.Context() ...
    gfx.EndFrame();
}

That gfx.Context() accessor is our doorway to the future. When we build a mesh renderer, it will ask the GraphicsDevice for the context and issue its own draw calls — but it never needs to know how the swap chain was built or how depth is configured. The abstraction holds, and every new system we add leans on it.

Recap
  • A GraphicsDevice class owns the device, context, swap chain, RTV, and DSV, hiding raw D3D11 behind a small surface.
  • Microsoft::WRL::ComPtr releases every GPU resource automatically, so the class needs no destructor.
  • Initialize does one-time setup; CreateRenderTargets is shared with Resize.
  • BeginFrame binds and clears the targets; EndFrame presents the swap chain.
  • Device() and Context() let other engine systems record draw calls without touching setup code.
