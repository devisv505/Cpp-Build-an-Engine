Smart Pointers & Modern Memory

Manual memory management with new and delete is the classic way to leak memory and crash an engine. Modern C++ gives us smart pointers — small wrapper types that own memory for you and free it automatically. In this lesson we replace raw pointers with std::unique_ptr, std::shared_ptr and std::weak_ptr, the exact tools we'll lean on when our engine stores components, meshes and textures.

What you'll learn
  • Why RAII lets an object clean up its own memory — no manual delete.
  • How std::unique_ptr models sole ownership with std::make_unique.
  • How std::shared_ptr shares ownership through reference counting.
  • How std::weak_ptr breaks reference cycles that would otherwise leak.
  • How to convert a leaky raw-pointer example into safe, modern code.

The problem with raw pointers

When you write new, you promise the program you will call delete exactly once, later, on every path out of the function. Forget it and you leak memory; do it twice and you corrupt the heap; do it and then keep using the pointer and you get a crash. In a real engine that allocates thousands of objects per frame, this bookkeeping is impossible to do by hand.

Here is a small but realistic leak. Imagine a Mesh resource that we load and use:

#include <iostream>
#include <string>

struct Mesh {
    std::string name;
    explicit Mesh(std::string n) : name(std::move(n)) {
        std::cout << "Loaded mesh: " << name << "\n";
    }
    ~Mesh() {
        std::cout << "Freed mesh: " << name << "\n";
    }
};

void loadLevel() {
    Mesh* player = new Mesh("player");   // we own this now
    if (player->name.empty()) {
        return;                          // BUG: early return leaks 'player'
    }
    // ... use the mesh ...
    delete player;                       // only runs on the happy path
}

The early return skips delete, so the Mesh is never freed and its destructor never runs. Every leaked path like this is a slow drip that eventually exhausts memory. Smart pointers make this class of bug disappear.

What is RAII?

Resource Acquisition Is Initialization means a resource's lifetime is tied to an object's lifetime. When the object goes out of scope, its destructor runs and releases the resource. Smart pointers are RAII wrappers around heap memory — the whole idea is that cleanup happens automatically.

std::unique_ptr — sole ownership

A std::unique_ptr<T> owns a single heap object and frees it automatically when the unique_ptr is destroyed. It is unique: exactly one unique_ptr owns the object at a time, so you never wonder who is responsible for deletion. Always create one with std::make_unique, which allocates and constructs in one step.

Here is the leaky example rewritten. Notice there is no delete at all:

#include <iostream>
#include <memory>   // std::unique_ptr, std::make_unique
#include <string>

struct Mesh {
    std::string name;
    explicit Mesh(std::string n) : name(std::move(n)) {
        std::cout << "Loaded mesh: " << name << "\n";
    }
    ~Mesh() {
        std::cout << "Freed mesh: " << name << "\n";
    }
};

void loadLevel() {
    auto player = std::make_unique<Mesh>("player");  // owns the Mesh
    if (player->name.empty()) {
        return;   // safe: unique_ptr frees the Mesh on the way out
    }
    // ... use player like a normal pointer: player->name, *player ...
}   // player goes out of scope here -> ~Mesh() runs, no matter which path

int main() {
    loadLevel();
    return 0;
}

No matter how the function exits — normal return, early return, or an exception thrown midway — the unique_ptr's destructor runs and the Mesh is freed exactly once. You get the same output on every path.

Because a unique_ptr is the sole owner, you cannot copy it — that would create two owners. You can move it to transfer ownership, which is how we hand resources between systems:

auto a = std::make_unique<Mesh>("terrain");
// auto b = a;              // ERROR: cannot copy a unique_ptr
auto b = std::move(a);      // OK: ownership moves from a to b
// now 'a' is empty (nullptr), 'b' owns the terrain mesh
Make this your default

Reach for std::unique_ptr first — it is the cheapest smart pointer (no overhead versus a raw pointer) and it clearly documents ownership. Our engine will store each entity's components in a std::vector<std::unique_ptr<Component>>, so an entity owns its components and destroying it frees them all.

std::shared_ptr — shared ownership

Sometimes several parts of the engine genuinely need to share one object. A single Texture in GPU memory might be referenced by dozens of materials — none of them "the" owner. A std::shared_ptr<T> handles this by keeping a reference count: each copy increments it, each destruction decrements it, and when the count hits zero the object is freed. Create one with std::make_shared.

#include <iostream>
#include <memory>
#include <string>

struct Texture {
    std::string path;
    explicit Texture(std::string p) : path(std::move(p)) {
        std::cout << "GPU upload: " << path << "\n";
    }
    ~Texture() {
        std::cout << "GPU release: " << path << "\n";
    }
};

int main() {
    auto brick = std::make_shared<Texture>("brick.dds");
    std::cout << "count = " << brick.use_count() << "\n";  // 1

    {
        auto wallMaterial = brick;   // copy -> shares ownership
        std::cout << "count = " << brick.use_count() << "\n";  // 2
    }   // wallMaterial destroyed here -> count drops back to 1

    std::cout << "count = " << brick.use_count() << "\n";  // 1
    return 0;
}   // brick destroyed -> count hits 0 -> ~Texture() runs, GPU released

The Texture is uploaded once and released exactly once, at the moment the last owner disappears. That is exactly the behaviour a resource cache wants.

Shared is not free

A shared_ptr carries a small overhead: it allocates a control block for the count, and updating that count is thread-safe (atomic), which costs a little. Do not reach for it by default. Use it only when ownership is genuinely shared; otherwise a unique_ptr plus plain references or raw non-owning pointers is leaner.

std::weak_ptr — breaking cycles

Reference counting has one trap: a cycle. If object A holds a shared_ptr to B, and B holds a shared_ptr back to A, each keeps the other's count at 1 forever. Neither ever reaches zero, so neither is freed — a leak, even with smart pointers. This happens naturally in scene graphs, where a node points to its children and each child points back to its parent.

A std::weak_ptr<T> is a non-owning observer of a shared_ptr. It does not raise the reference count, so it cannot keep an object alive. To use the object you call lock(), which returns a shared_ptr if the object still exists, or an empty one if it has been freed. Make the "back" edge weak and the cycle is broken.

#include <iostream>
#include <memory>
#include <vector>
#include <string>

struct Node {
    std::string name;
    std::vector<std::shared_ptr<Node>> children;  // parent OWNS children
    std::weak_ptr<Node> parent;                  // child OBSERVES parent (no cycle)

    explicit Node(std::string n) : name(std::move(n)) {}
    ~Node() { std::cout << "Destroyed: " << name << "\n"; }
};

int main() {
    auto root  = std::make_shared<Node>("root");
    auto child = std::make_shared<Node>("child");

    root->children.push_back(child);   // strong: root keeps child alive
    child->parent = root;             // weak: does NOT keep root alive

    // Safely access the parent through the weak_ptr:
    if (auto p = child->parent.lock()) {
        std::cout << child->name << "'s parent is " << p->name << "\n";
    }
    return 0;   // both nodes are correctly destroyed, no leak
}

If the parent edge were a shared_ptr, root and child would keep each other alive forever and their destructors would never print. With weak_ptr the counts fall to zero cleanly.

Never dereference a weak_ptr directly

A weak_ptr has no -> or *, because the object it watches may already be gone. Always call lock() and check the result before use. Treating a dangling reference as valid is exactly the crash smart pointers exist to prevent.

Choosing the right pointer

Most ownership questions come down to a single decision. Use this table as a quick guide:

SituationUseWhy
One clear ownerstd::unique_ptrZero overhead, frees automatically
Genuinely shared ownershipstd::shared_ptrRef-counted; freed when last owner leaves
Observe without owningstd::weak_ptrBreaks cycles; check with lock()
Borrow, no ownership at allreference or raw T*Cheapest; caller must outlive the callee

These patterns are the backbone of the memory model we'll build on. An entity will hold unique_ptrs to the components it owns; a shared resource cache will hand out shared_ptrs to meshes and textures; and back-references in our scene graph will be weak_ptrs so parents and children can point at each other without leaking. From here on, you should rarely write new or delete again.

Recap
  • RAII ties a resource's lifetime to an object, so cleanup is automatic — smart pointers apply it to heap memory.
  • std::unique_ptr (via std::make_unique) is a zero-overhead sole owner; move it to transfer ownership, you can't copy it.
  • std::shared_ptr (via std::make_shared) uses a reference count and frees the object when the last owner is gone.
  • std::weak_ptr observes a shared_ptr without owning it; call lock() to use it safely and to break reference cycles.
  • Default to unique_ptr; use shared_ptr only for real shared ownership; use weak_ptr for back-references.
