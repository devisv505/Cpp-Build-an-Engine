Templates & Generic Programming

Last lesson you taught your own types how to behave with operators like + and ==. Now we go one level higher: writing code that works for any type without copy-pasting. Templates are how std::vector stores ints, floats, or your own Vector3 with a single implementation — and they are exactly how our engine will one day offer a clean GetComponent<Transform>().

What you'll learn
  • Why generics give you type-safe reuse instead of duplicated code
  • How to write function templates with template <typename T>
  • How template argument deduction figures out T for you
  • How to write a class template — a tiny generic Array<T>
  • That std::vector<T> is a class template, and how this leads to GetComponent<T>()

The problem: one idea, many types

Suppose you need a function that returns the larger of two values. You could write one for int, then another for double, then another for std::string. They would be identical except for the type. That is tedious, error-prone, and every bug fix has to be applied three times.

An older trick was to use macros, but macros are just blind text substitution — they don't understand types, don't respect scope, and give terrible error messages. Templates solve this properly: you write the logic once with a placeholder type, and the compiler stamps out a correct, fully type-checked version for each type you actually use.

Function templates

A function template starts with template <typename T>. Inside the function, T is a stand-in for whatever concrete type gets used. Here is a generic Max:

#include <iostream>
#include <string>

// A single template that works for any type T
template <typename T>
T Max(const T& a, const T& b) {
    return (a > b) ? a : b;   // requires that T supports operator>
}

int main() {
    std::cout << Max(3, 7) << "\n";            // T deduced as int    -> 7
    std::cout << Max(2.5, 1.5) << "\n";         // T deduced as double -> 2.5
    std::cout << Max(std::string("apple"),
                     std::string("pear")) << "\n"; // T is std::string -> pear
    return 0;
}

One template, three concrete functions generated on demand. Notice we take arguments by const T& (a const reference) so we don't copy large objects unnecessarily — the same habit you'd use for a Vector3.

typename vs class

You will see both template <typename T> and template <class T>. For a type parameter they mean exactly the same thing. This course prefers typename because it reads more clearly — T can be any type, not only a class.

Template argument deduction

You may have noticed we never wrote Max<int>(3, 7) — just Max(3, 7). The compiler looked at the arguments, saw two ints, and deduced that T is int. This is called template argument deduction, and it's why templates feel effortless most of the time.

Sometimes deduction can't decide — for example if the two arguments have different types. Then you can name the type explicitly:

// Ambiguous: is T int or double? The compiler refuses to guess.
// Max(3, 4.5);          // error

// Fix it by stating T yourself:
Max<double>(3, 4.5);      // force T = double, 3 becomes 3.0 -> 4.5
Templates are compile-time

A template is a recipe, not code by itself. The compiler generates real functions only for the types you actually call with. This means zero runtime cost for the abstraction — the generated Max<int> is exactly as fast as a hand-written one.

Class templates

Templates aren't just for functions. An entire class can be parameterized on a type. This is the big one, because it's how containers work. Let's build a tiny growable array — a stripped-down cousin of std::vector — to see the machinery.

#include <cstddef>   // for std::size_t

template <typename T>
class Array {
public:
    Array() : m_data(nullptr), m_size(0), m_capacity(0) {}

    ~Array() { delete[] m_data; }   // free the buffer we own

    void Add(const T& value) {
        if (m_size == m_capacity) {
            Grow();                 // make room before writing
        }
        m_data[m_size] = value;
        ++m_size;
    }

    // Access by index, both mutable and read-only
    T& operator[](std::size_t index)             { return m_data[index]; }
    const T& operator[](std::size_t index) const { return m_data[index]; }

    std::size_t Size() const { return m_size; }

private:
    void Grow() {
        std::size_t newCapacity = (m_capacity == 0) ? 4 : m_capacity * 2;
        T* newData = new T[newCapacity];
        for (std::size_t i = 0; i < m_size; ++i) {
            newData[i] = m_data[i];       // copy existing elements over
        }
        delete[] m_data;
        m_data = newData;
        m_capacity = newCapacity;
    }

    T*          m_data;      // buffer of T, sized to m_capacity
    std::size_t m_size;      // how many are used
    std::size_t m_capacity;  // how many can fit before we must grow
};

Everywhere you see T, the compiler substitutes the concrete type when you create an Array<int> or an Array<Vector3>. The same source produces a correct, fully type-checked container for each. Here it is in action:

#include <iostream>
#include "Array.h"

int main() {
    Array<int> numbers;      // T is int for this instance
    numbers.Add(10);        // first Add allocates the buffer: Grow() -> capacity 4
    numbers.Add(20);
    numbers.Add(30);        // still room in the buffer; no reallocation yet

    for (std::size_t i = 0; i < numbers.Size(); ++i) {
        std::cout << numbers[i] << "\n";
    }
    return 0;
}
Keep templates in headers

Because the compiler needs the full template definition to generate code for each type, class and function templates almost always live entirely in a header file. Splitting them into a separate .cpp the usual way leads to "unresolved external" linker errors. Our Array lives in Array.h for exactly this reason.

You already use templates: std::vector

Our Array was a teaching toy. In real code you'll reach for std::vector<T> from the standard library — a battle-tested class template that does everything ours does and much more (bounds-safe access, iterators, efficient moves). The angle brackets are the giveaway that it's a class template:

#include <vector>

std::vector<int>   ids;      // a growable array of int
std::vector<float> speeds;   // the SAME class, specialized for float

ids.push_back(42);
ids.push_back(1337);
// ids now holds two elements; it grew its buffer for us, just like Array::Grow

Every time you write std::vector<something>, the compiler instantiates that template for you. You've been using generic programming all along — now you understand the mechanism underneath it.

Where this points: GetComponent<T>()

Here's the payoff for the engine. Unity and Unreal let you ask an entity for a component by type — entity.GetComponent<Transform>(). That clean, type-safe call is a member function template. We'll build the real version in a later module, but the shape is already within your reach:

// A sketch of what we'll build later in the engine
class Entity {
public:
    // A function template *inside* a class
    template <typename T>
    T* GetComponent();   // returns a T* if this entity has one, else nullptr
};

// Usage reads beautifully and stays fully type-safe:
Transform* t = entity.GetComponent<Transform>();
Mesh*      m = entity.GetComponent<Mesh>();

No casting, no string lookups, no guessing — the type you ask for is the type you get back, checked at compile time. That is the power of generic programming, and templates are the language feature that makes it possible.

Requirements come from usage

A template silently requires whatever operations it uses. Max needs operator>; Array needs T to be copy-assignable. If you instantiate a template with a type that lacks a needed operation, the error appears at the point of use — another reason to keep your own types well-behaved with the operators from the previous lesson.

Recap
  • Templates let you write logic once and have the compiler generate a type-checked version for each concrete type — reuse without duplication.
  • template <typename T> introduces a function or class template; T is a placeholder for a real type filled in at compile time.
  • Argument deduction usually infers T from the call; when it can't, name it explicitly like Max<double>(...).
  • Class templates power containers: our tiny Array<T> mirrors how the standard std::vector<T> works.
  • Templates typically live in headers, and they set up the type-safe GetComponent<T>() we'll add to the engine.
