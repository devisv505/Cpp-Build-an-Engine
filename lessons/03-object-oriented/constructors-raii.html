Constructors, Destructors & RAII

In the previous lesson you built classes and created objects from them. But how does an object come to life in a valid state, and what happens when it goes away? In this lesson we answer both questions and meet RAII, the single most important idiom in C++. RAII is the reason C++ can manage GPU buffers, file handles, and memory safely without a garbage collector — and it is the foundation every resource-owning class in our engine will rest on.

What you'll learn
  • Write default and parameterized constructors to initialize objects safely.
  • Use a member initializer list and understand why it beats assignment in the body.
  • Write a destructor that runs automatically when an object is destroyed.
  • Apply the RAII idiom: acquire a resource in the constructor, release it in the destructor.
  • See why RAII is how our engine will own GPU and system resources without leaks.

What a constructor is for

A constructor is a special member function that runs automatically the moment an object is created. Its job is to put the object into a valid, usable state before anyone touches it. A constructor has the same name as the class and no return type — not even void.

If you write no constructor at all, the compiler generates a trivial one for you. But relying on that leaves members uninitialized, which is one of the most common sources of bugs in C++. Take control by writing your own.

// A tiny 3D vector — the kind of type our engine uses everywhere.
struct Vec3 {
    float x;
    float y;
    float z;

    // Default constructor: no arguments, sensible zero state.
    Vec3() {
        x = 0.0f;
        y = 0.0f;
        z = 0.0f;
    }

    // Parameterized constructor: caller supplies the components.
    Vec3(float px, float py, float pz) {
        x = px;
        y = py;
        z = pz;
    }
};

Vec3 origin;              // calls Vec3()        -> (0, 0, 0)
Vec3 up(0.0f, 1.0f, 0.0f); // calls Vec3(float,...) -> (0, 1, 0)

Notice we have two constructors with the same name but different parameters. That is overloading, and it lets one type support several ways of being created. The compiler picks the right one by matching the arguments you pass.

Give every member a value

If your class has data members, initialize all of them in every constructor. An uninitialized float or pointer holds garbage, and the bug it causes may not appear until much later — often somewhere unrelated.

Member initializer lists

Assigning to members inside the constructor body works for simple types, but there is a better tool: the member initializer list. It appears after a colon, before the body, and initializes each member directly instead of default-constructing it and then overwriting it.

struct Vec3 {
    float x, y, z;

    // Preferred: initialize members directly, in declaration order.
    Vec3() : x(0.0f), y(0.0f), z(0.0f) {}

    Vec3(float px, float py, float pz) : x(px), y(py), z(pz) {}
};

For a float the difference is invisible, but the initializer list becomes essential the moment a member is a type that must be initialized at birth — a reference, a const member, or a class that has no default constructor. Those cannot be assigned later, so the initializer list is the only way to set them.

Order matters

Members are initialized in the order they are declared in the class, not the order you list them in the initializer list. Keep the two orders in sync to avoid confusing surprises — some compilers will warn you when they differ.

Destructors: the automatic cleanup

The mirror image of a constructor is the destructor. It runs automatically when an object's lifetime ends — when a local variable goes out of scope, or when you delete a heap object. Its name is the class name with a leading tilde (~), and like a constructor it takes no arguments and returns nothing.

For a plain Vec3 you would never write a destructor; there is nothing to clean up. Destructors earn their keep when an object owns something that must be released: heap memory, a file, a network socket, or a GPU buffer. That ownership idea is the heart of the next section.

RAII: the big idea

RAII stands for Resource Acquisition Is Initialization. The name is clumsy, but the rule is simple and powerful:

  • Acquire the resource in the constructor.
  • Release it in the destructor.

Because the destructor runs automatically and guarantees to run — whether the function returns normally, hits an early return, or unwinds because of an exception — the resource is always released exactly once. You never write manual cleanup at every exit point, and you can never forget to. The object's lifetime is the resource's lifetime.

A class that owns a resource

Let's make this concrete with a class that owns a block of heap memory. This models exactly what our engine will later do for vertex data before we hand it to the GPU. The class allocates in its constructor and frees in its destructor — no caller ever has to remember to clean up.

#include <iostream>
#include <cstddef> // std::size_t

// Owns a raw block of vertex memory. Acquire in ctor, release in dtor.
class VertexBuffer {
public:
    // Constructor ACQUIRES the resource.
    explicit VertexBuffer(std::size_t count)
        : m_count(count), m_data(new float[count]) {
        std::cout << "Allocated " << m_count << " floats\n";
    }

    // Destructor RELEASES the resource — runs automatically.
    ~VertexBuffer() {
        delete[] m_data;
        std::cout << "Freed the buffer\n";
    }

    void set(std::size_t i, float value) { m_data[i] = value; }
    float get(std::size_t i) const { return m_data[i]; }
    std::size_t size() const { return m_count; }

private:
    std::size_t m_count;
    float* m_data;
};

void uploadTriangle() {
    VertexBuffer vb(9); // 3 vertices * 3 floats — memory acquired here
    vb.set(0, 1.0f);
    std::cout << "First value: " << vb.get(0) << "\n";
} // vb goes out of scope here -> ~VertexBuffer() runs -> memory freed

int main() {
    uploadTriangle();
    std::cout << "Back in main, buffer already cleaned up.\n";
    return 0;
}

Run this and you will see the allocation message, the value, then "Freed the buffer" — all without a single manual cleanup call in uploadTriangle. The moment vb leaves scope, its destructor fires. That is RAII doing its job.

The Rule of Three

Once a class owns a raw resource and has a real destructor, the compiler-generated copy behavior becomes dangerous: copying our VertexBuffer would leave two objects pointing at the same memory, and both destructors would try to free it — a double-free crash. When you write a destructor that releases a resource, you usually also need to handle copying (delete or define the copy constructor and copy assignment). We'll tackle copy and move semantics properly in a later module; for now, avoid copying resource-owning objects.

Why the engine leans on RAII

Manual cleanup does not scale. A real render frame might touch dozens of resources, and any early exit that skips a cleanup call leaks memory or GPU handles. RAII removes that entire class of bug: the compiler emits the cleanup for you at every exit path.

The C++ standard library is already built this way, and you have been relying on it without noticing. When you use std::vector, its constructor allocates and its destructor frees — a vector is a textbook RAII type. std::string, std::fstream, and smart pointers like std::unique_ptr all follow the same pattern.

ResourceAcquired in constructorReleased in destructor
Heap memorynew[] / allocatordelete[] / deallocate
File handlefopen / ifstream openfclose / stream close
GPU bufferD3D CreateBuffer or Metal newBufferCOM release on Windows or Metal object release on macOS
Std containerstd::vector ctorstd::vector dtor

GPU resources need explicit ownership on both platforms. Direct3D uses reference-counted COM objects, commonly held by Microsoft::WRL::ComPtr. Metal objects on macOS are held by Objective-C strong references or metal-cpp ownership helpers. Our Mesh and Texture wrappers use RAII so the selected backend releases resources automatically.

Prefer the library's RAII types

You rarely need raw new/delete. Reach for std::vector to own an array and std::unique_ptr to own a single heap object. They implement RAII correctly, including the tricky copy and move rules, so you get safe ownership for free. We wrote a raw buffer above only to see the mechanism up close.

Recap
  • A constructor runs at creation and puts the object into a valid state; you can overload it for default and parameterized creation.
  • A member initializer list (: x(px), y(py)) initializes members directly and is required for references, const members, and types without a default constructor. Members initialize in declaration order.
  • A destructor (~ClassName) runs automatically at the end of an object's lifetime and is where cleanup belongs.
  • RAII ties a resource's lifetime to an object's: acquire in the constructor, release in the destructor, and cleanup becomes automatic and leak-proof.
  • Writing a resource-releasing destructor triggers the Rule of Three — handle copying too, or avoid copying the object for now.
  • Standard types like std::vector and std::unique_ptr are RAII, and both graphics backends wrap GPU resources the same way.
