Encapsulation, const & Access

In the last lesson your Transform class learned to construct and clean up after itself with RAII. Now we make it trustworthy. Encapsulation is the art of hiding a class's internals behind a small, well-behaved public interface so the rest of the engine can only touch it in safe, intentional ways. Get this right and every future system — rendering, physics, scene graphs — becomes far easier to reason about.

What you'll learn
  • How public, private and protected control who can touch each member
  • Why hiding data behind methods keeps invariants safe
  • How to write sensible getters and setters — and when to skip them
  • What const member functions promise, and why const-correctness matters
  • How encapsulation shapes a clean, dependable engine API

Access specifiers: who is allowed in

Every member of a class lives behind an access specifier. There are three, and each answers the question "who is allowed to see or change this?"

SpecifierAccessible fromUse it for
publicAnywhereThe interface — methods callers are meant to use
privateOnly this classData and helpers that are implementation details
protectedThis class and derived classesMembers subclasses need (more on this next lesson)

The one difference between class and struct in C++ is the default: members of a class are private unless you say otherwise, while a struct's members default to public. That is why we reach for class when a type has invariants to protect.

An invariant is a promise

An invariant is a rule that must always hold true for an object — for example, "a health value is never negative." Encapsulation exists so no outside code can quietly break that promise.

Hiding data behind methods

Imagine a component with a public field anyone can write. Nothing stops a bug elsewhere from stuffing a nonsense value into it:

// Everything is public — no protection at all.
struct Health {
    int current;
    int max;
};

Health hp{ 100, 100 };
hp.current = -9999;   // Oops. Nothing stopped this.
hp.max = 0;           // Now current > max, and divide-by-zero looms.

When data is private, the only way to change it is through methods you control — and those methods can enforce the rules every single time:

#include <algorithm>  // std::clamp, std::max

class Health {
public:
    Health(int maxHp)
        : m_max(std::max(1, maxHp)), m_current(std::max(1, maxHp)) {}

    // Mutators keep the invariant 0 <= current <= max intact.
    void takeDamage(int amount) {
        if (amount < 0) return;                 // ignore nonsense
        m_current = std::max(0, m_current - amount);
    }

    void heal(int amount) {
        if (amount < 0) return;
        m_current = std::min(m_max, m_current + amount);
    }

private:
    int m_max;
    int m_current;
};

Now m_current can never fall below zero or climb above m_max, no matter who calls what. The class is the single source of truth for its own rules. Notice the m_ prefix on the fields — a common convention that makes it obvious at a glance which names are member data.

Getters and setters

Sometimes callers legitimately need to read a value, or set one under controlled conditions. That is what getters and setters are for. A getter is usually a tiny method that returns a copy or reference; a setter validates before storing.

class Health {
public:
    Health(int maxHp)
        : m_max(std::max(1, maxHp)), m_current(m_max) {}

    // Getters: read-only access, no way to corrupt state.
    int current() const { return m_current; }
    int max() const     { return m_max; }
    bool isAlive() const { return m_current > 0; }

    // Setter: validates, then stores.
    void setMax(int newMax) {
        m_max = std::max(1, newMax);
        m_current = std::min(m_current, m_max); // keep it in range
    }

private:
    int m_max;
    int m_current;
};
Don't add getters and setters reflexively

A getter/setter pair that just reads and writes a field with no validation is barely better than a public field — but it does preserve your freedom to add rules later without breaking callers. Expose behavior first (takeDamage), raw accessors only where they earn their place.

const member functions

Look again at the getters above — each ends with const after the parameter list. A const member function promises not to modify the object it is called on. The compiler enforces that promise: inside a const method you cannot assign to a member or call a non-const method.

class Counter {
public:
    int value() const {
        // m_count = 0;   // ERROR: cannot modify in a const method
        return m_count;   // reading is fine
    }
    void increment() { ++m_count; } // non-const: allowed to change state

private:
    int m_count = 0;
};

This is the difference between a query (asks a question, changes nothing) and a command (does something). Marking queries const documents your intent and lets the compiler catch mistakes.

Why const-correctness matters

The payoff appears the moment you pass objects around by const& — which you will do constantly in an engine to avoid copying big objects. You can only call const methods on a const reference:

// Passing by const reference: fast (no copy) and safe (no changes).
void printHealth(const Health& hp) {
    // hp.current() works — it's a const method.
    std::cout << hp.current() << " / " << hp.max() << '\n';
    // hp.takeDamage(5); // ERROR: takeDamage is non-const, hp is const
}
Const spreads — start early

If a getter isn't marked const, you can't call it through a const&, so callers are forced to take non-const references, and the "can't be changed" guarantee unravels across your whole codebase. Mark every query const from day one; retrofitting it later is painful.

Refining our Transform component

Let's pull it all together by hardening the Transform from the previous lesson. Position and rotation become private; we expose read access with const getters, controlled mutation with setters, and a const query that reports whether the cached world matrix is stale.

#pragma once
#include <DirectXMath.h>

// A component-based engine leans on encapsulation: systems talk to
// Transform only through its public interface, never its raw fields.
class Transform {
public:
    Transform() = default;

    // --- Queries: const, because they never change the object ---
    const DirectX::XMFLOAT3& position() const { return m_position; }
    const DirectX::XMFLOAT3& rotation() const { return m_rotation; }
    const DirectX::XMFLOAT3& scale()    const { return m_scale; }
    bool isDirty() const { return m_dirty; }

    // --- Commands: non-const, they mutate and flag a rebuild ---
    void setPosition(const DirectX::XMFLOAT3& p) {
        m_position = p;
        m_dirty = true;              // world matrix must be rebuilt
    }
    void setScale(const DirectX::XMFLOAT3& s) {
        m_scale = s;
        m_dirty = true;
    }

    // Builds the model matrix. const: reading state, returning a value.
    DirectX::XMMATRIX worldMatrix() const {
        using namespace DirectX;
        XMMATRIX s = XMMatrixScaling(m_scale.x, m_scale.y, m_scale.z);
        XMMATRIX r = XMMatrixRotationRollPitchYaw(
            m_rotation.x, m_rotation.y, m_rotation.z);
        XMMATRIX t = XMMatrixTranslation(
            m_position.x, m_position.y, m_position.z);
        return s * r * t;            // scale, then rotate, then translate
    }

private:
    DirectX::XMFLOAT3 m_position{ 0.0f, 0.0f, 0.0f };
    DirectX::XMFLOAT3 m_rotation{ 0.0f, 0.0f, 0.0f }; // radians
    DirectX::XMFLOAT3 m_scale{ 1.0f, 1.0f, 1.0f };
    bool m_dirty = true;
};

Because the fields are private, the only way to move an object is through setPosition — which guarantees the m_dirty flag flips so our renderer knows to recompute the world matrix. No system can forget to set the flag, because no system can reach the data directly. That is encapsulation earning its keep: the class enforces its own correctness.

Interface vs. implementation

Callers depend on the public methods — the interface. How Transform stores its data — the implementation — stays yours to change. Swap XMFLOAT3 for a quaternion rotation later and, as long as the public methods still behave, not a single caller needs editing.

Why a clean engine API relies on this

Unity and Unreal expose thousands of objects through small, disciplined interfaces. You never poke a private field of a Unity Transform; you call SetPosition and trust it to update everything downstream. Your engine will grow the same way. Every component you build — meshes, materials, cameras — becomes a black box with a tidy public face and a protected interior. When a subsystem misbehaves you look inside one class, not the whole program, because nothing else could have touched its data.

Recap
  • public, private and protected decide who can access each member; class defaults to private.
  • Hiding data behind methods lets a class enforce its invariants so no outside code can corrupt its state.
  • Getters give safe read access and setters validate before storing — but expose behavior rather than raw fields where you can.
  • A const member function promises not to modify the object; mark every query const so it works through const&.
  • Encapsulation separates a stable interface from a changeable implementation — the foundation of a clean engine API.
