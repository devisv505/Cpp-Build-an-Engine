The Standard Template Library

Every engine juggles collections of things: meshes, textures, entities, the components attached to each object. Rather than reinventing dynamic arrays and lookup tables by hand, C++ ships them in the Standard Template Library — the STL. This lesson is a guided tour of the containers and algorithms you will reach for constantly, so that when we start wiring up components and resources you already speak the language of std::vector and std::unordered_map.

What you'll learn
  • The four containers you will use most: std::vector, std::array, std::map, and std::unordered_map.
  • How iterators and the range-based for loop let you walk any container the same way.
  • Reaching into <algorithm> for std::sort, std::find, and std::for_each.
  • Engine-flavored patterns, like a name-to-resource lookup table for our asset manager.

What "template" means here

The "T" in STL stands for template. A template is a container recipe that works for any type you plug in. std::vector<int> is a growable array of integers; std::vector<Mesh> is a growable array of meshes. You write the type in angle brackets and the compiler generates a specialized, fully type-safe version. Because everything lives in the std namespace, you will see the std:: prefix throughout — or a using declaration to shorten it.

std::vector — the workhorse

std::vector is a dynamic array: contiguous in memory, resizable at runtime, and cache-friendly. If you are unsure which container to use, the honest answer is almost always "start with a vector." We will store our scene's entities in one.

#include <iostream>
#include <vector>
#include <string>

int main() {
    std::vector<std::string> entities;   // empty, grows on demand

    entities.push_back("Player");        // append to the end
    entities.push_back("Enemy");
    entities.push_back("Pickup");

    std::cout << "Entity count: " << entities.size() << "\n";
    std::cout << "First entity: " << entities[0] << "\n";

    entities.pop_back();                 // remove the last element
    std::cout << "After pop: " << entities.size() << "\n";

    return 0;
}

Key members you will use daily: push_back appends, size reports the count, operator[] and at access elements (the latter bounds-checks), front and back grab the ends, and clear empties it. When you know roughly how many items you need, call reserve up front to avoid repeated reallocations.

Reserve when you can

If you're about to load 5,000 vertices, call verts.reserve(5000) first. The vector allocates once instead of growing piecemeal — a real win in hot loading paths.

std::array — fixed size, zero overhead

When the count is known at compile time, std::array<T, N> is a thin, safe wrapper around a C-style array. No heap allocation, no resizing — just a fixed block with the same friendly interface (size, at, iterators). It is perfect for things like a 4×4 matrix's 16 floats, or the RGBA channels of a color.

#include <array>
#include <iostream>

int main() {
    // A clear color: red, green, blue, alpha — always exactly 4 floats.
    std::array<float, 4> clearColor = { 0.1f, 0.2f, 0.4f, 1.0f };

    std::cout << "Channels: " << clearColor.size() << "\n";  // always 4
    std::cout << "Blue: "     << clearColor[2]     << "\n";

    return 0;
}
array vs. vector

Use std::array when the size never changes and is known while compiling. Use std::vector when the size is decided at runtime or grows over time. DirectX will later hand us buffers whose size we already know — std::array shines there.

Associative containers: map and unordered_map

Vectors answer "give me the item at index 3." Maps answer "give me the item named brick_wall." They store key → value pairs, which is exactly what an asset manager needs: look up a texture or mesh by its name.

  • std::map keeps keys sorted and is backed by a balanced tree. Lookups are O(log n). Use it when you need keys in order.
  • std::unordered_map is a hash table. Lookups are average O(1) but the order is undefined. For a resource cache, this is usually what you want.

Here is a small engine-flavored resource cache mapping a resource name to a made-up handle id.

#include <iostream>
#include <string>
#include <unordered_map>

int main() {
    // name -> texture handle (a stand-in for a real GPU resource id)
    std::unordered_map<std::string, int> textures;

    textures["brick_wall"] = 101;
    textures["grass"]      = 102;
    textures["player_skin"] = 103;

    // Look up by name.
    std::string wanted = "grass";
    auto it = textures.find(wanted);
    if (it != textures.end()) {
        // it->first is the key, it->second is the value.
        std::cout << wanted << " -> handle " << it->second << "\n";
    } else {
        std::cout << wanted << " not loaded\n";
    }

    // count() gives a quick yes/no for existence.
    if (textures.count("brick_wall")) {
        std::cout << "brick_wall is cached\n";
    }

    return 0;
}
operator[] inserts

Reading a missing key with textures["ghost"] does not error — it silently creates the entry with a default value. To check whether a key exists without inserting it, use find or count, as above.

Iterators and the range-based for loop

An iterator is a generalized pointer into a container. Every STL container exposes begin() (points at the first element) and end() (points just past the last). Algorithms are written against iterators, which is why one std::sort works on a vector, an array, or any other sequence.

Ninety percent of the time you won't touch iterators directly — the range-based for loop does it for you. Use const auto& to read without copying, or auto& to modify in place.

#include <iostream>
#include <vector>
#include <string>

int main() {
    std::vector<std::string> scene = { "Player", "Enemy", "Pickup" };

    // Read-only pass: const reference avoids copying each string.
    for (const auto& name : scene) {
        std::cout << "Spawned: " << name << "\n";
    }

    // The classic iterator form does the same thing.
    for (auto it = scene.begin(); it != scene.end(); ++it) {
        std::cout << "  visited: " << *it << "\n";  // *it dereferences to the value
    }

    return 0;
}

Algorithms from <algorithm>

The header <algorithm> is a toolbox of reusable operations that act on iterator ranges. Three you will use often:

  • std::sort — orders a range in place.
  • std::find — returns an iterator to the first matching element (or end()).
  • std::for_each — applies a function to every element.
#include <iostream>
#include <vector>
#include <algorithm>

int main() {
    std::vector<int> drawOrder = { 40, 10, 30, 20 };

    // Sort ascending (default). Great for depth-sorting later.
    std::sort(drawOrder.begin(), drawOrder.end());

    // Sort descending with a lambda comparator.
    std::sort(drawOrder.begin(), drawOrder.end(),
              [](int a, int b) { return a > b; });

    // Find a value; compare against end() to test success.
    auto found = std::find(drawOrder.begin(), drawOrder.end(), 30);
    if (found != drawOrder.end()) {
        std::cout << "Found 30 in the list\n";
    }

    // Apply an action to every element.
    std::for_each(drawOrder.begin(), drawOrder.end(),
                  [](int layer) { std::cout << "layer " << layer << "\n"; });

    return 0;
}

Those [](...) { ... } snippets are lambdas — small unnamed functions written inline. They pair naturally with algorithms: std::sort takes one to decide ordering, and std::for_each takes one as the action. We will lean on this pattern when sorting renderables front-to-back for our engine's draw loop.

Half-open ranges

STL ranges are half-open: [begin, end) includes the first element but stops before end. This is why an empty container has begin() == end(), and why loops terminate cleanly.

Putting it together

None of these tools live in isolation. A resource manager might hold an unordered_map from name to a vector of mesh handles, sort that vector with std::sort, and iterate it with a range-based for. Because they all share the iterator interface, they compose without friction. In the next module we shift gears from data structures to math — coordinate systems and spaces — but we will keep storing that data in exactly these containers.

Recap
  • std::vector is your default growable array; std::array is its fixed-size, zero-overhead sibling.
  • std::map and std::unordered_map store key → value pairs — ideal for name-based resource lookups.
  • Iterators (begin()/end()) unify traversal; the range-based for loop is the everyday way to walk a container.
  • std::sort, std::find, and std::for_each from <algorithm> operate on any iterator range, often paired with lambdas.
