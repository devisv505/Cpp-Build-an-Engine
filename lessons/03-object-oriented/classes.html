Classes & Objects

Up to now our programs have been loose collections of variables and free functions. That works for tiny examples, but a 3D engine is made of things — a transform, a mesh, a timer, a camera — each with its own data and its own behavior. The C++ tool for modeling a "thing" is the class. In this lesson you'll learn how a class bundles data and functions together, and how to stamp out real objects from it. This is the single most important idea behind how Unity and Unreal are built.

What you'll learn
  • The difference between a class (the blueprint) and an object (a live instance).
  • How to declare data members and member functions (methods) inside a class.
  • How to create objects and call their methods with the . operator.
  • What the hidden this pointer is and why every method has one.
  • How public and private control what the outside world can touch.

Class vs. object: blueprint vs. building

A class is a description — a blueprint. It says "an object of this type will hold these pieces of data and will be able to do these things." Writing a class does not create anything you can use yet, just like a blueprint for a house is not a house you can live in.

An object (also called an instance) is a real, concrete value built from that blueprint. From one class you can create as many objects as you like, and each one carries its own copy of the data. One Vector2 class, a thousand Vector2 objects — each with its own x and y.

Declaring your first class

Let's build a small Vector2 — a 2D point or direction. This is a type we'll use constantly in the engine for screen positions, UV coordinates, and velocities. Here's the whole thing:

#include <iostream>
#include <cmath>

// A class declaration: the blueprint for a 2D vector.
class Vector2 {
public:                    // everything below is usable from outside
    // Data members: each object gets its own x and y.
    float x = 0.0f;
    float y = 0.0f;

    // Member function (method): computes the vector's length.
    float length() const {
        // We can use x and y directly — they belong to this object.
        return std::sqrt(x * x + y * y);
    }

    // Another method: shift this vector by another one.
    void add(const Vector2& other) {
        x += other.x;
        y += other.y;
    }
};

int main() {
    // Create an object (an instance) of Vector2.
    Vector2 position;      // x and y start at 0.0f
    position.x = 3.0f;     // reach in and set the data members
    position.y = 4.0f;

    // Call a method with the dot operator.
    std::cout << "Length: " << position.length() << "\n";  // 5

    Vector2 velocity;
    velocity.x = 1.0f;
    velocity.y = -2.0f;

    position.add(velocity);   // position becomes (4, 2)
    std::cout << "Now at: " << position.x << ", " << position.y << "\n";
    return 0;
}

Run it and you'll see Length: 5 followed by Now at: 4, 2. Let's unpack what just happened.

Don't forget the semicolon

A class declaration ends with a semicolon after the closing brace: };. This trips up nearly every beginner. If you get a confusing error on the line after your class, a missing ; is the usual culprit.

Data members and member functions

Inside a class we declare two kinds of things:

TermWhat it isIn our example
Data member (field)A variable that belongs to each objectx, y
Member function (method)A function that operates on an object's datalength(), add()

The key insight is that a method automatically has access to the object's data members by name. Inside length() we wrote x * x + y * y without saying which object's x — because the method always runs on a specific object, and x means "this object's x." That is what makes a class more than a bag of loose variables: the data and the code that works on it live together.

const methods

Notice float length() const. The const after the parameter list promises "this method does not modify the object." length() only reads x and y, so it's marked const. add() changes them, so it is not. Marking read-only methods const is a good habit — the compiler will stop you if you accidentally change data you promised not to, and it lets you call the method on const objects.

Creating instances and calling methods

Once the class exists, you create objects just like any other variable — the type name followed by a variable name:

Vector2 position;   // one object
Vector2 velocity;   // a second, completely independent object

To reach the data or behavior of an object, use the dot operator .:

  • position.x — access a data member.
  • position.length() — call a method on that object.

Because each object owns its data, position.add(velocity) changes only position. velocity is untouched. If you created ten Vector2 objects, each would have its own private pair of floats sitting in memory.

The this pointer

When you call position.length(), how does the method know it should use position's data and not velocity's? Behind the scenes, every non-static member function receives a hidden pointer called this, which points at the object the method was called on. Writing x inside a method is really shorthand for this->x.

class Timer {
public:
    float elapsed = 0.0f;   // seconds accumulated so far

    // Advance the timer by dt seconds and return the new total.
    Timer& tick(float dt) {
        this->elapsed += dt;   // 'this' points at the object we called tick on
        return *this;          // return the object itself, so calls can chain
    }

    void reset() {
        elapsed = 0.0f;        // same as this->elapsed = 0.0f;
    }
};

int main() {
    Timer frameTimer;
    // Because tick returns *this, we can chain calls:
    frameTimer.tick(0.016f).tick(0.016f).tick(0.016f);
    // frameTimer.elapsed is now ~0.048f
}

You rarely need to type this-> — plain x works. But it matters in two common cases: when a parameter has the same name as a data member (so you write this->x = x; to tell them apart), and when you want a method to return the object itself with return *this; so calls can be chained, as in tick(...).tick(...).

this is a pointer, so use ->

this is a pointer to the current object, which is why you write this->elapsed (arrow) rather than this.elapsed (dot). To hand back the object itself rather than the pointer, dereference it with *this.

public vs. private

You've seen the public: label. Members after it are usable from outside the class. C++ also gives you private:, which hides members so only the class's own methods can touch them. This is called encapsulation, and it's how you keep an object's internal state consistent — the outside world can only interact through the methods you choose to expose.

class Health {
public:
    void takeDamage(int amount) {
        current -= amount;
        if (current < 0) current = 0;   // clamp — never go negative
    }
    int value() const { return current; }

private:
    int current = 100;   // outside code can't set this to a bad value
};

Here nothing outside Health can set current directly to -9999. The only way in is takeDamage(), which enforces the "never below zero" rule. We'll lean on this heavily when we design engine components.

class vs struct: same tool, different default

C++ also has struct. It is almost identical to class — the only real difference is that members are public by default in a struct and private by default in a class. By convention many engines use struct for plain data holders and class for things with behavior and hidden state.

Why this matters for the engine

Component-based engines like Unity and Unreal are, at their core, giant collections of classes. A Transform class holds position, rotation, and scale plus methods to move and rotate. A Mesh class holds vertex data plus a method to draw itself. Our Vector2 (and soon a Vector3) will be the tiny building block those larger classes are made of. Every time you write class Something { ... };, you're defining a new type of "thing" your engine can create and manipulate — exactly the mental model you'll use for the rest of this course.

Right now our objects start with hardcoded default values and we set the data member by member after creation. That's clumsy. In the next lesson we'll fix it with constructors — special functions that build an object into a valid state the moment it's created — and their partner, the destructor.

Recap
  • A class is a blueprint; an object (instance) is a concrete value built from it, with its own copy of the data.
  • Classes contain data members (variables) and member functions / methods (behavior) bundled together.
  • Create an object like a normal variable, then use the . operator to read members and call methods.
  • Every method receives a hidden this pointer to the object it runs on; x means this->x, and return *this; hands back the object for chaining.
  • Mark read-only methods const, and use private to hide internal state so objects stay valid — the essence of encapsulation.
  • Don't forget the semicolon after the closing } of a class.
