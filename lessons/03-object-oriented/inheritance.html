Inheritance

Last lesson you learned to lock a class down with encapsulation and const. Now we take the next step: building new classes on top of existing ones. Inheritance lets a class absorb the data and behavior of another and then add its own. This is the exact mechanism engines like Unity and Unreal use to say "a Transform, a MeshRenderer, and a RigidBody are all Components" — and it is how we will structure our own engine.

What you'll learn
  • How to define a base class and a derived class with public inheritance
  • What the is-a relationship means and when inheritance is the right tool
  • The difference between private, protected, and public access for subclasses
  • How to call a base constructor from a derived class
  • How this sets up a shared Component base class for our engine

The is-a relationship

Inheritance answers one question: "Is this new thing a kind of that existing thing?" If the honest answer is yes, X is a Y, inheritance may fit. A Dog is an Animal. A Button is a Widget. In our engine, a Transform is a Component, and a MeshRenderer is a Component too.

When you write class Dog : public Animal, the Dog automatically receives every member of Animal: its data, and its functions. You did not copy-paste anything — the base part lives inside every derived object. We call Animal the base class (or parent), and Dog the derived class (or child).

Is-a, not has-a

If you catch yourself saying "has a" instead of "is a", reach for a member variable instead. A Car has an Engine, so Engine is a member of Car, not a base class of it. Getting this distinction right keeps designs clean.

A first base and derived class

Let's make a small, complete program. We'll build a base Animal that stores a name and can describe() itself, then derive a Dog that adds barking. Notice that Dog uses name and describe() without redefining them.

#include <iostream>
#include <string>

class Animal {
public:
    // The base constructor sets shared state.
    Animal(const std::string& name, int legs)
        : m_name(name), m_legs(legs) {}

    void describe() const {
        std::cout << m_name << " has " << m_legs << " legs.\n";
    }

protected:
    std::string m_name;   // protected: derived classes may read/write this
    int m_legs;

private:
    int m_secretId = 0;   // private: NOT visible to derived classes
};

// "Dog is an Animal" -> public inheritance.
class Dog : public Animal {
public:
    Dog(const std::string& name)
        : Animal(name, 4) {}   // call the base constructor first

    void bark() const {
        // m_name is accessible because it is protected in Animal.
        std::cout << m_name << " says: Woof!\n";
    }
};

int main() {
    Dog rex("Rex");
    rex.describe();  // inherited from Animal
    rex.bark();      // added by Dog
    return 0;
}

Running this prints:

Rex has 4 legs.
Rex says: Woof!

Dog never declared m_name, m_legs, or describe() — it inherited all three. It only added bark(). That is reuse without duplication, which is the whole point.

protected: the middle ground

You already know public (anyone can touch it) and private (only the class itself). Inheritance adds a third level: protected. A protected member is hidden from outside code, exactly like private, but it is visible to derived classes.

AccessSame classDerived classOutside code
publicYesYesYes
protectedYesYesNo
privateYesNoNo

In the example above, Dog::bark() could read m_name because it was protected. If we had left it private, the compiler would reject bark() with an error like "m_name is a private member of Animal". Keep members private by default; promote them to protected only when a subclass genuinely needs direct access.

Don't over-expose

Making everything protected so subclasses can reach it feels convenient but leaks your internal layout to every child class. That coupling makes the base hard to change later. Prefer a small protected helper function over a pile of protected data.

Calling the base constructor

A derived object is built in two stages: first the base part, then the derived part. Because the base part must be fully constructed before the derived body runs, you initialize it in the derived constructor's member initializer list — the part after the colon.

Dog(const std::string& name)
    : Animal(name, 4)   // 1. construct the Animal base with these arguments
{
    // 2. Dog-specific setup would go here (the body runs second)
}

If Animal had a default constructor (one taking no arguments), you could omit this and the compiler would call it for you. But our Animal requires a name and a leg count, so we must pass them explicitly. Forgetting to do this is a common early error:

error: constructor for 'Dog' must explicitly initialize the
       base class 'Animal' which does not have a default constructor
Order of construction

Bases are always constructed before members, and both happen before the constructor body. Destruction runs in the exact reverse order: derived first, then base. You get this correct automatically as long as you pass the base its arguments.

Sharing behavior across the hierarchy

The real payoff is that shared behavior lives in one place. Add a method to the base, and every derived class gets it instantly. Suppose we give Animal a sleep() function:

class Animal {
public:
    Animal(const std::string& name, int legs)
        : m_name(name), m_legs(legs) {}

    void describe() const {
        std::cout << m_name << " has " << m_legs << " legs.\n";
    }

    // New shared behavior — every derived class gets it for free.
    void sleep() const {
        std::cout << m_name << " is sleeping...\n";
    }

protected:
    std::string m_name;
    int m_legs;
};

class Cat : public Animal {
public:
    Cat(const std::string& name) : Animal(name, 4) {}
    void purr() const { std::cout << m_name << " purrs.\n"; }
};

int main() {
    Cat c("Milo");
    c.describe();  // from Animal
    c.sleep();     // from Animal, added later — no change to Cat needed
    c.purr();      // from Cat
}

We never touched Cat, yet Milo can now sleep. Multiply that across a dozen component types and you can feel how much duplication inheritance saves.

Foreshadowing: a Component base class

Here is where this pays off for our engine. In a component-based design, a game object (often called an entity) is a lightweight container that holds a list of components. Each component is a small piece of behavior or data. A Transform stores position; a MeshRenderer stores what to draw. They are all different, but they are all a Component — the textbook is-a relationship.

So we give them a common base:

#include <string>

// The shared base for every component in our engine.
class Component {
public:
    Component(const std::string& name) : m_name(name) {}

    const std::string& name() const { return m_name; }
    bool isEnabled() const { return m_enabled; }
    void setEnabled(bool on) { m_enabled = on; }

protected:
    std::string m_name;      // derived components can read their own name
    bool m_enabled = true;
};

// A Transform IS A Component.
class Transform : public Component {
public:
    Transform() : Component("Transform") {}

    void setPosition(float x, float y, float z) {
        m_x = x; m_y = y; m_z = z;
    }

private:
    float m_x = 0.0f, m_y = 0.0f, m_z = 0.0f;
};

// A MeshRenderer IS A Component too.
class MeshRenderer : public Component {
public:
    MeshRenderer() : Component("MeshRenderer") {}
    // ...mesh and material data will live here later
};

Every component now shares a name and an enabled flag, and each derived type calls Component's constructor to pass its own label. Later we will store many different components together and ask each one to update itself. Doing that cleanly needs one more tool — virtual functions — which is exactly the next lesson.

Keep hierarchies shallow

Deep chains (A : B : C : D) get hard to reason about fast. Real engines keep the tree flat: one Component base and many direct children. When in doubt, favor composition (an object holding other objects) over tall inheritance towers.

Recap
  • Inheritance models an is-a relationship: class Dog : public Animal makes Dog a kind of Animal.
  • A derived class inherits the base's data and functions, then adds its own — reuse without duplication.
  • protected members are hidden from outside code but visible to derived classes; keep members private unless a subclass truly needs them.
  • Call the base constructor in the derived class's member initializer list; the base part is built first.
  • We built a Component base that Transform and MeshRenderer derive from — the backbone of our engine, ready for virtual functions next.
