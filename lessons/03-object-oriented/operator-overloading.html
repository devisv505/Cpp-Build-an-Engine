Operator Overloading

Wouldn't it be nice to write a + b when a and b are 3D vectors, instead of calling Add(a, b) everywhere? Operator overloading lets your own types plug into the same symbols the built-in types use. In this lesson we build a small Vector3 — the exact type that will power the Transform component of our engine — and teach it how to add, subtract, scale, and compare.

What you'll learn
  • What operator overloading is and the syntax for operator+, operator-, and friends
  • When to write an operator as a member function versus a free (non-member) function
  • How to build a reusable Vector3 with +, -, scalar *, and ==
  • How to add a Dot() product and a printable operator<<
  • Why you should overload operators sparingly and keep their meaning intuitive

What is an operator, really?

In C++ an operator like + is just a function with a special name. When you write 3 + 4 the compiler already knows how to add two ints. But if you write a + b where both are your own Vector3 type, the compiler has no built-in meaning — so it looks for a function named operator+ that accepts two Vector3s. Overloading simply means providing that function.

This matters for game math. A 3D position, a direction, a velocity — these are all vectors, and the math reads far more clearly as position + velocity * deltaTime than as a pile of nested function calls.

You can't invent new symbols

You can only overload operators that already exist in C++ (+, -, *, ==, <<, and so on). You cannot create a brand-new operator like **, and you cannot change the number of operands or their precedence.

Member vs. free functions

There are two places you can define an operator, and choosing well makes your type easier to use.

  • Member function: the left-hand operand is the object itself (this), and you write only the right-hand side as a parameter. Natural for operators that modify or belong to one object, like += or ==.
  • Free (non-member) function: both operands are parameters. This is the friendlier choice for symmetric operators such as +, and it is required when the left operand is a type you don't own — for example the stream in std::cout << v, where the left side is std::ostream.

Building a Vector3

Let's start with the data and a couple of members. We use a struct so the fields are public by default — for a small math type where everyone reads and writes x, y, z directly, that is perfectly idiomatic.

#pragma once
#include <iostream>

struct Vector3 {
    float x = 0.0f;
    float y = 0.0f;
    float z = 0.0f;

    // Default + component constructor
    Vector3() = default;
    Vector3(float x_, float y_, float z_) : x(x_), y(y_), z(z_) {}

    // Compound assignment as a MEMBER: modifies *this, returns a reference
    Vector3& operator+=(const Vector3& rhs) {
        x += rhs.x;
        y += rhs.y;
        z += rhs.z;
        return *this;   // allow chaining: a += b += c;
    }
};

Notice operator+= returns Vector3& (a reference to the modified object). That mirrors how the built-in += behaves and lets you chain operations. Now the symmetric operators, written as free functions. We pass by const& to avoid copying and to promise we won't modify the inputs.

// Vector + Vector  (free function)
inline Vector3 operator+(const Vector3& a, const Vector3& b) {
    return Vector3(a.x + b.x, a.y + b.y, a.z + b.z);
}

// Vector - Vector
inline Vector3 operator-(const Vector3& a, const Vector3& b) {
    return Vector3(a.x - b.x, a.y - b.y, a.z - b.z);
}

// Vector * scalar  (scales the vector)
inline Vector3 operator*(const Vector3& v, float s) {
    return Vector3(v.x * s, v.y * s, v.z * s);
}

// scalar * Vector  (so 2.0f * v also works — same math, args flipped)
inline Vector3 operator*(float s, const Vector3& v) {
    return v * s;   // reuse the version above
}

// Equality: are two vectors the same?
inline bool operator==(const Vector3& a, const Vector3& b) {
    return a.x == b.x && a.y == b.y && a.z == b.z;
}

inline bool operator!=(const Vector3& a, const Vector3& b) {
    return !(a == b);   // define != in terms of == to stay consistent
}
Provide both orders for scalar multiply

We wrote operator* twice — once for v * 2.0f and once for 2.0f * v. C++ won't automatically flip the operands for you, so if you only define one, the other form fails to compile. The second version just reuses the first.

The dot product

Not every operation deserves an operator. The dot product combines two vectors into a single number that tells you how aligned they are — it is central to lighting and to checking whether something faces the camera. There is no obvious symbol for it, so we give it a plain, well-named function instead.

// Dot product: a scalar measuring alignment of two vectors.
inline float Dot(const Vector3& a, const Vector3& b) {
    return a.x * b.x + a.y * b.y + a.z * b.z;
}
Comparing floats exactly is risky

Our operator== uses exact == on floats. That is fine for testing "is this the untouched value?", but after real math (rotations, additions) rounding error creeps in and two "equal" vectors may differ in the last bit. For engine code you'll usually compare with a small tolerance (an epsilon) instead. We'll cover that in the math module.

Printing with operator<<

To debug vectors you want to std::cout them directly. Because the left operand is a std::ostream (a type we don't own), this must be a free function. It takes the stream by reference, writes into it, and returns it so you can chain more <<.

// Stream output — note the ostream& return so chaining works.
inline std::ostream& operator<<(std::ostream& os, const Vector3& v) {
    os << "(" << v.x << ", " << v.y << ", " << v.z << ")";
    return os;
}

Putting it together

Here is a full program that uses every operator we defined. It compiles as-is with a C++17 compiler.

#include "Vector3.h"
#include <iostream>

int main() {
    Vector3 position(0.0f, 1.0f, 0.0f);
    Vector3 velocity(2.0f, 0.0f, -1.0f);
    float   deltaTime = 0.5f;   // half a second

    // Reads just like the math: p = p + v * dt
    position = position + velocity * deltaTime;
    std::cout << "New position: " << position << "\n";

    // Compound assignment (member operator+=)
    Vector3 gravity(0.0f, -9.8f, 0.0f);
    velocity += gravity * deltaTime;
    std::cout << "New velocity: " << velocity << "\n";

    // Equality
    Vector3 origin;                 // (0, 0, 0)
    std::cout << "At origin? " << (position == origin ? "yes" : "no") << "\n";

    // Dot product: how aligned are these two directions?
    Vector3 forward(0.0f, 0.0f, 1.0f);
    Vector3 toTarget(0.0f, 0.0f, 1.0f);
    std::cout << "Alignment: " << Dot(forward, toTarget) << "\n";

    return 0;
}

Expected output:

New position: (1, 1, -0.5)
New velocity: (2, -4.9, -1)
At origin? no
Alignment: 1

That last line is the payoff. position + velocity * deltaTime is exactly the update every physics engine runs each frame, and thanks to overloading it reads like the equation from a textbook. When we build the Transform component later, this Vector3 is what stores its position and scale.

Don't overdo it

Operator overloading is powerful, and that is precisely why it is easy to abuse. The golden rule: an overloaded operator should do what a reader already expects that symbol to do. If + did something surprising — say, appended to a log file — every reader of your code would be misled.

Overload only when the meaning is obvious

Adding two vectors with +? Great. Using * to mean "cross product" versus "scale"? Ambiguous — reach for a named function like Cross() instead. If you have to explain what an operator does, it probably shouldn't have been an operator. Named functions (like our Dot()) are never wrong.

A few practical guidelines to carry forward:

  • Keep operators cheap and side-effect-free (except the compound-assignment family like +=).
  • Preserve expected relationships: if you write ==, also write != in terms of it.
  • Take arguments by const& to avoid needless copies of large types.
  • When in doubt, prefer a clearly named function over a clever operator.

Recap
  • An operator is a specially named function; overloading gives it meaning for your own types.
  • Write symmetric operators (+, -, *) as free functions; write in-place ones (+=) as members.
  • operator<< must be a free function because its left operand is std::ostream.
  • Provide both v * s and s * v; C++ does not flip operands for you.
  • Some operations (like the dot product) belong in named functions such as Dot(), not operators.
  • Overload sparingly — only when the symbol's meaning is intuitive and unsurprising.
