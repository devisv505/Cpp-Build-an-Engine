Virtual Functions & Polymorphism

In the last lesson you learned how inheritance lets a derived class reuse a base class. Now we make inheritance do work: with virtual functions, a single loop over base-class pointers can call the correct behavior for every different object it holds. This is exactly how Unity and Unreal update thousands of components each frame — and it is the mechanism we will build our own component system on top of.

What you'll learn
  • What virtual and override do, and how dynamic dispatch picks the right function at runtime.
  • The vtable mental model — why calling through a base pointer is not magic.
  • Pure virtual functions to define an abstract base class (an interface).
  • The essential virtual destructor rule that prevents subtle memory leaks.
  • How to iterate mixed Component objects through one list of base pointers.

The problem: one loop, many behaviors

Imagine an engine object (an "entity") that owns several components. One component moves it, another renders it, another checks physics. Every frame the engine wants to say: "everybody, update yourselves." It should not know or care what kind of component each one is — it just calls Update() and each component does its own thing.

Without virtual functions, C++ decides which function to call based on the type of the pointer, not the type of the object it points to. That default is called static dispatch, and it gets in our way:

#include <iostream>

struct Component {
    void Update() { std::cout << "Component::Update\n"; }
};

struct Transform : Component {
    void Update() { std::cout << "Transform moved!\n"; }
};

int main() {
    Transform t;
    Component* c = &t;   // a base pointer aimed at a Transform
    c->Update();         // prints "Component::Update" — NOT what we wanted
}

Because c is declared as a Component*, the compiler picks Component::Update at compile time. The actual object being a Transform is ignored. To make the object's real type win, we opt in to dynamic dispatch.

virtual and override

Mark the base function virtual. That tells the compiler: "when this is called through a pointer or reference, look up the real object's version at runtime." In each derived class, repeat the signature and add override.

#include <iostream>

struct Component {
    virtual void Update() { std::cout << "Component::Update\n"; }
};

struct Transform : Component {
    void Update() override { std::cout << "Transform moved!\n"; }
};

int main() {
    Transform t;
    Component* c = &t;
    c->Update();   // prints "Transform moved!" — the object's real type wins
}

That one keyword changes everything. The call is now resolved from the object, so the same c->Update() line runs different code depending on what c actually points to. That is polymorphism: "many shapes" behind one interface.

Always write override

It is not required, but it makes the compiler verify that you are really overriding a base virtual with a matching signature. If you mistype the name or the parameters, you get a clear compile error instead of a function that silently never runs.

The vtable, in plain terms

How does the program know, at runtime, which Update() to call? You do not have to implement this, but the mental model kills a lot of confusion.

When a class has at least one virtual function, the compiler builds a small hidden table — the vtable — one per class. It holds the address of each virtual function's real implementation for that class. Every object of that class silently stores a hidden pointer (the vptr) to its class's vtable. A virtual call becomes: follow the object's vptr to the table, look up the slot for Update, and jump there.

The cost is tiny: one pointer per object and one indirect jump per call. In exchange you get behavior that adapts to the real object type. For an engine updating a list of components, that trade is exactly what we want.

Pure virtual functions and abstract classes

Sometimes the base class has no sensible default. What does it mean for a generic Component to "update"? Nothing on its own — only concrete components know. We express that with a pure virtual function, written = 0:

struct Component {
    virtual void Update(float dt) = 0;   // pure virtual: no body here
    virtual ~Component() = default;      // (explained in the next section)
};

A class with at least one pure virtual function is abstract: you cannot create an instance of it directly. It exists to define a contract — an interface — that derived classes must fulfill. Any class that does not implement every pure virtual stays abstract too.

Interface = pure virtual base

C++ has no separate interface keyword like C# or Java. An abstract class whose functions are all pure virtual is the C++ way to declare an interface. Our Component is one: it promises every component can Update, without saying how.

The virtual destructor rule

This is the one that bites everyone. When you delete an object through a base pointer, C++ only runs the base destructor unless the base destructor is virtual. If a derived class allocated resources, they leak silently.

#include <iostream>
#include <vector>

struct Component {
    // WITHOUT the "virtual" below, deleting via Component* skips ~Mesh.
    virtual ~Component() { std::cout << "~Component\n"; }
};

struct Mesh : Component {
    std::vector<float> vertices{1.0f, 2.0f, 3.0f}; // owns memory
    ~Mesh() override { std::cout << "~Mesh (frees vertices)\n"; }
};

int main() {
    Component* c = new Mesh();
    delete c;   // virtual dtor => runs ~Mesh THEN ~Component. Correct.
}
If you delete through a base pointer, the base destructor MUST be virtual

Forgetting it is undefined behavior — in practice a leak, because the derived destructor never runs. Rule of thumb: any class meant to be inherited from and deleted polymorphically gets a virtual ~Base() = default;. Our Component already has one.

Putting it together: a component system

Now the payoff. We define the abstract Component, write two real components, store them as base pointers in a std::vector, and update them all in one loop. Because Update is virtual, each pointer runs its own code. We use std::unique_ptr (from an earlier lesson) so cleanup is automatic and correct.

#include <iostream>
#include <memory>
#include <vector>
#include <string>

// --- The interface every component must satisfy ---
struct Component {
    virtual void Update(float dt) = 0;   // pure virtual: no default behavior
    virtual ~Component() = default;      // safe deletion via Component*
};

// --- A concrete component: moves an entity each frame ---
struct TransformComponent : Component {
    float x = 0.0f, y = 0.0f;
    float vx = 1.0f, vy = 0.5f;          // velocity units per second

    void Update(float dt) override {
        x += vx * dt;
        y += vy * dt;
        std::cout << "  Transform -> (" << x << ", " << y << ")\n";
    }
};

// --- Another concrete component: reports what it would draw ---
struct RenderComponent : Component {
    std::string mesh;
    explicit RenderComponent(std::string m) : mesh(std::move(m)) {}

    void Update(float dt) override {
        std::cout << "  Render   -> drawing '" << mesh << "'\n";
    }
};

int main() {
    // One list holds different component types as base pointers.
    std::vector<std::unique_ptr<Component>> components;
    components.push_back(std::make_unique<TransformComponent>());
    components.push_back(std::make_unique<RenderComponent>("cube.obj"));

    const float dt = 0.016f;   // ~16 ms, one frame at 60 FPS

    for (int frame = 0; frame < 2; ++frame) {
        std::cout << "Frame " << frame << ":\n";
        for (const auto& c : components) {
            c->Update(dt);     // dynamic dispatch: right Update per object
        }
    }
    // unique_ptr destroys each component; virtual ~Component ensures
    // the correct derived destructor runs. No manual delete needed.
}

Expected output:

Frame 0:
  Transform -> (0.016, 0.008)
  Render   -> drawing 'cube.obj'
Frame 1:
  Transform -> (0.032, 0.016)
  Render   -> drawing 'cube.obj'

Notice what main does not do: it never checks "is this a Transform or a Render?" It just calls Update. To add a PhysicsComponent later, you write a new derived class and push one more pointer — the update loop never changes. That extensibility is the whole point, and it is why real engines are built this way.

Don't store polymorphic objects by value

Virtual dispatch only works through a pointer or reference. If you put derived objects into a std::vector<Component> (by value), the derived part is sliced off — called object slicing — and you lose the overridden behavior. Store std::unique_ptr<Component> (or references) instead.

When you need the real type back

Occasionally you hold a Component* but need to know if it is specifically a TransformComponent. The safe tool is dynamic_cast, which returns nullptr when the object is not that type:

Component* c = components[0].get();
if (auto* t = dynamic_cast<TransformComponent*>(c)) {
    t->x = 100.0f;   // only runs if c really points to a TransformComponent
}

Reach for this sparingly — most engine code should just call virtual functions and let polymorphism do the work. But it is the correct, safe way when you genuinely need the concrete type.

Recap
  • virtual turns on dynamic dispatch, so a call through a base pointer runs the real object's overridden function.
  • Always add override on derived functions to let the compiler catch signature mistakes.
  • The vtable is the runtime mechanism: one hidden pointer per object, one lookup per virtual call.
  • A pure virtual (= 0) makes a class abstract — the C++ way to define an interface like our Component.
  • Give any polymorphic base a virtual destructor, or deleting through a base pointer leaks the derived part.
  • Store components as std::unique_ptr<Component> and iterate them through one loop — the foundation of our component-based engine.
